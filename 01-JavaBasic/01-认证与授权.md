# **认证** 与 **授权**

在 Java 生态中，用于 **认证（Authentication）** 与 **授权（Authorization）** 的框架或技术方案非常丰富，覆盖从传统单体应用到现代微服务、云原生架构的多种场景。以下是全面且结构清晰的整理，包含主流及新兴方案：

------

## 一、主流综合安全框架

### 1. **Spring Security**

- **适用**：Spring / Spring Boot 应用（事实标准）

- 特点：

  - 支持表单登录、HTTP Basic、JWT、OAuth2、OIDC、SAML、LDAP 等
- 方法级安全（`@PreAuthorize`, `@Secured`）
  - 内置 CSRF、CORS、Session 管理、密码加密等
  - 高度可扩展（自定义 `AuthenticationProvider`、`UserDetailsService` 等）
  
- **注意**：5.7+ 版本弃用 `WebSecurityConfigurerAdapter`，推荐函数式配置。

### 2. **Apache Shiro**

- **适用**：非 Spring 项目或轻量级需求

- 特点：

  - API 简洁，学习成本低
- 支持认证、授权、会话、加密四大核心功能
  - 可用于任何 Java 环境（Web、桌面、CLI）
  
- **局限**：对 OAuth2/OIDC 原生支持弱，社区活跃度下降。

### 3. **Sa-Token** ✅（国产新兴框架）

- **适用**：快速开发、前后端分离、RESTful API 场景

- 特点

  ：

  - 极简 API（如 `StpUtil.login(userId)`、`@SaCheckPermission("xxx")`）
  - 内置分布式 Session（支持 Redis）
  - 支持单点登录（SSO）、多端登录控制、Token 自动续期
  - 无 Cookie 依赖，默认基于 Token（Header/Param）
  - 零配置起步，YAML 可调

- **优势**：开发效率高，代码可读性强，适合中小团队。

- **官网**：https://sa-token.cc/

## 二、基于标准协议的身份解决方案

### 4. **OAuth2 / OpenID Connect (OIDC)**

- 典型实现

  ：

  - **Keycloak**（Red Hat 开源 IAM 平台）
    → 提供用户管理、SSO、OAuth2/OIDC/SAML，Java 应用通过适配器或标准协议集成。
  - **Auth0 / Okta / AWS Cognito**（商业云服务）
    → 托管身份服务，提供 SDK 和管理界面。
  - **Spring Authorization Server**
    → Spring 官方维护的 OAuth2 授权服务器实现（替代旧版 Spring Security OAuth）。

> Java 应用通常作为 **Resource Server**，验证 JWT 并执行权限控制。

### 5. **JWT（JSON Web Token）**

- **说明**：不是框架，而是 Token 格式标准（RFC 7519）

- 常用库：

  - `io.jsonwebtoken:jjwt`（最流行）
- `com.auth0:java-jwt`
  
- **使用方式**：常与 Spring Security 或 Sa-Token 结合，实现无状态认证。

## token认证

### **Token 认证**

**具体实现 = 基于 JWT 的 Token 认证（最主流）**

#### 一、完整流程（你描述的 exactly 就是这个）

1. 用户登录（账号密码）
2. 后端验证通过 → **生成 Token（一般是 JWT）**
3. 前端拿到 Token → **存到请求头 Header 里**（通常是 `Authorization: Bearer xxx`）
4. 后续每次请求前端**自动带上 Header**
5. 后端拦截器/过滤器 **解析 Token → 验证合法性 → 获取用户信息**

---

#### 二、它的官方标准名称（面试必须说对）

##### 1. **Token 认证（Token-Based Authentication）**

##### 2. **无状态认证（Stateless Authentication）**

##### 3. **基于 JWT 的认证（最常用）**

##### 4. **Bearer Token 认证（标准规范）**

##### ✅ 面试官问：

“用户登录生成token，放在header里带给后端，这是什么认证？”

##### ✅ 你直接答：

**这是 Token 认证，也叫无状态认证，目前主流使用 JWT 实现，前端通过 Authorization 请求头传递 Token。**

---

#### 三、它和「Session 认证」的区别（必考点）

##### 1. **Session 认证（传统）**

- 基于 Cookie
- 服务端**存储 Session**（有状态）
- 分布式需要 Session 共享（Redis）

##### 2. **Token 认证（现在微服务全用这个）**

- 基于 **Header** 传递
- 服务端**不存用户状态（无状态）**
- 信息全部存在 **JWT 里面**
- 天然支持分布式、微服务、前后端分离

---

#### 四、标准请求头长这样（你一定见过）

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

- `Authorization`：请求头名
- `Bearer`：认证类型（固定标准）
- 后面一串：**JWT Token**

---

#### 五、SpringBoot 里实现这套的技术栈（真实开发）

1. **JWT**（jjwt 库）生成/解析 token
2. **自定义拦截器 / Filter** 校验 header 中的 token
3. **ThreadLocal / SecurityContext** 存储当前用户
4. 配合框架：
   - **Sa-Token**（国内最简单）
   - **Spring Security**（标准安全框架）

---

#### 六、最终极简总结（背这一句就够）

**用户登录生成 Token，放在请求头 Authorization 中传递给后端，
这就是 SpringBoot + 前后端分离 + 微服务架构中最主流的：
👉 Token 认证（无状态认证，常用 JWT 实现）。**



### **双 Token 认证**

#### 一、双 Token 是什么？

**双 Token = 两个 Token**

1. **Access Token（短有效期，5~30分钟）**
2. **Refresh Token（长有效期，7天~30天）**

合称：**Access/Refresh Token 认证机制**
也叫：**令牌刷新机制、双令牌认证、JWT 双Token**

#### 二、为什么要有双 Token？

**单 JWT Token 有致命缺陷：**

- 有效期太短：用户频繁登录，体验差
- 有效期太长：一旦泄露，**被盗用很久都能用，极不安全**

##### 双Token完美解决：

- **Access Token 短有效**：即使泄露，风险小
- **Refresh Token 长有效**：用来**静默刷新 Access Token**
- 用户不用重新登录，安全 + 体验兼顾

#### 三、完整流程（超级重要，面试必考）

1. 用户登录 → 验证成功
2. 后端生成**两个 Token**：
   - **AccessToken（短）**
   - **RefreshToken（长）**
3. 前端同时保存两个Token
4. **每次正常请求**：
   Header 只带 **AccessToken**
5. **AccessToken 过期**：
   前端**自动用 RefreshToken 请求刷新接口**
6. 后端验证 RefreshToken 有效 → **返回新的 AccessToken**
7. 前端替换新 AccessToken → **继续请求，用户无感知**
8. RefreshToken 也过期 → **用户重新登录**

---

#### 四、核心特点（必须背）

1. **无状态认证（依然 JWT，不存 Session）**
2. **安全 + 体验平衡**
3. **微服务、前后端分离、APP 最主流认证方案**
4. 不属于 OAuth2，但 **OAuth2 核心就是双Token**

---

#### 五、双Token 属于什么认证？

##### 标准答案：

**仍然是 Token 认证（无状态认证），
是 JWT 单Token 的升级版、增强安全版。
也可以理解为：OAuth2 标准的核心机制。**

一句话：
**双Token = 更安全、更完善的 JWT 认证。**

---

#### 六、面试满分回答（背这段）

**双Token认证是指使用 AccessToken 和 RefreshToken 两个令牌的无状态认证机制：
AccessToken 短有效期，用于正常接口鉴权；
RefreshToken 长有效期，用于静默刷新 AccessToken。
既解决了单Token有效期过短导致频繁登录的问题，又降低了长期Token泄露的安全风险，
是目前前后端分离、微服务、移动端最主流的用户认证方案。**

---

#### 七、超级精简记忆

- **单Token：简单，但不安全、体验差**
- **双Token：安全、体验好、企业级标准**
- **本质：JWT 无状态认证的升级版**

### 使用 **双Token** 前端、后端实现

我讲得**超级清晰、一步到位**，面试/开发都能直接说。

---

#### 一、后端要做的修改（重点）

##### 1. 登录接口改造

**原来：只返回一个 token**
**现在：返回两个 token**
```json
{
  "accessToken": "eyxxx",    // 短有效：15~30分钟
  "refreshToken": "eyyyy"    // 长有效：7~30天
}
```

##### 2. 新增一个 **刷新token接口**（必加）

```
POST /user/refresh-token
参数：refreshToken
返回：新的 accessToken
```
作用：accessToken过期时，前端调用此接口**静默获取新token**。

##### 3. JWT 工具类修改

- 生成 **accessToken**（短过期）
- 生成 **refreshToken**（长过期）
- 两个 token 可以**secret不同**，更安全

##### 4. 拦截器（Interceptor）逻辑修改

**原来：**
- 校验token是否有效 → 有效放行
- 无效直接返回401

**现在：**
- 校验 **accessToken**
- **过期不直接返回401，而是返回固定码（如 401 或 4001）**
- 让前端知道：是**过期**，不是非法，前端可以去刷新

##### 5. 安全增强（可选但推荐）

- refreshToken 存入 Redis（黑名单/白名单）
- 支持**强制下线**、**刷新token过期**、**顶号下线**
- 因为 JWT 本身无法作废，refreshToken存Redis才能实现登出

---

#### 二、前端要做的修改（重点）

##### 1. 登录后存储两个token

- localStorage / pinia / vuex 存储：
  - accessToken
  - refreshToken

##### 2. 请求头统一携带 **accessToken**

```js
config.headers.Authorization = 'Bearer ' + accessToken
```

##### 3. **axios响应拦截器核心改造（最重要）**

**前端双Token灵魂在这里：**

流程：
1. 接口返回 **401/自定义过期码**
2. 前端判断：是否 access_token 过期
3. **如果是：**
   - 立即请求 `/refresh-token` 接口
   - 用 refreshToken 获取 **新accessToken**
   - **替换本地accessToken**
   - **自动重试刚才失败的请求**
4. 用户完全无感知（静默刷新）

##### 4. 特殊处理：防止同时多次刷新（锁）

避免并发请求同时过期，发送多次refresh。
用**锁/标志位**保证同一时间只刷新一次。

##### 5. refreshToken也过期 → 跳登录

- 刷新接口也失败 → 清除token → 跳login

---

#### 三、前后端改动总结（极简版）

##### 后端改动

1. 登录返回 **accessToken + refreshToken**
2. 增加 **/refresh-token** 刷新接口
3. 拦截器区分：**token过期 / token非法**
4. refreshToken 存入Redis（支持作废、登出、顶号）

##### 前端改动

1. 存储两个token
2. 请求头只带 accessToken
3. axios响应拦截实现：
   **过期 → 刷新 → 重试**（静默刷新）
4. 处理刷新失败 → 跳登录

---

#### 四、面试官最爱问的一句话（你这样答满分）

使用双Token时，后端需要增加刷新接口、区分token过期状态、登录返回双Token；

前端需要在响应拦截器中实现自动刷新token并重试失败请求，实现无感刷新。
同时为了解决JWT无法作废问题，refreshToken通常存入Redis实现安全控制。

## 用户认证 / 会话管理方案

我给你整理**极度清晰、面试直接背、不会混淆**的**三者对比思路**，结构简单、逻辑闭环，**面试官听了直接觉得你基础很稳**。

### 一、三者本质一句话

1. **传统单机 Session**
   服务器**内存**存用户状态，Cookie 带 `JSESSIONID`，**不支持集群**。

2. **Spring Session + Redis（分布式 Session）**
   把 Session 存 **Redis**，Cookie 带 `SESSION`（= Token），**支持集群、共享登录**。

3. **JWT Token**
   **服务端不存任何状态**，用户信息**加密放在 Token 里**，前端每次携带，**无状态**。

---

### 二、核心结构对比（最重要）

#### 1. 传统单机 Session

- 存储位置：**Tomcat 内存**
- Cookie：`JSESSIONID=xxxx`
- 结构：
  - Key：sessionId
  - Value：用户信息
- 特点：
  - 单机有效
  - 集群不同步 → **登录状态丢失**
- **缺点：微服务环境完全不能用**

---

#### 2. Spring Session + Redis（分布式Session）

- 存储位置：**Redis**
- Cookie：`SESSION=xxxx` 👉 **这个就是 Token**
- 结构：
  - Key：`spring:session:sessions:{sessionId}`（Token）
  - Value：用户信息、过期时间、最后访问时间
- 特点：
  - **多服务、多实例完全共享登录**
  - 有状态、可踢人、可下线、可查在线人数
- **本质：基于 Token 的有状态分布式登录**

---

#### 3. JWT Token

- 存储位置：**服务端不存储**
- 放在：Header / Cookie / LocalStorage
- 结构：
  - Header.Payload.Signature
  - Payload 里**直接存用户ID、权限、过期时间**
- 特点：
  - **无状态**，不依赖Redis
  - 水平扩展极强
  - **无法主动过期、无法踢人**

---

### 三、最关键区别（面试必问）

#### 1. 是否存储状态？

- 传统Session：**存内存（有状态）**
- SpringSession+Redis：**存Redis（有状态）**
- JWT：**不存储（无状态）**

#### 2. 是否支持分布式/集群？

- 传统Session：❌ **不支持**
- SpringSession+Redis：✅ **完美支持**
- JWT：✅ **天然支持**

#### 3. 能不能踢人、强制下线？

- 传统Session：能
- SpringSession+Redis：**能（删Redis即可）**
- JWT：**不能（除非配合黑名单，复杂）**

#### 4. 前端携带的是什么？

- 传统Session：JSESSIONID
- SpringSession：SESSION（就是Token）
- JWT：JWT字符串（自包含令牌）

---

### 四、最简单记忆口诀

**传统Session：内存存，单机用，集群废。**
**Redis Session：Token当Key，Redis存状态，集群共享，能踢人。**
**JWT：自身带信息，不存状态，扩展强，难踢人。**

---

### 五、企业真实使用场景（面试加分）

- **后台管理系统、内部系统：Spring Session + Redis（主流）**
- **网关、微服务对外接口、第三方登录：JWT**
- **传统单体老项目：单机Session**

---

### 六、你最关心的问题最终答案

#### **Redis 分布式 Session 是不是存 Token？**

**是！完全是！**
- Spring 自动生成一个**随机字符串作为 Token**
- 以这个 Token 为 Key 存 Redis
- 前端通过 Cookie 携带这个 Token
- 只是框架封装好了，你**不用自己生成、解析、存储**

所以：
**Spring Session = 全自动封装版的 Token+Redis 登录**

# SSO

## SSO（单点登录）

**SSO（Single Sign-On，单点登录）**，核心是**一次登录，全网（信任系统）通行**，是企业/多系统架构的核心身份认证方案。

---

### 一、核心概念（3个关键角色）

- **IdP（Identity Provider，身份提供方/认证中心）**
  唯一登录入口，负责**账号密码校验、签发令牌、维护全局会话**（如企业统一认证中心、CAS、Keycloak、Okta）。
- **SP（Service Provider，服务提供方/业务系统）**
  各个子系统（如OA、CRM、ERP、邮件），**不做登录校验，只信任IdP的令牌**。
- **Token（令牌）**
  IdP签发的**加密身份凭证**（如JWT、SAML Assertion、CAS Ticket），含用户ID、过期时间、签名，防篡改。

---

### 二、标准登录流程（Web最经典，5步闭环）

#### 场景：首次访问系统A（SP1）

1. **用户访问SP1（如OA）** → SP1检测无本地会话 → **302重定向到IdP**，并携带`redirect_uri`（登录后回跳地址）。
2. **用户在IdP登录** → 输入账号密码 → IdP校验通过 → **创建全局会话（IdP Cookie）+ 签发临时Token**（如JWT）。
3. **IdP重定向回SP1** → URL携带Token（如`?token=xxx`）。
4. **SP1验签Token** → 校验签名/过期时间/issuer → 合法则**创建本地会话（SP1 Cookie）**，返回业务资源。
5. **后续访问SP2/SP3** → 重定向到IdP → IdP发现已有全局会话 → **自动签发新Token** → SP2验通过直接登录，**无需重复输密码**。

#### 一句话流程

**访问SP → 跳转IdP → IdP登录发Token → 回SP验签 → 访问其他SP免密**。



### SSO 单点登录【通俗完整流程】

场景：公司有3个系统，**统一认证中心 = 认证中心(IdP)**
- 系统A：OA 办公
- 系统B：CRM 客户系统
- 系统C：财务系统

三个系统是**不同域名**，需要**只登一次，全部免登**。

#### 第一步：你第一次访问 OA系统（系统A）

1. 浏览器访问：`oa.xxx.com`
2. OA 后端**发现你没登录**（没有本地会话）
3. OA 自动**302重定向**到统一认证中心：
```
sso.xxx.com/login?redirect=oa.xxx.com
```

---

#### 第二步：在 SSO 认证中心 登录

1. 认证中心看你：**没有全局登录Cookie**
2. 展示登录页，你输入：账号+密码
3. 认证中心校验账号密码正确：
   - 生成**全局会话**（种下 sso.xxx.com 顶级Cookie）
   - 生成**一次性临时令牌**（Ticket / 授权码）

---

#### 第三步：认证中心 跳转回 OA

重定向回到：
```
oa.xxx.com/callback?code=授权码
```

---

#### 第四步：OA 拿到授权码，去认证中心兑换用户信息

1. OA 拿着 `code` 调用 SSO 接口：**换取用户信息+令牌**
2. 认证中心校验 code 合法，返回：用户ID、账号、JWT
3. OA 验证令牌，**自己创建本地登录会话**
4. 你正常进入 OA 首页，登录成功

---

#### 第五步：关键！再访问 CRM（系统B），体验 SSO

1. 你访问：`crm.xxx.com`
2. CRM 检测未登录，同样重定向到：
```
sso.xxx.com/login?redirect=crm.xxx.com
```
3. **重点：认证中心发现你已经有全局Cookie**
   - 不用输密码！
   - 直接生成新授权码，自动跳转回 CRM

---

#### 第六步：CRM 自动登录

CRM 拿到 code → 去SSO校验 → 拿到用户信息 → 建立自己会话
👉 **全程不用输密码，直接登录**

---

#### 第七步：退出登录（全局注销）

你在任意系统点「退出」：
1. 跳转到 SSO 注销接口
2. 清除 **全局Cookie**
3. 所有系统的登录态全部失效
下次进任何系统，都要重新输密码

---

#### 极简总结版（面试直接背）

1. 访问子系统 → 未登录 → 重定向到统一认证中心
2. 认证中心登录，生成**全局Cookie**+下发授权码
3. 携带授权码跳转回业务系统
4. 业务系统凭码换取用户信息，建立本地会话
5. 访问其他子系统，自动携带全局Cookie，免密登录
6. 登出销毁全局Cookie，全系统下线

---

#### 一句话区分

- **分布式Session**：同一个项目集群，共享Redis会话
- **SSO单点登录**：**多个不同系统、不同域名**，共用一个认证中心

结合刚才 **SSO 完整流程**，我给你精准拆解：
**IdP认证中心Cookie、各个业务系统Session、业务系统Cookie**，分别干什么、缺一不可。

---

#### cookie/session

1. **Cookie**
客户端（浏览器）存储，key=value，随请求自动携带，**解决「无状态HTTP」问题**。

2. **Session**
服务端存储（Redis/内存），存用户登录信息，**靠 Cookie 里的 JSESSIONID 关联**。

---

##### 一、SSO 里 两种核心 Cookie + 两种 Session

###### 1. 【IdP 全局 Cookie】`sso.xxx.com`

✅ 作用：**记录你「全局是否登录」**
- 你在认证中心登录后，浏览器种下 `sso 域名 Cookie`
- 后续访问任何子系统（OA/CRM/财务）
- 跳转去 sso.xxx.com 时，**浏览器自动带上这个Cookie**
- IdP 一看：有全局Cookie → 你已登录 → 直接发授权码，不用输密码

👉 **这就是 SSO 免密登录的核心**

---

###### 2. 【业务系统本地 Cookie】`oa.xxx.com` / `crm.xxx.com`

每个业务系统独立域名，**不能共享 IdP 的 Cookie**。
业务系统登录成功后，会给自己种下：
- `JSESSIONID` 或 `satoken`

✅ 作用：
当前系统**短期免登**，下次直接进首页，不用每次都跳转到认证中心。

---

###### 3. 【IdP 全局 Session】（服务端）

认证中心服务端保存：
- 全局登录态、过期时间、用户基础信息

✅ 作用：
校验全局Cookie是否合法、控制**强制下线、踢人、过期**。

---

###### 4. 【业务系统本地 Session】（服务端）

OA/CRM 各自服务端存：
- 当前登录用户ID、权限、角色
✅ 作用：
接口鉴权、获取当前用户、权限控制。

---

##### 二、套入刚才 SSO 流程，逐环节对应

1. 访问 OA 没登录
→ OA 无本地Cookie/Session → 重定向 IdP

2. IdP 登录成功
👉 写入：**IdP全局Cookie + IdP全局Session**

3. 访问 CRM 系统
→ 重定向 IdP
→ 浏览器自动携带 **IdP全局Cookie**
→ IdP 查询全局Session：已登录
→ 直接放行，下发授权码

4. CRM 拿到码登录
👉 写入：**CRM本地Cookie + CRM本地Session**

---

##### 三、关键区别（面试必问）

1. **IdP Cookie（跨系统）**
统一登录凭证，所有系统共用，实现**一次登录，多系统通行**

2. **业务系统 Cookie（独立）**
各玩各的，只负责自己系统短期免登

3. **Session 都在服务端**
- 全局Session：认证中心管
- 本地Session：各个业务系统自己管

---

##### 四、极简背诵版（直接背）

1. **IdP全局Cookie**：
浏览器保存，访问认证中心自动携带，**标记全局登录态，实现免密跳转**。
2. **业务系统Cookie**：
保存当前系统会话标识，**单个系统免登**。
3. **全局Session**：
认证中心服务端存储，管控全局登录、过期、下线。
4. **本地Session**：
各业务系统存储，用于接口鉴权、获取当前用户信息。

---

##### 五、延伸：为什么不能只用 JWT 代替 Cookie？

JWT 无状态，**无法主动注销、踢下线**；
SSO 必须靠「**全局Cookie + 服务端Session**」才能实现退出全系统下线。

---

### 三、核心原理：信任链+双会话

#### 1. 双会话机制（安全关键）

- **全局会话（IdP）**：IdP的Cookie，**一次登录全局有效**，控制所有SP的免密权限。
- **本地会话（SP）**：各SP独立Cookie，**仅当前系统有效**，避免全局会话泄露导致所有系统沦陷。

#### 2. 令牌防篡改

Token（如JWT）**Header+Payload+签名**，SP用IdP的**公钥/密钥**重算签名对比，**篡改即验签失败（401）**。

---

### 四、主流实现协议（企业必知）

#### 1. CAS（Central Authentication Service，最经典）

- 流程：**SP→IdP→Ticket→SP→IdP验Ticket→登录**
- 特点：**简单、轻量、Java生态友好**，老企业常用。

#### 2. SAML 2.0（企业级标准）

- 基于XML，**跨域/跨公司SSO首选**（如企业登录第三方SaaS）
- 特点：**安全成熟、支持联邦身份**，但移动端适配差。

#### 3. OIDC（OpenID Connect，现代主流）

- 基于OAuth2.0，用**JWT作ID Token**，**移动端/APP友好**
- 特点：**轻量、JSON格式、支持无密码登录**，互联网公司首选。

#### 4. 社交SSO（如微信/支付宝/Google登录）

- 本质是**OIDC简化版**，第三方IdP（微信）签发Token，业务系统验签登录。

---

### 五、优缺点（面试高频）

#### ✅ 优点

1. **体验拉满**：一次登录，多系统免密，告别密码疲劳。
2. **安全可控**：**集中认证+统一MFA（双因素）**，离职一键注销所有权限。
3. **成本降低**：不用每个系统开发登录模块，用户/权限集中管理。

#### ❌ 缺点

1. **单点故障**：IdP挂了，**所有系统都无法登录**（需集群部署）。
2. **安全依赖**：IdP被攻破，**所有信任系统沦陷**（需强加密+IP白名单）。
3. **跨域复杂**：多域名/跨公司场景，需处理**Cookie跨域、回调地址配置**。

---

### 六、和JWT的关系（别混淆）

- **JWT是Token格式**（可存用户信息、签名防篡改）。
- **SSO是认证方案**，**常用JWT作Token**（OIDC默认用JWT）。
- 简单说：**SSO是“方案”，JWT是“工具”**。

---

### 七、常见面试题

1. **SSO和分布式Session的区别？**
   - SSO：**跨系统/跨域名**，依赖IdP+Token；
   - 分布式Session：**同域名/同集群**，Session共享（如Redis），无法跨域。

2. **IdP的Cookie安全吗？**
   - 必须设**HttpOnly+Secure+SameSite=Strict**，防止XSS窃取和CSRF攻击。

3. **Token为什么要设过期时间？**
   - 防止Token泄露后长期有效，**短有效期（5-15分钟）+ 刷新Token**平衡安全和体验。

---

## 单点登录下如何保存用户登录状态

  目前市面上很多公司是分布式系统架构，这种架构下采用的是单点登录方式来实现用户的一次登录多处使用，那么单点登录下使用什么方案来存储用户的登录状态呢？下面介绍几种常见的实现方案。

### **1、认识单点登录**   

  单点登录是为了实现在分布式系统当中，用户登录了其中的一个子系统后就可以正常的去访问其他的子系统，如下图所示：

![图片](https://mmbiz.qpic.cn/sz_mmbiz_png/brUWWGkv917KLC7geiaib1GhQjrTLicTYWqeFy7zRSmmyXrXPrnzfm05LKicNC6BjaKNWxPsuIEazmOBbr54nxibfIg/640?wx_fmt=png&from=appmsg&tp=wxpic&wxfrom=5&wx_lazy=1#imgIndex=0)

  用户在A系统中登录之后，再去请求B系统的时候就无需用户再次执行登录操作，大大的提高了用户的体验。

### **2、cookie+session实现用户登录状态的存储**

  实现单点登录最简单的是cookie+session的方案，方案的实现原理图如下图所示：

![图片](https://mmbiz.qpic.cn/sz_mmbiz_png/brUWWGkv917KLC7geiaib1GhQjrTLicTYWq3ErmpXAYvibPX79icnI7IQC2xBdkUQHRZmwdaCX4NibO4iaPEibicAhTZDSg/640?wx_fmt=png&from=appmsg&tp=wxpic&wxfrom=5&wx_lazy=1#imgIndex=1)

  当用户请求后端系统的时候，此时在分布式系统环境下，用户的请求首先会到达网关进行统一的认证授权，网关解析请求的cookie中存储的sessionId，如果sessionId不存在，或者在认证中心检测到sessionId无效，以及session已经过期等情况下，系统会重定向到登录页面要求用户登录。

  当用户执行登录请求的时候会将用户名密码发送给认证中心进行用户名和密码的验证，如果验证成功，认证中心会存储session信息。然后认证中心将sessionId存入到客户端的cookie当中，这样就完成了用户状态的存储。

  当用户再次访问后端系统，网关首先获取cookie中的sessionId，然后携带sessionId去请求认证中心，认证中心验证sessionId是否有效，如果验证成功会返回用户信息给网关，然后网关再去转发请求到后端对应的服务上执行。

  虽然cookie+session实现单点登录是一种非常简单的实现方案，但是这样方案存在一定的缺陷，如果网站的用户量非常的大，此时认证中心服务的压力变大，为了保证认证中心服务的高可用，此时需要对认证中心做进行集群部署，如下图所示：

![图片](https://mmbiz.qpic.cn/sz_mmbiz_png/brUWWGkv917KLC7geiaib1GhQjrTLicTYWqc5oe9kfZILTt1HT3dF4fj5rjicBaeBmK1ejT5BbuY4PMMFzibc5nxBOg/640?wx_fmt=png&from=appmsg&tp=wxpic&wxfrom=5&wx_lazy=1#imgIndex=2)

  对认证中心做集群部署后，每个认证中心服务上的session信息是不同步的，此时cookie+session实现单点登录就会存在问题。

# JWT

### JWT （JSON Web Token）

JWT（JSON Web Token）是 **RFC 7519** 定义的开放标准，用于在网络中**安全、紧凑、自包含**地传递 JSON 格式的声明信息，核心用于**身份认证与授权**，解决传统 Session-Cookie 的分布式、跨域、移动端适配难题。

---

#### 一、核心定位与特点

##### 1. 核心作用

- **身份认证**：登录后签发，客户端携带，服务端验签即识用户
- **授权**：令牌内包含角色/权限，网关/微服务直接校验
- **数据交换**：安全传递用户ID、租户ID等非敏感业务数据

##### 2. 关键特性

- **无状态**：服务端不存会话，只验签名，水平扩展无压力
- **自包含**：令牌自带用户信息，无需查询中心
- **跨平台/跨域**：基于 HTTP 头传递，适配 Web/APP/小程序
- **防篡改**：签名保证内容不可改，改则验签失败

---

#### 二、JWT 结构（三段式：Header.Payload.Signature）

格式：`eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c`

##### 1. Header（头部）：元数据

指定**令牌类型**与**签名算法**，JSON → Base64Url 编码。
```json
{
  "alg": "HS256", // 签名算法：HS256(对称)/RS256(非对称)
  "typ": "JWT"     // 令牌类型，固定为JWT
}
```

##### 2. Payload（负载）：核心数据（Claims）

存**声明信息**，分三类，JSON → Base64Url 编码（**不加密，禁存密码**）。
###### （1）注册声明（标准预定义）

| 字段  | 说明                              |
| ----- | --------------------------------- |
| `iss` | 签发者（如 auth-server）          |
| `exp` | 过期时间（Unix 时间戳，必须校验） |
| `sub` | 主题（通常是用户ID）              |
| `aud` | 受众（接收方系统）                |
| `iat` | 签发时间                          |
| `jti` | 令牌ID（防重放/黑名单）           |

###### （2）公共声明：自定义公开字段

`name`、`role`、`tenant_id` 等。

###### （3）私有声明：双方约定的业务字段

如 `department`、`permissions` 等。

示例 Payload：
```json
{
  "sub": "user_123",
  "name": "张三",
  "role": "admin",
  "iat": 1710000000,
  "exp": 1710009000
}
```

##### 3. Signature（签名）：安全核心

防止前两段被篡改，生成公式（以 HS256 为例）：
```
Signature = HMAC-SHA256(
  Base64Url(Header) + "." + Base64Url(Payload),
  密钥（Secret）
)
```
- **对称加密（HS256）**：签发与验签共用同一密钥，适合单体/信任内网
- **非对称加密（RS256）**：私钥签发、公钥验签，适合微服务/跨系统

---

#### 三、工作流程（登录→请求→验签）

1. **用户登录**：账号密码 → 认证中心校验通过
2. **签发 JWT**：生成 Header+Payload → 签名 → 返回给客户端
3. **客户端存储**：存于 localStorage / Cookie（HttpOnly 更安全）
4. **请求携带**：每次请求在 Header 加 `Authorization: Bearer <token>`
5. **服务端验签**：
   - 拆分三段，Base64Url 解码 Header/Payload
   - 用密钥/公钥重算签名，对比是否一致
   - 校验 `exp`（未过期）、`iss`/`aud`（合法）
6. **处理响应**：验签通过 → 执行业务；失败 → 401 未授权



#### **JWT 到底是怎么“验签”的？为什么能防篡改？**

##### 一、先记住一句话

**验签 = 重新算一遍签名，和传过来的签名对比。
一样 = 没被改；不一样 = 被篡改！**

---

##### 二、JWT 长这样（三段）

```
Header.Payload.Signature
```

例子：
```
aaaaa.bbbbb.ccccc
```

- 第一段 `aaaaa` = Header（Base64）
- 第二段 `bbbbb` = Payload（Base64）
- 第三段 `ccccc` = **签名**

---

##### 三、服务端验签的 5 个真实步骤（最核心）

###### 第1步：把 Token 按 `.` 拆成 3 段

```
aaaaa . bbbbb . ccccc
```

得到：
- headerStr = `aaaaa`
- payloadStr = `bbbbb`
- clientSignature = `ccccc`

---

###### 第2步：服务端用自己的密钥，重新计算签名

这是**最关键的一步**！

服务端做：
```
新签名 = 加密算法( headerStr + "." + payloadStr, 密钥 )
```

比如 HS256：
```
新签名 = HMAC-SHA256( "aaaaa.bbbbb", "你的密钥" )
```

---

###### 第3步：对比 两个签名

```
新签名 === 客户端传过来的签名(ccccc) ？
```

**结果只有两种：**

✅ 一样 = 验签通过

说明：
- Header 和 Payload **从头到尾没被改过**
- 令牌是**合法签发**的

 ❌ 不一样 = 验签失败

说明：
- Header 或 Payload **被人篡改过**
- 或令牌是**伪造的**
- 直接拒绝请求！

---

###### 第4步：验签通过后，再校验过期时间

检查 Payload 里的：
```
exp = 过期时间戳
```

当前时间 < exp → 有效
当前时间 > exp → 过期，拒绝

---

###### 第5步：全部通过 → 信任这个令牌

- 取出用户ID
- 放行请求
- 执行业务逻辑

---

##### 四、最关键的灵魂问题：

###### **为什么这样就能防篡改？**

我用最简单逻辑告诉你：

###### 假如黑客篡改 Payload

把
```
{"userId":"1001"}
```
改成
```
{"userId":"9999（管理员）"}
```

会发生什么？

1. Payload 变了
2. 拼接字符串 `header.payload` 也变了
3. **黑客不知道你的密钥，无法生成正确签名**
4. 服务端一对比 → 签名对不上 → 直接拒绝！

###### 结论：

**只要不知道密钥，任何人改任何内容，都会导致验签失败！**

---

##### 五、一张图看懂验签（超清晰）

```
前端传来：
Header.Payload.签名

服务端做：
1. 切分成三段
2. 用 Header+Payload+密钥 重新计算签名
3. 对比：新签名 == 旧签名？
   相等 → 没被改 → 合法
   不等 → 被改/伪造 → 拒绝
4. 校验过期时间 exp
5. 通过 → 放行
```

---

##### 六、最精简总结（背会就够）

**JWT 验签就 3 步：**

1. **拆分三段**
2. **重新计算签名**
3. **对比是否一致**

一致 = 安全
不一致 = 伪造/篡改

#### 四、优缺点

##### 优点

- ✅ **无状态易扩展**：服务端无会话，集群部署无压力
- ✅ **跨域适配强**：Header 传递，无 Cookie 同源限制
- ✅ **微服务友好**：网关统一验签，下游服务直接取用户信息
- ✅ **移动端适配好**：APP/小程序无需处理 Cookie

##### 缺点

- ❌ **令牌不可撤销**：无状态导致无法主动作废（需黑名单/短过期）
- ❌ **Payload 不加密**：禁存密码、手机号等敏感信息
- ❌ **体积较大**：比 Session ID 大，频繁请求略耗带宽
- ❌ **过期时间难平衡**：太短频繁登录，太长安全风险高

---

#### 五、安全最佳实践

1. **用 HTTPS**：全程加密，防中间人劫持
2. **设置短过期**：`exp` 设 15–60 分钟，配合 Refresh Token
3. **敏感数据禁入 Payload**：密码、身份证绝不存
4. **优先非对称加密**：微服务用 RS256，私钥保密、公钥分发
5. **加 JTI 防重放**：配合 Redis 黑名单实现撤销
6. **Header 防注入**：校验 `alg` 为指定算法，禁止 `none`
7. **Refresh Token 长效+安全存储**：HttpOnly Cookie 存 Refresh Token，降低泄露风险

#### **JWT 到底是什么**

JWT 全称 JSON Web Token，说白了就是一段字符串，用点号分成三段：Header、Payload、Signature。

JWT的三段结构：Header（算法类型）、Payload（用户信息）、Signature（签名），三部分Base64编码后用点号连接

- **Header** ：声明算法类型（比如 HS256）
- **Payload** ：存用户信息（用户 ID、角色、过期时间）。只是 Base64 编码，**不是加密** ——任何人拿到 token 都能解码看到内容，密码绝对不能放
- **Signature** ：用 Header + Payload + 服务端密钥算出来的签名，作用是防篡改

#### 整个验证流程：

JWT认证完整流程：用户登录→服务端签发Token→客户端存储→请求时带上Token→服务端验签

登录成功 → 服务端签发 Token → 客户端保存（一般放 localStorage）→ 每次请求在 Header 里带上 `Authorization: Bearer <token>` → 服务端验签，从 Payload 取用户信息。**全程不查数据库、不查 Redis，自给自足。** 这就是"无状态"的意思。

#### **JWT"无状态"的代价：四个致命场景**

无状态听起来很美，上线后就知道了。

**场景一：账号被盗，无法立即踢人** 。Token 有效期 2 小时，现在还剩 1 小时 45 分钟。怎么让它立刻失效？纯 JWT 做不到——已签发的 token 在过期前一直有效。

**场景二：封禁用户不生效** 。账号封了，但手里的 token 还能用。

**场景三：改密码不踢其他设备** 。用户改了密码，想让其他设备全部失效。纯 JWT 做不到。

**场景四：token 过期吞表单** 。用户填了 20 多分钟的表单，点提交，token 过期了，整个内容全丢。

**这四个场景加在一起，几乎是所有正经业务系统都会碰到的问题。**

#### **为什么要把 Token 放 Redis**

解决上面这些问题，最直接的方案是**黑名单** 。

用户被踢下线时，把 token 加进黑名单。每次请求先查黑名单，在里面就拒。黑名单存哪？内存不行（多台服务器不共享，重启丢），数据库太慢（每个请求都查一次受不了），**Redis 最合适** ——查询不到 1ms，分布式共享。

Redis 只存"已失效"的 token，平时没什么写入，查询也快。

还有更彻底的方案：**直接把 token 存 Redis，每次验证都去 Redis 查。** 这和传统 Session 没太大区别，只是借了 JWT 格式当 Session ID 用。

#### **名单 vs 白名单：两种方案的本质区别**

| 方案                     | 存什么         | 查几次 Redis | 能否主动吊销 | 是否无状态 |
| :----------------------- | :------------- | :----------- | :----------- | :--------- |
| 纯 JWT                   | 不存           | 0 次         | 不能         | 是         |
| JWT+黑名单               | 存失效的 token | 1 次         | 能           | 不完全是   |
| token 存 Redis（白名单） | 存所有 token   | 1 次         | 能           | 否         |

**黑名单** 相对轻量，正常请求不写 Redis，只有踢人时才写，多数业务选这个。**白名单** 控制力更强，能做限制同时在线设备数、实时查看在线状态，代价是每次请求都依赖 Redis。

#### **双 Token 续期方案**

Token 到期了，用户正在操作怎么办？

- **AccessToken** ：有效期短（30 分钟），做实际鉴权
- **RefreshToken** ：有效期长（7 天），只用来换 AccessToken

AccessToken 过期，前端用 RefreshToken 换新的，对用户无感。RefreshToken 存 Redis，用户改密码就删掉，自动让他下次操作时重新登录。

#### **Redis 本身可靠吗**

有人担心引入单点故障。Redis 的高可用方案跑了十几年：

- **Sentinel（哨兵模式）** ：主节点挂了自动选新主
- **Cluster（集群模式）** ：数据分片，横向扩展

Redis Cluster架构：数据分片分布在多个节点，支持横向扩展和自动故障转移

性能方面 Redis 单次查询 1ms 以内，单机 QPS 十几万。说"不能放 Redis 因为会单点故障"——这逻辑说不通。

#### **面试官到底在嘲笑什么**

两种情况。

**第一种** ：你用了 JWT 但验证时完全不走签名，每次都去 Redis 查 token 是否存在。这等于只借了 JWT 格式，没用它的能力——不如直接生成随机字符串当 Session ID，还省了编解码开销。面试官嘲笑的是这里。

**第二种** ：你确实有业务需求需要主动吊销，选了 JWT+Redis 方案。但面试里只说了"放 Redis"，没说为什么。面试官误以为你不懂无状态设计。

第一种要改方案，第二种要改表达。

#### **纯 JWT 什么时候真的合适**

**跨服务调用** ：网关层做一次验签，用户信息往下游传。订单服务、支付服务直接从 JWT 取 userId，不用再查 Redis。这里 JWT 承担的是身份传递角色，不是管理登录会话。

**一次性凭证** ：邮箱验证链接、密码重置链接、临时分享链接。签一次用一次，不需要续期和吊销。

**安全要求不高的内部工具** ：用户量少，不需要踢人下线。纯 JWT 省一层依赖。

#### **面试怎么答**

别上来就说"放 Redis"，也别上来就说"JWT 无状态"。

先说业务需求：需不需要主动踢人？改密码后需不需要其他设备立刻失效？需不需要限制同时在线设备数？

然后根据需求说方案：需要的话选 JWT+Redis 黑名单，说清楚纯 JWT 做不了主动吊销；不需要的话纯 JWT 够了，说清楚签名验证流程和密钥管理。

**面试官真正想听的不是你选了哪个，是你知不知道每个方案的边界。** "Token 放 Redis"这件事本身没问题，问题是说不出来为什么放。

## **Sa‑Token 完整登录流程**

下面我把 **Sa‑Token 完整登录流程**按“前端请求 → 后端校验 → 生成会话 → 返回 Token → 后续请求鉴权”一步步讲清楚，结合源码核心逻辑，尽量通俗、完整、贴合实际开发。

---

### 一、一句话总览

Sa‑Token 登录核心：  
**校验账号密码 → StpUtil.login(userId) → 生成唯一 Token → 存 Redis → 写 Cookie / 返回 Token → 后续请求自动鉴权**

---

### 二、完整流程（分 7 步）

#### 1. 前端提交登录请求

前端（浏览器/APP）发送：
```
POST /login
{ username: "xxx", password: "xxx" }
```

#### 2. 后端校验账号密码（业务层）

```java
// 伪代码
User user = userMapper.selectByUsername(username);
if (user == null || !BCrypt.checkpw(password, user.getPassword())) {
    return "账号或密码错误";
}
// 校验通过，进入 Sa‑Token 登录
```

#### 3. 调用 Sa‑Token 登录：`StpUtil.login(userId)`

这一行背后 Sa‑Token 做了很多事（源码核心逻辑）：

```java
// StpLogic.login(id, model)
public void login(Object id, SaLoginModel loginModel) {
    // 1. 检查是否已登录、是否互斥登录、是否踢人下线
    checkLoginSafety(id, loginModel);

    // 2. 生成唯一 token（UUID + 配置）
    String token = createToken(id, loginModel);

    // 3. 构建会话信息（Session）
    SaSession session = createSession(id, token, loginModel);

    // 4. 存储到 Redis（或内存）
    saveSessionToCache(id, token, session, loginModel.getTimeout());

    // 5. 绑定 token → userId 反向映射
    bindTokenToUserId(token, id);

    // 6. 写入 Cookie（或不写，由前端存）
    setTokenToResponse(token, loginModel);
}
```

关键细节：
- **互斥登录**：同一账号只能一端登录，新登录踢掉旧登录（默认）
- **并发登录**：可配置允许多端同时登录
- **Token 生成**：默认 UUID，可自定义、可 JWT 模式
- **存储**：默认 **Redis**（分布式、可集群、支持过期、可踢人）

#### 4. 返回 Token 给前端（两种模式）

##### 模式 A：Cookie 模式（默认，浏览器）

- 自动写入 Cookie：`satoken=xxx`
- 前端下次请求自动携带，无需手动处理

##### 模式 B：Header 模式（前后端分离/APP）

- 后端返回 JSON：
```json
{ "code": 200, "token": "xxxx" }
```
- 前端存在 `localStorage`，后续请求头携带：
```
Authorization: Bearer xxxx
```

#### 5. 后续请求鉴权（核心）

每次请求进入后端：
1. **拦截器/过滤器**自动从 Cookie 或 Header 取出 Token
2. **查 Redis**：
   - 存在且未过期 → 已登录，放行
   - 不存在/过期 → 抛出 `NotLoginException` → 401
3. 可通过注解或代码校验权限：
```java
@SaCheckPermission("user:list")
public List<User> list() { ... }
```

#### 6. 会话续期

- 每次请求通过鉴权时，**自动刷新 Redis 过期时间**（默认 30 分钟）
- 长时间不操作自动掉线

#### 7. 注销登录

```java
StpUtil.logout();
```
- 删除 Redis 会话
- 清除 Cookie
- 前端 Token 失效

---

### 三、Sa‑Token 与 JWT 的区别（简单说）

- **JWT**：自包含、无状态、不存服务端 → 无法主动踢人、无法撤销
- **Sa‑Token**：**Token + Redis 会话** → 可踢人、可禁用、可续期、支持单点登录、微服务网关鉴权

```java
}
```

### 二、权限认证模块

**解决问题**：判断用户是否拥有某权限/某角色，控制接口访问

**解决问题**：精确控制用户能访问哪些接口、能做哪些操作，实现**权限码校验、角色校验、注解式鉴权、多维度权限组合、自定义权限规则**。

---

#### 1. 权限码校验（细粒度控制）

##### 核心说明

- 权限码是**字符串标识**，如 `user:list`、`order:edit`、`goods:delete`
- 一个用户可以拥有**N个权限码**
- 框架只做**校验**，权限码的查询需要你实现 `StpInterface` 接口

##### 核心 API

```java
// 判断：是否拥有【单个权限】，返回布尔值
StpUtil.hasPermission("user:add");        

// 判断：是否拥有【全部权限】，多个权限必须同时满足
StpUtil.hasPermission("user:edit", "user:delete");  

// 强制校验：无此权限直接抛出异常
StpUtil.checkPermission("user:delete");   

// 判断：是否拥有【任意一个权限】
StpUtil.hasPermissionOr("user:list", "admin:list");
```

---

#### 2. 角色校验（粗粒度控制）

##### 核心说明

- #### 角色是**身份分组**，如 `admin`、`teacher`、`user`、`guest`
- 一个用户可以拥有**多个角色**
- 角色与权限分离：**角色绑定权限，用户绑定角色**

##### 核心 API

```java
// 判断：是否拥有【单个角色】
StpUtil.hasRole("admin");        

// 强制校验：无此角色直接抛异常
StpUtil.checkRole("super-admin");

// 判断：是否拥有【全部角色】
StpUtil.hasRole("admin", "operation");  

// 判断：是否拥有【任意一个角色】
StpUtil.hasRoleOr("admin", "user");
```

---

#### 3. 注解式权限控制（最常用、最优雅）

直接加在**Controller接口/方法**上，无需写代码逻辑。

```java
// 1. 必须登录才能访问（无权限要求，只要登录）
@SaCheckLogin                

// 2. 必须拥有【user:list】权限才能访问
@SaCheckPermission("user:list")

// 3. 必须是【admin】角色才能访问
@SaCheckRole("admin")        

// 4. 组合校验：登录 + 拥有权限 + 拥有角色
@SaCheckLogin
@SaCheckPermission("user:edit")
@SaCheckRole("admin")
@GetMapping("/user/update")
public Result update() {
    return Result.success();
}
```

##### 高级注解用法

```java
// 同时校验多个权限（AND 关系：必须全部拥有）
@SaCheckPermission({"user:add", "user:delete"})

// 同时校验多个角色（AND 关系）
@SaCheckRole({"admin", "operation"})

// OR 关系：满足任意一个即可
@SaCheckPermission(value = {"user:list", "admin:list"}, mode = SaMode.OR)
```

#### 4. 必须实现：权限数据源（StpInterface）

**关键！**
Sa-Token 不存储权限/角色，需要你**自定义查询**（从数据库查）。

##### 实现类（直接复制可用）

```java
    // 返回一个用户拥有的【权限码集合】
    @Override
    public List<String> getPermissionList(Object loginId, String loginType) {
        // 这里写你的业务逻辑：根据 userId 查询权限列表
        long userId = Long.parseLong(loginId.toString());
        return userMapper.getPermissionListByUserId(userId);
    }

    // 返回一个用户拥有的【角色集合】
    @Override
    public List<String> getRoleList(Object loginId, String loginType) {
        long userId = Long.parseLong(loginId.toString());
        return userMapper.getRoleListByUserId(userId);
    }
}
```

### 四、微服务分布式会话

**解决问题**：集群/微服务下**会话共享**，登录状态多服务器同步

**支持模式**

1. Redis 共享（最常用）
2. JWT 分布式
3. 自定义持久层

**特点**

- 无侵入接入
- 自动同步登录状态
- 分布式环境无缝使用

**解决问题**：集群/微服务、多服务器节点环境下，实现**登录状态共享、会话互通、数据同步**，保证用户在任何一台服务器登录，所有节点都能同步感知。

---

#### 1. 核心说明

微服务环境下，默认的内存会话会**不同步**，导致：
- 一台服务登录，另一台服务显示未登录
- 重启服务会话丢失
- 集群环境无法使用

Sa-Token 分布式会话就是为了解决这个问题。

---

#### 2. 三种分布式会话模式

##### ① Redis 共享模式（企业最常用、推荐）

**原理**：所有服务连接同一个 Redis，会话统一存储在 Redis 中。
**优点**：功能完整、支持踢人、封禁、在线用户、Session 存储。
**缺点**：依赖 Redis

##### ② JWT 分布式模式

**原理**：会话信息存在 Token 里，服务端无状态。
**优点**：不依赖中间件
**缺点**：无法主动踢人、无法封禁、无法统计在线用户

##### ③ 自定义持久层

**原理**：自己实现会话存储（MySQL、MongoDB 等）
**优点**：高度自定义
**缺点**：需要自己编码实现

---

#### 3. Redis 分布式会话（实战完整版）

##### ① 引入依赖

```xml

</dependency>
```

##### ② 配置 yml

```yaml

  redis:
    host: 127.0.0.1
    port: 6379
    database: 0

  # 是否开启分布式会话
  is-read-write-session: true
```

##### ③ 效果

- 登录状态**所有服务实时同步**
- 踢人、封禁、在线用户、Session 全部互通
- 重启服务会话不丢失
- 集群/微服务环境完美运行

#### 5. 核心特性（微服务必备）

##### ① 无侵入接入

只需要加依赖 + 配置，**原有代码一行不用改**。

##### ② 登录状态自动同步

##### ③ 会话持久化

重启服务、发布新版本，**会话不丢失**。

##### ④ 统一会话控制

在任意服务执行踢人/封禁，所有服务立即生效。

##### ⑤ 支持所有微服务架构

Spring Cloud、Spring Cloud Alibaba、Dubbo、Solon 等全都支持。

#### 6. 核心 API（分布式环境完全通用）

#### 1. 核心概念

- **SSO**：Single Sign On 单点登录
- **认证中心（Server）**：统一登录页面，统一鉴权
- **客户端（Client）**：业务系统A、B、C
- **效果**：登录一个系统 → 所有系统自动登录

---

#### 2. 支持的三种模式

##### ① Sa-Token 原生 SSO（推荐、最简单）

- 轻量、无冗余、开箱即用
- 支持：统一登录、统一注销、ticket 校验
- 无需复杂配置，适合内部系统

##### ② OAuth2.0 标准 SSO

- 遵循国际标准
- 适合开放平台、第三方接入
- 支持 scope 权限、refresh_token

##### ③ 第三方登录（微信/支付宝/钉钉/QQ）

- 社会化登录
- 扫码登录、快捷登录
- 由 Sa-Token 全封装

---

#### 3. 核心角色

1. **SSO Server（认证中心）**
   - 提供登录页
   - 鉴权、发放票据
   - 管理全局会话
2. **SSO Client（客户端系统）**
   - 业务系统
   - 未登录时自动跳转到认证中心
   - 登录成功自动创建本地会话

**效果：**
访问系统A → 未登录 → 自动跳转认证中心登录 → 登录成功自动跳回系统A → 再访问系统B/C/D **无需登录**

---

#### 4. 核心流程（必懂）

1. 客户端用户未登录 → 重定向到认证中心
2. 认证中心检查是否已登录：
   - 已登录 → 直接发放票据 ticket
   - 未登录 → 显示登录页，登录后发放 ticket
3. 客户端拿着 ticket 去认证中心校验
4. 校验通过 → 客户端创建本地会话 → 登录成功

---

#### 5. 核心 API（极简）

##### 服务端（认证中心）

```java
// 构建 SSO 认证中心
@SaCheckLogin
@RequestMapping("sso/auth")
public String auth(String redirect, String... otherParams){
    return SaSsoUtil.buildRedirectUrl(loginId, redirect);
}
```

##### 客户端（业务系统）

```java
// SSO 回调校验
@RequestMapping("sso/callback")
public Object callback(String ticket, String back){
    SaSsoUtil.checkTicket(ticket);
    return back;
}
```

---

#### 6. 统一注销（一处退出、全平台退出）

```java
// 认证中心注销
StpUtil.logout();

// 客户端注销（自动同步所有系统）
StpUtil.logout();
```

#### 7. 特点

- 无侵入接入
- 支持集群/分布式
- 支持 Redis 会话共享
- 支持前后端分离
- 支持小程序、APP、Web
- 安全：ticket 一次性、防重放、防伪造

#### 8. 适用场景

- 企业内部多管理系统
- 微服务多子系统
- 多品牌、多站点互通

### 六、OAuth2.0 认证授权

**解决问题**：第三方应用授权登录、开放平台授权

**支持 OAuth2.0 四种模式**

1. 授权码模式（最常用）
2. 隐藏式模式
3. 密码模式
4. 客户端模式

**核心能力**

- 创建 Client
- 生成 Token
- 刷新 Token
- 权限范围控制

**解决问题**：为第三方应用、开放平台、外部系统提供**安全授权登录**，实现“用XX账号登录”功能，严格遵循 OAuth2.0 国际标准。

---

#### 1. 核心概念

- **开放平台**：提供用户数据的一方（如微信、QQ、你的统一认证中心）
- **第三方应用**：需要调用用户数据的外部系统
- **授权**：第三方应用**只能访问用户允许的权限**，无法拿到账号密码
- **Token**：授权凭证，第三方使用 Token 调用接口

---

#### 2. 支持 OAuth2.0 四种标准模式（完整版）

##### ① 授权码模式（最安全、最常用、企业首选）

**流程**：
1. 第三方跳转开放平台登录页
2. 用户登录、授权
3. 开放平台返回一次性**授权码 code**
4. 第三方用 code 换取**Access Token**
5. 使用 Token 调用用户信息

**适用**：官网登录、开放平台、第三方授权登录

##### ② 隐藏式模式

- 直接返回 Token，无授权码
- 适用于纯前端应用

##### ③ 密码模式

- 直接提交账号密码获取 Token
- 适用于**自家内部系统**信任授权

##### ④ 客户端模式

- 无需用户，应用直接凭 Client ID/Secret 获取 Token
- 适用于服务间调用

---

#### 3. 核心能力（Sa-Token 完整实现）

- 创建、管理第三方 Client（应用）
- 生成 AccessToken / RefreshToken
- 刷新 Token 续期
- 权限范围 scope 控制
- 授权记录管理
- 自动过期、自动回收

---

#### 4. 核心 API（直接可用于开发）

##### 1）创建 Client 应用

```java
SaOAuth2Util.createClient(
    "app1",                // clientId
    "123456",              // clientSecret
    "http://localhost:8080/callback",  // 回调地址
    "userinfo,order,phone"            // 权限范围
);
```

##### 2）生成授权码（Authorization Code）

```java
String code = SaOAuth2Util.authorize(
    "app1",
    "userinfo,order",
    loginId,    // 当前用户ID
    "state"     // 随机值
);
```

##### 3）用授权码换取 Token

```java
SaOAuth2Token token = SaOAuth2Util.getTokenByCode(
    "app1", 
    "123456", 
    code
);

// 获取结果
token.getAccessToken();  // 访问令牌
token.getRefreshToken(); // 刷新令牌
token.getExpiresIn();    // 过期时间
token.getScope();        // 权限范围
```

##### 4）刷新 Token

```java
SaOAuth2Token newToken = SaOAuth2Util.refreshToken("app1", refreshToken);
```

##### 5）校验 Token 并获取用户ID

```java
Object loginId = SaOAuth2Util.getLoginIdByToken(accessToken);
```

##### 6）回收 Token（强制失效）

```java
SaOAuth2Util.revokeToken(accessToken);
```

---

##### 5. 权限范围 scope 控制

```java
// 校验是否拥有某个权限
SaOAuth2Util.checkScope(accessToken, "userinfo");

// 校验必须拥有多个权限
SaOAuth2Util.checkScope(accessToken, "userinfo", "order");
```

---

#### 6. 注解式 OAuth2 鉴权（接口保护）

```java
// 必须携带有效的 AccessToken 才能访问
@SaCheckOAuth2

// 必须携带 Token 且拥有 userinfo 权限
@SaCheckOAuth2(scope = "userinfo")

// 必须同时拥有 userinfo 和 order 权限
@SaCheckOAuth2(scope = {"userinfo", "order"})
@GetMapping("/user/info")
public Result userInfo() {
    return Result.success();
}
```

### 七、Token 生成与管理

**解决问题**：自定义 Token 格式、有效期、续签策略

**Token 风格**

- UUID（默认）
- 简单Token
- JWT Token
- 自定义 Token

**自动续签**

- 活跃会话自动延长有效期
- 不活跃自动过期

**解决问题**：自定义 Token 格式、生成策略、存储方式、有效期控制、自动续签、安全加固，全面掌控登录凭证生命周期。

---

#### 1. Token 风格（4 种官方内置）

##### ① UUID Token（默认，最安全）

- 随机 32 位字符串，无规律、不可伪造
- 服务端存储，安全度最高
- 支持踢人、封禁、在线用户统计
```
token: 621e6b9ad5c94a36b4c86781f6f87d49
```

##### ② 简单 Token（短 Token）

- 更短、体积更小，适合传输
- 安全性略低于 UUID
```
token: Q6G1K8Z4
```

##### ③ JWT Token（无状态分布式）

- 自包含信息，服务端无需存储
- 适合无状态微服务、网关
- **不支持踢人、在线用户**
```
token: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

##### ④ 自定义 Token（完全可控）

- 自己定义生成规则
- 支持业务前缀、时间戳、用户ID 拼接
```
token: USER_10001_1738764722988_S6sK2z
```

---

#### 2. Token 生成策略（配置即可）

##### application.yml 配置

```yaml
sa-token:
  # Token 风格：1=UUID 2=简单Token 3=JWT 4=自定义
  token-style: 1

  # Token 名称（请求头/参数名）
  token-name: Authorization

  # Token 前缀（如 Bearer）
  token-prefix: Bearer

  # JWT 秘钥（只有 token-style=3 才需要）
  jwt-secret-key: asdf5j4k32n5l2k34n5k23n5l2k3n45
```

---

#### 3. 自动续签（核心机制）

##### 作用

用户只要在操作，就**永不掉线**；长时间不操作，自动过期。

##### 核心配置

```yaml
sa-token:
  # Token 有效期（秒）默认30天
  timeout: 2592000

  # 活跃超时时间（秒）：2小时无操作则过期
  activity-timeout: 7200

  # 是否自动续签：默认true
  auto-renew: true
```

##### 机制

- 每次请求接口 → 自动刷新 **最后活跃时间**
- 活跃时间内 → 续期 Token
- 超过活跃时间 → 会话失效

---

#### 4. Token 管理核心 API

##### 获取当前 Token

```java
// 获取当前用户 Token
String token = StpUtil.getTokenValue();

// 获取 Token 剩余有效期（秒）

```

##### 手动续签

```java
// 手动续签 Token

```

##### 强制失效 Token

```java
// 让一个 Token 立即失效（踢人）
StpUtil.logoutByTokenValue(token);
```

##### 检查 Token 是否有效

```java
boolean isValid = StpUtil.isValidToken(token);
```

### 八、路由拦截鉴权

**解决问题**：全局拦截接口，统一校验登录/权限

支持所有 Java Web 框架：
- SpringBoot
- SpringMVC
- Solon
- JFinal
- Play

**示例（SpringBoot）**
```java
@Configuration
public class SaTokenConfig implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new SaInterceptor()).addPathPatterns("/**");
    }
}
```

**解决问题**：不用在每个接口写校验代码，通过**全局路由拦截**统一校验：登录状态、权限、角色、注解鉴权，实现安全控制。

---

#### 1. 核心作用

- 拦截所有请求，统一鉴权
- 自动生效 `@SaCheckLogin`、`@SaCheckPermission` 等注解
- 统一处理：未登录、无权限、无效Token
- 支持白名单放行（登录接口、验证码、静态资源）

---

#### 2. SpringBoot 完整版配置（可直接复制）

```java
import cn.dev33.satoken.interceptor.SaInterceptor;
import cn.dev33.satoken.stp.StpUtil;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.InterceptorRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
public class SaTokenConfig implements WebMvcConfigurer {

    /**
     * 注册 Sa-Token 全局拦截器
     */
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new SaInterceptor(handler -> {
            // 自定义校验逻辑（写在这里就会全局校验）
            StpUtil.checkLogin(); // 全局校验登录
        }))
        .addPathPatterns("/**")           // 拦截所有路由
        .excludePathPatterns(             // 放行白名单
            "/user/login",                // 登录接口
            "/user/register",             // 注册接口
            "/captcha/**",                // 验证码接口
            "/file/**",                   // 文件资源
            "/test/**",                   // 测试接口
            "/swagger-resources/**",      // swagger
            "/webjars/**",
            "/v3/api-docs/**"
        );
    }
}
```

### 九、全局异常处理

Sa-Token 提供统一异常，方便返回前端规范格式

```java
@RestControllerAdvice
public class GlobalException {
    @ExceptionHandler(NotLoginException.class)
    public Result<String> handler(NotLoginException e) {
        return Result.fail("未登录");
    }
}
```

**解决问题**：捕获 Sa-Token 所有认证/权限/会话异常，统一返回标准格式给前端，避免系统抛出原生异常、页面报错或乱码。

---

#### 1. 核心作用

- 捕获所有 Sa-Token 异常
- 统一返回 JSON 格式
- 前端可根据状态码做跳转（如未登录自动跳登录页）
- 不侵入业务代码，全局一份配置即可

---

#### 2. 完整版全局异常处理（直接复制可用）

包含：**未登录、无权限、无角色、封禁、踢人、二级认证、限流、自定义异常**全覆盖。

```java
import cn.dev33.satoken.exception.*;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@RestControllerAdvice
public class SaGlobalException {

    // =================== 1. 未登录 ===================
```

## sa-token 方法

### **为什么 StpInterfaceImpl 只能放在 Auth、Biz 服务，网关绝对不能放？**

---

#### 一、先记住一句话

**网关只认 Token，不认权限；
权限是业务服务自己的事。**

---

#### 二、3 个核心原因（一看就懂）

##### 1. 网关是**响应式（Reactor）**，不支持普通 StpInterface

- Gateway 用的是：`sa-token-reactor`
- 而 `StpInterface` 是给 **Servlet 普通Web服务**用的
- 网关里写了**也不会生效，还会报错**

##### 2. 网关**只做登录校验，不做权限校验**

网关只干一件事：
**你带 Token 了吗？带了就放行，没带就拦截。**

网关不管：
- 你是什么角色
- 你有什么权限
- 你能访问什么接口

**权限 → 是业务服务自己决定的**

##### 3. 每个微服务权限不一样，网关管不过来

微服务架构：
- order 服务：有订单权限
- user 服务：有用户权限
- pay 服务：有支付权限

网关怎么可能知道所有服务的权限？
**不可能，也不应该。**

所以：
**谁需要校验权限，谁就自己写 StpInterfaceImpl**

---

#### 三、最清晰的分工（背会就永远不踩坑）

##### ✅ Gateway 网关（只做登录）

- 校验：**是否登录**
- 不做：**角色、权限**
- 不放：`StpInterfaceImpl`

##### ✅ Auth 认证服务（登录签发Token）

- 登录成功后需要知道：**你有什么权限**
- 所以必须放：`StpInterfaceImpl`

##### ✅ Biz 业务服务（细粒度权限）

- 接口上加了：
  - `@SaCheckPermission`
  - `@SaCheckRole`
- 必须知道当前用户权限 → 必须放：`StpInterfaceImpl`

---

#### 四、用流程讲一遍（秒懂）

1. 前端请求 → 网关
2. 网关：**看一眼 Token，合法就放行**
3. 请求到达业务服务
4. 业务服务：
   - 调用 `StpInterfaceImpl`
   - 查询当前用户**角色、权限**
   - 判断你能不能访问这个接口

---

#### 五、最终结论（超级精简版）

##### **网关 = 登录校验（不关心权限）**

##### **业务服务 = 权限校验（必须 StpInterface）**

所以：
- **Gateway：不放 StpInterfaceImpl**
- **Auth、Biz：必须放 StpInterfaceImpl**

### StpInterfaceImpl 

（结合 SpringCloud + Sa-Token 底层原理）

#### 一、核心接口是什么

```java
public interface StpInterface {
    // 获取当前用户【权限集合】
    List<String> getPermissionList(Object loginId, String loginType);

    // 获取当前用户【角色集合】
    List<String> getRoleList(Object loginId, String loginType);
}
```
- 由 **Sa-Token 框架提供**
- 属于**扩展回调接口**，框架**只定义、不实现**
- 必须由**开发者手动实现**并注入 Spring 容器

---

#### 二、核心作用（本质）

Sa-Token 只会从 Redis 读取：
- `loginId` 登录ID
- 登录状态、Token 有效期

👉 **Sa-Token 框架本身没有任何业务库，不知道：**
1. 当前用户有哪些权限？
2. 当前用户有哪些角色？

**`StpInterface` 唯一作用：**
> 当执行「角色/权限校验」时，**回调此接口**，让开发者**手动给框架返回当前用户的权限、角色数据**。

---

#### 三、什么时候会触发这个接口？

只要用到以下注解/代码，**必触发回调**：

##### 1. 注解方式（项目最常用）

```java
@SaCheckRole("admin")                // 校验角色
@SaCheckPermission("system:add")      // 校验权限
```

##### 2. 代码手动校验

```java
// 校验权限
StpUtil.checkPermission("system:add");
// 校验角色
StpUtil.checkRole("admin");

// 判断是否拥有
StpUtil.hasPermission("xxx");
StpUtil.hasRole("xxx");
```

##### 3. 关键结论

- 只做**登录校验** `@SaCheckLogin` / `StpUtil.checkLogin()`
  ✅ **不会调用 StpInterface**
- 只要涉及**角色、权限**
  ✅ **一定会调用 StpInterface**

---

#### 四、方法参数详解

##### 1. 入参

- `loginId`：当前登录用户唯一标识（你登录时 `StpUtil.login(10001)` 存的ID）
- `loginType`：多账号体系标识（默认 `login`，多用户类型才用到）

##### 2. 执行流程

1. 拦截器/注解触发权限校验
2. Sa-Token 从上下文拿到当前 `loginId`
3. 自动调用你实现的 `StpInterfaceImpl`
4. 你根据 `loginId` 查询数据库/缓存，返回权限、角色集合
5. 框架内部对比：`所需权限` vs `用户拥有权限`
6. 比对失败 → 抛出 `NotPermissionException / NotRoleException`

---

#### 五、为什么网关 **不需要、也不能** 实现？

##### 1. 架构分工

- **Gateway 网关**
  只做：Token 有效性校验、是否登录、路由、跨域
  不做：业务权限、角色判断
- **下游微服务（Auth / Biz / Order / User）**
  各自业务权限隔离，谁要做权限校验，谁自己实现

##### 2. 技术不兼容

- Gateway 基于 **WebFlux 响应式**
- 普通 `StpInterface` 是 **Servlet 环境** 接口
- 网关使用 `sa-token-reactor` 响应式依赖
- 强行实现会：报错、不生效、上下文丢失

##### 3. 数据无法获取

网关一般**不连接业务数据库**，无法根据 loginId 查询权限；
权限数据只能在**业务服务**内查询。

---

#### 六、为什么 Auth服务、Biz服务 必须实现？

##### 1. Auth 认证服务

- 存放用户核心信息、角色权限数据源
- 后续微服务可以**远程调用 Auth 服务**拉取权限

##### 2. Biz 业务服务

接口使用了：
`@SaCheckPermission` / `@SaCheckRole`
👉 框架必须回调此接口拿权限数据，**不实现直接报错/放行失效**

---

#### 八、微服务最佳方案（企业级）

##### 问题

每个业务服务都写一遍 SQL 查权限，冗余、不好维护。

##### 标准方案

1. **统一权限数据源：Auth 认证服务**
2. 所有业务服务的 `StpInterfaceImpl` 不查库
3. 通过 **Feign远程调用 Auth服务**，根据 loginId 获取权限角色
4. 加入 Redis 缓存，减少远程调用、提升性能

```text
Biz服务权限校验 
→ 回调 StpInterfaceImpl 
→ Feign调用 Auth服务 
→ 查询库/缓存 
→ 返回权限集合
→ Sa-Token完成比对
```

#### 九、常见坑（高频报错）

1. **只加注解，不实现 StpInterface**
   结果：权限直接放行、拦截失效
2. **网关中实现此类**
   结果：环境不兼容、空指针、不回调
3. **多服务 token-name / Redis 不一致**
   结果：loginId 读取不到，权限为空
4. **未加 @Component**
   结果：框架无法注入，接口不会被回调

---

#### 十、一句话极简总结

1. `StpInterface` 是 **Sa-Token 权限数据回调接口**
2. 作用：**给框架提供当前用户的角色、权限列表**
3. 只在 **Servlet 微服务**生效（Auth、Biz）
4. **Gateway 响应式环境不支持、不需要、不实现**
5. 只要用 `@SaCheckRole / @SaCheckPermission`，必须实现该接口。

###  `SaTokenGatewayConfig` 

（Gateway 专用，Reactor 环境）。

---

#### 一、SaTokenGatewayConfig 是什么？

它是 **Spring Cloud Gateway + Sa-Token 的核心配置类**，作用：

1. 注册 **SaReactorFilter**（Gateway 专用过滤器）
2. 统一**网关鉴权（登录校验）**
3. 统一**跨域、全局异常、路由放行**
4. 对接 **Redis 分布式会话**
5. **不做权限校验**（权限交给下游服务的 `StpInterfaceImpl`）

一句话：
> **网关只在这里做“登录拦截 + 路由放行 + 统一异常”，不碰权限。**

---

#### 二、Gateway 与普通服务的本质区别

| 环境                           | 过滤器            | 接口               | 用途                |
| ------------------------------ | ----------------- | ------------------ | ------------------- |
| **Gateway（WebFlux/Reactor）** | `SaReactorFilter` | 无 `StpInterface`  | 只校验 **是否登录** |
| **Auth/Biz（Servlet）**        | `SaServletFilter` | `StpInterfaceImpl` | 校验 **角色/权限**  |

所以：
- 网关用 **SaReactorFilter**
- 网关 **绝对不能** 写 `StpInterfaceImpl`

---

#### 三、完整可上线的 SaTokenGatewayConfig（带注释）

```java
import cn.dev33.satoken.context.SaHolder;
import cn.dev33.satoken.reactor.filter.SaReactorFilter;
import cn.dev33.satoken.router.SaRouter;
import cn.dev33.satoken.stp.StpUtil;
import cn.dev33.satoken.util.SaResult;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.http.HttpStatus;
import org.springframework.web.server.ServerWebExchange;
import reactor.core.publisher.Mono;

/**
 * Gateway 网关 Sa-Token 配置
 * 职责：登录校验、路由放行、跨域、全局异常
 * 不做：权限校验（交给下游服务 StpInterfaceImpl）
 */
@Configuration
public class SaTokenGatewayConfig {

    /**
     * 注册 Gateway 专用过滤器：SaReactorFilter
     */
    @Bean
    public SaReactorFilter saReactorFilter() {
        return new SaReactorFilter()
                // 1. 拦截所有请求
                .addInclude("/**")

                // 2. 放行白名单（登录、注册、swagger、健康检查）
                .addExclude(
                        "/auth/login",
                        "/auth/register",
                        "/favicon.ico",
                        "/doc.html",
                        "/webjars/**",
                        "/swagger-resources/**",
                        "/v3/api-docs/**",
                        "/actuator/health"
                )

                // 3. 认证逻辑：只校验登录，不校验权限
                .setAuth(obj -> {
                    // 所有请求（除白名单）必须登录
                    SaRouter.match("/**", r -> StpUtil.checkLogin());
                })

                // 4. 全局异常处理（未登录/无效token）
                .setError(this::handleError)

                // 5. 跨域处理（网关统一处理，避免OPTIONS预检被拦截）
                .setBeforeAuth(this::handleCors);
    }

    /**
     * 全局异常处理
     */
    private Mono<SaResult> handleError(Throwable e) {
        // 未登录 / Token 无效
        if (e instanceof cn.dev33.satoken.exception.NotLoginException) {
            return Mono.just(SaResult.code(HttpStatus.UNAUTHORIZED.value()).msg("未登录或Token已过期"));
        }
        // 其他异常
        return Mono.just(SaResult.error("认证失败：" + e.getMessage()));
    }

    /**
     * 跨域处理（Gateway 层统一处理，非常关键）
     */
    private void handleCors(ServerWebExchange exchange) {
        SaHolder.getResponse()
                .setHeader("Access-Control-Allow-Origin", "*")
                .setHeader("Access-Control-Allow-Methods", "GET,POST,PUT,DELETE,OPTIONS")
                .setHeader("Access-Control-Allow-Headers", "token,Content-Type,Authorization")
                .setHeader("Access-Control-Max-Age", "3600");

        // OPTIONS 预检请求直接放行
        if ("OPTIONS".equalsIgnoreCase(exchange.getRequest().getMethodValue())) {
            exchange.getResponse().setStatusCode(HttpStatus.OK);
        }
    }
}
```

#### 五、核心方法逐行详解

##### 1. `SaReactorFilter`（Gateway 专属）

- 替代 Servlet 环境的 `SaServletFilter`
- 适配 WebFlux 响应式流
- 必须在网关容器中注册

##### 2. `.addInclude("/**")`

- 拦截**所有请求**
- 必须配合 `.addExclude()` 放行白名单

##### 3. `.addExclude(...)` 白名单

必须放行：
- 登录/注册接口
- Swagger 文档
- 健康检查
- 静态资源

##### 4. `.setAuth(...)` 认证逻辑

```java
SaRouter.match("/**", r -> StpUtil.checkLogin());
```
- 只做 **StpUtil.checkLogin()**
- **不写** `checkPermission` / `checkRole`
- 网关**不查询权限**，也**没有 StpInterface**

##### 5. `.setError(...)` 全局异常

统一返回：
- 401 未登录
- Token 过期
- 格式统一，前端好处理

##### 6. `.setBeforeAuth(...)` 跨域

**Gateway 必须在这里处理跨域**，否则：
- OPTIONS 预检请求被拦截
- 前端跨域报错
- 下游服务不用再写跨域

---

#### 六、网关鉴权完整流程（一眼看懂）

1. 前端请求 → Gateway
2. `SaReactorFilter` 拦截
3. 检查是否白名单 → 是 → 放行
4. 非白名单 → 校验 Token（Redis 会话）
5. Token 有效 → 路由到下游服务（Auth/Biz）
6. 下游服务 → `StpInterfaceImpl` → 查询权限
7. 下游服务 → `@SaCheckPermission` → 校验权限

一句话：
> **网关管“进门”，服务管“干活权限”。**

---

#### 七、常见坑（必看）

1. **网关写了 StpInterfaceImpl**
   → 报错、不生效、上下文丢失
2. **网关配置了 checkPermission**
   → 无效，网关拿不到权限数据
3. **跨域写在下游服务**
   → OPTIONS 被网关拦截，前端报错
4. **Token 名称和业务服务不一致**
   → 登录成功，网关不认

---

#### 八、一句话总结

**SaTokenGatewayConfig = Gateway 登录过滤器 + 白名单 + 跨域 + 全局异常**  
**只做登录，不做权限；只拦截 Token，不查数据库。**
