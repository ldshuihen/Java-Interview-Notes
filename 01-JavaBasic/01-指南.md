# 代码命名规范指南

**干净的命名 = 代码即文档 = 可读性 = 可维护性**
这份指南覆盖：类、方法、变量、常量、接口、布尔、集合、异常、测试、通用禁忌，**面试、开发、CodeReview 全能用**。

---

## 一、通用核心规则（所有命名必须遵守）

1. **必须英文**，禁止拼音、禁止中英混合
2. **见名知意**，不缩写、不模糊
3. **大小写严格遵守**
4. **不使用 $、_ 开头/结尾**（除常量、常量例外）
5. **不使用 a、b、c、tmp、flag 这种无意义名称**
6. **统一技术语境**：增删改查、业务名词统一

---

## 二、大小写规范（Java 死规则）

### 1. 大驼峰（UpperCamelCase）

**类、枚举、接口、注解**

- `UserController`
- `OrderService`
- `UserDTO`
- `SensitiveWordFilter`

### 2. 小驼峰（lowerCamelCase）

**方法名、变量名、参数名**
- `userName`
- `orderList`
- `createUser()`

### 3. 全大写 + 下划线

**常量、静态常量**
- `MAX_SIZE`
- `DEFAULT_TIMEOUT`
- `TOKEN_PREFIX`

### 4. 全小写 + 下划线

**包名、配置文件、数据库表/字段**
- `com.company.user.service`
- `application.yml`
- `user_info`
- `order_id`



### 5. 蛇形命名法（snake_case）

- 规则：**全小写 + 下划线分隔**
- 别名：下划线命名法
- 适用场景（**超级重要**）：
  1. **数据库表名、字段名**（MySQL、Oracle）
  2. **包名不使用，Java 不用**
  3. **配置文件 key**
  4. **URL 路径不推荐，但有些公司用**
  5. Python、Go、C++ 变量 / 函数常用
- 示例：
  - `user_name`
  - `order_id`
  - `create_time`
  - `user_info_table`

------

### 6. 串式命名法（kebab-case）

- 规则：**全小写 + 横线分隔**
- 别名：短横线命名法、烤肉串命名法
- 适用场景：
  1. **URL 路径**
  2. **前端 CSS 类名**
  3. **前端组件文件名**
  4. **域名、静态资源**
  5. **Java 绝对不能用！**
- 示例：
  - `/user-info/list`
  - `.btn-primary`
  - `user-login-component`
  - `aliyun-oss-file`

## 五、变量命名指南

### 1. 普通变量

**名词/形容词+名词**
- `userId`
- `userName`
- `orderAmount`
- `loginToken`

### 2. 集合/数组

**名词 + 复数 / List / Map**
- `userList`
- `orderList`
- `idList`
- `userMap`（key-value）
- `userNameList`

禁止：
- `list`
- `data`
- `map`

### 3. 布尔变量（超级重要）

**必须用 is / can / has / allow 开头**
- `isSuccess`
- `isDeleted`
- `hasPermission`
- `canEdit`

禁止：
- `success`
- `delete`
- `flag`

### 4. 临时变量（允许短命名）

- 页码：`pageNum`、`pageSize`
- 索引：`index`
- 总数：`total`

   - 错：`uInfo`、`odr`

6. **禁止超长但没必要**
7. **禁止拼音首字母**

## Java 后端 100% 强制规范（总结）

1. **Java 代码：只用驼峰**
   - 类：大驼峰
   - 方法/变量：小驼峰
   - 常量：全大写下划线（`MAX_SIZE`）

2. **数据库：全用蛇形（snake_case）**
   - 表：`user_info`
   - 字段：`user_id`, `create_time`

3. **URL / 前端：全用串式（kebab-case）**
   - `/user/login`
   - `/order/list`

4. **配置文件：蛇形或小驼峰**
   - `spring.datasource.url`

---

## 最容易犯的错误（千万避开）

1. ❌ Java 变量用蛇形：`user_name` → 错
2. ❌ Java 变量用串式：`user-name` → **语法错误**
3. ❌ 数据库用驼峰：`userName` → 错、不规范、易错
4. ❌ URL 用驼峰：`/userInfo` → 不推荐
5. ❌ 混合命名：`user_Name` → 灾难

# 代码重构完整指南

## 一、重构核心定义
**重构**：在**不改变外部功能、不修改业务逻辑**的前提下，对代码内部结构、写法、逻辑、设计进行优化；
目的：**提升可读性、可维护性、可扩展性、降低Bug率、方便迭代**。

> 核心原则：**行为不变，结构优化；小步迭代，持续重构。**

---

## 二、什么时候需要重构（坏代码信号）
### 1. 代码异味（Smell Code）
- 方法超长、类超大、上千行巨型类
- 重复代码遍地、复制粘贴开发
- 嵌套层级过深（if/for 套娃 3层以上）
- 大量硬编码、魔法值、无注释
- 模糊命名、缩写、拼音、无意义变量
- 超大参数列表、过多全局变量
- 分支爆炸：if-else / switch 巨多
- 实体臃肿、一个类承载多种职责

### 2. 业务迭代痛点
- 改一处、崩多处（牵一发而动全身）
- 新增需求只能堆砌代码，无法扩展
- BUG频发、逻辑混乱、难以排查
- 新人接手看不懂、上手成本极高
- 单元测试无法编写、耦合严重

---

## 三、重构核心原则（必遵守）
1. **最小改动原则**
   不改动对外接口、不改动业务结果，只优化内部。
2. **单一职责原则**
   一个类/一个方法只做一件事。
3. **开闭原则**
   对扩展开放、对修改关闭，少改老代码，多新增扩展。
4. **依赖倒置 / 低耦合**
   依赖抽象、不依赖具体实现，减少硬编码耦合。
5. **小步重构、随时可测**
   不要一次性大改，分批优化、每步自测、防止线上事故。
6. **先注释、再优化**
   复杂逻辑先补注释，再梳理拆分。

---

## 四、通用落地重构手段（最常用）
### 1. 命名重构（成本最低、收益最大）
- 模糊名称 → 语义完整名称
- 缩写/拼音 → 标准英文
- 布尔统一 `is/has/can/allow` 前缀
- 集合加后缀 List/Map/Set
- 杜绝：`data、info、temp、flag、obj`

### 2. 抽取方法 / 拆分巨型方法
- 超过 **50行** 方法必须拆分
- 独立逻辑、可复用逻辑抽为私有方法
- 一个方法只保留**主流程**，细节下沉

**优化前**
```java
public void createOrder() {
    // 校验
    if (userId == null) { ... }
    // 库存
    // 计价
    // 入库
    // 消息推送
}
```
**优化后**
```java
public void createOrder() {
    checkOrderParam();
    lockStock();
    calculateOrderAmount();
    saveOrder();
    sendOrderNotice();
}
```

### 3. 消除嵌套、减少层级
- 优先**卫语句**：提前 return / continue / break
- 反向判断，减少 if 嵌套

**优化前**
```java
if (user != null) {
    if (user.getStatus() == 1) {
        // 业务逻辑...
    }
}
```
**优化后**
```java
if (user == null) {
    return;
}
if (user.getStatus() != 1) {
    return;
}
// 业务逻辑
```

### 4. 消除重复代码
- 重复逻辑抽**工具类 / 公共方法 / 父类**
- 相同条件判断、相同转换逻辑统一封装
- 批量操作抽取通用模板方法

### 5. 魔法值、硬编码统一改造
- 所有固定数值、固定文案 → 常量/枚举
- 接口地址、密钥、配置 → 配置文件 / 配置中心
- 禁止代码里写死 `0、1、2、"normal"、"delete"`

### 6. 复杂 if-else 重构
1. 简单分支：用**枚举 + 策略**
2. 大量业务分支：策略模式 + 工厂模式
3. 条件判断过多：合并条件、拆分条件

### 7. 参数优化
- 超长参数列表 → 封装 DTO/Query 对象
- 多余参数、无用参数删除
- 参数校验统一收口（JSR校验+手动校验）

### 8. 注释与日志重构
- 核心业务、复杂算法、特殊逻辑必须加注释
- 日志分级：info/debug/error 合理使用
- 禁止打印完整敏感数据、超长报文裸打
- 关键节点记录入参、出参、异常堆栈

### 9. 异常处理重构
- 统一全局异常捕获
- 禁止大量 `try-catch` 嵌套
- 禁止 catch 后空处理、吞异常
- 自定义业务异常，区分系统异常/业务异常

### 10. 集合、循环、Stream 优化
- 普通 for 复杂遍历 → 简化为 Stream
- 集合初始化指定容量，减少扩容
- 避免频繁创建集合、对象
- 增强for / lambda 简化冗余写法

---

## 五、分层架构重构（后端项目重点）
### 1. 分层职责清晰
- **Controller**：接收参数、校验、转发、组装返回
- **Service**：核心业务逻辑、事务、流程编排
- **Mapper/DAO**：只做数据库CRUD
- **Domain/Entity**：纯数据载体，无业务逻辑
- **Util**：通用工具，无业务侵入

### 2. 杜绝跨层乱写
- Controller 不写复杂业务
- Service 不直接操作 Http、不拿 Request
- Mapper 不写复杂业务判断

### 3. DTO/VO/DO 分层隔离
- 入参：DTO
- 数据库：DO/Entity
- 返回前端：VO
- 禁止**实体类直接透传**，字段暴露、安全隐患

### 4. 业务与技术解耦
- 缓存、MQ、异步、定时任务抽独立组件
- 第三方调用、支付、推送抽独立 Service

---

## 六、典型场景重构方案
### 1. 超大接口、超级Controller
- 按**业务模块**拆分多个接口
- 按**职责**拆分多个 Controller
- 复杂操作拆成多个单一接口

### 2. 状态判断超多（订单、流程）
- 状态码 → 枚举
- 状态流转规则 → 统一策略类
- 大量 if-else → 状态机模式

### 3. 批量操作、导入导出
- 抽取统一批量处理模板
- 分页分批处理，防止OOM、长事务
- 公共校验、公共结果封装统一

### 4. 老旧遗留代码
- 不一次性推翻重写
- 新增需求走新写法
- 迭代中逐步局部替换、平滑改造

---

## 七、重构避坑指南（关键）
1. **禁止边重构边加新功能**
   重构只改结构，新需求单独开发。
2. **禁止大跨度全量重构**
   分批、分模块、单文件逐步优化。
3. **重构后必须自测**
   边界条件、异常场景、历史兼容全部回归。
4. **避免过度设计**
   小业务不要强行上设计模式，够用即可。
5. **锁、事务、分布式逻辑谨慎修改**
   事务传播、锁范围、幂等不要随意动。

---

## 八、常用设计模式 重构落地场景
- **策略模式**：替换大量 if-else 业务分支
- **工厂模式**：多类型实现统一创建
- **模板方法**：统一流程，不同步骤自定义
- **装饰器**：功能叠加、扩展增强
- **适配器**：老旧接口、第三方接口兼容改造

---

## 九、重构验收标准（重构完怎么判断合格）
1. 代码行数精简、逻辑清晰
2. 无重复代码、无魔法值、无深层嵌套
3. 命名规范、见名知意
4. 职责单一、分层清晰
5. 异常、日志、注释完善
6. 原有功能完全一致，无BUG
7. 扩展性提升，新增需求无需大幅改旧代码

---

## 十、面试高频背诵总结
1. 重构定义：**不改变外部行为，优化内部结构，提升可维护性**。
2. 常见代码异味：巨型类、长方法、重复代码、嵌套过深、大量ifelse、硬编码。
3. 核心重构手段：拆分方法、抽取公共、优化命名、卫语句降嵌套、枚举替代状态、分层解耦。
4. 重构原则：**小步迭代、行为不变、单一职责、开闭原则**。
5. 复杂分支优化：if-else → 枚举 + 策略模式 + 工厂模式。
6. 后端重点：分层解耦、DTO隔离、业务与工具分离、事务与异常统一。

# 单元测试完整指南

## 一、单元测试核心概念
### 1. 定义
**单元测试**：对程序中**最小代码单元**（方法/类/组件）进行隔离、独立、自动化测试。
- 后端：以 **Service、工具类、公共方法、核心算法** 为核心
- 不依赖数据库、不依赖Redis、不依赖MQ、不依赖第三方接口
- 只测**业务逻辑、分支、边界、异常**

### 2. 核心价值
1. 验证逻辑正确性，提前发现Bug
2. 代码重构、迭代升级**保障回归不报错**
3. 强制写出「高内聚、低耦合」代码
4. 减少联调、测试、线上问题
5. 提升代码质量，方便新人理解逻辑

### 3. 测试金字塔
- **单元测试（底层）**：数量最多、成本最低、速度最快
- 集成测试（中层）：测接口、数据库、中间件联动
- 自动化/手工测试（顶层）：全链路业务场景

> 原则：**单元测试越多，整体质量越稳**

---

## 二、技术栈（Java 标配）

1. **Junit 5（Jupiter）**：主流测试框架
2. **SpringBoot Test**：Spring 环境整合
3. **Mockito**：Mock 依赖、隔离外部组件
4. **AssertJ / Junit Assert**：优雅断言
5. **H2 内存库**：集成测试轻量数据库

**JUnit 5** 是Java生态的**通用单元测试框架**，而 **Spring Test** 是Spring生态提供的**集成测试扩展**，两者是**基础框架与扩展插件**的关系，常结合使用。

### 一、核心定位与关系

- **JUnit 5**：独立的通用测试框架，提供**测试生命周期、断言、注解、扩展机制**等基础能力。
- **Spring Test**：**依赖JUnit 5**，通过`SpringExtension`扩展JUnit，专为**Spring应用**提供**上下文管理、依赖注入、事务控制、Web测试**等能力。

### 二、关键区别对比
#### 1. 核心功能
- **JUnit 5**
  - 基础测试注解：`@Test`、`@BeforeEach`、`@AfterEach`
  - 断言：`Assertions.*`（`assertEquals`、`assertThrows`等）
  - 扩展机制：`@ExtendWith`、自定义扩展
  - 参数化测试、条件测试、嵌套测试

- **Spring Test**
  - **Spring上下文加载**：`@ContextConfiguration`、`@SpringBootTest`
  - **依赖注入**：`@Autowired`、`@MockBean`
  - **事务管理**：`@Transactional`（默认回滚）
  - **Web测试**：`MockMvc`、`TestRestTemplate`
  - **上下文缓存**：避免重复加载`ApplicationContext`

#### 2. 适用场景
- **JUnit 5**：**纯Java单元测试**（无Spring依赖）、简单逻辑验证
- **Spring Test**：**Spring应用集成测试**（Service/Controller/Repository）、依赖注入验证、事务测试

#### 3. 注解体系
- **JUnit 5**：`@Test`、`@BeforeEach`、`@AfterEach`、`@ExtendWith`
- **Spring Test**：`@SpringBootTest`、`@ContextConfiguration`、`@Transactional`、`@MockBean`

#### 4. 依赖关系
- **JUnit 5**：独立依赖（`junit-jupiter-api`）
- **Spring Test**：依赖JUnit 5（`spring-test`）

### 三、使用示例
#### 1. 纯JUnit 5测试
```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

public class SimpleTest {
    @Test
    void testAdd() {
        assertEquals(2, 1 + 1);
    }
}
```

#### 2. Spring Test + JUnit 5
```java
import org.junit.jupiter.api.Test;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.beans.factory.annotation.Autowired;
import static org.junit.jupiter.api.Assertions.*;

@SpringBootTest
public class UserServiceTest {
    @Autowired
    private UserService userService;

    @Test
    void testCreateUser() {
        User user = userService.createUser("test");
        assertNotNull(user);
    }
}
```

### 四、总结
- **JUnit 5**：**通用测试基础**，提供测试的**核心骨架**。
- **Spring Test**：**Spring专用扩展**，解决**Spring上下文、依赖注入、事务**等集成测试问题。

**一句话**：**JUnit 5是“测试的引擎”，Spring Test是“Spring专用的引擎插件”**。

## 三、单元测试分类（必须分清）
### 1. 纯单元测试（Unit Test）✅【推荐】
- 不启动Spring容器
- 不连库、不连缓存
- 所有依赖通过 **Mockito Mock**
- 速度极快、适合核心业务、工具类、算法

### 2. 集成测试（Integration Test）
- 启动Spring上下文
- 真实连接数据库、Redis、配置
- 适合：Mapper、完整流程、事务、多组件联动

### 3. 接口测试（Controller Test）
- 使用 `MockMvc` 模拟请求
- 不启动Tomcat，模拟HTTP接口调用

## 四、单元测试规范（开发强制遵守）
### 1. 命名规范
- 测试类：`业务类 + Test`
  `UserServiceTest`、`OrderUtilTest`
- 测试方法：
  `被测方法_场景_预期结果()`
  ```java
  void calculatePrice_折扣场景_返回优惠后金额()
  void checkUser_用户不存在_抛出异常()
  ```

### 2. 目录规范
```
src/main/java    业务代码
src/test/java    单元测试代码
包结构与main完全一致
```

### 3. 被测范围
#### ✅ 必须写单元测试
- 核心业务Service
- 工具类、脱敏、加密、日期、算法
- 复杂条件判断、状态机、订单流程
- 公共组件、通用方法
- 经常迭代、易出Bug代码

#### ❌ 不需要写
- 简单CRUD Mapper（SQL简单无逻辑）
- 单纯参数转发Controller
- 简单枚举、常量类

### 4. 用例覆盖要求
1. **正常场景**：正向流程
2. **边界场景**：极值、空值、临界值
3. **异常场景**：非法参数、异常分支、报错
4. **分支覆盖**：if/else、switch、循环全部覆盖

## 五、SpringBoot 单元测试标准写法
### 1. 依赖
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>
```

### 2. ① 纯单元测试（Mockito 不启Spring）
适合：Service 逻辑、工具类
```java
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    // 被测试类
    @InjectMocks
    private UserService userService;

    // 外部依赖：全部Mock
    @Mock
    private UserMapper userMapper;
    @Mock
    private RedisUtil redisUtil;

    @Test
    void getUserById_正常查询_返回用户() {
        // 1. 准备Mock数据
        when(userMapper.selectById(1L)).thenReturn(new User(1L,"张三"));
        
        // 2. 执行方法
        User user = userService.getUserById(1L);
        
        // 3. 断言
        Assertions.assertNotNull(user);
        Assertions.assertEquals("张三", user.getName());
        
        // 4. 验证依赖是否执行
        verify(userMapper, times(1)).selectById(any());
    }
}
```

### 3. ② 集成测试（启动Spring容器）
适合：事务、数据库、完整流程
```java
@SpringBootTest
class OrderServiceIntegrationTest {

    @Autowired
    private OrderService orderService;

    @Test
    void createOrder_正常下单_创建成功(){
        // 执行 + 断言
    }
}
```

### 4. ③ Controller 接口单元测试（MockMvc）
```java
@WebMvcTest(UserController.class)
class UserControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private UserService userService;

    @Test
    void userInfo_请求接口_返回成功() throws Exception {
        mockMvc.perform(get("/user/info")
                        .param("userId","1"))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.code").value(200));
    }
}
```

## 六、Mockito 核心常用能力
1. `@Mock`：模拟依赖，不走真实实现
2. `@InjectMocks`：注入被测试对象
3. `when().thenReturn()`：模拟返回值
4. `doThrow().when()`：模拟异常抛出
5. `verify()`：校验方法调用次数、参数
6. `ArgumentCaptor`：捕获传入参数，校验参数正确性

## 九、单元测试最佳实践
1. **测试方法短小单一**，一个方法只测一个场景
2. 避免硬编码，测试数据统一管理
3. 禁止写大量 `sysout`，使用断言
4. 测试代码同样需要：规范命名、分层、整洁
5. 单元测试**不写业务**，只做校验
6. 频繁修改的核心模块，优先保障单测覆盖
7. 集成测试尽量使用 **H2内存库**，不污染生产/测试库
8. 禁止单元测试写死线上配置、真实密钥

## 十、单元测试常见误区
### 误区1：单元测试要连数据库
错：**纯单元测试必须隔离数据库**，用Mock

### 误区2：单元测试写了没用
错：重构、升级、修复Bug时，单测是**回归保障**

### 误区3：单元测 = 接口测试
错：
- 单元测试：测内部逻辑
- 接口测试：测网络、参数、序列化、权限

### 误区4：单测代码随便写
错：测试代码也是工程代码，同样需要维护

## 十一、面试高频问题（必背）
### 1. 什么是单元测试？
对最小代码单元（方法/类）进行**隔离、独立、自动化**测试，保障逻辑正确性，快速回归、降低Bug。

### 2. 单元测试和集成测试区别？
- 单元测试：Mock依赖、不启容器、不连中间件、速度快、测逻辑
- 集成测试：启动Spring、真实连库/缓存、测组件联动与事务

### 3. 为什么要用 Mock？
隔离外部依赖（数据库、Redis、第三方接口），**只关注当前方法内部逻辑**，不受外部环境影响。

### 4. 单元测试要覆盖哪些场景？
正常场景、空值边界、非法参数、异常分支、多条件分支全覆盖。

### 5. 如何测试异常？
使用 `assertThrows` 断言指定异常，捕获Lambda执行方法抛出的业务异常。
