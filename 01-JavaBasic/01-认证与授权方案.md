# **认证（Authentication）** 与 **授权（Authorization）**

在 Java 生态中，用于 **认证（Authentication）** 与 **授权（Authorization）** 的框架或技术方案非常丰富，覆盖从传统单体应用到现代微服务、云原生架构的多种场景。以下是全面且结构清晰的整理，包含主流及新兴方案：

------

## 一、主流综合安全框架

### 1. **Spring Security**

- **适用**：Spring / Spring Boot 应用（事实标准）

- 特点：

  - 支持表单登录、HTTP Basic、JWT、OAuth2、OIDC、SAML、LDAP 等
- 方法级安全（`@PreAuthorize`, `@Secured`）
  - 内置 CSRF、CORS、Session 管理、密码加密等
  - 高度可扩展（自定义 `AuthenticationProvider`、`UserDetailsService` 等）
  
- **注意**：5.7+ 版本弃用 `WebSecurityConfigurerAdapter`，推荐函数式配置。

### 2. **Apache Shiro**

- **适用**：非 Spring 项目或轻量级需求

- 特点：

  - API 简洁，学习成本低
- 支持认证、授权、会话、加密四大核心功能
  - 可用于任何 Java 环境（Web、桌面、CLI）
  
- **局限**：对 OAuth2/OIDC 原生支持弱，社区活跃度下降。

### 3. **Sa-Token** ✅（国产新兴框架）

- **适用**：快速开发、前后端分离、RESTful API 场景

- 特点

  ：

  - 极简 API（如 `StpUtil.login(userId)`、`@SaCheckPermission("xxx")`）
  - 内置分布式 Session（支持 Redis）
  - 支持单点登录（SSO）、多端登录控制、Token 自动续期
  - 无 Cookie 依赖，默认基于 Token（Header/Param）
  - 零配置起步，YAML 可调

- **优势**：开发效率高，代码可读性强，适合中小团队。

- **官网**：https://sa-token.cc/

## 二、基于标准协议的身份解决方案

### 4. **OAuth2 / OpenID Connect (OIDC)**

- 典型实现

  ：

  - **Keycloak**（Red Hat 开源 IAM 平台）
    → 提供用户管理、SSO、OAuth2/OIDC/SAML，Java 应用通过适配器或标准协议集成。
  - **Auth0 / Okta / AWS Cognito**（商业云服务）
    → 托管身份服务，提供 SDK 和管理界面。
  - **Spring Authorization Server**
    → Spring 官方维护的 OAuth2 授权服务器实现（替代旧版 Spring Security OAuth）。

> Java 应用通常作为 **Resource Server**，验证 JWT 并执行权限控制。

### 5. **JWT（JSON Web Token）**

- **说明**：不是框架，而是 Token 格式标准（RFC 7519）

- 常用库：

  - `io.jsonwebtoken:jjwt`（最流行）
- `com.auth0:java-jwt`
  
- **使用方式**：常与 Spring Security 或 Sa-Token 结合，实现无状态认证。

## token认证

### **Token 认证（Token 认证机制）**

**具体实现 = 基于 JWT 的 Token 认证（最主流）**

#### 一、完整流程（你描述的 exactly 就是这个）

1. 用户登录（账号密码）
2. 后端验证通过 → **生成 Token（一般是 JWT）**
3. 前端拿到 Token → **存到请求头 Header 里**（通常是 `Authorization: Bearer xxx`）
4. 后续每次请求前端**自动带上 Header**
5. 后端拦截器/过滤器 **解析 Token → 验证合法性 → 获取用户信息**

---

#### 二、它的官方标准名称（面试必须说对）

##### 1. **Token 认证（Token-Based Authentication）**

##### 2. **无状态认证（Stateless Authentication）**

##### 3. **基于 JWT 的认证（最常用）**

##### 4. **Bearer Token 认证（标准规范）**

##### ✅ 面试官问：

“用户登录生成token，放在header里带给后端，这是什么认证？”

##### ✅ 你直接答：

**这是 Token 认证，也叫无状态认证，目前主流使用 JWT 实现，前端通过 Authorization 请求头传递 Token。**

---

#### 三、它和「Session 认证」的区别（必考点）

##### 1. **Session 认证（传统）**

- 基于 Cookie
- 服务端**存储 Session**（有状态）
- 分布式需要 Session 共享（Redis）

##### 2. **Token 认证（现在微服务全用这个）**

- 基于 **Header** 传递
- 服务端**不存用户状态（无状态）**
- 信息全部存在 **JWT 里面**
- 天然支持分布式、微服务、前后端分离

---

#### 四、标准请求头长这样（你一定见过）

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

- `Authorization`：请求头名
- `Bearer`：认证类型（固定标准）
- 后面一串：**JWT Token**

---

#### 五、SpringBoot 里实现这套的技术栈（真实开发）

1. **JWT**（jjwt 库）生成/解析 token
2. **自定义拦截器 / Filter** 校验 header 中的 token
3. **ThreadLocal / SecurityContext** 存储当前用户
4. 配合框架：
   - **Sa-Token**（国内最简单）
   - **Spring Security**（标准安全框架）

---

#### 六、最终极简总结（背这一句就够）

**用户登录生成 Token，放在请求头 Authorization 中传递给后端，
这就是 SpringBoot + 前后端分离 + 微服务架构中最主流的：
👉 Token 认证（无状态认证，常用 JWT 实现）。**

### **双 Token 认证**

我给你讲**最标准、最直白、面试必背**的**双 Token 认证**，一次性讲透：是什么、为什么、流程、和单Token区别。

#### 一、双 Token 是什么？

**双 Token = 两个 Token**

1. **Access Token（短有效期，5~30分钟）**
2. **Refresh Token（长有效期，7天~30天）**

合称：**Access/Refresh Token 认证机制**
也叫：**令牌刷新机制、双令牌认证、JWT 双Token**

#### 二、为什么要有双 Token？

**单 JWT Token 有致命缺陷：**

- 有效期太短：用户频繁登录，体验差
- 有效期太长：一旦泄露，**被盗用很久都能用，极不安全**

##### 双Token完美解决：

- **Access Token 短有效**：即使泄露，风险小
- **Refresh Token 长有效**：用来**静默刷新 Access Token**
- 用户不用重新登录，安全 + 体验兼顾

#### 三、完整流程（超级重要，面试必考）

1. 用户登录 → 验证成功
2. 后端生成**两个 Token**：
   - **AccessToken（短）**
   - **RefreshToken（长）**
3. 前端同时保存两个Token
4. **每次正常请求**：
   Header 只带 **AccessToken**
5. **AccessToken 过期**：
   前端**自动用 RefreshToken 请求刷新接口**
6. 后端验证 RefreshToken 有效 → **返回新的 AccessToken**
7. 前端替换新 AccessToken → **继续请求，用户无感知**
8. RefreshToken 也过期 → **用户重新登录**

---

#### 四、核心特点（必须背）

1. **无状态认证（依然 JWT，不存 Session）**
2. **安全 + 体验平衡**
3. **微服务、前后端分离、APP 最主流认证方案**
4. 不属于 OAuth2，但 **OAuth2 核心就是双Token**

---

#### 五、双Token 属于什么认证？

##### 标准答案：

**仍然是 Token 认证（无状态认证），
是 JWT 单Token 的升级版、增强安全版。
也可以理解为：OAuth2 标准的核心机制。**

一句话：
**双Token = 更安全、更完善的 JWT 认证。**

---

#### 六、面试满分回答（背这段）

**双Token认证是指使用 AccessToken 和 RefreshToken 两个令牌的无状态认证机制：
AccessToken 短有效期，用于正常接口鉴权；
RefreshToken 长有效期，用于静默刷新 AccessToken。
既解决了单Token有效期过短导致频繁登录的问题，又降低了长期Token泄露的安全风险，
是目前前后端分离、微服务、移动端最主流的用户认证方案。**

- **双Token：安全、体验好、企业级标准**
- **本质：JWT 无状态认证的升级版**

### 使用 **双Token** 前端、后端实现

我讲得**超级清晰、一步到位**，面试/开发都能直接说。

---

#### 一、后端要做的修改（重点）

##### 1. 登录接口改造

**原来：只返回一个 token**
**现在：返回两个 token**
```json
{
  "accessToken": "eyxxx",    // 短有效：15~30分钟
  "refreshToken": "eyyyy"    // 长有效：7~30天
}
```

##### 2. 新增一个 **刷新token接口**（必加）

```
POST /user/refresh-token
参数：refreshToken
返回：新的 accessToken
```
作用：accessToken过期时，前端调用此接口**静默获取新token**。

##### 3. JWT 工具类修改

- 生成 **accessToken**（短过期）
- 生成 **refreshToken**（长过期）
- 两个 token 可以**secret不同**，更安全

##### 4. 拦截器（Interceptor）逻辑修改

**原来：**
- 校验token是否有效 → 有效放行
- 无效直接返回401

**现在：**
- 校验 **accessToken**
- **过期不直接返回401，而是返回固定码（如 401 或 4001）**
- 让前端知道：是**过期**，不是非法，前端可以去刷新

##### 5. 安全增强（可选但推荐）

- refreshToken 存入 Redis（黑名单/白名单）
- 支持**强制下线**、**刷新token过期**、**顶号下线**
- 因为 JWT 本身无法作废，refreshToken存Redis才能实现登出

---

#### 二、前端要做的修改（重点）

##### 1. 登录后存储两个token

- localStorage / pinia / vuex 存储：
  - accessToken
  - refreshToken

##### 2. 请求头统一携带 **accessToken**

```js
config.headers.Authorization = 'Bearer ' + accessToken
```

##### 3. **axios响应拦截器核心改造（最重要）**

**前端双Token灵魂在这里：**

流程：
1. 接口返回 **401/自定义过期码**
2. 前端判断：是否 access_token 过期
3. **如果是：**
   - 立即请求 `/refresh-token` 接口
   - 用 refreshToken 获取 **新accessToken**
   - **替换本地accessToken**
   - **自动重试刚才失败的请求**
4. 用户完全无感知（静默刷新）

##### 4. 特殊处理：防止同时多次刷新（锁）

避免并发请求同时过期，发送多次refresh。
用**锁/标志位**保证同一时间只刷新一次。

##### 5. refreshToken也过期 → 跳登录

- 刷新接口也失败 → 清除token → 跳login

---

#### 三、前后端改动总结（极简版）

##### 后端改动

1. 登录返回 **accessToken + refreshToken**
2. 增加 **/refresh-token** 刷新接口
3. 拦截器区分：**token过期 / token非法**
4. refreshToken 存入Redis（支持作废、登出、顶号）

##### 前端改动

1. 存储两个token
2. 请求头只带 accessToken
3. axios响应拦截实现：
   **过期 → 刷新 → 重试**（静默刷新）
4. 处理刷新失败 → 跳登录

---

#### 四、面试官最爱问的一句话（你这样答满分）

##### **使用双Token时，后端需要增加刷新接口、区分token过期状态、登录返回双Token；

前端需要在响应拦截器中实现自动刷新token并重试失败请求，实现无感刷新。
同时为了解决JWT无法作废问题，refreshToken通常存入Redis实现安全控制。**

## 用户认证 / 会话管理方案

我给你整理**极度清晰、面试直接背、不会混淆**的**三者对比思路**，结构简单、逻辑闭环，**面试官听了直接觉得你基础很稳**。

## 一、三者本质一句话

1. **传统单机 Session**
   服务器**内存**存用户状态，Cookie 带 `JSESSIONID`，**不支持集群**。

2. **Spring Session + Redis（分布式 Session）**
   把 Session 存 **Redis**，Cookie 带 `SESSION`（= Token），**支持集群、共享登录**。

3. **JWT Token**
   **服务端不存任何状态**，用户信息**加密放在 Token 里**，前端每次携带，**无状态**。

---

## 二、核心结构对比（最重要）

### 1. 传统单机 Session

- 存储位置：**Tomcat 内存**
- Cookie：`JSESSIONID=xxxx`
- 结构：
  - Key：sessionId
  - Value：用户信息
- 特点：
  - 单机有效
  - 集群不同步 → **登录状态丢失**
- **缺点：微服务环境完全不能用**

---

### 2. Spring Session + Redis（分布式Session）

- 存储位置：**Redis**
- Cookie：`SESSION=xxxx` 👉 **这个就是 Token**
- 结构：
  - Key：`spring:session:sessions:{sessionId}`（Token）
  - Value：用户信息、过期时间、最后访问时间
- 特点：
  - **多服务、多实例完全共享登录**
  - 有状态、可踢人、可下线、可查在线人数
- **本质：基于 Token 的有状态分布式登录**

---

### 3. JWT Token

- 存储位置：**服务端不存储**
- 放在：Header / Cookie / LocalStorage
- 结构：
  - Header.Payload.Signature
  - Payload 里**直接存用户ID、权限、过期时间**
- 特点：
  - **无状态**，不依赖Redis
  - 水平扩展极强
  - **无法主动过期、无法踢人**

---

## 三、最关键区别（面试必问）

### 1. 是否存储状态？

- 传统Session：**存内存（有状态）**
- SpringSession+Redis：**存Redis（有状态）**
- JWT：**不存储（无状态）**

### 2. 是否支持分布式/集群？

- 传统Session：❌ **不支持**
- SpringSession+Redis：✅ **完美支持**
- JWT：✅ **天然支持**

### 3. 能不能踢人、强制下线？

- 传统Session：能
- SpringSession+Redis：**能（删Redis即可）**
- JWT：**不能（除非配合黑名单，复杂）**

### 4. 前端携带的是什么？

- 传统Session：JSESSIONID
- SpringSession：SESSION（就是Token）
- JWT：JWT字符串（自包含令牌）

---

## 四、最简单记忆口诀

**传统Session：内存存，单机用，集群废。**
**Redis Session：Token当Key，Redis存状态，集群共享，能踢人。**
**JWT：自身带信息，不存状态，扩展强，难踢人。**

---

## 五、企业真实使用场景（面试加分）

- **后台管理系统、内部系统：Spring Session + Redis（主流）**
- **网关、微服务对外接口、第三方登录：JWT**
- **传统单体老项目：单机Session**

---

## 六、你最关心的问题最终答案

### **Redis 分布式 Session 是不是存 Token？**

**是！完全是！**
- Spring 自动生成一个**随机字符串作为 Token**
- 以这个 Token 为 Key 存 Redis
- 前端通过 Cookie 携带这个 Token
- 只是框架封装好了，你**不用自己生成、解析、存储**

所以：
**Spring Session = 全自动封装版的 Token+Redis 登录**

## 一、登录认证模块（最核心）
**解决问题**：判断用户是否登录、会话管理、踢人下线、自动续签

### 1. 基础登录认证

- 支持**单端登录**、**多端登录**
- 自动创建会话、存储用户ID、自动过期
- 前后端分离/传统Web均支持

**核心API**
```java
StpUtil.login(10001);        // 登录
StpUtil.checkLogin();        // 校验是否登录，未登录抛异常
StpUtil.isLogin();           // 判断是否登录，返回布尔值
StpUtil.logout();            // 登出
StpUtil.getLoginId();        // 获取当前登录用户ID
```

### 2. 多账号体系认证
支持：**用户端、管理员端、商家端**多类型账号同时登录互不干扰

```java
StpUtil.login(10001, "user");    // 用户端
```

### 1. 权限码校验（细粒度控制）

#### 核心说明

- 权限码是**字符串标识**，如 `user:list`、`order:edit`、`goods:delete`
- 一个用户可以拥有**N个权限码**
- 框架只做**校验**，权限码的查询需要你实现 `StpInterface` 接口

#### 核心 API

```java
// 判断：是否拥有【单个权限】，返回布尔值

// 判断：是否拥有【全部权限】，多个权限必须同时满足
StpUtil.hasPermission("user:edit", "user:delete");  

// 强制校验：无此权限直接抛出异常
StpUtil.checkPermission("user:delete");   

// 判断：是否拥有【任意一个权限】
StpUtil.hasPermissionOr("user:list", "admin:list");
```

### 2. 角色校验（粗粒度控制）

#### 核心说明

- #### 角色是**身份分组**，如 `admin`、`teacher`、`user`、`guest`
- 一个用户可以拥有**多个角色**
- 角色与权限分离：**角色绑定权限，用户绑定角色**

#### 核心 API

```java
// 判断：是否拥有【单个角色】

// 强制校验：无此角色直接抛异常
StpUtil.checkRole("super-admin");

// 判断：是否拥有【全部角色】
StpUtil.hasRole("admin", "operation");  

// 判断：是否拥有【任意一个角色】
StpUtil.hasRoleOr("admin", "user");
```

### 3. 注解式权限控制（最常用、最优雅）

直接加在**Controller接口/方法**上，无需写代码逻辑。

## 四、微服务分布式会话
**解决问题**：集群/微服务下**会话共享**，登录状态多服务器同步

### 支持模式
1. Redis 共享（最常用）
2. JWT 分布式
3. 自定义持久层

**特点**
- 无侵入接入
- 自动同步登录状态
- 分布式环境无缝使用

**解决问题**：集群/微服务、多服务器节点环境下，实现**登录状态共享、会话互通、数据同步**，保证用户在任何一台服务器登录，所有节点都能同步感知。

---

### 1. 核心说明

微服务环境下，默认的内存会话会**不同步**，导致：
- 一台服务登录，另一台服务显示未登录
- 重启服务会话丢失
- 集群环境无法使用

Sa-Token 分布式会话就是为了解决这个问题。

---

### 2. 三种分布式会话模式

#### ① Redis 共享模式（企业最常用、推荐）

**原理**：所有服务连接同一个 Redis，会话统一存储在 Redis 中。
**优点**：功能完整、支持踢人、封禁、在线用户、Session 存储。
**缺点**：依赖 Redis

#### ② JWT 分布式模式

**原理**：会话信息存在 Token 里，服务端无状态。
**优点**：不依赖中间件
**缺点**：无法主动踢人、无法封禁、无法统计在线用户

#### ③ 自定义持久层

**原理**：自己实现会话存储（MySQL、MongoDB 等）
**优点**：高度自定义
**缺点**：需要自己编码实现

---

### 3. Redis 分布式会话（实战完整版）

#### ① 引入依赖

```xml
<!-- Redis 依赖 -->

    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>

<!-- Sa-Token 插件：整合Redis -->

    <version>1.37.0</version>
</dependency>
```

```yaml

    host: 127.0.0.1
    port: 6379

```

### 3. 自动续签（核心机制）

#### 作用

用户只要在操作，就**永不掉线**；长时间不操作，自动过期。

#### 核心配置

```yaml
sa-token:

  timeout: 2592000

  # 活跃超时时间（秒）：2小时无操作则过期
  activity-timeout: 7200

  auto-renew: true
```

#### 机制

- 每次请求接口 → 自动刷新 **最后活跃时间**

### 2. SpringBoot 完整版配置（可直接复制）

```java
import cn.dev33.satoken.interceptor.SaInterceptor;
import cn.dev33.satoken.stp.StpUtil;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.InterceptorRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
public class SaTokenConfig implements WebMvcConfigurer {

    /**
     * 注册 Sa-Token 全局拦截器
     */
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new SaInterceptor(handler -> {
            // 自定义校验逻辑（写在这里就会全局校验）
            StpUtil.checkLogin(); // 全局校验登录

        .addPathPatterns("/**")           // 拦截所有路由
        .excludePathPatterns(             // 放行白名单
            "/user/login",                // 登录接口
            "/user/register",             // 注册接口
            "/captcha/**",                // 验证码接口
            "/file/**",                   // 文件资源
            "/test/**",                   // 测试接口
            "/swagger-resources/**",      // swagger
            "/webjars/**",
            "/v3/api-docs/**"
        );
    }
}
```

### 3. 网关全局过滤器（统一鉴权）

```java
@Component
public class SaTokenFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().toString();
        
        // 白名单放行
        if(path.equals("/user/login") || path.equals("/captcha")){
            return chain.filter(exchange);
        }

        // 校验登录
        StpUtil.checkLogin();

        // 把用户ID传给下游微服务
        ServerHttpRequest newRequest = exchange.getRequest().mutate()
                .header("loginId", StpUtil.getLoginIdAsString())
                .build();

        return chain.filter(exchange.mutate().request(newRequest).build());
    }

    @Override
    public int getOrder() {
        return -100;
    }
}
```
