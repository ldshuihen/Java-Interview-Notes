## 一、Java 基础类源码（基础必问）
### 1. String / StringBuffer / StringBuilder

**核心源码点**
- `String` 底层：`final char[]`（JDK9+ 改为 `byte[] + coder` 优化内存），**不可变**
- 不可变的好处：线程安全、常量池复用、哈希值缓存
- `String.equals()`：先比较引用，再逐字符比较
- `String.intern()`：字符串常量池原理
- `StringBuffer`（线程安全，`synchronized`） vs `StringBuilder`（非线程安全）

**面试问法**：String为什么不可变？+、concat、intern原理



#### 一、底层源码结构

1. **String**
- JDK8及以前：底层 `private final char[] value`
- JDK9+ 优化：改为 `byte[] value + byte coder`
  - coder 标记编码：0=Latin1单字节、1=UTF-16双字节，大幅节省内存
- **被final修饰、数组引用final**，赋值后无法修改原数组，天然**不可变**

2. **StringBuffer**
- 继承 `AbstractStringBuilder`
- 所有增删改方法**加 synchronized 锁**，线程安全
- 扩容逻辑：默认容量16，超容量自动扩容

3. **StringBuilder**
- 同继承 `AbstractStringBuilder`，底层可变char数组
- 方法**无锁**，线程不安全，性能更高
- 日常字符串拼接首选

---

#### 二、核心源码关键方法

##### 1. equals() 源码逻辑

```java
// 核心流程
1. == 引用比较，相同直接true
2. 判断是否为String类型
3. 长度不一致直接false
4. 逐位遍历字符数组对比
```

##### 2. 拼接原理

- **+ 拼接**
  编译期优化：少量拼接 → 常量折叠；
  循环内+拼接 → 底层自动创建 `StringBuilder` 拼接，效率低。
- **concat()**
  底层：复制原数组+新字符串数组，**新建String对象**，不修改原字符串。

##### 3. intern() 源码作用

- 将字符串主动放入**字符串常量池**
- 池中存在：返回池内引用；
- 池中不存在：存入常量池再返回引用；
- 作用：复用对象、减少内存占用。

#### 三、高频面试题标准答案

##### 1. String 为什么不可变？

1. 存储数组 `value` 被 **private + final** 修饰，外部无法访问修改；
2. 所有修改操作（substring/concat/replace）都会**新建字符串对象**，不改动原数组；
3. 类被final修饰，无法被继承破坏不可变性。

##### 2. 不可变的优势

- 线程安全：多线程只读，无需加锁；
- 常量池缓存：字符串复用，节省内存；
- hashCode 缓存：只需计算一次，适合做HashMap的key；
- 安全可靠：防止数据被恶意篡改。

##### 3. 三者区别

| 类            | 线程安全   | 效率 | 底层       | 适用场景                   |
| ------------- | ---------- | ---- | ---------- | -------------------------- |
| String        | 安全       | 低   | 不可变数组 | 少量字符串、常量           |
| StringBuffer  | 安全(sync) | 中   | 可变数组   | 多线程拼接                 |
| StringBuilder | 不安全     | 极高 | 可变数组   | **单线程频繁拼接**（主流） |

### 2. 包装类（Integer、Long等）

**核心源码点**
- **缓存机制**：`IntegerCache` 缓存 `-128 ~ 127`，Long/Short同理
- 自动装箱/拆箱：`Integer.valueOf()` / `intValue()`
- 为什么高频数字要缓存：减少对象创建，提升性能

**面试问法**：`Integer a=128; Integer b=128; System.out.println(a==b);` 结果？



#### 一、核心源码

##### 1. 自动装箱 & 拆箱

- **装箱**：`Integer num = 10;`
  编译后底层：`Integer num = Integer.valueOf(10);`
- **拆箱**：`int n = num;`
  编译后底层：`int n = num.intValue();`

##### 2. IntegerCache 缓存源码逻辑

Integer 内部静态内部类 `IntegerCache`
- 默认缓存范围：**-128 ~ 127**
- 范围之内：直接返回缓存池中**复用对象**
- 范围之外：`new Integer()` 新建对象
- Long、Short、Byte 同理；Character 缓存 0~127；
- Double、Float **没有缓存**。

##### 3. 缓存设计目的

高频小整数使用极多，缓存复用对象，减少GC、节省内存、提升性能。

#### 二、经典面试题标准答案

##### 题目

```java
Integer a = 128;
Integer b = 128;
System.out.println(a == b);

Integer c = 127;
Integer d = 127;
System.out.println(c == d);
```
##### 结果

- `a == b` → **false**
- `c == d` → **true**

##### 原理

1. `127` 在 `-128~127` 缓存区间，`valueOf` 返回同一个缓存对象，引用相同；
2. `128` 超出缓存范围，会创建两个不同 `Integer` 新对象，引用不同；
3. `==` 对于包装类：**比较对象地址**，不是数值。

#### 三、扩展考点

1. 手动修改缓存上限
JVM 参数：`-XX:AutoBoxCacheMax=200`，可上调缓存最大值。
2. 包装类与基本类型对比
`Integer i = 100; int j = 100;`
`i == j` 会触发**自动拆箱**，比较数值，永远相等。
3. 阿里规范
包装类对象比较数值，**一律使用 equals**，杜绝缓存坑。

### 3. Object 类

**核心源码点**
- `equals()` / `hashCode()` 规范（必须同时重写）
- `toString()`
- `wait()` / `notify()` / `notifyAll()`（必须配合`synchronized`使用）
- `clone()` 浅拷贝/深拷贝

#### 核心源码点

1. `equals()`
默认实现：**直接 `==` 比较对象地址**。
```java
public boolean equals(Object obj) {
    return (this == obj);
}
```

2. `hashCode()`
本地方法，根据对象内存地址生成哈希码；
**契约规范**：
- `equals` 相等 → `hashCode` 必须相等；
- `hashCode` 相等 → `equals` 不一定相等。
**业务实体必须同时重写 `equals & hashCode`**。

3. `wait() / notify() / notifyAll()`
- 三者都是 **final 本地方法**；
- 必须在 **synchronized 同步代码块/方法** 中调用；
- `wait()`：释放锁，进入等待队列；
- `notify()`：随机唤醒一个等待线程；
- `notifyAll()`：唤醒全部等待线程。

4. `clone()`
- 实现 `Cloneable` 接口才可调用，否则抛异常；
- 默认是 **浅拷贝**：基本类型直接复制，引用类型只复制地址；
- 深拷贝需手动递归重写 / 序列化实现。

5. `toString()`
默认打印：**类名@十六进制哈希值**，业务开发建议重写。

2. wait 和 sleep 区别？
3. 浅拷贝、深拷贝区别与实现方式？

## 二、集合框架源码（面试最高频）
### 1. List 家族
#### ✅ ArrayList（最常问）
- 底层结构：**可变 Object 数组**，基于数组实现
- 初始化机制：JDK7+ 无参构造默认空数组，首次 add 初始化容量为 10
- 扩容规则：容量不足时**1.5倍扩容**，核心源码：`oldCapacity + (oldCapacity >> 1)`
- 特性：**线程不安全**；随机访问查询快，中间/头部增删需要移动元素，效率低
- 新增细节：尾部直接添加、扩容拷贝；`modCount` 记录修改次数，支持快速失败

#### ✅ LinkedList
- 底层结构：**双向链表**，JDK1.7 废弃循环链表，使用头尾节点
- 实现接口：实现 `Deque`，可作为栈、队列、双端队列使用
- 特性：**线程不安全**；任意位置增删只修改指针，效率高；遍历查询需要从头结点遍历，查询慢
- 额外：内存占用更高，每个节点存数据、前驱、后继指针

#### ✅ Vector / Stack

- Vector：底层数组，**所有方法加 synchronized**，线程安全但并发性能极差，基本淘汰
- 扩容：默认**2倍扩容**，区别于 ArrayList 的1.5倍
- Stack：继承 Vector，实现栈先进后出；设计老旧、效率低，官方推荐使用 `ArrayDeque` 替代

---
**高频面试点**
1. ArrayList 为什么线程不安全？多线程并发 add 会出现元素覆盖、数组越界
2. ArrayList 与 LinkedList 适用场景区分
3. 快速失败 fail-fast 原理：modCount 版本号校验





### 2. Set 家族
#### ✅ HashSet
- 底层依托 **HashMap** 实现，元素存 Map 的 key，value 统一为静态常量 `PRESENT`
- 去重规则：先判断 `hashCode()`，哈希冲突再执行 `equals()` 校验
- 无序、无索引、**线程不安全**，查询、增删效率高

#### ✅ TreeSet
- 底层依托 **TreeMap**，基于**红黑树**结构
- 天然有序：支持**自然排序**（元素实现Comparable）/ 自定义比较器Comparator
- 元素不可为null、**线程不安全**，适合排序去重场景

#### ✅ LinkedHashSet

- 继承HashSet，底层基于 **LinkedHashMap**
- 维护双向链表，可保证**插入顺序**
- 介于HashSet与TreeSet之间：去重+有序，性能优于TreeSet

---
**高频面试小结**
1. 三者都不可重复，去重核心：`hashCode + equals`
2. HashSet 无序；TreeSet 排序有序；LinkedHashSet 插入有序
3. 均非线程安全，并发场景需手动加锁或使用工具类



### 3. Map 家族（重中之重）
#### HashMap（JDK1.8 高频）

##### 核心源码考点

1. **底层数据结构**
JDK1.8：**数组 + 单向链表 + 红黑树**
- 数组：哈希桶，存放链表/树的头节点
- 链表：处理哈希冲突
- 红黑树：冲突严重时优化查询效率

2. **关键阈值（树化/退化）**
- 树化条件：**链表长度 ≥ 8 且 数组容量 ≥ 64**
- 只满足链表≥8、数组小于64：只触发**扩容**，不树化
- 树退化条件：红黑树节点数 **≤ 6**，退化为单向链表
> 6、8 中间留7作为缓冲，防止频繁树化/退化、性能抖动

3. **基础参数**
- 默认初始容量：`16`（必须是2的幂）
- 默认负载因子：`0.75`
- 扩容阈值 = 容量 × 负载因子
> 0.75 是空间与查询效率的平衡值

4. **扩容机制 resize()**
- 达到扩容阈值，触发**2倍扩容**
- 扩容后容量、阈值全部翻倍
- 重新计算hash：
  - 高位为0：留在原桶下标
  - 高位为1：迁移至「原下标 + 旧容量」位置
  - 无需全部重新哈希，效率大幅提升

5. **JDK1.8 核心优化**
- 由 **头插法 改为 尾插法**
  解决 JDK1.7 多线程扩容导致的**循环链表、死循环**问题
- 引入红黑树，高冲突下查询时间复杂度从 $O(n) \to O(\log n)$
- 优化hash扰动算法，减少哈希冲突
- 插入逻辑：先数组、再链表、最后树化

6. **哈希寻址 & 插入流程**
- 下标计算：`(hashCode ^ 高位) & (table.length - 1)`
- put流程：
  1. 数组为空 → 先初始化扩容
  2. 桶位置无元素 → 直接放入数组
  3. 桶位置是链表 → 遍历比对key，相等则覆盖，末尾追加
  4. 链表长度达标+数组达标 → 转为红黑树
  5. 桶位置是红黑树 → 树节点插入/覆盖

7. **线程不安全问题**
- 多线程并发 put：元素覆盖、数据丢失
- JDK1.7 头插法：扩容易产生循环链表，CPU 100%
- JDK1.8 尾插法解决死链，但**依旧线程不安全**
- 并发场景推荐：`ConcurrentHashMap`

---
##### 面试简答背诵版

HashMap1.8底层为数组+链表+红黑树，初始容量16、负载因子0.75；链表长度大于8且数组大于64树化，小于6退化为链表；扩容为2倍扩容，采用尾插法，优化了哈希重定位；非线程安全，多线程会数据覆盖，并发使用CHM。

#### HashMap 底层原理

（JDK1.8 面试完整版，简洁好背）

##### 一、底层结构

**数组 + 单向链表 + 红黑树**

- 数组：哈希桶，存放桶位头节点
- 链表：解决哈希冲突
- 红黑树：链表过长时优化查询性能

##### 二、核心常量

1. 默认初始容量：**16**（必须是 2 的幂）
2. 默认负载因子：**0.75**
3. 扩容阈值 = 容量 × 0.75
4. 树化阈值：链表长度 **≥8**
5. 树化前提：数组容量 **≥64**
6. 退化阈值：树节点 **≤6**

##### 三、下标计算原理

```java
hash = key.hashCode() ^ (hash >>> 16);  // 高低位扰动，减少冲突
index = hash & (table.length - 1);     // 替代取模，高效寻址
```

##### 四、Put 插入完整流程

1. 判断数组是否为空，为空则**初始化容量16**
2. 计算 key 哈希 & 桶下标，找到对应桶位
3. 桶位无元素：直接新建节点放入数组
4. 桶位有元素：
   - 首节点 key 相同：**覆盖 value**
   - 是链表：遍历链表，key 重复则覆盖，不重复则**尾插**
   - 链表长度≥8 & 数组≥64：**链表转红黑树**
   - 是红黑树：走树节点插入逻辑
5. 插入完成，判断元素个数是否超过扩容阈值，超过则执行 **resize 2倍扩容**

##### 五、扩容机制 resize()

1. 容量、阈值统一**扩大2倍**
2. 元素重新迁移：
   - 利用高位hash位判断，无需重新完整计算哈希
   - 元素只分两类：留在原桶 / 迁移到「原下标+旧容量」
3. JDK1.8 使用**尾插法**

##### 六、树化与退化

- 树化：链表过长查询太慢，红黑树时间复杂度 $O(\log n)$
- 退化：元素变少，树结构维护成本高，退回链表
- 6 和 8 中间留白，避免频繁转换、性能震荡

##### 七、JDK1.7  vs JDK1.8

1. JDK1.7：**数组+链表**，头插法，多线程扩容会死循环
2. JDK1.8：尾插法 + 红黑树，解决死链问题，但**仍然线程不安全**

##### 八、线程不安全表现

- 并发 put 导致**元素覆盖、数据丢失**
- 无加锁、无CAS，并发环境必须用 `ConcurrentHashMap`

#### ConcurrentHashMap

（并发核心）

- JDK1.7：**Segment分段锁**（ReentrantLock），细粒度锁
- JDK1.8：**CAS + synchronized**，锁链表/红黑树头节点
- 线程安全，性能优于Hashtable
- `size()`：无锁统计 + 冲突时加锁

##### 一、整体定位

`ConcurrentHashMap` 是**线程安全**的键值存储容器，
替代 `Hashtable`、同步HashMap；
**高并发、高性能**，分段锁升级，适合多线程场景。

---

##### 二、JDK1.7 与 JDK1.8 核心区别

###### JDK1.7

- 结构：**Segment分段锁 + 数组+链表**
- 锁粒度：Segment 段锁（ReentrantLock），默认16个分段
- 缺点：分段固定，并发上限有限，无红黑树

###### JDK1.8（主流面试重点）

- 结构：**数组 + 链表 + 红黑树**，和HashMap一致
- 锁粒度：**细化到桶位**
- 核心并发手段：**CAS + synchronized + 红黑树**
- 抛弃 Segment，并发性能大幅提升

---

##### 三、JDK1.8 底层核心结构

1. **table[]**：哈希桶数组，存放链表/红黑树根节点
2. **Node**：普通链表节点
3. **TreeNode**：红黑树节点
4. **ForwardingNode**：扩容转移标记节点（扩容时占位）
5. **核心控制变量**
   - `sizeCtl`：复合变量
     - 正数：下次扩容阈值
     - 负数：正在初始化 / 正在扩容

---

##### 四、加锁机制（源码核心）

###### 1. 并发控制三板斧

1. **CAS**：无锁操作，赋值、计数、占位
2. **synchronized**：只锁住**当前桶的头节点**
3. **volatile**：保证数组、节点引用可见性

###### 2. 锁粒度极大优化

- 不锁整个数组、不锁整个集合
- 只锁住**当前哈希桶的第一个节点**
- 不同桶位并发写入互不影响，并发度极高

> 区别：Hashtable 锁整张表，效率极低。

---

##### 五、put() 源码执行流程

1. 校验key/value非空
2. 判断数组是否未初始化，**CAS 懒加载初始化**
3. 计算hash & 桶下标
4. **桶位为空**：
   通过 **CAS 无锁** 直接写入新Node
5. **桶位正在扩容**（ForwardingNode）：
   协助扩容 `helpTransfer()`
6. **桶位已有数据**：
   对**桶头节点加 synchronized 锁**
   - 链表：遍历，key相同覆盖，末尾追加
   - 链表长度≥8 → 转为红黑树
   - 红黑树：走树节点插入
7. 插入完成，更新元素数量，判断是否需要扩容

---

##### 六、扩容机制 transfer()

1. 触发条件：元素数量达到阈值
2. 容量**2倍扩容**，和HashMap一致
3. 支持**多线程协助扩容**
   - 一个线程扩容，其他写入线程发现扩容标记，主动帮忙迁移数据
   - 大幅减少单线程扩容耗时
4. 迁移期间桶位标记为 `ForwardingNode`，防止写入冲突

---

##### 七、元素计数原理（size）

1. 不使用全局锁统计总数
2. 采用 **BaseCount + CounterCell 数组** 分段计数
   - 低竞争：直接累加 baseCount
   - 高竞争：写入不同 CounterCell 单元格，减少CAS冲突
3. 最终总数 = baseCount + 所有CounterCell累加

---

##### 八、树化/退化规则（同HashMap）

- 链表长度 ≥ 8 && 数组长度 ≥ 64 → 红黑树
- 树节点 ≤ 6 → 退化为链表
- 时间复杂度：链表O(n) → 红黑树O(logn)

---

##### 九、线程安全细节 & 常见问题

1. **为什么不用 ReentrantLock？**
`synchronized` 后期JVM优化（偏向锁/轻量级锁/重量级锁），性能更好，开销更低。
2. **弱一致性**
迭代器、size 是**弱一致**，不加锁，不会抛并发修改异常。
3. **key/value 不可为 null**
防止二义性，区别于HashMap。

---

##### 十、高频面试简答背诵版

JDK1.8 ConcurrentHashMap 底层为**数组+链表+红黑树**；
采用 **CAS + synchronized 锁桶头节点** 实现线程安全，锁粒度极细；
懒加载初始化、多线程协助扩容；
通过 baseCount+CounterCell 分段统计元素总数；
不允许key、value为null，高并发性能远优于Hashtable。

#### Hashtable

- 全方法`synchronized`，线程安全但低效
- 不允许key/value为null（HashMap允许1个null key）

#### TreeMap

- 红黑树实现，key有序
- 必须实现Comparable或传入Comparator

##### 核心源码点

1. **底层结构**
底层基于 **红黑树（自平衡二叉查找树）**，
元素以 key 为依据排序存储。

2. **有序特性**
- 自然排序：key 实现 `Comparable` 接口
- 自定义排序：构造器传入 `Comparator`
- 全局**升序有序**，区别于 HashMap 无序、LinkedHashMap 插入有序

3. **去重规则**
通过**比较器 compare 返回值**判断重复，
而非 `equals / hashCode`；
compare 返回 0 判定为同一个 key，覆盖 value。

4. **线程安全**
**非线程安全**；
并发场景可使用 `Collections.synchronizedSortedMap`。

5. **核心源码机制**
- 插入、删除后自动触发**红黑树自平衡**（变色、左旋、右旋）
- 增删查时间复杂度：$O(\log n)$
- 不允许 key 为 null（会空指针），value 允许 null

6. **遍历特性**
默认**中序遍历**，输出结果天然有序。

---
##### 高频面试对比

1. HashMap：数组+链表+红黑树，无序，查询快
2. LinkedHashMap：有序（插入/访问），LRU缓存常用
3. TreeMap：红黑树，**key排序有序**，适合排序场景

---
##### 极简背诵版

TreeMap 底层是**红黑树**，依靠 Comparable/Comparator 实现 key 排序；
增删节点自动平衡，时间复杂度 $O(\log n)$；
依靠比较器去重，非线程安全，key 不能为 null，适合需要有序的业务场景。

## 三、并发编程源码（中高级必问）
### 1. AQS（AbstractQueuedSynchronizer）
**核心源码点**
- 核心：`volatile int state` + **CLH双向队列**
- 核心方法：`tryAcquire`、`tryRelease`、`acquireQueued`
- 基于**CAS + 自旋**实现无锁/轻量级锁竞争

**面试问法**：AQS原理？CountDownLatch/ReentrantLock基于AQS怎么实现？

### 2. ReentrantLock
- 可重入：`state`变量记录重入次数
- 公平锁/非公平锁：`lock()`时是否直接尝试CAS
- 底层：AQS独占锁实现

### 3. ThreadPoolExecutor 线程池
**核心源码点**
- 核心参数：核心线程、最大线程、队列、拒绝策略
- 线程池运行流程：核心线程 → 队列 → 最大线程 → 拒绝策略
- `ctl`变量：**高3位存状态，低29位存线程数**
- 5种状态：RUNNING、SHUTDOWN、STOP、TIDYING、TERMINATED

**面试问法**：线程池执行流程？为什么不建议用Executors创建？

### 4. ThreadLocal
- 底层：`ThreadLocalMap`（Entry继承WeakReference）
- 内存泄漏原因：key为弱引用，value强引用
- 解决方案：使用完`remove()`

### 5. LockSupport
- `park()` / `unpark()`：线程阻塞/唤醒
- 比wait/notify更灵活，无需锁、无顺序依赖

### 6. 原子类（AtomicInteger等）
- 底层：**Unsafe类 + CAS**
- CAS：compare and swap，无锁乐观锁
