# 计算机网络



### **TCP 三次握手 & 四次挥手 详解**（通俗易懂版）

TCP 是面向连接的可靠传输协议，建立连接和断开连接都必须经过“握手”来协商参数。

---

#### 1. TCP 三次握手（Three-Way Handshake）

**目的**：建立连接 + 双方确认各自的初始序列号（ISN），为可靠传输做准备。

##### 过程图解：

```text
客户端                  服务端
   |                       |
   |  SYN = 1, seq = x     |  ------>   (第一次握手)
   |                       |
   |                       |  <------   SYN=1, ACK=1, seq=y, ack=x+1  (第二次握手)
   |                       |
   |  ACK=1, seq=x+1, ack=y+1 | ------>  (第三次握手)
   |                       |
   ✅ 连接建立成功
```

**详细步骤**：

1. **第一次握手**（客户端 → 服务端）  
   - 客户端发送 SYN 报文（SYN=1），携带客户端初始序列号 `seq = x`。  
   - 状态：客户端进入 `SYN_SENT`。

2. **第二次握手**（服务端 → 客户端）  
   - 服务端回复 SYN+ACK 报文（SYN=1, ACK=1）。  
   - `seq = y`（服务端初始序列号），`ack = x+1`（确认收到客户端的 SYN）。  
   - 状态：服务端进入 `SYN_RCVD`。

3. **第三次握手**（客户端 → 服务端）  
   - 客户端回复 ACK 报文（ACK=1）。  
   - `seq = x+1`，`ack = y+1`（确认收到服务端的 SYN）。  
   - 状态：客户端进入 `ESTABLISHED`，服务端收到后也进入 `ESTABLISHED`。

**为什么需要三次？**  
- 两次不够：无法确认客户端的接收能力（第二次握手后，服务端不确定客户端是否收到自己的 SYN）。  
- 三次刚好可以让双方都确认：**自己能发、对方能收、对方能发、自己能收**。

---

#### 2. TCP 四次挥手（Four-Way Wave）

**目的**：关闭连接，释放双方资源。

##### 过程图解：

```text
客户端（主动关闭）         服务端（被动关闭）
   |                          |
   |  FIN=1, seq=u          |  ------>   (第一次挥手)
   |                          |
   |                        |  <------   ACK=1, ack=u+1   (第二次挥手)
   |                          |
   |  （进入 FIN_WAIT_2）     |
   |                          |
   |                        |  <------   FIN=1, seq=v, ACK=1   (第三次挥手)
   |                          |
   |  ACK=1, ack=v+1        |  ------>   (第四次挥手)
   |                          |
   ✅ 客户端进入 TIME_WAIT（等待 2MSL）
   ✅ 服务端进入 CLOSED
```

**详细步骤**：

1. **第一次挥手**：客户端发送 FIN 报文（FIN=1），表示“我要关闭了”，`seq = u`。  
   客户端进入 `FIN_WAIT_1`。

2. **第二次挥手**：服务端回复 ACK（ACK=1），`ack = u+1`。  
   服务端进入 `CLOSE_WAIT`（此时服务端可能还有数据要发送）。

3. **第三次挥手**：服务端发送 FIN 报文（FIN=1），表示“我也关闭了”。  
   服务端进入 `LAST_ACK`。

4. **第四次挥手**：客户端回复 ACK（ACK=1），`ack = v+1`。  
   客户端进入 `TIME_WAIT` 状态，等待 **2MSL**（Maximum Segment Lifetime，通常 2×2分钟 = 4分钟）后关闭。

---

#### 3. 常见面试问题解答

| 问题                             | 答案                                                         |
| -------------------------------- | ------------------------------------------------------------ |
| **为什么是三次握手？**           | 保证双方都能确认对方的收发能力，避免历史连接干扰             |
| **为什么是四次挥手？**           | 因为 TCP 是全双工，双方都要单独关闭自己的发送通道（FIN），关闭是分两次完成的 |
| **TIME_WAIT 状态的作用？**       | 防止最后一次 ACK 丢失；避免新连接复用老连接的序列号导致混乱  |
| **为什么 TIME_WAIT 要等 2MSL？** | 确保最后一个 ACK 能到达对方，同时让本次连接的所有报文在网络中消失 |
| **半关闭状态是什么？**           | 发送 FIN 后还能接收数据（CLOSE_WAIT）                        |

---

#### 4. 状态变迁总结

**三次握手状态流**：
`CLOSED → SYN_SENT → ESTABLISHED`

`CLOSED → SYN_RCVD → ESTABLISHED`

**四次挥手状态流**：
`ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED`

`ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED`

---

**记忆口诀**：
- **握手**：SYN → SYN+ACK → ACK（三次确认）
- **挥手**：FIN → ACK → FIN → ACK（双方各自说再见）

### **OSI 网络模型（Open System Interconnection）详解**

OSI 模型是国际标准化组织（ISO）在 1984 年提出的**7层网络通信模型**，它是网络协议设计和故障排查的**理论基础**，帮助我们理解网络中数据的流动过程。

#### 1. OSI 七层模型（从上到下）

| 层级（中文/英文）              | 主要功能                   | 关键协议/技术                       | 传输单位          | 核心职责                             |
| ------------------------------ | -------------------------- | ----------------------------------- | ----------------- | ------------------------------------ |
| **7. 应用层**<br>Application   | 提供网络服务给应用程序     | HTTP, HTTPS, FTP, SMTP, DNS, Telnet | -                 | 用户接口、文件传输、邮件等           |
| **6. 表示层**<br>Presentation  | 数据格式转换、加密解密     | SSL/TLS, JPEG, ASCII, EBCDIC        | -                 | 翻译、压缩、加密                     |
| **5. 会话层**<br>Session       | 建立、管理、终止会话       | NetBIOS, RPC, SIP                   | -                 | 对话控制、同步、检查点               |
| **4. 传输层**<br>Transport     | 端到端可靠传输             | TCP, UDP, SCTP                      | **段（Segment）** | 端口号、可靠传输、流量控制、拥塞控制 |
| **3. 网络层**<br>Network       | 逻辑寻址和路由选择         | IP, ICMP, OSPF, BGP                 | **包（Packet）**  | IP地址、路由、逻辑寻址               |
| **2. 数据链路层**<br>Data Link | 物理寻址、帧同步、错误检测 | Ethernet, PPP, ARP, VLAN, Wi-Fi     | **帧（Frame）**   | MAC地址、交换机、错误校验            |
| **1. 物理层**<br>Physical      | 比特流的传输               | 网线、光纤、Hub、中继器             | **比特（Bit）**   | 电压、信号、光信号、接口标准         |

---

#### 2. 各层通俗理解（记忆口诀）

**从下往上记**（推荐）：

**物链网输会表应**

- **物理层**：看得见摸得着的硬件（网线、光纤、电压、0和1的传输）。
- **数据链路层**：在同一局域网内通信（MAC地址、交换机、帧）。
- **网络层**：跨网络通信（IP地址、路由器、选择最佳路径）。
- **传输层**：可靠送达（TCP保证不丢不重，UDP快速但不保证）。
- **会话层**：管理“对话”（建立连接、保持会话、断开）。
- **表示层**：翻译官（数据格式转换、加密、压缩）。
- **应用层**：我们直接使用的（浏览器、邮件客户端等）。

**记忆口诀**：**请 勿 让 传 输 会 表 应**  
（物链网输会表应）

---

#### 3. 数据封装与解封装过程（重要！）

数据从上层向下层流动时，每层都会**加上自己的头部**（封装）：

```
应用层数据
    ↓ + 表示层头
表示层数据
    ↓ + 会话层头
会话层数据
    ↓ + 传输层头（TCP/UDP头 + 端口）
传输层段 (Segment)
    ↓ + 网络层头（IP头）
网络层包 (Packet)
    ↓ + 数据链路层头尾（MAC头 + FCS校验）
数据链路层帧 (Frame)
    ↓ 
物理层比特流 (Bit)
```

接收端从下往上**去掉头部**（解封装）。

---

#### 4. OSI vs TCP/IP 模型对比

| OSI 7层    | TCP/IP 4层模型 | 对应关系 |
| ---------- | -------------- | -------- |
| 应用层     | 应用层         | 合并     |
| 表示层     | 应用层         | 合并     |
| 会话层     | 应用层         | 合并     |
| 传输层     | 传输层         | 相同     |
| 网络层     | 网络层         | 相同     |
| 数据链路层 | 网络接口层     | 合并     |
| 物理层     | 网络接口层     | 合并     |

**实际工作中**更常用的是 **TCP/IP 模型**，但 OSI 模型更细，适合教学和深入排查。

---

#### 5. 各层常用设备

- **物理层**：网线、Hub、中继器、光模块
- **数据链路层**：交换机、网卡（MAC地址工作）
- **网络层**：路由器、三层交换机
- **传输层及以上**：主机、防火墙、负载均衡器

### **TCP vs UDP 详细对比**（面试/开发必备）

TCP 和 UDP 都是传输层（OSI 第 4 层）的协议，但设计理念完全不同。

#### 1. 核心区别总结

| 对比维度     | **TCP (Transmission Control Protocol)** | **UDP (User Datagram Protocol)**   |
| ------------ | --------------------------------------- | ---------------------------------- |
| **中文名称** | 传输控制协议                            | 用户数据报协议                     |
| **连接性**   | **面向连接**（三次握手建立连接）        | **无连接**（直接发数据）           |
| **可靠性**   | **可靠**（保证送达、不重复、不乱序）    | **不可靠**（尽力而为，不保证送达） |
| **速度**     | 较慢（有确认、重传、拥塞控制）          | **极快**（开销小）                 |
| **传输顺序** | 有序（按序到达）                        | 不保证顺序                         |
| **流量控制** | 有（滑动窗口）                          | 无                                 |
| **拥塞控制** | 有                                      | 无                                 |
| **头部开销** | 20~60 字节                              | **8 字节**（极小）                 |
| **适用场景** | 需要高可靠性的场景                      | 需要高速度、低延迟的场景           |

---

#### 2. 详细特性对比

**TCP 的优势（可靠传输）**：

- 三次握手建立连接 + 四次挥手断开连接
- 序列号（Seq）和确认号（Ack）机制
- 超时重传
- 滑动窗口（流量控制）
- 拥塞控制（慢开始、拥塞避免、快重传、快恢复）
- 校验和（Checksum）检测错误

**UDP 的优势（简单高效）**：
- 无需建立连接，延迟极低
- 资源消耗少
- 支持一对一、一对多、多对一、多对多通信
- 适合实时性要求高的业务

---

#### 3. 常见使用场景

**TCP 典型应用**：
- HTTP / HTTPS（网页浏览）
- FTP（文件传输）
- SMTP（邮件发送）
- SSH、Telnet（远程登录）
- MySQL、Redis（大部分数据库连接）
- 任何需要**数据完整性**的场景

**UDP 典型应用**：
- 视频会议、直播（实时音视频）
- 在线游戏（王者荣耀、吃鸡等）
- DNS 查询（大部分情况）
- DHCP（动态主机配置）
- VoIP（网络电话）
- IoT（物联网）设备通信
- 实时监控、日志推送

**两者都用的场景**：
- QUIC 协议（基于 UDP 的可靠传输，HTTP/3 使用）
- 很多应用会同时使用 TCP + UDP（如游戏中控制指令用 TCP，位置同步用 UDP）

---

#### 4. 报文头部对比

- **TCP Header**：最小 20 字节，包含源端口、目标端口、序列号、确认号、窗口大小、标志位（SYN、ACK、FIN、RST 等）
- **UDP Header**：仅 8 字节，只有源端口、目标端口、长度、校验和

---

#### 5. 端口号（Port）

- 两者都使用 **0~65535** 的端口号
- 知名端口（0-1023）：如 80（HTTP）、443（HTTPS）、53（DNS）、3306（MySQL）
- 注册端口（1024-49151）
- 动态端口（49152-65535）

---

#### 6. 实际开发选择建议

**选择 TCP**：
- 需要保证数据不丢失、不出错（如订单、支付、用户数据）
- 可以接受一定延迟

**选择 UDP**：
- 追求极致速度和实时性
- 允许少量数据丢失（丢几帧画面没关系，但卡顿更难受）
- 可在应用层自己实现可靠性（如重传机制）

**现代趋势**：
- HTTP/3 使用 **QUIC**（基于 UDP + 自定义可靠性），兼顾了速度和可靠性。
- 很多实时应用正在从 TCP 转向 UDP + 应用层重传。

---

**记忆口诀**：
- **TCP**：稳定、慢、可靠，像**打电话**（要先拨号、确认对方接听）
- **UDP**：快、不靠谱，像**寄快递**（扔过去不管是否收到）

**HTTP 协议体系详解**（Java 面试最高频部分）

以下内容按面试重要程度和出现频率整理，建议重点掌握。

### 1. HTTP 1.0 / 1.1 / 2.0 / 3.0 主要区别

| 版本         | 发布年份 | 连接方式           | 传输格式   | 多路复用     | 头部压缩 | Server Push | 队头阻塞     | 底层传输       |
| ------------ | -------- | ------------------ | ---------- | ------------ | -------- | ----------- | ------------ | -------------- |
| **HTTP/1.0** | 1996     | 短连接             | 文本       | 不支持       | 不支持   | 不支持      | 存在         | TCP            |
| **HTTP/1.1** | 1999     | **长连接**（默认） | 文本       | 管道化（差） | 不支持   | 不支持      | 存在         | TCP            |
| **HTTP/2.0** | 2015     | 长连接             | **二进制** | **支持**     | **支持** | **支持**    | 部分解决     | TCP            |
| **HTTP/3.0** | 2022     | 长连接             | 二进制     | **支持**     | 支持     | 支持        | **彻底解决** | **QUIC (UDP)** |

**核心区别总结**：

- **HTTP/1.1** 相比 1.0 最大改进：**默认长连接**（Connection: keep-alive）和 Host 头成为必传字段。
- **HTTP/2** 革命性变化：
  - 二进制协议（更高效、更不易出错）
  - 多路复用（同一个连接上可同时发送多个请求）
  - 头部压缩（HPACK）
  - Server Push（服务器主动推送）
- **HTTP/3**：基于 QUIC 协议，完全解决 TCP 的队头阻塞问题，目前主流浏览器和 CDN 已广泛支持。

---

### 2. HTTP 请求方法语义和幂等性

**幂等性**：同一个请求执行多次，**服务器状态不变**（不会产生副作用）。

| 方法        | 语义             | 是否幂等 | 是否安全（不修改资源） | 常见用途         |
| ----------- | ---------------- | -------- | ---------------------- | ---------------- |
| **GET**     | 获取资源         | 是       | 是                     | 查询             |
| **POST**    | 创建/提交资源    | 否       | 否                     | 新增、提交表单   |
| **PUT**     | **替换**资源     | 是       | 否                     | 更新（全量）     |
| **DELETE**  | 删除资源         | 是       | 否                     | 删除             |
| **PATCH**   | **部分更新**资源 | 否       | 否                     | 局部更新         |
| **HEAD**    | 只获取响应头     | 是       | 是                     | 检查资源是否存在 |
| **OPTIONS** | 预检请求（CORS） | 是       | 是                     | 跨域检查         |
| **TRACE**   | 回显请求（诊断） | 是       | 是                     | 已基本废弃       |

**重点记忆**：
- **安全方法**：GET、HEAD、OPTIONS（不会修改服务器资源）
- **幂等方法**：GET、HEAD、PUT、DELETE、OPTIONS
- **非幂等**：POST、PATCH（多次调用可能产生不同结果）

---

### 3. 常见 HTTP 状态码

**2xx（成功）**：
- **200 OK**：请求成功（最常见）
- **204 No Content**：成功但无返回体
- **206 Partial Content**：范围请求返回（断点续传）

**3xx（重定向）**：
- **301 Moved Permanently**：**永久重定向**（SEO 友好）
- **302 Found**：**临时重定向**（最常用）
- **303 See Other**：通常用于 POST 重定向到 GET
- **304 Not Modified**：**协商缓存命中**（非常重要）
- **307 Temporary Redirect**：临时重定向（保持原请求方法）

**4xx（客户端错误）**：
- **400 Bad Request**：请求参数错误
- **401 Unauthorized**：未认证（需要登录）
- **403 Forbidden**：已认证但无权限
- **404 Not Found**：资源不存在
- **405 Method Not Allowed**：方法不允许
- **429 Too Many Requests**：请求频率过高（限流）

**5xx（服务端错误）**：
- **500 Internal Server Error**：服务器内部错误（最常见）
- **502 Bad Gateway**：网关/代理后端服务不可用
- **503 Service Unavailable**：服务器过载或维护
- **504 Gateway Timeout**：网关超时

---

### 4. 常用 HTTP 请求头和响应头

**请求头（Request Headers）**：
- **Host**：必传，指定服务器域名和端口
- **User-Agent**：客户端信息
- **Accept**：客户端能接收的内容类型
- **Content-Type**：请求体格式（`application/json`、`application/x-www-form-urlencoded`、`multipart/form-data`）
- **Content-Length**：请求体长度
- **Cookie**：客户端 Cookie
- **Authorization**：认证信息（Bearer Token、Basic Auth）
- **Referer**：请求来源页面
- **Cache-Control**：缓存控制
- **If-None-Match / If-Modified-Since**：协商缓存

**响应头（Response Headers）**：
- **Set-Cookie**：服务器设置 Cookie（可多个）
- **Content-Type**：响应体类型
- **Content-Length**：响应体长度
- **Location**：重定向地址（3xx 使用）
- **Cache-Control**：强缓存控制（max-age、no-cache、no-store）
- **ETag**：实体标签（协商缓存）
- **Expires**：过期时间（强缓存）
- **Access-Control-Allow-Origin**：CORS 跨域
- **Connection: keep-alive**：长连接

---

### 5. HTTPS 原理

**HTTPS = HTTP + TLS/SSL**

**核心解决两个问题**：
1. **加密传输**（防止窃听）
2. **身份认证**（防止中间人攻击）

**加密方式组合**：
- **非对称加密**（RSA / ECDHE）：用于密钥交换
- **对称加密**（AES）：用于实际数据加密
- **CA 证书 + 数字签名**：验证服务器身份

**TLS 握手过程（TLS 1.2 简化版）**：
1. ClientHello（客户端支持的版本、密码套件、随机数）
2. ServerHello（服务器选定版本、密码套件、随机数 + 证书）
3. 客户端验证证书 → 生成对称密钥（用服务器公钥加密后发送）
4. 双方切换成对称加密，开始传输数据

**TLS 1.3** 比 1.2 更快（握手减少到 1-RTT）。

---

### 6. HTTP 长连接、管道化、队头阻塞

- **长连接**（Keep-Alive）：一个 TCP 连接上可以发送多个 HTTP 请求，减少三次握手开销。
- **管道化**（Pipelining）：客户端可以在一个连接上连续发送多个请求，但**服务器必须按顺序返回**。
- **队头阻塞**（Head of Line Blocking）：前一个请求阻塞，后面的请求即使已经处理完也无法返回（HTTP/1.1 最大问题）。

**HTTP/2 通过多路复用彻底解决**：一个连接上多个 Stream 独立传输，互不干扰。

---

### 1. 5 种 IO 模型（Linux 底层视角）

| IO 模型                            | 是否阻塞 | 是否需要轮询 | 内核准备数据 | 应用复制数据 | 代表实现                 | 性能 |
| ---------------------------------- | -------- | ------------ | ------------ | ------------ | ------------------------ | ---- |
| **阻塞 IO** (Blocking)             | 阻塞     | 不需要       | 阻塞等待     | 阻塞         | 传统 BIO                 | 差   |
| **非阻塞 IO** (Non-blocking)       | 不阻塞   | 需要         | 立即返回     | 阻塞         | Socket 设置 `O_NONBLOCK` | 一般 |
| **IO 多路复用** (I/O Multiplexing) | 阻塞     | 不需要       | 内核通知     | 阻塞         | select / poll / epoll    | 好   |
| **信号驱动 IO** (Signal-driven)    | 不阻塞   | 不需要       | 信号通知     | 阻塞         | SIGIO                    | 较好 |
| **异步 IO** (Asynchronous)         | 不阻塞   | 不需要       | 内核完成     | 不阻塞       | aio_read / io_uring      | 最好 |

**通俗理解**：

- **阻塞 IO**：你去饭店点餐，站在柜台前一直等到饭做好（全程阻塞）。
- **非阻塞 IO**：你点完餐后不断问“好了吗？好了吗？”（忙轮询）。
- **IO 多路复用**：服务员（内核）同时看着很多桌，等哪桌饭好了通知你（epoll 最强大）。
- **异步 IO**：你点完餐直接走，饭做好服务员给你打电话送过来（真正异步）。

**Java 中对应关系**：
- **BIO** → 阻塞 IO
- **NIO** → IO 多路复用（Selector + epoll）
- **AIO** → 异步 IO（NIO 2.0）

---

### 2. Java 中 BIO、NIO、AIO 对比

| 维度       | **BIO** (Blocking IO) | **NIO** (New IO / Non-blocking IO) | **AIO** (Asynchronous IO) |
| ---------- | --------------------- | ---------------------------------- | ------------------------- |
| 出现版本   | JDK 1.0+              | JDK 1.4+                           | JDK 1.7+                  |
| 编程模型   | 同步阻塞              | 同步非阻塞                         | 异步非阻塞                |
| 线程模型   | 一连接一线程          | 一个 Selector 管理多个连接         | 回调或 CompletionHandler  |
| 适用场景   | 连接数少、并发低      | 高并发（几千~几万连接）            | 超高并发（十万以上）      |
| CPU 利用率 | 低                    | 中                                 | 高                        |
| 复杂度     | 最低                  | 中                                 | 较高                      |
| 吞吐量     | 低                    | 高                                 | 最高                      |
| 底层实现   | Socket + ServerSocket | Channel + Selector + Buffer        | AsynchronousChannelGroup  |

**BIO 典型代码**：
```java
ServerSocket server = new ServerSocket(8080);
while(true) {
    Socket socket = server.accept(); // 阻塞
    new Thread(() -> handle(socket)).start();
}
```

**NIO 核心**：**Channel（双向） + Buffer（数据载体） + Selector（多路复用）**

---

### 3. NIO 三大核心组件工作原理

- **Channel（通道）**：双向，支持读写（FileChannel、SocketChannel、ServerSocketChannel）。
- **Buffer（缓冲区）**：数据读写必须经过 Buffer（ByteBuffer 最常用）。
  - 关键属性：`capacity`、`position`、`limit`、`mark`
  - 常用方法：`flip()`、`clear()`、`compact()`、`rewind()`
- **Selector（选择器）**：核心！一个线程可以监控**多个 Channel** 的就绪状态（Accept、Read、Write）。
  - `select()` / `selectNow()`：阻塞/非阻塞检测就绪事件
  - `SelectionKey`：代表 Channel + Selector 的注册关系，包含事件类型

**NIO 工作流程**：
1. ServerSocketChannel 注册到 Selector（监听 Accept 事件）
2. Selector 调用 `select()` 等待事件就绪
3. 遍历 SelectionKey，处理对应事件（accept → 注册 Read，read → 处理业务）

---

### 4. Netty 核心组件与 Reactor 模型

**Netty 是 NIO 的最佳实践封装**，极大简化了开发。

#### **核心组件**：
- **EventLoop**：一个线程 + Selector，负责处理 IO 事件（类似 Reactor）。
- **EventLoopGroup**：一组 EventLoop（Boss Group + Worker Group）。
- **Channel**：Netty 对 JDK Channel 的封装（NioSocketChannel 等）。
- **ChannelPipeline**：责任链模式，存放一系列 ChannelHandler（双向链表）。
- **ChannelHandler**：业务处理器（Inbound / Outbound）。
- **ByteBuf**：Netty 自定义的更强大的 Buffer（支持零拷贝、引用计数、池化）。

#### **Reactor 模型**（Netty 默认使用）：

| 模型               | Boss 线程数 | Worker 线程数 | 说明                       |
| ------------------ | ----------- | ------------- | -------------------------- |
| **单线程 Reactor** | 1           | 1（共用）     | 简单场景，性能低           |
| **多线程 Reactor** | 1           | N             | 推荐，Boss 负责 Accept     |
| **主从 Reactor**   | M（通常1）  | N             | **生产最常用**，高并发最佳 |

**Netty 默认是 主从多线程 Reactor 模型**。

---

### 5. 零拷贝（Zero Copy）

**核心目标**：减少 CPU 把数据从内核态拷贝到用户态的次数。

**常见实现方式**：

1. **mmap + write**（传统优化）
2. **sendfile**（Linux 2.1+）—— 经典零拷贝
3. **sendfile + DMA Gather Copy**（Linux 2.4+）—— 真正零拷贝
4. **Netty 中的零拷贝**：
   - **ByteBuf** 切片（`slice()`、`retainedSlice()`）
   - `CompositeByteBuf`（逻辑合并多个 Buffer）
   - `FileRegion`（直接发送文件，无需经过用户态）
   - `WrappedByteBuf`

**Netty 零拷贝 vs Linux 零拷贝**：
- Linux 零拷贝：内核态完成数据传输
- Netty 零拷贝：用户态减少对象拷贝和 GC

### 1. Cookie 与 Session 的区别、实现原理

| 对比维度 | **Cookie**                            | **Session**                            |
| -------- | ------------------------------------- | -------------------------------------- |
| 存储位置 | 客户端（浏览器）                      | 服务端（内存 / Redis / 数据库）        |
| 存储内容 | 少量文本数据（最大约 4KB）            | 任意对象（用户对象、购物车等）         |
| 安全性   | 较低（明文，可篡改）                  | 较高（服务端控制）                     |
| 实现原理 | 响应头 `Set-Cookie` + 请求头 `Cookie` | 服务端生成 SessionId，通过 Cookie 传递 |
| 生命周期 | 可设置过期时间                        | 通常有超时时间（默认 30 分钟）         |
| 跨域支持 | 较复杂                                | 依赖 Cookie，跨域较麻烦                |

**工作流程**：
1. 用户第一次请求 → 服务端创建 Session（生成 JSESSIONID）→ 通过 `Set-Cookie` 返回给浏览器
2. 后续请求 → 浏览器自动携带 Cookie（JSESSIONID）→ 服务端根据 ID 找到对应 Session

**分布式 Session 解决方案**（高频考点）：

- **Session 复制**：Tomcat 集群复制（适合小规模，不推荐）
- **Session 粘滞**（IP Hash）：负载均衡把同一 IP 固定到同一台机器（扩展性差）
- **Session 共享**（**推荐**）：把 Session 集中存到 **Redis**（Spring Session + Redis 最流行）
- **JWT Token**：无状态方案（后文详解）
- **数据库存储**：大公司有时使用

### 2. Token 机制（JWT）与 OAuth2.0

#### **JWT（JSON Web Token）**

**特点**：自包含、无状态、支持分布式

**结构**：`Header.Payload.Signature`（三段，用 `.` 分隔）

- **Header**：算法类型（HS256 / RS256）
- **Payload**：存放用户信息（iss、exp、sub、roles 等），**不要放敏感信息**
- **Signature**：签名防篡改

**优点**：
- 服务端无状态，易水平扩展
- 支持跨域和移动端
- 可设置过期时间

**缺点**：
- 无法主动失效（解决办法：黑名单 + 短过期 + Refresh Token）
- Payload 较大，占用带宽
- 不适合存放大量数据

**典型使用方式**：Authorization: Bearer xxx

#### **OAuth2.0 基本流程**（授权协议）

常见授权模式：

1. **Authorization Code（授权码模式）** —— 最安全，Web 应用常用
2. **Implicit** —— 已基本废弃
3. **Password** —— 信任客户端（如内部系统）
4. **Client Credentials** —— 服务间调用
5. **Refresh Token** —— 刷新访问令牌

**授权码模式核心流程**：
1. 用户点击“用微信/GitHub 登录”
2. 跳转到第三方授权页
3. 用户同意 → 第三方返回授权码（code）
4. 后端用 code + client_secret 换取 Access Token + Refresh Token
5. 用 Access Token 调用第三方资源接口

### 4. 长轮询 vs WebSocket vs SSE 对比

| 技术                            | 通信方向                 | 实现方式                                     | 延迟 | 资源消耗 | 适用场景                   | 推荐指数 |
| ------------------------------- | ------------------------ | -------------------------------------------- | ---- | -------- | -------------------------- | -------- |
| **长轮询**<br>(Long Polling)    | 单向（模拟双向）         | 客户端发请求，服务端 hold 住连接直到有新消息 | 中   | 高       | 简单实时通知               | ★★☆☆☆    |
| **SSE**<br>(Server-Sent Events) | **服务器 → 客户端** 单向 | HTTP 长连接 + text/event-stream              | 低   | 中       | 实时数据推送（通知、监控） | ★★★★☆    |
| **WebSocket**                   | **全双工**               | 独立协议（TCP）                              | 极低 | 低       | 聊天、在线协作、实时游戏   | ★★★★★    |

**详细对比**：

- **长轮询**：客户端发请求 → 服务端没数据就挂起 → 有数据或超时返回 → 客户端立即再发请求。消耗服务器连接资源。
- **SSE**：基于 HTTP 的单向推送，自动重连，简单易用（EventSource API）。
- **WebSocket**：最强大，支持双向实时通信，但实现相对复杂（需处理心跳、断线重连）。

**现代推荐**：
- 单向推送（通知、监控）→ **SSE**
- 双向实时交互（聊天、协作）→ **WebSocket**
- 极致性能要求 → WebSocket + Protobuf

### 1. DNS 解析过程（递归 + 迭代查询）

**DNS 解析核心流程**（以访问 `www.example.com` 为例）：

1. **浏览器缓存** → 本地 Hosts 文件 → 操作系统缓存
2. **本地 DNS 服务器**（通常是运营商 DNS 或 8.8.8.8）
3. **根域名服务器**（Root Server，全世界只有 13 组）
4. **顶级域名服务器**（TLD，如 `.com`）
5. **权威域名服务器**（Authoritative Name Server，域名实际注册的服务器）
6. 返回 IP 地址，浏览器建立 TCP 连接

**两种查询方式**：

- **递归查询**（Recursive Query）：
  - 客户端询问本地 DNS → 本地 DNS 负责完成后续所有查询 → 最终返回结果给客户端。
  - 特点：本地 DNS 很“辛苦”，客户端很“轻松”。

- **迭代查询**（Iterative Query）：
  - 本地 DNS 依次询问根 → TLD → 权威服务器，每一步只返回“下一步该问谁”。
  - 特点：本地 DNS 像“指路人”，逐步迭代查找。

**实际解析通常是**：客户端到本地 DNS 是**递归**，本地 DNS 到根/TLD/权威服务器是**迭代**。

### 2. ARP 协议工作原理

**ARP（Address Resolution Protocol）**：**IP 地址 → MAC 地址** 的解析协议。

**工作过程**（同一局域网内）：

1. 发送方主机检查 ARP 缓存（IP-MAC 映射表），有则直接使用。
2. 没有则广播 **ARP Request**（目标 IP 地址，谁是这个 IP？请告诉我你的 MAC）。
3. 目标主机收到后单播回复 **ARP Reply**（我的 MAC 地址是 XXX）。
4. 发送方将映射关系存入 ARP 缓存（默认保留几分钟）。

**特点**：
- ARP 是**广播**的（同一网段）。
- 有 ARP 欺骗/攻击风险（中间人攻击）。
- RARP（反向 ARP）已基本淘汰，被 BOOTP/DHCP 替代。

### 3. IP 地址分类、子网划分、CIDR

#### **IP 地址分类**（传统分类，已过时但基础仍考）

| 类   | 第一字节范围 | 网络位 | 主机位 | 最大主机数 | 用途     |
| ---- | ------------ | ------ | ------ | ---------- | -------- |
| A    | 1-126        | 8位    | 24位   | 16,777,214 | 大型网络 |
| B    | 128-191      | 16位   | 16位   | 65,534     | 中型网络 |
| C    | 192-223      | 24位   | 8位    | 254        | 小型网络 |
| D    | 224-239      | -      | -      | -          | 组播     |
| E    | 240-255      | -      | -      | -          | 保留     |

#### **子网划分 & CIDR**

**CIDR（Classless Inter-Domain Routing）**：无类域间路由，解决 IP 地址浪费问题。

表示方式：`192.168.1.0/24`（`/24` 表示前 24 位是网络位）

**子网掩码计算**：
- `/24` → 255.255.255.0
- 主机数 = 2^(32 - 前缀长度) - 2（减去网络地址和广播地址）

**示例**：把 `192.168.1.0/24` 划分成 4 个子网 → 每个子网 `/26`

### 4. 负载均衡算法（高频）

| 算法                    | 原理                                   | 优点                 | 缺点                  | 适用场景         |
| ----------------------- | -------------------------------------- | -------------------- | --------------------- | ---------------- |
| **轮询（Round Robin）** | 依次分发请求                           | 简单、公平           | 不考虑服务器性能      | 服务器配置相同   |
| **加权轮询**            | 按权重分配（性能好的权重高）           | 考虑服务器差异       | 配置权重需人工调整    | 服务器配置不同   |
| **最少连接**            | 把请求发给当前连接数最少的服务器       | 动态负载均衡         | 实现稍复杂            | 长连接场景       |
| **加权最少连接**        | 结合权重 + 当前连接数                  | 效果最好             | 实现复杂              | 生产环境推荐     |
| **一致性哈希**          | 根据请求特征（IP、用户ID）哈希到服务器 | 扩容缩容时命中率高   | 部分节点故障影响较大  | 分布式缓存、网关 |
| **IP Hash**             | 根据客户端 IP 哈希                     | 同一 IP 固定到同一台 | IP 分布不均时负载不均 | Session 粘滞     |
| **随机**                | 随机选择                               | 简单                 | 可能分配不均          | 辅助算法         |

### 5. CDN 工作原理

**CDN（Content Delivery Network）**：内容分发网络。

**核心思想**：将静态资源（图片、视频、JS、CSS）缓存到离用户最近的**边缘节点**（Edge Node）。

**访问流程**：

1. 用户访问 `static.example.com/image.jpg`
2. DNS 解析时被智能调度到最近的 CDN 节点（GSLB 全局负载均衡）
3. CDN 边缘节点检查是否有缓存：
   - 有 → 直接返回（命中）
   - 没有 → 回源到源站（Origin Server）拉取 → 缓存并返回

**主要作用**：
- 加速访问（减少延迟）
- 减轻源站压力
- 防御 DDoS 攻击（流量清洗）

### 6. 跨域问题（CORS）产生原因及解决方式

**产生原因**：
浏览器**同源策略**（Same-Origin Policy）限制：
- 同协议（http/https）、同域名、同端口 才算同源。
- 不同源的 XMLHttpRequest / Fetch 请求会被浏览器拦截。

**CORS（Cross-Origin Resource Sharing）解决方式**：

**服务端返回响应头**（核心）：

```http
Access-Control-Allow-Origin: https://example.com     # 或 *
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true               # 允许携带 Cookie
Access-Control-Max-Age: 7200                         # 预检结果缓存时间
```

**两种请求**：
- **简单请求**：GET、POST（特定 Content-Type），直接发送。
- **预检请求**（OPTIONS）：复杂请求会先发 OPTIONS 询问服务器是否允许。

**其他解决方式**：
- JSONP（只支持 GET，已过时）
- 反向代理（Nginx / Gateway 代理）
- iframe + postMessage（特殊场景）

### 1. TIME_WAIT 过多的问题及解决办法

**TIME_WAIT 状态**：主动关闭连接的一方（通常是客户端）在收到对方的 FIN 后，发送 ACK 后进入的状态，持续 **2MSL**（Maximum Segment Lifetime，通常 2×30秒 = 60秒 或 4分钟）。

**问题现象**：
- 大量 TIME_WAIT 占用端口（默认端口范围 28232 个）
- 导致新连接无法建立（`Cannot assign requested address` 或 `Address already in use`）
- 服务器资源占用高

**根本原因**：
- 高并发短连接场景（如 Nginx / API 网关 / 爬虫）
- 服务器主动关闭连接

**解决办法**（按优先级排序）：

1. **客户端优化**（最推荐）：
   - 使用连接池（HttpClient、数据库连接池、Redis 连接池等）
   - 复用连接，不要频繁建立/关闭

2. **TCP 参数调优**（服务器端）：
   ```bash
   # 允许重用 TIME_WAIT 端口
   net.ipv4.tcp_tw_reuse = 1          # 客户端
   net.ipv4.tcp_tw_recycle = 0        # 已废弃，慎用
   
   # 快速回收
   net.ipv4.tcp_fin_timeout = 15      # 缩短 TIME_WAIT 时间
   net.ipv4.tcp_max_tw_buckets = 5000 # 限制 TIME_WAIT 数量
   ```

3. **负载均衡策略**：后端使用 `keep-alive`，让客户端关闭连接（客户端进入 TIME_WAIT）

4. **使用长连接**：HTTP/2、WebSocket、RPC 框架

### 2. SYN Flood 攻击及防御

**SYN Flood** 是最经典的 DDoS 攻击之一。

**攻击原理**：
- 攻击者伪造大量 SYN 包发送给服务器
- 服务器为每个 SYN 分配资源（半连接队列），回复 SYN+ACK
- 攻击者不回复 ACK，导致服务器半连接队列被占满，无法处理正常请求

**防御措施**（多层防御）：

| 防御方式           | 原理                               | 效果   |
| ------------------ | ---------------------------------- | ------ |
| **SYN Cookies**    | 不立即分配资源，用 Cookie 记录信息 | 非常好 |
| **增加半连接队列** | `tcp_max_syn_backlog`              | 一般   |
| **SYN Proxy**      | 防火墙/负载均衡设备代理三次握手    | 很好   |
| **限流**           | iptables / Nginx limit_conn        | 有效   |
| **云防护**         | 阿里云/腾讯云 DDoS 清洗            | 最实用 |

**Linux 关键参数**：
- `net.ipv4.tcp_syncookies = 1`

### 3. TCP 粘包/半包问题及解决

**原因**：TCP 是**面向字节流**的协议，没有消息边界。

- **粘包**：多个小包被合并成一个大数据包发送
- **半包**：一个完整包被拆分成多个包发送

**解决办法**（应用层）：

1. **固定长度**（最简单）
2. **分隔符**（如 `\n`、`\r\n`，HTTP 就是用这个）
3. **协议头 + 长度**（**推荐**，工业级常用）：
   - 协议格式：`[包头(魔数)][长度字段(4字节)][业务数据][校验码]`
4. **Netty 方案**：
   - `LineBasedFrameDecoder`
   - `LengthFieldBasedFrameDecoder`（最常用）
   - `DelimiterBasedFrameDecoder`

**Netty 推荐使用 `LengthFieldBasedFrameDecoder`** 处理粘包/半包。

### 4. 流量控制与拥塞控制（重点）

#### **① 流量控制（Flow Control） - 滑动窗口**

- 由**接收方**控制发送方的发送速度
- 核心机制：**滑动窗口**（Sliding Window）
- 通过 TCP 头部 `Window` 字段通告自己还能接收多少字节
- 防止接收方缓冲区溢出

#### **② 拥塞控制（Congestion Control）**

由**发送方**主动控制发送速率，防止网络拥塞。

**四大算法**（必须掌握）：

| 算法         | 触发时机            | 核心行为                             | 窗口变化 |
| ------------ | ------------------- | ------------------------------------ | -------- |
| **慢启动**   | 连接建立初期        | 拥塞窗口（cwnd）指数增长（1→2→4→8…） | 快速增大 |
| **拥塞避免** | cwnd 达到慢启动阈值 | 线性增长（每个 RTT +1）              | 缓慢增长 |
| **快重传**   | 收到 3 个重复 ACK   | 立即重传丢失的包                     | -        |
| **快恢复**   | 快重传后            | cwnd = ssthresh/2 + 3，继续线性增长  | 快速恢复 |

**经典图示记忆**：慢启动 → 拥塞避免 → 遇到丢包 → 快重传 + 快恢复 → 继续拥塞避免。

### 1. 服务发现（Service Discovery）

**核心作用**：解决服务实例动态上下线问题，实现服务之间“解耦”。

#### **常见服务发现组件对比**

| 特性         | **Eureka**（Netflix） | **Nacos**（阿里）    | **Consul**      | **Zookeeper** |
| ------------ | --------------------- | -------------------- | --------------- | ------------- |
| CAP 理论     | AP（强可用）          | CP/AP 可切换         | CP              | CP            |
| 服务发现     | 原生支持              | 优秀（集成配置中心） | 优秀            | 一般          |
| 配置中心     | 不支持                | **原生支持**         | 支持            | 可实现        |
| 健康检查     | 客户端心跳            | 服务端 + 客户端      | 服务端          | 临时节点      |

3. 消费者通过**负载均衡**（Ribbon / Spring Cloud LoadBalancer）选择实例调用
4. 心跳续约 + 实例剔除机制

### 2. RPC 通信（Dubbo vs gRPC）

**RPC（Remote Procedure Call）**：远程过程调用，让远程调用像本地调用一样简单。

| 维度       | **Dubbo**（阿里）              | **gRPC**（Google）            |
| ---------- | ------------------------------ | ----------------------------- |
| 通信协议   | Dubbo / HTTP / REST            | **HTTP/2**                    |
| 序列化     | Hessian / Kryo / Protobuf      | **Protobuf**（默认）          |
| 性能       | 优秀                           | **极高**                      |
| 多语言支持 | Java 为主，多语言较弱          | **优秀**（官方支持 10+ 语言） |
| 服务治理   | 非常完善（路由、权重、限流等） | 基础，需要配合其他框架        |
| 适用场景   | Java 生态微服务                | **跨语言**、高性能内部通信    |
| 学习难度   | 中                             | 稍高（Protobuf）              |

**gRPC 核心优势**：
- 基于 HTTP/2（多路复用、头部压缩、流式传输）
- Protobuf 序列化（体积小、速度快）
- 支持四种调用模式：一元、服务器流、客户端流、双向流

**Dubbo 优势**：生态成熟，服务治理能力强（路由、权重、标签路由等）。

### 3. 服务间调用超时、重试、熔断机制

这是**微服务稳定性**的核心保障机制。

#### **服务调用链路保护三大核心**：

1. **超时（Timeout）**
   - 必须设置**超时时间**，防止线程被长时间占用
   - 建议：业务超时 < 调用链路最上层超时
   - 常见配置：`ribbon.ReadTimeout`、`feign.client.config`、`dubbo.timeout`

2. **重试（Retry）**
   - **幂等接口**才能重试（GET、PUT、DELETE）
   - POST 类接口重试需特别小心（重复下单风险）
   - 常用策略：指数退避 + 抖动（避免惊群）

3. **熔断（Circuit Breaker） + 降级**
   - **熔断器三种状态**：Closed（正常）、Open（熔断）、Half-Open（半开尝试）
   - **常用框架**：
     - **Resilience4j**（Spring Cloud 推荐）
     - **Sentinel**（阿里，国内主流）
     - **Hystrix**（已进入维护模式）

**最佳实践**：
- 超时 → 重试 → 熔断 → 降级（返回默认值或缓存数据）
- 线程隔离（信号量或线程池隔离）
- 限流（QPS 控制）

### 4. 分布式链路追踪（Distributed Tracing）

**核心目的**：在微服务架构中跟踪一次请求经过的所有服务，快速定位慢请求、故障点。

#### **主流方案对比**

| 方案           | 语言   | 存储           | 侵入性                 | 国内使用   | 特点             |
| -------------- | ------ | -------------- | ---------------------- | ---------- | ---------------- |
| **Zipkin**     | 多语言 | ES / MySQL     | 低                     | 较多       | 经典，轻量       |
| **SkyWalking** | Java强 | ES / H2        | **极低**（字节码增强） | **最流行** | APM 全链路监控强 |

```java
public class BioServer {
    public static void main(String[] args) throws Exception {
        ServerSocket serverSocket = new ServerSocket(8080);
        System.out.println("服务器启动，监听端口 8080...");

        while (true) {
            Socket socket = serverSocket.accept(); // 阻塞等待客户端连接

    private static void handleClient(Socket socket) {
        try (BufferedReader reader = new BufferedReader(
                new InputStreamReader(socket.getInputStream()));
             PrintWriter writer = new PrintWriter(socket.getOutputStream(), true)) {

            while ((line = reader.readLine()) != null) {
                System.out.println("收到客户端消息: " + line);
                writer.println("服务器回复: " + line.toUpperCase());
            }
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```

### 2. NIO 实现简单聊天室（核心代码）

**NIO 聊天室服务端核心实现**（使用 Selector）：

```java
public class NioChatServer {
    private Selector selector;
    private ServerSocketChannel serverChannel;
    private static final int PORT = 9000;

    public void start() throws Exception {
        selector = Selector.open();
        serverChannel = ServerSocketChannel.open();
        serverChannel.configureBlocking(false);
        serverChannel.bind(new InetSocketAddress(PORT));
        serverChannel.register(selector, SelectionKey.OP_ACCEPT);

        System.out.println("NIO 聊天室启动，端口: " + PORT);

        while (true) {
            selector.select(); // 阻塞直到有事件就绪
            Iterator<SelectionKey> iterator = selector.selectedKeys().iterator();

            while (iterator.hasNext()) {
                SelectionKey key = iterator.next();
                iterator.remove();

                if (key.isAcceptable()) {
                    handleAccept(key);
                } else if (key.isReadable()) {
                    handleRead(key);
                }
            }
        }
    }

    private void handleAccept(SelectionKey key) throws Exception {
        ServerSocketChannel ssc = (ServerSocketChannel) key.channel();
        SocketChannel clientChannel = ssc.accept();
        clientChannel.configureBlocking(false);
        clientChannel.register(selector, SelectionKey.OP_READ);
        System.out.println("客户端连接: " + clientChannel.getRemoteAddress());
    }

    private void handleRead(SelectionKey key) throws Exception {
        SocketChannel channel = (SocketChannel) key.channel();
        ByteBuffer buffer = ByteBuffer.allocate(1024);
        int len = channel.read(buffer);

        if (len > 0) {
            buffer.flip();
            String message = Charset.defaultCharset().decode(buffer).toString();
            System.out.println("收到消息: " + message);
            broadcast(message, channel); // 广播给其他客户端
        } else {
            System.out.println("客户端断开: " + channel.getRemoteAddress());
            key.cancel();
            channel.close();
        }
    }

    private void broadcast(String message, SocketChannel self) throws Exception {
        for (SelectionKey key : selector.keys()) {
            Channel target = key.channel();
            if (target instanceof SocketChannel && target != self) {
                SocketChannel sc = (SocketChannel) target;
                sc.write(ByteBuffer.wrap(message.getBytes()));
            }
        }
    }
}
```

**NIO 优点**：一个线程处理多个连接。

### 1. HTTP 状态码 304 是什么意思？什么情况下会出现？

**回答模板：**
304 Not Modified 表示**协商缓存命中**。  
当浏览器携带 `If-None-Match`（ETag）或 `If-Modified-Since` 请求头向服务器发起请求时，如果服务器发现资源没有发生变化，就会返回 304 状态码，告诉浏览器“资源未修改，你可以直接使用本地缓存”。  
此时服务器**不会返回响应体**，从而节省带宽和加快页面加载速度。

### 2. 302 和 307 的区别？301 和 302 呢？

**回答模板：**

- **301 Moved Permanently**：永久重定向。表示资源**永久**转移到新地址，搜索引擎会更新索引，**下次不会再请求老地址**。
- **302 Found**：临时重定向。最常用。表示资源**临时**转移，下次请求还是会访问老地址。
- **307 Temporary Redirect**：临时重定向（HTTP/1.1）。和 302 功能类似，但**严格保留原请求方法**（302 在某些情况下可能会把 POST 变成 GET，而 307 不会）。

**总结**：301 是永久的，302 和 307 是临时的，307 比 302 更严格遵守 HTTP 规范。

### 3. GET 和 POST 的主要区别？哪个更安全？

**回答模板：**

| 维度     | GET                         | POST          |
| -------- | --------------------------- | ------------- |
| 语义     | 获取资源                    | 提交/修改资源 |
| 是否幂等 | 是                          | 否            |
| 是否安全 | 是（不修改服务器状态）      | 否            |
| 参数位置 | 在 URL 中                   | 在请求体中    |
| 数据长度 | 受 URL 长度限制（约 2K-8K） | 几乎无限制    |
| 缓存     | 可被缓存                    | 一般不可缓存  |

**安全性**：  
从传输层面来说，**POST 相对更安全**，因为参数不在 URL 中，不会在浏览器历史记录、服务器日志中明文显示。但从本质上讲，**两者都不安全**，因为 HTTP 是明文传输，真正安全要使用 **HTTPS**。

### 4. HTTP 和 HTTPS 的区别是什么？

**回答模板：**

- HTTP 是超文本传输协议，明文传输，不安全。
- HTTPS = HTTP + SSL/TLS，在 HTTP 基础上增加了加密、身份认证和数据完整性保护。

**主要区别**：
1. **安全性**：HTTPS 安全，HTTP 不安全。
2. **端口**：HTTP 默认 80，HTTPS 默认 443。
3. **性能**：HTTPS 需要 TLS 握手，首次连接稍慢（可通过 Session Resumption 优化）。
4. **证书**：HTTPS 需要 CA 颁发的数字证书。
5. **SEO**：搜索引擎更偏好 HTTPS 网站。

### 6. TCP 三次握手过程？为什么是三次而不是两次？

**回答模板：**

**三次握手过程**：
1. 客户端发送 SYN=1，seq=x（第一次）
2. 服务端回复 SYN=1，ACK=1，seq=y，ack=x+1（第二次）
3. 客户端回复 ACK=1，seq=x+1，ack=y+1（第三次）

**为什么是三次？**  
主要是为了**双方都能确认对方的收发能力正常**，同时防止历史连接（旧的 SYN 包）干扰新连接。如果只有两次握手，服务端无法确认客户端是否收到了自己的 SYN 包，容易出现连接错误。

### 7. TCP 四次挥手过程？为什么需要 TIME_WAIT？

**回答模板：**

**四次挥手过程**：
1. 客户端发送 FIN（第一次）
2. 服务端回复 ACK（第二次）
3. 服务端发送 FIN（第三次）
4. 客户端回复 ACK（第四次）

**TIME_WAIT 作用**：
- 确保最后一个 ACK 能被对方收到（防止对方重传 FIN）
- 让本次连接的所有报文在网络中消失，避免与下次新连接混淆

客户端会等待 **2MSL**（通常 1-4 分钟）后才真正关闭。

### 8. Cookie 和 Session 的区别？

**回答模板：**

- **Cookie**：保存在**客户端**的小型文本数据，最大约 4KB。
- **Session**：保存在**服务端**的会话信息。

**工作机制**：服务端生成 Session 后，把 SessionId 通过 Cookie（通常叫 JSESSIONID）返回给客户端，后续请求客户端自动携带该 Cookie，服务端据此找到对应 Session。

**分布式环境下**：Session 需要做共享（常用 Redis），而 JWT Token 是无状态方案。

### 1. HTTPS 握手过程详细说一下？（极高频）

**回答模板：**

HTTPS 握手本质是 **TLS 握手**，以 TLS 1.2 为例，主要分为以下 4 步：

1. **Client Hello**：客户端发送支持的 TLS 版本、加密套件、随机数等信息。
2. **Server Hello**：服务端选定 TLS 版本、加密套件，并返回自己的随机数 + **服务器证书**。
3. **密钥交换**：
   - 客户端验证服务器证书（使用 CA 公钥验证）
   - 客户端生成预主密钥（Premaster Secret），用服务器公钥加密后发送
   - 双方根据随机数 + 预主密钥生成会话对称密钥
4. **Finished**：双方交换 Finished 消息，验证握手过程是否被篡改，握手完成。

TLS 1.3 做了优化，握手可减少到 **1-RTT**（1 个往返），性能更好。

### 2. HTTPS 中为什么要使用 CA 证书？如果没有 CA 证书会怎样？

**回答模板：**

使用 CA 证书是为了**解决中间人攻击**问题。

- **非对称加密**虽然能加密数据，但无法验证对方身份。
- CA（Certificate Authority）作为权威第三方机构，对服务器证书进行数字签名。
- 客户端通过操作系统或浏览器内置的 CA 根证书验证服务器证书的合法性。

**如果没有 CA 证书（自签名证书）**：
- 浏览器会提示“不安全连接”，用户可以选择继续访问。
- 存在严重中间人攻击风险：攻击者可以伪造证书，劫持通信，窃取敏感信息（账号、密码、支付信息等）。
- 生产环境**绝对不允许**使用自签名证书。

### 4. HTTP 长连接和短连接的区别？什么是队头阻塞？

**回答模板：**

- **短连接**：每次请求都建立一次 TCP 连接，请求完立即关闭（HTTP/1.0 默认）。
- **长连接**（Keep-Alive）：建立一次 TCP 连接后，可传输多个 HTTP 请求（HTTP/1.1 默认）。

**队头阻塞（Head of Line Blocking）**：
在 HTTP/1.1 中，即使后面请求已经处理完，也必须等待前面的请求返回，导致后面的请求被阻塞。  
HTTP/2 通过**多路复用**（Stream）解决了这个问题。

### 5. 解释一下 TCP 的滑动窗口机制。

**回答模板：**

滑动窗口是 TCP **流量控制**的核心机制。

- 发送方和接收方各自维护一个滑动窗口。
- 接收方通过 TCP 报文头的 **Window 字段** 告诉发送方自己还能接收多少字节的数据。
- 发送方根据这个窗口值控制发送速度，避免接收方缓冲区溢出。
- 窗口会随着数据发送和确认不断“滑动”。

滑动窗口实现了**动态流量控制**，极大提高了 TCP 的传输效率。

---

### 6. TCP 的拥塞控制有哪些算法？简单说说慢启动和快重传。

**回答模板：**

TCP 拥塞控制主要有四大算法：

1. **慢启动**：连接初期，拥塞窗口（cwnd）从 1 开始指数增长（1→2→4→8…），快速试探网络容量。
2. **拥塞避免**：当 cwnd 达到慢启动阈值后，改为线性增长（每个 RTT +1），避免拥塞。
3. **快重传**：当发送方收到 **3 个重复 ACK** 时，立即重传丢失的报文，而不用等待超时。
4. **快恢复**：快重传后，cwnd 设为阈值的一半，继续线性增长，而不是从 1 开始。

### 7. BIO、NIO、AIO 的区别和适用场景？

**回答模板：**

| 模型 | 同步/异步 | 阻塞/非阻塞 | 适用场景             | 并发能力 |
| ---- | --------- | ----------- | -------------------- | -------- |
| BIO  | 同步      | 阻塞        | 连接数少、业务简单   | 低       |
| NIO  | 同步      | 非阻塞      | 高并发（几千~几万）  | 高       |
| AIO  | 异步      | 非阻塞      | 超高并发（十万以上） | 最高     |

- **BIO**：一连接一线程，简单但扩展性差。
- **NIO**：基于 Selector（IO 多路复用），一个线程管理多个连接（Netty 底层）。
- **AIO**：真正异步，适合长连接、大文件传输等。

### 8. NIO 中 Selector 的作用是什么？工作原理？

**回答模板：**

**Selector（选择器）** 是 NIO 实现 IO 多路复用的核心组件。

**作用**：一个线程可以监控**多个 Channel** 的就绪状态（Accept、Read、Write 等）。

**工作原理**：
1. 将多个 Channel 注册到 Selector 上，并指定监听的事件。
2. 调用 `selector.select()` 方法，线程阻塞，直到有 Channel 就绪。
3. 通过 `selectedKeys()` 拿到就绪的 SelectionKey，处理对应事件。

这样就实现了**一个线程处理多个连接**，大大提升了并发能力。

### 9. JWT 的原理和优缺点？

**回答模板：**

**JWT（JSON Web Token）** 由三部分组成：`Header.Payload.Signature`

- **Header**：算法类型
- **Payload**：存放用户信息和过期时间（不要放敏感信息）
- **Signature**：签名，防止篡改

**优点**：
- 无状态，服务端不需要存储 Session
- 易水平扩展，支持分布式
- 跨语言、移动端友好

**缺点**：
- Token 无法主动失效（需配合黑名单或短过期 + Refresh Token）
- Payload 较大，占用带宽
- 一旦泄露，任何人可用（必须用 HTTPS）

### 10. WebSocket 握手过程是怎样的？和 HTTP 什么关系？

**回答模板：**

WebSocket 握手**基于 HTTP 协议**完成协议升级。

过程：
1. 客户端发送 HTTP 请求，带上 `Upgrade: websocket`、`Connection: Upgrade`、`Sec-WebSocket-Key` 等头。
2. 服务端返回 `HTTP/1.1 101 Switching Protocols`，完成协议升级。
3. 之后双方使用 WebSocket 协议（ws:// 或 wss://）进行**全双工**通信。

### 11. 负载均衡有哪些常见算法？一致性哈希的优缺点？

**回答模板：**

常见负载均衡算法：

- **轮询（Round Robin）**：最简单
- **加权轮询**：按服务器性能分配
- **最少连接**：动态分配给连接最少的服务器
- **IP Hash**：同一 IP 固定到一台服务器
- **一致性哈希**：根据请求特征哈希到服务器环上

**一致性哈希优缺点**：
- **优点**：扩容/缩容时，只有少量请求被重新路由，命中率高。
- **缺点**：部分节点故障时，可能影响较多请求（需引入虚拟节点优化）。

### 1. Netty 的线程模型详细说一下？（Boss + Worker）

**回答模板：**

Netty 默认采用 **主从 Reactor 多线程模型**（Master-Slave Reactor）：

- **Boss Group**（通常 1 个线程）：只负责处理 `OP_ACCEPT` 事件，即客户端的连接请求。接受连接后将 SocketChannel 注册到 Worker Group。
- **Worker Group**（默认 CPU 核数 × 2 个线程）：负责处理 `OP_READ`、`OP_WRITE` 等 IO 事件，以及业务 Handler 的执行。

**优点**：
- 职责分离，Boss 线程负载极低
- Worker 线程可水平扩展
- EventLoop（每个线程绑定一个 Selector）实现线程安全的事件驱动

实际项目中，Boss Group 通常设置为 1 个线程，Worker Group 根据机器配置和业务特点调整。

---

### 2. Reactor 模型和 Proactor 模型的区别？

**回答模板：**

| 维度       | **Reactor**（Netty 使用）      | **Proactor**                   |
| ---------- | ------------------------------ | ------------------------------ |
| 编程模型   | 同步非阻塞                     | 异步非阻塞                     |
| 事件类型   | IO 就绪事件（可读、可写）      | IO 完成事件                    |
| 内核支持   | select / epoll / kqueue        | aio（Linux 原生 AIO 支持较差） |
| 实现复杂度 | 中等                           | 较高                           |
| 典型框架   | Netty、Redis（单线程 Reactor） | Windows IOCP                   |

**核心区别**：Reactor 是“**我告诉你可以读写了**”，Proactor 是“**我帮你读写完了再通知你**”。

目前主流高性能网络框架大多采用 **Reactor 模型**，因为 Proactor 在 Linux 下成熟度较低。

---

### 3. Netty 如何解决 TCP 粘包和半包问题？

**回答模板：**

Netty 提供了多种 `ByteToMessageDecoder` 实现来解决粘包/半包：

1. **LengthFieldBasedFrameDecoder**（**最推荐**）：在消息头中预留长度字段，根据长度自动拆包。
2. **LineBasedFrameDecoder**：以换行符 `\n` 或 `\r\n` 作为分隔符。
3. **DelimiterBasedFrameDecoder**：自定义分隔符。
4. **FixedLengthFrameDecoder**：固定长度消息。

实际项目中，通常在 Pipeline 最前面加上 `LengthFieldBasedFrameDecoder + LengthFieldPrepender`，从协议层面彻底解决粘包问题。

---

### 4. Netty 的零拷贝是怎么实现的？

**回答模板：**

Netty 的零拷贝分为**两类**：

1. **用户态零拷贝**（Netty 层面）：
   - `ByteBuf` 的 `slice()`、`retainedSlice()`：避免数据拷贝
   - `CompositeByteBuf`：逻辑合并多个 Buffer
   - `WrappedByteBuf`

2. **内核态零拷贝**（底层）：
   - 使用 `FileRegion` 实现 `transferTo()`，底层调用 `sendfile()` 系统调用
   - 避免了数据从内核态到用户态再回到内核态的多次拷贝

Netty 零拷贝主要优化的是**用户态**的数据拷贝和对象创建，极大降低了 GC 压力。

---

### 5. 什么是 IO 多路复用？epoll 和 select 的区别？

**回答模板：**

**IO 多路复用**：一个线程可以同时监听多个文件描述符（Socket），当其中某个就绪时，内核通知线程进行处理。

**select vs epoll**：

| 特性       | select                 | epoll                       |
| ---------- | ---------------------- | --------------------------- |
| 最大连接数 | 1024（可修改但有限）   | 几乎无限制                  |
| 底层实现   | 数组                   | 红黑树 + 就绪链表           |
| 事件复杂度 | O(n)（每次都要遍历）   | O(1)（只返回就绪的事件）    |
| 内存拷贝   | 每次都需要拷贝 fd 集合 | 只拷贝一次（mmap 共享内存） |
| 适用场景   | 小并发                 | **高并发**（Linux 主流）    |

Netty 在 Linux 环境下默认使用 epoll。

### 6. 如何设计一个支持百万长连接的系统？

**回答模板：**

核心思路是**降低单机资源消耗** + **水平扩展**：

1. **网络层**：使用 Netty + 主从 Reactor 模型 + epoll
2. **内存优化**：每个连接对象尽量轻量（减少不必要的属性）
3. **心跳检测**：IdleStateHandler + 30~60 秒读空闲关闭
4. **线程模型**：Worker 线程数控制在 CPU 核数 1~2 倍
5. **业务线程池隔离**：IO 线程与业务线程分离
6. **水平扩展**：多机部署 + 一致性哈希 + 服务发现
7. **连接管理**：使用 Map + 双向索引，快速踢人、广播
8. **监控**：连接数、内存、GC、Watermark 等关键指标

实际项目中，单机稳定支持 20~50 万长连接是比较合理的。

### 7. 大量 TIME_WAIT 产生的原因和优化方案？

**回答模板：**

**主要原因**：高并发**短连接**场景下，客户端或服务器主动关闭连接。

**优化方案**（优先级从高到低）：

1. **根本优化**：使用连接池（HttpClient、Dubbo、Redis 等），尽量复用连接
2. **让客户端关闭连接**（推荐）
3. **服务器参数调优**：
   - `net.ipv4.tcp_tw_reuse = 1`
   - `net.ipv4.tcp_fin_timeout = 15`
   - `net.ipv4.tcp_max_tw_buckets` 适当调大
4. **使用长连接**（HTTP/2、WebSocket、RPC）

### 8. SYN Flood 攻击原理及防御手段？

**回答模板：**

**原理**：攻击者伪造大量 SYN 包，占满服务器的半连接队列（SYN Queue），导致正常连接无法建立。

**防御手段**：
- 开启 `tcp_syncookies = 1`
- 增大 `tcp_max_syn_backlog`
- 使用 SYN Proxy（负载均衡设备或云厂商实现）
- 限流 + IP 黑名单
- 购买云 DDoS 防护

### 9. 服务发现的实现原理？Nacos 和 Eureka 有什么区别？

**回答模板：**

**核心原理**：服务提供者注册 → 注册中心存储实例信息 → 服务消费者拉取/订阅实例列表 → 客户端负载均衡。

**Nacos vs Eureka**：
- Eureka：纯 AP 设计，强调可用性，自我保护机制强
- Nacos：支持 CP/AP 切换，同时具备**服务发现 + 配置中心**，健康检查更灵活
- Nacos 目前在国内使用更为广泛

### 10. 分布式链路追踪（SkyWalking）的实现原理？TraceId 如何传递？

**回答模板：**

SkyWalking 采用**字节码增强（Java Agent）**的无侵入方案。

**核心原理**：
- Agent 拦截入口方法，生成 TraceId 和 SpanId
- 通过插件采集各种中间件调用信息
- 数据异步上报到 Collector

**TraceId 传递**：通过协议头（HTTP Header、Dubbo Attachment、gRPC Metadata 等）在服务间传递，保证一次请求链路唯一标识。

---

### 11. Dubbo 和 gRPC 的区别？分别在什么场景使用？

**回答模板：**

- **Dubbo**：服务治理能力极强（路由、权重、标签路由、限流等），适合**Java 生态**大型微服务系统。
- **gRPC**：基于 HTTP/2 + Protobuf，性能更高，多语言支持优秀，适合**跨语言**、高性能内部通信场景（如 Go、Python、Java 混用）。

**选择建议**：纯 Java 系统选 Dubbo；多语言或追求极致性能选 gRPC。

---

### 12. 如何设计微服务间的调用超时、重试、熔断机制？

**回答模板：**

采用**分层防护**策略：

1. **超时**：每个调用必须设置超时时间（建议 200ms~2s），避免线程堆积
2. **重试**：仅对**幂等接口**进行有限重试（指数退避 + 抖动）
3. **熔断降级**：使用 Sentinel / Resilience4j，实现熔断器模式（Closed → Open → Half-Open）
4. **线程隔离**：使用信号量或线程池隔离
5. **监控告警**：结合链路追踪，及时发现慢调用和异常

### 1. 百万长连接如何设计？（高频架构题）

**优化方案**（优先级从高到低）：

### 3. 如何优化深分页接口的网络和数据库性能？

**回答模板：**

**核心思路**：避免 `LIMIT offset, size` 的大偏移量扫描。

**优化方案**：
- **推荐**：**游标分页**（Cursor Pagination），使用 `WHERE id < lastId ORDER BY id DESC LIMIT size`
- **次选**：延迟关联（先查主键，再关联查完整记录）
- **覆盖索引**：建立 `status + create_time + id` 复合索引
- **禁止超大页码**：业务上限制最大页码（如不超过 500 页）
- **其他**：静态数据使用 Elasticsearch；热点数据 Redis 缓存

### 4. 服务之间调用频繁超时，应该怎么排查和解决？

2. **查看日志**：慢 SQL、外部调用、Redis、锁等待等
3. **监控指标**：CPU、内存、线程池队列、数据库连接数、GC 频率
4. **网络层面**：检查是否出现大量 TIME_WAIT、重传、丢包

**解决措施**：
- 设置合理的超时 + 重试 + 熔断（Sentinel）
- 慢 SQL 优化 + 增加缓存
- 线程池/数据库连接池参数调优
- 异步化 + 降级方案
- 限流 + 扩容

### 7. 分布式 Session 有哪些解决方案？各有什么优缺点？

### 8. 如何防止接口被重复请求（幂等性设计）？

**回答模板：**

**常用方案**：
1. **唯一业务 ID**（推荐）：前端生成全局唯一 ID（TraceId + UUID），后端用 Redis `SETNX` 实现防重
2. **数据库唯一索引**：如订单号唯一索引
3. **Token 令牌机制**：提交前获取令牌，提交时校验并删除（适用于页面表单）
4. **版本号 / 乐观锁**：更新操作使用 version 字段

核心原则：**核心业务操作必须做幂等**。

### 9. 高并发下如何进行服务降级和熔断？

**回答模板：**

- **熔断**：基于错误率、慢请求比例，超过阈值进入熔断状态（Open）
- **降级**：熔断后返回默认值、缓存数据或简化逻辑
- **限流**：QPS 限流、线程数限流、排队等待
- **多级降级**：接口层 → 服务层 → 数据层

同时结合**灰度发布**和**开关配置中心**实现动态控制。

### 10. 如何对一个慢接口进行全链路性能优化？

**回答模板：**

采用**分层优化**方法：

1. **链路追踪**：SkyWalking 定位最耗时环节
2. **数据库层**：EXPLAIN 分析 + 加索引 + SQL 重写 + 读写分离 + 缓存
3. **应用层**：异步化、并行调用、线程池优化、减少锁粒度
4. **网络层**：压缩、协议优化、连接池、减少调用次数
5. **缓存层**：本地缓存 + 分布式缓存 + 多级缓存

### 1. “你说完了三次握手，那如果第三次握手报文丢失了会怎样？”

**优秀回答模板：**

如果第三次握手（客户端发送的 ACK）丢失，会出现以下情况：

- 服务端会认为自己的 SYN+ACK 可能丢失，于是**重传 SYN+ACK**（默认重传 5 次，间隔指数退避）。
- 客户端因为已经进入 `ESTABLISHED` 状态，会直接回复 ACK。
- 如果服务端在重传次数用完后仍未收到 ACK，则会放弃连接，进入 `CLOSED` 状态。
- 客户端这边如果后续没有数据发送，可能长时间处于 `ESTABLISHED` 状态（半打开连接），直到应用层超时或发送数据时发现 RST。

**总结**：TCP 的可靠性机制（重传 + 序列号）保证了最终双方状态能达成一致，只是会多消耗一些网络资源。这也是为什么需要三次握手来保障双方状态同步。

---

### 2. “你提到 JWT，那 JWT 的 Token 泄露了怎么办？”

**优秀回答模板：**

JWT Token 泄露是它的主要风险之一，我会从几个层面进行防护：

**预防措施**：
- 必须使用 **HTTPS** 传输，杜绝中间人窃取。
- 设置较短的过期时间（例如 Access Token 15~60 分钟）。
- 使用 **Refresh Token** 机制（Refresh Token 存 Redis，可主动失效）。
- 重要的操作（如支付、修改密码）二次验证（短信/密码/二次 Token）。

**泄露后的应对**：
- 将 Token 加入**黑名单**（Redis Set，设置过期时间）。
- 每次请求校验 Token 是否在黑名单中。
- 采用 `kid`（Key ID）机制，支持密钥轮换。
- 敏感系统可结合 IP + User-Agent + 设备指纹进行风控。

**总结**：JWT 是**无状态**的，所以我们通过**短过期 + Refresh Token + 黑名单**的方式，在可用性和安全性之间取得平衡。

### 3. “Netty 解决了粘包，那你具体用了哪个 Handler？”

**优秀回答模板：**

在实际项目中，我主要使用 **`LengthFieldBasedFrameDecoder`** 来解决粘包/半包问题。

**具体配置示例**：
```java
pipeline.addLast(new LengthFieldBasedFrameDecoder(
    1024*1024,     // 最大帧长度
    0,             // 长度字段偏移量
    4,             // 长度字段占4字节（int）
    0,             // 长度字段后跳过的字节数
    4));           // 解码后跳过的字节数（去掉长度字段）
```

同时配合 `LengthFieldPrepender` 在编码时自动添加长度头。

**为什么选它？**
- 支持自定义协议，灵活性高
- 性能好，Netty 官方推荐
- 能完美处理半包和粘包

如果消息是文本协议，也会配合 `LineBasedFrameDecoder` 使用。

### 4. “你用了 Redis 做 Session 共享，那 Redis 挂了怎么办？”

**优秀回答模板：**

这是一个很好的问题，我在设计时会考虑**多重保障**：

**1. 架构层面**：
- Redis 使用**主从 + Sentinel** 或 **Redis Cluster** 模式，实现高可用。
- 应用层使用 **Redisson** 或 **Spring Session**，配置连接池 + 超时 + 重试。

**2. 降级策略**：
- Redis 不可用时，**降级到本地 Session**（单机）或直接使用 **JWT Token**（无状态）。
- 采用**多级缓存**：本地 Caffeine 缓存 + Redis 分布式缓存。
- 实现**熔断机制**（Sentinel），Redis 超时或异常达到阈值后自动熔断，减少雪崩风险。

**3. 数据一致性**：
- Session 写入采用异步 + 本地缓存双写策略。
- Redis 恢复后，通过**异步同步**机制把本地 Session 回写。

**总结**：不会把 Redis 当成单点依赖，而是通过**高可用架构 + 降级方案 + 熔断**来保证系统整体稳定性。
