### **性能优化**

#### 一、Redis 缓存高频数据的应用场景（用户端 + 管理端）

Redis 缓存高频访问数据**并非只用于用户端**，而是在**用户端（小程序）和管理端（Web后台）均有核心应用**，具体场景如下：

##### 1. 用户端（核心场景）

- 菜品/套餐列表（用户点餐时高频查询，如热门菜品、分类菜品）
- 店铺基础信息（营业时间、配送范围、起送价等）
- 用户地址缓存（减少频繁查询数据库）
- 优惠券/活动信息（营销活动数据，访问量高）

##### 2. 管理端（辅助场景）

- 订单统计数据（如今日订单数、销售额，高频刷新）
- 菜品分类/标签（后台操作时频繁查询）
- 员工权限信息（登录后缓存，减少权限校验查库）

#### 二、SpringCache + Redis 的缓存问题处理方案

在订餐系统中使用 SpringCache 集成 Redis 时，**必须处理缓存击穿、穿透、雪崩问题**，且高频访问数据（如菜品列表）建议做缓存预热，以下是具体实现方案：

##### 1. 缓存预热（建议实施）

**适用场景**：系统启动时加载高频访问的基础数据（如菜品列表、店铺信息）到 Redis，避免首次访问缓存未命中导致的数据库压力。
**实现代码**：

```java
@Component
@Slf4j
public class CacheWarmUpRunner implements CommandLineRunner {

    @Autowired
    private DishService dishService;
    @Autowired
    private StringRedisTemplate redisTemplate;
    @Autowired
    private ObjectMapper objectMapper;

    // 缓存前缀
    private static final String DISH_CACHE_KEY = "dish:list:";

    @Override
    public void run(String... args) throws Exception {
        log.info("开始缓存预热：加载菜品数据");
        // 1. 查询所有店铺的基础菜品列表（按店铺ID分组）
        List<Long> shopIds = dishService.listShopIds();
        for (Long shopId : shopIds) {
            List<DishVO> dishList = dishService.listByShopId(shopId);
            // 2. 序列化后存入Redis（设置过期时间，避免雪崩）
            String key = DISH_CACHE_KEY + shopId;
            redisTemplate.opsForValue().set(
                key,
                objectMapper.writeValueAsString(dishList),
                12, // 过期时间12小时
                TimeUnit.HOURS
            );
        }
        log.info("缓存预热完成");
    }
}
```

##### 2. 缓存穿透处理（必做）

**问题**：查询不存在的数据（如ID=-1的菜品），每次都穿透到数据库，导致DB压力。
**解决方案**：缓存空值 + 请求参数校验

```java
@Service
public class DishServiceImpl extends ServiceImpl<DishMapper, Dish> implements DishService {

    @Cacheable(
        value = "dish", 
        key = "#id",
        unless = "#result == null" // 结果为空时不缓存（结合下方逻辑）
    )
    public Dish getById(Long id) {
        // 1. 参数校验：拦截非法ID
        if (id == null || id <= 0) {
            return null;
        }
        Dish dish = baseMapper.selectById(id);
        // 2. 缓存空值（有效期短，如5分钟）
        if (dish == null) {
            redisTemplate.opsForValue().set(
                "dish:" + id,
                "", // 空值占位
                5,
                TimeUnit.MINUTES
            );
        }
        return dish;
    }
}
```

##### 3. 缓存击穿处理（必做）

**问题**：热点Key（如爆款菜品ID）过期瞬间，大量请求同时穿透到数据库。
**解决方案**：互斥锁 + 热点Key永不过期（主动更新）

```java
@Service
public class DishServiceImpl extends ServiceImpl<DishMapper, Dish> implements DishService {

    private static final String LOCK_KEY = "lock:dish:";

    public Dish getHotDishById(Long id) {
        String cacheKey = "dish:hot:" + id;
        // 1. 先查缓存
        String dishJson = redisTemplate.opsForValue().get(cacheKey);
        if (StringUtils.isNotBlank(dishJson)) {
            return JSON.parseObject(dishJson, Dish.class);
        }

        // 2. 缓存未命中，加锁查询数据库
        String lockKey = LOCK_KEY + id;
        boolean locked = false;
        try {
            // 分布式锁：3秒过期，防止死锁
            locked = redisTemplate.opsForValue().setIfAbsent(lockKey, "1", 3, TimeUnit.SECONDS);
            if (locked) {
                // 3. 锁成功，查数据库
                Dish dish = baseMapper.selectById(id);
                if (dish != null) {
                    // 热点Key设置永不过期，通过定时任务主动更新
                    redisTemplate.opsForValue().set(cacheKey, JSON.toJSONString(dish));
                }
                return dish;
            } else {
                // 4. 锁失败，重试（短休眠后查缓存）
                Thread.sleep(50);
                return getHotDishById(id);
            }
        } catch (Exception e) {
            log.error("查询热点菜品失败", e);
            return null;
        } finally {
            // 5. 释放锁
            if (locked) {
                redisTemplate.delete(lockKey);
            }
        }
    }
}

// 定时任务主动更新热点缓存（避免永不过期导致数据不一致）
@Component
public class HotDishCacheTask {

    @Autowired
    private DishService dishService;
    @Autowired
    private StringRedisTemplate redisTemplate;

    // 每30分钟更新一次热点菜品缓存
    @Scheduled(cron = "0 0/30 * * * ?")
    public void refreshHotDishCache() {
        log.info("开始更新热点菜品缓存");
        List<Long> hotDishIds = dishService.listHotDishIds(); // 查询爆款菜品ID列表
        for (Long id : hotDishIds) {
            Dish dish = dishService.getById(id);
            redisTemplate.opsForValue().set("dish:hot:" + id, JSON.toJSONString(dish));
        }
        log.info("热点菜品缓存更新完成");
    }
}
```

##### 4. 缓存雪崩处理（必做）

**问题**：大量缓存Key同时过期，导致数据库瞬间压力激增。
**解决方案**：过期时间添加随机值 + 服务熔断/降级

```java
@Configuration
@EnableCaching
public class RedisCacheConfig {

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
        // 1. 配置缓存过期策略
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofHours(12)) // 基础过期时间12小时
            .serializeKeysWith(RedisSerializationContext.SerializationPair.fromSerializer(new StringRedisSerializer()))
            .serializeValuesWith(RedisSerializationContext.SerializationPair.fromSerializer(new GenericJackson2JsonRedisSerializer()));

        // 2. 为不同缓存设置随机过期时间（避免同时过期）
        Map<String, RedisCacheConfiguration> cacheConfigMap = new HashMap<>();
        // 菜品缓存：12小时 ± 10分钟
        cacheConfigMap.put("dish", config.entryTtl(Duration.ofHours(12).plusMinutes(new Random().nextInt(20) - 10)));
        // 店铺缓存：24小时 ± 30分钟
        cacheConfigMap.put("shop", config.entryTtl(Duration.ofHours(24).plusMinutes(new Random().nextInt(60) - 30)));

        // 3. 创建缓存管理器
        return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .withInitialCacheConfigurations(cacheConfigMap)
            .build();
    }
}

// 结合Sentinel做服务降级（可选，防止DB压力过大）
@Service
public class DishServiceImpl {
    @SentinelResource(value = "dishService", fallback = "getDishFallback")
    public Dish getById(Long id) {
        // 原有逻辑
    }

    // 降级逻辑：缓存和DB都异常时，返回默认空数据
    public Dish getDishFallback(Long id, Throwable e) {
        log.error("查询菜品降级，ID：{}", id, e);
        return new Dish(); // 返回空对象，避免前端报错
    }
}
```

#### 三、SpringCache 优化配置（适配订餐系统）

```java
@Configuration
@EnableCaching
public class SpringCacheConfig {

    // 解决SpringCache默认不支持TTL、序列化问题
    @Bean
    public RedisCacheConfiguration redisCacheConfiguration() {
        return RedisCacheConfiguration.defaultCacheConfig()
            // 设置缓存默认过期时间（30分钟）
            .entryTtl(Duration.ofMinutes(30))
            // 禁用空值缓存（结合手动处理穿透）
            .disableCachingNullValues()
            // Key序列化
            .serializeKeysWith(RedisSerializationContext.SerializationPair.fromSerializer(new StringRedisSerializer()))
            // Value序列化（支持对象）
            .serializeValuesWith(RedisSerializationContext.SerializationPair.fromSerializer(new GenericJackson2JsonRedisSerializer()));
    }
}
```

#### 总结

1. **缓存应用范围**：Redis 缓存不仅用于用户端（菜品、店铺信息），也用于管理端（统计数据、权限信息），核心是降低高频查询的DB压力。
2. **缓存问题处理**：
   - 缓存穿透：参数校验 + 缓存空值（短有效期）；
   - 缓存击穿：分布式锁 + 热点Key永不过期（定时更新）；
   - 缓存雪崩：过期时间加随机值 + 服务降级；
3. **缓存预热**：针对菜品、店铺等核心高频数据，建议系统启动时预热，提升首次访问性能。

以上方案完全适配订餐系统的业务场景，既保证了缓存性能，又避免了缓存常见问题导致的系统稳定性风险。





### 一、实时通信（WebSocket）设计思路（基于苍穹外卖文档）

**核心目标**：管理端**秒级接收新订单、用户催单**，替代轮询，降低服务端压力，提升商家响应速度。

这里应该简单说明一下：没有使用微信支付的api，而是直接在账户上扣除的。

#### 1. 整体设计思路

1. **通信选型**
   采用 **WebSocket 全双工长连接**，一次握手后持久连接，服务端可主动向管理端推送消息，解决 HTTP 被动请求、无效轮询问题。
2. **连接架构**
   - 管理端（浏览器）登录后自动与服务端建立 WebSocket 长连接
   - 服务端用 `SessionMap` 维护所有在线商家客户端
   - 支持**群发推送**：新订单/催单通知所有在线管理员
3. **消息协议**
   统一 JSON 格式，字段固定：
   - `type`：1=新订单提醒；2=用户催单
   - `orderId`：订单ID
   - `content`：订单号/提示文本
4. **触发时机**
   - **新订单提醒**：用户支付成功 → 修改订单状态 → 立即推送
   - **用户催单**：小程序点击催单 → 接口请求 → 立即推送
5. **前端交互**
   接收消息 → 弹窗提示 + 语音播报，商家无感知延迟。

#### 2. 关键实现步骤

1. 引入 `spring-boot-starter-websocket` 依赖
2. 编写 `WebSocketServer` 组件，实现 `@OnOpen`/`@OnClose`/`@OnMessage`
3. 用 `Map<String, Session>` 管理所有客户端连接
4. 提供 `sendToAllClient(String message)` 群发方法
5. 订单支付成功/用户催单时，调用推送接口
6. 管理端 Web 页面监听消息，做提示与播报

### 二、自动化处理（SpringTask）设计思路（基于苍穹外卖文档）

**核心目标**：自动修正订单状态，避免脏数据、人工漏处理，保证业务闭环准确。

#### 1. 整体设计思路

1. **定时框架**
   使用 Spring 内置 `SpringTask`，轻量无依赖，通过 `@Scheduled(cron="...")` 定义执行规则。
2. **两大核心任务**
   - **超时未支付订单自动取消**
   - **派送中订单自动完成**
3. **执行规则**
   - 支付超时：**每分钟执行一次**，检查下单超过15分钟未支付订单
   - 派送完成：**每天凌晨1点执行**，自动完成长期未确认的派送中订单
4. **数据处理逻辑**
   - 按**状态 + 下单时间**筛选订单
   - 批量更新状态、取消原因/完成时间
   - 保证事务一致性，避免状态异常

#### 2. 关键实现步骤

1. 启动类加 `@EnableScheduling` 开启定时任务
2. 新建 `OrderTask` 定时任务类
3. 编写 Mapper 方法：`getByStatusAndOrdertimeLT`
4. 实现两个定时方法：
   - `processTimeoutOrder`：处理待支付超时订单
   - `processDeliveryOrder`：处理派送中自动完成订单
5. 配置 cron 表达式，控制执行频率
6. 日志记录 + 事务保证，确保数据安全可追溯



### JMeter 验证核心点餐接口响应耗时从 200ms 降至 30ms 内，

要通过 JMeter 验证核心点餐接口响应耗时从 200ms 降至 30ms 内，需遵循**对比测试思路**（缓存未启用 vs 缓存启用），并通过标准化的测试步骤保证结果可信。以下是详细的操作流程：

#### 一、测试前准备

##### 1. 环境与数据准备

- **测试环境**：搭建与生产一致的环境（数据库、Redis、应用服务），避免开发环境干扰；
- **数据准备**：
  - 初始化测试数据：提前录入高频访问的菜品、店铺数据（模拟真实点餐场景）；
  - 清空 Redis 缓存（测试“未启用缓存”阶段），确保首次请求走数据库；
  - 记录基准接口：明确核心点餐接口（如 `GET /api/dish/list?shopId=1001`）。

##### 2. JMeter 工具准备

- 下载安装 JMeter（建议 5.5+ 版本），配置 JDK 环境；
- 关闭 JMeter 图形化界面的非必要功能（如监听器实时统计），减少性能损耗。

#### 二、核心测试步骤（分 2 轮对比）

##### 第一轮：测试“未启用缓存”的接口响应耗时（目标：≈200ms）

###### 步骤 1：创建 JMeter 测试计划

1. 打开 JMeter，新建**测试计划**，添加**线程组**（模拟用户并发）：
   - 线程数：50（模拟50个并发用户）；
   - Ramp-Up 时间：5秒（5秒内启动所有线程，避免瞬间压测）；
   - 循环次数：100（每个线程请求100次，累计5000次请求，保证数据样本足够）；
   - 勾选“永远”（可选，测试持续稳定性）。

2. 添加**HTTP 请求** sampler，配置接口信息：
   - 协议：HTTP/HTTPS；
   - 服务器名称/IP：应用服务地址（如 `192.168.1.100`）；
   - 端口号：应用端口（如 8080）；
   - 请求方法：GET（核心点餐接口多为GET）；
   - 路径：`/api/dish/list`；
   - 参数：`shopId=1001`（高频访问的店铺ID，模拟真实场景）。

3. 添加**配置元件**：
   - HTTP 请求默认值：统一配置服务器地址、端口，避免重复设置；
   - 查看结果树（调试用，正式测试可关闭）；
   - 聚合报告（核心监听器，统计响应时间）；
   - 用表格查看结果（辅助分析单次请求耗时）。

##### 步骤 2：执行“未启用缓存”测试

1. 确保 Redis 缓存未启用（或清空缓存），重启应用服务；
2. 点击 JMeter“启动”按钮，执行测试；
3. 测试完成后，查看**聚合报告**，记录核心指标：
   - 平均值（Average）：未启用缓存时的平均响应耗时（目标≈200ms）；
   - 中位数（Median）：50%请求的响应耗时，更贴近真实用户体验；
   - 90% 百分位（90% Line）：90%请求的耗时，反映大部分用户的体验。

##### 第二轮：测试“启用缓存”的接口响应耗时（目标：≤30ms）

###### 步骤 1：启用 Redis 缓存并预热

1. 开启应用的 Redis 缓存功能，执行缓存预热（加载菜品/店铺数据到 Redis）；
2. 验证缓存：通过 Redis 客户端（如 redis-cli）查看 `dish:list:1001` 键是否存在，确保缓存生效。

###### 步骤 2：复用 JMeter 测试计划，执行测试

1. 保持线程组、HTTP 请求等配置与第一轮完全一致（保证对比公平）；
2. 点击“启动”执行测试，完成后查看**聚合报告**：
   - 平均值（Average）：启用缓存后的平均响应耗时（目标≤30ms）；
   - 90% Line：确保90%的请求耗时也在30ms内（仅看平均值不够，需保证大部分请求达标）。

#### 三、关键监听器与指标解读

| 监听器         | 核心指标                  | 含义与解读                                                   |
| -------------- | ------------------------- | ------------------------------------------------------------ |
| 聚合报告       | 平均值（Average）         | 所有请求的响应时间平均值，核心对比指标（未缓存≈200ms → 缓存≤30ms）； |
|                | 90% Line                  | 90%的请求耗时≤该值，反映“大部分用户”的体验，需≤30ms；        |
|                | 最小值/最大值             | 辅助判断是否有极端耗时请求（如缓存击穿导致的最大值偏高）；   |
| 用表格查看结果 | 响应时间（Response Time） | 可筛选单次请求的耗时，定位偶尔的慢请求（如缓存未命中的请求）； |
| 图形结果       | 响应时间趋势图            | 直观展示耗时变化（未缓存时曲线在200ms左右，缓存后降至30ms以下）； |

#### 四、避坑要点（保证测试结果可信）

1. **避免“冷启动”干扰**：
   - 测试前先执行1-2轮“预热请求”（如10个线程循环10次），避免应用首次加载类、建立数据库连接导致的耗时偏高；
2. **控制测试环境**：
   - 测试期间关闭其他无关服务（如日志采集、监控工具），避免资源抢占；
   - 数据库、Redis 单独部署，避免应用与数据库/Redis 争用服务器资源；
3. **保证请求幂等**：
   - 核心点餐接口为GET请求（无写操作），避免POST/PUT等写请求导致数据变化，影响测试结果；
4. **多次测试取平均值**：
   - 每轮测试执行3次，取平均值作为最终结果，避免单次测试的偶然性；
5. **排查缓存未命中**：
   - 若缓存启用后耗时仍高，检查 Redis 缓存是否真的命中（可在接口日志中打印“缓存命中/未命中”标识）。

#### 五、测试结果验证示例

| 测试场景          | 平均响应耗时 | 90% Line | 结论               |
| ----------------- | ------------ | -------- | ------------------ |
| 未启用 Redis 缓存 | 205ms        | 220ms    | 符合基准（≈200ms） |
| 启用 Redis 缓存   | 28ms         | 30ms     | 达标（≤30ms）      |

#### 总结

1. JMeter 测试核心是**对比测试**（缓存未启用 vs 启用），需保证两轮测试的并发、请求参数完全一致；
2. 核心关注**平均值**和**90% Line** 两个指标，验证从 200ms 降至 30ms 内；
3. 测试前清空/预热缓存、控制环境干扰，确保结果能真实反映 Redis 缓存的优化效果。

### 并发安全解决方案（超卖/重复下单/数据不一致）

我给你整理**清晰、可直接写在简历/面试里的完整解决思路**，从**为什么会出问题 → 怎么解决 → 多层兜底方案**，逻辑完整、技术专业、符合真实高并发订餐场景。

#### 一、核心问题场景

在订餐系统高并发场景下，会出现三大致命问题：

1. **超卖**：商品/套餐库存只有10份，并发下单时卖出15份
2. **重复下单**：同一用户重复点击、网络重试，生成多笔相同订单
3. **数据不一致**：库存扣减、订单创建、支付状态不同步

#### 二、整体解决思路（多层防护架构）

采用 **Redis 预扣减 + 分布式锁控制并发 + 数据库最终兜底** 的**三层强一致性架构**，彻底杜绝并发问题。

##### 第一层：Redis + Lua 实现**原子化库存预扣减 + 一人一单**

###### 解决目标：快速拦截无效请求，减少数据库压力

1. **库存预扣减（防超卖）**
   - 把商品库存同步到 Redis，下单前**先扣 Redis 库存**
   - 使用 **Lua 脚本**保证：**库存判断 + 扣减** 是原子操作
   - 扣减成功才允许进入下单流程，扣减失败直接返回“库存不足”

2. **一人一单（防重复下单）**
   - Redis Key 设计：`order:user:{userId}:{dishId}`
   - Lua 脚本中判断：**同一用户 + 同一商品** 是否已存在未支付订单
   - 存在则直接拒绝，从源头拦截重复下单

###### 优势

- Lua 原子性 = 无并发竞争
- Redis 性能极高 = 支持上万 QPS 拦截
- 无效请求直接在缓存层拒绝，不访问数据库

##### 第二层：Redisson 分布式锁（强并发控制）

###### 解决目标：保证同一用户/同一商品**同一时间只有一个下单请求执行**

1. **锁粒度精细化**
   - 锁 key：`lock:order:{userId}:{dishId}`
   - 只锁“当前用户+当前商品”，不影响其他用户，并发性能最大化

2. **Redisson 核心特性**
   - 自动**重试**、自动**续期**（看门狗机制）

3. **锁执行流程**

```lua
-- 1. 获取参数
local dishId = KEYS[1]
local userId = KEYS[2]

local orderKey = "order:user:" .. userId .. ":" .. dishId

end

-- 4. 原子扣减库存
redis.call("decr", stockKey)
-- 5. 标记用户已下单（10分钟超时未支付自动释放）
redis.call("setex", orderKey, 600, "1")

return 0  -- 成功
```

```java
public class MybatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        return interceptor;
    }
}
```

**乐观锁插件用于自动实现数据库乐观锁机制。**

它会在执行更新操作时，**自动拼接版本号（version）判断条件**，并自动版本号 + 1，**无需手动编写复杂 SQL**，就能解决高并发下的**超卖、数据覆盖、数据不一致**问题，是数据库层最终的并发安全兜底方案。

###### 乐观锁原理：

用一个 `version` 版本号控制：

- 只有**版本号匹配**，才允许更新
- 更新成功 → 版本号 +1
- 版本号不匹配 → 更新失败（返回 0 行）

###### 插件到底干了什么？

 它自动帮你做 3 件事：

1. **自动把 version 字段带进 SQL**
2. **自动拼接 WHERE version = ?**
3. **自动让 version + 1**

 **你只需要做**：

1. 实体类加 `@Version`
2. 配置这个插件
3. **正常调用 updateById ()**

插件就会**自动把普通更新 → 变成乐观锁安全更新**。

###### 没有这个插件会怎样？

即使你加了 `@Version`，**乐观锁也不会生效！**

Mybatis-Plus 不会自动处理版本号，并发更新依然会超卖。

###### 最直观对比

 **没插件（普通更新）**

```sql
UPDATE dish SET stock = 0 WHERE id = 1
```

❌ 并发会超卖、数据错乱

**有插件（乐观锁更新）**

```sql
UPDATE dish 
SET stock = 0, version = version + 1 
WHERE id = 1 AND version = 1
```

✅ 并发安全、绝对不超卖

#### 五、核心下单业务代码（完整可运行）

##### `OrderServiceImpl.java`

```java
import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.core.io.ClassPathResource;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.data.redis.core.script.DefaultRedisScript;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.util.Collections;

@Service
public class OrderServiceImpl {

    @Autowired
    private StringRedisTemplate stringRedisTemplate;
    @Autowired
    private RedissonClient redissonClient;
    @Autowired
    private DishMapper dishMapper;
    @Autowired
    private OrderMapper orderMapper;

    // 加载 Lua 脚本
    private static final DefaultRedisScript<Long> SCRIPT;
    static {
        SCRIPT = new DefaultRedisScript<>();
        SCRIPT.setLocation(new ClassPathResource("stockLock.lua"));
        SCRIPT.setResultType(Long.class);
    }

    /**
     * 并发安全下单方法
     */
    @Transactional(rollbackFor = Exception.class)
    public Result createOrder(Long userId, Long dishId) {
        // ===================== 第1步：Redis+Lua 原子拦截 =====================
        Long result = stringRedisTemplate.execute(
                SCRIPT,
                Collections.singletonList(dishId.toString()),  // KEY1: 菜品ID
                Collections.singletonList(userId.toString()) // KEY2: 用户ID
        );

        int code = result.intValue();
        if (code == 1) {
            return Result.fail("您已下单，请勿重复操作");
        }
        if (code == 2) {
            return Result.fail("菜品库存不足");
        }

        // ===================== 第2步：Redisson 分布式锁 =====================
        String lockKey = "lock:order:" + userId + ":" + dishId;
        RLock lock = redissonClient.getLock(lockKey);
        boolean isLock = lock.tryLock();

        if (!isLock) {
            // 缓存库存回滚（重要）
            stringRedisTemplate.opsForValue().increment("dish:stock:" + dishId);
            stringRedisTemplate.delete("order:user:" + userId + ":" + dishId);
            return Result.fail("系统繁忙，请稍后再试");
        }

        try {
            // ===================== 第3步：数据库乐观锁扣库存 =====================
            Dish dish = dishMapper.selectById(dishId);
            if (dish.getStock() <= 0) {
                throw new RuntimeException("库存不足");
            }
            dish.setStock(dish.getStock() - 1);
            // 乐观锁更新：自动带 version 条件
            int rows = dishMapper.updateById(dish);
            if (rows == 0) {
                throw new RuntimeException("并发更新失败，请重试");
            }

            // ===================== 第4步：创建订单（唯一索引防重复） =====================
            Order order = new Order();
            order.setUserId(userId);
            order.setDishId(dishId);
            order.setOrderStatus(0); // 0-未支付
            orderMapper.insert(order);

            return Result.success("下单成功");
        } finally {
            lock.unlock(); // 释放锁
        }
    }
}
```

##### 4. AOP 切面核心实现（防重+分布式锁）

```java
import cn.hutool.core.util.Md5Util;
import lombok.RequiredArgsConstructor;
import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.reflect.MethodSignature;
import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.springframework.stereotype.Component;
import org.springframework.web.context.request.RequestAttributes;
import org.springframework.web.context.request.RequestContextHolder;
import org.springframework.web.context.request.ServletRequestAttributes;

import javax.servlet.http.HttpServletRequest;
import java.lang.reflect.Method;
import java.util.concurrent.TimeUnit;

@Aspect
@Component
@RequiredArgsConstructor
public class NoRepeatSubmitAspect {

    private final RedissonClient redissonClient;

    // 防重 Redis Key 前缀
    private static final String LOCK_KEY_PREFIX = "no_repeat_submit:";

    @Around("@annotation(com.xxx.annotation.NoRepeatSubmit)")
    public Object around(ProceedingJoinPoint point) throws Throwable {
        MethodSignature signature = (MethodSignature) point.getSignature();
        Method method = signature.getMethod();
        NoRepeatSubmit annotation = method.getAnnotation(NoRepeatSubmit.class);
        int lockTime = annotation.lockTime();
        NoRepeatSubmit.KeyType keyType = annotation.keyType();

        // 1. 获取 request
        RequestAttributes attributes = RequestContextHolder.getRequestAttributes();
        HttpServletRequest request = ((ServletRequestAttributes) attributes).getRequest();

        // 2. 生成唯一防重 key
        String lockKey = generateLockKey(request, point, keyType);

        // 3. Redisson 分布式锁：尝试加锁
        RLock lock = redissonClient.getLock(lockKey);
        boolean tryLock = lock.tryLock(0, lockTime, TimeUnit.SECONDS);
        if (!tryLock) {
            // 加锁失败 = 重复提交
            return R.repeatSubmit();
        }

        try {
            // 执行目标方法
            return point.proceed();
        } finally {
            // 释放锁：避免业务异常导致锁永久占用
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    /**
     * 生成防重Key
     */
    private String generateLockKey(HttpServletRequest request, ProceedingJoinPoint point, NoRepeatSubmit.KeyType keyType) {
        String requestUri = request.getRequestURI();
        switch (keyType) {
            case USER_ID:
                // 从Token/Header获取当前登录用户ID，示例
                String userId = request.getHeader("userId");
                return LOCK_KEY_PREFIX + userId + ":" + requestUri;
            case PARAMS:
                // 请求参数MD5，相同参数拦截
                String params = point.getArgs().toString();
                String md5 = Md5Util.md5Hex(params);
                return LOCK_KEY_PREFIX + requestUri + ":" + md5;
            case TOKEN:
                // 前端传唯一防重token
                String submitToken = request.getHeader("submit-token");
                return LOCK_KEY_PREFIX + submitToken;
            default:
                return LOCK_KEY_PREFIX + requestUri;
        }
    }
}
```

#### 四、常见问题解决

1. **业务执行很慢，锁提前过期？**
   调整 `lockTime` 为合理业务时长，或使用 Redisson 续期锁（看门狗）。

2. **集群部署失效？**
   使用 Redisson 分布式锁，所有服务共用 Redis，天然集群生效。

3. **同一接口不同操作误拦截？**
   使用 `USER_ID + 接口URI` 维度，不要只用接口路径。

##### 1. 注入 RedissonClient

```java
import org.redisson.api.RRateLimiter;
import org.redisson.api.RateIntervalUnit;
import org.redisson.api.RateType;
import org.redisson.api.RedissonClient;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import javax.annotation.Resource;

@RestController
public class LimitController {

    @Resource
    private RedissonClient redissonClient;

    @GetMapping("/testLimit")
    public String testLimit() {
        // 1. 创建限流器 key（唯一标识）
        RRateLimiter rateLimiter = redissonClient.getRateLimiter("my-limiter");

        // 2. 设置限流规则
        // RateType.OVERALL = 全局限流（集群共用）
        // 2 = 令牌数，1 = 时间单位，SECONDS = 秒
        // 意思：每秒生成 2 个令牌
        rateLimiter.trySetRate(RateType.OVERALL, 2, 1, RateIntervalUnit.SECONDS);

        // 3. 尝试获取 1 个令牌
        boolean canAccess = rateLimiter.tryAcquire(1);

        if (!canAccess) {
            return "请求频繁，请稍后再试！";
        }

        return "请求成功！";
    }
}
```

###### 规则说明

- `tryAcquire(1)`：尝试拿**1个令牌**，不等待
- `RateType.OVERALL`：分布式全局限流
- `RateType.PER_CLIENT`：单机限流
- `2,1,SECONDS` = **每秒允许 2 个请求**

```java
import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.reflect.MethodSignature;
import org.redisson.api.RRateLimiter;
import org.redisson.api.RateType;
import org.redisson.api.RedissonClient;
import org.springframework.stereotype.Component;

import javax.annotation.Resource;
import java.lang.reflect.Method;

@Aspect
@Component
public class RedissonLimitAspect {

    @Resource
    private RedissonClient redissonClient;

    @Around("@annotation(com.yourpackage.RedissonLimit)") // 改为你的注解包路径
    public Object limit(ProceedingJoinPoint pjp) throws Throwable {
        MethodSignature signature = (MethodSignature) pjp.getSignature();
        Method method = signature.getMethod();

        // 获取注解
        RedissonLimit limit = method.getAnnotation(RedissonLimit.class);
        String key = limit.key() + method.getName();
        long permits = limit.permits();
        long time = limit.time();
        TimeUnit unit = limit.unit();

        // 创建限流器
        RRateLimiter rateLimiter = redissonClient.getRateLimiter(key);
        rateLimiter.trySetRate(RateType.OVERALL, permits, time, unit);

        // 尝试获取令牌
        boolean acquire = rateLimiter.tryAcquire();
        if (!acquire) {
            return "系统繁忙，请稍后再试！";
        }

        return pjp.proceed();
    }
}
```

### 防重复提交 VS Redisson限流 核心区别（通俗易懂+生产级对比）

结合你刚才两套代码：**Redisson限流(RRateLimiter)**、**防重复提交(RLock分布式锁)**，从核心算法、目的、规则、场景、粒度逐一拆解。

#### 一、核心本质不同

##### 1. 限流（RRateLimiter）

- 底层：**令牌桶算法**
- 核心：**控制单位时间内总请求量**
- 逻辑：每秒/每分钟只放行 N 个请求，超出直接拒绝
- 关键词：**削峰、抗并发、防雪崩、控QPS**

##### 2. 防重复提交（Redisson分布式锁 RLock）

- 底层：**互斥锁 + 时间锁**
- 核心：**短时间内禁止同一用户/同一参数重复操作**
- 逻辑：5秒内同一个人不能点两次提交，和总并发无关
- 关键词：**防连点、防表单重复、防误操作、保证幂等**

---

#### 二、关键维度完整对比表

| 对比维度         | 限流（RRateLimiter）              | 防重复提交（RLock 锁）               |
| ---------------- | --------------------------------- | ------------------------------------ |
| **核心目标**     | 限制接口整体QPS，保护服务不被打垮 | 限制**单人单次操作短时间重复触发**   |
| **控制粒度**     | 全局/接口整体、全用户统一配额     | 按用户ID / 请求参数 / 维度隔离       |
| **时间规则**     | 固定频率：每秒N次、每分钟N次      | 固定锁定时长：3s/5s/10s 禁止二次提交 |
| **底层实现**     | 令牌桶、平滑限流                  | Redis 互斥分布式锁 + 过期key         |
| **业务场景**     | 秒杀、接口过载、高并发、爬虫拦截  | 下单、付款、新增、修改、表单提交     |
| **是否允许并发** | 允许正常并发，只是上限封顶        | **禁止同一人短时间并发操作**         |
| **失效方式**     | 令牌自动恢复、循环生成            | 锁超时自动释放 / 代码手动释放        |
| **典型提示**     | 请求过于频繁、系统繁忙、访问受限  | 请勿重复提交、操作请勿频繁点击       |
| **集群效果**     | 分布式全局限流，集群共用配额      | 分布式全局防重，多节点统一拦截       |

---

#### 三、举实例秒懂区别

##### 场景1：限流

配置：**每秒允许 5 个请求**
- 10个用户同时疯狂点接口
- 前5个正常通行，第6~10个直接拦截
- **不管是谁，超总量就拦**
👉 作用：保护服务器CPU、数据库，防止压垮

##### 场景2：防重复提交

配置：**同一个用户5秒内只能提交1次**
- 用户A连续点3次下单
- 第一次成功，后两次直接拦截
- 用户B、用户C正常提交不受影响
👉 作用：防止用户手抖连击、网络超时重试产生**重复订单、重复扣款、重复数据**

### 二、后端模块化设计（横向）

后端采用Maven多模块工程，拆分核心职责，降低模块耦合，便于团队协作和版本管理：

#### 1. 模块依赖关系

```
sky-server（核心服务） 
├── 依赖 sky-common（通用能力）
└── 依赖 sky-pojo（数据模型）
sky-common（通用能力）
└── 依赖 sky-pojo（数据模型）
sky-pojo（数据模型）
└── 无下游依赖（最基础模块）
```
- **sky-pojo**：最底层模块，仅定义数据结构，无业务逻辑，供所有后端模块依赖，保证数据模型统一；
- **sky-common**：通用能力模块，依赖sky-pojo，封装工具类、异常、配置等，避免重复编码；
- **sky-server**：业务核心模块，依赖前两个模块，实现所有外卖业务逻辑，是后端对外提供服务的唯一入口。

#### 2. 模块内部目录规范（推测，基于Java企业级项目通用规范）

每个模块的`src/`目录遵循Maven标准目录结构：
```
src/
├── main/
│   ├── java/        # 源码目录
│   │   └── com/sky/ # 包名前缀（推测）
│   │       ├── pojo/      # sky-pojo：实体/VO/DTO
│   │       ├── common/    # sky-common：工具类/异常/常量
│   │       └── server/    # sky-server：控制器/服务/持久层
│   └── resources/   # 资源目录
│       ├── application.yml # 配置文件（数据库、端口、日志等）
│       ├── mapper/         # MyBatis映射文件（sky-server）
│       └── static/         # 静态资源（可选）
└── test/            # 测试目录（单元测试、集成测试）
```

### 三、核心业务流程与技术交互

以“用户下单”为例，说明各模块/组件的交互逻辑：
1. **前端触发**：用户在WeiXinApp小程序选择商品，提交订单，小程序通过uni-app封装的请求API，将订单数据（DTO格式）发送到Nginx地址；
2. **Nginx转发**：Nginx根据请求路径（如`/api/order`）反向代理到sky-server的后端接口（如`127.0.0.1:8080/api/order`）；
3. **后端处理**：
   - sky-server的`OrderController`接收请求，校验参数（基于sky-common的校验工具）；
   - `OrderService`处理业务逻辑：扣减库存、生成订单号、关联用户信息（基于sky-pojo的OrderDTO/OrderEntity）；
   - 调用MyBatis将订单数据持久化到MySQL（通过sky_take_out.sql定义的订单表）；
   - 若出现异常，抛出sky-common定义的业务异常，由全局异常处理器返回统一格式响应；
4. **响应返回**：sky-server将处理结果（VO格式）返回给Nginx，再由Nginx转发到小程序；
5. **前端渲染**：小程序接收响应，更新页面状态（如显示订单成功、跳转订单详情）。

### 四、部署架构

项目提供完整的部署配套组件，支持单机/集群部署（单机为例）：
```
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│  微信客户端   │    │   Nginx服务器  │    │  后端服务节点  │
│  （小程序）   │◄──►│  (Web/nginx)  │◄──►│  (sky-server) │
└───────────────┘    └───────────────┘    └───────────────┘
                                                  │
                                                  ▼
                                          ┌───────────────┐
                                          │   MySQL数据库 │
                                          │  (sky_take_out.sql) │
                                          └───────────────┘
```
1. **部署环境**：Linux服务器（推荐），安装JDK（运行Java后端）、MySQL（数据存储）、Nginx（代理/静态资源）；
2. **打包流程**：
   - 后端：通过Maven打包（`mvn clean package`），生成sky-server的jar包；
   - 前端：小程序通过微信开发者工具打包，或uni-app编译为微信小程序包；
3. **运行流程**：
   - 启动MySQL，执行sky_take_out.sql初始化表结构和数据；
   - 启动sky-server（`java -jar sky-server.jar`），加载application.yml配置；
   - 启动Nginx，配置反向代理规则（指向sky-server的端口）；
   - 小程序配置请求域名（指向Nginx地址），即可正常访问。

### 五、架构设计亮点

1. **模块化拆分**：将数据模型、通用能力、业务核心拆分为独立模块，降低耦合，便于单独维护和版本升级（如替换sky-common的工具类不影响业务）；
2. **前后端分离**：前端专注交互，后端专注业务，通过接口解耦，支持多端扩展（如后续增加H5、APP端，仅需开发前端，复用后端接口）；
3. **工程化规范**：
   - 统一Maven管理依赖，避免版本冲突；
   - 完整的数据库设计文档和SQL脚本，降低协作成本；
   - .gitignore/.gitattributes保证版本控制规范；
4. **可扩展性**：
   - 后端：基于Spring Boot可快速集成Redis（缓存）、RabbitMQ（消息队列）、Elasticsearch（搜索）等中间件；
   - 部署：Nginx支持多后端节点负载均衡，可扩展为集群部署；
5. **可维护性**：通用能力抽离、异常体系统一、数据模型标准化，降低后期维护成本。

### 六、总结

该外卖系统架构是典型的“轻量级企业级应用架构”，以Java后端为核心，结合微信小程序前端，通过Nginx实现网关层能力，覆盖外卖业务全流程。架构设计兼顾了“开发效率”和“可维护性”，既适合中小型外卖平台快速落地，也可通过扩展中间件、拆分微服务（如拆分为用户服务、订单服务、商品服务）适配大型平台的需求。

## **用户端（小程序） + 管理员端（Web）双端登录与权限实现原理**

全程通俗、不绕弯，一看就懂👇

### 一、核心结论（先记住）

你的项目是**双身份体系、两套登录、互不干扰**：
1. **Web管理端** → 管理员/员工 → **账号密码登录**
2. **小程序端** → C端用户 → **微信一键授权登录**

权限通过 **JWT令牌 + 拦截器** 统一控制。

---

### 二、Web 管理端（管理员/员工）登录与权限实现

#### 1. 登录流程

1. 管理员在网页输入 **账号 + 密码**
2. 后端接收请求，**校验密码**（BCrypt加密比对）
3. 校验通过 → **生成JWT令牌（token）**
4. 把 token 返回给前端
5. 前端把 token 存在 **localStorage**
6. 后续所有请求**请求头携带token**

#### 2. 权限如何控制？

后端有一个 **LoginCheckInterceptor 登录拦截器**：
- 所有后台接口（除登录外）**必须经过拦截器**
- 拦截器检查请求头有没有 token
- 有token → 解析出 **员工ID、员工姓名、角色**
- 无token / token过期 → **直接拒绝访问**
- 权限不足 → 返回 401

#### 3. 权限细分

- **超级管理员**：所有权限（增删改查）
- **普通员工**：只能看订单、接单，不能删菜品、改配置
- **禁用员工**：无法登录

权限通过 **员工表（employee）** 中的 `status` 字段控制：
- 1 = 正常
- 0 = 禁用

---

### 三、小程序端（C端用户）登录与权限实现

#### 1. 登录流程

1. 用户打开小程序
2. 点击**微信授权登录**
3. 小程序获取 **微信code**
4. 把 code 发给后端
5. 后端调用微信接口 → 获取 **openid**（用户唯一标识）
6. 查看数据库：
   - 有这个openid → 登录
   - 没有 → **自动注册用户**
7. 生成 **JWT令牌** 返回给小程序
8. 小程序把 token 存在 **storage**
9. 后续请求头携带token

#### 2. 权限控制

同样使用 **拦截器**：
- 小程序用户只能访问 **用户端接口**
- 不能访问管理员接口
- 未登录用户：不能下单、不能看订单、不能管理地址

---

### 四、最关键：双端登录为什么不会冲突？

因为：
#### 1. 两套表

- **管理员表：employee**
- **用户表：user**

#### 2. 两套登录接口

- 管理端：`admin/employee/login`
- 用户端：`user/user/login`

#### 3. 两套拦截规则

- 管理端接口：`/admin/**`
- 用户端接口：`/user/**`
- 拦截器会**区分路径**，判断是哪种身份

#### 4. 同一套 JWT 技术

虽然登录方式不同，但**技术实现一样**：
- 都是 token 鉴权
- 都是请求头携带
- 都是后端统一解析
