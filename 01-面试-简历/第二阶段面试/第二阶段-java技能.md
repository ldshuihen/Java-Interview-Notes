## RabbitMQ

### RabbitMQ 消息丢失、重复、幂等性解决方案

这是**RabbitMQ面试必问核心考点**，面试官重点考察你对MQ可靠性设计、生产实战问题的理解，我按照**问题场景→原因→解决方案→代码实现**整理，直接背就能用。

##### 一、先明确核心前提

RabbitMQ 消息流转链路：
**生产者 → 交换机(Exchange) → 队列(Queue) → 消费者**
消息丢失/重复，一定发生在这三个环节之一。

---

##### 二、消息丢失问题（3个环节+完整解决方案）

###### 1. 生产者丢失消息（发送到MQ失败）

**原因**：网络波动、MQ宕机，生产者发送消息后，不知道MQ是否收到。
**解决方案**：
1. **开启生产者确认机制（ConfirmCallback）**
   消息成功到达交换机，返回ack；失败返回nack。
2. **开启路由失败回退机制（ReturnCallback）**
   消息到达交换机，但**路由不到队列**时，触发回退。

###### 2. MQ自身丢失消息（交换机/队列宕机）

**原因**：MQ重启、断电，内存中的消息丢失。
**解决方案**：
1. **交换机、队列、消息都设置持久化**
   - 队列：`durable=true`
   - 消息：`deliveryMode=2`（持久化）
2. **不要用自动应答，配合手动应答**
   避免MQ刚发消息就删除，消费者还没处理完。

###### 3. 消费者丢失消息（处理失败/异常）

**原因**：MQ默认**自动确认**，消费者拿到消息后立即宕机，MQ认为消费成功，消息丢失。
**解决方案**：
**关闭自动ack，开启手动确认**
- 处理成功：`channel.basicAck()` 手动确认
- 处理失败：`channel.basicNack()` 重回队列/丢弃

---

##### 三、消息重复问题（必现问题，无法100%避免）

###### 原因

1. 生产者重试发送重复消息
2. 消费者网络波动，MQ未收到ack，重新投递
3. 消费者异常重启，MQ重发消息

###### 核心结论

**重复无法避免，只能保证消费端【幂等性】**（重复消费和消费一次结果一致）。

---

##### 四、幂等性解决方案（面试重点！5种方案）

幂等性：**同一消息消费多次，业务结果不变**。

###### 方案1：唯一ID + 分布式锁（最通用、推荐）

**流程**：
1. 生产者发送消息时，携带**全局唯一ID**（业务ID/UUID）
2. 消费者拿到消息，**先查Redis/DB是否存在该ID**
3. 存在：直接丢弃，不处理
4. 不存在：执行业务逻辑 → 写入唯一ID → 手动ack

**优点**：通用、简单、高性能
**缺点**：多一次查询

###### 方案2：数据库唯一约束（简单粗暴）

**适用**：插入数据场景
给业务表加**唯一索引**（订单号、用户ID+业务ID），重复插入直接报错，不影响数据。

###### 方案3：Redis 分布式锁（Redisson）

**流程**：
1. 消费者拿到消息，用消息ID加锁
2. 加锁成功：执行业务
3. 加锁失败：说明正在处理/已处理，直接丢弃

###### 方案4：状态机幂等（业务层设计）

**适用**：订单、支付等状态流转业务
给数据加**状态字段**（待支付→已支付→已完成），只有状态符合才执行，重复消息直接跳过。

###### 方案5：全局消息表（最可靠，适合金融/支付）

**流程**：
1. 建一张`msg_log`表，字段：msg_id、status、create_time
2. 消费时：开启事务 → 插入消息日志（唯一约束）→ 执行业务 → 提交事务
3. 重复消息：插入失败，事务回滚，直接丢弃

**优点**：绝对可靠
**缺点**：性能稍差，需要额外表

---

##### 五、面试满分回答模板（直接背诵）

###### 问题1：RabbitMQ消息丢失怎么解决？

我从**生产者、MQ服务、消费者**三个环节解决：
1. **生产者端**：开启ConfirmCallback确认机制+ReturnCallback回退机制，确保消息可靠投递到交换机和队列，失败则重试/告警。
2. **MQ服务端**：交换机、队列、消息全部设置**持久化**，配合磁盘存储，避免MQ重启丢失数据。
3. **消费者端**：关闭自动ack，使用**手动确认**，处理成功再应答，异常则拒绝/重回队列，防止消费失败丢失消息。

###### 问题2：消息重复和幂等性怎么解决？

消息重复无法100%避免，核心是保证**消费端幂等**，我常用方案：
1. **唯一ID+Redis判重**（首选）：生产者带全局唯一ID，消费者先查Redis，存在则丢弃，不存在再处理。
2. **数据库唯一索引**：插入场景直接用唯一约束防重。
3. **分布式锁**：用Redisson加锁，保证同一消息只处理一次。
4. **业务状态机**：通过状态字段控制，重复消息不执行逻辑。

---

##### 六、Java代码核心实现（关键片段）

###### 1. 生产者 确认+回退 配置

```yaml
spring:
  rabbitmq:
    publisher-confirm-type: correlated # 开启确认
    publisher-returns: true # 开启回退
    template:
      mandatory: true
```

###### 2. 消费者 手动ack + 幂等性（Redis判重）

```java
@RabbitListener(queues = "testQueue")
public void receive(Message message, Channel channel) throws IOException {
    long tag = message.getMessageProperties().getDeliveryTag();
    // 1. 获取消息唯一ID
    String msgId = message.getMessageProperties().getHeader("spring_returned_message_correlation");
    
    try {
        // 2. 幂等判断：Redis不存在才执行
        Boolean flag = stringRedisTemplate.opsForValue().setIfAbsent("msg:" + msgId, "1", 10, TimeUnit.HOURS);
        if (!flag) {
            // 重复消息，直接确认
            channel.basicAck(tag, false);
            return;
        }

        // 3. 执行业务逻辑
        System.out.println("消费消息：" + new String(message.getBody()));

        // 4. 手动确认
        channel.basicAck(tag, false);
    } catch (Exception e) {
        // 5. 异常：拒绝消息，可选择重回队列
        channel.basicNack(tag, false, true);
    }
}
```

---

##### 总结

1. **消息丢失**：生产者确认 + MQ持久化 + 消费者手动ack
2. **消息重复**：无法避免，只能做**消费端幂等**
3. **幂等首选**：唯一ID + Redis分布式锁/判重（简单高效）
4. **最高可靠**：消息日志表 + 事务（适合核心业务）

这套答案覆盖**面试80%以上RabbitMQ可靠性问题**，条理清晰、实战性强，面试官直接给高分。

## RocketMQ

### RocketMQ 消息丢失、重复投递、幂等性解决方案

适配Java后端面试，**分层拆解原因+对应方案+高频背诵话术+区别RabbitMQ关键点**，直接背。

#### 一、消息流转链路

生产者 → NameServer → Broker → 消费者
消息问题分三层：**生产端、Broker服务端、消费端**

---

#### 一、RocketMQ 消息丢失 原因+完整解决方案

##### 1. 生产端消息丢失

###### 丢失原因

1. 同步发送超时、网络抖动、Broker宕机，消息发出去无响应；
2. 使用**单向发送 sendOneway**，无任何应答，直接丢失；
3. 未处理发送异常、失败无重试。

###### 解决方案

1. **拒绝 oneway 单向发送**，核心业务使用：
   - 同步发送 `send()`
   - 异步发送 `sendAsync()`
2. **开启生产者重试**
   生产者内置重试机制，默认重试次数、可自定义重试次数。
3. **校验发送结果**
   判断 `SendResult` 状态，非 `SEND_OK` 手动重试、记录日志、落库告警。
4. **设置超时时间**，避免阻塞丢失。

---

##### 2. Broker 服务端消息丢失

##### 丢失原因

1. 默认**异步刷盘**，消息写入PageCache就返回成功，机器断电/宕机，内存数据丢失；
2. 主从架构异常，主节点挂掉、未同步完数据就切换从节点；
3. 磁盘满、文件损坏、存储故障。

##### 解决方案

1. **调整刷盘策略**
   - 普通业务：**异步刷盘**（性能高）
   - 金融/核心业务：**同步刷盘 `SYNC_FLUSH`**，落盘成功才返回ACK
2. **开启主从复制 + 高可用集群**
   部署Master+Slave，保证数据副本；
   主节点故障自动故障转移，避免单节点数据丢失。
3. 定期清理磁盘、监控Broker存储、日志告警。

---

##### 3. 消费端消息丢失

###### 丢失原因

1. 消费者收到消息，**业务异常、程序崩溃**，未提交消费位点Offset；
2. 手动提交Offset时机错误，提前提交；
3. 批量消费批量提交，部分成功部分失败导致数据丢失。

###### 解决方案

1. **严格遵循：先执行业务，后提交Offset**
2. 使用RocketMQ两种消费模式合理选择：
   - **集群模式**：默认，保证消息只被一个消费者消费
   - 顺序消息必须保证顺序消费+顺序提交
3. 消费异常时：
   - 不提交Offset
   - 重试队列重试/人工补偿，**不能直接丢弃消息**
4. 关闭自动提交，使用**手动提交Offset**（核心业务推荐）

---

#### 二、RocketMQ 消息重复投递 原因

RocketMQ **天生无法彻底避免重复**，常见场景：
1. **消费成功但Offset提交失败**，Broker未收到位点，下次重启重新投递；
2. 消费者宕机、网络断连，Broker重复推送消息；
3. 生产者超时重试，产生重复消息；
4. 集群负载均衡、重平衡触发重发；
5. 消息重试队列多次投递。

> 面试金句：
> RocketMQ 保证的是 **At-Least-Once（至少一次投递）**，必然会出现消息重复，业务必须做幂等。

---

#### 三、RocketMQ 幂等性 5种落地方案（面试核心）

##### 方案1：全局唯一消息ID / 业务唯一键 + Redis去重【最常用】

##### 实现

1. 发送消息携带唯一标识：
   - 原生：`msgId`（RocketMQ自动生成）
   - 业务：订单号、支付ID、流水号
2. 消费前先查Redis：
   - 已存在 → 直接消费忽略，不执行业务
   - 不存在 → 执行业务 + 写入Redis标记 + 设置过期时间

##### 优点

轻量、高性能、接入简单，互联网项目首选。

---

##### 方案2：数据库唯一索引约束【插入类业务首选】

订单表、流水表、交易表，对**业务唯一字段**加唯一索引：
- 订单号
- 交易流水号
重复消息插入直接报错，数据库拦截，无脏数据。

---

##### 方案3：状态机控制幂等【订单/流程类业务】

适用于有明确状态流转场景：
> 待处理 → 处理中 → 已完成/已关闭

消费逻辑：
只处理**初始状态**数据，
已完成/处理中数据直接跳过，天然防重。

---

##### 方案4：分布式锁 实现幂等（Redisson）

以消息唯一ID为Key加分布式锁：
- 加锁成功：执行业务
- 加锁失败：说明正在处理/已处理，直接跳过
适合并发高、强一致性场景。

---

##### 方案5：消息消费记录表 + 事务【金融/支付高可靠】

新建一张 **消息消费记录表**：
字段：msgId、业务ID、消费状态、时间
1. 消费开启本地事务
2. 先插入消息日志（唯一主键）
3. 再执行业务操作
4. 统一提交事务

重复消息插入日志失败，事务回滚，杜绝重复业务执行。

---

#### 四、面试标准背诵版（一问一答，直接默写）

##### 1. 如何解决RocketMQ消息丢失？

分三层处理：
1. **生产端**：禁用单向发送，使用同步/异步发送，判断发送结果，失败自动重试+日志记录；
2. **Broker端**：核心业务开启同步刷盘，部署主从集群保证数据副本，防止宕机丢数据；
3. **消费端**：业务执行完成后再提交Offset，异常不提交位点，避免消费失败消息丢失。

##### 2. RocketMQ为什么会消息重复？怎么解决？

RocketMQ是至少一次投递机制，消费offset提交失败、服务重启、生产者重试、集群重平衡都会导致重复。
解决方案是业务层实现**幂等性**，常用：
唯一ID+Redis去重、数据库唯一索引、业务状态机、分布式锁、消息消费日志表。

---

#### 五、补充：RocketMQ 与 RabbitMQ 关键区别（加分项）

1. 可靠性模型
- RabbitMQ：默认内存、依赖持久化+手动ACK
- RocketMQ：原生主从复制、刷盘策略可控，企业级高可用更强
2. 重复投递
两者都是 At-Least-Once，都必须业务幂等
3. 性能与场景
RocketMQ 更适合**高并发、大数据量、金融电商**场景

## Kafka

### Kafka 消息丢失、重复投递、幂等性解决方案

适配Java后端面试，分层讲**原因+解决方案+核心原理+背诵话术**，同时对比RabbitMQ、RocketMQ，一套直接背。

#### 前置基础

Kafka投递语义：**At-Least-Once 至少一次**
天然会重复，**无法杜绝重复，必须业务做幂等**
链路：生产者 → Broker分区 → 消费者(消费+提交offset)

---

#### 一、Kafka 消息丢失 三层原因+解决方案

##### 1. 生产者端 消息丢失

###### 丢失原因

1. 生产者发送消息后直接结束，未等待Broker应答；
2. `acks=0` 只管发不管收，网络抖动直接丢消息；
3. 重试次数为0、失败无回调、异常不捕获。

###### 解决方案

1. **配置acks应答级别**
- 核心业务：`acks=all`（最高可靠，全部副本落盘才返回成功）
- 普通业务：`acks=1`（leader写入成功即返回）
- 禁止核心业务用 `acks=0`

2. **开启生产者重试**
```properties
retries=3   # 发送失败自动重试
```

3. **使用同步发送 / 异步+回调**
- 同步：`send().get()` 阻塞等待结果
- 异步：添加 `Callback`，处理失败日志、告警、本地落库补发

4. **设置消息发送超时、避免静默失败**

---

##### 2. Broker服务端 消息丢失

###### 丢失原因

1. leader副本写入内存页缓存就返回，**未刷盘**，机器断电丢数据；
2. 副本同步不及时，leader宕机，未同步的消息丢失；
3. 分区副本数=1，无备份，单点故障直接丢数据。

###### 解决方案

1. **提高副本机制**
分区副本数 `replication-factor ≥ 3`，保证多副本冗余。

2. **配合 acks=all**
必须等待**leader+所有follower副本**写入成功，才确认发送完成。

3. **调整刷盘策略**
- 主动刷盘：`log.flush.interval.messages` 控制落盘时机
- 高可靠场景缩短刷盘间隔，减少内存数据丢失风险

4. **开启ISR机制、合理配置副本同步参数**

---

##### 3. 消费者端 消息丢失【最常见】

###### 丢失原因

1. **自动提交offset**：消费者拉取到消息立刻提交偏移量，业务还没执行完就宕机，消息直接丢失；
2. 批量消费提前批量提交offset；
3. 消费异常直接捕获吃掉，不重试、不补偿。

###### 解决方案

1. **关闭自动提交offset**
```properties
enable.auto.commit=false
```

2. **手动提交offset，顺序严格保证**
> **先消费处理业务 → 业务成功 → 再手动提交offset**

3. 消费异常：
- 不提交offset
- 本地重试/死信队列，下次重启重新消费
- 禁止异常直接吞掉导致位点前移

---

#### 二、Kafka 消息重复投递 原因

1. 消费者**业务成功、offset提交失败**，重启后从旧位点重新消费；
2. 消费者组重平衡(rebalance)，触发重复消费；
3. 生产者重试发送，网络超时导致重复推送；
4. 批量消费部分成功、位点提交错乱；

核心结论：
**Kafka 只保证 至少一次投递，重复不可避免，必须做消费端幂等**

---

#### 三、Kafka 幂等性 主流5种解决方案（面试高频）

##### 方案1：Kafka 原生幂等生产者 + 事务（官方最优）

Kafka 0.11+ 原生支持：
1. **幂等生产者**
配置：
```properties
enable.idempotence=true
```
自动生成 **Producer ID+Sequence序列号**，Broker自动去重，**杜绝生产者重试导致的重复消息**。

2. **Kafka事务**
配合 `transactional.id`，实现：
- 生产+消费原子性
- 跨分区事务
彻底解决生产阶段重复、数据一致性问题。

> 适用：金融、支付、核心交易场景

---

##### 方案2：全局唯一ID + Redis 去重（业务最常用）

1. 每条消息携带唯一标识：
   - 业务ID：订单号、交易流水号
   - 全局唯一Key：UUID、雪花ID
2. 消费逻辑：
   - 消费前查询Redis是否存在该ID
   - 存在 → 重复消息，直接跳过
   - 不存在 → 执行业务 + Redis写入标记 + 设置过期时间

优点：轻量、低侵入、高性能、通用所有MQ

---

##### 方案3：数据库唯一索引（插入类业务）

对业务唯一字段建立**唯一索引**：
- 订单编号
- 支付流水号
重复消息插入数据库直接报错，天然拦截重复数据，简单高效。

---

##### 方案4：业务状态机幂等（流程类业务）

适用于订单、订单支付、退款等场景：
设计状态：待处理→处理中→已完成
消费判断：
**只有初始状态才执行业务，已完成/处理中直接忽略**

---

##### 方案5：消息日志表 + 本地事务（强一致）

新建 `kafka_msg_record` 消息记录表，msg_id唯一约束：
1. 消费开启事务
2. 先插入消息记录
3. 执行业务操作
4. 整体提交事务
重复消息插入失败，事务回滚，保证绝不重复执行业务。

#### 四、面试标准满分回答（直接背诵）

##### 1. Kafka如何解决消息丢失？

分三层保障：
1. **生产者端**：配置 `acks=all`、开启重试机制，使用异步回调或同步发送，捕获发送异常，失败记录日志告警；
2. **Broker端**：分区设置多副本，依靠ISR副本同步机制，配合高可靠刷盘策略，避免单点故障与内存数据丢失；
3. **消费端**：关闭自动提交offset，采用**先业务处理、后手动提交偏移量**，消费异常不提交位点，防止消息丢失。

##### 2. Kafka消息重复怎么产生？如何解决？

Kafka是至少一次投递模型，消费者offset提交失败、消费者组重平衡、生产者重试都会造成消息重复。
解决方案以**业务幂等**为主：
1. 使用Kafka原生幂等生产者，解决生产重试重复；
2. 业务层通过唯一ID+Redis去重、数据库唯一索引、状态机、消息记录表等方式，保证重复消费结果一致；
3. 核心场景结合Kafka事务实现生产消费原子性。
