### 2.中缀转逆波兰表达式

波兰表达式（前缀表达式）和逆波兰表达式（后缀表达式）是对中缀表达式的两种“无括号”改写形式，核心区别在于**运算符的位置**，但二者都能彻底解决中缀表达式的优先级/括号歧义问题，是编译器、计算器等场景的核心基础。

#### 一、核心定义与直观对比

先通过一个简单例子建立认知（中缀表达式：`3 + 4 × 2`）：
| 表达式类型           | 形式        | 核心特征                   | 读法/计算逻辑                                    |
| -------------------- | ----------- | -------------------------- | ------------------------------------------------ |
| 中缀表达式           | `3 + 4 × 2` | 运算符在两个操作数之间     | 先算4×2，再算3+结果（需优先级/括号）             |
| 波兰表达式（前缀）   | `+ 3 × 4 2` | 运算符在对应操作数**之前** | 先找`×`，作用于4和2；再找`+`，作用于3和`×`的结果 |
| 逆波兰表达式（后缀） | `3 4 2 × +` | 运算符在对应操作数**之后** | 先读3、4、2；遇到`×`算4×2=8；遇到`+`算3+8=11     |

#### 二、关键区别（核心维度）

| 维度           | 波兰表达式（前缀表达式）                               | 逆波兰表达式（后缀表达式）                             |
| -------------- | ------------------------------------------------------ | ------------------------------------------------------ |
| 运算符位置     | 运算符在**操作数前面**                                 | 运算符在**操作数后面**                                 |
| 计算顺序       | 从右往左扫描，遇到运算符就取其右侧最近的两个操作数计算 | 从左往右扫描，遇到运算符就取其左侧最近的两个操作数计算 |
| 手工计算难度   | 较难（需反向扫描）                                     | 较易（符合人从左到右的阅读习惯）                       |
| 计算机处理效率 | 需先缓存操作数（反向扫描）                             | 更高效（栈操作更顺，无需反向）                         |
| 经典应用       | 编译器语法分析、Lisp语言核心                           | 计算器实现、Java虚拟机指令集、SQL执行计划              |
| 中缀转换复杂度 | 稍高（需反向处理运算符优先级）                         | 更低（主流实现方案）                                   |

#### 三、核心特性详解（结合例子）

##### 1. 波兰表达式（前缀）

- **核心逻辑**：每一个运算符后面跟着它的两个操作数（操作数可以是单个值，也可以是另一个前缀表达式）。
- 示例：中缀 `(3 + 4) × (5 - 2)` → 波兰表达式 `× + 3 4 - 5 2`。
- 计算过程（从右往左）：
  1. 扫描到`2`，缓存；扫描到`5`，缓存；遇到`-`，计算`5-2=3`，替换`- 5 2`为`3`；
  2. 扫描到`4`，缓存；扫描到`3`，缓存；遇到`+`，计算`3+4=7`，替换`+ 3 4`为`7`；
  3. 遇到`×`，计算`7×3=21`，最终结果为21。

##### 2. 逆波兰表达式（后缀）

- **核心逻辑**：每一个运算符前面是它的两个操作数（操作数可以是单个值，也可以是另一个后缀表达式）。
- 示例：中缀 `(3 + 4) × (5 - 2)` → 逆波兰表达式 `3 4 + 5 2 - ×`。
- 计算过程（从左往右，栈实现）：
  1. 3入栈 → 4入栈 → 遇到`+`，弹出4、3，计算7，7入栈；
  2. 5入栈 → 2入栈 → 遇到`-`，弹出2、5，计算3，3入栈；
  3. 遇到`×`，弹出3、7，计算21，最终结果为21。

#### 四、统一优势：无括号、无优先级歧义

中缀表达式的痛点是需要括号和运算符优先级（如`×`优先级高于`+`），而波兰/逆波兰表达式完全规避了这个问题：
- 中缀 `3 + (4 × 2)` 和 `3 + 4 × 2` 的逆波兰都是 `3 4 2 × +`；
- 中缀 `(3 + 4) × 2` 的逆波兰是 `3 4 + 2 ×`；
- 仅通过运算符的位置，就能唯一确定计算顺序，无需额外规则。

#### 五、为什么逆波兰表达式更常用？

虽然二者都能解决问题，但逆波兰表达式在工程实现上更友好：
1. **扫描方向符合直觉**：从左到右扫描即可，无需反向处理；
2. **栈操作更高效**：计算时只需“遇到操作数入栈，遇到运算符出栈计算，结果入栈”，逻辑简单，代码实现成本低；
3. **硬件适配性好**：CPU的指令执行、栈式虚拟机（如JVM）都天然适配后缀表达式的执行逻辑。

#### 总结

1. **核心区别**：波兰表达式运算符在操作数前（前缀），逆波兰在操作数后（后缀），计算扫描方向分别为“从右往左”和“从左往右”；
2. **统一优势**：都无需括号和优先级规则，能唯一确定计算顺序；
3. **工程实践**：逆波兰表达式更常用，因为扫描和栈操作逻辑更简单，是计算器、虚拟机等场景的主流选择。

### 逆波兰表达式（后缀表达式）的核心应用场景

主要有两个：**中缀表达式转逆波兰表达式**、**逆波兰表达式求值**。我会为你提供这两个场景的完整、可运行的Java代码，并详细解释核心逻辑，同时补充边界情况处理（如多位数、负数、括号），满足面试/实际开发需求。

#### 一、场景1：逆波兰表达式求值（核心高频）

##### 1. 核心思路

- 利用**栈**实现：遇到操作数入栈，遇到运算符弹出栈顶两个操作数（注意顺序：后弹出的是左操作数，先弹出的是右操作数），计算后将结果入栈；
- 最终栈中剩余的唯一元素就是表达式结果。

##### 2. 完整代码（支持多位数、正负号、加减乘除）

```java
import java.util.Deque;
import java.util.LinkedList;

public class RPNCalculator {
    /**
     * 逆波兰表达式求值
     * @param tokens 逆波兰表达式数组（如 ["3","4","2","*","+"]）
     * @return 表达式计算结果
     */
    public static int evalRPN(String[] tokens) {
        // 用Deque实现栈（LinkedList比Stack更推荐）
        Deque<Integer> stack = new LinkedList<>();
        
        for (String token : tokens) {
            // 判断是否为运算符
            if (isOperator(token)) {
                // 弹出右操作数（先弹的是右侧）
                int num2 = stack.pop();
                // 弹出左操作数（后弹的是左侧）
                int num1 = stack.pop();
                // 计算并将结果入栈
                int result = calculate(num1, num2, token);
                stack.push(result);
            } else {
                // 操作数：直接转数字入栈（支持负数，如 "-123"）
                stack.push(Integer.parseInt(token));
            }
        }
        // 最终栈顶就是结果
        return stack.pop();
    }

    // 判断是否为运算符
    private static boolean isOperator(String s) {
        return "+".equals(s) || "-".equals(s) || "*".equals(s) || "/".equals(s);
    }

    // 执行运算（注意num1是左操作数，num2是右操作数）
    private static int calculate(int num1, int num2, String op) {
        return switch (op) {
            case "+" -> num1 + num2;
            case "-" -> num1 - num2;
            case "*" -> num1 * num2;
            case "/" -> num1 / num2; // 题目默认除数不为0，实际开发可加异常处理
            default -> throw new IllegalArgumentException("不支持的运算符：" + op);
        };
    }

    // 测试用例
    public static void main(String[] args) {
        // 测试1：3 + 4 * 2 = 11
        String[] rpn1 = {"3", "4", "2", "*", "+"};
        System.out.println(evalRPN(rpn1)); // 输出 11

        // 测试2：(3 + 4) * (5 - 2) = 21
        String[] rpn2 = {"3", "4", "+", "5", "2", "-", "*"};
        System.out.println(evalRPN(rpn2)); // 输出 21

        // 测试3：支持负数 10 - (6 - (-2)) = 2
        String[] rpn3 = {"10", "6", "-2", "-", "-"};
        System.out.println(evalRPN(rpn3)); // 输出 2
    }
}
```

##### 3. 核心细节解释

- **栈的操作顺序**：运算符作用于“栈顶两个数”，但注意 `num1 - num2` 而非 `num2 - num1`（因为栈是后进先出，先弹的是右操作数）；
- **多位数/负数处理**：直接用 `Integer.parseInt()` 即可，无需额外拆分（如 `"123"`、`"-45"` 能直接转数字）；
- **异常处理**：实际开发中可补充“除数为0”“表达式格式错误”（如栈元素不足）的异常捕获，面试中可提及该优化点。

#### 二、场景2：中缀表达式转逆波兰表达式（面试高频）

##### 1. 核心思路

- 初始化**运算符栈**和**结果列表**；
- 遍历中缀表达式字符：
  - 操作数（多位数/负数）：直接加入结果列表；
  - 左括号`(`：入栈；
  - 右括号`)`：弹出栈顶运算符到结果列表，直到遇到左括号（左括号弹出但不加入结果）；
  - 运算符：按优先级比较，栈顶优先级≥当前则弹出，直到栈空/栈顶是左括号，再入栈；
- 遍历结束后，弹出栈中剩余运算符到结果列表。

##### 2. 完整代码（支持多位数、负数、括号、加减乘除）

```java
import java.util.ArrayList;
import java.util.Deque;
import java.util.LinkedList;
import java.util.List;

public class InfixToRPN {
    // 定义运算符优先级（数字越大优先级越高）
    private static int getPriority(String op) {
        return switch (op) {
            case "+", "-" -> 1;
            case "*", "/" -> 2;
            case "(" -> 0; // 左括号优先级最低，确保先入栈
            default -> -1; // 非法运算符
        };
    }

    /**
     * 中缀表达式转逆波兰表达式
     * @param infix 中缀表达式字符串（如 "3+4*2/(1-5)"）
     * @return 逆波兰表达式列表
     */
    public static List<String> infixToRPN(String infix) {
        List<String> result = new ArrayList<>(); // 结果列表
        Deque<String> opStack = new LinkedList<>(); // 运算符栈
        int i = 0;
        int len = infix.length();

        while (i < len) {
            char c = infix.charAt(i);
            // 跳过空格（兼容带空格的表达式，如 "3 + 4 * 2"）
            if (Character.isWhitespace(c)) {
                i++;
                continue;
            }
            // 情况1：数字（支持多位数、负数）
            if (Character.isDigit(c) || (c == '-' && (i == 0 || infix.charAt(i-1) == '(' || isOperator(String.valueOf(infix.charAt(i-1)))))) {
                StringBuilder num = new StringBuilder();
                // 处理负号
                if (c == '-') {
                    num.append(c);
                    i++;
                }
                // 处理多位数
                while (i < len && Character.isDigit(infix.charAt(i))) {
                    num.append(infix.charAt(i));
                    i++;
                }
                result.add(num.toString());
                continue;
            }
            // 情况2：左括号
            if (c == '(') {
                opStack.push(String.valueOf(c));
                i++;
            }
            // 情况3：右括号
            else if (c == ')') {
                while (!opStack.isEmpty() && !"(".equals(opStack.peek())) {
                    result.add(opStack.pop());
                }
                opStack.pop(); // 弹出左括号，不加入结果
                i++;
            }
            // 情况4：运算符（+、-、*、/）
            else if (isOperator(String.valueOf(c))) {
                String op = String.valueOf(c);
                // 弹出栈顶优先级≥当前的运算符
                while (!opStack.isEmpty() && getPriority(opStack.peek()) >= getPriority(op)) {
                    result.add(opStack.pop());
                }
                opStack.push(op);
                i++;
            } else {
                // 非法字符
                throw new IllegalArgumentException("不支持的字符：" + c);
            }
        }
        // 弹出栈中剩余运算符
        while (!opStack.isEmpty()) {
            result.add(opStack.pop());
        }
        return result;
    }

    // 判断是否为运算符
    private static boolean isOperator(String s) {
        return "+".equals(s) || "-".equals(s) || "*".equals(s) || "/".equals(s);
    }

    // 测试用例
    public static void main(String[] args) {
        // 测试1：3+4*2/(1-5) → [3,4,2,*,1,5,-,/,+]
        String infix1 = "3+4*2/(1-5)";
        System.out.println(infixToRPN(infix1)); // 输出 [3, 4, 2, *, 1, 5, -, /, +]

        // 测试2：(3+4)*(5-2) → [3,4,+,5,2,-,*]
        String infix2 = "(3+4)*(5-2)";
        System.out.println(infixToRPN(infix2)); // 输出 [3, 4, +, 5, 2, -, *]

        // 测试3：支持负数 10-(6-(-2)) → [10,6,-2,-,-]
        String infix3 = "10-(6-(-2))";
        System.out.println(infixToRPN(infix3)); // 输出 [10, 6, -2, -, -]
    }
}
```

##### 3. 核心细节解释

- **负数处理**：判断 `-` 是否为负号（开头/左括号后/运算符后），而非减号；
- **多位数处理**：通过循环拼接数字字符，避免拆分“123”为“1”“2”“3”；
- **空格兼容**：跳过表达式中的空格，适配更通用的输入场景。

#### 三、场景3：整合场景1+2（中缀表达式→求值）

将“中缀转逆波兰”和“逆波兰求值”结合，实现中缀表达式直接求值：
```java
public class InfixCalculator {
    public static void main(String[] args) {
        // 中缀表达式：3+4*2/(1-5) = -1
        String infix = "3+4*2/(1-5)";
        // 步骤1：转逆波兰
        List<String> rpnList = InfixToRPN.infixToRPN(infix);
        // 步骤2：转数组求值
        String[] rpnArray = rpnList.toArray(new String[0]);
        int result = RPNCalculator.evalRPN(rpnArray);
        System.out.println("中缀表达式 " + infix + " 的结果：" + result); // 输出 -1
    }
}
```

1. **逆波兰求值**：核心是栈的“入栈-出栈计算”，关键注意操作数顺序（左操作数后弹出）；
2. **中缀转逆波兰**：核心是运算符栈的优先级比较，重点处理括号、多位数、负数；
3. **面试关键点**：能解释栈的作用、运算符优先级规则、负数/多位数的处理逻辑，代码需保证健壮性（异常处理、边界情况）。

### 3. HTTP请求过程（从输入域名到浏览器显示页面）

从输入域名（如`www.baidu.com`）到页面展示，核心分为**7个步骤**，涵盖网络层、应用层、渲染层：

#### （1）步骤拆解

1. **DNS解析**（域名→IP地址）：
   - 浏览器先查本地DNS缓存 → 系统hosts文件 → 本地DNS服务器 → 根DNS服务器 → 顶级域DNS服务器（.com）→ 权威DNS服务器，最终获取目标服务器IP；
2. **建立TCP连接**（三次握手）：
   - 客户端→服务端：SYN（请求连接）；
   - 服务端→客户端：SYN+ACK（确认连接）；
   - 客户端→服务端：ACK（最终确认）；
   - （HTTP/3基于UDP，无需TCP握手）；
3. **发送HTTP请求**：
   - 客户端通过TCP连接向服务端发送请求报文（包含请求行、请求头、请求体）；
   - 示例请求行：`GET /index.html HTTP/1.1`；
4. **服务端处理请求**：
   - 服务端（如Tomcat/Nginx）解析请求，处理业务逻辑（查数据库、调用接口等），生成响应数据；
5. **服务端返回HTTP响应**：
   - 服务端发送响应报文（包含状态行、响应头、响应体）；
   - 示例状态行：`HTTP/1.1 200 OK`，响应体为HTML/CSS/JS等资源；
6. **关闭TCP连接**（四次挥手，HTTP/1.1默认Keep-Alive可复用连接）：
   - 客户端→服务端：FIN（请求关闭）；
   - 服务端→客户端：ACK（确认关闭）；
   - 服务端→客户端：FIN（准备关闭）；
   - 客户端→服务端：ACK（最终关闭）；
7. **浏览器渲染页面**：
   - 解析HTML生成DOM树 → 解析CSS生成CSSOM树 → 合并为渲染树 → 布局（Layout）→ 绘制（Paint）→ 合成（Composite），最终展示页面。

#### （2）补充

- 若涉及HTTPS，还需增加**TLS握手**步骤（验证证书、协商加密算法、生成会话密钥）；
- 浏览器会对静态资源（CSS/JS/图片）做缓存，减少重复请求。

### 4. TCP和UDP的区别

TCP（传输控制协议）和UDP（用户数据报协议）是传输层核心协议，核心差异围绕“可靠性”和“效率”展开：

| 维度     | TCP                                          | UDP                                         |
| -------- | -------------------------------------------- | ------------------------------------------- |
| 连接性   | 面向连接（三次握手建立连接）                 | 无连接（无需握手，直接发送）                |
| 可靠性   | 可靠传输（重传、确认、排序）                 | 不可靠（丢包不重传，无确认）                |
| 有序性   | 保证数据按序到达                             | 不保证有序（可能乱序）                      |
| 拥塞控制 | 有（慢启动、拥塞避免等）                     | 无（发送速率不受限）                        |
| 首部开销 | 大（20~60字节）                              | 小（8字节）                                 |
| 适用场景 | 要求可靠的场景（HTTP/HTTPS、文件传输、邮件） | 追求效率的场景（直播、视频通话、游戏、DNS） |
| 数据边界 | 无（流式数据，需应用层处理）                 | 有（以数据报为单位，一次发送一次接收）      |

### 5. TCP如何保证可靠传输

TCP通过**7大核心机制**实现可靠传输，核心思路是“确认+重传+有序+控速”：

#### （1）核心机制

1. **校验和**：发送方计算数据的校验和，随报文发送；接收方重新计算，若不一致则判定数据损坏，直接丢弃（触发重传）；
2. **序列号与确认应答（ACK）**：
   - 每个TCP报文段都有唯一序列号（标识数据字节位置）；
   - 接收方收到数据后，发送ACK报文，告知发送方“已收到序列号≤X的所有数据，期望接收下一个字节为X+1”；
3. **重传机制**：
   - 超时重传：发送方发送数据后启动定时器，超时未收到ACK则重传；
   - 快速重传：接收方收到乱序数据时，连续发送3个重复ACK，发送方立即重传（无需等待超时）；
4. **流量控制**：
   - 接收方通过TCP报文头的“窗口大小”字段，告知发送方“当前可接收的最大数据量”，避免发送方发送过快导致接收方缓冲区溢出；
5. **拥塞控制**：
   - 避免网络拥塞（如路由器缓存满丢包），核心算法：慢启动→拥塞避免→快速恢复；
   - 动态调整发送方的“拥塞窗口”，控制发送速率；
6. **报文段排序**：
   - 接收方按序列号重组数据，乱序的报文段暂存于缓冲区，等待缺失报文段到达后再按序交付给应用层；
7. **超时重传计时器**：
   - 动态调整超时时间（RTO），基于往返时间（RTT）计算，避免超时时间过短/过长导致频繁重传或延迟。

---

### 6. MySQL MVCC与隔离级别（实现+解决的问题）
#### （1）先明确：MySQL的4种隔离级别（SQL标准）
| 隔离级别                     | 脏读 | 不可重复读 | 幻读     | 备注                   |
| ---------------------------- | ---- | ---------- | -------- | ---------------------- |
| 读未提交（Read Uncommitted） | 可能 | 可能       | 可能     | 性能最高，安全性最低   |
| 读已提交（Read Committed）   | 解决 | 可能       | 可能     | Oracle默认级别         |
| 可重复读（Repeatable Read）  | 解决 | 解决       | 基本解决 | MySQL InnoDB默认级别   |
| 串行化（Serializable）       | 解决 | 解决       | 解决     | 性能最低，完全串行执行 |

#### （2）各隔离级别解决的问题
- **脏读**：读取到其他事务**未提交**的数据（如事务A修改数据但未提交，事务B读取到该数据，若A回滚，B读到的是“脏数据”）；
- **不可重复读**：**同一事务内**，多次读取同一数据，结果不一致（如事务A读取数据后，事务B修改并提交，A再次读取数据不同）；
- **幻读**：**同一事务内**，多次执行同一查询，结果集行数不同（如事务A查询“id<10”的记录有5条，事务B插入一条id=8的记录并提交，A再次查询得到6条）。

#### （3）MVCC如何实现隔离级别（InnoDB核心）
MVCC（多版本并发控制）是InnoDB实现隔离级别的核心，通过**隐藏列+undo log+read view** 实现“读写不阻塞”：

##### ① 核心基础
- **隐藏列**：InnoDB每行数据包含3个隐藏列：
  - `DB_TRX_ID`：插入/更新该行的事务ID（递增）；
  - `DB_ROLL_PTR`：指向undo log的指针（用于回滚/版本回溯）；
  - `DB_ROW_ID`：行唯一标识（无主键时自动生成）；
- **undo日志**：记录数据的历史版本，形成版本链（每次修改数据，都会生成新版本，并通过`DB_ROLL_PTR`关联旧版本）；
- **Read View**：事务读取数据时的“可见性视图”，包含4个核心字段：
  - `m_ids`：当前活跃的事务ID集合；
  - `min_trx_id`：活跃事务最小ID；
  - `max_trx_id`：下一个要分配的事务ID；
  - `creator_trx_id`：创建Read View的事务ID。

##### ② 不同隔离级别的实现逻辑
1. **读未提交（Read Uncommitted）**：
   - 实现：不生成Read View，直接读取数据的**最新版本**（即使未提交）；
   - 问题：脏读、不可重复读、幻读都可能发生；
2. **读已提交（Read Committed）**：
   - 实现：**每次查询都生成新的Read View**；
   - 解决：脏读（只读取已提交的版本）；
   - 未解决：不可重复读（同一事务多次查询生成不同Read View，可能读到新提交的版本）、幻读；
3. **可重复读（Repeatable Read）**：
   - 实现：**事务启动时生成一次Read View，全程复用**；
   - 解决：脏读、不可重复读（同一事务Read View不变，只能读到事务启动前已提交的版本）；
   - 幻读：InnoDB通过**间隙锁+Next-Key锁** 基本解决（MVCC+锁结合）；
4. **串行化（Serializable）**：
   - 实现：放弃MVCC，直接对所有读取的行加锁（读锁），写操作加写锁，事务串行执行；
   - 解决：所有并发问题，但性能极差。

### CORS 跨域资源共享 工作原理简述

1. **同源策略限制**
浏览器出于安全，强制执行**同源策略**：协议、域名、端口三者必须完全一致，否则浏览器会拦截跨域请求的响应。

2. **CORS 核心作用**
CORS 是**跨域资源共享**标准，由**服务端**通过响应头声明「允许哪些跨域来源访问」，让浏览器放行合法跨域请求。

3. **两类请求处理逻辑**
- **简单请求**
满足请求方法（GET/POST/HEAD）、请求头限制，浏览器**直接先发请求**，服务端返回 `Access-Control-*` 系列响应头；浏览器校验响应头合法，才允许前端读取响应数据，否则拦截。

- **预检请求（OPTIONS）**
非简单请求（PUT/DELETE、自定义请求头、带Cookie等），浏览器会**先发送 OPTIONS 预检请求**，询问服务端是否允许该跨域请求；服务端返回允许跨域的配置后，浏览器再发送真实业务请求。

4. **关键响应头**
- `Access-Control-Allow-Origin`：允许的跨域域名（或`*`）
- `Access-Control-Allow-Methods`：允许的请求方法
- `Access-Control-Allow-Headers`：允许的自定义请求头
- `Access-Control-Allow-Credentials`：是否允许携带 Cookie

5. **整体流程总结**
跨域请求发起 → 浏览器识别跨域 → 简单请求直接请求/复杂请求先OPTIONS预检 → 服务端配置CORS响应头 → 浏览器校验通过 → 正常交互；校验失败则浏览器抛出跨域错误。

### 什么是长连接和短连接？它们各有什么优缺点？

#### 一、短连接

**定义**
客户端每次**一次请求+一次响应**后，立即断开TCP连接；下次通信需重新建立连接。
常见：HTTP1.0、普通接口请求。

**优点**
1. 服务端无需维护大量空闲连接，资源占用低；
2. 逻辑简单，适合**短频、一次性**请求场景；
3. 天然规避连接超时、僵死连接问题。

**缺点**
1. 每次通信都要**三次握手、四次挥手**，开销大、延迟高；
2. 频繁并发场景下，频繁创建销毁连接，性能损耗严重。

---

#### 二、长连接

**定义**
TCP连接建立后，**持续保持不断开**，多次请求复用同一个连接；空闲时通过心跳保活。
常见：HTTP1.1、WebSocket、TCP即时通讯、MQ。

**优点**
1. 一次握手，多次通信，减少握手开销，**响应更快**；
2. 高并发高频交互场景性能更强；
3. 支持服务端主动推送数据（如WebSocket）。

**缺点**
1. 长期占用服务端文件描述符、内存，连接数上限受限；
2. 需要额外处理**心跳保活、断线重连、连接隔离**；
3. 空闲长连接易被防火墙、路由切断，维护复杂。

---

#### 三、场景总结

- **短连接**：后台管理、普通查询接口、低频访问业务
- **长连接**：即时聊天、消息推送、直播、实时监控、高频交互系统

### MySQL 数据库命名规范

#### 一、MySQL 数据库命名规范（通用面试版）

1. **统一小写**
库名、表名、字段名、索引名全部使用**小写字母**，避免大小写敏感问题（Linux 下 MySQL 区分大小写）。

2. **下划线分隔**
采用**蛇形命名**，单词之间用下划线 `_` 分隔，禁止驼峰、空格、特殊字符。
例：`user_info`、`order_log`

3. **见名知意、语义清晰**
名称具备业务含义，杜绝拼音、简写、无意义字符。
例：`user_address` 优于 `dz`、`addr1`

4. **禁止关键字与保留字**
不使用 `order`、`user`、`desc`、`status` 等 MySQL 关键字，必要时加业务前缀区分。

5. **合理加前缀、区分业务**
不同业务模块加统一前缀，便于归类管理。
例：用户模块 `user_xxx`、订单模块 `order_xxx`

6. **字段命名统一**
- 状态统一：`status`
- 创建时间：`create_time`
- 更新时间：`update_time`
- 逻辑删除：`deleted`
保持全库字段语义、命名口径一致。

7. **索引命名规范**
- 普通索引：`idx_表名_字段名`
- 唯一索引：`uk_表名_字段名`
- 主键：默认 `PRIMARY`

8. **长度适中**
名称不宜过长，兼顾可读性与存储效率。

---

#### 二、为什么要遵守命名规范

1. **降低理解成本**
结构语义直观，新人快速看懂库表、字段含义，减少沟通与文档依赖。

2. **避免语法与环境问题**
统一小写、不用关键字，规避 Linux 大小写敏感、关键字报错、SQL 语法异常。

3. **便于维护与迭代**
业务模块划分清晰，方便分库分表、迁移、备份、权限管理、问题排查。

4. **减少Bug与线上事故**
字段统一（时间、状态、删除标记），避免代码字段映射混乱、逻辑不一致。

5. **提升团队协作效率**
统一编码标准，多人开发、联调、代码合并更顺畅，减少个性化写法带来的冲突。

6. **利于后期优化与治理**
规范的索引、表命名，方便慢查询优化、索引梳理、数据盘点与运维管理。

---

#### 面试极简背诵版

MySQL 命名规范：**小写、蛇形命名、见名知意、禁用关键字、业务前缀、字段统一、索引规范**。
遵守规范可：**规避环境差异问题、提升可读性、降低维护成本、统一团队标准、减少线上Bug、方便后期优化迭代**。



### Java 静态变量 & 静态方法 详解

#### 一、静态变量（static 修饰成员变量）

1. **定义**
用 `static` 修饰的成员变量，属于**类**，不属于对象。

2. **核心特点**
- 随**类加载**初始化，优先于对象创建；
- **全局唯一**：所有对象共享同一份静态变量；
- 存储在**方法区**，而非堆内存；
- 可通过 **类名.变量名** 直接访问，也可通过对象访问（不推荐）；
- 生命周期：**随类销毁而销毁**。

3. **使用场景**
常量、计数器、全局配置、共享数据。

---

#### 二、静态方法（static 修饰成员方法）

1. **定义**
用 `static` 修饰的方法，属于**类级别**方法。

2. **核心特点**
- 不需要 new 对象，直接 **类名.方法名()** 调用；
- 内部**不能直接使用 this、super**；
- **只能访问静态成员**（静态变量、静态方法），不能直接调用普通成员变量/普通方法；
- 加载时机早，常驻方法区。

3. **使用场景**
工具类方法（如 `Math`、`StringUtils`）、工具通用逻辑、单例获取、常量操作。

---

#### 三、关键区别 & 面试高频点

1. 归属不同：
静态 → 类；非静态 → 实例对象。
2. 调用方式：
静态：类名调用；实例：必须 new 对象调用。
3. 访问限制：
静态方法**不能直接访问实例资源**，实例方法可以访问静态资源。
4. 内存：
静态全局共享，节省内存；实例变量每个对象独立一份。

---

#### 极简背诵版

1. **静态变量**：`static` 修饰，属于类、对象共享、类加载时初始化，全局唯一，用类名访问。
2. **静态方法**：`static` 修饰，属于类，无需实例化调用，只能访问静态成员，不能使用 this。

### GC

#### 一、什么是 Java 垃圾回收（GC）

1. **核心定义**
Java 垃圾回收（Garbage Collection，GC）是 JVM 自动执行的**内存管理机制**，用于**自动回收堆内存中不再被引用、失效的对象**，释放内存空间，避免内存泄漏、内存溢出，无需开发者手动释放内存。

2. **回收目标**
主要回收**堆内存**中没有有效引用的对象；
栈、方法区、本地方法栈一般不由 GC 负责。

3. **核心作用**
- 自动清理无效对象，防止内存浪费；
- 规避手动 free/delete 带来的野指针、内存泄漏问题；
- 保证程序内存稳定。

---

#### 二、如何触发垃圾回收

##### 1. 自动触发（主流、常态）

- 堆内存不足，新生代空间占满时，**自动触发 Minor GC**；
- 老年代空间不足、空间分配担保失败，触发 **Full GC**；
- 程序运行空闲、系统资源充足时，JVM 后台自动定时回收。

##### 2. 手动「建议」触发（不保证立即执行）

只是**建议**JVM 执行 GC，JVM 可拒绝、延迟执行：
```java
System.gc();
// 或
Runtime.getRuntime().gc();
```

##### 3. 特殊场景触发

- 调用 `finalize()` 方法前会尝试回收；
- 年轻代对象晋升老年代、元空间不足时触发 Full GC；
- 显式调用 `System.runFinalization()` 辅助回收。

---

#### 三、面试精简背诵版

1. **垃圾回收**：JVM 自动机制，自动清理堆中**无引用的废弃对象**，释放内存，实现自动化内存管理。
2. **触发方式**
- 自动：堆空间不足、分代区域占满，JVM 自动执行 Minor GC / Full GC；
- 手动：调用 `System.gc()` 仅为建议，不强制立即回收。

### 栈和队列有什么区别？两者的应用场景

#### 一、核心区别

##### 1. 进出规则不同

- **栈（Stack）**：**后进先出** LIFO（Last In First Out）
- **队列（Queue）**：**先进先出** FIFO（First In First Out）

##### 2. 操作端不同

- **栈**：只在**一端**操作（栈顶）
- **队列**：在**一端进、另一端出**

##### 3. 常用操作

- 栈：push 入栈、pop 出栈、peek 查看栈顶
- 队列：offer/enqueue 入队、poll/dequeue 出队、peek 查看队头

---

#### 二、应用场景（高频考点）

##### 栈的应用场景

1. **方法调用栈 / 递归**
   函数调用、返回地址保存，后进先出。
2. **括号匹配、表达式求值**
   如有效括号、逆波兰表达式（计算器）。
3. **浏览器前进后退**
   后退就是弹出栈顶。
4. **DFS 深度优先搜索**
   递归本质就是栈，非递归用 Stack/Deque。
5. **单调栈**
   解决：下一个更大元素、接雨水、柱状图最大矩形。
6. **撤销操作（undo）**

##### 队列的应用场景

1. **任务排队、公平调度**
   打印队列、线程池任务队列。
2. **BFS 广度优先搜索**
   层序遍历、最短路径。
3. **消息队列、异步处理**
   RabbitMQ、Kafka 都是队列思想。
4. **缓冲区、流量控制**
   防止瞬间流量打垮系统。
5. **LRU 近似实现（FIFO 淘汰）**
6. **滑动窗口**
   用队列维护窗口内元素。

---

#### 三、一句话总结

- **栈：后进先出，适合回溯、递归、逆序处理**
- **队列：先进先出，适合排队、顺序处理、缓冲**

#### 一、分库分表到底怎么设计？（标准流程）

分库分表的核心目标：**把一张超大表，拆成 N 个小表 + N 个小库**，让查询、写入都变快。

##### 1. 确定拆分维度（最关键）

分两种：

1. **垂直拆分**（按列拆）
   - 把大字段、不常用字段拆到单独表
   - 例：用户表 = 用户基础信息表 + 用户详情表
   - 适合：表字段太多

2. **水平拆分**（按行拆，主流方案）
   - 把数据按某个**分片键（sharding-key）**分散到不同库/表
   - 例：按 user_id 分成 user_0、user_1…user_15
   - 适合：数据量千万、亿级

##### 2. 选择分片键（Sharding Key）

必须满足：

- **查询最频繁的字段**
- **写入均匀**
- **避免跨库join**
- **业务唯一**（如 user_id、order_id、shop_id）

##### 3. 选择分片算法

最常用 3 种：

1. **取模（mod）**：id % 16 → 表 0~15
2. **范围（range）**：1~100万放表0，100万~200万放表1
3. **哈希（hash）**：hash(id) % 16

##### 4. 确定分片数量

经验公式：

- 单表数据量 **≤ 1000万**
- 单表大小 **≤ 10GB**
- 分片数一般是 **2的幂（8、16、32、64）**，方便扩容

##### 5. 架构选型

- **客户端分片**（Sharding-JDBC、Mybatis 拦截器）
- **代理分片**（Sharding-Proxy、MyCat）
- **数据库原生**（MySQL 8.0 分片表）

**90%公司用：Sharding-JDBC**

#### 二、用自增ID分库分表，如何避免数据不一致？

你问的是：**用数据库自增ID做分片键，会有什么坑？怎么解决？**

##### 1. 自增ID做分片键的问题

- 多个库无法全局唯一
- 主从切换、主主同步会产生重复ID
- 导致**数据路由错误、数据丢失、数据重复**

##### 2. 解决方案（生产标准方案）

###### 方案A：使用**全局唯一ID**代替数据库自增（推荐）

替代方案：

- **雪花算法（Snowflake）**
- **百度 UidGenerator**
- **美团 Leaf**
- **Redis 自增**
- **号段模式（Segment）**

**优点：全局唯一、不重复、无冲突**

###### 方案B：如果必须用数据库自增

设置不同库的**自增起始值 + 步长**
例如 2 个库：

- 库1：1,3,5,7…（auto_increment_offset=1, step=2）
- 库2：2,4,6,8…（auto_increment_offset=2, step=2）

**缺点：库数量固定，无法轻松扩容**

##### 3. 数据一致性如何保证？

- 使用**分布式事务**（Seata AT/TCC/SAGA）
- 最终一致性 + 消息队列（RocketMQ/Kafka）
- 幂等设计（防重复提交）
- 分布式锁（Redisson）

#### 三、取余分库分表后，数据不均衡（数据倾斜）怎么解决？

`id % 16` 最常见问题：**某些表数据特别多，某些特别少**，比如：

- user_0 1000万
- user_1 10万

这叫**数据倾斜**。

#### 解决方案（4种实战方案）

##### 1. 改用哈希分片（最常用）

`hash(user_id) % 16`
比直接取模分布更均匀。

##### 2. 扩容 + 重新分片（数据迁移）

比如从 16 张表 → 32 张表
需要做：

- 数据迁移
- 双写
- 数据校验
- 切流量

##### 3. 冷热数据分离

热数据（最近3个月）→ 小分片
冷数据（历史）→ 单独归档库

##### 4. 自定义分片策略（最灵活）

根据业务规则强制打散：

```
if (userId < 100000) → 表0
else if (userId < 500000) → 表1
...
```

#### 四、如何避免分库分表后的热点问题？

热点 = **某一个库/表被疯狂读写**，CPU/IO 打满。

典型场景：

- 大主播、大商家
- 热门活动
- 爆款商品

#### 1. 热点key识别

- 监控 QPS
- 慢查询
- 数据库连接数
- 磁盘IO

#### 2. 解决热点问题（5种硬核方案）

##### 方案1：热点数据单独分片

把大商家/大主播单独放一个库表，不跟普通用户混在一起。

##### 方案2：缓存兜底（Redis）

所有热点读走缓存，不打数据库。

##### 方案3：分片键加盐（最有效）

给热点ID加随机后缀：

```
hot_id = 热点ID + "_" + random(0~9)
```

让数据分散到 10 个表，解决单表热点。

##### 方案4：读写分离

- 主库写
- 从库读
- 热点读走从库

##### 方案5：限流 + 降级

保护数据库不被打崩。

#### 五、最标准的生产级分库分表设计（直接照抄）

```
1. 分片键：user_id / order_id（业务主键）
2. 分片算法：HASH % 32
3. 全局ID：雪花算法
4. 分片数量：32张表 / 4个库
5. 一致性：Seata 分布式事务
6. 热点处理：缓存 + 加盐分片
7. 数据均衡：HASH分片 + 定期巡检
8. 扩容策略：双倍扩容（32→64）+ 平滑迁移
```

#### 总结（你最关心的4个问题答案）

1. **分库分表怎么设计？**
   选分片键 → 选分片算法 → 定分片数量 → 选分片框架

2. **自增ID分库分表如何避免不一致？**

3. **取余后数据不均衡怎么办？**

### 设计一个实时的天气预报系统，给出架构设计？如果使用API调用现有天气数据，有API调用次数限制，如何确保实时

#### 一、系统整体架构设计（实时天气预报）

##### 1. 架构分层（从上到下）

1. **接入层**
   - Nginx / Gateway
   - 限流、鉴权、IP 黑白名单
2. **应用服务层**
   - 天气查询服务（核心）
   - 地区管理服务（城市编码、经纬度）
   - 定时调度服务（主动拉取天气）
   - 缓存预热 & 刷新服务
3. **数据中间件层**
   - Redis（热点缓存、实时数据）
   - MQ（异步更新、削峰）
4. **数据持久层**
   - MySQL（城市信息、历史天气、用户订阅）
   - MongoDB/ES（海量天气历史归档）

#### 二、核心流程：如何保证“实时”又不超 API 限额？

##### 核心思路

**不每次用户查询都调用 API**，而是：

1. **主动定时拉取**，而不是被动触发调用
2. **多级缓存 + 过期策略**
3. **按城市热度分级刷新**
4. **API 配额严格管控**

##### 1. 数据刷新策略（最关键）

把城市分为 3 类，不同刷新频率：

###### ① 热点城市（北京、上海、广州等）

- 刷新频率：**5～10 分钟/次**
- 方式：**定时任务主动拉取**
- 存储：Redis 永久有效 + 主动覆盖

###### ② 普通地级市

- 刷新频率：**30 分钟/次**

###### ③ 冷门县级市/海外城市

- 刷新频率：**1～2 小时/次**
- 额外兜底：用户首次查询时懒加载一次，之后走定时

这样：
**API 调用次数 = 城市数 × 每日刷新次数**
完全可控，不会因为用户访问暴增而超配额。

---

##### 2. 缓存架构（保证实时性的关键）

使用 **Redis 多级缓存结构**：

1. **实时天气缓存**
   - key：`weather:city:110000`
   - value：JSON 天气对象
   - TTL：根据城市热度设置（10min/30min/2h）
   - 注意：**TTL 略长于刷新间隔**，防止缓存击穿

2. **城市编码映射缓存**
   - key：`city:code:北京`
   - 永久缓存

3. **API 调用频次计数器**
   - key：`api:call:limit:hour`
   - 防止超配额

---

##### 3. 高并发下保证实时性（用户查询流程）

```
用户请求 → 查 Redis
   命中 → 直接返回（毫秒级）
   未命中 →
      1. 用分布式锁防止并发刷 API
      2. 调用第三方 API 获取最新天气
      3. 写入 Redis + 异步落库历史天气
      4. 返回结果
```

关键点：

- **缓存永远不过期，只主动更新** → 实时性最强
- 用**分布式锁**避免缓存击穿导致大量 API 调用
- 用**MQ异步更新**，不阻塞用户请求

---

##### 4. API 配额不够用时的兜底策略

###### 方案1：多个天气厂商互为备用

- 主：和风天气
- 备：彩云天气 / OpenWeatherMap
- 当主 API 达到日限额，自动切备用

- 不影响用户体验，又不超 API

- 夜间 02:00~06:00 API 便宜/余量多，集中更新

### list和set的区别

---

#### 一、核心区别总览

| 特性           | List                                           | Set                                                    |
| :------------- | :--------------------------------------------- | :----------------------------------------------------- |
| **有序性**     | 有序（按插入顺序/索引排序）                    | 无序（HashSet），部分实现有序（LinkedHashSet/TreeSet） |
| **重复性**     | 允许元素重复                                   | 不允许元素重复                                         |
| **索引访问**   | 支持通过索引（下标）直接访问元素               | 不支持索引访问                                         |
| **常用实现类** | `ArrayList`、`LinkedList`、`Vector`            | `HashSet`、`LinkedHashSet`、`TreeSet`                  |
| **底层实现**   | 动态数组/双向链表                              | 基于`HashMap`/`TreeMap`实现                            |
| **使用场景**   | 数据需要按顺序存储、允许重复、需频繁按索引访问 | 数据需要去重、不关心顺序（或需特定排序）               |

---

#### 二、关键细节拆解（面试必讲）

##### 1. 有序性与重复性

- **List**：
  - 有序：`ArrayList`/`LinkedList` 都会保留元素的插入顺序；
  - 允许重复：可以存储多个相同的元素（比如 `[1,2,2,3]`）。
- **Set**：
  - 无序（默认）：`HashSet` 不保证插入顺序；
  - 有序实现：`LinkedHashSet` 保留插入顺序，`TreeSet` 按元素自然顺序/自定义比较器排序；
  - 不允许重复：通过元素的 `equals()` + `hashCode()` 方法判断重复，无法存储相同元素。

##### 2. 访问与遍历方式

- **List**：
  - 支持通过索引 `get(int index)` 快速访问元素（`ArrayList` 随机访问快，`LinkedList` 随机访问慢）；
  - 遍历方式多：for循环、增强for、迭代器、stream流都支持。
- **Set**：
  - 没有索引，不能用下标访问元素；
  - 只能通过增强for循环、迭代器、stream流遍历。

##### 3. 常用实现类与底层

- **List**：
  - `ArrayList`：基于动态数组，随机访问快、增删慢；
  - `LinkedList`：基于双向链表，增删快、随机访问慢；
  - `Vector`：线程安全的动态数组，性能较低，不推荐使用。
- **Set**：
  - `HashSet`：基于`HashMap`实现，无序，查询/添加/删除性能高；
  - `LinkedHashSet`：继承`HashSet`，底层是`Linked`
  - `.HashMap`，保留插入顺序；
  - `TreeSet`：基于`TreeMap`实现，元素按自然序/比较器排序，查询性能略低于`HashSet`。

##### 4. 核心应用场景

- 用 `List`：
  - 需要按顺序存储数据，且允许重复；
  - 需要频繁通过索引访问元素；
  - 比如用户消息列表、订单列表。
- 用 `Set`：
  - 需要对数据去重；
  - 不关心顺序，或需要自动排序；
  - 比如统计用户浏览过的商品ID、去重后的标签集合。

---

#### 三、面试精简版回答（直接背）

> 面试官您好，List和Set的核心区别主要在三点：
> 1. **有序性与重复性**：List是有序且允许元素重复的，Set默认无序且不允许重复；
> 2. **访问方式**：List支持通过索引访问，Set没有索引，只能通过迭代器遍历；
> 3. **底层实现与场景**：List常用ArrayList/LinkedList，适合有序存储、需要索引访问的场景；Set常用HashSet/TreeSet，适合数据去重或需要自动排序的场景。

---

#### 四、加分补充（面试官追问时用）

- Set去重的原理：
  `HashSet` 添加元素时，会先调用元素的 `hashCode()` 计算哈希值定位桶，再调用 `equals()` 判断元素是否相等，两者都相同才判定为重复，拒绝添加。
- List和Set互转的常见场景：
  比如需要对List去重，可以先转成Set，再转回List（但会丢失顺序，想保留顺序可以用`LinkedHashSet`）。



### hashmap底层结构

#### 一、底层结构（JDK1.8 重点）

**数组 + 链表 + 红黑树**
1. 底层主体：**哈希表（数组）**，数组每一个位置称为「桶」；
2. 哈希冲突少时：桶下挂**单向链表**；
3. 哈希冲突严重、链表过长时：链表转为**红黑树**，提高查询效率。

---

#### 二、核心参数

1. **默认初始容量**：16
2. **负载因子**：0.75
3. **扩容阈值**：容量 × 负载因子
4. **树化阈值**：链表长度 ≥ **8**，且数组长度 ≥ **64**，链表转红黑树
5. **退化阈值**：树节点数量 ≤ **6**，红黑树退化为链表

---

#### 三、存取流程（put 流程，必背）

1. 根据 key 计算 `hashCode()`，经过扰动算法优化哈希值；
2. 通过 `(容量-1) & hash` 计算数组下标，定位桶位置；
3. 判断当前桶是否为空：
   - 为空：直接新建节点存入；
4. 桶不为空，发生哈希冲突：
   - **key 完全相同**：覆盖旧 value；
   - **key 不同**：遍历链表/红黑树尾部插入；
5. 插入完成后，判断链表长度是否达到树化条件，满足则树化；
6. 判断元素个数是否超过扩容阈值，超过则**两倍扩容 + 重新哈希迁移**。



#### ConcurrentHashMap JDK1.8 put流程

1. 计算key哈希值，扰动优化，算出数组下标
2. **判断当前桶位置是否为空**
    - 为空：**CAS无锁插入**节点，成功结束；失败自旋重试
3. 桶不为空，判断**头节点是否是扩容标记(FWD)**
    - 是：协助一起**扩容迁移**
4. 既不空也不扩容：**对头节点加synchronized锁**
5. 锁内遍历链表/红黑树
    - key相同：**覆盖value**
    - key不同：链表尾插 / 红黑树插入
6. 插入后判断链表长度≥8，数组长度≥64 → **链表转红黑树**
7. 使用**CAS更新元素数量size**
8. 元素总数超过扩容阈值 → **触发两倍扩容**

---
##### 核心锁机制（必记）

- 空桶：**CAS** 无锁写入
- 有数据桶：**synchronized 锁桶首节点**（细粒度锁）
- 计数、扩容、协助迁移：大量**CAS + volatile**
- 全程**不锁整个数组**，并发性能极高

##### 一句话速背

先算哈希定下标，空桶CAS直接放，遇扩容就帮忙，有头节点上锁遍历，同key覆盖异key尾插，达标树化超量扩容，CAS统计元素个数。

#### ConcurrentHashMap 中 **volatile** 作用

##### 一、volatile 修饰的核心变量

1. `transient volatile Node<K,V>[] table` 哈希数组
2. `volatile int sizeCtl` 扩容控制阈值
3. `volatile Node<K,V> next` 链表下一跳

---
##### 二、三大核心作用（必背）

###### 1. 保证**可见性**

多线程之间变量修改**立刻刷新主内存**，一个线程改完，其他线程马上可见
- 数组table被修改、扩容后，所有线程实时感知
- sizeCtl状态变更，线程立刻知道是否要扩容

###### 2. 禁止**指令重排**

防止JVM编译器乱序执行，保证并发下对象初始化、节点插入、扩容顺序不乱

###### 3. 配合CAS实现**无锁并发**

volatile 不保证原子性，但**保证读写可见**，是CAS自旋操作的前提
- CAS 读取内存最新值，全靠volatile

---
##### 三、分字段精准作用

1. **volatile table 数组**
    数组扩容、初始化完毕后，其他线程立即看到最新数组，避免拿到旧数组操作

2. **volatile sizeCtl**
    控制扩容状态：正数(阈值)、0(未初始化)、负数(正在扩容)
    线程实时识别扩容状态，决定是写入数据还是**协助扩容**

3. **volatile Node.next**
    链表节点链接实时可见，遍历链表不会读到陈旧节点，并发遍历安全

---
##### 四、总结答题话术

ConcurrentHashMap 使用 **volatile** 主要保证**内存可见性、禁止指令重排序**，让多线程实时感知哈希表数组、扩容状态、链表节点的最新数据；
**volatile不保证原子性**，所以计数、修改值依靠**CAS + synchronized**配合实现线程安全。

volatile保可见、禁重排，实时同步数组与扩容状态，支撑CAS无锁操作，弥补无原子性短板，搭配锁实现并发安全。

---

#### 四、哈希冲突解决方式

- JDK1.7：**数组 + 单向链表**，头插法
- JDK1.8：**数组+链表+红黑树**，尾插法
> 为什么改尾插？避免多线程下**循环链表死循环**问题。

### 索引失效的场景

#### 一、核心背诵版（面试口述）

1. **字段类型隐式转换**：字符串不加引号、数字和字符串对比，索引失效。
2. **索引列使用函数/运算**：`left()`、`substr()`、`date_format()`、加减乘除运算。
3. **模糊查询左模糊/全模糊**：`%xx`、`%xx%` 导致索引失效，只有 `xx%` 走索引。
4. **or 连接无索引字段**：or 两边，只要一侧没有索引，整体索引失效。
5. **违背联合索引最左前缀原则**：联合索引(a,b,c)，跳过a直接查b、c，索引失效。
6. **not in、not exists、!=、<>**：会导致索引失效，触发全表扫描。
7. **order by 字段无索引/字段混乱**：排序字段不在索引中，出现 Using filesort。
8. **where 条件使用 is null / is not null**（部分场景），大量null会导致索引失效。
9. **数据量差距过大**：索引列离散度低（如性别、状态只有2种值），优化器自动放弃索引。
10. **手动 select * 、不必要字段**，优化器选择全表扫描更优。

---

#### 二、逐个举例（好理解，防追问）

1. **隐式类型转换**
```sql
-- phone 是字符串索引字段，没加引号，索引失效
where phone = 13800138000
```

2. **索引字段做函数/计算**
```sql
where date(create_time) = '2026-04-28'
where age + 1 > 20
```

3. **左模糊、全模糊**
```sql
where name like '%张三'
where name like '%张三%'
```

4. **or 滥用**
```sql
-- age无索引，整条SQL不走索引
where id = 1 or age = 18
```

5. **联合索引最左匹配失效**
联合索引 `idx_a_b_c(a,b,c)`
```sql
-- 跳过a，直接查b、c，索引失效
where b=1 and c=2
```

6. **!= 、not in**
```sql
where status != 1
where id not in (1,2,3)
```

7. **order by 字段无索引**
额外排序，产生文件排序，索引失效。

8. **字段大量NULL**
索引选择性太差，MySQL 优化器认为全表扫描更快。

---

#### 三、加分优化方案（面试官追问必答）

1. 杜绝索引列做函数、运算，**把函数移到值一侧**；
2. 字符串类型**必须加单引号**，避免隐式转换；
3. 禁止左模糊，能用**全文索引/ES**替代；
4. or 场景改用 `union all` 拆分；
5. 严格遵循**联合索引最左前缀**；
6. 低基数字段（性别、状态）不建索引。

### 我有10张表，查询很慢，如何优化

---

#### 一、先给结论（面试第一句必须说）

> 10张表关联查询慢，核心原因是**联表过多、索引缺失、数据量大、查询逻辑复杂、没有做分页/缓存**。
> 优化思路是：**先查SQL → 加索引 → 减少联表 → 拆分查询 → 加缓存 → 分库分表**。

---

#### 二、完整可背诵回答（3分钟版）

> 面试官您好，如果项目中出现**10张表关联查询很慢**的情况，我会从以下几个方向逐步优化：

##### 1. 第一步：定位问题（ Explain 执行计划）

- 先用 `EXPLAIN` 分析SQL，看是否**全表扫描**、是否**没有走索引**、是否出现**Using filesort / Using temporary**。
- 找到慢的原因：是没索引？还是联表太多？还是数据量太大？

##### 2. 第二步：加索引（最直接、最有效）

- 给**where条件、join关联字段、order by、group by**字段建立**联合索引**。
- 遵循**最左前缀原则**，避免索引失效。
- 大表必须建索引，否则联表一定会全表扫描。

##### 3. 第三步：减少联表数量（10张表太多了！）

- **不要一次join 10张表**，这是数据库大忌。
- 把大SQL**拆分成多个小SQL**，在代码里用**Java内存组装结果**。
- 能冗余字段就冗余，避免为了查一两个字段去join多张表。

##### 4. 第四步：优化SQL逻辑

- 只查**需要的字段**，不要 `select *`。
- 提前加**limit分页**，避免一次性加载全量数据。
- 去掉不必要的**子查询、distinct、排序**。

##### 5. 第五步：加缓存（高并发必备）

- 不经常变化的数据（如题目列表、分类、标签）
- 放入 **Redis 缓存**，设置合理过期时间
- 直接绕开数据库，性能提升10～100倍

##### 6. 第六步：分库分表 / 读写分离（终极方案）

- 单表数据量过大（百万以上）→ **分表**
- 读多写少 → **读写分离**
- 热点数据放缓存，从缓存读取

---

#### 三、最精简版（面试 30 秒背这个）

> 10张表联表查询慢，我会先通过explain分析SQL，然后**给关联字段和where条件加索引**；
> 再**减少联表数量**，把大SQL拆成多个小SQL在代码里拼接；
> 然后**只查需要字段、加分页**；
> 最后把不常变的数据放入**Redis缓存**，从根本提升查询速度。

---

#### 四、你可以说的亮点（加分项！）

这些话一说，面试官会觉得你**真正做过优化**：

1. **联表不超过3-4张是最佳实践**，10张表必须拆分。
2. **联合索引比单字段索引更重要**。
3. **能冗余就不联表**，用空间换时间。
4. **Java内存拼接比数据库join快得多**。
5. **缓存+异步+分页**是解决慢查询的三驾马车。



### inner join和left join

#### 一、核心区别

1. **INNER JOIN（内连接）**
只返回**两张表中完全匹配**的数据，两边条件都满足才保留，**舍弃不匹配数据**。

2. **LEFT JOIN（左连接 / 左外连接）**
以**左表为基准**，左表所有数据**全部保留**；
右表匹配不到的字段，自动补 **NULL**。

---

#### 二、通俗举例

- A表（员工）、B表（部门）
- **内连接**：只查「有部门的员工」，没分配部门的员工直接过滤掉。
- **左连接**：查**所有员工**，哪怕员工没有部门，员工信息还在，部门字段显示null。

---

#### 三、使用场景

##### INNER JOIN

- 需要**两边数据都存在**的业务
- 比如：查询「已绑定部门的员工列表」

##### LEFT JOIN

- 需要**保留左表全部数据**
- 比如：查询**所有员工**，附带所属部门（有无部门都要展示）

---

#### 四、高频面试加分点

1. `left join` 筛选**右表条件**要写在 `on` 后，不要写在 `where`，否则会失效变成内连接效果。
2. 多表联表查询，优先用 `left join` 保证数据不丢失，业务更安全。
3. 右连接 `right join` 极少用，开发中一般调换表顺序改用左连接。

---

#### 五、30秒极简口述版（直接背）

inner join 是内连接，只返回两张表匹配的数据，不匹配的数据会过滤；
left join 是左外连接，以左表为主，左表数据全部保留，右表没有匹配数据就补null。
日常开发查询列表大多使用left join，避免有效数据被过滤。
