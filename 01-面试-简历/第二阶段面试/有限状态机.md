# Java 有限状态机（FSM）：设计 + 使用场景

**有限状态机（Finite State Machine, FSM）** 是一种**行为设计模式**，核心思想：**一个对象在任意时刻只处于一种确定的状态，状态之间通过事件/条件触发切换**，非常适合处理**复杂流程、状态流转、业务规则**。

我会用**最易懂的方式**讲清楚：**怎么设计**、**怎么写Java代码**、**实际业务用在哪**。

---

## 一、核心概念（极简版）
一个完整的 FSM 必须包含 3 个要素：
1. **状态（State）**：当前所处的阶段（如：待支付、已支付、已取消）
2. **事件（Event）**：触发状态变化的动作（如：支付、取消、退款）
3. **转移规则（Transition）**：状态 + 事件 → 新状态

> 一句话：**状态 + 事件 = 下一个状态**

---

## 二、Java 中 FSM 的 4 种设计方案（从简单到高级）
### 方案 1：if-else / switch（最简单，适合简单状态）
适合**状态少、逻辑简单**的场景，快速实现。

```java
// 状态枚举
enum OrderState {
    TO_PAY, PAID, CANCELLED, REFUNDED
}

// 事件枚举
enum OrderEvent {
    PAY, CANCEL, REFUND
}

public OrderState nextState(OrderState currentState, OrderEvent event) {
    return switch (currentState) {
        case TO_PAY -> switch (event) {
            case PAY -> OrderState.PAID;
            case CANCEL -> OrderState.CANCELLED;
            default -> throw new IllegalStateException("非法事件");
        };
        case PAID -> switch (event) {
            case REFUND -> OrderState.REFUNDED;
            default -> throw new IllegalStateException("非法事件");
        };
        default -> throw new IllegalStateException("当前状态无法操作");
    };
}
```

**优点**：简单、无依赖、开箱即用
**缺点**：状态多了代码臃肿

### 方案 2：状态表/枚举驱动（推荐，企业常用）

用**枚举 + 转移表**定义所有状态流转，**可读性最强、维护最简单**。

```java
public enum OrderStateMachine {
    // 原状态    事件        目标状态
    TO_PAY_PAY(OrderState.TO_PAY, OrderEvent.PAY, OrderState.PAID),
    TO_PAY_CANCEL(OrderState.TO_PAY, OrderEvent.CANCEL, OrderState.CANCELLED),
    PAID_REFUND(OrderState.PAID, OrderEvent.REFUND, OrderState.REFUNDED);

    private final OrderState currentState;
    private final OrderEvent event;
    private final OrderState targetState;

    // 根据 当前状态 + 事件 查找下一个状态
    public static OrderState getNextState(OrderState current, OrderEvent event) {
        for (OrderStateMachine t : values()) {
            if (t.currentState == current && t.event == event) {
                return t.targetState;
            }
        }
        throw new RuntimeException("状态不允许流转");
    }
}
```

**这是 Java 业务开发最常用的写法**，清晰、易维护、可扩展。

#### 一、核心思想

把**所有合法状态流转**提前枚举定义成一张**状态转移表**，存入枚举常量；
业务中只需要传入「当前状态 + 触发事件」，查表自动得到**目标状态**，非法流转直接抛异常。

优点：
1. 流转规则集中统一，一眼看懂全流程
2. 零冗余if-else，新增流程只加枚举行
3. 易文档、易校验、好维护，业务开发首选
4. 可扩展前置校验、后置动作、备注说明

#### 二、完整结构分层

1. **状态枚举**：定义业务所有状态
2. **事件枚举**：定义触发状态变更的所有动作
3. **状态转移枚举（状态表核心）**：存储 `原状态+事件→目标状态` 映射
4. **工具类**：提供统一查询、状态流转校验方法
5. **业务实体**：挂载状态，调用状态机流转

##### 3. 核心：状态转移枚举（状态表）

```java
import lombok.Getter;

/**
 * 订单状态转移表 - 枚举驱动FSM核心
 * 格式：源状态 + 触发事件 = 目标状态
 */
@Getter
public enum OrderStateTransition {

    // 1. 待付款 可执行事件
    WAIT_PAY_PAY(OrderStatus.WAIT_PAY, OrderEvent.PAY, OrderStatus.PAID, "用户付款"),
    WAIT_PAY_CANCEL(OrderStatus.WAIT_PAY, OrderEvent.CANCEL, OrderStatus.CANCELLED, "未付款取消订单"),

    // 2. 已付款 可执行事件
    PAID_SHIP(OrderStatus.PAID, OrderEvent.SHIP, OrderStatus.SHIPPED, "商家发货"),
    PAID_REFUND(OrderStatus.PAID, OrderEvent.APPLY_REFUND, OrderStatus.REFUNDED, "付款后申请退款"),

    // 3. 已发货 可执行事件
    SHIPPED_CONFIRM(OrderStatus.SHIPPED, OrderEvent.CONFIRM, OrderStatus.FINISHED, "用户确认收货");

    /** 源状态 */
    private final OrderStatus source;
    /** 触发事件 */
    private final OrderEvent event;
    /** 目标状态 */
    private final OrderStatus target;
    /** 流转描述 */
    private final String desc;

    OrderStateTransition(OrderStatus source, OrderEvent event, OrderStatus target, String desc) {
        this.source = source;
        this.event = event;
        this.target = target;
        this.desc = desc;
    }

    /**
     * 核心查询方法：根据 当前状态 + 事件 获取下一个状态
     */
    public static OrderStatus getNextStatus(OrderStatus currentStatus, OrderEvent event) {
        for (OrderStateTransition transition : values()) {
            if (transition.getSource() == currentStatus && transition.getEvent() == event) {
                return transition.getTarget();
            }
        }
        // 查表无匹配 = 非法状态流转
        throw new IllegalArgumentException(
                String.format("当前状态[%s]不支持执行事件[%s]，非法流转", currentStatus, event)
        );
    }

    /**
     * 判断是否允许流转
     */
    public static boolean isAllowTransition(OrderStatus current, OrderEvent event) {
        try {
            getNextStatus(current, event);
            return true;
        } catch (IllegalArgumentException e) {
            return false;
        }
    }
}
```

##### 4. 业务订单实体使用

```java
import lombok.Data;

@Data
public class Order {
    private Long orderId;
    private OrderStatus status;

    /**
     * 执行事件，自动变更订单状态
     */
    public void fireEvent(OrderEvent event) {
        // 查表获取新状态
        OrderStatus nextStatus = OrderStateTransition.getNextStatus(this.status, event);
        // 更新状态
        this.status = nextStatus;
        // 可扩展：状态变更日志、回调、业务动作
        System.out.println("订单状态变更为：" + nextStatus);
    }
}
```

#### 五、核心优势详解

1. **规则中心化**
所有流转全部写在 `OrderStateTransition` 枚举里，产品、测试、开发统一看这一张表。

2. **彻底消灭臃肿if-else**
状态再多、分支再多，只新增枚举常量，不用改业务判断逻辑。

3. **天然防脏数据**
不在枚举里定义的流转一律禁止，从代码层面杜绝乱改状态。

4. **易于扩展附加属性**
可自由加字段：
- 流转权限角色
- 前置校验标识
- 超时自动流转
- 操作渠道

5. **便于生成文档/流程图**
直接遍历枚举所有值，就能导出完整业务流程。

```java
    if (currentStatus == OrderStatus.FINISHED) {

    }
    // 查表逻辑...
}
```

##### 2. 绑定业务执行动作

给枚举加业务执行接口：
```java
@FunctionalInterface
public interface TransitionAction {
    void execute(Order order);
}
```
枚举中注入对应动作，流转时自动执行业务逻辑。

##### 3. 状态回退、逆向流程

直接在状态表补充逆向流转枚举即可。

#### 七、适用 & 不适用场景

##### 适合

- 订单、支付、售后、工单、审批流
- 流程固定、流转规则明确
- 中小体量状态机（状态数 20 个以内最优）

##### 不适合

- 超复杂分布式状态、带延时/定时流转
- 流程动态配置（需要数据库配置流转规则）
这类场景改用 **Spring Statemachine / 数据库配置状态流**

#### 八、总结口诀

状态分两类，状态+事件；
流转写成表，全部放枚举；
遇事查表格，合法就跳转；
无配直接抛，脏流全拦住；
业务最简单，企业最常用。

### 方案 3：状态模式（面向对象，适合复杂逻辑）

每个状态**独立成一个类**，处理自己的行为，符合**开闭原则**。

结构：
```
State 接口
 ├── ToPayState
 ├── PaidState
 ├── CancelledState
```

调用：
```java
OrderContext context = new OrderContext(OrderState.TO_PAY);
context.handleEvent(OrderEvent.PAY); // 自动切换到 PAID
```

**优点**：逻辑解耦，适合复杂业务
**缺点**：类会变多



#### 一、核心思想

**状态模式**：把**每一种状态单独抽成独立类**，状态内部封装自身可执行的行为，上下文统一调度。
- 核心：**行为随状态改变而改变**
- 职责拆分：
  1. **抽象状态角色**：定义所有状态统一行为接口
  2. **具体状态角色**：每个状态实现自己能处理的事件、并负责切换到下一个状态
  3. **上下文角色**：持有当前状态、对外暴露统一调用入口，统一管理状态切换
- 优势：完全符合**开闭原则**，新增状态只加新类，不改原有代码；复杂业务逻辑彻底解耦

#### 二、整体类结构

```
1. OrderState 抽象状态接口（统一行为）
├─ WaitPayState 待付款状态
├─ PaidState    已付款状态
├─ ShippedState 已发货状态
├─ FinishedState 已完成状态
└─ CancelledState 已取消状态

2. OrderContext 上下文（持有当前状态，对外调用）
3. 事件枚举 OrderEvent
```

#### 三、完整代码实现

##### 1. 定义事件枚举

```java
public enum OrderEvent {
    PAY,        // 付款
    SHIP,       // 发货
    CONFIRM,    // 确认收货
    CANCEL,     // 取消订单
    REFUND      // 退款
}
```

##### 2. 抽象状态接口（核心规范）

所有状态必须实现该接口，处理对应事件
```java
public interface OrderState {
    /**
     * 处理事件，由当前状态决定跳转至哪个新状态
     * @param context 上下文
     * @param event 触发事件
     */
    void handleEvent(OrderContext context, OrderEvent event);

    /** 获取当前状态名称 */
    String getStateName();
}
```

##### 3. 上下文类（统一入口、持有状态）

```java
public class OrderContext {
    // 持有当前状态
    private OrderState currentState;

    // 初始化默认状态：待付款
    public OrderContext() {
        this.currentState = new WaitPayState();
    }

    // 切换状态
    public void switchState(OrderState state) {
        this.currentState = state;
        System.out.println("订单状态切换为：" + state.getStateName());
    }

    // 对外统一触发事件
    public void fireEvent(OrderEvent event) {
        currentState.handleEvent(this, event);
    }

    // 获取当前状态
    public String getCurrentState() {
        return currentState.getStateName();
    }
}
```

##### 4. 具体状态实现（逐个编写）

###### 4.1 待付款状态

```java
public class WaitPayState implements OrderState {
    @Override
    public void handleEvent(OrderContext context, OrderEvent event) {
        switch (event) {
            case PAY:
                // 付款 → 切换已付款
                context.switchState(new PaidState());
                break;
            case CANCEL:
                // 取消 → 切换已取消
                context.switchState(new CancelledState());
                break;
            default:
                throw new RuntimeException("待付款状态不支持该操作：" + event);
        }
    }

    @Override
    public String getStateName() {
        return "待付款";
    }
}
```

###### 4.2 已付款状态

```java
public class PaidState implements OrderState {
    @Override
    public void handleEvent(OrderContext context, OrderEvent event) {
        switch (event) {
            case SHIP:
                context.switchState(new ShippedState());
                break;
            case REFUND:
                context.switchState(new RefundedState());
                break;
            default:
                throw new RuntimeException("已付款状态不支持该操作：" + event);
        }
    }

    @Override
    public String getStateName() {
        return "已付款";
    }
}
```

###### 4.3 已发货状态

```java
public class ShippedState implements OrderState {
    @Override
    public void handleEvent(OrderContext context, OrderEvent event) {
        if (event == OrderEvent.CONFIRM) {
            context.switchState(new FinishedState());
        } else {
            throw new RuntimeException("已发货仅支持确认收货");
        }
    }

    @Override
    public String getStateName() {
        return "已发货";
    }
}
```

###### 4.4 已完成状态

```java
public class FinishedState implements OrderState {

        throw new RuntimeException("订单已完成，禁止任何操作");

    @Override
    public String getStateName() {
        return "已完成";
    }
}
```

#### 四、状态模式核心优缺点

##### 优点

1. **彻底解耦**：每个状态逻辑独立，互不干扰
2. **开闭原则极强**：新增业务状态，只新建`XXXState`类，原有代码不用改
3. **消除大量臃肿if-else/switch**，代码可读性极高
4. 状态内部可以**封装专属业务逻辑**（比如待付款超时自动取消、付款后发消息等）
5. 职责单一，易于单元测试

##### 缺点

1. **状态多则类爆炸**：状态越多，创建的具体状态类越多
2. 结构比枚举状态表更重，轻量小流程略显繁琐
3. 状态之间跳转逻辑分散在各自状态类中，全局流转规则不够集中查看

#### 五、和「枚举状态表方案2」对比

| 对比项   | 枚举状态表                   | 状态模式                             |
| -------- | ---------------------------- | ------------------------------------ |
| 代码体量 | 轻量、少类                   | 类多、结构重                         |
| 流转规则 | 集中在一张表，一目了然       | 分散在各个状态类                     |
| 业务逻辑 | 逻辑写在工具类/业务层        | 逻辑内嵌在状态类中                   |
| 扩展状态 | 加枚举常量即可               | 新建独立状态类                       |
| 适用场景 | 流程固定、状态不多、快速开发 | 状态行为复杂、每个状态有独立业务动作 |
| 维护难度 | 简单                         | 复杂大型流程更易维护                 |

#### 六、适用场景

1. **每个状态都有独立业务行为**，不只是单纯状态跳转
2. 订单、售后、审批、工单等**流程复杂、后续频繁新增状态**的业务
3. 需要给不同状态绑定**前置校验、后置通知、定时任务**等专属逻辑
4. 业务规则经常迭代，需要频繁新增状态分支

#### 七、进阶优化写法

##### 1. 状态类单例优化

状态对象无成员变量，全局只用一个实例，避免频繁new
```java
public class WaitPayState implements OrderState {
    // 单例
    public static final WaitPayState INSTANCE = new WaitPayState();
    private WaitPayState(){}
    // ...
}
```
使用时直接：`context.switchState(WaitPayState.INSTANCE)`

##### 2. 统一状态工厂

通过枚举快速获取对应状态实例，简化调用
##### 3. 状态变更监听

在`switchState`方法中加入事件监听、日志记录、消息推送

#### 八、面试口述精简版

状态模式是行为型设计模式，核心是**将不同状态封装为独立类**，把状态对应的行为放到状态类中，上下文只负责持有当前状态并对外提供统一调用入口。
调用时触发事件，由**当前状态类自行决定下一步跳转状态**，从而实现行为随状态自动变化。
优点是解耦强、符合开闭，适合复杂多状态业务；缺点是会产生大量状态子类，轻量流程一般用枚举状态表代替。

### 方案 4：使用框架（Spring Statemachine，企业级）

Spring 官方状态机框架，适合**大型项目、分布式、持久化状态**。

核心依赖：
```xml
<dependency>
    <groupId>org.springframework.statemachine</groupId>
    <artifactId>spring-statemachine-core</artifactId>
</dependency>
```

配置式定义状态流转，支持：
- 状态持久化

#### 一、Spring Statemachine 是什么？

Spring 官方出品的**分布式、可配置、可扩展、可持久化**的状态机框架。

**企业为什么要用它？**
- 状态流转**集中配置**，不用写一堆 if-else
- 支持**事务、监听、拦截、异常、状态持久化**
- 适合**订单、支付、审批、工单、物流**等复杂业务
- 分布式环境下安全控制状态流转

**核心三要素**
1. **状态（State）**：订单状态
2. **事件（Event）**：触发动作
3. **配置（Config）**：状态 + 事件 → 目标状态

#### 三、定义 状态 + 事件

##### 1. 订单状态

```java
public enum OrderState {
    WAIT_PAY,    // 待支付
    PAID,        // 已支付
    SHIPPED,     // 已发货
    FINISHED,    // 已完成
    CANCELLED    // 已取消
}
```

##### 2. 订单事件

```java
public enum OrderEvent {
    PAY,         // 支付
    SHIP,        // 发货
    CONFIRM,     // 确认收货
    CANCEL       // 取消
}
```

#### 四、核心：状态机配置（最关键）

```java
@Configuration
@EnableStateMachine
public class OrderStateMachineConfig extends StateMachineConfigurerAdapter<OrderState, OrderEvent> {

    /**
     * 配置所有状态
     */
    @Override
    public void configure(StateMachineStateConfigurer<OrderState, OrderEvent> states) throws Exception {
        states
            .withStates()
            .initial(OrderState.WAIT_PAY)  // 初始状态
            .state(OrderState.PAID)
            .state(OrderState.SHIPPED)
            .state(OrderState.FINISHED)
            .state(OrderState.CANCELLED);
    }

    /**
     * 配置状态流转（核心！）
     */
    @Override
    public void configure(StateMachineTransitionConfigurer<OrderState, OrderEvent> transitions) throws Exception {
        transitions
            // 待支付 → 支付 → 已支付
            .withExternal()
            .source(OrderState.WAIT_PAY).target(OrderState.PAID).event(OrderEvent.PAY)
            
            .and()
            // 待支付 → 取消 → 已取消
            .withExternal()
            .source(OrderState.WAIT_PAY).target(OrderState.CANCELLED).event(OrderEvent.CANCEL)
            
            .and()
            // 已支付 → 发货 → 已发货
            .withExternal()
            .source(OrderState.PAID).target(OrderState.SHIPPED).event(OrderEvent.SHIP)
            
            .and()
            // 已发货 → 确认收货 → 已完成
            .withExternal()
            .source(OrderState.SHIPPED).target(OrderState.FINISHED).event(OrderEvent.CONFIRM);
    }
}
```

#### 五、状态监听器（业务处理核心）

状态切换时**自动执行业务逻辑**：记录日志、发送消息、更新数据库…

```java
@Component
public class OrderStateListener {

    @WithStateMachine
    public void onTransition(StateContext<OrderState, OrderEvent> context) {
        OrderState source = context.getSource().getId();
        OrderState target = context.getTarget().getId();
        OrderEvent event = context.getEvent();

        System.out.println("状态切换：" + source + " → " + target + "，触发事件：" + event);
    }

    // 监听支付成功
    @OnStateChanged(target = "PAID")
    public void onPaySuccess() {
        System.out.println("【业务动作】支付成功，准备发货");
    }

    // 监听发货成功
    @OnStateChanged(target = "SHIPPED")
    public void onShip() {
        System.out.println("【业务动作】商家已发货，通知用户");
    }
}
```

#### 六、使用状态机（Service 层）

```java
@Service
public class OrderService {

    @Autowired
    private StateMachine<OrderState, OrderEvent> stateMachine;

    // 触发事件
    public boolean sendEvent(OrderEvent event) {
        return stateMachine.sendEvent(event);
    }

    // 获取当前状态
    public OrderState getCurrentState() {
        return stateMachine.getState().getId();
    }
}
```

#### 八、企业级核心功能（必须掌握）

##### 1. 状态持久化（存数据库）

生产环境必须用，否则重启服务状态丢失。

实现方式：
- 实现 `StateMachinePersister`
- 把状态存到 MySQL / Redis

##### 2. 状态流转校验

```java
@Bean
public StateMachineInterceptor<OrderState, OrderEvent> interceptor() {
    return new StateMachineInterceptor<>() {
        @Override
        public boolean preEvent(StateContext<OrderState, OrderEvent> context) {
            // 前置校验：比如订单已取消不能再支付
            return true; // true 允许流转
        }
    };
}
```

##### 3. 分布式锁

防止并发操作导致状态错乱。

##### 4. 状态机工厂

多订单、多实例，每个订单一个独立状态机。

#### 十、和前三种方案对比

| 方案              | 优点                               | 缺点           | 适用             |
| ----------------- | ---------------------------------- | -------------- | ---------------- |
| if/else           | 简单                               | 乱、难维护     | 小功能           |
| 枚举状态表        | 轻量、清晰                         | 不支持复杂监听 | 中小流程         |
| 状态模式          | 面向对象、解耦                     | 类太多         | 复杂状态         |
| **Spring 状态机** | **企业级、稳定、可持久化、分布式** | 重、学习成本高 | **生产项目首选** |

## 三、使用场景（高频业务场景）
### 1. 订单流程（最经典）
**状态**：待支付 → 已支付 → 待发货 → 已发货 → 完成 / 退款
**事件**：支付、发货、确认收货、申请退款

FSM 能**严格控制流转**，防止非法状态（比如已完成不能再发货）。

---

### 2. 支付流程
状态：创建 → 待支付 → 支付成功 → 支付失败
控制：只有待支付才能执行支付操作。

---

### 3. 审批流程
员工提交 → 主管审批 → 经理审批 → 通过 / 驳回
FSM 天然适合**线性 + 分支流程**。

---

### 4. 任务/工单系统
待处理 → 处理中 → 已解决 → 已关闭
防止重复操作、状态错乱。

---

### 5. 游戏角色状态
待机 → 行走 → 攻击 → 死亡
严格控制动作切换，避免逻辑混乱。

---

### 6. 网络连接状态
未连接 → 连接中 → 已连接 → 断开
适合 TCP、WebSocket 等连接管理。

---

### 7. 工作流、电商履约、物流
物流：已下单 → 已揽收 → 运输中 → 派送中 → 已签收

---

## 四、什么时候必须用 FSM？
满足下面任意一点，就该用 FSM：
1. **对象有明确的多状态**
2. **状态流转有严格规则**
3. **防止非法状态跳转**
4. **业务流程复杂，if-else 难以维护**
5. **需要统一管理状态变更日志**

---

## 五、FSM 设计最佳实践
1. **状态和事件都用枚举**
2. **统一维护状态转移表**
3. **禁止非法状态流转**
4. **状态切换时加日志/监控**
5. **分布式场景下状态要加锁/事务**
6. **复杂流程用 Spring Statemachine**
