####  一、流量治理 具体指什么

微服务中，**对请求流量进行管控、调度、防护**，避免突发流量打垮系统。

你项目里实际落地的点：

1. 限流

   接口限流、单机限流，防止秒杀 / 点餐高峰期大量请求瞬间压垮数据库、服务。

2. 熔断

   下游服务（如订单、支付）卡顿 / 超时，直接熔断，不再无限调用，避免连锁雪崩。

3. 降级

   流量高峰时，关闭非核心功能（如菜品推荐、评价），优先保证下单、支付核心链路。

4. 网关路由、黑白名单

   网关层统一路由转发、拦截非法请求、统一鉴权，统一管控入口流量。

> 一句话口语：
>
> 流量治理就是**针对高并发场景，做限流、熔断、路由管控，保护系统不被突发流量打垮**。

#### 二、服务容错 具体指什么

微服务调用链条长，极易出现超时、宕机、调用失败；

**服务容错 = 保证服务调用失败后，系统不崩溃、业务可用**。

你项目落地：

1. 超时控制

   远程调用设置超时时间，避免接口卡死。

2. 重试机制

   临时网络抖动、偶尔调用失败，自动重试一次，提高成功率。

3. 降级兜底

   依赖服务异常时，返回兜底数据、友好提示，不让页面报错 500。

4. 分布式事务兜底

   订单、库存、扣减，异常时回滚 / 补偿，保证数据一致。

5. 资源隔离

   不同业务线程池隔离，一个业务炸了不影响其他模块。

> 一句话口语：
>
> 服务容错就是**解决服务调用超时、失败、宕机问题，通过超时、重试、降级、隔离，保证系统稳定不雪崩**。

#### 四、面试官可能追问的「高频问题+应答思路」

提前准备好这些，就不怕面试官深挖了：

1.  **为什么用 Sa-Token 不用 Spring Security？**
    答：Sa-Token 上手成本更低，API 更简洁，对分布式会话、权限控制的支持更开箱即用，适合快速开发；而且和 Redis 集成方便，网关层鉴权更简单，减少了重复配置。
2.  **用户上下文透传的具体实现？**
    答：网关层从 Token 中解析出用户信息，存入 ThreadLocal，同时通过 Feign 拦截器，把用户信息放到请求头里，下游服务从请求头中取出再存入 ThreadLocal，实现全链路透传。
3.  **CompletableFuture 优化的场景和效果？**
    答：比如查询题目列表时，需要同时查分类、标签、用户点赞状态等信息，用 CompletableFuture 并行查询，相比串行查询，接口响应时间大幅缩短，性能提升约80%。
4.  **Redis + RabbitMQ 解决点赞一致性的思路？**
    答：点赞请求先写 Redis，再通过 RabbitMQ 异步同步到 DB，同时做了消息确认机制和重试机制，避免数据丢失；定时任务会做 Redis 和 DB 的数据对账，保证最终一致性。

  - 性能达到 O(1) 级别，支撑高并发抢券

##### 3. 座位管理：状态机 + 乐观锁 + 定时任务（业务难点）

  使用 **SQL条件更新（WHERE 库存>0）** 实现乐观锁，无锁化并发控制，性能远超分布式锁。
- **僵尸锁座自动清理**：
  用 XXL-Job 定时扫描超过15分钟未支付的锁座，自动释放座位、恢复库存，保证资产高效流转。

##### 1. 你们项目的分布式事务是怎么解决的？

我们项目是分布式微服务架构，支付、订单、库存、用户余额是独立服务。
我采用了两种方案：

2. **超时关单**：使用 **RocketMQ 延迟队列**，超时未支付自动回滚，保证最终一致性。
既保证了数据安全，又避免了强一致性带来的性能损耗。

##### 2. 秒杀如何保证高并发和不超卖？

秒杀核心是 **Redis+Lua原子脚本**，把判断库存和扣库存做成原子操作，避免并发问题；
同时做了**库存预热**到Redis，减轻数据库压力；
使用 **Sentinel限流降级**，防止压垮服务；
数据库层面用**乐观锁+条件更新**做最终兜底，双重保证**零超卖**。

##### 3. 座位锁座怎么防止超卖？

1. 座位用**有限状态机**，只能按固定流程流转；
2. 锁座使用 **SQL条件更新**：`update seat set status=1 where id=? and status=0`
3. 数据库乐观锁保证并发安全；
4. 定时任务清理超时未支付的锁座，释放资源。

2. 熟练运用大语言模型做业务方案设计，精通Trae、Claude Code、Codex等AI编程工具，研发效率高；
3. 做事严谨负责，注重代码质量与线上稳定性，善于沟通协作，执行力强；
4. 懂得理性使用AI工具，能甄别工具生成代码优劣，自主完成逻辑校验与优化。

**缺点**
1. 目前更多偏向大模型应用层开发，大模型底层微调、训练实战经验不足；
2. 运用AI工具时偶尔会依赖快速生成代码，正在刻意加强底层手写编码能力，平衡效率与基本功；
3. 大规模AI智能系统整体架构统筹经验还需进一步积累。

第三看重团队氛围与带教氛围，希望同事乐于交流技术，共同精进；
第四看重业务前景与薪资匹配度，希望平台长期发展稳定，薪资与个人能力、付出对等，能够长期稳定深耕发展。
