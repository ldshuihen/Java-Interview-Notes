####  一、流量治理 具体指什么

微服务中，**对请求流量进行管控、调度、防护**，避免突发流量打垮系统。

你项目里实际落地的点：

1. 限流

   接口限流、单机限流，防止秒杀 / 点餐高峰期大量请求瞬间压垮数据库、服务。

2. 熔断

   下游服务（如订单、支付）卡顿 / 超时，直接熔断，不再无限调用，避免连锁雪崩。

3. 降级

   流量高峰时，关闭非核心功能（如菜品推荐、评价），优先保证下单、支付核心链路。

4. 网关路由、黑白名单

   网关层统一路由转发、拦截非法请求、统一鉴权，统一管控入口流量。

> 一句话口语：
>
> 流量治理就是**针对高并发场景，做限流、熔断、路由管控，保护系统不被突发流量打垮**。

#### 二、服务容错 具体指什么

微服务调用链条长，极易出现超时、宕机、调用失败；

**服务容错 = 保证服务调用失败后，系统不崩溃、业务可用**。

你项目落地：

1. 超时控制

   远程调用设置超时时间，避免接口卡死。

2. 重试机制

   临时网络抖动、偶尔调用失败，自动重试一次，提高成功率。

3. 降级兜底

   依赖服务异常时，返回兜底数据、友好提示，不让页面报错 500。

4. 分布式事务兜底

   订单、库存、扣减，异常时回滚 / 补偿，保证数据一致。

5. 资源隔离

   不同业务线程池隔离，一个业务炸了不影响其他模块。

> 一句话口语：
>
> 服务容错就是**解决服务调用超时、失败、宕机问题，通过超时、重试、降级、隔离，保证系统稳定不雪崩**。

#### 四、面试官可能追问的「高频问题+应答思路」

提前准备好这些，就不怕面试官深挖了：

1.  **为什么用 Sa-Token 不用 Spring Security？**
    答：Sa-Token 上手成本更低，API 更简洁，对分布式会话、权限控制的支持更开箱即用，适合快速开发；而且和 Redis 集成方便，网关层鉴权更简单，减少了重复配置。
2.  **用户上下文透传的具体实现？**
    答：网关层从 Token 中解析出用户信息，存入 ThreadLocal，同时通过 Feign 拦截器，把用户信息放到请求头里，下游服务从请求头中取出再存入 ThreadLocal，实现全链路透传。
3.  **CompletableFuture 优化的场景和效果？**
    答：比如查询题目列表时，需要同时查分类、标签、用户点赞状态等信息，用 CompletableFuture 并行查询，相比串行查询，接口响应时间大幅缩短，性能提升约80%。
4.  **Redis + RabbitMQ 解决点赞一致性的思路？**
    答：点赞请求先写 Redis，再通过 RabbitMQ 异步同步到 DB，同时做了消息确认机制和重试机制，避免数据丢失；定时任务会做 Redis 和 DB 的数据对账，保证最终一致性。

- **多渠道支付统一**：使用**策略模式**封装支付宝、微信、余额三种支付方式，对外提供统一接口，新增支付方式无需修改原有代码，符合开闭原则。
- **全链路幂等设计**：支付、订单、退款全部做**防重**：

  - 数据库唯一索引 + 状态机判断

  2. 若出票失败，自动触发补偿回滚
  3. 延迟队列监听未支付订单，**超时自动关单、释放库存、回滚余额**
  4. 保证**余额扣减、订单创建、出票**三者最终一致。
- **复杂退款编排**：支持原路退款、积分扣回、优惠券原路返还，链式事务补偿，保证资金和资产安全。

##### 2. 秒杀链路：Redis + Lua 高并发零超卖（高并发重点）

- 秒杀核心采用 **Redis + Lua 原子脚本**：
  - 把**库存判断 + 扣减**放在一个Lua脚本执行，保证原子性
  - 性能达到 O(1) 级别，支撑高并发抢券
- **库存惰性预热**：活动开始前异步把库存加载到Redis，减轻DB压力
- **Sentinel动态流量防护**：限流、降级、熔断，防止流量打垮服务
- **双模式优惠券**：定时生效+立即生效，配合状态机管理生命周期
- 最终实现：**高并发下零超卖、无库存错误、服务不崩溃**。

##### 3. 座位管理：状态机 + 乐观锁 + 定时任务（业务难点）

- **有限状态机(FSM)** 严格管控座位状态：
  可选 → 锁定 → 已售 → 退款回收
  禁止非法状态跳转，从业务逻辑上杜绝超卖。
- **高并发锁座方案**：
  使用 **SQL条件更新（WHERE 库存>0）** 实现乐观锁，无锁化并发控制，性能远超分布式锁。
- **僵尸锁座自动清理**：
  用 XXL-Job 定时扫描超过15分钟未支付的锁座，自动释放座位、恢复库存，保证资产高效流转。

##### 4. 评论嵌套性能优化（算法/代码优化亮点）

- 原问题：评论无限嵌套递归查询，时间复杂度 O(N^k)，深层嵌套极慢
- 优化方案：
  1. **单次SQL查询所有数据**
  2. 使用 **Java Stream + groupingBy** 构建父ID哈希索引
  3. 内存O(N)线性组装评论树

##### 1. 你们项目的分布式事务是怎么解决的？

我们项目是分布式微服务架构，支付、订单、库存、用户余额是独立服务。

既保证了数据安全，又避免了强一致性带来的性能损耗。

##### 2. 秒杀如何保证高并发和不超卖？

数据库层面用**乐观锁+条件更新**做最终兜底，双重保证**零超卖**。

##### 3. 座位锁座怎么防止超卖？

1. 座位用**有限状态机**，只能按固定流程流转；
2. 锁座使用 **SQL条件更新**：`update seat set status=1 where id=? and status=0`
3. 数据库乐观锁保证并发安全；
4. 定时任务清理超时未支付的锁座，释放资源。

##### 难点三：批量导入大量文件，接口性能与并发阻塞问题

**问题**

**收获**

1. 这个岗位日常核心工作职责主要是什么？
2. 目前这个岗位面临最主要的工作挑战是什么？
3. 公司对这个岗位**3个月试用期**的考核标准是什么？

### 二、了解**团队氛围&架构**（技术岗首选）

1. 咱们团队目前人员规模、分工模式是怎样的？
2. 团队现在主流技术栈、项目架构是什么？
3. 日常开发流程、代码评审、迭代节奏大概如何？

### 三、了解**成长晋升**（体现上进心）

1. 新人入职后会有导师带教吗？培养体系如何？
2. 这个岗位后续晋升路径和发展方向是怎样的？
3. 公司会提供技术培训、项目轮岗学习机会吗？

---

### 精简万能组合（每次挑2-3个问就行）

#### 通用稳妥版

1. 想了解下这个岗位日常主要负责哪些工作内容？
2. 新人入职会有带教安排吗？

#### 技术岗专业版

1. 团队目前使用的技术栈和项目架构是什么？
2. 该岗位现阶段主要攻坚的业务难点是什么？
3. 岗位后续技术成长方向是怎样规划的？

#### 求职走心版

1. 您认为胜任这份工作最需要具备哪些能力？
2. 团队整体工作氛围和协作模式怎么样？

---

### 避坑千万别问

- 一上来就问**月薪、加班费、年假、请假**
- 质疑公司业务、吐槽行业
- 一次性问五六个，显得急躁
- 避免一下问是否容易被辞，温和问考核要求。
