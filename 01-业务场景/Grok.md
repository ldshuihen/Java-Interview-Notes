### 1. 双11 / 秒杀场景，瞬间10w+ QPS 如何不雪崩？

核心思想：**漏斗式流量收敛 + 多级缓存 + 削峰 + 容量规划 + 防护兜底**

主流分层防护模型（从外到内，越往里QPS越低）：

| 层级            | 手段                             | 预期过滤后QPS     | 备注 / 常见组件                    |
| --------------- | -------------------------------- | ----------------- | ---------------------------------- |
| 浏览器 / App    | 动静分离 + 前端限流 + 按钮置灰   | 原始 → 70–80%     | CDN缓存静态页、倒计时控制          |
| CDN / 边缘节点  | 页面/图片全缓存、WAF拦截异常IP   | → 30–50%          | 阿里云/腾讯云CDN + Bot防护         |
| Nginx / 网关    | limit_req + 全局QPS限流 + 黑名单 | → 10–20万 → 5–8万 | Nginx + OpenResty + Lua限流        |
| Sentinel / 网关 | 集群维度的QPS/线程数/热点限流    | → 3–5万           | 阿里Sentinel、Hystrix/Resilience4j |
| 应用层          | 本地缓存（Caffeine/Guava）命中   | → 几千–几万       | 热点key本地缓存 + 预热             |
| Redis           | 预减库存 + Lua脚本原子扣减       | → 几百–几千       | Redis Cluster + 热点key分片        |
| MQ              | 异步下单、削峰填谷               | → 几百–一千       | RocketMQ / Kafka 高吞吐分区        |
| DB              | 最终只有成功请求落库             | < 几百            | 读写分离 + 批量写                  |

**最常被问的对比点**：
- 先限流还是先缓存？ → 先限流保护系统，再缓存提升效率
- 为什么很多公司最终Redis层只有几百QPS？ → 因为真正有效的请求（有库存+用户合法）极少
- 雪崩的最后一根稻草通常是？ → Redis热点key打穿 → DB，或MQ积压导致级联

### 2. 突发流量10倍、50倍、100倍打上来怎么处理？

分**事前、事中、事后**三个阶段回答：

**事前（容量规划 + 预案）**
- 常态容量压测到多少？（例如日常3k，压到8k–10k）
- 活动预热：缓存预热、机器预扩容、HPA/ESS自动伸缩
- 多活/异地容灾预案、降级开关（非核心接口降级）

**事中（快速止血三板斧）**
1. **限流 / 拒绝**（最快生效，1秒内）  
   → 全局限流 > 接口限流 > IP/用户限流 > 热点参数限流
2. **降级 / 熔断**（保护下游）  
   → 非核心依赖直接降级（推荐、排行榜、凑单）
3. **临时扩容**（K8s HPA / 阿里云ESS，5–15分钟生效）

**事后复盘**
- 流量来源分析（爬虫？羊毛？正常用户？）
- 哪个环节最先顶不住？为什么？
- 容量模型是否需要升级？

**一句话总结**：  
“10倍靠预案+限流，50倍靠降级+临时扩容，100倍靠上帝保佑+事后快速回滚+全链路降级”

| -------- | -------------------------------- | ---------------------------------------------- | ---------------- | -------------------------------- |
| 接入层   | WAF / Bot管理（行为指纹+JS挑战） | UA、指纹、Headless、请求速率、轨迹异常         | 60–80%           | 阿里云WAF、腾讯云Bot、Cloudflare |
| 前端对抗 | 动态Token、指纹采集、行为验证码  | 无JS执行、指纹缺失、轨迹直线、点击间隔均匀     | 70–90%           | 自研或腾讯防水墙、阿里聚安全     |

### 4. 接口防刷 + 限流 + 防重，如何设计多层次方案？

推荐**四层递进**防御模型（2025年主流做法）：

1. **接入层限流**（Nginx / 网关）  
   - 全局QPS、单IP QPS、单用户QPS  
   - 令牌桶 / 漏桶 / 滑动窗口

2. **业务层防刷**（频率 + 行为）  
   - Redis ZSET / HyperLogLog 统计用户/接口/维度频率  
   - 例如：用户维度 1min 10次、商品维度 1min 50次

3. **防重 / 幂等**（防止重复扣款/领券/下单）  
   - 客户端：唯一请求ID（UUID）+ 前端防抖  
   - 服务端：Redis setnx / Lua + 过期（防重5–30min）  
   - 数据库唯一索引 / 防重表（最终兜底）

4. **风控兜底**（动态规则 + 评分）  
   - 设备指纹 + 行为链路 + 风控评分 > 阈值直接拦截

**层次对比**（面试最爱问）：

| 层级      | 速度 | 成本 | 误杀率 | 防护对象          |
| --------- | ---- | ---- | ------ | ----------------- |
| Nginx     | 最快 | 最低 | 中     | 粗颗粒刷          |
| Redis频率 | 快   | 中   | 低–中  | 中颗粒刷          |
| 幂等      | 中   | 中   | 极低   | 重复提交/重试     |
| 风控评分  | 慢   | 高   | 最低   | 智能羊毛/团伙作弊 |

### 5. 直播间万人同时点赞/送礼并发怎么做？

直播场景特点：**读多写多、实时性强、允许少量丢、最终一致优先**

主流架构组合（2025–2026年主流）：

1. **接入 & 限流**  
   - WebSocket / HTTP长轮询 + 网关限流（单直播间QPS限流）

2. **削峰 & 聚合**  
   - 客户端：本地防抖 + 批量上报（每3–5秒上报一次）  
   - 服务端：内存聚合 / Redis INCR + pipeline（1s聚合一次）

3. **存储 & 广播**  
   - 点赞/礼物数 → Redis INCR（最终一致）  
   - 排行榜 → Sorted Set + 定时落地（或使用Dragonfly/Redisson）  
   - 实时推送 → Kafka / RocketMQ Topic（分区按roomId）→ 消费后推WebSocket

4. **高可用 & 一致性**  
   - Redis Cluster + 多副本  
   - 允许短暂不准（延迟<3s），最终靠MQ + DB对账  
   - 礼物金额 → 异步写DB + 幂等 + 补偿任务

- 万人同时送最贵礼物，怎么防止刷屏？
- 排行榜实时性要求<1s，怎么做？
- 怎么防止刷礼物薅平台补贴？

### 1. 秒杀商品超卖怎么彻底解决？（多种方案对比）

核心目标：**不超卖**（库存≥0）、**尽量不少卖**、**高性能**（QPS万级~数十万）

| 方案                   | 核心技术                  | 是否彻底防超卖 | 性能（QPS）  | 是否防少卖   | 一致性保障            | 适用场景            | 主流公司典型用法            | 主要缺点                        |
| ---------------------- | ------------------------- | -------------- | ------------ | ------------ | --------------------- | ------------------- | --------------------------- | ------------------------------- |

| Redis分布式锁 + DB     | Redisson / 自研锁 + 事务  | 是             | 几千~万      | 是           | 最终一致（补偿）      | 中等并发秒杀        | 部分中小厂                  | 锁竞争严重，网络抖动易超时      |

### 2. 12306 余票“区间库存”怎么扣不超卖？

12306最经典的难点：**动态区间库存**（A→D 买票会扣 A→B、B→C、C→D 三个原子区间），不是简单一个商品库存。

主流理解的几种方案（非官方，仅业界公开分析与猜测，2024–2026年主流说法）：

1. **读扩散（主流猜测）**  
   - 存储方式：只存**原子区间**库存（相邻两站之间，如A-B、B-C、C-D）  
   - 扣减：用户买A→D，只扣三个原子区间各自-1（用分布式锁或Lua批量原子扣）  
   - 查询余票：动态计算路径所有原子区间的最小值（min）作为余票  
   → 优点：扣减简单、原子性好  
   → 缺点：查询复杂（需遍历路径所有区间）

2. **写扩散（较少用）**  
   - 每卖一张A→D，同时写扩散扣减所有受影响的组合区间库存（A-C、A-D、B-D等）  
   → 优点：查询极快（直接读）  
   → 缺点：扣减时写放大，性能差

3. **混合 / 预分配 + Buffer**（12306风格高可用思路）  
   - Redis存统一原子区间库存  
   - 每台机器本地内存预分配一部分票 + Buffer冗余（防单机/Redis抖动）  
   - 扣票流程：先本地内存减 → 成功再远程Redis原子减（Lua）  
   - 宕机：Buffer兜底，少量可少卖但不超卖  
   → 核心：**本地内存 + Redis双保险 + Buffer容灾**

4. **分桶 / 热点分片**（2025–2026年优化方向）  
   - 热门区间用Redis分多个桶（hash tag分片）  
   - 扣减时随机/轮询桶扣减 → 降低单key热点

一句话总结12306风格：**“原子区间 + 读扩散 + Redis/Lua原子扣 + 本地Buffer + 运营预留区间票”**，彻底防超卖，接受极少量少卖（远途优先）。

### 3. 下单减库存、支付减库存、取消加库存，怎么保证最终一致？

电商最常见的三种库存阶段 + 一致性诉求：

| 阶段      | 操作        | 主流时机选择                 | 一致性挑战                  | 保障最终一致主流方案                                         |
| --------- | ----------- | ---------------------------- | --------------------------- | ------------------------------------------------------------ |
| 下单      | 预占/减库存 | 主流：下单时减（占坑）       | 用户不支付 → 库存被长期占用 | 订单超时自动回补（定时任务/MQ延时消息）                      |
| 支付成功  | 真减/确认   | 极少直接支付减（用户体验差） | —                           | —                                                            |
| 取消/退款 | 加回库存    | 主动取消 / 超时 / 退款成功   | 回补失败 → 少卖             | 1. MQ异步可靠回补<br>2. 库存补偿任务（扫库存流水）<br>3. 对账job（日/小时级） |

**主流组合方案**（2025–2026年）：

1. **下单减库存（主流） + 超时回补**  
   - 下单成功 → Redis/MySQL减库存 → 生成待支付订单  
   - 支付成功 → 无需再动库存（已减）  
   - 超时/取消 → MQ发延时消息 / 定时任务回补库存  
   → 优点：简单，用户看到库存准  
   → 缺点：恶意下单占库存（防刷 + 风控）

2. **下单预占 + 支付真减**（部分票务/预约场景）  
   - 下单 → 预占（Redis setnx + 过期）  
   - 支付成功 → 真减（异步写DB）  
   - 超时 → 自动过期释放  
   → 适合选座/预约类

3. **最终一致核心三板斧**：
   - **幂等**：库存流水表（orderId + 操作类型唯一）防重
   - **补偿**：定时扫描“已支付未真减”“已取消未回补”的订单
   - **对账**：T+1对账job，比对订单流水 vs 库存变化

### 4. 分布式高并发场景下库存扣减方案对比（锁、Lua、消息、预减等）

再精简对比（高并发视角）：

| 方案                 | 原子性保障      | QPS上限      | 网络开销 | 复杂度 | 少卖风险 | 推荐场景                   |
| -------------------- | --------------- | ------------ | -------- | ------ | -------- | -------------------------- |
| 分布式锁（Redisson） | 锁内操作        | 中（万级）   | 高       | 中     | 低       | 中并发 + 复杂逻辑          |
| Redis Lua脚本        | 脚本整体原子    | 高（10w+）   | 低       | 低     | 中       | 纯扣减（主流秒杀首选）     |
| 预减 + MQ异步落库    | 预减原子 + 异步 | 极高（20w+） | 中       | 高     | 中–高    | 双11/极致秒杀              |
| 本地缓存分段预减     | 本地原子 + 补   | 极高         | 中       | 高     | 低       | 超高并发（如12306借鉴）    |
| MQ削峰 + 串行扣      | MQ顺序消费      | 低–中        | 高       | 中     | 低       | 对顺序强依赖（如严格排队） |

**一句话总结大厂选择逻辑**（2025–2026）：
- 普通商品 → 数据库乐观锁
- 中等秒杀 → Redis Lua原子扣
- 双11/百万QPS → Redis预减 + MQ异步 + 强补偿对账
- 极致（如12306） → 本地内存预分配 + Redis统一补 + Buffer

### 1. 用户手抖重复点击下单/支付，怎么防？

核心：**前端防抖 + Token机制 + 服务端多层校验**（防用户快速连点、F5刷新、网络重试）

主流分层方案对比（2025–2026年主流）：

| 层级        | 方案                             | 拦截时机    | 误杀率 | 实现复杂度 | 大厂典型用法         | 缺点/局限性                 |
| ----------- | -------------------------------- | ----------- | ------ | ---------- | -------------------- | --------------------------- |
| 前端        | 按钮置灰 + 防抖/节流（lodash）   | 点击瞬间    | 低     | 低         | 几乎所有App/H5       | 刷新页面、后退前进失效      |
| 前端        | 一次性Token（页面加载时获取）    | 提交前      | 低     | 中         | 京东/拼多多/美团外卖 | Token泄露风险小，需防重放   |
| 服务端-网关 | 请求唯一ID（UUID） + Redis setnx | 网关/拦截器 | 中     | 中         | 字节/腾讯部分接口    | 全局开销大，适合粗颗粒      |
| 服务端-业务 | 防重Token（推荐）                | AOP/注解    | 极低   | 中–高      | 阿里/美团/京东主流   | 需要提前发Token，流程稍复杂 |
| 服务端-兜底 | 数据库唯一索引 + 状态机校验      | 事务内      | 极低   | 低         | 所有公司最终防线     | 性能瓶颈，适合最终一致      |

**最推荐组合（2025主流）**：
1. 前端：按钮禁用 + 防抖 500ms–1s + loading遮罩
2. 页面加载/进入支付页 → 调用`/api/token`接口获取一次性Token（Redis setnx + 5–30min过期）
3. 提交时携带Token → 服务端AOP/拦截器校验：Redis get + del（原子Lua）
4. 业务层：订单号/用户+商品维度做唯一索引 + 状态机（待支付 → 已支付）
5. 兜底：支付成功回调时用乐观锁 `update set status=PAID where status=UNPAID`

**常问追问**：
- Token怎么防重放攻击？（绑定用户ID+IP+设备指纹+短过期）
- 高并发下Token生成压力大？（Redis pipeline + 本地缓存预热）

### 2. 支付回调幂等如何设计？网络抖动重复回调怎么办？

支付回调（微信/支付宝）典型特点：**异步通知、可能无限重试、必须返回特定字符串（如success）否则不停重发**

核心原则：**先查后写 + 唯一流水号 + 只处理一次**

主流设计（微信/支付宝官方推荐 + 大厂实践）：

1. **幂等键**：第三方流水号（微信transaction_id / 支付宝trade_no）+ 商户订单号 组合（唯一索引）
2. **流程**（伪代码风格）：
   - 接收回调 → 验签 → 解析参数
   - 用第三方流水号查`payment_callback_record`表（或Redis）
     - 已存在 → 直接返回`success`（阻止第三方继续重发）
     - 不存在 → 继续
   - 检查订单状态（乐观锁方式）：
     - `update orders set status='PAID', version=version+1 where order_no=? and status='UNPAID' and version=?`
     - 影响行=1 → 执行业务（发货、加积分、通知商户）
     - 影响行=0 → 说明已被其他回调处理过，返回`success`
   - 插入回调记录（防重表），返回`success`

**多层保障**：
- Redis快速判断（毫秒级） → 防高频重试
- 数据库唯一索引兜底（防Redis失效）
- 状态机 + 乐观锁（防并发回调）
- 补偿任务：定时扫描“支付成功但未发货”的订单（T+1对账）

**微信/支付宝特有处理**：
- 必须返回`<xml><return_code><![CDATA[SUCCESS]]></return_code></xml>`或`success`
- 失败返回fail/其他 → 平台会按间隔重试（最长48小时或更多）

**常问追问**：
- 回调网络抖动导致返回失败？ → 第三方会重试，幂等设计天然兼容
- 怎么防止“假成功”？ → 回调里查单 + 对账

### 3. 消息消费幂等最推荐的几种做法？

MQ（RocketMQ/Kafka）重复消费不可避免（At-Least-Once），必须**业务层幂等**

2025–2026年大厂最推荐的4种做法（优先级从高到低）：

| 优先级 | 做法                            | 核心手段                                                     | 适用场景                   | 优点               | 缺点/注意点            | 大厂主流度 |
| ------ | ------------------------------- | ------------------------------------------------------------ | -------------------------- | ------------------ | ---------------------- | ---------- |
| 1      | 数据库唯一约束/防重表           | 订单号/业务ID + 唯一索引                                     | 插入/状态变更              | 最可靠、最终兜底   | 性能瓶颈（高并发写DB） | ★★★★★      |
| 2      | 业务状态机判断                  | if (status == "待处理") → 处理并改状态                       | 状态流转（支付→发货→完成） | 业务友好、无额外表 | 高并发下需结合乐观锁   | ★★★★☆      |
| 3      | Redis去重（setnx / set + 过期） | 业务唯一ID作为key，setnx成功才消费                           | 临时去重、短时效           | 高性能、简单       | Redis宕机/过期可能漏   | ★★★★       |
| 4      | 消费流水表 + 状态               | 插入消费记录（msgId + 业务ID + status=PROCESSING → SUCCESS） | 复杂场景、需补偿           | 支持重试控制、补偿 | 表设计复杂、写放大     | ★★★        |

**最推荐组合**（阿里/美团主流）：
- **业务唯一ID**（订单号/流水号）作为幂等键（**不要只用msgId**，因为重投内容相同msgId不同）
- 先Redis setnx（快速判断） → 成功则消费 + 设过期（防内存爆炸）
- 消费中插入/更新业务表（唯一索引/状态机）
- 消费成功后更新Redis状态或插入防重表
- 补偿Job：扫描“PROCESSING”超时的记录，重试或告警

**RocketMQ/Kafka额外Tips**：
- RocketMQ：用msg Key携带业务唯一ID
- Kafka：幂等生产者 + 业务层去重（exactly-once需事务）

### 4. 接口重复调用、超时重试、MQ重投，怎么统一做幂等？

目标：**统一幂等框架**，减少每个接口重复造轮子

主流统一方案（2025–2026年大厂趋势）：

1. **注解 + AOP + Redis/Token**（最流行）
   - 自定义`@Idempotent`注解（timeout、keyPrefix、keySpEL）
   - AOP切面：解析SpEL生成key → Redis setnx + del
   - 支持：接口超时重试、MQ重投、前端重复提交
   - 示例key：`idempotent:user:{userId}:order:create:{goodsId}`

2. **统一幂等中间件**（字节/阿里内部风格）
   - 网关/拦截器统一生成requestId
   - 业务方法加`@Idempotent` → 自动查Redis/数据库去重
   - 支持降级：Redis失效走数据库唯一索引

3. **多层组合拳**（生产环境最稳）
   - 网关层：requestId + 短时限流
   - 业务层：@Idempotent（Redis + Lua原子）
   - 数据层：唯一索引 + 状态机 + 乐观锁
   - MQ层：业务唯一ID + 防重表

**统一原则**（面试高分回答）：
- 优先业务唯一标识（订单号 > requestId > msgId）
- 多层防御：前端 → 网关 → 业务 → 数据
- 最终一致 + 补偿机制（定时对账/补偿Job）
- 监控告警：幂等命中率、重复请求量

### 1. Redis缓存与MySQL双写一致性怎么保证？

**核心结论**：**强一致几乎不可能**（CAP），大厂全部接受**最终一致**，追求“读尽量准、写不丢”。

主流双写一致性方案对比（2025–2026年实际使用比例）：

| 方案                 | 写操作流程                             | 一致性级别 | 性能影响 | 复杂度 | 实时性 | 大厂主流使用场景                   | 主要风险/缺点                   |
| -------------------- | -------------------------------------- | ---------- | -------- | ------ | ------ | ---------------------------------- | ------------------------------- |
| Cache Aside（主流）  | 先写DB → 删Redis（或延迟双删）         | 最终一致   | 最低     | 低     | 中     | 99%普通业务（阿里/美团/字节）      | 可能短暂读到旧数据（毫秒~秒级） |
| Read/Write Through   | 写Redis（同步写DB或异步）              | 强/最终    | 中       | 高     | 高     | 极少（只有金融对账场景）           | Redis挂了影响DB，事务难         |
| Write Behind（异步） | 先写Redis → 异步批量刷DB               | 最终一致   | 最高     | 高     | 低     | 日志/计数类（部分字节/腾讯）       | 数据丢失风险，需强对账          |
| Binlog订阅（Canal）  | 写DB → Canal监听Binlog → 异步更新Redis | 最终一致   | 极低     | 中–高  | 中–高  | 高并发读多写少（美团/京东/拼多多） | 延迟1–5s，Canal本身运维成本     |
| 多级缓存 + 延迟双删  | 写DB → 删Redis → 延时1–3s再删一次      | 最终一致   | 低       | 低     | 高     | 大促秒杀/热点商品（阿里双11主力）  | 需MQ/定时任务支持               |

**最推荐组合（2025主流）**：
- 普通接口：**Cache Aside + 延迟双删 + 状态机**
- 高并发读：**Canal/Binlog订阅**（异步解耦，最稳）
- 金融/订单：**数据库 + Redis双写事务 + 最终对账Job**

**一句话总结**：  
“先写DB后删缓存”是底线，**Canal + 延迟双删**是性价比之王，**对账Job**是最终兜底。

### 2. 先删缓存还是先改库？Cache Aside、Delay Double Delete、Canal、Binlog订阅谁更好？

**经典面试陷阱**：面试官最爱问“先删还是先改”背后的**并发异常场景**。

四种方案详细对比（2025–2026年真实推荐）：

| 方案                                | 操作顺序                                       | 并发异常场景（最坏情况）                              | 一致性窗口 | 性能/实时性 | 复杂度 | 推荐场景 & 大厂选择理由            |
| ----------------------------------- | ---------------------------------------------- | ----------------------------------------------------- | ---------- | ----------- | ------ | ---------------------------------- |
| 先改库后删缓存（经典Cache Aside）   | 1. 更新DB<br>2. 删除Redis                      | 线程A改DB→删缓存；线程B读缓存miss→读旧DB→写旧缓存     | 毫秒~秒级  | 最高        | 最低   | 普通业务（字节/腾讯主流）          |
| 先删缓存后改库                      | 1. 删除Redis<br>2. 更新DB                      | 线程A删缓存；线程B读miss→读旧DB→写旧缓存；线程A再改DB | 秒级       | 高          | 低     | 几乎不用（风险比第一种大）         |
| **Delay Double Delete**（延迟双删） | 1. 更新DB<br>2. 删Redis<br>3. 延时1–3s再删一次 | 极大程度规避上面两种异常                              | <1秒       | 高          | 低     | 大促热点商品（阿里/美团/京东最爱） |
| **Canal / Binlog订阅**（异步）      | 只写DB，Canal监听后异步更新Redis               | 无并发异常（DB是权威）                                | 1–5秒      | 最高（读）  | 中     | 高QPS读场景（拼多多/美团2025主力） |

**面试高分回答**：
- **“先改库后删缓存”** 是理论最优（Cache Aside模式），但要配合**延迟双删**解决并发读写窗口。
- **Canal** 是目前性价比最高的大规模方案（解耦、异步、不侵入业务代码），2025年后90%+新项目倾向用Canal + Redis多级缓存。
- 谁更好？**没有绝对最好**：
  - 普通业务 → Cache Aside + Delay Double Delete
  - 大促/高并发 → Canal（或自研Binlog）+ 本地Caffeine
  - 金融强一致 → 数据库事务 + 同步双写（极少）

### 3. 缓存穿透、击穿、雪崩在真实项目中分别怎么处理？

**三兄弟对比 + 大厂真实打法**（2025–2026年）：

| 类型     | 触发原因                            | 危害         | 主流防护手段（优先级排序）                                   | 大厂典型组合（2025）                              |
| -------- | ----------------------------------- | ------------ | ------------------------------------------------------------ | ------------------------------------------------- |
| **穿透** | 查询不存在的key（恶意攻击）         | 全部打穿到DB | 1. Bloom Filter（布隆过滤器）<br>2. 空值缓存（null + 短TTL 30s–5min）<br>3. 接口参数校验 | Redis Bloom + 空值缓存（阿里/字节主力）           |
| **击穿** | 热点key突然过期，大量线程同时打DB   | DB瞬间被打死 | 1. 互斥锁（本地synchronized或Redis分布式锁）<br>2. 热点key永不过期 + 后台异步刷新<br>3. 本地Caffeine二级缓存 | 互斥锁 + 本地缓存（美团/京东最稳）                |
| **雪崩** | 大量key同一时间过期（大促预热失效） | DB雪崩       | 1. 随机TTL（基础TTL + 随机30–300s）<br>2. 多级缓存（Caffeine + Redis + 本地堆）<br>3. 降级开关 + 限流 | 随机TTL + 多级缓存 + Sentinel降级（阿里双11标配） |

**真实项目最常用组合拳**：
- 穿透：**布隆 + 空值缓存**
- 击穿：**本地Caffeine + Redis互斥锁**（单机QPS 10w+不打DB）
- 雪崩：**随机TTL + 多级缓存 + 熔断降级**

### 4. 大促预热缓存、热点key、缓存击穿的组合拳怎么打？

**双11/618真实打法**（2025–2026年阿里/美团/京东标准流程）：

**四层组合拳**（从外到内）：

1. **预热阶段（活动前1–24小时）**
   - 脚本/Job批量把秒杀/热门商品全量刷进Redis（Lua pipeline）
   - 同时刷本地Caffeine（每台机器本地热点）
   - 预热完成后打开“预热标记”

2. **热点key检测 & 分片**
   - Redis自带热点key发现（Redis 6+ hotkey）或自研监控
   - 热点key自动分片（hash tag）或本地Caffeine + Redis双写
   - 极热key用“本地内存 + 异步刷新”模式（永不过期）

3. **击穿防护（实时）**
   - 读流程：先本地Caffeine → Redis → DB
   - Redis miss时用**分布式锁**（Redisson或自研Lua锁），只让一个线程回源
   - 回源成功后双写本地 + Redis

4. **兜底 & 监控**
   - Sentinel热点参数限流
   - 缓存命中率/回源QPS实时大盘
   - 降级开关：缓存全挂 → 直接读DB + 限流

**一句话总结大促组合拳**：
**“活动前全量预热 + 本地Caffeine二级缓存 + Redis热点分片 + 互斥锁回源 + 随机TTL + Sentinel降级”**  
→ 实测可扛百万QPS，DB回源<1%。

### 1. 分布式主键ID生成方案在海量订单场景下怎么选？

海量订单（日亿级、累计百亿级）核心要求：**全局唯一**、**高性能（10w+ TPS）**、**趋势递增**（利于索引/B+树）、**可水平扩容**、**可嵌入分片信息**。

2025–2026年大厂主流方案对比表：

| 方案                  | 生成方式                     | TPS上限 | 是否递增 | 是否全局唯一     | 是否可嵌入分片 | 运维成本 | 大厂典型用法（2025）     | 主要缺点                           |
| --------------------- | ---------------------------- | ------- | -------- | ---------------- | -------------- | -------- | ------------------------ | ---------------------------------- |
| DB自增（单库）        | MySQL auto_increment         | 几千    | 是       | 否（分库后冲突） | 否             | 低       | 极早期小项目             | 分库后ID冲突，性能差               |
| UUID / Snowflake变种  | 随机/算法                    | 极高    | 部分     | 是               | 可             | 低       | 辅助方案                 | 无序（索引碎片）、存储大（128bit） |
| **Snowflake（标准）** | 时间戳+机器ID+序列           | 40w+    | 趋势递增 | 是               | 可             | 中       | 字节/腾讯/拼多多普通订单 | 时钟回拨、机器ID上限（1024）       |
| **Leaf（美团）**      | 号段模式 + Snowflake         | 50w+    | 趋势递增 | 是               | 可             | 中       | 美团/阿里部分订单        | 依赖DB/ZK，号段耗尽风险            |
| **数据库号段模式**    | DB预分配号段（批量取）       | 10w+    | 是       | 是               | 可             | 中       | 阿里早期、京东部分       | DB压力、扩容需重置                 |
| **自定义嵌入分片**    | Snowflake + 分片位（订单号） | 30w+    | 趋势递增 | 是               | 强支持         | 低       | **淘宝/京东主力**        | 需业务定制                         |

**海量订单选型逻辑（2025主流）**：
- **首选**：Snowflake变种或Leaf（高性能 + 趋势递增 + 全局唯一）
- **订单号特殊要求**：必须**嵌入分片键**（如用户ID后几位），方便路由 → 淘宝/京东最爱
- **极致场景**：Leaf + 本地缓存号段（美团风格）
- **兜底**：对账Job + 补偿生成（防止ID冲突）

**一句话总结**：普通业务用Snowflake，订单业务必须“Snowflake + 分片信息嵌入”。

### 2. 为什么雪花算法在某些场景下不合适？有什么变种？

**Snowflake不合适场景**（面试高频追问）：
1. **时钟回拨**：机器时间被NTP/手动调整回拨 → ID重复或乱序（最致命）
2. **机器ID上限**：默认5bit（32个机房）+5bit（32台机器）=1024台上限，高可用集群不够
3. **ID可预测**：连续递增 → 安全风险（订单号被猜、刷库）
4. **严格全局单调递增**：跨节点无法保证（仅单机递增）
5. **高QPS + 强递增需求**：每毫秒4096个上限，极端秒杀可能耗尽
6. **无分片信息**：纯订单场景需要路由信息时不友好

**2025–2026年主流变种**（大厂真实使用）：

| 变种                | 改进点                              | 解决的问题         | 大厂使用场景       | 代表实现                 |
| ------------------- | ----------------------------------- | ------------------ | ------------------ | ------------------------ |
| Meituan Leaf        | 号段 + Snowflake + DB容灾           | 时钟回拨、号段耗尽 | 美团订单、阿里部分 | Leaf-segment + Snowflake |
| Baidu UID           | 时间戳 + worker + sequence + 扩展位 | 机器ID上限、回拨   | 百度内部订单       | UIDGenerator             |
| Sonyflake           | 缩短时间戳位、增加序列位            | 机器上限、ID长度   | 部分日企/国内中厂  | Sonyflake                |
| **淘宝/京东订单号** | Snowflake + 买家/卖家ID后N位嵌入    | 路由 + 可预测问题  | 淘宝/京东亿级订单  | 自定义（订单号规则）     |
| 阿里自研（内部）    | 时钟回拨自动漂移 + 多机房位         | 回拨 + 扩容        | 阿里双11订单       | 内部版Snowflake          |

**高分回答**：  
“标准Snowflake在时钟回拨和高可用集群场景下容易出问题，所以大厂普遍用**Leaf**或**嵌入分片信息的定制Snowflake**，同时配合时钟监控 + 备用节点兜底。”

### 3. 淘宝/京东订单表怎么分库分表？（按用户？按商家？按时间？）

**核心结论**（2025–2026年公开分析 + 大厂实践）：**不单用一种**，而是**买家库 + 卖家库双库 + 多维度冗余**。

**淘宝真实做法**（最经典）：
- **垂直拆库**：买家库（buyer_db） + 卖家库（seller_db）
- **水平分片键**：
  - 买家库 → **按 buyer_id（用户ID）哈希分库分表**（支持“我的订单”查询）
  - 卖家库 → **按 seller_id（商家ID）哈希分库分表**（支持“我的销售”查询）
- **数据冗余**：一个订单同时写两份（买家库 + 卖家库），通过**MQ异步同步**状态变更
- **订单号设计**：嵌入 **买家ID后2位 + 卖家ID后2位** → 路由时直接从订单号取出分片信息（无需额外查路由表）
- **冷热分离**：热数据（近90天）在分库分表MySQL，冷数据归档到ES/HBase/TiDB

**京东做法**（类似但更灵活）：
- **主分片键**：**buyer_id（用户ID）哈希**（个人订单查询为主）
- **辅助维度**：同时按 **order_id**、**seller_id**、**create_time** 冗余存储多份（或用ShardingSphere广播/绑定表）
- **分表策略**：通常 **32/64/128库 × 32/64表**（按业务峰值估算，单表控制在5000万行以内）
- 2025年趋势：**ShardingSphere** 做路由 + 多维度查询加速

**为什么不单纯按时间？**
- 按时间分 → “我的订单”查询跨库跨表爆炸（分页/聚合惨）
- 按用户/商家 → 查询友好，但热点用户（大V商家）倾斜 → 解决办法：**一致性Hash + 扩容双写**

**一句话总结**：淘宝 = 买家库(user_id) + 卖家库(seller_id) + 订单号嵌入分片；京东 = buyer_id主分片 + 多维度冗余 + ShardingSphere。

### 1. RocketMQ/Kafka双11峰值消息积压怎么处理？

双11峰值积压（百万~千万条常见）核心：**快速消费 + 动态扩容 + 监控预警**，目标是把积压从“小时级”压到“分钟级”。

**2025–2026年大厂主流处理流程**（分事前/事中/事后）：

**事前准备（必做）**
- Topic分区规划：提前按峰值QPS估算（RocketMQ建议分区数 = 消费者数 × 2~4）
- 消费者线程池：批量消费（batch 10~100）+ 线程池并行（RocketMQ ConsumeConcurrently）
- 消费超时设置：单个消息处理 < 1min（否则重投）

**事中应急三板斧**（5–15分钟止血）
1. **水平扩容消费者**（最快）  
   - K8s HPA / 阿里云ESS 自动扩容消费组（实例数翻2–5倍）
   - RocketMQ：动态添加消费者实例（零停机）
   - Kafka：增加消费者实例 + rebalance（注意rebalance风暴）
2. **临时提速**  
   - 开启**跳过堆积消息**（RocketMQ skip策略）或**批量加大**（Kafka max.poll.records=5000）
   - 降级非核心Topic（营销消息、日志类先丢）
3. **消息分流**  
   - 热点队列单独消费组（RocketMQ Message Queue分片）
   - Kafka用多个消费组并行拉取

**事后复盘**
- RocketMQ控制台 / Kafka Eagle 查看堆积曲线、消费延迟
- 优化点：消费端慢查询、DB压力、锁竞争 → 加本地缓存、异步落库、批量写

**一句话总结**（面试高分）：
“双11积压靠**事前分区规划 + 事中HPA自动扩容 + 批量消费**，RocketMQ比Kafka更友好（支持跳过堆积、不重平衡风暴）。”

### 2. 保证消息至少一次、至多一次、恰好一次分别怎么做？

**三种语义对比表**（2025–2026年主流实现）：

| 语义                                  | 保证机制           | 实现方式（RocketMQ / Kafka）                                 | 性能影响 | 适用场景                       | 大厂主流选择（2025）      |
| ------------------------------------- | ------------------ | ------------------------------------------------------------ | -------- | ------------------------------ | ------------------------- |
| **At Least Once**（至少一次，最常见） | 消费成功后ACK      | RocketMQ：ack后commit<br>Kafka：enable.auto.commit=false，手动commit offset后处理 | 最低     | 订单、支付、库存（可重试）     | 99%业务（阿里/美团主力）  |
| **At Most Once**（至多一次）          | 先ACK再处理        | RocketMQ：收到即commit<br>Kafka：auto.commit=true，offset先提交再消费 | 最高     | 日志、监控、统计（丢了无所谓） | 日志类、埋点（字节/腾讯） |
| **Exactly Once**（恰好一次，最难）    | 幂等 + 事务 + 去重 | RocketMQ：事务消息 + 消费端幂等<br>Kafka：事务API + EOS（exactly-once semantics） + 幂等 | 最高     | 金融、资金、订单状态变更       | 金融场景（京东/拼多多）   |

**Exactly Once落地细节**（面试最爱问）：
- RocketMQ：生产端事务消息（半消息 + 本地事务 + 回查） + 消费端幂等（业务唯一ID）
- Kafka：Kafka Streams / Flink + 事务 + 幂等消费者（producer.enable.idempotence=true + 事务）
- 通用兜底：**消费流水表 + 状态机**（永远不丢不重）

**高分回答**：  
“生产环境99%用At Least Once + 消费端幂等实现Exactly Once效果；纯Exactly Once只在金融核心链路用Kafka事务或RocketMQ事务消息。”

### 3. 严格顺序消息（如下单→支付→发货）怎么保证？

**核心原理**：**同一个消息队列 + 单线程消费**（生产有序 + 消费有序）。

**RocketMQ vs Kafka对比**（2025–2026年做法）：

| 维度          | RocketMQ方案                               | Kafka方案                             | 优缺点对比                            | 大厂真实选择                |
| ------------- | ------------------------------------------ | ------------------------------------- | ------------------------------------- | --------------------------- |
| 分区/队列控制 | 同一个Message Queue（Orderly消息）         | 同一个Partition                       | RocketMQ支持MessageQueue级别更细粒度  | RocketMQ更常用（阿里/美团） |
| 生产端        | producer.send(Orderly, hashKey) → 同一队列 | producer.send(相同partition)          | 必须用相同hashKey/partition           | 相同                        |
| 消费端        | MessageListenerOrderly（单线程串行消费）   | 单个consumer单线程拉取该partition     | RocketMQ自动保证顺序，Kafka需手动控制 | RocketMQ更省心              |
| 失败重试      | 失败后整个队列暂停消费（保序）             | 失败后暂停partition消费               | 都会阻塞后续消息                      | 配合死信队列兜底            |
| 扩容          | 无法动态扩分区（顺序消息不支持）           | 无法动态扩partition（顺序消息不支持） | 两者都牺牲了水平扩容                  | 业务拆分 + 多Topic          |

**严格顺序实战落地**（下单→支付→发货）：
1. 订单ID作为hashKey → 生产时路由到同一队列/分区
2. 消费端用Orderly监听器（RocketMQ）或单线程consumer（Kafka）
3. 业务状态机 + 幂等：即使重试也不会乱序
4. 极端情况：队列积压 → 报警 + 人工介入或拆分Topic（不同业务用不同Topic）

**高分回答**：  
“严格顺序只能牺牲并行度，用**同一队列 + 单线程消费**实现。RocketMQ原生支持Orderly消息，比Kafka更友好；高并发下通常拆业务，避免全局顺序。”

### 4. 消费端幂等 + 死信队列 + 延迟队列怎么组合使用？

**三者组合拳**（2025–2026年标准流程，最稳方案）：

**整体流程图（伪代码级）**：
1. 消息生产 → 带业务唯一ID（orderNo）
2. 消费端：
   - **幂等第一关**：Redis setnx(业务ID, "PROCESSING", 5min) 或查防重表
     - 已处理 → 直接ACK（跳过）
     - 处理中 → 延迟重试（放延迟队列）
     - 未处理 → 执行 + 成功后更新状态
3. **失败处理**：
   - 消费失败（异常/超时）→ 进入**重试队列**（RocketMQ 18次重试机制）
   - 重试仍失败 → 进入**死信队列（DLQ）**
4. **延迟队列**：
   - 超时订单回补库存、支付回调重试 → 用RocketMQ定时消息（delayLevel）或Kafka + 时间轮
   - 死信消息 → 定时任务扫描DLQ → 延迟队列重投（人工审核或自动补偿）

**组合最佳实践**（大厂真实配置）：
- RocketMQ：消费失败自动重试16次 → 第17次进DLQ → DLQ消费端再投延迟队列（delay 1h/6h/24h）
- Kafka：手动实现重试 + DLQ Topic + 延迟Topic（或用Quartz/K8s CronJob）
- 监控：DLQ积压告警 + 幂等命中率大盘
- 补偿Job：T+1扫描“处理中”超时的消息 → 强制重投延迟队列

**一句话总结**：
“**幂等防重 + 死信兜底失败 + 延迟队列做渐进重试/补偿**”是消费端最稳组合，覆盖了丢失、重试、顺序、最终一致全部场景。”

这些回答基本覆盖了大厂P6–P8所有追问（包括RocketMQ事务消息细节、Kafka EOS实现、DLQ监控报警、顺序消息扩容痛点等）。

### 1. 分布式锁在哪些真实业务场景必须用？怎么选型（Redis、Zookeeper、etcd、数据库）？

- 全局唯一任务调度（定时任务防多节点同时执行）

**选型决策树**（面试最爱问，按优先级排序）：

| 锁类型        | 性能（QPS）  | 可靠性（CP/AP） | 阻塞等待 | 实现复杂度 | 适用场景（2025主流）               | 大厂典型用法              | 主要缺点                        |
| ------------- | ------------ | --------------- | -------- | ---------- | ---------------------------------- | ------------------------- | ------------------------------- |
| **Redis**     | 最高（10w+） | AP（最终一致）  | 支持     | 最低       | 秒杀、抢红包、限流、99%业务        | 阿里/美团/字节/京东主力   | 网络抖动可能锁失效（用RedLock） |

| **Redisson**  | 极高         | AP+RedLock      | 支持     | 最低       | 需要可重入/公平锁的秒杀场景        | 美团/京东秒杀标配         | 依赖Redis，集群模式需注意       |

“高并发读写 → Redis/Redisson；强一致领导选举 → ZK/etcd；简单低并发 → 数据库乐观锁；云原生 → etcd。秒杀99%选Redis + RedLock。”

### 2. 秒杀场景分布式锁的性能瓶颈怎么解决？

**性能瓶颈本质**：锁粒度太大 → 所有请求串行 → 单机QPS瓶颈在几千。

**2025–2026年大厂真实优化组合拳**（从“有锁”到“无锁”演进）：

1. **最快止血（单机优化）**
   - 锁粒度细化：从“商品ID”锁 → “商品ID + 桶号”分段锁（10个桶并发10倍）
   - Redisson公平锁换成非公平 + 本地ReentrantLock前置过滤

2. **主流无锁方案（性能提升10倍+）**
   - **Redis Lua原子扣减**（最推荐）：DECR + Lua脚本判断库存>0，一次RTT完成，无锁！
   - **本地内存预分配 + Redis补**（12306风格）：每台机器本地扣Buffer → 成功再远程Redis原子扣
   - **预减库存 + MQ异步**：Redis先预减（原子），成功再MQ下单，彻底无锁

3. **极端优化**
   - 热点key分片（Redis hash tag）
   - 本地Caffeine + Redis双层（本地锁 + 远程原子）
   - Sentinel热点参数限流 + 降级

**性能对比**（真实压测数据）：
- 纯Redisson分布式锁 → 峰值QPS ≈ 8000
- Lua原子扣减 → QPS 8–12万
- 本地Buffer + Redis → QPS 20万+

**高分回答**：  
“秒杀锁的瓶颈是串行竞争，核心解法是**Lua无锁原子操作**或**本地内存预分配**，把分布式锁降级为最终兜底。”

### 3. TCC、Seata、XA、MQ最终一致、柔性事务在什么业务场景下选哪个？

**事务方案选型表**（2025–2026年大厂真实决策）：

| 方案           | 一致性   | 性能 | 业务侵入度 | 回滚复杂度 | 适用场景（典型例子）                     | 大厂主流选择（2025）      | 核心缺点                          |
| -------------- | -------- | ---- | ---------- | ---------- | ---------------------------------------- | ------------------------- | --------------------------------- |
| **XA**         | 强一致   | 极低 | 无         | 自动       | 银行核心转账、金融强一致                 | 极少（仅传统银行）        | 2PC阻塞、性能差、跨库不支持       |
| **TCC**        | 强一致   | 中   | 高         | 手动写     | 订单+库存+积分（需Try/Confirm/Cancel）   | 阿里/美团部分金融链路     | 业务代码侵入重、空回滚/悬挂难处理 |
| **Seata AT**   | 最终一致 | 高   | 低         | 自动       | 跨库订单、优惠券核销、普通电商           | 阿里/字节/京东主力（90%） | 依赖undo_log，脏回滚风险          |
| **Seata TCC**  | 强一致   | 中   | 中         | 手动       | 需要强一致但不想XA的场景                 | 美团/拼多多金融           | 需手写Try/Confirm/Cancel          |
| **MQ最终一致** | 最终一致 | 最高 | 低         | 补偿Job    | 支付回调、积分、营销、异步通知           | 所有公司99%非核心链路     | 需幂等 + 补偿 + 对账              |
| **柔性事务**   | 最终一致 | 最高 | 中         | 业务补偿   | 库存扣减、秒杀、优惠券（允许短暂不一致） | 阿里双11秒杀、京东库存    | 需业务设计可补偿逻辑              |

**选型口诀**：
- 要强一致 + 数据库支持 → XA（极少）
- 要强一致 + 可接受侵入 → TCC/Seata TCC
- 普通电商跨库 → **Seata AT**（最香）
- 高并发异步 → **MQ最终一致 + 补偿Job**
- 库存/营销 → **柔性事务**（允许少量不一致）

### 4. 跨库转账、积分扣减、优惠券核销怎么保证强/最终一致？

**三大场景具体落地**（2025真实方案）：

1. **跨库转账（A账户→B账户）**（最严）
   - **强一致**：Seata AT（自动undo_log）或TCC（Try冻结→Confirm扣→Cancel解冻）
   - **最终一致**：MQ异步 + 资金流水表 + T+1对账Job（阿里/京东金融主力）
   - 兜底：补偿任务 + 人工介入

2. **积分扣减**（高并发）
   - 下单扣积分 → Redis预减（Lua）+ MQ异步写DB
   - 失败 → 死信队列 + 补偿Job（扫描未扣积分订单）
   - 一致性：**柔性事务 + 幂等 + 对账**（允许短暂积分显示延迟）

3. **优惠券核销**（最常见）
   - 使用券 → Redis setnx（幂等）+ MQ异步核销状态
   - 支付失败回滚 → MQ延时消息 + 补偿
   - 一致性：**MQ最终一致 + 券状态机 + 每日对账**（防止券被多次核销）

**统一组合拳**（大厂标配）：
- Seata（强一致场景） + MQ（高并发场景）
- 所有场景必备：**幂等 + 状态机 + 补偿Job + 对账**
- 监控：事务失败率、补偿成功率、资金对账差额

**一句话总结**：
“转账用Seata/TCC，积分/优惠券用MQ + 柔性 + 对账，核心是**提前设计补偿逻辑**，没有完美的强一致，只有最合适的最终一致。”

### 1. 调用第三方接口经常超时，怎么优化用户体验？

**核心思路**：**同步变异步 + 快速返回 + 兜底降级 + 主动推送**，让用户感觉“秒响应”而不是卡死。

**2025–2026年主流分层优化方案**（从前端到后端）：

| 层级      | 优化手段                          | 用户感知效果   | 实现复杂度 | 大厂典型用法             | 关键技术/组件                   |
| --------- | --------------------------------- | -------------- | ---------- | ------------------------ | ------------------------------- |
| 前端      | 立即返回“处理中” + Loading/骨架屏 | 秒开页面       | 低         | 所有App/H5               | React/Vue + 轮询/WebSocket      |
| 网关/业务 | 设置合理超时（3–5s）+ 异步化调用  | 同步变异步     | 中         | 阿里/美团第三方支付/物流 | CompletableFuture / WebClient   |
| 容错      | 熔断 + 降级 + 限流                | 失败不雪崩     | 中         | 字节/腾讯所有外部调用    | Resilience4j / Sentinel         |
| 异步补偿  | MQ异步回调 + 主动推送             | 最终结果秒达   | 中         | 京东物流、拼多多支付     | RocketMQ + WebSocket / App Push |
| 兜底      | 本地缓存/默认值 + 人工/重试机制   | 最坏情况可接受 | 低         | 所有金融/外部接口        | 本地Caffeine + 补偿Job          |

**推荐组合拳**（面试高分）：
1. 接口立即返回“请求已受理，稍后通知您”（HTTP 202）
2. 后端用`CompletableFuture`或`WebClient` + `timeout(3s)`异步调用第三方
3. 超时/失败 → 走降级（返回默认值或“稍后查询”）+ 发MQ
4. 第三方回调成功 → MQ消费后推送用户（WebSocket / App Push）
5. 前端轮询接口（指数退避）或长连接监听

**常问追问**：
- 第三方一直不回调？ → 定时补偿Job + 对账
- 如何防止用户重复提交？ → 结合前面幂等Token

一句话总结：**“前端秒回 + 异步MQ + 熔断降级 + 主动推送”** 是用户体验最优解。

### 2. 10w条数据导入/导出/批量操作怎么异步化？

**核心**：**不能同步阻塞主线程**，目标是“前端秒开 + 后台慢慢跑 + 进度可查”。

**2025主流异步化方案对比**：

| 场景     | 推荐方案                         | 处理速度      | 进度反馈        | 复杂度 | 大厂典型用法          | 关键实现                            |
| -------- | -------------------------------- | ------------- | --------------- | ------ | --------------------- | ----------------------------------- |
| 导入     | MQ + 分批消费 + 异步任务         | 10w条/分钟级  | 实时进度条      | 中     | 美团/京东批量导入     | RocketMQ + XXL-Job / PowerJob       |
| 导出     | 异步任务 + Excel Streaming + OSS | 10w条/5–15min | 进度 + 下载链接 | 中     | 阿里/字节运营后台导出 | EasyExcel + CompletableFuture + OSS |
| 通用批量 | Spring @Async + ThreadPool + MQ  | 高            | 任务ID轮询      | 低     | 腾讯/拼多多内部工具   | @Async + TaskExecutor + Redis进度   |

**落地流程**（最推荐）：
1. 前端提交 → 立即返回`taskId`
2. 后端：`@Async` 或 XXL-Job调度任务 → 写Redis进度（0% → 100%）
3. 分批处理（每批1000–5000条）+ 批量插入/更新（JDBC batch / MyBatis batch）
4. 导出：EasyExcel Streaming写OSS，完成后回调前端（WebSocket或轮询下载链接）
5. 失败：进入死信 + 补偿Job + 邮件/短信通知运营

**性能Tips**：
- 用虚拟线程（Java 21+ Project Loom）或线程池动态扩容
- 限流 + 熔断防止把DB打挂
- 大文件用分片上传 + 并行处理

一句话总结：**“前端拿taskId + 后端MQ/异步任务 + Redis进度 + OSS存储”**，10w条轻松搞定，用户无感知。

### 3. 削峰填谷典型场景（下单、券领取、物流状态推送）怎么做？

**核心**：**把峰值流量“削”到MQ，再“填”到低谷消费**，让系统始终平稳。

**三大场景具体打法**（2025–2026年主流）：

| 场景         | 削峰手段                         | 填谷手段                | 关键技术                       | 大厂典型用法       |
| ------------ | -------------------------------- | ----------------------- | ------------------------------ | ------------------ |
| **下单**     | 前端防抖 + 网关限流 + Redis预占  | MQ异步扣库存 + 异步发货 | RocketMQ延迟消息 + 令牌桶      | 阿里/京东双11下单  |
| **券领取**   | Sentinel热点限流 + Redis Lua抢占 | MQ异步核销 + 补偿Job    | 令牌桶 + 延迟队列              | 美团/拼多多抢券    |
| **物流推送** | 第三方回调先入MQ（不直接推业务） | 消费端批量 + 限流推送   | Kafka分区 + 消费线程池动态扩容 | 京东物流、淘宝物流 |

**通用削峰填谷模型**：
1. **入口限流**：Nginx/Sentinel全局 + 单用户限流
2. **削峰**：所有请求先写Redis/MQ（瞬间扛住峰值）
3. **填谷**：消费者按固定速率消费（线程池 + 令牌桶 / RateLimiter）
4. **兜底**：堆积监控 + 自动扩容消费者 + 降级开关

一句话总结：**“入口全进MQ + 消费端令牌桶控速 + 动态扩容”**，峰值10倍也能稳。

### 4. 异步任务重试、任务堆积、任务依赖怎么管理？

**核心框架**：**任务中心 + 重试策略 + 堆积告警 + DAG依赖**。

**2025主流全生命周期管理方案**：

### 1. 电商订单列表、物流轨迹、用户行为日志怎么快速分页？

**核心痛点**：传统 `limit offset` 在千万级表上会全表扫描，偏移量越大越慢。

**2025–2026年主流分页方案对比**：

| 场景             | 推荐方案                             | 核心技术                                  | 查询速度（百万级） | 支持排序/过滤 | 大厂典型用法         | 缺点/注意点                        |
| ---------------- | ------------------------------------ | ----------------------------------------- | ------------------ | ------------- | -------------------- | ---------------------------------- |
| **订单列表**     | **ES + MySQL双写** 或 **最后ID分页** | ES搜索 + MySQL最后ID（> last_id limit N） | 毫秒级             | 全支持        | 京东/淘宝/拼多多主力 | ES同步延迟需控制在1s内             |
| **物流轨迹**     | ES + 宽表冗余                        | ES按order_id/轨迹时间倒排                 | 毫秒级             | 全支持        | 京东物流、美团外卖   | 轨迹更新用MQ异步更新ES             |
| **用户行为日志** | **ClickHouse / ES**                  | ClickHouse MergeTree + ES倒排             | 毫秒级（亿级）     | 全支持        | 字节/阿里埋点分析    | ClickHouse适合聚合，ES适合全文搜索 |
| **通用兜底**     | ShardingSphere + 绑定表              | 按user_id/时间分片 + 最后ID分页           | 亚秒级             | 支持          | 中小型分库分表项目   | 跨分片聚合需ES辅助                 |

**推荐组合拳**（面试高分）：
1. **前端**：滚动分页（scroll / last_id），不传pageNum/offset
2. **后端**：先走ES模糊/复合查询 → 拿到主键列表 → 再回MySQL/ClickHouse拿详情（ES只存必要字段）
3. **订单/物流**：订单号嵌入分片信息 + 按user_id分片 + 最后ID + 业务时间倒序
4. **日志**：ClickHouse按时间+user_id分区 + ES全文索引
5. **兜底**：Redis缓存最近7天热数据 + 降级读DB

一句话总结：**“ES负责搜索 + 最后ID分页 + 分片路由 + 宽表冗余”**，千万级列表毫秒返回。

### 2. 千万级甚至亿级数据模糊搜索怎么做？

**核心**：MySQL `like '%xx%'` 彻底废掉，必须上搜索引擎。

**2025主流架构**（分实时 vs 离线）：

| 方案                  | 搜索引擎        | 分词器/索引类型                 | 实时性        | QPS上限 | 大厂典型用法                      | 扩展能力                      |
| --------------------- | --------------- | ------------------------------- | ------------- | ------- | --------------------------------- | ----------------------------- |
| **ES（最主流）**      | Elasticsearch   | IK / ngram / pinyin / edgeNGram | 秒级（Canal） | 10w+    | 阿里/京东/美团/字节搜索主力       | 聚合、排序、向量搜索（ES 8+） |
| **ClickHouse**        | ClickHouse      | ngram / token / inverted        | 毫秒级        | 百万+   | 字节/阿里日志搜索                 | 适合海量聚合，不适合复杂全文  |
| **Doris / StarRocks** | Doris           | inverted + bitmap               | 秒级          | 百万+   | 新兴实时数仓（部分美团/京东）     | 强聚合，搜索稍弱              |
| **ES + Vector**       | ES + Milvus混合 | 向量 + 文本                     | 秒级          | 高      | 推荐搜索、语义搜索（字节/拼多多） | 支持AI语义搜索                |

**落地流程**（最推荐ES方案）：
1. **数据同步**：Canal / Flink CDC / MQ实时同步MySQL → ES（或离线Spark批量）
2. **索引设计**：多字段组合（商品名 + 标题 + 标签 + pinyin），用ngram防错别字
3. **查询优化**：bool查询 + function_score + 异步聚合 + 缓存热词
4. **亿级优化**：热冷分离（近30天热索引 + 老数据归档）、滚动索引、跨集群搜索
5. **兜底**：布隆过滤器 + 缓存 + 降级返回“相关推荐”

一句话总结：**“Canal/Flink实时同步 + ES ngram/pinyin倒排 + bool多字段查询 + 向量扩展”**，亿级模糊搜索毫秒级。

### 3. 运营后台导出百万级订单/用户数据怎么不卡死数据库？

**核心**：**绝不能一条SQL limit百万**，必须**流式 + 异步 + 分批**。

**2025主流导出方案对比**：

| 方案                              | 核心技术                         | 导出速度（百万级） | 是否卡DB | 内存占用 | 大厂典型用法           | 关键防崩手段           |
| --------------------------------- | -------------------------------- | ------------------ | -------- | -------- | ---------------------- | ---------------------- |
| **EasyExcel Streaming**（最推荐） | EasyExcel + JDBC Stream / Cursor | 5–15分钟           | 不卡     | 极低     | 阿里/字节/京东运营后台 | 分页流式读取 + 写OSS   |
| **异步任务 + 分片**               | XXL-Job / PowerJob + 多线程分片  | 3–10分钟           | 不卡     | 低       | 美团/拼多多            | 按user_id/时间分片并行 |
| **ClickHouse导出**                | ClickHouse SELECT INTO OUTFILE   | 1–5分钟            | 不卡     | 低       | 日志/宽表导出          | ClickHouse原生高吞吐   |
| **Doris / Spark**                 | Spark SQL + 分布式导出           | 分钟级             | 不卡     | 中       | 大数据团队             | 离线计算集群           |

**完整落地流程**（面试最爱）：
1. 前端提交导出 → 返回taskId（立即响应）
2. 后台XXL-Job异步任务：按时间/用户分片（每批1–5万）
3. 用`ResultSet`流式读取（fetchSize=1000）或JDBC Cursor
4. EasyExcel Streaming边读边写Excel → 直接上传OSS
5. 进度实时写Redis，用户轮询下载链接

- 读写分离从库
- 限流 + 熔断（单任务最大并发）

**事后修复（10分钟–几天）**（三板斧）：
1. **快速修复**：写补偿Job（扫描不一致记录 → 重试或回滚）
2. **批量修复**：用Flink / Spark离线Job全量对账（订单流水 vs 库存 vs 资金）

- 每日T+1对账Job（必备）

- 状态机 + 幂等 + 补偿表

**高分回答**：

### 1. 单体 → 微服务拆分边界怎么划？怎么控制拆分节奏？

**核心结论**：**业务驱动 + 渐进式**，绝不一步到位。2025–2026年大厂统一采用**五步安全演进路径**（阿里云开发者社区最新总结）。

**拆分边界划定原则**（7大黄金原则，高频考点）：
- 高内聚低耦合（一个限界上下文 = 一个微服务）
- 数据自治（每个服务独占DB，避免跨库事务）
- 业务域对齐（DDD子域划分：核心域/支撑域/通用域）
- 变更频率隔离（高频变更域优先拆）
- 团队规模匹配（每个服务2–8人可维护）
- 技术异构友好（不同服务可不同语言/框架）
- 可独立部署 + 可观测

**拆分边界实战方法**（美团/阿里风格）：
1. **战略设计**：事件风暴 / DDD建模 → 画出限界上下文
2. **垂直拆分**：先按“用户/订单/商品/支付”等大域拆（最稳）
3. **水平拆分**：再按读写分离、热冷数据拆

**控制拆分节奏**（五步渐进式，零翻车）：
1. 单体 → 模块化（代码分层 + 包隔离，3–6个月）
2. 垂直拆库（DB先拆，代码后拆）
3. 非核心服务先拆（营销、日志、配置）
4. 支撑域拆（库存、优惠券）
5. 核心域最后拆（订单、支付）+ 多活/灰度验证

**节奏控制三板斧**：
- 每次只拆1–2个服务
- 双写 + 灰度 + 回滚预案
- 拆完一个验证一个（链路监控 + 压测）

**高分回答**：  
“拆分边界用DDD限界上下文，节奏用五步渐进式：先模块化 → 垂直拆库 → 非核心 → 核心。阿里双11前都是这样演进的，避免休克式疗法。”

### 2. 老系统百万行代码，怎么逐步重构不影响线上？

**核心模式**：**Strangler Pattern（绞杀者模式）** + **并行运行新老系统**，2025年大厂主流做法。

**百万行代码重构四步法**（阿里/京东真实案例）：
1. **止血 + 隔离**（第1–2个月）
   - 画出当前系统全链路图 + 依赖矩阵
   - 新功能全部走新架构，老功能维持
   - 引入Facade层/防腐层隔离新老代码

2. **逐步绞杀**（第3–12个月）
   - 按优先级（高频变更 → 核心 → 历史）一个个模块替换
   - 每个模块：新服务 + 双写 + 读切换 + 写切换
   - 每次只上线一个服务，用灰度流量验证

3. **数据与流量切换**（见第3题）
   - 历史数据迁移 + 双写
   - 流量逐步从老系统切到新系统（1% → 10% → 100%）

4. **下线老系统**（第12–18个月）
   - 老系统只读 → 完全下线

**不影响线上关键技术**：
- 配置中心一键开关（Nacos/Apollo）
- 灰度 + 金丝雀发布（Istio/K8s）
- 链路追踪 + 自动回滚（SkyWalking + Argo Rollouts）
- 每步必做压测 + 混沌工程

**高分回答**：  
“用Strangler绞杀者模式，百万行代码分18个月逐步替换：新老并行 + 双写 + 灰度流量切换 + 配置开关。京东订单系统就是这么从单体重构到微服务的。”

### 3. 历史数据迁移、双写、读写分离、流量切换怎么做到无感知？

**核心**：**双写 + 逐步切换 + 对账兜底**，实现“用户零感知、业务零中断”。

**2025–2026年标准四阶段无感知迁移方案**（阿里/美团/ClickHouse迁移案例）：

| 阶段                | 操作                      | 技术手段                           | 用户感知 | 风险控制               |
| ------------------- | ------------------------- | ---------------------------------- | -------- | ---------------------- |
| **1. 双写准备**     | 新库建表 + 业务代码改双写 | Canal监听Binlog + MQ异步补历史数据 | 无       | 先双写验证一致性       |
| **2. 历史数据迁移** | 存量数据一次性全量同步    | DataX / Flink CDC / Spark离线Job   | 无       | 分批 + 校验 + 补偿     |
| **3. 读写分离切换** | 先切读（90%流量）→ 再切写 | 配置中心开关 + ShardingSphere路由  | 无       | 灰度1%→100% + 回滚预案 |
| **4. 停止双写**     | 验证一致性后停老库写      | 对账Job（T+1全量比对） + 监控告警  | 无       | 补偿Job + 人工兜底     |

**关键无感知技巧**：
- **读切换**：先切读流量（新库读 + 老库兜底）
- **写切换**：双写期间新库优先，成功率>99.99%后再切
- **流量切换**：Istio/K8s金丝雀 + 按用户/百分比
- **兜底**：每日对账Job + 补偿任务 + 死信队列

**高分回答**：  
“历史数据用Flink CDC + 双写，读写分离用配置中心灰度切换，最后对账Job兜底。阿里双11库存库迁移就是这样零感知完成的。”

### 4. 域驱动设计在实际项目中怎么落地？（有赞/美团案例风格）

**核心**：**DDD不是银弹，而是把“业务架构”映射成“系统架构”**。美团/有赞/阿里都用**战略设计 + 战术设计**两步落地。

**美团/有赞真实落地四阶段**（2024–2026美团技术沙龙 + 有赞生产项目总结）：

1. **战略设计（限界上下文划分）**
   - 事件风暴workshop → 划分子域（核心域：订单/支付；支撑域：库存/优惠券；通用域：用户/通知）
   - 每个限界上下文对应一个微服务（美团交易系统、大众点评订单平台）

2. **战术设计（领域建模）**

### 1. 外卖/打车派单算法（距离、评分、抢单、指派）怎么优化？

**核心**：**多维度评分 + 实时索引 + 抢单/指派混合模式**，目标是“最优派单 + 最低空驶率”。

**2025–2026年主流优化方案**（美团外卖/滴滴打车真实做法）：

| 维度         | 优化手段                                                     | 技术实现                            | 效果提升            | 大厂典型用法                  |
| ------------ | ------------------------------------------------------------ | ----------------------------------- | ------------------- | ----------------------------- |
| **距离**     | 网格索引 + GeoHash + 路网距离                                | Redis Geo / Geohash + 路网预计算    | 距离计算从O(n)→O(1) | 美团/滴滴主力                 |
| **评分**     | 多因子加权模型（距离30% + 评分25% + 接单率20% + 预计时长15% + 优惠10%） | 实时机器学习打分（Flink + XGBoost） | 接单率提升15–25%    | 美团外卖、滴滴                |
| **抢单模式** | 附近骑手/司机广播 + 抢单限流                                 | Kafka广播 + Redis分布式锁           | 高峰期并发10w+      | 美团早期抢单                  |
| **指派模式** | 最优匹配（匈牙利算法简化版 / 贪心）                          | Flink实时计算 + 动态规划            | 空驶率降低20%       | 美团/滴滴2025主力（智能派单） |
| **混合**     | 抢单失败自动指派（3秒超时）                                  | Redis ZSET排序 + 定时任务           | 覆盖率99.9%         | 美团/饿了么标配               |

**高并发优化**：
- 骑手/司机位置每5秒上报 → Redis GeoHash分桶 + 本地缓存
- 订单进来 → 先本地匹配Top50 → 再Flink全局打分
- 兜底：兜底池（人工派单或备用骑手）

**常问追问**：高峰期怎么防骑手抢不到单？→ 动态调整广播半径 + 优先级队列。

### 2. 社交/feed流推荐（点赞、评论、转发）并发怎么设计？

**核心**：**推拉结合 + 聚合写 + 读扩散**，解决“写扩散爆炸 + 读延迟”。

**2025字节/小红书/抖音真实架构**：

| 模式         | 写方式                   | 读方式               | 并发能力 | 适用场景          | 大厂典型用法        |
| ------------ | ------------------------ | -------------------- | -------- | ----------------- | ------------------- |
| **推模式**   | 写扩散（写粉丝Timeline） | 直接读本地Timeline   | 写压力大 | 粉丝少（大V除外） | 早期微信朋友圈      |
| **拉模式**   | 只写自己发件箱           | 读时拉取关注人发件箱 | 读压力大 | 大V多             | 抖音/小红书主力     |
| **推拉结合** | 普通用户推 + 大V拉       | 读时合并本地 + 拉大V | 平衡     | 99%场景           | 字节/小红书2025标配 |

**高并发设计**（点赞/评论/转发）：
- **写端**：所有互动先写Kafka → Flink聚合（1秒批处理）→ Redis Sorted Set（时间+热度）
- **读端**：先读本地Redis Timeline（ZSET）→ 再拉大V最新10条 → 合并返回
- **计数**：Redis INCR + 异步落库（防热点）
- **防刷**：风控评分 + 设备指纹（参考前面防刷模块）
- **实时性**：WebSocket推送 + 长连接（亿级在线用Kafka + 消费分组）

**一句话总结**：**“普通用户推 + 大V拉 + Kafka聚合 + Redis ZSET”**，抖音Feed毫秒级返回。

### 3. 直播间礼物排行榜、实时在线人数怎么做？

**核心**：**Redis实时聚合 + 秒级落地**，允许短暂不一致。

**快手/抖音/淘宝直播2025主流方案**：

| 功能             | 存储与计算                        | 实时性 | 高并发方案                          | 大厂典型用法   |
| ---------------- | --------------------------------- | ------ | ----------------------------------- | -------------- |
| **礼物排行榜**   | Redis Sorted Set（礼物价值+时间） | 毫秒级 | 客户端批量上报 + 服务端1s聚合       | 快手/抖音主力  |
| **实时在线人数** | Redis INCR + HyperLogLog估算      | 秒级   | 心跳上报（5s一次）+ 本地缓存        | 淘宝直播、快手 |
| **组合优化**     | 本地Map聚合 + Redis原子操作       | <1s    | Kafka削峰 + Dragonfly/Redis Cluster | 所有直播平台   |

**具体流程**：
1. 礼物/在线心跳 → 先本地内存聚合（每台机器）
2. 每1秒pipeline刷Redis（ZINCRBY / INCR）
3. 排行榜读：直接ZREVRANGE（Top50）
4. 在线人数：HyperLogLog估算 + 精确计数兜底
5. 落地：每10秒异步写ClickHouse（事后分析）

**防刷**：礼物金额风控 + 设备指纹 + 限流。

### 4. 金融场景：对账、资金路由、幂等、防重、冻结解冻

**金融核心要求**：**强/最终一致 + 可审计 + 防重**。

**阿里/京东/支付宝2025真实做法**：

| 场景          | 方案                            | 核心技术                          | 一致性   | 大厂典型用法            |

| **幂等/防重** | 业务唯一ID + Redis + 状态机     | Redis Lua + 防重表 + 乐观锁       | 强一致   | 前面幂等模块 + 金融强化 |

### 5. IM/聊天系统：离线消息、已读回执、撤回、分布式存储

**腾讯/字节IM2025主流架构**（微信/企业微信风格）：

| 功能           | 方案                                   | 存储                            | 高并发/一致性 | 大厂典型用法   |
| -------------- | -------------------------------------- | ------------------------------- | ------------- | -------------- |
| **离线消息**   | 客户端拉取 + 服务端Push                | Redis List（最近100条）+ DB归档 | 亿级在线      | 微信/企业微信  |
| **已读回执**   | 服务端维护已读位点                     | Redis Bitmap / Sorted Set       | 最终一致      | 字节飞书、腾讯 |
| **撤回**       | 软删除 + 消息版本                      | DB + Redis缓存 + 广播           | 强一致        | 所有IM         |
| **分布式存储** | Kafka分区（按会话ID） + TiKV/Cassandra | Kafka持久化 + 多副本            | 高可用        | 快手/字节IM    |

**整体流程**：
- 发送 → Kafka → 消费端写Redis + DB
- 离线：用户上线拉取未读（Redis List + DB）
- 撤回/已读：发命令消息（特殊类型）→ 全员广播

**高并发优化**：会话分片（按userId % N）+ 多机房同步。
