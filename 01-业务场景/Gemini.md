# Gemini

在大厂（如阿里、腾讯、字节、美团等）的 Java 面试中，**业务场景题**主要考察候选人将技术理论落地到复杂业务逻辑的能力，通常没有标准答案，更看重方案的完备性、扩展性和对极端情况的处理。

为了方便你系统化复习，我将这些高频场景分为以下 **6 大类**：

------

## 1. 高并发与突发流量（秒杀/抢购类）

这类题目是“大厂标配”，核心考察点在于**抗压**、**限流**与**一致性**。

- **秒杀系统设计：** 如何设计一个抗住百万级 QPS 的秒杀系统？
- **超卖问题：** 在分布式环境下，如何通过技术手段保证库存不扣减为负数？
- **热点数据（Hot Key）：** 微博热搜或大促期间，某个商品瞬间成为极端热点，如何避免单点 Redis 挂掉？
- **防刷与风控：** 如何识别并拦截刷子脚本、黄牛抢购？



针对这一类“高并发与突发流量”的业务场景，面试官其实是在考查你对**分层过滤**和**保护核心资源**的理解。

面对百万级 QPS，核心思想只有八个字：**流量削峰、层层拦截**。

------

### 1. 秒杀系统设计：如何抗住百万级 QPS？

处理如此巨大的流量，绝对不能让请求直接冲击数据库。我们需要构建一个“漏斗型”架构：

- **客户端优化：** 资源静态化（CDN 缓存）、按钮置灰（防止重复点击）、动态链接隐藏（防止提前猜测 URL）。
- **接入层（网关/Nginx）：** 配置限流（如令牌桶算法），将 99% 的无效流量拦截在业务逻辑之外。
- **应用层（业务削峰）：** 使用消息队列（MQ）进行异步处理。用户请求进入后先入队，系统根据处理能力匀速消费，实现“流量削峰填谷”。
- **缓存层：** 读写全在 Redis 中完成，不直接操作 MySQL。

### 2. 超卖问题：分布式环境下如何保证库存一致性？

防止库存扣成负数（超卖）是秒杀最核心的挑战，通常有三种递进方案：

- **方案 A：数据库乐观锁（适合中低并发）**

  使用 SQL 判断：`UPDATE stock SET num = num - 1 WHERE id = 1 AND num > 0`。但在百万 QPS 下，数据库连接池会瞬间被打满。

- **方案 B：Redis Lua 脚本（推荐）**

  利用 Redis 的单线程原子性，将“查询库存 + 判断库存 + 扣减库存”写在一段 Lua 脚本中。这样可以保证操作的原子性，且性能极高。

- **方案 C：预扣库存 + 最终一致性**

  在 Redis 中预扣库存，异步发送 MQ 消息到数据库。若后续支付失败，再通过回滚机制返还 Redis 里的库存。

### 3. 热点数据（Hot Key）：如何避免单点 Redis 挂掉？

当全网用户都在抢同一个商品时，流量会集中在 Redis 的某个分片（Slot）上，导致单点处理能力达到瓶颈。

- **多级缓存（本地缓存）：** 在 JVM 进程内使用 Caffeine 或 Guava Cache 缓存热点商品信息，请求优先访问本地内存，不走网络 IO。
- **热点发现机制：** 使用监控组件（如阿里 Sentinel 或京东 hotkey）实时发现热点 Key。
- **Key 备份（散列化）：** 给热点 Key 加上随机后缀（如 `sku_1001_1`, `sku_1001_2`），将流量分散到多个 Redis 实例上。

### 4. 防刷与风控：如何识别并拦截黄牛？

如果 100 万流量里有 90 万是机器刷的，后端做得再好也是徒劳。

- **行为特征分析：** 同一 IP 或同一 UserID 在极短时间内发出大量请求，直接封禁。
- **验证码挑战：** 引入滑块验证、点击图中文字等交互，将流量由“突发”转为“匀速”。
- **设备指纹：** 通过前端采集设备唯一标识，识别出一台机器多开模拟器的行为。
- **黑名单机制：** 对接公司内部的风控系统，利用历史画像直接拦截已知黑灰产账号。

## 2. 数据一致性与分布式事务

当单机数据库无法承载业务时，分布式带来的数据一致性问题是面试重灾区。

- **双写一致性：** 数据库（MySQL）和缓存（Redis）的数据如何保证最终一致？先删缓存还是先更新数据库？
- **分布式事务：** 在微服务架构下，跨库转账如何保证 A 扣钱 B 一定加钱？（考察 TCC、Saga、二阶段提交等）
- **幂等性设计：** 用户在网络波动时点击了两次支付，或者 MQ 投递了两次消息，如何保证业务只处理一次？
- **订单状态机：** 复杂的订单流转（待支付、已支付、退款中...），如何防止状态跳变或并发更新？



分布式事务和数据一致性是架构设计的“深水区”。面试官不仅看你知不知道这些名词，更看你是否了解不同方案在**性能、一致性、实现复杂度**之间的权衡（Trade-off）。

------

### 1. 双写一致性：MySQL 与 Redis 如何同步？

这是一个经典的“CAP原则”折中问题。

- **先删缓存，再改数据库：** 风险在于高并发下，线程A删了缓存，线程B查询发现没缓存去读旧数据库并写回缓存，导致缓存永远是旧值。
  - *解决方案：* **延时双删**（更新完数据库后，过几百毫秒再删一次缓存）。
- **先改数据库，再删缓存（推荐）：** 风险在于数据库改成功了，但缓存删除失败。
  - *解决方案：* **消息队列重试** 或 **订阅 Binlog 异步删除**（使用 Canal 伪装成从库，监听到数据库变动后异步清理 Redis）。
- **强一致性要求：** 如果业务要求绝对一致，建议使用 **读写锁（ReentrantReadWriteLock）**，但这会牺牲性能。

### 2. 分布式事务：跨库转账如何保证 A 扣 B 加？

在微服务下，本地事务失效，需要引入分布式事务中间件（如 Seata）：

- **2PC / 3PC（强一致性）：** 准备阶段 + 提交阶段。性能差，有锁定资源时间过长的问题，互联网大促场景极少使用。
- **TCC (Try-Confirm-Cancel)：** 业务层的二阶段提交。需要人工写代码实现 Try（预留资源）、Confirm（确认执行）、Cancel（回滚释放）。
  - *场景：* 支付、核心账务系统。
- **Saga（最终一致性）：** 长事务拆分成多个本地事务，如果某个环节失败，则执行反向的“补偿操作”。
- **可靠消息最终一致性：** 利用消息队列（如 RocketMQ 事务消息）。A 扣钱成功后发消息，B 消费消息加钱，失败则不断重试。

### 3. 幂等性设计：如何防止重复处理？

幂等（Idempotency）的核心是：**同一个请求执行多次，结果必须相同。**

- **数据库唯一索引：** 最简单的方案。如流水号作为唯一索引，第二次插入直接报错。
- **Token 机制：** 提交前先申请一个 Token，提交时携带 Token，后端校验并立即在 Redis 中删除（原子操作）。
- **状态机判断：** 见下文。处理前先检查业务状态。
- **分布式锁：** 请求进来先占坑，处理完再释放。

### 4. 订单状态机：如何防止状态乱序？

订单状态多、并发高时，容易出现“已评价”居然在“已发货”之前到达的情况。

- **状态流转约束：** 在 SQL 更新时带上当前状态，例如：`UPDATE order SET status = 'PAID' WHERE id = 123 AND status = 'UNPAID'`。如果状态不匹配，更新行数为 0，直接报错。
- **版本号（Optimistic Locking）：** 给订单表加一个 `version` 字段，每次更新版本号加 1，防止并发覆盖。
- **分库分表下的处理：** 确保同一订单的所有操作通过 `order_id` 路由到同一个数据库分片，利用本地事务保证状态变更的原子性。

## 3. 海量数据存储与检索

这类题目考察你对“大数据量”的处理直觉，不仅仅是会写 SQL。

- **分库分表：** 数据量达到千万/亿级后，如何选择分片键（Sharding Key）？扩容时如何实现不停机数据迁移？
- **慢查询优化：** 在复杂的 B 端业务中，多表关联查询极慢，如何通过架构手段（如 ES、Cansal）解决？
- **分布式 ID：** 在分库分表环境下，如何生成全局唯一且趋势递增的 ID？
- **历史数据归档：** 几年前的订单数据如何处理？如何平衡查询性能与存储成本？

当数据量从百万级跨越到**亿级**，单机数据库的 B+ 树索引深度会增加，磁盘 I/O 成为瓶颈。这时候面试官考查的不再是简单的 SQL 调优，而是**架构设计和工程落地的能力**。

------

### 1. 分库分表：如何分？如何扩？

这是处理海量数据的核心手段。

- **分片键（Sharding Key）选择：**
  - **原则：** 必须能涵盖 80% 以上的核心查询场景，避免“全表路由”。
  - **电商示例：** 订单表通常按 `user_id` 取模。这样同一个用户的所有订单都在一个库/表里，方便查询“我的订单”。
  - **痛点：** 如果商家想查自己店铺的订单怎么办？**冗余方案：** 再存一份按 `merchant_id` 分片的“索引表”，或者同步到 ES。
- **不停机数据迁移（双写方案）：**
  1. **第一步：** 开启双写（旧库读写，新库只写），历史数据开始后台静默迁移（用 Canal 监听 binlog）。
  2. **第二步：** 校验新旧库数据一致性。
  3. **第三步：** 灰度切读，将部分流量切到新库。
  4. **第四步：** 停止旧库写入，彻底切换。

### 2. 慢查询优化：告别多表 Join

在分布式环境下，跨库 Join 几乎是不可能的。

- **宽表处理：** 针对 B 端复杂的报表查询，利用 **Canal** 监听各业务表的 Binlog，实时同步数据并进行“打平”处理，构建成一张包含所有字段的**宽表**。
- **搜索引擎（Elasticsearch）：** 将非核心事务查询全部路由到 ES。利用 ES 的倒排索引处理多维度组合筛选、全文检索和聚合统计。
- **读写分离：** 主库负责事务性写，从库或 ES 负责复杂读。

### 3. 分布式 ID：如何生成“唯一且有序”的身份证？

普通的自增 ID 在分表后会冲突，UUID 太长且无序（导致 B+ 树索引频繁分裂）。

- **雪花算法（Snowflake）：** 64 位 Long 型。由时间戳（保证递增）+ 工作机器 ID（保证唯一）+ 序列号（保证同毫秒并发）组成。这是目前大厂最通用的方案。
- **号段模式（Leaf/TinyID）：** 从数据库批量获取一段 ID 加载到内存。性能极高，但对数据库有短时间依赖。
- **Redis 生成：** 利用 `INCR` 命令，但存在单点风险及持久化延迟导致的重复风险。

### 4. 历史数据归档：冷热分离

并不是所有数据都值得留在昂贵的 SSD 分布式数据库里。

- **热数据（近 6 个月）：** 留在分库分表的 MySQL 或 TiDB 中，保证高性能交易。
- **冷数据（2 年前）：** 迁移到归档库（如压缩比极高的列式存储 ClickHouse，或者便宜的云存储 OSS/S3）。
- **查询代理层：** 在代码逻辑层做一层路由判断：如果用户查询的时间范围跨度太大，自动引导请求去冷数据库查询，并提示用户“历史订单加载较慢”。

## 4. 系统稳定性与高可用

考察系统在“出问题”时如何自保。

- **雪崩/击穿/穿透：** 缓存失效后的三重灾难如何应对？
- **限流与熔断：** 当下游接口响应慢时，如何防止把自己的线程池拖垮？
- **延迟队列：** 如何实现“订单下单 15 分钟未支付自动取消”？（考查时间轮、延时队列）
- **全链路压测：** 如何模拟真实流量来验证系统的极限？

在大厂面试中，“稳定性”是区分初中级与高级开发者的分水岭。这部分考察的是你对**防御性编程**和**故障自愈**的理解。

------

### 1. 缓存三兄弟：穿透、击穿、雪崩

这三个概念极易混淆，核心区别在于**谁不在了**：

- **缓存穿透（数据根本不存在）：** 恶意攻击查询 `id=-1` 的数据，请求直接打到数据库。
  - *对策：* **布隆过滤器 (Bloom Filter)** 快速拦截；或者对空结果也进行缓存，但过期时间设短一点。
- **缓存击穿（热点 Key 过期）：** 某个超热点 Key 突然过期，瞬间万级并发涌向数据库。
  - *对策：* 设置热点数据**永不过期**；或者加载数据时使用**互斥锁 (Mutex Lock)**，只允许一个请求去查库，其他等待。
- **缓存雪崩（大面积过期）：** 大量 Key 同时过期，或者 Redis 宕机，整个系统瘫痪。
  - *对策：* 过期时间加上**随机扰动值 (Jitter)**，避免集体失效；使用 Redis 集群提高可用性。

### 2. 限流与熔断：系统的“保险丝”

当外部压力过大或下游崩溃时，系统必须学会“断臂求生”。

- **限流 (Rate Limiting)：** 限制入口流量。
  - *算法：* **令牌桶**（允许短时间突发流量）或 **漏桶**（绝对平滑流出）。常用工具如 Sentinel 或 Guava RateLimiter。
- **熔断 (Circuit Breaker)：** 针对下游服务。当下游接口错误率超过阈值，直接触发熔断，后续请求快速失败（Return Fast），不再调用下游。
- **隔离 (Bulkhead)：** 核心思想是**线程池隔离**。比如给 A 业务分配 20 个线程，给 B 业务分配 30 个，防止 A 业务把整个系统的 CPU 和线程资源耗尽。

### 3. 延迟队列：订单 15 分钟超时取消

这题在考查你对“定时任务”在海量数据下的性能考量。

- **方案 A：死信队列 (Dead Letter Exchange)**
  - 利用 RabbitMQ 的消息 TTL（生存时间）。消息过期后进入死信队列，消费者监听死信队列处理逻辑。
- **方案 B：时间轮算法 (Time Wheel)**
  - 这是 Kafka、Netty 底层使用的算法。像时钟一样转动，刻度代表时间单位，每个刻度挂载任务链表。优点是插入和执行效率极高 ($O(1)$)。
- **方案 C：Redis ZSet**
  - 将订单号作为 `value`，过期时间戳作为 `score`。轮询 `ZRANGEBYSCORE` 获取到期的订单。

### 4. 全链路压测：如何在线上“演习”？

大厂为了迎接双 11，会直接在生产环境压测，但不能影响真实用户。

- **流量染色：** 在请求头标记 `is_test=true`。
- **影子库/表：** 压测流量产生的订单、支付数据，自动路由到专用的“影子数据库”（结构与线上一致），压测结束后一键清理。
- **挡板 (Mock)：** 对于无法真实压测的外部接口（如银联支付、短信接口），通过 Mock 挡板模拟返回成功。

## 5. 消息队列（MQ）实战

消息队列不只是异步，还有很多坑。

- **消息积压：** 消费者挂了，MQ 堆积了几千万条消息，如何快速消费掉？
- **消息丢失：** 消息从发送端到存储端再到消费端，哪个环节可能丢数据？如何实现可靠性传输？
- **顺序消费：** 比如 binlog 同步，如何保证消息按顺序处理？

在分布式架构中，消息队列（MQ）是系统的“呼吸机”和“缓冲池”。面试官问这类问题，主要是看你是否具备**处理线上事故的经验**以及对**中间件底层原理**的掌握。

------

### 1. 消息积压：千万级消息堆积怎么办？

这通常是由于消费者（Consumer）处理逻辑过慢或下游挂掉导致的。

- **临时扩容方案（最快）：**
  1. 先修复消费者的问题，确保不再增加积压。
  2. **增加 Topic 的 Partition（分区/队列）数量。**
  3. 新建一个临时的 Topic，并部署 10 倍于平时的消费者。
  4. 写一个临时的“分发程序”，把老 Topic 里的消息转发到新 Topic。
  5. 利用 10 倍的机器资源，在几个小时内把消息消化掉，最后切换回原架构。
- **服务降级：** 积压期间，如果消息不是核心业务（如日志、积分），可以考虑临时修改 Consumer 代码，直接丢弃非核心消息。

### 2. 消息丢失：如何保证 100% 可靠性传输？

一条消息的生命周期分为三个阶段，每个阶段都要防丢：

- **生产者阶段（Producer -> Broker）：**
  - *方案：* 使用 **带回调的发送接口**。开启 RabbitMQ 的 `confirm` 模式或 RocketMQ 的事务消息，确保 Broker 明确回复“已收到”。
- **存储阶段（Broker 内部）：**
  - *方案：* 开启 **持久化**（写入磁盘）。
  - *进阶：* 开启 **多副本同步**。只有当消息同步到从节点（Follower）后，才向生产者返回成功，防止主节点宕机导致数据丢失。
- **消费者阶段（Broker -> Consumer）：**
  - *方案：* **关闭自动 ACK，改为手动 ACK**。只有在业务逻辑真正执行完后，才告知 MQ 删掉这条消息。

### 3. 顺序消费：如何保证“先发先到”？

分布式环境下，消息往往是并发处理的，顺序很容易乱（例如：订单先创建、再支付、再完成）。

- **分区有序（局部有序）：**
  - *核心逻辑：* 将需要保证顺序的消息（如同一个 `order_id`）通过 **Hash 取模** 发送到同一个 Partition/Queue 中。
  - *原因：* 每一个分区内的消息是先进先出的。
- **单消费者处理：**
  - 确保一个分区只对应一个消费者实例，并且消费者内部不能开启多线程并发处理（否则线程间的执行顺序无法保证）。
- **异常重试处理：**
  - 如果某条消息处理失败，不能跳过它去处理下一条，必须停下来重试，直到成功，否则顺序会断掉。

### 4. MQ 选型对比（加分项）

大厂面试经常会问你为什么选这个 MQ：

| 特性 | RabbitMQ | RocketMQ | Kafka |

| :--- | :--- | :--- | :--- |

| **单机吞吐量** | 万级 | 十万级 | 百万级 |

| **时效性** | 微秒级（极快） | 毫秒级 | 毫秒级 |

| **可靠性** | 高 | 高（经过双11考验） | 一般（会有丢失风险） |

| **适用场景** | 中小型公司、对延迟敏感 | 金融级业务、大规模分布式系统 | 日志采集、大数据实时计算 |

### 1. 百万级实时排行榜：Redis ZSet 是最优解吗？

对于百万级用户的积分排行，直接用数据库 `ORDER BY` 会导致严重的磁盘扫描。

- **核心方案：Redis ZSet (Sorted Set)**。
  - **原理：** ZSet 内部使用**跳表 (SkipList)**，插入和查询的时间复杂度均为 $O(\log N)$。
  - **实现：** 使用 `ZADD` 更新分数，`ZREVRANGE` 获取前 N 名，`ZREVRANK` 获取用户当前排名。
- **痛点处理：**
  - **同分排名：** ZSet 默认按成员字典序排。如果要求同分时先达成的排前面，可以将 `分数 = 实际分数 + (未来某个时间戳 - 当前时间戳)`。
  - **超大规模（千万级以上）：** 建议**分片排行**。将用户 ID 按 Hash 路由到多个 ZSet 中，取全局前 10 名时，从每个分片各取前 10 名，在内存中聚合。

### 2. 附近的人/店铺：LBS（地理位置服务）

处理经纬度查询，核心挑战在于**二维空间索引**。

- **方案 A：Redis GeoHash（推荐）**
  - **原理：** 将二维的经纬度通过算法转换为一维的字符串（如 `wx4g0s`），字符串越接近，位置越近。
  - **API：** `GEOADD` 添加位置，`GEORADIUS` 查询半径 $N$ 公里内的点。
- **方案 B：Elasticsearch Geo Distance**
  - **适用场景：** 如果除了位置，还要过滤“人均消费”、“店铺类型”等多个维度。ES 支持复杂的地理位置聚合和过滤。
- **方案 C：四叉树 (Quadtree) / Google S2**
  - **原理：** 将地球表面不断切割成小格子。外卖平台常用 S2 算法进行蜂窝状的服务区覆盖判断。

### 3. 全链路灰度发布：如何精准控制流量？

灰度发布（Canary Release）不仅仅是部署两个版本，关键在于**流量染色**和**环境隔离**。

- **流量染色（网关层）：**
  - 在最外层网关（如 Spring Cloud Gateway / Kong）根据用户 ID、城市、设备 ID 或特殊的 Header 命中 1% 的流量。
  - 给这些请求打上一个标签：`gray_version = true`。
- **全链路透传：**
  - 使用 **ThreadLocal** 或 **分布式链路追踪 (SkyWalking/Jaeger) 的 Context**，将灰度标签在整个微服务链路中传递。
- **路由决策：**
  - 每个微服务的负载均衡器（如 Ribbon/LoadBalancer）根据标签判断：如果有灰度标签，优先访问灰度版本的服务实例（V2.0），否则访问稳定版（V1.0）。
- **数据库隔离（进阶）：**
  - 对于涉及数据Schema变更的灰度，可能需要配合“影子表”或逻辑字段区分，防止灰度版本产生的测试数据污染正式生产环境。
