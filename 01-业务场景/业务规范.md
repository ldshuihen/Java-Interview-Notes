# 标准通用响应

日常开发里，**后端统一返回 Response（响应体）** 分**通用基础字段**、**扩展业务字段**，分主流格式给你整理，附示例。

## 一、标准通用响应（前后端最常用）

### 1. 基础四件套（必选）

| 字段名    | 类型        | 说明                                   |
| --------- | ----------- | -------------------------------------- |
| `code`    | 数字/字符串 | 状态码（核心）                         |
| `message` | 字符串      | 提示信息                               |
| `data`    | 任意类型    | 业务数据（成功返回主体，失败可为null） |
| `success` | 布尔        | 简化判断：true=成功，false=失败        |

#### 示例 JSON

```json
// 成功响应
{
  "success": true,
  "code": 200,
  "message": "操作成功",
  "data": {
    "id": 1001,
    "name": "测试数据"
  }
}

// 失败响应
{
  "success": false,
  "code": 500,
  "message": "服务器内部错误",
  "data": null
}
```

---

### 2. 分页场景 额外字段（列表接口必加）

分页接口在基础上增加：
- `total`：总数据条数
- `pages`：总页数
- `current`：当前页码
- `size`：每页条数
- `records`：当前页数据列表

**分页示例**
```json
{
  "success": true,
  "code": 200,
  "message": "查询成功",
  "data": {
    "current": 1,
    "size": 10,
    "total": 56,
    "pages": 6,
    "records": [
      {"id":1,"name":"A"},
      {"id":2,"name":"B"}
    ]
  }
}
```

---

## 二、常见扩展字段（根据项目按需加）

1. **timestamp / ts**
   时间戳（服务器响应时间，防缓存、日志排查）
2. `traceId`
   链路追踪ID（微服务/日志排查问题必备）
3. `error` / `errors`
   详细错误信息、字段校验错误（表单校验专用）
4. `token` / `tokenHead`
   登录/刷新接口返回令牌
5. `warn`
   警告信息（非报错，只是提醒）

---

## 三、主流状态码规范（配套 code 使用）

### HTTP 风格码（最通用）

- `200`：请求成功
- `400`：参数错误
- `401`：未登录/Token失效
- `403`：权限不足
- `404`：接口不存在
- `500`：服务异常

#### 自定义业务码（大型项目）

如 `10001=账号不存在`、`10002=密码错误` 等。

---

## 四、Java 实体类模板（SpringBoot 直接用）

```java
@Data
public class Result<T> {
    private Boolean success;
    private Integer code;
    private String message;
    private T data;

    // 成功静态方法
    public static <T> Result<T> success(T data) {
        Result<T> r = new Result<>();
        r.setSuccess(true);
        r.setCode(200);
        r.setMessage("成功");
        r.setData(data);
        return r;
    }

    // 失败静态方法
    public static <T> Result<T> fail(Integer code, String message) {
        Result<T> r = new Result<>();
        r.setSuccess(false);
        r.setCode(code);
        r.setMessage(message);
        r.setData(null);
        return r;
    }
}
```

---

## 总结

- **简单接口**：`success + code + message + data` 足够
- **分页接口**：多加 `current/size/total/records`
- **微服务/线上项目**：额外加 `traceId、timestamp`
- **表单校验**：增加 `errors` 字段



# 响应格式

---

## 1. 为什么会用 `result` 代替 `data`？

这两个字段**作用一模一样**：
- 都是放**接口返回的业务数据**
- 只是命名风格不同

### 最常见两种响应格式

#### ① 用 data（国内最主流）

```json
{
  "success": true,
  "code": 200,
  "message": "查询成功",
  "data": {
    "id": 1,
    "name": "张三"
  }
}
```

#### ② 用 result（国外/老项目/简洁风格）

```json
{
  "success": true,
  "code": 200,
  "message": "查询成功",
  "result": {
    "id": 1,
    "name": "张三"
  }
}
```

---

## 2. 两者区别（一句话）

**data = 数据**
**result = 结果**

**功能完全一样，只是名字不同。**

---

## 3. 什么时候用 result 更合适？

这些场景大家更喜欢用 `result`：
- 接口返回**单一结果**（如登录结果、计算结果）
- 前端习惯用 `res.data.result`
- 老项目、PHP / Python / Node 项目
- 追求简洁、语义化

---

## 4. 统一响应实体类（支持 result 版）

如果你想项目里用 `result` 替代 `data`，实体类这样写：

```java
@Data
public class ApiResponse<T> {
    private Boolean success;
    private Integer code;
    private String message;
    private T result; // 这里用 result 代替 data

    // 成功
    public static <T> ApiResponse<T> success(T data) {
        ApiResponse<T> r = new ApiResponse<>();
        r.setSuccess(true);
        r.setCode(200);
        r.setMessage("操作成功");
        r.setResult(data);
        return r;
    }

    // 失败
    public static <T> ApiResponse<T> fail(int code, String msg) {
        ApiResponse<T> r = new ApiResponse<>();
        r.setSuccess(false);
        r.setCode(code);
        r.setMessage(msg);
        r.setResult(null);
        return r;
    }
}
```

---

## 5. 行业小总结

- **国内公司 90% 用 `data`**
- **国外/开源项目常用 `result`**
- **功能完全一样，不影响逻辑**

---

## 最终结论

你完全可以放心：
**用 data 或 result 都可以，项目统一就行，没有对错。**
你看到别人用 `result` 替换 `data`，就是这个原因。

# .gitkeep 文件的作用
`.gitkeep` 是**Git 版本控制工具**里的一个**约定俗成的空文件**，**不是 Git 官方命令/配置**，只是开发者通用的小技巧。

## 核心作用
**让 Git 追踪并提交空文件夹**。

---

### 为什么需要它？
Git 有一个硬性规则：
> **Git 只追踪文件，不追踪空文件夹**
> 空文件夹无法被 `git add`、无法提交到仓库，克隆仓库时也不会被创建。

但实际项目中，**有些空文件夹必须保留**，比如：
- 日志目录 `logs/`
- 上传文件目录 `uploads/`
- 临时文件目录 `tmp/`
- 预留的目录结构（后续会放文件）

如果不做处理，这些空文件夹会被 Git 忽略，团队协作时别人克隆代码就看不到这些目录。

---

### 怎么用？
1. 在**需要保留的空文件夹**里，新建一个名为 `.gitkeep` 的空文件
2. 正常提交这个文件到 Git 即可

示例结构：
```
project/
├── logs/
│   └── .gitkeep   ✅ 保留空文件夹
├── uploads/
│   └── .gitkeep
└── src/
```

---

### 重要补充
1. **文件名固定**：必须叫 `.gitkeep`，这是全球通用约定
2. **文件内容**：必须是空文件，不需要写任何内容
3. **和 .gitignore 无关**：
   - `.gitignore` 是**忽略文件**
   - `.gitkeep` 是**保留空文件夹**
4. 文件夹一旦有了真实文件，`.gitkeep` 可以删除（不删也不影响）

---

### 总结
1. **Git 不识别空文件夹**，空目录无法提交
2. **.gitkeep 是空文件**，放在空目录里让 Git 追踪目录
3. 用于保留项目必须的空目录结构（日志、上传、临时文件夹等）

# md签名

`md5(AK+SK+timestamp)` 这种签名方式**几乎专门用于：服务端 ↔ 服务端**的接口鉴权，**绝对不会用在普通前端/客户端**。

我给你用最清晰、最直白的方式讲清楚👇

---

## 1. 它到底是什么？

这是**服务间接口签名认证**的标准写法：
- **AK** = Access Key（公钥，类似用户名）
- **SK** = Secret Key（密钥，类似密码，**绝不传输**）
- **timestamp** = 时间戳（防重放）
- **md5** = 把三者拼接后做哈希，生成签名串

最终请求一般长这样：
```
?ak=xxx&timestamp=123456&sign=md5(ak+sk+timestamp)
```

---

## 2. 为什么说它**一定是服务间使用**？

因为：
### ✔ **SK 密钥绝对不能暴露在前端/APP/小程序里**

前端代码会被反编译、抓包，SK 一旦泄露，别人就能伪造请求。

### ✔ 这种签名逻辑**只能在后端安全执行**

调用方是：
- 你的后端服务
- 第三方公司后端
- 内部微服务
- 开放平台接口调用

**普通用户端（H5/小程序/App）绝对不会用这种方式。**

---

## 3. 典型使用场景（你一看就懂）

- 内部微服务调用鉴权
- 开放平台 API 授权（像支付宝、微信支付、阿里云 API 都是这种风格）
- 企业间系统对接
- 供应链 / 财务 / 费控系统之间的接口（你之前的 Capex 场景非常典型）

---

## 4. 一句话总结

**md5(AK+SK+timestamp) = 服务与服务之间的“接口钥匙”**
不是给用户用的，是**给系统用的**。

# **服务间接口签名（AK+SK+签名）** 用在**场景**

---

## 一、**最核心、最常见的 5 大使用场景**

### 1. **企业内部微服务之间调用（最常用）**

比如：
- 订单服务 → 调用库存服务
- 采购系统 → 调用财务系统
- 用户中心 → 调用权限中心

**为什么用？**
内部服务不能裸奔，必须验证“谁在调用我”。

---

### 2. **第三方系统对接（企业间接口）**

比如：
- 你们公司对接 **银行、支付公司、物流、供应商系统、ERP 厂商**
- 别人调用你们的开放接口，你们给对方发 **AK/SK**

这是**企业对接标准方案**。

---

### 3. **开放平台 / Open API**

阿里云、腾讯云、微信支付、支付宝、抖音开放平台，**全部用这种方式**。

结构一模一样：
```
ak + sk + 时间戳 + 签名
```

---

### 4. **后台管理系统、运维系统、内部工具**

比如：
- 运营后台调用核心服务接口
- 数据大屏拉取数据
- 定时任务、脚本调用接口

---

### 5. **不使用 Session / Cookie 的无状态鉴权**

服务集群、分布式、跨域、跨机房场景，**不能用 Cookie**，
只能用 **AK/SK 签名**。

---

## 二、**什么时候必须用这种签名方式？（判断口诀）**

只要满足下面**任意一条**，就必须用签名：

1. **调用方是后端服务、脚本、定时任务、第三方系统**
2. **不能登录、没有用户态、没有 Cookie**
3. **接口很重要，不能被随便调用**
4. **需要防篡改、防重复提交**
5. **跨公司、跨部门、跨系统对接**

---

## 三、**一句话总结（最精准）**

### **服务间签名 = 后端与后端之间的“身份证 + 门禁卡”**

**凡是没有用户登录、没有前端页面、系统与系统对话的场景，都用它。**
