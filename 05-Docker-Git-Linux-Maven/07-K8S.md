#### 二、K8s 核心架构（必须理解）

先别安装。

先理解。

------

##### 1. Pod

K8s 最小运行单位。

不是容器。

而是：

```text
Pod = 一个或多个容器
```

例如：

```text
一个Java服务
+
一个日志sidecar
```

放一个 Pod。

------

##### 2. Node

节点（服务器）。

比如：

```text
Node1
Node2
Node3
```

每个 Node 上跑很多 Pod。

------

##### 3. Cluster

整个 K8s 集群。

------

##### 4. Deployment

管理 Pod。

例如：

```yaml
我要3个订单服务实例
```

Deployment 自动维护。

------

##### 5. Service

服务发现。

因为：

```text
Pod IP 会变
```

Service 提供固定访问入口。

#### 十八、最核心 YAML 示例

------

##### deployment.yaml

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: nginx-deployment

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
        - name: nginx
          image: nginx
          ports:
            - containerPort: 80
```

------

##### 应用 YAML

```bash
kubectl apply -f deployment.yaml
```

### `Docker` 和 `K8s`

经常一起出现，但它们解决的问题完全不同。

你可以把它们理解成：

- Docker：负责“装箱”
- K8s：负责“管理很多箱子”

------

#### 一、先理解 Docker 是什么

Docker 本质上是：

> 一个“容器运行平台”

它解决的问题：

##### 传统部署的问题

以前部署 Java 项目：

```text
开发环境：
JDK17 + Maven3.9 + MySQL8

测试环境：
JDK11

生产环境：
JDK8
```

结果：

```text
开发能跑
上线就炸
```

经典问题：

```text
“在我电脑上是好的”
```

------

##### Docker 的核心思想

Docker 把：

- JDK
- 项目
- 依赖
- 环境

全部打包成一个“容器”。

类似：

```text
一个可移动的运行箱子
```

无论在哪：

- Linux
- 云服务器
- 本地
- 测试环境

都能一致运行。

------

#### 二、Docker 的核心能力

Docker 主要干：

##### 1. 容器化

比如：

```bash
docker run nginx
```

启动一个 Nginx 容器。

------

##### 2. 环境隔离

每个容器：

- 文件系统隔离
- 网络隔离
- 进程隔离

类似轻量虚拟机。

但：

> 比虚拟机轻很多

因为：

Docker 共享宿主机内核。

------

##### 3. 镜像机制

Docker Image：

```text
镜像 = 模板
容器 = 运行实例
```

例如：

```text
Java镜像
MySQL镜像
Redis镜像
```

------

#### 三、Docker 的问题

Docker 非常适合：

- 单机
- 小项目
- 开发环境

但企业级会出现问题。

------

#### 四、假设有 100 个容器

比如：

```text
订单服务：20个容器
支付服务：30个容器
用户服务：50个容器
```

这时候会出现：

------

##### 问题1：容器挂了怎么办？

```text
docker stop xxx
```

服务没了。

谁自动恢复？

Docker 不负责。

------

##### 问题2：如何扩容？

双十一来了：

```text
订单服务从20扩成200
```

Docker 没有“集群调度能力”。

------

##### 问题3：容器放哪台机器？

你有：

```text
10台服务器
```

哪个容器：

- 放机器1？
- 放机器5？
- 如何均衡？

Docker 不管。

------

##### 问题4：服务通信

```text
user-service
↓
order-service
↓
pay-service
```

IP 经常变化。

如何服务发现？

Docker 解决不了。

------

#### 五、K8s 是什么

Kubernetes 是：

> 容器编排平台（Container Orchestration）

它是：

```text
管理大量 Docker 容器的系统
```

K8s 不负责创建容器技术。

它负责：

```text
管理容器集群
```

------

#### 六、K8s 的核心能力

------

##### 1. 自动部署

你告诉 K8s：

```yaml
我要运行：
订单服务 3个实例
```

K8s 自动创建。

------

##### 2. 自动扩缩容

高峰期：

```text
CPU > 80%
```

K8s 自动：

```text
3个实例
→
30个实例
```

低峰：

```text
30
→
3
```

------

##### 3. 自动恢复

某个容器挂了：

```text
K8s 自动拉起新容器
```

无需人工。

------

##### 4. 服务发现

K8s 内部：

```text
order-service
pay-service
```

自动 DNS。

服务间直接调用。

------

##### 5. 负载均衡

请求自动分发：

```text
用户请求
↓
10个订单服务实例
```

K8s 自动均衡。

------

##### 6. 滚动更新

升级系统：

```text
v1 → v2
```

K8s：

- 先启动新版本
- 再关闭旧版本

实现：

```text
不停机升级
```

------

#### 七、最核心区别

| 对比     | Docker   | K8s          |
| -------- | -------- | ------------ |
| 本质     | 容器引擎 | 容器编排平台 |
| 作用     | 运行容器 | 管理容器集群 |
| 适合     | 单机     | 大规模集群   |
| 自动恢复 | 不擅长   | 强           |
| 自动扩容 | 没有     | 强           |
| 服务发现 | 弱       | 强           |
| 负载均衡 | 很弱     | 强           |
| 滚动更新 | 基础     | 企业级       |
| 集群管理 | 无       | 核心能力     |

------

#### 八、关系图（非常重要）

```text
应用
 ↓
Docker（负责容器）
 ↓
Kubernetes（负责管理容器）
 ↓
服务器集群
```

所以：

> K8s 通常是管理 Docker 容器的。

虽然现在：

K8s 已逐渐不直接依赖 Docker。

底层更多使用：

- containerd
- CRI-O

但思想完全一致。
