# Web 实时消息推送

Web 实时消息推送核心是**服务器主动向浏览器发送数据**，主流方案按演进顺序为：**短轮询 → 长轮询 → SSE → WebSocket**，另有 **MQTT+WebSocket、WebHook** 等扩展方案。下面从原理、代码、优缺点、选型做完整详解。

---

## 一、短轮询（Short Polling）

**原理**：客户端定时发 HTTP 请求，服务器立即响应（有无更新都返回）。


**前端代码**
```javascript
// 每秒拉一次
setInterval(async () => {
  const res = await fetch('/api/messages');
  const data = await res.json();
  render(data);
}, 1000);
```

**优点**
- 实现最简单、兼容性 100%
- 无需特殊服务端支持

**缺点**
- 延迟高（取决于间隔）
- 大量无效请求、浪费带宽与服务器资源

**适用场景**
- 实时性要求低、用户量小
- 简单页面数据刷新

---

## 二、长轮询（Long Polling）
**原理**：客户端发请求 → 服务器**挂起不响应** → 有数据/超时才返回 → 客户端收到后**立即重连**。


**伪代码流程**
```
客户端 → 请求 → 服务器（挂起等待）
→ 有新消息 → 响应 → 客户端处理 → 立刻重发请求
→ 超时（如30s）→ 响应空 → 立刻重发请求
```

**优点**
- 比短轮询**减少无效请求**、实时性更好
- 兼容老浏览器、无需新协议

**缺点**
- 服务器需维持大量挂起连接，**资源占用高**
- 编程与运维复杂（超时、并发控制）

**适用场景**
- 兼容老旧环境、无法用 WebSocket/SSE
- 通知、简单 IM 等中等实时需求

---

## 三、SSE（Server-Sent Events）
**原理**：基于 HTTP 长连接，**单向（服务器→客户端）** 流式推送，MIME `text/event-stream`。


**前端代码（原生 EventSource）**
```javascript
const es = new EventSource('/api/sse');
es.onmessage = (e) => {
  console.log('消息：', e.data);
};
es.onerror = (err) => {
  console.error('SSE 错误', err);
};
```

**服务端（Node.js 示例）**
```javascript
// Express
app.get('/api/sse', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');
  // 每隔2秒推一条
  setInterval(() => {
    res.write(`data: ${new Date().toISOString()}\n\n`);
  }, 2000);
});
```

**优点**
- **原生自动重连**、API 极简
- 基于 HTTP、易穿透代理、兼容 HTTP2
- 轻量、仅维护单方向连接

**缺点**
- **仅单向**（客户端发消息需另发 AJAX）
- 部分老浏览器（IE）不支持

**适用场景**
- 日志流、股票行情、新闻推送、系统通知
- 只需服务端单向推送的场景

---

## 四、WebSocket（主流首选）
**原理**：HTTP 握手升级 → 建立**全双工持久连接**（ws/wss），双方随时互发数据。


**握手关键请求头**
```
Connection: Upgrade
Upgrade: websocket
Sec-WebSocket-Key: xxx
```

**前端代码**
```javascript
const ws = new WebSocket('wss://your.domain/ws');
ws.onopen = () => ws.send('hello');
ws.onmessage = (e) => console.log('收到：', e.data);
ws.onerror = (err) => console.error(err);
ws.onclose = () => { /* 重连逻辑 */ };
```

**优点**
- **全双工、极低延迟、高吞吐**
- 支持文本/二进制、消息边界清晰
- 现代浏览器全覆盖、标准成熟

**缺点**
- 需独立协议处理、**服务器连接开销较大**
- 要自己处理：心跳、断线重连、跨域、消息有序

**适用场景**
- 聊天、在线游戏、协同编辑、直播弹幕、实时交易

---

## 五、技术方案对比表
| 技术          | 通信方向      | 协议   | 延迟       | 资源 | 重连     | 适用场景         |
| ------------- | ------------- | ------ | ---------- | ---- | -------- | ---------------- |
| 短轮询        | 半双工        | HTTP   | 高（秒级） | 极高 | 手动     | 低实时、小用户   |
| 长轮询        | 半双工        | HTTP   | 中（亚秒） | 高   | 手动     | 兼容老环境       |
| **SSE**       | 单向（服→客） | HTTP   | 低（毫秒） | 中   | **自动** | 通知、行情、日志 |
| **WebSocket** | 全双工        | WS/WSS | **极低**   | 中   | 手动     | 聊天、游戏、协作 |

---

## 六、进阶方案
### 1. MQTT over WebSocket
- 轻量发布订阅、适合 IoT/大量设备
- 常用库：Paho MQTT、EMQX

### 2. 成熟推送服务
- 第三方：Firebase Cloud Messaging、极光、个推
- 云厂商：阿里云消息推送、腾讯云 TIM

### 3. 高并发架构（生产常用）
```
客户端 → LB → WebSocket/SSE 接入层 → Redis Pub/Sub / Kafka → 业务服务
```
- 接入层水平扩展
- 消息队列削峰、广播、持久化

---

## 七、选型建议
1. **只需单向推送、简单轻量** → **SSE**
2. **双向强交互、低延迟** → **WebSocket**
3. **兼容 IE/老旧环境** → **长轮询**
4. **极低实时要求、快速原型** → **短轮询**

---

## 八、常见问题与实践要点
- **跨域**：WebSocket/SSE 均受跨域限制，服务端配置 `Access-Control-Allow-Origin`
- **断线重连**：WebSocket 必须自己实现；SSE 浏览器自动处理
- **心跳保活**：WebSocket 定时发 ping，避免代理断开连接
- **消息可靠性**：重要消息需**序号、ACK、重发、去重**机制

# Java 定时任务 

## 整体总览（全分类，无遗漏）

Java 定时任务按**实现形态 & 适用场景**分为五大类，覆盖你关心的：**单机原生、Spring 单机、Redis 实现、MQ 延迟/定时、分布式自研、专业调度框架**，完整闭环：
1. 单机原生定时（JDK 基础）
2. Spring 单体定时（Spring Task）
3. 基于 Redis 实现分布式定时/延迟任务
4. 基于 MQ 实现定时/延迟任务（RabbitMQ/RocketMQ/Kafka）
5. 分布式统一任务调度（简易自研 + 专业框架）

核心核心区分：
- **周期定时**：每隔5分钟、每天凌晨2点执行（报表、数据同步、对账）
- **延迟任务**：下单30分钟未支付自动关单、短信延时通知（非固定周期）

---

## 一、单机定时任务（单实例部署）

### 1.1 JDK 原生 Timer / TimerTask

#### 原理

基于单线程+最小堆任务队列，串行执行任务。
#### 缺陷

1. 单线程阻塞：一个任务卡顿/异常，所有任务全部卡死；
2. 异常无隔离：任务抛异常，调度器直接停止；
3. 不支持集群、无重试、无监控。
#### 结论

**彻底淘汰**，仅作技术了解。

### 1.2 JDK 原生 ScheduledExecutorService（推荐单机原生）

#### 原理

线程池 + 延迟队列，多线程调度，任务隔离。
#### 核心两种调度模式

1. `scheduleAtFixedRate`：**固定频率**
   不受上一次执行耗时影响，到时间强制执行，适合短耗时任务。
2. `scheduleWithFixedDelay`：**固定间隔**
   上一次任务执行完毕后，再间隔指定时间执行，适合长耗时任务。
#### 优点

线程隔离、异常互不影响、性能稳定、无框架依赖。
#### 缺点

集群多实例下**重复执行**，无分布式协调能力。

### 1.3 Spring Task（@Scheduled）【单体项目最常用】

#### 核心使用

启动器加 `@EnableScheduling`，业务方法注解声明规则：
- `cron`：灵活自定义周期（年月日时分秒）
- `fixedRate`：固定频率
- `fixedDelay`：执行完成后固定间隔
#### 优点

接入极简、Spring 原生集成、无需额外中间件。
#### 致命短板

1. 集群/多实例部署，**每台机器都会独立执行 → 任务重复**；
2. 无失败重试、无任务日志、无可视化、无故障转移；
3. 不支持任务分片、动态启停。

---

## 二、基于 Redis 实现 分布式定时 & 延迟任务

适用场景：**轻量分布式、延迟类业务、不想引入重型调度框架**
### 2.1 Redis Key 过期监听

#### 原理

1. 业务数据新增时，写入带过期时间的 Redis Key；
2. 开启 Redis 配置：`notify-keyspace-events Ex`；
3. 服务订阅 Key 过期事件，监听回调执行任务。
#### 优缺点

- 优点：实现简单、无额外编码；
- 缺点：时间不准（惰性删除）、任务易丢失、无法重试、不支持长周期任务。

### 2.2 Redis ZSet 有序集合（自研延迟任务 工业级方案）

#### 核心原理

1. ZSet 的 `score` 存储**任务执行时间戳**；
2. 后台定时轮询，拉取 `score ≤ 当前时间` 的到期任务；
3. 消费成功后删除任务，保证单次执行；
4. 多实例通过 Redis 原子命令抢占，避免重复。
#### 优点

时间精准、支持取消任务、可持久化、轻量化、无中间件依赖。
#### 缺点

需要手动封装轮询、ACK、失败重试、防重复逻辑。

### 2.3 Redisson 延迟队列（Redis 方案最优解）

#### 原理

基于 Redis 封装 **延迟队列**，底层整合 ZSet+阻塞队列，开箱即用。
#### 核心能力

- 支持指定延迟时间；
- 天然分布式、多实例消费互斥；
- 自带持久化、任务不丢失、支持重试。
#### 适用

中小型项目、订单超时、延时提醒等**短中期延迟任务**。

---

## 三、基于 MQ 实现 定时 & 延迟任务

MQ 方案核心价值：**解耦、削峰、高可靠、高并发、天然异步**，是互联网延迟业务标准方案。

### 3.1 RabbitMQ

1. **DLX+TTL 死信队列（经典方案）**
   消息设置过期时间，到期未消费转入死信队列，业务消费死信实现延迟。
   缺点：队列级 TTL，无法精准控制单条消息延迟时间。
2. **延迟交换机插件**
   安装插件后，直接发送带延迟时长的消息，精准度更高，推荐使用。

### 3.2 RocketMQ（MQ 定时/延迟 最强方案）

#### 核心能力

1. **原生支持定时消息 + 延迟消息**，无需额外改造；
2. 内置多级延迟等级，也支持自定义时间戳；
3. 自带重试机制、死信队列、事务消息，保证数据一致性。
#### 适用

生产级高并发、高可靠延迟场景：订单关闭、支付超时、营销延时推送。

### 3.3 Kafka

Kafka **无原生延迟/定时能力**，无官方支持。
只能通过业务层手动实现：时间戳过滤、分区时间分片轮询，**不推荐**做定时/延迟任务。

---

## 四、分布式简易方案（解决集群重复执行）

### 4.1 SpringTask + 分布式锁（最低成本改造）

#### 原理

保留原有 `@Scheduled` 定时逻辑，执行前加**分布式锁**：
- Redis 分布式锁（Redisson 推荐）
- MySQL 唯一索引/行级锁
- Zookeeper 临时节点锁
#### 执行逻辑

多实例同时触发定时，只有一台机器抢到锁并执行，其余直接放弃。
#### 优点

改造成本极低、无需引入新框架、无缝兼容老项目。
#### 缺点

无任务管理、无监控、无分片、无告警，只解决**重复执行**单一问题。

### 4.2 Quartz 集群版

#### 原理

老牌定时框架，集群模式基于 **数据库锁** 抢占执行权，多节点互斥。
#### 优缺点

- 优点：cron 能力强大、成熟稳定、支持集群；
- 缺点：无可视化控制台、运维困难、分片能力弱、代码侵入高。

---

## 五、专业分布式任务调度框架（企业级周期定时首选）

针对**复杂周期任务、批量数据处理、定时运维、任务分片、监控告警**场景，统一解决方案。

### 5.1 XXL-JOB（国内最主流、首选）

#### 架构

调度中心(Admin) + 执行器(Executor) 分离架构，中心化调度。
#### 核心特性

1. 可视化 Web 管理：任务增删改查、启停、日志查看；
2. 集群高可用：调度中心、执行器均可集群部署；
3. 路由策略、分片广播、失败重试、邮件/钉钉告警；
4. 接入简单、文档完善、社区成熟、适配所有Spring项目。
#### 适用

绝大多数企业级**周期性定时任务**：日终对账、数据同步、报表统计。

### 5.2 Elastic-Job（当当开源）

#### 架构

无中心化，基于 Zookeeper 做服务注册、任务协调。
#### 核心优势

**动态分片能力极强**，天然适合海量数据拆分处理、弹性扩容。
#### 缺点

依赖 ZK、运维复杂、接入成本高于 XXL-JOB。

### 5.3 PowerJob（新一代高性能框架）

#### 核心亮点

无锁化调度、DAG 任务编排、MapReduce 分片、云原生适配、性能更强。
#### 适用

复杂任务依赖、大规模集群、高性能定时调度场景。

## 七、最终业务选型规则（直接落地）

1. **单体项目、无集群**
直接使用：**Spring Task**

2. **集群部署、原有定时不能重构、只防重复**
使用：**SpringTask + Redisson 分布式锁**

3. **延迟类业务（关单、提醒、延时推送）**
- 小项目、无MQ：**Redisson 延迟队列**
- 中大型、高并发：**RocketMQ 定时消息**
- 已有RabbitMQ：**RabbitMQ 延迟插件/DLX**

4. **周期批处理（每天/每小时定时、报表、同步、对账）**
禁止用 Redis/MQ，统一选用：**XXL-JOB**

5. **海量数据、需要任务分片、弹性扩容**
选用：**Elastic-Job**

6. **有任务依赖、流程编排、复杂DAG场景**
选用：**PowerJob**

## 八、核心概念最终区分

1. **周期定时**
固定时间重复执行 → SpringTask、Quartz、XXL-JOB、Elastic-Job
2. **延迟任务**
事件触发后延后执行 → Redis、MQ 所有方案
3. **分布式核心矛盾**
单机定时 = 简单；集群定时 = 防重复 + 高可用 + 运维管控
4. **Redis vs MQ vs 调度框架**
- Redis：轻量、低成本，主打**延迟**
- MQ：高可靠、解耦，主打**异步延迟**
- 调度框架：专业化、可运维，主打**周期定时**

# 数据安全 

## 一、核心定义
**数据安全**：通过**技术+管理+规范**，保障数据在**采集、传输、存储、使用、共享、销毁**全生命周期内的
- 机密性（不泄露）
- 完整性（不篡改）
- 可用性（不丢失、可访问）
- 不可否认性（操作可追溯）

覆盖范围：用户隐私数据、业务数据、配置密钥、日志数据、数据库数据、文件/OSS数据、接口传输数据等。

---

## 二、数据安全 全生命周期划分（核心框架）
1. 数据采集安全
2. 数据传输安全
3. 数据存储安全
4. 数据使用&访问安全
5. 数据共享&对外输出安全
6. 数据销毁&脱敏下线安全

> 所有数据安全方案，都围绕这6个环节落地。

---

## 三、1. 数据采集安全
### 核心风险
- 违规采集、过度收集用户隐私
- 前端明文提交、参数篡改、越权提交
- 敏感数据未做入参校验

### 解决方案
1. **最小采集原则**
只收集业务必需字段，禁止过度收集手机号、身份证、住址、人脸等。
2. 前端参数校验 + 后端二次强校验
防止恶意参数、SQL注入、XSS、非法特殊字符。
3. 敏感数据前端初步脱敏
手机号、身份证、银行卡前端隐藏展示。
4. 接入隐私合规规范

## 敏感词过滤方案

核心目标：对用户输入（评论、昵称、内容、私信等）进行检测，拦截/替换违规词汇（色情、暴力、政治敏感、辱骂、广告等），保障内容合规，规避法律和平台风险。

核心评价维度：**检测速度、准确率、扩展性、维护成本、误判率**，不同方案适配不同业务场景（小流量单体、高并发分布式、多语言多场景）。

### 一、敏感词过滤 核心原理（通用）

所有方案本质都是「关键词匹配」，核心差异在于「匹配算法」和「存储方式」，核心流程：

1. 敏感词预处理：去重、归一化（如全角转半角、大小写统一、繁简体转换）；
2. 用户输入预处理：同样做归一化，避免因格式差异漏检；
3. 匹配检测：用对应算法比对输入内容与敏感词库；
4. 结果处理：拦截（拒绝提交）、替换（用*遮挡）、标记（人工审核）。

### 二、主流敏感词过滤方案（按落地优先级排序）

#### 方案1：基于字符串匹配（入门级，小流量适用）

##### 1.1 核心实现

将敏感词库存入集合（List/Set），遍历敏感词，用字符串匹配方法（contains、indexOf）检测输入内容是否包含敏感词。

Java 简易示例：

```java
// 敏感词库（实际中从配置/数据库加载）
private static final Set<String> SENSITIVE_WORDS = new HashSet<>(Arrays.asList("敏感词1", "敏感词2", "违规词"));

// 检测方法
public boolean containsSensitive(String content) {
    // 预处理：全角转半角、小写
    String processedContent = normalize(content);
    for (String word : SENSITIVE_WORDS) {
        if (processedContent.contains(word)) {
            return true;
        }
    }
    return false;
}

// 替换方法
public String replaceSensitive(String content) {
    String processedContent = normalize(content);
    for (String word : SENSITIVE_WORDS) {
        processedContent = processedContent.replaceAll(word, "***");
    }
    return processedContent;
}

// 归一化处理
private String normalize(String content) {
    if (content == null) return "";
    // 全角转半角、大写转小写、去除空格
    content = content.replaceAll("１", "1").replaceAll("Ａ", "A").toLowerCase().trim();
    return content;
}
```

##### 1.2 优缺点

- **优点**：实现极简、零依赖、开发成本低、易调试，适合敏感词数量少（几百个以内）、流量低的场景（如小型管理系统）。
- **缺点**：效率极低（遍历匹配，时间复杂度O(n*m)，n为敏感词数量，m为输入长度）；不支持模糊匹配、偏旁变体；敏感词增多后性能急剧下降。

##### 1.3 适用场景

小型单体项目、后台管理系统、敏感词数量少（≤1000个）、低并发（QPS≤100）场景。

#### 方案2：基于Trie树（字典树，推荐中小型项目）

##### 2.1 核心原理

Trie树（前缀树）是一种多叉树结构，将敏感词的每个字符作为节点，敏感词的结尾节点标记为「敏感节点」。遍历用户输入的每个字符，沿Trie树节点查找，若走到敏感节点，则判定包含敏感词。

核心优势：预处理敏感词为Trie树后，匹配速度只与「输入内容长度」有关（时间复杂度O(m)，m为输入长度），与敏感词数量无关。

##### 2.2 关键优化（落地必加）

1. 敏感词预处理：去重、归一化、补全变体（如“赌”和“堵”、“枪”和“炝”）；
2. Trie树优化：压缩节点（减少内存占用）、支持多模式匹配（一次遍历检测多个敏感词）；
3. 输入预处理：除归一化外，过滤特殊字符（如“敏*感*词”中的*，避免绕开检测）。

##### 2.3 优缺点

- **优点**：匹配速度快、支持大量敏感词（万级以上）、内存占用可控；Java可直接手写实现，无额外依赖。
- **缺点**：不支持模糊匹配（如偏旁、谐音）；敏感词更新后需重新构建Trie树（可做定时刷新）；对变体敏感词（如“微信”→“微x”）漏检。

##### 2.4 适用场景

中小型项目、敏感词数量中（1万~10万）、中并发（QPS≤1000）场景（如电商评论、用户昵称）。

#### 方案3：基于DFA算法（确定有穷自动机，工业级首选）

##### 3.1 核心原理

DFA是Trie树的优化版，将敏感词库转换为「状态转移表」，每个状态代表一个字符，通过状态转移判断是否匹配敏感词。相比Trie树，DFA进一步压缩内存、提升匹配速度，支持多敏感词同时检测。

核心逻辑：输入字符→根据当前状态，查找下一个状态→若状态为“结束状态”，则匹配到敏感词。

##### 3.2 落地方式（Java）

无需手写，直接使用成熟开源工具：

- HuTool工具类：`WordFilter`（基于DFA，开箱即用）；
- IKAnalyzer：结合分词，支持更灵活的匹配；
- 自定义DFA：适合需定制化（如特殊字符过滤、变体匹配）的场景。

HuTool 简易示例：

```java
// 1. 初始化敏感词库
WordFilter wordFilter = new DfaWordFilter();
wordFilter.addWords(Arrays.asList("敏感词1", "敏感词2", "违规词"));

// 2. 检测是否包含敏感词
boolean hasSensitive = wordFilter.contains("输入内容包含敏感词1");

// 3. 替换敏感词
String result = wordFilter.replace("输入内容包含敏感词1", '*');
```

##### 3.3 优缺点

- **优点**：匹配速度极快（比Trie树更快）、内存占用低、支持万级~10万级敏感词、易集成；
- **缺点**：仍不支持谐音、偏旁等模糊匹配；敏感词更新需重建状态表（可做热更新优化）。

##### 3.4 适用场景

中大型项目、高并发（QPS≥1000）、敏感词数量多（1万~10万）场景（如社交平台、内容社区、短视频评论）。

#### 方案4：基于分词+关键词匹配（提升准确率，减少误判）

##### 4.1 核心原理

先对用户输入进行分词（如使用IK分词、HanLP分词），将输入拆分为多个词语，再用敏感词库匹配分词结果，避免“误判”（如“敏感”在“敏感词”中是敏感词，但在“敏感话题讨论”中可能无需拦截）。

核心优化：结合上下文，过滤“误触发”场景，提升准确率。

##### 4.2 实现流程

1. 输入预处理→分词（拆分词语）；
2. 将分词结果与敏感词库匹配（用DFA/Trie树）；
3. 结合上下文判断：若敏感词是独立词语，且非语境允许的场景（如“警察”在正常句子中无需拦截，在辱骂语境中需拦截），则判定为违规。

##### 4.3 优缺点

- **优点**：误判率低、准确率高；支持长句、复杂语境的检测；
- **缺点**：增加分词开销，性能略低于纯DFA；分词精度影响检测效果；开发成本高于纯DFA。

##### 4.4 适用场景

对准确率要求高、语境复杂的场景（如长文本评论、私信、文章内容）。

#### 方案5：基于AI/深度学习（复杂场景，大厂适用）

##### 5.1 核心原理

利用自然语言处理（NLP）模型，训练敏感内容识别模型，不仅能检测明确的敏感词，还能识别「语义敏感」（如无敏感词，但内容涉及违规话题）、谐音、偏旁、隐晦表达（如“加我v”→广告）。

##### 5.2 落地方式

- 自研模型：基于BERT、RoBERTa等预训练模型，结合业务敏感数据微调；
- 第三方API：百度AI、阿里云、腾讯云的内容安全API（开箱即用，按调用量收费）；
- 混合方案：AI检测+敏感词库兜底（兼顾准确率和效率）。

##### 5.3 优缺点

- **优点**：能检测隐晦、变体、语义敏感内容；误判率极低；适配复杂场景；
- **缺点**：成本高（自研需算力、数据；第三方需付费）；有调用延迟；小项目无需使用。

##### 5.4 适用场景

大厂、高并发、对内容安全要求极高的场景（如社交平台、短视频、直播弹幕）。

#### 方案6：第三方成熟组件（快速落地，无需自研）

除了HuTool、IKAnalyzer，还有以下成熟组件，直接集成即可：

- Sensitive-Word：开源敏感词过滤组件，支持DFA、Trie树，可自定义敏感词库、替换规则；
- IKAnalyzer+扩展：在分词基础上扩展敏感词过滤，支持自定义词典；
- 云服务商API：百度内容安全、阿里云绿网，适合不想维护敏感词库的场景。

### 三、敏感词库 维护方案（落地关键）

敏感词过滤的效果，核心依赖「敏感词库的完整性和时效性」，推荐落地方案：

1. **敏感词库存储**：存入数据库（MySQL/Redis），支持增删改查，避免硬编码；
2. **热更新机制**：定时从数据库加载最新敏感词库，重建Trie树/DFA状态表（如每小时刷新一次），无需重启服务；
3. **敏感词分类**：按违规等级（普通、严重、禁止）分类，对应不同处理逻辑（替换、拦截、人工审核）；
4. **定期更新**：关注监管政策，定期补充新敏感词（如新增违规词汇、网络热词）；
5. **误判反馈**：建立人工审核机制，收集误判案例，优化敏感词库（如添加白名单）。

### 四、常见问题 & 解决方案（面试高频）

1. **问题1：用户用特殊字符绕开检测（如“敏\*感\*词”“敏_感词”）** 解决方案：输入预处理时，过滤所有特殊字符（*、_、空格、表情等），统一归一化；
2. **问题2：敏感词更新后，服务需要重启** 解决方案：实现敏感词库热更新，定时从数据库加载，重建匹配模型；      
3. **问题3：误判率高（如“中南海”在正常新闻中被拦截）** 解决方案：结合分词+上下文判断；添加白名单（如新闻类内容的合法词汇）；按场景差异化检测；      
4. **问题4：高并发场景下，过滤性能不足** 解决方案：用DFA算法；敏感词库预加载到内存；多线程并行检测；添加缓存（缓存已检测过的合规内容）；      
5. **问题5：谐音、偏旁变体漏检（如“赌”→“堵”、“微信”→“微x”）** 解决方案：敏感词库补充变体词汇；用AI模型检测；结合拼音匹配（如“赌”和“堵”拼音相同）。

### 六、面试高频问答（必背）

1. **敏感词过滤的核心难点是什么？** 答：① 性能（高并发下快速检测）；② 准确率（减少误判、漏检）；③ 时效性（敏感词库及时更新）；④ 规避绕开检测（特殊字符、变体）。      
2. **为什么不推荐用字符串contains做敏感词过滤？** 答：时间复杂度高（O(n*m)），敏感词数量增多后，性能急剧下降；不支持批量检测，易漏检变体敏感词。      
3. **DFA和Trie树的区别是什么？** 答：两者都是基于前缀匹配，DFA是Trie树的优化版；DFA用状态转移表压缩内存，匹配速度更快，更适合高并发、大量敏感词场景。      
4. **如何解决敏感词热更新问题？** 答：将敏感词库存入数据库，定时（如每小时）从数据库加载最新敏感词，重建DFA/Trie树模型，无需重启服务；也可通过消息通知，实时触发模型更新。      
5. **高并发场景下，敏感词过滤如何优化性能？** 答：① 用DFA算法，降低时间复杂度；② 敏感词库预加载到内存，避免频繁读取数据库；③ 输入预处理异步执行；④ 添加缓存，缓存合规内容；⑤ 多线程并行检测。     

## 数据脱敏方案

核心目标：在不影响业务正常使用的前提下，对**敏感数据**（用户隐私、商业机密等）进行变形处理，实现「可用不可见」，既保障数据安全、符合合规要求（《个人信息保护法》《数据安全法》），又避免敏感信息泄露（如开发/测试环境、日志、接口返回）。

核心敏感数据范围：手机号、身份证号、银行卡号、姓名、邮箱、地址、密码、手机号、企业核心数据（如营收、客户信息）等。

### 一、数据脱敏 核心原则（必守）

所有脱敏方案必须遵循以下4个原则，避免脱敏失效或影响业务：

1. **最小必要原则**：只对敏感字段脱敏，非敏感字段不处理；脱敏程度以“满足业务需求”为限，不过度脱敏。
2. **不可逆原则**：展示脱敏（如接口返回、日志）必须不可逆（无法通过脱敏后的数据反推原文）；存储脱敏按需选择可逆/不可逆。
3. **业务无侵入原则**：脱敏不影响业务逻辑（如手机号脱敏后仍能判断运营商，身份证脱敏后仍能判断省份）。
4. **全生命周期覆盖原则**：脱敏需覆盖数据「采集、传输、存储、使用、共享、销毁」全流程，不遗漏任何泄露场景。

### 二、数据脱敏 分类（按场景+实现方式）

核心分为两大类型：**展示脱敏**（最常用）和**存储脱敏**，延伸出日志脱敏、文件脱敏、开发测试环境脱敏，覆盖后端所有高频场景。

#### 类型1：展示脱敏（接口返回/页面展示，最核心场景）

定义：数据在数据库中存储原始值，仅在「对外输出」（接口返回、前端展示、报表导出）时进行脱敏处理，不改变原始数据，不影响业务查询（如根据手机号后4位查询）。

核心要求：不可逆、不影响业务展示、适配不同敏感字段的格式。

##### 1.1 常见敏感字段 脱敏规则（Java实现示例）

##### 1.2 展示脱敏 实现方式（Java后端，按推荐优先级排序）

###### 方式1：注解式脱敏（推荐，无侵入、易维护）

核心：自定义脱敏注解，结合Spring AOP或Jackson序列化，在接口返回时自动脱敏，无需修改业务代码。

常用开源组件：

- HuTool工具类：`@Desensitized` 注解，支持多种脱敏类型（手机号、身份证等），开箱即用；
- 自定义注解：适配业务个性化脱敏规则（如特殊字段格式）。

示例（HuTool注解使用）：

```java
// 实体类字段添加注解，接口返回时自动脱敏
public class User {
    private Long id;
    // 手机号脱敏
    @Desensitized(type = DesensitizedType.MOBILE_PHONE)
    private String mobile;
    // 身份证脱敏
    @Desensitized(type = DesensitizedType.ID_CARD)
    private String idCard;
    // 姓名脱敏
    @Desensitized(type = DesensitizedType.CHINESE_NAME)
    private String name;
    // getter/setter
}
```

###### 方式2：工具类脱敏（手动调用，适配特殊场景）

核心：封装通用脱敏工具类，在需要脱敏的地方（如报表导出、日志打印）手动调用，灵活可控。

示例（通用脱敏工具类）：

```java
/**
 * 数据脱敏工具类（可直接复制使用）
 */
public class DesensitizationUtil {
    // 手机号脱敏
    public static String desensitizeMobile(String mobile) {
        if (StringUtils.isBlank(mobile) || mobile.length() != 11) return mobile;
        return mobile.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
    }

    // 身份证脱敏
    public static String desensitizeIdCard(String idCard) {
        if (StringUtils.isBlank(idCard)) return idCard;
        return idCard.replaceAll("(\\d{6})\\d+(\\d{4})", "$1********$2");
    }

    // 通用脱敏（保留首尾，中间用*遮挡）
    public static String desensitizeCommon(String str, int start, int end) {
        if (StringUtils.isBlank(str) || str.length() < start + end) return str;
        String regex = "(.{" + start + "})(.+)(" + ".{" + end + "})";
        return str.replaceAll(regex, "$1" + String.join("", Collections.nCopies(str.length() - start - end, "*")) + "$3");
    }
}
```

###### 方式3：数据库查询脱敏（不推荐，侵入性强）

核心：通过SQL函数（如MySQL的SUBSTRING、REPLACE）在查询时直接脱敏，返回脱敏后的数据。

示例（MySQL SQL）：

```sql
-- 手机号脱敏：保留前3后4
SELECT 
    id,
    CONCAT(SUBSTRING(mobile, 1, 3), '****', SUBSTRING(mobile, 8, 4)) AS mobile,
    name
FROM user;
```

缺点：SQL冗余、维护成本高、无法复用，仅适合简单查询场景。

#### 类型2：存储脱敏（数据库存储，高安全场景）

定义：数据在「入库前」进行脱敏处理，数据库中存储的是脱敏后的数据，即使数据库泄露，也无法获取原始敏感信息。

核心要求：按需选择「可逆/不可逆」，不可逆用于密码等绝对敏感字段，可逆用于需偶尔获取原始数据的场景（如客服查询）。

##### 2.1 存储脱敏 两种方式

###### 方式1：不可逆存储脱敏（推荐，高安全）

适用场景：密码、支付密码等，无需获取原始数据的字段，核心用「哈希摘要算法」（不可逆）。

核心实现：密码存储用BCrypt（首选）、SHA256+盐值，禁止用MD5（不安全）。

Java示例（BCrypt加密存储密码）：

```java
// 密码加密（入库前）
String rawPassword = "123456";
String encryptedPassword = BCrypt.hashpw(rawPassword, BCrypt.gensalt()); // 自动生成盐值

// 密码校验（登录时）
boolean isMatch = BCrypt.checkpw(rawPassword, encryptedPassword);
```

###### 方式2：可逆存储脱敏（谨慎使用）

适用场景：需偶尔获取原始数据（如客服核实用户信息），但又要保障存储安全，核心用「对称加密算法」（AES、SM4）。

核心实现：入库前用AES加密，出库后（需原始数据时）用密钥解密，密钥需统一托管（如KMS），禁止硬编码。

Java示例（AES加密存储手机号）：

```java
// AES加密工具类（简化版）
public class AesUtil {
    private static final String KEY = "1234567890abcdef"; // 实际用KMS托管
    private static final String ALGORITHM = "AES/ECB/PKCS5Padding";

    // 加密（入库前）
    public static String encrypt(String content) throws Exception {
        SecretKeySpec keySpec = new SecretKeySpec(KEY.getBytes(), "AES");
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.ENCRYPT_MODE, keySpec);
        return Base64.getEncoder().encodeToString(cipher.doFinal(content.getBytes()));
    }

    // 解密（需原始数据时）
    public static String decrypt(String encryptedContent) throws Exception {
        SecretKeySpec keySpec = new SecretKeySpec(KEY.getBytes(), "AES");
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.DECRYPT_MODE, keySpec);
        return new String(cipher.doFinal(Base64.getDecoder().decode(encryptedContent)));
    }
}

// 入库时加密
String mobile = "13812345678";
String encryptedMobile = AesUtil.encrypt(mobile);
// 存入数据库：encryptedMobile（密文）

// 需原始数据时解密
String rawMobile = AesUtil.decrypt(encryptedMobile);
```

注意：可逆存储脱敏需严格管控密钥，避免密钥泄露导致脱敏失效。

#### 类型3：日志脱敏（易遗漏场景）

核心问题：日志中常打印敏感数据（如用户手机号、身份证），若日志泄露，会造成敏感信息泄露，需针对性脱敏。

##### 3.1 实现方式

- **方式1：日志框架拦截脱敏**（推荐）：用Logback/Log4j的拦截器，对日志内容进行正则替换，过滤敏感字段。
- **方式2：手动脱敏后打印**：打印日志前，用脱敏工具类处理敏感字段，再输出日志。
- **方式3：禁止日志打印敏感字段**：规范日志打印规范，禁止直接打印手机号、身份证等敏感信息。

示例（Logback拦截器脱敏）：

```java
// 自定义Logback脱敏拦截器
public class LogDesensitizationFilter extends TurboFilter {
    // 敏感字段正则（手机号、身份证）
    private static final Pattern MOBILE_PATTERN = Pattern.compile("1[3-9]\\d{9}");
    private static final Pattern ID_CARD_PATTERN = Pattern.compile("\\d{17}[0-9Xx]|\\d{15}");

    @Override
    public FilterReply decide(Marker marker, Logger logger, Level level, String format, Object[] params, Throwable t) {
        // 对日志内容进行脱敏替换
        if (params != null && params.length > 0) {
            for (int i = 0; i < params.length; i++) {
                if (params[i] instanceof String) {
                    String content = (String) params[i];
                    // 手机号脱敏
                    content = MOBILE_PATTERN.matcher(content).replaceAll("138****5678");
                    // 身份证脱敏
                    content = ID_CARD_PATTERN.matcher(content).replaceAll("***********1234");
                    params[i] = content;
                }
            }
        }
        return FilterReply.NEUTRAL;
    }
}
```

#### 类型4：开发/测试环境脱敏（合规重点）

核心风险：生产数据直接导入开发/测试环境，会导致敏感信息泄露（如开发人员接触用户隐私），需做「数据漂白/脱敏」。

##### 4.1 落地方案

1. **方案1：数据脱敏工具批量处理**：用开源工具（如DataMasker）对生产导出的数据进行批量脱敏，再导入测试环境；
2. **方案2：测试数据生成**：用工具（如Mockaroo、EasyMock）生成模拟测试数据，不使用真实生产数据；
3. **方案3：数据库层面脱敏**：通过数据库触发器，在测试环境数据插入时自动脱敏；
4. **方案4：脱敏规则统一**：开发/测试环境与生产环境使用相同的脱敏规则，避免脱敏不一致导致的业务问题。

#### 类型5：文件/OSS脱敏（附件场景）

适用场景：用户上传的证件照片（身份证、银行卡）、合同文件等，需脱敏后存储/展示。

##### 实现方案

- 图片脱敏：用工具（如OpenCV）对图片中的敏感信息（姓名、身份证号）进行遮挡、模糊处理；
- 文件脱敏：对文件中的敏感字段进行替换（如PDF、Word中的手机号、身份证），再上传至OSS；
- 权限管控：OSS文件设置私有权限，仅授权用户可访问，结合临时URL时效控制。

### 三、数据脱敏 全流程落地架构（企业级）

结合后端业务，形成“全流程、无死角”的脱敏架构，核心分为5个环节：

1. **采集环节**：前端提交敏感数据时，初步脱敏（如手机号隐藏展示），后端二次校验；
2. **存储环节**：密码用BCrypt不可逆加密，核心敏感字段（如身份证）按需用AES可逆加密；
3. **使用环节**：接口返回用注解式脱敏，日志用拦截器脱敏，报表导出用工具类脱敏；
4. **共享环节**：对外提供数据（如第三方接口）时，强制脱敏，裁剪非必要敏感字段；
5. **运维环节**：开发/测试环境数据漂白，数据库备份脱敏，密钥统一托管（KMS）。

### 四、常见问题 & 解决方案（面试高频）

1. **问题1：脱敏后影响业务查询（如根据手机号后4位查询用户）** 解决方案：展示脱敏不改变数据库原始数据，查询时用原始字段查询，返回时再脱敏；存储脱敏需保留可查询的特征（如手机号后4位）。      
2. **问题2：可逆脱敏的密钥管理问题** 解决方案：密钥不硬编码，用KMS（密钥管理系统）统一托管，定期轮换密钥；禁止明文存储密钥。      
3. **问题3：日志脱敏不彻底（如JSON格式日志中的敏感字段漏检）** 解决方案：日志拦截器支持JSON解析，对JSON中的敏感字段（如mobile、idCard）进行精准替换；规范日志打印格式。      
4. **问题4：开发/测试环境脱敏成本高，数据不一致** 解决方案：开发自动化脱敏脚本，生产数据导出后自动脱敏，再导入测试环境；统一脱敏规则，确保测试数据与生产数据格式一致。      
5. **问题5：脱敏与数据统计冲突（如按手机号归属地统计）** 解决方案：脱敏时保留统计所需的特征（如手机号前3位），不影响统计逻辑；对统计用的数据单独处理，避免脱敏影响。

### 六、面试高频问答（必背）

1. **数据脱敏的核心原则是什么？** 答：最小必要、不可逆（展示脱敏）、业务无侵入、全生命周期覆盖。      
2. **展示脱敏和存储脱敏的区别是什么？** 答：① 展示脱敏：数据库存原始数据，仅对外输出时脱敏，可逆/不可逆均可（推荐不可逆）；② 存储脱敏：入库前脱敏，数据库存脱敏后数据，不可逆用于密码，可逆用于需获取原始数据的场景。      
3. **密码存储为什么不能用MD5？推荐用什么？** 答：MD5可碰撞、易被彩虹表破解，不安全；推荐用BCrypt，自带随机盐值、自适应慢哈希，天然防暴力破解。     
4. **如何实现接口返回的自动脱敏？** 答：用注解式脱敏（如HuTool的@Desensitized），结合Spring AOP或Jackson序列化，在接口返回时自动拦截并脱敏，无需修改业务代码。      
5. **开发/测试环境为什么要脱敏？如何实现？** 答：避免开发人员接触生产敏感数据，符合合规要求；实现方式：生产数据批量脱敏后导入、生成模拟测试数据、数据库触发器自动脱敏。      
6. **可逆脱敏的密钥如何管理？** 答：密钥不硬编码，用KMS（密钥管理系统）统一托管，定期轮换密钥；禁止明文存储、传输密钥，仅授权服务可访问密钥。

这是**后端/全栈面试必考**、**项目必做**的三层安全防御体系，我给你整理成**最清晰、最落地、最容易背诵**的版本。

### 一、三层校验总览（一句话记住）

1. **前端校验**：给用户看的，提升体验，**不能防攻击**
2. **后端校验**：真正的安全防线，**必须做**
3. **权限校验**：控制“谁能访问、能操作什么数据”，**最后一道防线**

口诀：
**前端拦外行，后端防坏人，权限管范围**

### 二、前端校验（Frontend Validation）

#### 1. 作用

- 减少无效请求，减轻服务器压力
- 即时提示，提升用户体验
- **不能作为安全依据**（可被绕过）

#### 2. 校验内容

- 非空校验
- 长度、格式校验（手机号、邮箱、身份证）
- 密码强度
- 输入合法性（数字、日期范围）
- 防重复提交

#### 3. 常见实现

- Vue：VeeValidate、Element Plus 表单校验
- React：Formik、Ant Design 表单校验
- 原生：HTML5 + 正则

#### 4. 关键点

**前端校验 = 体验优化 ≠ 安全保障**
抓包、改参数、调用接口可直接绕过前端。

---

### 三、后端校验（Backend Validation）

#### 1. 作用

- **真正的安全校验**
- 防止非法数据入库
- 防止SQL注入、XSS、非法参数
- **所有接口必须做**

#### 2. 校验内容

- 非空、长度、格式
- 数据合法性、业务规则
- 敏感词过滤
- 数据脱敏
- 防止重复提交

#### 3. 主流实现（Java）

- **JSR380 / Hibernate Validator**（注解式）
  - `@NotBlank`、`@NotNull`、`@Length`、`@Pattern`
- 全局异常捕获
- AOP 统一校验
- 手动工具类校验

#### 4. 关键点

**后端校验 = 安全底线 = 必须做**
**前端可以没有，后端绝对不能没有！**

---

### 四、权限校验（Permission Validation）

#### 1. 作用

控制：**谁能访问？能访问哪些数据？能做什么操作？**

#### 2. 权限三要素

1. **认证（Authentication）**：你是谁？（登录、token）
2. **授权（Authorization）**：你能做什么？（角色、权限）
3. **鉴权（Access Control）**：是否允许操作？（拦截校验）

#### 3. 权限模型

- **RBAC（最常用）**
  用户 → 角色 → 权限
- 数据权限
  只能看自己/本部门/本公司数据

#### 4. 实现方式

- 登录认证：JWT、Session、OAuth2
- 接口权限：注解 `@PreAuthorize`、拦截器
- 数据权限：AOP + 数据范围过滤
- 网关统一鉴权（Spring Cloud Gateway）

#### 5. 两种越权

- **垂直越权**：普通用户访问管理员接口
- **水平越权**：用户A访问用户B的数据

**权限校验就是为了防止越权！**

---

### 五、三者关系与执行顺序

#### 执行流程

1. 用户输入 → **前端校验**（体验）
2. 请求到达后端 → **后端校验**（安全）
3. 校验通过 → **权限校验**（范围）
4. 最后才执行业务逻辑

#### 安全重要性

**权限校验 > 后端校验 > 前端校验**

---

### 六、核心区别（面试必背版）

#### 前端校验

- 目的：体验、减少请求
- 特点：可被绕过
- 安全性：低
- 必须吗：可选

#### 后端校验

- 目的：数据安全、防攻击
- 特点：不可绕过
- 安全性：高
- 必须吗：**必须！**

#### 权限校验

- 目的：防止越权、控制访问范围
- 特点：接口最后一道关卡
- 安全性：最高
- 必须吗：**必须！**

---

### 七、最经典面试题

#### 1. 前端校验了，后端为什么还要校验？

因为前端可以被**抓包、篡改、禁用JS、直接调用接口**绕过，后端不校验就会出现：
- 非法数据入库
- 漏洞攻击
- 系统崩溃
**后端校验是安全底线。**

#### 2. 后端校验了，为什么还要权限校验？

后端校验只管**数据对不对**，
权限校验管**你能不能操作**。
没有权限校验会导致：
- 垂直越权（普通用户变管理员）
- 水平越权（看别人数据）

#### 3. 三者谁最重要？

**权限校验 > 后端校验 >>> 前端校验**

## 密码存储方案 + 完整登录流程 

### 一、核心前置认知

1. **密码绝对禁止明文存储**
2. **禁止弱加密**：MD5、SHA1、单纯SHA256 全部不安全
3. 密码存储核心要求：**单向不可逆、防彩虹表、防暴力破解、防撞库**
4. 核心术语
- 摘要算法：单向、不可解密
- 盐值Salt：随机字符串，混淆密码、防止彩虹表
- 慢哈希：拉长计算耗时，抵御暴力破解

---

### 二、密码存储方案演进 & 对比

#### 1. 明文存储（彻底淘汰）

- 数据库直接存 `123456`
- 风险：拖库直接泄露，严重违规
- 结论：违法、绝对禁止

#### 2. 单纯MD5（淘汰）

```
md5(123456)
```
- 问题：彩虹表可直接反向破解、碰撞漏洞、极易被爆破
- 结论：老旧系统遗留，新项目禁用

#### 3. MD5 / SHA256 + 固定盐（不安全）

```
sha256(密码 + 固定salt)
```
- 问题：盐值全局固定，一旦泄露，全量密码可批量破解
- 无法抵御撞库、批量爆破

#### 4. 随机盐 + 哈希（基础合格方案）

```
密文 = SHA256(原始密码 + 随机唯一Salt)
```
- 每个用户**独立随机盐**
- 即使同密码，加密结果不同
- 优点：防彩虹表、防批量撞库
- 缺点：哈希计算太快，**暴力破解、字典破解效率极高**

#### 5. 慢哈希算法（生产推荐✅）

##### 主流三种

1. **BCrypt（Java 首选、最常用）**
2. PBKDF2（安全合规、政务金融常用）
3. SCrypt / Argon2（更高安全、复杂度高）

##### 核心优势

- 自带**随机盐**、内置**迭代次数/算力消耗**
- 计算速度慢，大幅降低暴力破解效率
- 不可逆、无解密方式
- 不用手动维护盐值，算法自动生成、自动存储

#### 6. 可逆加密（严禁用于登录密码）

- AES、SM4 对称加密
- 特点：可解密还原明文
- 适用：手机号、身份证**可逆存储**
- 严禁：登录密码、支付密码

---

### 三、生产级推荐：BCrypt 完整方案

#### 1. 核心特点

1. 自动生成随机盐，每个用户盐不同
2. 盐值直接**拼接在密文中**，无需单独存数据库
3. 自适应慢哈希，算力开销可控
4. 算法成熟、Spring 原生支持、兼容性强

#### 2. 加密&校验原理

- 注册/改密：
  1. 生成随机盐
  2. 原始密码 + 盐值 → 慢速哈希计算
  3. 最终密文入库
- 登录校验：
  1. 根据用户名查询数据库密文
  2. 用数据库密文中自带的盐值，对前端传入密码二次哈希
  3. 比对两次哈希结果是否一致

#### 3. Java 代码示例

```java
// 1. 密码加密（注册/重置密码）
String rawPwd = "123456";
// 自动生成盐 + 加密
String encodePwd = BCrypt.hashpw(rawPwd, BCrypt.gensalt());

// 2. 密码校验（登录）
String inputPwd = "123456";
String dbPwd = "数据库密文";
boolean match = BCrypt.checkpw(inputPwd, dbPwd);
```

### 四、各类密码存储方案横向对比

| 方案          | 可逆 | 防彩虹表 | 防暴力破解 | 推荐度     |
| ------------- | ---- | -------- | ---------- | ---------- |
| 明文          | ✅    | ❌        | ❌          | 淘汰       |
| MD5           | ❌    | ❌        | ❌          | 淘汰       |
| MD5+固定盐    | ❌    | ⭐        | ❌          | 淘汰       |
| SHA256+随机盐 | ❌    | ⭐⭐       | ❌          | 一般       |
| **BCrypt**    | ❌    | ⭐⭐⭐      | ⭐⭐⭐        | ✅首选      |
| PBKDF2        | ❌    | ⭐⭐⭐      | ⭐⭐⭐        | ✅金融/合规 |
| AES/SM4 加密  | ✅    | ⭐⭐       | ❌          | 仅敏感字段 |

#### 结论

- **登录密码、系统密码**：统一使用 `BCrypt`
- **支付密码**：PBKDF2 / 国密改造
- **隐私字段（手机号/身份证）**：AES/SM4 可逆加密

### 五、完整密码登录流程（标准企业版）

#### 1. 整体流程

```
前端输入账号密码
→ 前端简单校验
→ HTTPS 传输到后端
→ 后端参数校验 + 防暴力拦截
→ 根据用户名查询数据库密文
→ 密码哈希比对
→ 校验通过：生成 Token / Session
→ 返回令牌，后续请求携带鉴权
```

#### 2. 分步详解

##### ① 前端处理

1. 基础校验：非空、密码长度、格式
2. **前端绝不加密密码**（避免算法暴露、无意义）
3. 通过 **HTTPS** 加密传输，防止抓包窃听

##### ② 网络传输

- 全站 HTTPS(TLS)
- 禁止 HTTP 明文传输密码
- 防重放：时间戳 + 请求签名（高频登录接口）

##### ③ 后端接收 & 安全校验

1. 接口参数合法性校验（@NotBlank 等）
2. 防暴力破解：
   - 单账号错误次数限制（5次锁定）
   - 登录接口限流、IP 黑名单
3. 敏感词、恶意参数过滤

##### ④ 数据库查询

- 根据用户名/手机号查询用户
- **只查密文，绝不查明文**
- 用户不存在：模糊返回「账号或密码错误」（防止遍历枚举账号）

##### ⑤ 密码比对核心逻辑

1. 拿到数据库中 BCrypt 密文
2. 使用 `BCrypt.checkpw(前端明文, 库中密文)`
3. 一致 → 登录成功；不一致 → 登录失败

> 关键点：**后端永远只校验哈希，全程不出现明文密码落地存储**

##### ⑥ 登录成功后

1. 生成身份凭证：
- 分布式：**JWT / Token + Redis 黑名单**
- 单体：Session
2. 记录登录日志：IP、设备、时间、地点
3. 重置错误次数、解锁账号

##### ⑦ 后续接口鉴权

- 每次请求携带 Token
- 网关/拦截器统一认证 + 权限校验
- 过期自动下线、异地登录挤下线

### 八、面试高频必背问答

#### 1. 为什么不能用 MD5 存密码？

MD5 算法速度快、可彩虹表破解、存在碰撞漏洞，极易被字典爆破，完全无法保障密码安全。

#### 2. BCrypt 为什么安全？

自带随机盐、慢哈希设计，增加破解算力成本；盐值内嵌密文，无需单独存储，安全性与可维护性兼顾。

#### 3. 前端要不要对密码加密？

不需要。前端加密算法容易被逆向破解，真正安全依靠 **HTTPS传输加密 + 后端单向哈希存储**。

#### 4. 密码登录为什么只能用不可逆算法？

即使数据库被拖库，黑客也无法还原用户原始密码，防止撞库、用户其他平台密码泄露。

#### 5. 加盐的作用是什么？

防止彩虹表攻击；相同密码生成不同密文，避免批量撞库破解。

## 十一、加密算法分类（面试高频）

1. **对称加密**
AES、SM4、DES（淘汰）
特点：加解密同一密钥，速度快，适合大段数据/报文
2. **非对称加密**
RSA、SM2
特点：公钥加密、私钥解密，安全性高，速度慢，适合加密密钥
3. **哈希摘要（不可逆）**
MD5（不安全）、SHA256、SM3、BCrypt
用途：密码存储、数据完整性校验
4. **国密算法**
SM1/SM2/SM3/SM4，政务、金融、国企强制要求



面试必背+原理+特点+使用场景+优劣对比，全覆盖。

### 一、对称加密算法

#### 1. 核心原理

加密、解密**使用同一个密钥**；
收发双方必须提前约定并保管同一密钥。

#### 2. 核心特点

- 加密速度**极快**、运算开销小、适合大数据/报文加密
- 密钥分发困难：多系统对接时密钥传递易泄露
- 可逆加密：密文可以还原明文

#### 3. 常见算法

##### （1）DES

- 早期经典对称算法
- 密钥长度短（56位），**暴力破解成本极低**
- 现状：**彻底淘汰，禁止生产使用**

##### （2）3DES（三重DES）

- 对DES做三次加密，提升强度
- 性能差、迭代老旧
- 现状：逐步淘汰，遗留老系统偶尔可见

##### （3）AES（高级加密标准）

- 目前**全球通用工业级标准**
- 密钥：128/192/256位，安全强度高
- 性能优秀、跨语言通用、Java/前端/移动端全支持
- 适用：**接口报文、敏感字段、文件加密**

##### （4）SM4（国密对称）

- 我国商用密码标准，对标AES
- 分组加密、安全强度与AES-128相当
- 适用：**政务、金融、国企、等保合规项目强制使用**

#### 4. 总结一句话

**大数量、高并发加密用对称；AES通用，SM4国产化合规。**

---

### 二、非对称加密算法

#### 1. 核心原理

生成**一对密钥**：
- 公钥（Public）：公开分发，用于**加密**
- 私钥（Private）：自己保管，用于**解密/签名**
公钥加密的数据，**只有私钥能解开**。

#### 2. 核心特点

- 安全等级极高，无密钥分发风险
- 运算**非常慢**、不适合加密大文本
- 可逆加密

#### 3. 常见算法

##### （1）RSA

- 最经典、使用最广的非对称算法
- 密钥常见：1024/2048/4096位
- 用途：
  1. 加密**对称密钥**（混合加密方案）
  2. 接口签名、验签
  3. HTTPS底层依赖RSA
- 缺点：长密钥性能差，短密钥易被破解

##### （2）SM2（国密非对称）

- 国产椭圆曲线加密算法（ECC）
- 同等安全强度下，**密钥更短、性能比RSA更强**
- 替代RSA，国企/银行/政务强制替换

#### 4. 典型使用方案：混合加密（工业标准）

1. 用**RSA/SM2（非对称）** 加密 随机AES/SM4密钥
2. 用**AES/SM4（对称）** 加密 业务明文数据
> 既保证安全，又保证性能。

#### 5. 总结一句话

**小数据、密钥传输、签名认证用非对称；RSA通用，SM2国产化。**

---

### 三、哈希摘要算法（单向散列 / 不可逆）

#### 1. 核心原理

任意长度明文 → 固定长度摘要字符串
**单向不可逆**：无法通过摘要反推原文。

#### 2. 核心特点

- 不可逆、不可解密
- 雪崩效应：原文改1个字符，摘要完全不同
- 主要用于：**完整性校验、密码存储、防篡改**

#### 3. 常见算法

##### （1）MD5

- 128位摘要
- 缺陷：碰撞严重、可彩虹表破解
- 现状：**完全不安全，禁止用于密码加密**
- 仅遗留场景：简单文件一致性校验

##### （2）SHA系列

- SHA1：已破解，淘汰
- **SHA256 / SHA512**：目前通用安全摘要算法
- 广泛用于：数据签名、文件校验、防篡改

##### （3）SM3（国密摘要）

- 国产哈希算法，对标SHA256
- 输出256位摘要
- 等保、涉密系统强制替代SHA256

##### （4）BCrypt 【密码存储最优解】

- 专门为**用户密码存储**设计
- 自带**随机盐值**、自适应算法、慢哈希
- 天然防暴力破解、防彩虹表
- 业务系统登录模块**首选**
> 重点：不要用MD5/SHA256存密码，必须 BCrypt

##### （5）HMAC（带密钥哈希）

- 如 HMAC-SHA256
- 在哈希基础上增加密钥
- 常用于：**接口签名、防篡改、防伪造**

#### 4. 总结一句话

**密码存储用BCrypt；数据防篡改用SHA256/SM3；接口签名用HMAC。**

### 四、国密算法（SM系列）面试必背

#### 1. 产生背景

国家商用密码标准，为摆脱国外算法依赖，
**政务、金融、国企、事业单位、等保三级及以上**项目强制要求接入国密。

#### 2. 四大国密算法对标关系

1. **SM1**
   - 对称加密，硬件级算法（不公开）
   - 对标：3DES
   - 多用于硬件加密机、涉密系统

2. **SM2**
   - 非对称加密（椭圆曲线ECC）
   - 对标：RSA
   - 用途：密钥交换、数字签名、身份认证

3. **SM3**
   - 哈希摘要算法
   - 对标：SHA256
   - 用途：密码摘要、数据完整性校验

4. **SM4**
   - 分组对称加密算法
   - 对标：AES
   - 用途：敏感数据加密、传输加密、存储加密

#### 3. 国密使用场景

- 等保测评、国企招标、银行系统、政务平台
- 接口改造：HTTPS 改为 国密SSL
- 数据库敏感字段：AES 替换为 SM4

### 六、面试高频问答

1. **为什么不用RSA加密大报文？**
RSA是非对称加密，运算极慢，只适合加密短数据（密钥、签名）。

2. **为什么密码不能用MD5？**
MD5可碰撞、彩虹表破解，无随机盐，极易被破解；推荐BCrypt。

3. **混合加密流程是什么？**
RSA加密AES密钥 + AES加密业务数据，兼顾安全与性能。

4. **AES和SM4区别？**
功能一致，AES国际通用，SM4国产国密，合规项目必须用SM4。

5. **SM2和RSA区别？**
SM2基于椭圆曲线，同等安全密钥更短、性能更好，国产化替代RSA。
