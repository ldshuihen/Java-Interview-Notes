
```yaml
version: '3.8'

services:
  # 1. Nacos 注册中心/配置中心
  nacos:
    image: nacos/nacos-server:v2.2.3
    container_name: nacos
    ports:
      - "8848:8848"
      - "9848:9848"
    environment:
      - MODE=standalone
    restart: always

  # 2. MySQL
  mysql:
    image: mysql:8.0
    container_name: mysql
    ports:
      - "3306:3306"
    environment:
      - MYSQL_ROOT_PASSWORD=root123
    volumes:
      - ./mysql/data:/var/lib/mysql
    restart: always

  # 3. Redis
  redis:
    image: redis:7
    container_name: redis
    ports:
      - "6379:6379"
    restart: always

  # 4. RabbitMQ（可选）
  rabbitmq:
    image: rabbitmq:3.12-management
    container_name: rabbitmq
    ports:
      - "5672:5672"
      - "15672:15672"
    restart: always
```

启动所有中间件：
```bash
docker-compose up -d
```

#### `common` 模块的处理方式

在 Java 微服务架构中，`common` 模块的处理方式与普通微服务（如 `auth-service`, `order-service`）有很大不同。以下是核心处理逻辑：

##### 1. `common` 不打包为“可执行”Jar

普通的微服务模块（含有 `main` 方法）会使用 Spring Boot Maven 插件打包成一个包含所有依赖的 **可执行 Jar (Executable JAR)**。
但是，`common` 是一个**库（Library）**，它不需要包含所有依赖，也不需要 `main` 方法。

*   **配置建议**：在 `common` 模块的 `pom.xml` 中，**不要**添加 `spring-boot-maven-plugin`，或者将其设置为不执行 `repackage`。它应该被打成一个普通的、瘦小的 **JAR** 包。

##### 2. 使用 `mvn install` 将其安装到本地仓库

由于其他微服务在编译时需要引用 `common` 里的类，你必须先让 Maven 知道 `common.jar` 在哪里：
*   **本地开发**：在根目录下运行 `mvn install`。Maven 会先编译 `common`，将其 JAR 包放入你电脑的本地仓库（通常是 `~/.m2/repository`）。这样其他模块在打包时就能正确引用它。
*   **整体打包**：如果你在父工程根目录运行 `mvn clean package`，Maven 会根据依赖关系自动计算顺序（Reactor Order），先打包 `common`，再打包依赖它的微服务。

##### 总结

*   **微服务** = 可执行 JAR（包含所有依赖，直接 `java -jar` 运行）。
*   **Common** = 普通 JAR（仅包含代码，作为依赖项被“塞”进微服务的 JAR 里）。
*   **操作顺序**：永远先 `mvn install` 根目录或 `common` 模块，再去打包具体的微服务。

#### 为什么要跳过测试 (`-DskipTests`)？

在执行 `mvn package` 或 `mvn install` 时加上 `-DskipTests`（或 `-Dmaven.test.skip=true`）主要有以下几个原因：

1.  **节省时间（最直接原因）**：
    微服务项目通常有几百甚至上千个单元测试。如果每次打包都要运行一遍，可能需要几分钟甚至更久。跳过测试可以让打包在几秒钟内完成。

2.  **环境依赖问题**：
    很多单元测试（尤其是集成测试）需要连接 **数据库（MySQL）**、**缓存（Redis）** 或 **消息队列（RabbitMQ）**。
    *   如果你在打包时，这些中间件没启动，或者你是在一台没有这些环境的服务器（如 Jenkins 编译机）上打包，测试就会失败，导致 JAR 包打不出来。
    *   为了能顺利打出包，我们会先跳过这些对环境敏感的测试。

3.  **测试已经做过了**：
    在标准的开发流程中，测试应该在 **提交代码前** 或者在 **CI (持续集成)** 阶段单独运行。
    当你确定代码逻辑没问题，只是想简单打个包部署到服务器看看效果时，重复运行测试就没有必要了。

4.  **测试代码本身有 Bug**：
    有时候你的业务代码是好的，但由于数据变动导致旧的测试用例挂了。在急着上线修复 Bug 的情况下，我们会选择先跳过测试，以后再回来修测试用例。

---

##### 总结建议：

*   **开发时**：建议**不要**跳过测试，养成跑一遍测试的好习惯，防止改坏东西。
*   **部署/打包镜像时**：通常会加上 `-DskipTests`，以追求最快的构建速度和最高的成功率。

常用命令：
`mvn clean install -DskipTests` （清理并安装到仓库，不跑测试）

### 3. 每个微服务创建 Dockerfile（示例）

以 `user-service` 为例：
`/home/service/user-service/Dockerfile`
```dockerfile
# 1. 基础镜像：使用 OpenJDK 17 精简版（运行 Java 项目必须的环境）
FROM openjdk:17-jdk-slim

# 2. 将宿主机的 user-service.jar 复制到容器内部，并重命名为 app.jar
#    左边 user-service.jar = 你服务器上的 jar 包
#    右边 app.jar = 容器里运行的包名
COPY user-service.jar app.jar

# 3. 声明容器对外暴露的端口（只是声明，不真正发布端口）
#    这里要和你微服务项目里的 server.port 一致
EXPOSE 8090

# 4. 容器启动命令：java -jar app.jar（启动 SpringBoot 项目）
ENTRYPOINT ["java","-jar","app.jar"]
```

### 4. 构建镜像 & 启动容器

```bash
# -------------------------------
# 1. 构建 Docker 镜像
# -------------------------------
# docker build       构建命令
# -t user-service    给镜像打标签（名字叫 user-service，方便识别）
# .                  表示使用当前目录下的 Dockerfile 构建
# -------------------------------
docker build -t user-service .

# -------------------------------
# 2. 启动 Docker 容器
# -------------------------------
docker run -d \                  # -d 后台运行容器（不占用命令行窗口）
  --name user-service \          # 给容器起名字（唯一标识，方便管理）
  --net host \                   # 关键：使用主机网络模式 → 直接用宿主机IP，能访问所有中间件
  # -- net-jc-club \
  user-service                   # 指定用哪个镜像启动（和上面构建的镜像名一致）
```

> 微服务推荐使用 `--net host` 网络模式，避免 Nacos 注册 IP 错误。

**微服务必须和中间件加入同一个 Docker 网络**

```nginx
server {

        root   /usr/share/nginx/html;
        index  index.html index.htm;
        # 解决 Vue history 路由刷新 404
        try_files $uri $uri/ /index.html;
    }

    # 代理后端网关
    # 重点：把 gateway 换成你 CentOS 真实IP！
    location /api/ {
        proxy_pass http://192.168.137.128:8080/;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }


    error_page   500 502 503 504  /50x.html;
    location = /50x.html {
        root   /usr/share/nginx/html;
    }
}

```

#### 1. 为什么容器内部 **永远可以是 80 端口**？

你现在：
- 容器A：里面 Nginx 监听 **80**

因为：
- 容器A 的 80 ≠ 容器B 的 80 ≠ 容器C 的 80

  - 小明家的门牌号 80
  - 小红家的门牌号 80
  - 小刚家的门牌号 80
**不冲突，因为不是同一栋房子！**

#### 错误原因

如果依然显示 404，通常是由于以下三个原因之一：

##### 1. Nginx 进程没有真正刷新（最常见原因）

Windows 上的 Nginx 有时会出现“僵尸进程”，导致你运行 `reload` 并没有真正加载新配置。
**请执行以下命令强力重启：**
1.  打开任务管理器，结束所有名为 `nginx.exe` 的进程。
2.  或者在命令行执行：
    ```powershell
    taskkill /f /im nginx.exe
    ```
3.  重新在 Nginx 目录下启动：
    ```powershell
    start nginx
    ```

##### 2. 权限问题

Nginx 作为一个服务，有时可能无法访问 `Desktop`（桌面）这种受保护的系统文件夹。
**尝试方法：**
1.  将 `theater-frontend/dist` 文件夹**复制**到盘符根目录，例如 `C:/theater-dist`。
2.  修改 `nginx.conf` 中的 `root` 为 `"C:/theater-dist";`。
3.  重启 Nginx 测试。

##### 3. 查看核心线索：`error.log`

Nginx 的日志会明确告诉你它在尝试访问哪个路径时失败了。
请打开 Nginx 安装目录下的 **`logs/error.log`**，找到最后一行报错，它看起来应该是这样的：
`".../dist/index.html" failed (3: The system cannot find the path specified)` 
或者是 `(13: Permission denied)`。

**请把 `error.log` 的最后几行报错贴出来，我能直接看出是路径写错了还是权限被拒绝了。**

##### 4. 检查是否有多个 `server` 块冲突

确认你的 `nginx.conf` 中没有其他 `listen 80;` 的配置。如果其他的配置排在前面，Nginx 会优先匹配那个。从你提供的代码看，目前只有一个。

---

**建议操作：** 先用 `taskkill` 杀掉所有进程再重启。如果还不行，请务必查看 `error.log`。

### ⚠️ 特别提醒：关于“后端接口”

如果您在 Windows 上只启动了前端，没有启动后端 Java 项目：
*   页面可以正常打开，UI 效果能看到。
*   但**登录、数据查询等功能会报错**（因为 Nginx 找不到后端网关）。

**建议操作：**
如果您想完整测试（含数据），建议直接在项目根目录运行我为您准备的：
```bash
docker-compose up -d --build
```
这样它会在您的 Windows 电脑上把前端、后端和数据库全部拉起来，您可以直接访问 `http://localhost` 看到一个完全可用的系统。
