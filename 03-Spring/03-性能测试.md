# 性能测试 

## 一、什么是性能测试
**性能测试**：通过工具模拟**多用户、高并发、大数据量**，
观测系统在不同压力下的：**响应速度、吞吐量、稳定性、资源消耗**，
提前发现瓶颈、预估容量、保障线上大促稳定。

---

## 二、核心四大指标（必背）
1. **TPS 每秒事务数**
   每秒处理的业务请求/事务量，代表**系统处理能力**
2. **QPS 每秒查询数**
   每秒接口请求数，纯查询接口常用
3. **RT 响应时间**
   单次请求耗时（平均RT、95%RT、99%RT），**95线最关键**
4. **并发数**
   同一时刻同时发起请求的用户/线程数

### 辅助指标
- 错误率：失败请求占比
- 资源监控：CPU、内存、磁盘IO、网络、GC、线程池
- 连接数：数据库连接、Redis连接、TCP连接

---

## 三、性能测试 分类（面试常问）
1. **压力测试 Stress**
   不断加压，找到系统**极限峰值、崩溃临界点**
2. **负载测试 Load**
   模拟**正常+预期峰值**流量，看稳定运行指标
3. **并发测试**
   模拟大量用户同时操作，发现并发锁、竞争、超卖问题
4. **稳定性测试/耐久测试**
   长时间（几小时/几天）持续压测，观察：内存泄漏、连接泄露、GC抖动
5. **容量测试**
   测算系统最大承载量、预估机器扩容数量
6. **弱网测试**
   模拟延迟、丢包，考察接口容错、超时、重试
7. **基准测试**
   迭代前后版本对比，验证优化是否生效

---

## 四、常用压测工具
1. **JMeter**：开源、Java技术栈最常用，功能全
2. **Postman / Apifox**：简单接口压测，轻量
3. **Gatling**：高并发、高性能，适合海量压测
4. **Locust**：Python编写脚本，分布式压测
5. **自研压测平台**：大厂内部，支持流量染色、全链路压测

---

## 五、标准性能测试流程
1. **需求梳理**
   确定核心接口、预期指标（TPS、RT、并发）
2. **测试环境准备**
   环境隔离、配置与线上对齐、预热缓存
3. **脚本编写**
   接口入参、请求头、关联、参数化、断言
4. **单接口基准压测**
   单个接口单独压，定位单点瓶颈
5. **混合场景压测**
   模拟真实用户链路（登录→浏览→下单→支付）
6. **监控采集**
   服务器+中间件+数据库全链路监控
7. **瓶颈分析 & 优化**
8. **出具报告**：指标数据、瓶颈点、优化建议

---

## 六、常见性能瓶颈 & 优化方向
### 1. 应用层瓶颈
- 接口逻辑冗余、循环查询
- 线程池耗尽、阻塞等待
- 频繁FullGC、内存泄漏
**优化**：代码优化、异步、线程池调优、JVM参数调优

### 2. 数据库瓶颈（最常见）
- 慢SQL、无索引、深分页
- 连接数打满、行锁竞争、事务过长
**优化**：加索引、SQL优化、分库分表、读写分离、缩短事务

### 3. 缓存瓶颈
- 缓存击穿/雪崩/穿透
- Redis 单节点瓶颈、热key、大key
**优化**：多级缓存、集群、本地缓存、Lua优化

### 4. 中间件瓶颈
- MQ堆积、消费能力不足
- 网关限流、转发瓶颈

### 5. 服务器资源瓶颈
CPU打满、内存不足、磁盘IO过高、网络带宽瓶颈

---

## 七、95%/99% 响应时间 为什么看这个？
- 平均RT会被大量正常请求**掩盖慢请求**
- **95%RT**：95%的请求都低于该耗时，代表用户真实体验
- 高并发下，长尾慢请求才是卡顿、超时的根源

---

## 八、全链路压测（大厂大促必备）
1. 线上隔离环境，使用**影子流量/压测账号**
2. 影子库、影子表，**不污染真实业务数据**
3. 模拟真实大促流量，全链路服务加压
4. 提前暴露集群、中间件、DB整体瓶颈

---

## 九、面试高频简答题（直接背）
1. **TPS和QPS区别**
QPS是接口请求数，TPS是完整业务事务；
一个事务可能包含多个接口请求。

2. **并发数≠TPS**
并发是同时线程数，TPS是单位时间处理能力；
并发高不代表吞吐量高，可能阻塞。

3. **压测报错常见原因**
连接超时、线程池满、DB连接耗尽、锁竞争、限流、GC频繁。

4. **如何定位性能瓶颈**
先看监控：CPU/GC/线程 → 再看日志慢接口 → 排查慢SQL → 检查缓存与中间件。

5. **稳定性测试关注点**
内存是否持续上涨、连接是否泄露、GC频率、接口成功率。

---

## 十、极简背诵总结
1. 四大指标：**TPS、QPS、RT、并发数**
2. 测试类型：负载、压力、并发、稳定性、容量
3. 工具：JMeter 主流
4. 瓶颈Top1：**数据库慢SQL + 索引**
5. 优化思路：缓存、异步、SQL、集群、资源扩容

# **JMeter 压测 + 线上调优**

## 一、JMeter 完整实操压测步骤（可直接照着操作）

### 1. 前期准备

1. 安装 JMeter + JDK8+
2. 关闭本机防火墙、杀毒，避免限制并发
3. 被测环境信息：接口地址、请求方式、Header、Token、参数
4. 确定压测指标：
   - 并发线程数：100/200/500
   - 压测时长：3~5分钟
   - 目标：TPS、95%RT、错误率＜0.01%

---

### 2. 新建测试计划 & 组件添加

1. 新建**测试计划**
2. 添加 **线程组**（核心）
   - 线程数：并发数
   - 循环次数：永久
   - 调度器：设置持续时间
3. 添加 **HTTP请求**
   - 协议/IP/端口/请求路径
   - GET/POST、JSON参数、表单参数
4. 添加 **请求头管理器**
   配置：`Content-Type: application/json`、token、鉴权头
5. 添加**结果查看器**（压测运行前关闭，防止占用内存）
   - 汇总报告
   - 察看结果树
   - 响应时间图形

---

### 3. 关键压测配置（必设）

#### （1）线程组设置

- 线程数：模拟并发用户
- 预热时间：30s（避免冷启动误差）
- 持续运行：180s~300s

#### （2）全局优化（JMeter 自身不卡顿）

`jmeter.properties`
```properties
# 关闭日志、减少IO
log_level.jmeter=WARN
# 关闭结果保存
jmeter.save.saveservice.output_format=csv
```

#### （3）分布式压测（高并发）

一台机器压力有限，多台机器分布式施压。

---

### 4. 三种常用压测模型

#### ① 并发测试（固定线程）

固定200线程一直请求，看稳定TPS、RT
> 适合：秒杀、抢购、接口并发竞争

#### ② 阶梯加压（梯度压测）

50→100→200→500 逐步加线程
> 适合：找系统性能拐点、最大TPS

#### ③ 恒定吞吐量压测

用「吞吐量控制器」固定QPS施压
> 适合：模拟线上平稳流量

---

### 5. 运行压测 & 指标观察

#### 核心看 5 个数据

1. **TPS/Throughput**：系统处理能力
2. **95% ResponseTime**：用户体验核心
3. **错误率**：正常必须接近0
4. **CPU/内存**：服务器资源
5. **GC、线程池、DB连接**：中间件&应用指标

> 建议：**命令行压测**，GUI只用来编写脚本
```bash
jmeter -n -t 测试计划.jmx -l result.jtl -e -o report
```

---

### 6. 压测后分析瓶颈

1. RT突然飙升 → 接口阻塞、锁竞争、慢SQL
2. TPS上不去 → 线程池满、DB瓶颈、Redis瓶颈
3. 错误率上升 → 连接超时、连接数耗尽、限流
4. CPU 100% → 代码死循环、频繁FullGC、序列化耗时

---

## 二、线上性能调优高频参数（面试+工作直接用）

### 1. JVM 调优参数（最常用）

#### 基础内存配置

```bash
-Xms2g -Xmx2g      # 堆内存固定，防止扩容缩容震荡
-XX:MetaspaceSize=512m
-XX:MaxMetaspaceSize=512m
```

#### GC 优化（G1 主流）

```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=50   # 最大停顿50ms
-XX:G1HeapRegionSize=16m
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/dump
```

#### 高频优化

- 禁止动态扩容：`-Xms = -Xmx`
- 减少FullGC：合理设置新生代比例
- 内存泄漏：开启OOM快照、定时分析dump

---

### 2. Tomcat / Web容器调优

```properties
# 核心线程池
server.tomcat.core-threads=200
server.tomcat.max-threads=800
server.tomcat.accept-count=200
server.tomcat.connection-timeout=20000
```
- core：核心常驻线程
- max：最大并发
- accept-count：等待队列

---

### 3. 线程池调优（Spring 业务线程池）

**CPU密集型**：核心线程数 = CPU核心数 + 1
**IO密集型**（接口/DB/Redis）：核心线程数 = CPU核心数 * 2 ~ 20

```java
ThreadPoolExecutor(
    corePoolSize = 50,
    maximumPoolSize = 200,
    keepAliveTime = 60,
    queueCapacity = 1000
)
```
原则：
- 队列不要过大，避免请求积压无拒绝
- 拒绝策略配置降级/排队，防止雪崩

---

### 4. MySQL 性能高频参数

```ini
# 连接池
max_connections = 1000
wait_timeout = 600
interactive_timeout = 600

# 缓冲
innodb_buffer_pool_size = 50%物理内存
innodb_log_file_size = 2G
innodb_flush_log_at_trx_commit = 1

# 并发
innodb_read_io_threads = 32
innodb_write_io_threads = 32
```
优化重点：
- 加大缓冲池、减少磁盘IO
- 合理超时，干掉闲置连接
- 避免长事务

---

### 5. Redis 调优参数

```conf
# 最大内存
maxmemory 4g
# 淘汰策略（最热）
maxmemory-policy allkeys-lru
# 网络
tcp-backlog 511
timeout 0
# 持久化平衡性能
appendfsync everysec
```
高频问题优化：
- 大key拆分、热key本地缓存
- 合理过期时间，避免批量过期

---

### 6. Linux 系统内核调优（线上必配）

```bash
# 最大文件句柄
ulimit -n 65535

# 网络连接优化
net.ipv4.tcp_syncookies = 1
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_fin_timeout = 30
net.core.somaxconn = 1024
```
作用：
- 解决高并发下**端口耗尽、time_wait 过多**
- 提升单机长连接上限

---

### 7. SpringBoot 应用优化

```yaml
spring:
  datasource:
    hikari:
      minimum-idle: 10
      maximum-pool-size: 30
      idle-timeout: 600000
  redis:
    lettuce:
      pool:
        max-active: 50
```
- 数据库连接池不要过大，防止DB压垮
- Redis 连接池合理限制，避免连接打爆
