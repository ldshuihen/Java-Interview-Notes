# Redis 8大数据类型 

Redis 核心**8种数据类型**：
**String、List、Hash、Set、Sorted Set、Bitmap、HyperLogLog、Geospatial**

## 一、String 字符串

### 特点

- 最基础类型，**key-value**
- 最大**512MB**
- 可以存：字符串、数字、二进制、JSON
- **底层是简单动态字符串SDS**

### 常用命令
```
set/get/del/expire
incr/decr  自增自减
mset/mget  批量读写
setnx      不存在才设置（分布式锁）
```
### 应用场景

- 普通缓存、用户信息、配置信息
- **计数器**：点赞、浏览量、订单数
- **分布式锁**：setnx
- 限流、全局唯一ID

### Redis String 底层原理

#### 一、先明确：Redis 的 String 底层不是 C 语言原生字符串

C 语言原生字符串：
- 以 **`\0` 空字符** 结尾
- 不记录长度，每次求长度要遍历找 `\0`
- 二进制不安全、容易缓冲区溢出

**Redis 自定义了一种结构：SDS（Simple Dynamic String，简单动态字符串）**
Redis 所有 String 类型、底层字符串都是用 **SDS** 实现。

---

#### 二、SDS 结构体源码（Redis 3.2 以后 5 种编码）

Redis 为了**节省内存**，设计了 **5 种 SDS 类型**，根据字符串长度自动选择：
```
sdshdr5
sdshdr8
sdshdr16
sdshdr32
sdshdr64
```
以 `sdshdr8` 为例结构：
```c
struct sdshdr8 {
    uint8_t len;     // 已使用长度
    uint8_t alloc;   // 已分配总容量（不含末尾\0）
    unsigned char flags; // 低3位标识类型，高5位保留
    char buf[];      // 真实字符数组，末尾自动补 \0
};
```
核心字段：
1. **len**：字符串实际长度
2. **alloc**：分配的总缓冲区容量
3. **flags**：标记是哪一种 SDS
4. **buf[]**：真正存数据的字节数组，**二进制安全**

---

#### 三、SDS 相比 C 原生字符串 4 大优势

##### 1. O(1) 获取字符串长度

C 字符串：`strlen` 遍历找 `\0`，O(n)
SDS：**直接读 len 属性**，O(1)

##### 2. 杜绝缓冲区溢出

C 字符串修改前不知道长度，容易溢出
SDS 修改前先判断容量，**不够自动扩容**，安全。

##### 3. 减少内存分配次数（预分配空间）

- **空间预分配**：字符串变长时，额外多分配冗余空间，避免每次修改都 realloc
- **惰性空间释放**：字符串缩短时，**不立即回收内存**，留着下次复用

大幅减少系统调用，性能更高。

##### 4. 二进制安全

C 字符串遇到 **`\0` 就截断**，不能存图片、序列化二进制数据。
SDS 以 **len 长度** 界定字符串，**不管中间有没有 \0**，可以存任意二进制数据。

---

#### 四、Redis String 三种底层编码

Redis 的 String 对外是一种类型，底层有 **3 种编码**，对象头 `encoding` 标记：

1. **INT 编码**
   存**整数**，且在 long 范围内，直接用数字存储，不存字符串，最省内存。

2. **EMBSTR 编码**
   短字符串（**长度 ≤ 44 字节**）
   SDS 结构体 + buf 连续一块内存，**一次内存分配**，内存紧凑、访问快。

3. **RAW 编码**
   长字符串（**长度 > 44 字节**）
   SDS 结构体 和 buf **分开两次分配**，支持更大长度和动态扩容。

> 关键点：**44 字节** 是 EMBSTR 和 RAW 分界线，面试常问。

---

#### 五、Redis String 限制

- 单个 String 最大 **512MB**
- 本质就是：**SDS 动态字符串**

---

#### 六、面试一句话总结

1. Redis String 底层是 **SDS 简单动态字符串**，不是 C 原生字符串。
2. SDS 优势：**O(1) 求长度、防溢出、空间预分配+惰性释放、二进制安全**。
3. 底层三种编码：**INT（整数）、EMBSTR（≤44字节短串）、RAW（>44字节长串）**。
4. 核心结构：**len 实际长度 + alloc 分配容量 + buf 字节数组**。

## 二、List 列表

### 特点
- **双向链表**，有序、可重复
- 两头进出，**左右都能操作**
- 插入删除极快，索引查询慢

### 常用命令
```
lpush/lpop  左进左出
rpush/rpop  右进右出
lrange      范围查询
llen        获取长度
```
### 应用场景
- **消息队列**、任务队列
- 朋友圈时间线、关注列表
- 栈、队列结构实现

### Redis List 底层原理

#### 一、核心结论

Redis **List 列表**，在 Redis 3.2 之后底层统一用：
**quicklist（快速链表）**

前身：
- Redis 3.2 之前：**ziplist 压缩列表 + adlist 双向链表**
- 3.2 之后直接合并优化为 **quicklist**

#### 二、先搞懂两个基础结构

##### 1. adlist 普通双向链表

特点：
- 每个节点单独分配内存
- 前后指针，**头尾操作快**
- 缺点：**内存碎片多、内存占用大**

##### 2. ziplist 压缩列表

连续一块内存，紧凑存储，无指针开销，省内存；
适合**元素少、值小**的场景。

缺点：元素多了**修改扩容成本高**。

---

#### 三、quicklist 是什么？

**quicklist = 双向链表 + 每个节点内部是 ziplist**

结构形象理解：
```
quicklist(双向链表)
    ↓
节点1 [ziplist存一批数据] <--> 节点2 [ziplist] <--> 节点3 [ziplist]
```
- 外层：**普通双向链表**（quicklistNode）
- 每个链表节点内部：**用 ziplist 紧凑存多条 list 数据**

既保留链表**头尾增删快**的优点，
又用 ziplist **省内存、减少内存碎片**。

---

#### 四、quicklist 源码结构

##### 1. quicklist 整体结构

```c
typedef struct quicklist {
    quicklistNode *head;  // 头节点
    quicklistNode *tail;  // 尾节点
    long count;           // 总元素个数
    int fill;             // 每个节点ziplist最大容纳数量
    int compress;         // 两端压缩深度（中间LZF压缩）
} quicklist;
```

##### 2. quicklistNode 链表节点

```c
typedef struct quicklistNode {
    struct quicklistNode *prev;
    struct quicklistNode *next;
    unsigned char *zl;   // 指向ziplist数据
    unsigned int sz;     // ziplist占用字节大小
    unsigned int count : 16; // 当前节点元素个数
    unsigned int encoding : 2; // 是否压缩
    unsigned int container : 2;
    unsigned int recompress : 1;
    // ... 其他标记位
} quicklistNode;
```

---

#### 五、工作机制

1. 插入数据时，往**头尾 ziplist** 里塞元素
2. 当一个 ziplist 元素数量超过 `fill` 阈值，**新建一个 quicklistNode 节点**
3. 中间节点可以开启 **LZF 压缩**，进一步节省内存
4. 首尾节点不压缩，保证**左右进出性能**

---

#### 六、List 特性为什么这么设计？

1. **首尾增删极快** O(1) → 适合队列、栈、时间线
2. **内存非常紧凑**，比纯双向链表省很多空间
3. **支持压缩**，海量列表数据也不占内存
4. **不适合中间随机访问**，遍历只能从头到尾/从尾到头

---

#### 七、Redis List 适用场景

- 消息队列、任务队列（lpush/rpop）
- 朋友圈、关注流、时间线
- 栈、排队列表

---

#### 八、面试必背总结

1. Redis List 3.2 后底层是 **quicklist**。
2. quicklist = **双向链表 + 每个节点内嵌 ziplist**。
3. 兼顾：**双向链表的头尾操作高性能** + **ziplist 低内存占用**。
4. 支持 **LZF 压缩**，中间节点压缩、首尾不压缩，平衡性能和内存。
5. 不适合随机索引访问，适合**左右进出、队列、时间线**场景。

## 三、Hash 哈希

### 特点
- 类似 **Map<String, Map<field,value>>**
- 一个key，存多个field-value
- 适合存**对象**，比String存JSON更省内存

### 常用命令
```
hset/hget
hmset/hmget
hgetall 获取所有字段
hdel 删除字段
```
### 应用场景
- 存储**用户信息、商品信息**
- 无需序列化，单独修改某个字段
- 购物车缓存

###  Redis Hash 底层原理

#### 一、Redis Hash 两种底层编码

Redis 的 Hash 类型有**两种底层编码**，会**自动切换**：
1. **ziplist（压缩列表）** —— 少量、小数据时
2. **hashtable（字典/哈希表）** —— 数量多、数据大时

#### 二、什么时候用 ziplist？

满足全部条件，底层用 **ziplist**：
1. Hash 元素**个数 ≤ 512 个**
2. 每个 `field` 和 `value` 的**字符串长度都 ≤ 64 字节**

超出任意一个阈值，**自动转成 hashtable**。

配置文件对应参数：
```hash
hash-max-ziplist-entries 512
hash-max-ziplist-value 64
```

---

#### 三、ziplist 压缩列表 底层结构

##### 存储形式

**连续内存数组**，紧凑排列，无多余指针开销。
Hash 在 ziplist 里存储顺序：
```
field1 -> value1 -> field2 -> value2 -> field3 -> value3 ...
```
按 **field、value 两两挨着** 顺序存放。

##### 优点

- 内存**连续紧凑**，节省空间
- 缓存命中率高
##### 缺点

- 不适合大量元素
- 中间增删会**引发内存重新分配、数据搬迁**，效率低

---

#### 四、hashtable 字典 底层结构

当 Hash 元素多、字段长，就升级为 **Redis 字典 dict**。

##### 整体结构

`dict` 里面依赖 **哈希表 + 链表** 解决哈希冲突。
结构层级：
```
dict
 ├── dictType  操作函数
 ├── ht[2]     两个哈希表（用于扩容/缩容）
 └── rehashidx 渐进式rehash标记
```

`ht[0]` 主哈希表，`ht[1]` 用于**扩容、缩容 rehash**。

每个哈希表里面是 **dictEntry 数组**：
```
dictEntry
    key     // hash的field
    val     // hash的value
    next    // 链表指针，解决哈希冲突
```
哈希冲突采用 **链地址法**，同下标元素连成单向链表。

---

#### 五、渐进式 Rehash（重点面试必问）

Redis Hash 底层字典扩容不是一次性搬完，而是**渐进式 rehash**：
1. 新建更大的哈希表 `ht[1]`
2. 维护 `rehashidx` 索引标记
3. **每次增删改查** 顺带搬一部分数据到新表
4. 所有数据迁移完，释放旧表，切换 `ht[1]` 为主表

**好处**：
避免一次性大量拷贝导致**主线程阻塞**，保证高性能不卡顿。

---

#### 六、两种编码对比

| 编码      | 适用场景       | 优点               | 缺点                   |
| --------- | -------------- | ------------------ | ---------------------- |
| ziplist   | 元素少、字段短 | 省内存、连续内存快 | 增删慢、不能存长字段   |
| hashtable | 元素多、字段长 | 增删改查 O(1)      | 内存占用高、有指针开销 |

---

#### 七、面试一句话总结

1. Redis Hash 底层两种编码：**ziplist、hashtable**，自动转换。
2. 小于 **512 个元素** 且 **field/value 都小于64字节** 用 ziplist。
3. ziplist 连续内存，按 **field-value 顺序紧凑存储**，省内存。
4. 超出阈值转 **dict 字典**，基于数组+单向链表解决哈希冲突。
5. 字典扩容采用 **渐进式 rehash**，分步迁移，不阻塞主线程。

## 四、Set 集合

### 特点
- **无序、不可重复**
- 底层：哈希表
- 支持**交集、并集、差集**

### 常用命令
```
sadd/smembers
sismember 判断是否存在
sinter 交集
sunion 并集
sdiff  差集
```
### 应用场景
- 点赞、收藏、去重
- 共同好友、共同关注
- 抽奖随机去重

###  Redis Set 集合底层原理

#### 一、Set 底层两种编码

Redis Set 有 **2 种底层编码**，自动转换：
1. **intset（整数集合）**
2. **hashtable（字典哈希表）**

---

#### 二、什么时候用 intset？

同时满足两个条件用 **intset**：
1. 集合中**所有元素都是整数**
2. 元素个数 ≤ **512 个**

任意一个不满足 → 自动转 **hashtable**

配置参数：
```
set-max-intset-entries 512
```

---

#### 三、intset 整数集合底层结构

##### 结构体

```c
typedef struct intset {
    uint32_t encoding;  // 编码层级：16位/32位/64位整数
    uint32_t length;    // 元素个数
    int8_t contents[];  // 有序紧凑存整数数组
} intset;
```

##### 特点

1. **连续内存、有序、无重复**
2. 只存**整数**，极度省内存
3. 底层是**有序数组**，查找用**二分查找 O(log n)**
4. 自动**升级编码**：存入更大数值自动从16位→32位→64位
5. **不支持降级**：升级后不能退回小编码

##### 存储形式

纯整数按从小到大有序排列，无指针开销，内存极致紧凑。

---

#### 四、什么时候转 hashtable？

出现以下任意情况：
- 加入**非字符串、非整数**元素
- 元素数量超过 **512**
- 存入超大整数超出当前 intset 编码范围

立刻转为 **dict 字典（hashtable）**

##### hashtable 存储方式

Set 底层复用 Redis 字典：
- **dict 的 key**：就是 Set 的元素
- **dict 的 value**：全部为 `NULL`

只利用字典 key **天然去重**特性，实现集合不可重复。
哈希冲突：**数组+单向链表 链地址法**。

---

#### 五、intset 和 hashtable 对比

| 编码      | 适用场景       | 底层结构     | 性能                    |
| --------- | -------------- | ------------ | ----------------------- |
| intset    | 全整数、数量少 | 有序连续数组 | 省内存、二分查找        |
| hashtable | 字符串/数量多  | 字典哈希表   | O(1) 增删查，占用内存大 |

---

#### 六、面试必背总结

1. Redis Set 底层两种编码：**intset、hashtable**。
2. 全是整数且元素≤512用 **intset**，否则转 **hashtable**。
3. intset：**有序整数数组、二分查找、自动编码升级、不降级**。
4. hashtable：复用字典，**key存元素、value为null**，利用字典去重。
5. intset 省内存，hashtable 适合复杂元素和大数据量。

## 五、Sorted Set / ZSet 有序集合

### 特点
- **不重复 + 自动排序**
- 每个元素带 **score 分数**，按分数排序
- 底层：跳表+哈希

### 常用命令
```
zadd  添加带分数
zrange 正序
zrevrange 倒序
zrank 排名
```
### 应用场景
- **排行榜**：热搜、积分榜、销量榜
- 延时队列、带权重任务排序



### Redis ZSet 底层原理

#### 一、ZSet 两种底层编码（自动切换）

ZSet 有 **两种编码**，根据元素数量和元素大小自动转换：
1. **ziplist 压缩列表** （数据少、分值小、元素短）
2. **skiplist 跳表 + dict 字典** （数据多、元素长）

切换阈值配置：
```conf
zset-max-ziplist-entries 128   # 元素个数 ≤128
zset-max-ziplist-value 64      # member 长度 ≤64 字节
```
**同时满足**用 ziplist；**任意超了**直接转跳表结构。

---

#### 二、编码1：ziplist 模式

##### 存储结构

ziplist 连续内存紧凑存放，成对保存：
```
member1 → score1 → member2 → score2 → member3 → score3 ...
```
- 按 **score 分值从小到大有序排列**
- 无指针开销，内存极度节省
- 不用额外索引，靠物理顺序维持有序

##### 优缺点

- 优点：内存连续、省空间、缓存命中率高
- 缺点：元素多/过长时，插入排序要**内存搬迁**，性能变差

---

#### 三、编码2：skiplist + dict 复合结构（重点）

超出阈值后，ZSet 底层是：
**zset 结构体 = 跳表(skiplist) + 字典(dict)**

##### 1. zset 整体结构

```c
typedef struct zset {
    zskiplist *sl;   // 跳表：负责按score排序、范围遍历
    dict *dict;      // 哈希字典：member -> score 映射
} zset;
```

**双结构分工**

- **跳表 skiplist**：按 score 排序，支持**范围查询、排行榜、正序倒序**
- **字典 dict**：通过 member **O(1) 快速查分值**、去重

##### 2. 跳表 zskiplist 结构

```c
typedef struct zskiplist {
    zskiplistNode *header; // 头结点
    zskiplistNode *tail;   // 尾结点
    unsigned long length;  // 元素个数
    int level;             // 当前最大层数
} zskiplist;
```

##### 3. 跳表节点 zskiplistNode

```c
typedef struct zskiplistNode {
    sds ele;           // 元素member
    double score;      // 分值
    struct zskiplistNode *backward; // 后退指针
    zskiplistLevel level[]; // 多层前进指针
} zskiplistNode;
```

**跳表核心特点**
- 多层索引结构，**空间换时间**
- 查询、插入、删除时间复杂度 **O(log n)**
- 每层都是有序链表，高层快速跳跃，底层全元素有序
- 有**后退指针**，支持从尾向前遍历

---

#### 四、为什么 ZSet 要用 跳表+字典 双结构？

1. 只靠跳表：根据 member 查 score 要遍历，太慢
2. 只靠字典：没法按 score 做范围排序、分页、排行榜
3. 组合后：
   - 查分值：走 dict O(1)
   - 排行、区间、分页：走 skiplist O(logn)
   - 天然去重 + 天然有序

---

#### 五、ziplist 和 跳表模式对比

| 编码          | 适用条件                  | 底层结构                      | 优势                     |
| ------------- | ------------------------- | ----------------------------- | ------------------------ |
| ziplist       | 元素≤128 && member≤64字节 | 压缩列表，按score有序紧凑存储 | 省内存、无指针开销       |
| skiplist+dict | 超出任一阈值              | 跳表排序 + 字典映射           | 增删改查快、支持范围排行 |

---

#### 六、面试必背总结

1. ZSet 底层两种编码：**ziplist、skiplist+dict**。
2. 元素**≤128个**且 **member长度≤64字节** 用 ziplist，否则转跳表。
3. ziplist 按 `member+score` 两两连续存放，**按分值物理有序**。
4. 跳表模式是**zset结构体**，包含：**skiplist 跳表 + dict 字典**。
5. 跳表负责**排序、范围查询、排行榜**；字典负责**member快速查score、去重**。
6. 跳表时间复杂度 **O(log n)**，Redis 用跳表不用红黑树：**实现简单、支持范围遍历、内存更友好**。

5大底层基础组件：

### 1. SDS 简单动态字符串

**作用**：Redis 所有字符串的底层实现，替代C原生字符串。
**特点**：

- O(1) 获取长度
- 二进制安全（能存图片、二进制、含`\0`数据）
- 空间预分配 + 惰性释放，减少扩容次数
**支撑**：String、Hash的field/value、Set元素、ZSet元素全是SDS。

#### 一、作用

Redis 自定义 **SDS（Simple Dynamic String，简单动态字符串）**，**替代 C 语言原生字符串**；
是 Redis 所有字符串底层实现，所有 key、value、field、集合元素**本质都是 SDS**。

#### 二、为什么不用 C 原生字符串

C 原生字符串：
- 以 `\0` 结尾，无长度字段
- `strlen` O(n) 遍历求值长度
- 遇到 `\0` 直接截断，**二进制不安全**
- 不做容量预估，容易缓冲区溢出、频繁内存分配

#### 三、SDS 源码结构体（Redis 3.2+ 5种规格）

为了**内存极致节省**，Redis 按字符串长度分 5 种 SDS 头部：
`sdshdr5`、`sdshdr8`、`sdshdr16`、`sdshdr32`、`sdshdr64`

以最常用 **sdshdr8** 为例源码：
```c
struct sdshdr8 {
    // 已使用的字符串实际长度
    uint8_t len;
    // 已分配缓冲区总容量（不含头部和末尾\0）
    uint8_t alloc;
    // 低3位标识SDS类型，高5位保留
    unsigned char flags;
    // 真实字符数组，末尾自动补 \0
    char buf[];
};
```
字段解释：
- **len**：实际有效长度 → **O(1) 直接获取长度**
- **alloc**：已分配总容量，记录冗余空间
- **flags**：区分 sdshdr5/8/16/32/64
- **buf[]**：真实数据缓冲区，**末尾主动加 \0**，兼容 C 函数

#### 四、核心特点（逐条详解）

##### 1. O(1) 获取字符串长度

C 原生：`strlen` 遍历到 `\0`，O(n)
SDS：直接读 `len` 字段，**O(1)**

##### 2. 二进制安全

C 字符串遇到中间 `\0` 就截断；
SDS 靠 **len 长度** 界定数据，不管内部有没有 `\0`，
可以存**图片、序列化二进制、特殊字符**。

##### 3. 空间预分配（减少扩容次数）

当 SDS 字符串变长、需要扩容时：
- 如果新长度 **< 1MB**：**额外预分配同等大小冗余空间**
- 如果新长度 **≥ 1MB**：固定预分配 **1MB 冗余**

下次追加内容不用立刻重新申请内存，**减少系统调用、提升性能**。

##### 4. 惰性空间释放

当字符串**缩短**时：
SDS **不立刻把多余内存还给操作系统**，
保留冗余空间，后续再追加可以直接复用，避免频繁内存分配释放。

##### 5. 杜绝缓冲区溢出

修改字符串前，先对比 `alloc` 容量，不够就**自动扩容**，
不会像 C 字符串直接往后覆盖内存，杜绝缓冲区溢出。

#### 五、Redis String 底层三种编码（基于SDS）

1. **INT**：纯数字且在 long 范围，直接存整数，不走SDS字符数组
2. **EMBSTR**：长度 **≤44字节**，SDS头+buf**连续一块内存**，一次分配
3. **RAW**：长度 **>44字节**，SDS头与buf**分开两次分配**，支持更大动态扩容

#### 六、支撑范围

所有 Redis 用到字符串的地方全部基于 SDS：
- String 类型底层
- Hash 的 field、value
- Set 集合元素
- ZSet 的 member 成员
- Redis 所有 key、日志、配置字符串

#### 七、面试极简背诵版

SDS 是 Redis 自研动态字符串，替代 C 原生字符串；
结构体含 **len、alloc、flags、buf**；
具备 **O(1) 求长度、二进制安全、空间预分配、惰性释放、防缓冲区溢出** 五大特性；
所有 Redis key、String、Hash、Set、ZSet 字符串底层全是 SDS。

### 2. ziplist 压缩列表

**作用**：**连续内存紧凑存储**，无指针开销，极度省内存。
**特点**：
- 一块连续内存，挨个存数据
- 适合**元素少、内容短**
- 增删会触发内存搬迁，量大性能差
**支撑**：
- List → quicklist 每个节点内部是 ziplist
- Hash → 元素少、字段短时用 ziplist
- ZSet → 元素少、member短时用 ziplist

#### 作用

Redis 自研**连续内存紧凑型数据结构**，无多余指针开销，**极致节省内存**；
专门适配**元素数量少、单个元素内容短**的场景，是 Redis 节省内存的核心基础结构。

#### 特点

- 整体是**一整块连续内存**，内存排布紧凑，无链表前后指针开销
- 顺序挨个存放数据，物理内存有序
- 适合：**元素少、值长度小** 的低容量场景
- 中间插入/删除会引发**内存重分配、整体数据搬迁**，元素越多性能越差
- 无需额外元数据指针，内存占用远低于普通双向链表

#### 支撑范围

- List：底层 quicklist **每个节点内部封装 ziplist**
- Hash：元素数量少、field/value 较短时，底层采用 ziplist
- ZSet：元素个数少、member 字符串较短时，底层采用 ziplist

---

#### 一、ziplist 整体内存布局（源码结构）

整体固定头部 + 末端结束标记：
```
zlbytes | zltail | zllen | entry1 | entry2 | ... | entryN | zlend
```

字段说明：
1. **zlbytes**：整个 ziplist 占用**总字节数**
2. **zltail**：最后一个 entry 的**偏移量**，支持快速定位尾节点
3. **zllen**：元素 entry 个数
4. **entry**：实际每条数据节点（可存整数/字符串）
5. **zlend**：结束标记，固定 `0xFF`

---

#### 二、单个 entry 节点内部结构

每个 entry 三层结构：
```
prevlen | encoding | content
```

1. **prevlen**：前一个 entry 的字节长度
   - 小于254字节：占1字节
   - 大于等于254字节：占5字节
   作用：**支持从后往前遍历**

2. **encoding**：编码类型
   - 标记当前 content 是**整数**还是**字符串**
   - 记录长度、格式，解析时用

3. **content**：真正存储的数据
   - 可以存整数、短字符串、长字符串

---

#### 三、核心工作原理

1. 所有数据**挤在一块连续内存**，没有链表 `prev/next` 指针，省大量内存
2. 靠 **prevlen** 实现双向遍历，不用指针
3. 数据按写入顺序物理排列
4. 当在**中间插入/删除**元素：
   - 必须重新分配整块内存
   - 后续所有元素整体向后/向前搬迁
   → **元素多了性能急剧下降**，所以只适合小数据量

---

#### 四、ziplist 触发编码降级/结构切换阈值

1. Hash
```
hash-max-ziplist-entries 512
hash-max-ziplist-value    64
```
元素超512 或 单个field/value超64字节 → 转 dict

2. ZSet
```
zset-max-ziplist-entries 128
zset-max-ziplist-value   64
```
元素超128 或 member超64字节 → 转 skiplist+dict

3. List
quicklist 每个节点 ziplist 元素数量由 `quicklist-fill` 控制，满了就新建节点

---

#### 五、面试极简总结

ziplist 是**连续内存压缩列表**，无指针、极度省内存；
由 **zlbytes、zltail、zllen、entry 列表、zlend** 组成；
单个 entry 由 **prevlen、encoding、content** 构成；
靠偏移量实现双向遍历，**中间增删会内存搬迁**；
为 List/Hash/ZSet 小数据场景做内存优化，超阈值自动切换为 dict/跳表。

### 3. intset 整数集合

**作用**：专门**只存整数**的有序数组。
**特点**：
- 有序、无重复、二分查找 O(logn)
- 自动升级编码（16→32→64位），**只能升级不能降级**
**支撑**：
- Set 集合：全整数且数量≤512时，底层用 intset

#### 作用

Redis 专门**纯存整数**的底层基础结构，**有序、去重、内存紧凑**，只为节省内存设计；
作为 **Set 集合** 在全整数、元素少时的底层实现。

#### 特点

- 底层是**有序整型数组**，元素从小到大排列、天然去重
- 查找采用**二分查找**，时间复杂度 **O(log n)**
- 支持**自动编码升级**：16位 → 32位 → 64位
- **只能升级、不能降级**，一旦存了大数，整体都升级为64位，无法退回小编码
- 连续内存存储，无指针开销，内存占用极低

#### 支撑范围

- **Set 集合**：
  所有元素都是整数 **且元素个数 ≤ 512** 时，底层使用 `intset`；
  不满足任意条件，自动转为 `dict 哈希表`。
  配置阈值：`set-max-intset-entries 512`

---

#### 一、intset 源码结构体

```c
typedef struct intset {
    // 编码类型：INTSET_ENC_16 / 32 / 64
    uint32_t encoding;
    // 集合元素个数
    uint32_t length;
    // 有序整数数组，真实存所有整数
    int8_t contents[];
} intset;
```

##### 字段详解

1. **encoding**
   三种编码级别：
   - 16位 ：`INTSET_ENC_INT16`
   - 32位 ：`INTSET_ENC_INT32`
   - 64位 ：`INTSET_ENC_INT64`

2. **length**
   数组中实际整数元素个数。

3. **contents[]**
   柔性数组，**连续内存、从小到大有序存放整数**；
   虽然类型是 `int8_t`，但按 encoding 规则解读为16/32/64位整数。

---

#### 二、核心工作原理

##### 1. 有序 + 去重

插入整数时，通过二分查找定位位置：
- 已存在 → 直接拒绝，保证**去重**
- 不存在 → 插入到对应下标，保持数组**全局有序**

##### 2. 自动编码升级

当插入一个**超出当前编码范围**的整数：
- 例如当前是16位，插入超16位范围的数
- 自动整体升级到32位，后续全部元素都按32位存
- 再遇更大数，继续升级到64位

**规则：只升不降**
一旦升级为64位，即使后面删掉大数，也不会退回到32/16位。

##### 3. 内存布局优势

整块连续内存，无链表指针、无哈希表冗余，
存纯整数场景**内存极致节省**。

---

#### 三、什么时候从 intset 转 dict

满足任一条件立即切换：
1. 插入**非整数字符串**元素
2. 元素总数 **超过 512**
3. 插入数值超出当前编码，触发升级仍放不下

转为 dict 后：
- dict 的 key 存 Set 元素
- dict 的 value 全部为 NULL
- 利用字典 key 天然去重特性

### 4. dict 字典（哈希表）

**作用**：Redis 自研哈希字典，**key-value 映射**。
**特点**：
- 底层数组+单向链表，**链地址法解决哈希冲突**
- **渐进式 rehash**：扩容/缩容分步迁移，不阻塞主线程
**支撑**：
- Hash 字段多/值超长 → 转 dict
- Set 非整数/超数量 → 转 dict（key存元素，value=null）
- ZSet 跳表模式里 搭配 dict 做 member→score 映射

#### 作用

Redis 自研**哈希字典**，是 Redis 最核心的 **key-value 映射底层结构**；
承担所有复杂键值映射、去重、关联查询，是 Redis 高层数据结构的底层基石。

#### 特点

- 底层：**数组 + 单向链表**，采用**链地址法**解决哈希冲突
- 扩容缩容采用**渐进式 rehash**，分步迁移数据，不阻塞主线程
- 查询、插入、删除时间复杂度 **O(1)**
- 自动扩容、自动缩容，负载均衡控制链表长度

#### 支撑范围

- Hash：元素超512个 / field/value 超长 → 切换为 dict
- Set：含非整数 / 元素超512个 → 切换为 dict（key=元素，value=null）
- ZSet：skiplist 模式下搭配 dict，做 **member → score** 快速映射
- Redis 全局所有 Key 空间、集群槽位、内部配置全部依赖 dict

---

#### 一、dict 三层源码结构

##### 1. 整体字典结构 dict

```c
typedef struct dict {
    dictType *type;       // 字典类型函数（哈希、比较、析构）
    void *privdata;       // 私有数据
    dictht ht[2];         // 两个哈希表：ht[0]主表，ht[1]rehash备用表
    long rehashidx;       // rehash 进度标记，-1 表示不在rehash
} dict;
```

##### 2. 哈希表 dictht

```c
typedef struct dictht {
    dictEntry **table;    // 哈希表数组，每个元素指向一条链表
    unsigned long size;   // 数组容量大小
    unsigned long sizemask;// 掩码，计算下标用：hash & sizemask
    unsigned long used;   // 已占用元素个数
} dictht;
```

##### 3. 哈希节点 dictEntry

```c
typedef struct dictEntry {
    void *key;            // 键
    union {
        void *val;
        uint64_t u64;
        int64_t s64;
        double d;
    } v;                  // 值（共用体，节省内存）
    struct dictEntry *next;// 单向链表指针，解决哈希冲突
} dictEntry;
```

---

#### 二、底层存储原理

1. 根据 key 计算**哈希值**
2. 哈希值 & sizemask 算出**数组下标**
3. 下标位置无数据 → 直接放入
4. 下标已有数据 → **挂单向链表尾部**（链地址法）
5. 链表过长触发扩容，开启 **渐进式 rehash**

---

#### 三、渐进式 rehash（面试核心重点）

##### 1. 什么时候触发 rehash

- 负载因子 `used/size > 阈值` 触发**扩容**
- 元素大量删除，负载因子过低触发**缩容**

##### 2. 渐进式 rehash 流程

1. 开辟新哈希表 **ht[1]**
2. 设置 `rehashidx = 0`，标记进入 rehash 状态
3. **每次增删改查命令**，顺带把 `rehashidx` 位置的链表迁移到 ht[1]，rehashidx++
4. 所有数据迁移完成后：
   - 释放 ht[0] 内存
   - 把 ht[1] 设为新 ht[0]
   - rehashidx 置为 -1，结束 rehash

##### 3. 为什么要用渐进式 rehash

- 不一次性全量迁移，**避免大量拷贝阻塞Redis主线程**
- 服务正常读写和数据迁移**并行执行**，高并发无卡顿

---

#### 四、dict 如何支撑各高层结构

1. **Hash**：field 当 key，value 当 value
2. **Set**：元素当 key，value 存 null，利用字典 key 天然去重
3. **ZSet**：member 当 key，score 当 value，实现 O(1) 根据成员查分数

---

#### 五、面试极简背诵版

dict 是 Redis 自研哈希字典，底层由 **dict、dictht、dictEntry** 三层组成；
采用**数组+单向链表、链地址法**解决哈希冲突；
扩容缩容使用**渐进式 rehash**，分步迁移不阻塞主线程；
支撑 Hash、Set、ZSet 复杂场景，是 Redis 核心键值映射结构。

### 5. skiplist 跳表

**作用**：**有序、高效范围查询**，替代红黑树。
**特点**：
- 多层索引，空间换时间
- 增删查 O(logn)
- 天然有序、支持前后遍历、范围分页
**支撑**：
- ZSet 超出阈值后：**skiplist 负责排序排行**



#### 作用

Redis 自研**有序跳表**，用来实现**高效排序、范围查询、排行榜**；
功能等价红黑树，但**实现更简单、范围遍历更友好**，所以 ZSet 不用红黑树，改用跳表。

#### 特点

- 多层索引结构，**空间换时间**
- 增、删、查时间复杂度 **O(log n)**
- 底层链表**天然有序**，支持正向、反向遍历
- 天然适合**区间范围查询、分页、排行榜**
- 实现比红黑树简单、无复杂旋转平衡逻辑

#### 支撑范围

- **ZSet 有序集合**：
  元素超过 `128` 个 或 member 长度超过 `64` 字节，
  由 ziplist 切换为 **skiplist + dict** 组合；
  **skiplist 专门负责排序、排行、范围查询**。

---

#### 一、跳表核心结构形象理解

普通单向链表：只能从头逐个遍历 O(n)
跳表：**多层链表建立索引**
```
第3层索引：1 ———————> 9
第2层索引：1 ——> 5 ——> 9
第1层索引：1 -> 3 -> 5 -> 7 -> 9
底层原始链表：全部元素有序排列
```
高层快速跳跃定位，底层精确遍历，实现二分查找效果。

---

#### 二、Redis 跳表 源码结构体

##### 1. 跳表整体 zskiplist

```c
typedef struct zskiplist {
    zskiplistNode *header;  // 头哨兵节点
    zskiplistNode *tail;    // 尾节点
    unsigned long length;  // 底层元素总个数
    int level;              // 当前最大层数
} zskiplist;
```

##### 2. 跳表节点 zskiplistNode

```c
typedef struct zskiplistNode {
    sds ele;                   // ZSet 的 member 元素
    double score;              // 排序分值
    struct zskiplistNode *backward; // 后退指针，支持反向遍历
    zskiplistLevel level[];    // 多层前进指针柔性数组
} zskiplistNode;
```

##### 3. 每层索引 zskiplistLevel

```c
typedef struct zskiplistLevel {
    struct zskiplistNode *forward; // 同层下一个节点
    unsigned long span;            // 当前层跨越节点数
} zskiplistLevel;
```

---

#### 三、核心工作原理

1. 所有节点**底层链表按 score 分值从小到大有序**
2. 插入节点时**随机生成层数**，层高概率递减
3. 查询从**最高层**开始：
   - 能往后跳就跳
   - 不能跳就下沉一层
   直到底层精准定位，O(logn)
4. 自带 **backward 后退指针**，支持从尾到头逆序遍历，做倒序排行榜

---

#### 四、ZSet 为什么用 跳表 不用 红黑树？（面试高频）

1. **跳表范围遍历更简单**
   底层就是有序链表，区间查询、分页直接遍历即可；
   红黑树范围遍历需要中序遍历，实现复杂。

2. **跳表实现简单**
   不用像红黑树做 **左旋、右旋、变色** 复杂平衡操作。

3. **内存开销可控**
   按概率随机层数，平均内存损耗小。

4. **支持反向遍历**
   自带后退指针，天然适合倒序榜单。

---

#### 五、ZSet 中 skiplist 与 dict 分工

```
skiplist → 负责：按score排序、范围查询、排行榜、正反遍历
dict     → 负责：member 查 score，O(1) 快速取值、去重
```
**两者配合，既有排序能力，又有随机查询性能**。

## 四、Redis 网络底层模型

**单线程 Reactor 模型**
- 多路复用：**epoll（Linux）**
- 事件驱动：文件事件 + 时间事件
- 核心工作线程**单线程**处理命令、数据结构、逻辑
- 持久化、刷缓冲区、集群心跳 放**后台子线程**

优点：无线程竞争、不用锁、极速、简单

### 核心概括

Redis 采用 **单线程 Reactor 事件驱动模型**
底层基于 **epoll 多路IO复用**，配合**文件事件 + 时间事件** 双事件机制；
**命令读写、协议解析、数据结构操作** 都在主线程串行执行；
持久化、异步任务、集群心跳等交给**后台子线程/子进程**。

### 1. 多路复用：epoll（Linux）

- Linux 下默认使用 **epoll**，Windows 是 select
- 同时监听**上万 socket 文件描述符**
- 阻塞等待，有事件就绪才唤醒，**空转不耗CPU**
- 高并发下内存、CPU 开销极低

#### 一、什么是 IO 多路复用

**IO 多路复用**：一个线程，通过 **epoll** 可以**同时监听多个 Socket 文件描述符(fd)**。
哪个 fd 有读/写/连接事件就绪，内核就通知应用，线程只处理就绪的，不用挨个轮询所有连接。

Redis 单线程能撑**十万并发连接**，核心就是靠 **epoll 多路复用**。

#### 二、为什么不用 阻塞IO / 非阻塞轮询

##### 1. 阻塞IO

一个线程只能管一个客户端，来一万连接就要一万线程，**线程爆炸、上下文切换爆炸、内存爆炸**。

##### 2. 非阻塞轮询 while(遍历所有fd)

不停循环遍历所有 socket，不管有没有事件都轮询，**空转狂耗CPU**。

##### 3. epoll 完美解决

- 不用多线程
- 不用空转轮询
- 内核帮你监听，**有事唤醒，无事休眠**

---

#### 三、epoll 三大核心函数

##### 1. epoll_create

创建一个 epoll 实例，返回监听句柄。

##### 2. epoll_ctl

把所有客户端 **socket fd** 注册到内核 epoll 树上，监听：
- 可读事件（客户端发命令过来）
- 可写事件（Redis 要返回响应）
- 连接事件（新客户端接入）

##### 3. epoll_wait

**阻塞等待**，直到有事件就绪，内核返回**就绪的 fd 列表**，
Redis 只遍历这一小部分就绪 fd 处理即可。

---

#### 四、epoll 两种工作模式

##### 1. LT 水平触发（默认）

只要 fd 有数据没读完，**每次 epoll_wait 都会一直通知**。

##### 2. ET 边缘触发

只有**状态发生变化瞬间**才通知一次，没读完也不再重复提醒；
要求必须**一次性把数据读完**，搭配非阻塞Socket。

**Redis 使用 ET 边缘触发 + 非阻塞 IO**
性能更高、适合高并发海量连接。

---

#### 五、epoll 相比 select/poll 优势

1. **文件描述符无上限**
select 有最大数量限制；epoll 只受系统内存限制，支持十几万连接。

2. **时间复杂度 O(1) 就绪通知**
select/poll 每次都要遍历全部 fd O(n)；
epoll 内核只返回**就绪的fd**，不用遍历全部。

3. **用户态内核态内存共享**
epoll 通过内存映射，减少数据拷贝，效率更高。

4. **空闲不耗CPU**
没有IO事件时，epoll_wait 阻塞休眠，**CPU 完全放空**。

---

#### 六、Redis 中 epoll 完整工作流程

1. Redis 启动，创建 **listen 监听套接字**，注册到 epoll。
2. 每来一个新客户端连接，生成新 socket fd，注册进 epoll。
3. 主线程循环调用 `epoll_wait` 阻塞等待。
4. 客户端发命令 → 内核感知可读事件 → 唤醒 Redis。
5. 主线程拿到就绪 fd → 解析命令、操作数据结构、执行逻辑。
6. 构造响应数据，注册可写事件，发给客户端。
7. 处理完继续回到 `epoll_wait` 阻塞，等待下一批事件。

**全程单线程、无锁、无轮询、高并发低CPU**

### 2. 事件驱动：两大类事件

#### ① 文件事件（File Event）

由 epoll 监听产生：
- 客户端**连接建立**
- 客户端**读请求**（接收命令）
- 服务端**写响应**（返回数据）
全程事件驱动，来了才处理，异步非阻塞。

#### ② 时间事件（Time Event）

定时轮询任务：
- 过期 key 定期删除
- 集群心跳、超时检测
- 定时持久化辅助任务

### 3. 核心工作线程 单线程串行处理

**主线程只做一件事**：循环从 epoll 拿事件，串行执行：
- 接收客户端连接
- 解析 Redis 命令
- 操作 5 大底层数据结构
- 执行增删改查逻辑
- 返回响应

**所有命令串行排队执行，不存在并发线程安全问题**，**不用加锁**。

### 4. 后台子线程/子进程分离耗时任务

耗时、阻塞、重IO操作**不占用主线程**，下放后台：
- RDB/AOF 持久化、AOF 重写
- 缓冲区刷盘、大文件IO
- 集群心跳、异步关闭连接
- 内存异步释放

### 单线程 Reactor 优点（面试必背）

1. **无线程竞争**，不用加各种锁，实现简单
2. **单线程串行无上下文切换**，CPU 利用率高、极速
3. 避免多线程死锁、竞态条件、调试复杂问题
4. IO 多路复用支撑**高并发海量连接**
5. 耗时任务剥离到子线程/子进程，不阻塞命令处理

### 面试一句话总结

Redis 网络底层是 **epoll 多路复用 + 单线程Reactor事件驱动**；
分**文件事件**处理网络IO、**时间事件**处理定时任务；
命令解析与数据操作**主线程单线程串行执行**，无需锁；
持久化、刷盘、集群心跳等耗时任务下放**后台子线程**，兼顾高性能与高并发。

- 自研内存管理：封装 **zmalloc / zfree**

### 1. 自研内存管理 zmalloc/zfree

Redis 不直接裸用 glibc 的 `malloc/free`，**自己封装一层内存分配器**：
- 统一封装 `zmalloc` / `zfree` / `zrealloc`
- 自带**内存统计**：记录已使用内存大小
- 做内存对齐、内存溢出检测
- 适配不同系统、兼容 tcmalloc/jemalloc 第三方分配器

**作用**：
精细化管控内存、统计占用、方便做内存上限控制、排查内存泄漏。

### 2. 内存淘汰策略 maxmemory

配置 `maxmemory` 设定 Redis 最大可用内存，**超出后触发淘汰策略**。

#### 8 种淘汰策略（必背）

1. **noeviction**：默认，不淘汰，写操作直接报错
2. **allkeys-lru**：所有key，淘汰**最近最少使用**
3. **volatile-lru**：只从**带过期**key中，淘汰LRU
4. **allkeys-lfu**：所有key，淘汰**使用频率最低**
5. **volatile-lfu**：过期key中淘汰LFU
6. **allkeys-random**：所有key随机删
7. **volatile-random**：过期key随机删
8. **volatile-ttl**：删掉**快要过期**的key

**企业常用**：`allkeys-lru`



### Redis 内存淘汰策略 ****

我直接给你讲 **Redis 到底在什么时候、用什么逻辑、怎么执行内存淘汰**，这是面试官最爱问的底层细节。

---

#### 一、内存淘汰 **触发时机**（代码执行点）

Redis **不是等内存满了才淘汰**，而是：
**每次执行写命令前，先检查内存是否超限 → 超限就执行淘汰**

伪代码逻辑：
```
执行命令前：
1. 检查是否设置了 maxmemory
2. 计算当前使用内存是否 > maxmemory
3. 如果 **超过** → 执行 **内存淘汰逻辑**
4. 淘汰后腾出空间 → 再执行写命令
5. 淘汰后还是不够 → 返回 OOM 错误
```

**核心入口函数**：
`freeMemoryIfNeeded()`  →  判断是否需要释放内存

---

#### 二、8 种策略 **代码实现分类**

Redis 内部把策略分成 **3 大类**：

1. **不淘汰**
   `noeviction`

2. **随机淘汰**
   `allkeys-random`
   `volatile-random`

3. **LRU / LFU 淘汰**
   `allkeys-lru` / `volatile-lru`
   `allkeys-lfu` / `volatile-lfu`

4. **按 TTL 淘汰**
   `volatile-ttl`

---

#### 三、**LRU 底层实现代码逻辑**（最常考）

##### 1. Redis 不使用标准 LRU（太耗内存）

标准 LRU 需要双向链表，Redis 用 **简化版 LRU**，**不额外占空间**。

##### 2. 实现原理

- 每个 key 的对象头里，记录一个字段：
  `lru` ：**最后一次访问时间戳**
- 不占额外内存，直接复用对象头空间

##### 3. 淘汰流程（代码逻辑）

```
1. 随机采样 5 个 key（可配置 maxmemory-samples 5）
2. 读取每个 key 的 lru 时间戳
3. 找到 **最久没访问** 的那个 key
4. 删除它
5. 再检查内存是否足够
6. 不够 → 继续采样删除 → 直到内存达标
```

**特点**：
- 近似 LRU
- 性能极高
- 不需要额外数据结构

---

#### 四、**LFU 底层实现代码逻辑**（4.0+）

##### 1. 原理

LFU = **Least Frequently Used** 最少使用频率
比 LRU 更精准：**淘汰访问次数最少的**，不是最久没访问的。

##### 2. 实现（复用对象头，不占额外内存）

对象头 24 bit 被拆成：
```
16 bit : lru 时间戳
8 bit  : 访问计数器（0~255）
```

##### 3. 计数器增长逻辑（代码核心）

- 访问 key → 计数器**不是一直+1**
- 用 **概率增长**：次数越多，增长越慢
- 防止冷数据突然变热

##### 4. 淘汰流程

```
1. 随机采样 5 个 key
2. 比较 8 bit 访问计数值
3. 删除 **计数最小** 的 key
4. 循环直到内存足够
```

---

#### 五、**volatile-ttl 实现**

```
1. 随机采样一批带过期时间的 key
2. 比较 TTL 值
3. 删除 **最快要过期** 的 key
```

---

#### 六、**随机淘汰 random**

最简单：
```
随机选 key → 直接删
```

---

#### 七、整体淘汰流程 **伪代码完整版**

```
void freeMemoryIfNeeded() {

    while (used_memory > maxmemory) {

        根据策略选择待删 key：

        if (策略 == LRU) {
            随机采5个key → 删 lru 最老的
        }
        else if (策略 == LFU) {
            随机采5个key → 删 计数最小的
        }
        else if (策略 == TTL) {
            随机采5个带过期key → 删 最快过期的
        }
        else if (策略 == RANDOM) {
            随机删一个
        }
        else if (策略 == NOEVICTION) {
            返回 OOM 错误
        }

        删完继续检查内存...
    }
}
```

### 4. 过期 Key 清理机制（重点面试）

采用**惰性删除 + 定期删除 组合策略**

#### ① 惰性删除

**什么时候访问过期key，什么时候立马删**
- 优点：**不耗CPU**，只在访问时清理
- 缺点：大量**冷过期key一直占内存**，不访问永远不删

#### ② 定期删除

Redis 定时轮询：
- 每隔一段时间**随机抽取一批带过期的key**
- 检查并删除已过期的
- 限制轮询耗时，不阻塞主线程

**作用**：
解决惰性删除遗留的**冷过期key内存占用**问题，平衡 CPU 和 内存。

#### 总结一句话

**惰性删除**防CPU浪费，**定期删除**防内存泄漏，两者配合平衡性能与内存。

## 六、Redis 持久化底层

1. **RDB**：二进制快照，全量备份
2. **AOF**：日志追加，逐条写命令，支持 rewrite 重写

### 核心两种持久化

1. **RDB**：二进制快照、全量备份
2. **AOF**：日志追加、记录写命令、支持 **AOF Rewrite 重写**

---

### 一、RDB 快照持久化

#### 原理

把**当前内存所有数据**，一次性生成**二进制压缩快照文件**（`.rdb`），保存到磁盘；
属于**全量备份**。

#### 触发方式

- **手动**：`save`、`bgsave`
- **自动**：配置 `save 秒数 改动次数` 自动触发

#### save 与 bgsave 区别

- **save**：**主线程阻塞**，一次性全量落盘，大数据量会卡
- **bgsave**：**fork 子进程**，子进程做快照，主线程继续处理命令（生产默认用）

#### 优点

- 二进制压缩文件，**体积小、恢复速度快**
- 适合**冷备份、灾难恢复**
- 对主线程影响小（bgsave）

#### 缺点

- 不是实时持久化，**会丢失最后一次快照后的数据**
- 大数据量 `fork` 子进程会占用内存、产生拷贝开销

#### 底层机制：写时复制 COW

fork 子进程后，采用 **Copy On Write 写时复制**：
- 子进程只读内存做快照
- 主线程修改数据时才复制页面，不重复浪费整块内存

---

### 二、AOF 日志持久化

#### 原理

把客户端**每条写命令**，以**协议日志形式**追加到 `.aof` 文件末尾；
重启时**从头回放命令**，恢复内存数据。

#### 三种刷盘策略（appendfsync）

1. **always**：每条命令都立即刷盘，**最安全、性能最差**
2. **everysec**：每秒刷一次，**折中（生产常用）**，最多丢1秒数据
3. **no**：交给操作系统自动刷盘，性能最好、不安全

#### AOF 重写 AOF Rewrite

随着命令不断追加，AOF 文件越来越大，产生大量冗余命令；
**AOF Rewrite**：
Redis fork 子进程，**直接遍历当前内存数据**，生成精简版最小化命令序列，重写新 AOF 文件，替换旧文件。

**作用**：
- 合并冗余命令、减少文件体积
- 加快重启数据恢复速度

#### 优点

- **数据更安全**，最多丢1秒数据
- 日志文本格式，可人工编辑

#### 缺点

- AOF 文件比 RDB 大
- 重启回放命令，**恢复速度比 RDB 慢**

---

### 三、RDB + AOF 混合模式

Redis 4.0+ 支持**混合持久化**：
- AOF Rewrite 时，**开头先写入 RDB 二进制快照**
- 后面追加增量 AOF 命令日志
兼顾：**RDB 恢复快 + AOF 数据安全**，企业推荐开启。

---

### 四、面试一句话总结

RDB 是**二进制全量快照**，体积小恢复快，会丢数据，靠 fork 子进程+COW 写时复制；
AOF 是**日志追加写命令**，数据更安全，支持每秒刷盘，通过 **AOF Rewrite** 压缩冗余日志；
生产一般**同时开启 RDB+AOF 混合持久化**。
