# LangSmith

LangSmith 是 LangChain 推出的**LLM应用全生命周期可观测平台**，定位为“AI 应用的 APM+调试器+评估系统”，核心解决大模型应用“黑盒不可控”问题，让 Chain/Agent/RAG 可追踪、可调试、可评估、可优化。

---



## 一、核心定位与价值

- **框架无关**：支持 LangChain、LangGraph、原生 OpenAI/Anthropic 调用，也可通过 OpenTelemetry 接入非 Python 服务。
- **全链路追踪**：记录一次请求从输入到输出的**完整调用链（Trace）**，含模型调用、工具调用、Prompt 渲染、中间结果、耗时、Token 消耗、成本。
- **生产级监控**：实时仪表盘（延迟、错误率、成本、Token）+ 告警 + 智能聚类分析。
- **评估闭环**：数据集管理、自动评估（LLM-as-judge/自定义指标）、人工反馈、版本对比。
- **协作与迭代**：Prompt 版本管理、Playground 可视化调试、团队协作评审。



---

## 二、核心概念（必懂）

- **Trace（调用链）**：一次用户请求的**完整执行树**，由多个 Run 组成。
- **Run（执行单元）**：Trace 的最小节点，如 LLM 调用、工具调用、Chain 调用、检索（Retriever）调用，记录输入/输出/参数/耗时/状态。
- **Project（项目）**：隔离不同应用/环境的 Trace，便于管理与权限控制。
- **Dataset（数据集）**：用于评估的样本集合，可由生产 Trace 导出或手动上传。
- **Experiment（实验）**：在数据集上运行不同 Prompt/模型/配置，对比效果。

### 1. Trace 调用链

**一次完整用户请求的全流程日志**
用户发一个问题，从进入程序 → 走提示词 → 调用大模型 → 调用工具 → 拼接结果 → 最终回复，**整条完整流程**就叫一条 Trace。
可视化呈现**树形结构**，能看清每一步怎么走的。

### 2. Run 执行单元

Trace 里面**最小独立执行节点**，Trace 由无数 Run 拼接而成。
常见 Run 类型：

- LLM Run：大模型调用
- Chain Run：链路执行
- Tool Run：工具调用
- Retriever Run：知识库检索
  每个 Run 单独记录：入参、出参、耗时、Token、报错信息。

### 3. Project 项目

用来**分类隔离**线上、测试、本地不同业务AI应用。
例：智能客服项目、RAG知识库项目、办公Agent项目分开建项目，日志互不混杂，方便团队管理、权限划分、环境区分。

### 4. Dataset 评估数据集

专门用来**测评AI效果**的测试题库。
来源两种：

1. 线上真实用户对话 Trace 直接导出
2. 手动批量编写标准问答样本
   作用：统一标准，批量测试模型好坏。

### 5. Experiment 实验

在同一个 Dataset 数据集上，**对照测试不同方案**。
比如：

- 测试Prompt A vs Prompt B
- 测试GPT-3.5 vs 国产模型
- 测试不同检索策略
  自动生成对比报表，直观看出哪个版本效果更好，快速迭代优化。

---

### 极简关系总结

**一次用户请求 = 1条 Trace**
**一条 Trace = 多个 Run 节点**
**所有 Trace 按 Project 项目归类**
**拿 Dataset 题库，跑 Experiment 实验做效果评测**

## 三、七大核心功能

### 1. 可观测性（Tracing+监控+Insights）

- **自动追踪**：一行代码开启，自动捕获 LLM/Chain/Agent/Tool 的全链路。
- **细节可视化**：每步输入/输出、Prompt 原文、工具参数、错误栈、延迟、Token、成本。
- **监控大盘**：P50/P99 延迟、错误率、成本趋势、Top 慢调用、Top 错误。
- **智能分析**：Trace 聚类、失败模式识别、根因定位、优化建议。
- **断点调试**：支持在任意 Run 前后打断点，单步执行，适合复杂 Agent 调试。

#### 1. 自动追踪

无需手动写日志，仅配置环境变量即可全局开启
自动抓取 **大模型、链式调用、智能代理、自定义工具** 所有执行流程，全程无侵入接入。

#### 2. 细节可视化

全流程信息一目了然，调试核心依据：

- 每一步**输入内容、最终输出结果**
- 真实渲染后的**完整Prompt**
- 工具调用传参、返回数据
- 程序报错堆栈、调用耗时
- 消耗输入/输出Token、接口调用成本

#### 3. 监控大盘

企业线上运维核心能力

- 统计请求**中位延迟P50、高耗时延迟P99**
- 全局错误请求占比、接口失败率
- 每日/每月模型调用成本走势
- 筛选最慢请求、高频报错请求，快速定位问题

#### 4. 智能分析

平台自动聚合数据分析

- 把相似异常调用归类聚合
- 总结高频失败原因、归纳问题模式
- 自动给出性能优化、Prompt优化、成本优化建议

#### 5. 断点调试

复杂Agent专属调试功能
可在任意**最小执行单元Run**前后暂停流程，单步运行排查
精准定位：工具调用异常、思考逻辑错误、多轮循环卡死、RAG检索失效等疑难问题。

---

#### 核心作用总结

把**黑盒大模型应用**彻底透明化，开发调试排错、线上运维监控、性能成本优化一站式搞定。

### 2. 评估系统（Evaluation）

- **离线评估**：在 Dataset 上批量运行，自动打分（相关性、正确性、有害性等）。
- **在线评估**：生产环境实时对 Trace 打分，监控质量变化。
- **LLM-as-Judge**：用 GPT-4 等作为评委，自动评估输出质量。
- **人工反馈**：支持标注团队直接在平台打分、评论、修正输出。
- **版本对比**：同一 Dataset 对比不同 Prompt/模型/代码版本的效果。

#### 1. 离线评估

提前准备好**测试数据集Dataset**，批量跑完整问答流程，自动批量打分评判。
评判维度：内容相关性、答案正确性、语句合规性、有无有害内容等，适合开发阶段批量验收效果。

#### 2. 在线评估

线上生产环境实时监测用户真实对话，对每条请求链路实时评分，持续监控AI回答质量波动，及时发现线上效果下滑问题。

#### 3. LLM-as-Judge 大模型裁判

借助更强的大模型（如GPT-4）充当人工评委，自动对AI输出内容进行打分、判错、挑漏洞，无需人工逐条审核，实现全自动质量校验。

#### 4. 人工反馈

平台支持业务标注人员直接在追踪记录里手动评分、写评语、修正错误回答，收集真实业务标准答案，用来优化模型与提示词。

#### 5. 版本对比

使用同一套测试数据集，横向对比**不同提示词、不同大模型、不同业务代码版本**的输出效果，直观选出最优方案，快速迭代优化。

**核心作用**：建立AI应用量化评价标准，告别主观感受，用数据衡量回答好坏，稳定上线质量。

### 3. Prompt 工程与协作

- **Playground**：可视化编辑、调试 Prompt，实时预览输出，支持多模型对比。
- **版本管理**：Prompt 历史版本回溯、分支管理、变更记录。
- **团队协作**：Prompt Canvas 支持多人在线编辑、评论、建议，直接合并优化。

#### 1. Playground 在线调试工作台

可视化编写提示词，即时运行查看模型返回结果；可一键切换多款大模型，**同问题横向对比不同模型回答效果**，快速择优选型。

#### 2. Prompt 版本管理

自动留存提示词每一次修改记录，支持历史版本回滚、分支区分、查看修改明细，避免优质提示词丢失，方便迭代回退。

#### 3. 团队协作开发

依托Prompt画布实现多人协同编辑，成员可在线批注、提修改意见、评审文案，统一团队Prompt规范，高效合力优化提示词，告别本地文件传来传去。

**核心作用**：规范化管理提示词，降低调试成本，实现团队协同共建优质提示词体系。

### 4. Agent 部署（Agent Server）

- **一键部署**：将 Agent/Chain 打包为可扩展的 HTTP 服务，支持负载均衡、自动扩缩容。
- **托管/自托管**：支持 LangSmith 云托管、私有化部署或混合架构，满足合规（HIPAA/SOC2/GDPR）。

#### 1. 一键部署

直接把写好的**智能体Agent、业务链路Chain**快速打包成标准HTTP接口服务
自带负载均衡、弹性扩缩容能力，并发高时自动扩容，空闲自动缩容，适配线上业务流量。

#### 2. 双部署模式

- **云托管**：LangSmith云端直接托管运行，不用自己维护服务器，开箱即用
- **私有化部署**：部署在企业内网/自有服务器，数据不外流
  满足行业数据安全、隐私合规、审计等各类合规要求。

**核心作用**：快速把本地调试完成的AI应用，低成本上线投产，正式对外提供服务。

### 5. Studio（可视化 IDE）

- **无代码/低代码**：拖拽式设计 Agent 流程，连接 LLM、工具、检索、条件分支。
- **端到端测试**：在可视化界面直接运行、调试、迭代整个应用。

#### 1. 无代码/低代码搭建

可视化拖拽编排智能体流程，直接拖拽**大模型、工具调用、知识库检索、逻辑分支**等组件，连线组合就能搭建完整Agent业务流程，大幅减少手写代码量，新手也能快速搭建AI应用。

#### 2. 端到端全流程测试

在可视化界面内直接发起请求，一站式完成运行、排错、流程调整，全程可视化查看每一步执行走向，快速迭代优化整体业务逻辑，不用反复本地调试。

**核心作用**：降低AI应用开发门槛，可视化梳理复杂业务流程，快速原型搭建与流程验证。

## 四、快速上手（5 步）

### 1. 安装与配置

```bash
pip install -U langsmith langchain-openai
```

```python
import os
# 注册：https://smith.langchain.com/
os.environ["LANGSMITH_API_KEY"] = "ls_xxx"
os.environ["LANGSMITH_TRACING"] = "true"  # 开启追踪
os.environ["LANGSMITH_PROJECT"] = "my-agent-app"  # 项目名
os.environ["OPENAI_API_KEY"] = "sk-xxx"
```

### 2. 追踪 LLM 调用

```python
from langchain_openai import ChatOpenAI
from langsmith import traceable

llm = ChatOpenAI(model="gpt-3.5-turbo")

@traceable  # 自动追踪此函数
def ask(question: str):
    return llm.invoke(question).content

ask("什么是 LangSmith？")
# 访问 https://smith.langchain.com/ 查看 Trace
```

### 3. 追踪 Agent 全链路

```python
from langchain.agents import AgentExecutor, create_openai_tools_agent
from langchain_core.tools import tool
from langchain_core.prompts import ChatPromptTemplate

@tool
def search(keyword: str) -> str:
    """联网搜索"""
    return f"{keyword} 的搜索结果"

tools = [search]
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是助手，可使用工具"),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}")
])
agent = create_openai_tools_agent(llm, tools, prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools)

agent_executor.invoke({"input": "LangSmith 有什么功能？"})
# LangSmith 会记录：思考→工具调用→结果→回答 全链路
```

### 1. 什么是 LangSmith？定位与核心价值

**答案**
LangSmith 是**LangChain 生态官方**的大模型应用全链路可观测、调试、评估、运维平台。
定位：LLM/Agent/RAG 应用专属监控运维平台。
核心价值：
1. 全链路追踪大模型调用流程
2. 可视化调试 Prompt、工具调用、检索流程
3. 离线+在线智能评估应用效果
4. 统计 Token 消耗、耗时、调用成本
5. 收集人工反馈，迭代优化智能应用

### 2. LangSmith 和 LangChain、LangGraph 的关系

**答案**
1. **LangChain**：大模型应用开发框架，负责写业务逻辑、编排流程
2. **LangGraph**：基于 LangChain 的**状态图智能体**框架，实现复杂多轮Agent流程
3. **LangSmith**：**上层运维观测平台**，不写业务代码，专门监控、追踪、调试、评估 LangChain / LangGraph 写出来的应用
三者关系：**开发用LangChain/LangGraph，运维观测用LangSmith**。

### 3. 为什么LLM应用需要专属可观测平台？传统日志/APM不够用？

**答案**
传统日志、SkyWalking、Pinpoint 只适合微服务接口调用，**不适合大模型场景**：
1. LLM 调用存在**Prompt、上下文、召回文档、模型输出**海量文本数据，普通日志无法结构化存储
2. 无法统计**Token 消耗、模型耗时、对话轮次、幻觉问题**
3. Agent 多工具嵌套调用链路复杂，普通链路追踪分不清大模型内部执行步骤
4. 缺少大模型专属评估能力，无法自动判断回答优劣、检索准确率
5. 无法一键回放对话流程、调试Prompt问题

### 4. 解释 Trace、Run、Project、Session 四大核心概念

**答案**
1. **Project 项目**：最大维度，用来区分业务模块、环境（开发/测试/生产）
2. **Session 会话**：对应**用户一次完整对话**，多轮对话归属同一个会话
3. **Trace 追踪链路**：一次完整请求全流程，包含所有执行节点
4. **Run 执行单元**：Trace 里最小执行节点，如LLM调用、向量检索、工具调用、Prompt组装都算一个Run

层级关系：`Project > Session > Trace > Run`

### 5. LangSmith Trace 和 OpenTelemetry Span 异同

**答案**
#### 相同点

都是分布式链路追踪，都有父子层级、耗时、调用链、标签属性。
#### 不同点

1. **用途不同**
OTel 通用微服务链路；LangSmith 专为**LLM/RAG/Agent**定制
2. **存储内容不同**
LangSmith 原生存储**完整Prompt、模型输入输出、召回文本、Token数**
3. **能力不同**
LangSmith 自带评估、Prompt调试、对话回放；OTel无大模型专属能力
4. **数据结构**

### 1. Evaluation作用 + 离线/在线评估区别

**作用**
解决大模型应用**效果不可量化、优劣无标准、迭代无依据**问题，实现AI应用标准化评测、版本灰度对比、质量把关。
- 离线评估：上线前批量评测数据集，提前发现缺陷
- 在线评估：线上实时监控用户请求质量，持续巡检

**离线评估**
提前导入标准测试数据集，批量执行评测，适合**Prompt调优、模型选型、版本回归测试**，不影响线上流量。

**在线评估**
线上真实流量实时打分，适合**生产环境质量巡检、风险拦截、数据回流**。

### 2. LLM-as-judge 含义 + Java自定义评估器

**LLM-as-judge**
用**大模型充当裁判**，按照指定评判规则，自动打分、判定回答合规性、准确性、逻辑性，替代人工评测。

**Java自定义评估器**
1. 基于LangChain4j实现`Evaluator`接口
2. 编写评判Prompt，定义打分维度与规则
3. 绑定LangSmith评估链路，自动接入评测流程
4. 自定义评分标准、错误标签、不合格拦截逻辑
5. 支持接入自有裁判模型，私有化离线评测

### 3. RAG召回质量评估 + 常用指标

**评估流程**
1. 输入标准问题+标准答案
2. 自动评测向量召回相关性、文档匹配度
3. 评测最终答案是否依赖检索内容、有无脱离原文

**常用核心指标**
- 召回准确率：召回文档与问题相关占比
- 上下文精准率：有效参考文档占比
- 答案忠实度：回答是否忠于检索内容
- 问答正确率：最终回答是否符合标准答案
- 幻觉率：无依据编造内容占比

### 5. 人工反馈接入LangSmith闭环

1. 前端/业务系统收集用户点赞、差评、纠错反馈
2. Java后端将**人工评分、错误原因、修正答案**上报LangSmith

1. 线上请求完成后自动触发LLM裁判评估

5. 如何用 LangSmith 分析 **token 成本分布**，优化 LLM 调用费用？



### 1. LangChain4j + LangSmith 实现RAG Trace链路+Run说明

#### 链路流程图

用户提问 → 向量化 → 向量检索 → 文档处理 → 提示词拼接 → LLM生成 → 结果返回
#### 对应Run节点

1. **UserRun**：接收用户请求，记录原始提问
2. **EmbeddingRun**：调用嵌入模型，将问题转为向量
3. **RetrievalRun**：执行向量库相似度查询，召回相关文档片段
4. **DocumentRun**：过滤、去重、精简召回文档内容
5. **PromptRun**：拼接系统提示词+历史对话+检索上下文
6. **LLMRun**：调用大模型执行推理，统计输入输出token
7. **FinalRun**：封装结果响应前端

**作用**：全流程可视化，每一步耗时、入参、出参、异常均可查看，一键排查RAG全流程问题。

### 2. Agent工具调用失败快速定位根因

1. 打开异常Trace，展开Agent树形调用链路
2. **参数错误**：查看工具调用入参Run，核对字段格式、传参内容是否不符合接口规范
3. **调用超时**：查看工具节点耗时，判定是第三方接口慢、网络延迟、还是本地阻塞
4. **逻辑错误**：查看Agent思考日志，确认是大模型选错工具，还是决策逻辑出错
5. **权限/业务异常**：查看工具原始返回报文，定位权限不足、业务报错、资源不存在等问题
6. 定位后修改Agent提示词、参数校验、超时时间、重试策略即可修复

### 3. 多轮对话关联同一会话所有Trace

1. 业务端生成**全局唯一SessionId**
2. 每一轮对话请求统一绑定同一个SessionId上报LangSmith
3. 平台自动以会话为维度聚合所有历史Trace
4. 可按会话ID查询完整对话时序、上下文拼接记录、历史记忆加载流程
5. 快速排查记忆丢失、上下文过长溢出、多轮对话逻辑错乱等线上问题

### 4. Java高并发下上报瓶颈与优化方案

**结论**：**同步上报会成为瓶颈**，阻塞业务线程，拉高接口RT；异步模式无明显瓶颈。
#### 优化方案

1. 全局开启**异步非阻塞上报**，脱离业务主线程执行
2. 启用本地队列缓冲，上报失败本地缓存，定时重试
3. 生产环境配置**链路采样**，只上报部分流量，降低上报压力
4. 精简上报字段，屏蔽无用文本、冗余日志，减少报文体积
5. 批量聚合上报，减少HTTP请求次数
6. 内网私有化部署LangSmith，消除公网网络延迟
7. 低优先级业务关闭追踪，只保留核心AI链路

### 5. Token成本分析与费用优化

1. 通过LangSmith统计**输入Token、输出Token、总Token**，按接口、模型、用户维度拆分成本分布
2. 定位高耗场景：超长历史对话、冗余检索文档、冗长Prompt模板
3. 优化手段
- 截断多轮对话历史，限制上下文最大轮数
- 控制向量召回数量，只保留高相似度文档
- 精简Prompt模板，剔除无效冗余语句
- 简单问题使用轻量低成本模型，复杂业务使用大模型
- 限制模型最大生成长度，避免无意义长文本输出

4. 搭建费用监控看板，提前预警超额消耗，控制AI调用总成本

## 六、对比与选型类

1. LangSmith vs Langfuse vs 自研日志平台，各自优缺点？
2. LangSmith 相比传统 APM（SkyWalking/Pinpoint）在 LLM 场景优势？
3. 什么情况下**不建议用 LangSmith**？（数据合规/内网隔离/成本敏感）

### 1. LangSmith vs Langfuse vs 自研日志平台 优缺点

#### LangSmith

**优点**
- 原生适配LangChain/LangGraph/LangChain4j，接入最简单
- 追踪、调试、评估、数据集一站式齐全
- 可视化链路强，Agent执行树展示最完善
- 社区成熟，文档与实战案例多

**缺点**
- 公有云数据外发，合规敏感场景受限
- 高阶功能收费，大规模使用成本偏高
- 自定义扩展自由度一般

#### Langfuse

**优点**
- 开源免费，支持自托管，数据完全可控
- 协议宽松，私有化部署友好
- 多框架兼容，不限于LangChain生态
- 轻量灵活，轻量化项目性价比高

**缺点**
- 生态联动弱，和LangGraph适配不如前者顺滑
- 内置评估能力较弱，需自行二次开发
- 企业级运维配套少，问题排查资料偏少

#### 自研日志平台

**优点**
- 完全自主可控，贴合业务定制
- 零第三方依赖，数据不出内网，合规性最强
- 按需裁剪功能，无多余冗余开销

**缺点**
- 开发维护成本极高，需自建埋点、解析、看板、统计
- 缺少LLM专属字段与评测能力，工作量大
- 迭代慢，跟不上大模型框架新版本特性
- 异常分析、Token统计、对话回放都要从零实现

### 2. LangSmith 对比传统APM(SkyWalking/Pinpoint) LLM场景优势

1. **数据维度不同**
传统APM只记录接口、耗时、异常、堆栈；LangSmith原生存储Prompt、问答内容、召回文档、Token用量、模型参数。
2. **链路语义不同**
APM只能看服务调用链路；LangSmith可识别**检索、思考、工具调用、模型生成**等AI专属流程。
3. **专属能力差异**
APM无幻觉检测、答案评估、Prompt调试、对话回放、成本统计能力。
4. **业务视角不同**
APM偏向运维监控；LangSmith偏向**AI应用效果调优、业务迭代优化**。
5. **字段原生支持**
内置embedding、向量检索、会话记忆、多轮对话等大模型专属字段，无需手动埋点扩展。

### 3. 不建议使用LangSmith的场景

1. **强数据合规场景**
涉密政务、金融、政企内网项目，禁止业务对话、Prompt数据外传，不能对接公有云SaaS。
2. **完全离线内网环境**
无公网、无法连通外网接口，即便自托管运维成本过高。
3. **极致成本敏感小型项目**
业务量小、预算有限，不愿承担平台服务费与流量上报成本。
4. **非LangChain技术栈重度项目**
技术栈独立、不用LangChain4j，接入繁琐且发挥不出最大价值。
5. **极简轻量AI应用**
仅简单调用LLM接口，无RAG、无Agent、无复杂流程，普通日志即可满足需求。
6. **超高并发极致低延迟业务**
哪怕异步上报仍有额外开销，追求零侵入零额外耗时场景优先自研轻量埋点。

## 七、工程化与安全

1. LangSmith API Key 如何安全管理？Java 中避免硬编码的方案？
2. 企业数据不上云，LangSmith 自托管如何部署？（K8s/Helm）
3. 如何控制 LangSmith 上报的数据范围，满足等保/数据合规？

### 1. LangSmith API Key 安全管理 + Java 避免硬编码方案

#### 安全管理原则

禁止明文存储、禁止代码提交、分级权限管控、定期轮换密钥。

#### Java 避硬编码方案

1. **环境变量注入**
线上服务器、容器直接配置环境变量，程序读取`System.getenv()`获取密钥。
2. **配置中心统一托管**
使用Nacos、Apollo、Spring Cloud Config集中存放密钥，动态拉取。
3. **系统配置文件外置**
将`application-env.yml`放在项目外，不打入Jar包，运维单独维护。
4. **密钥加密存储**
配置文件内使用AES加密，程序启动解密使用。
5. **云原生密钥管理**
K8s Secret、阿里云密钥管理服务托管API_KEY。
6. **权限隔离**
开发、测试、生产使用不同API密钥，权限互不通用。

### 2. 企业数据不上云，LangSmith 自托管部署方式

1. **Docker 单机部署**
适合测试、小型内网环境，一键拉起服务，快速离线使用。
2. **K8s 容器化部署**
企业生产主流方案，高可用、弹性扩缩容、稳定运维。
3. **Helm 一键部署**
官方提供Helm Chart，直接拉取Chart包内网部署，简化编排。
4. **内网镜像私有化**
拉取官方镜像推送到企业私有镜像仓库，全程无外网访问。
5. **数据库独立部署**
外置MySQL/PostgreSQL存储链路数据，数据全部留存企业内网。
**核心优势**：所有对话、Prompt、检索内容、Trace数据**全程不出企业内网**，满足涉密要求。

### 3. 控制上报数据范围，满足等保与数据合规

1. **字段裁剪上报**
关闭完整Prompt、用户隐私对话全量上报，仅上报摘要、耗时、Token数量。
2. **敏感数据前置脱敏**
Java层先对手机号、身份证、账号、隐私文本脱敏后，再执行LLM调用与上报。
3. **黑白名单过滤**
自定义拦截器，匹配敏感词汇直接丢弃不上报LangSmith。
4. **链路采样上报**
生产环境开启采样，只上报部分请求，减少敏感数据暴露面。
5. **关闭流式内容上报**
仅汇总最终结果上报，屏蔽中间流式分段内容。
6. **内网端点隔离**
修改`LANGCHAIN_ENDPOINT`指向内网私有部署地址，禁止访问公网地址。
7. **日志留存管控**
设置Trace自动过期清理策略，定期清理历史对话数据，符合数据留存规范。

## 八、常见面试追问（连环炮）

- 你在项目中怎么用 LangSmith 解决实际问题？举一个具体案例。
- 如果 Trace 量太大、费用太高，你会怎么裁剪？
- 如何用 LangSmith 做**回归测试**，防止 Prompt 改坏？

1. **项目中实际用LangSmith解决过什么问题？**
   答：通过链路追踪定位RAG召回不精准问题，拆分各节点耗时优化接口速度；利用评估功能批量对比Prompt效果，降低大模型幻觉率；统一统计Token消耗，大幅缩减AI调用成本。

2. Trace数据量太大、费用过高如何处理？
   答：开启异步上报、链路采样、精简上报字段、限制会话上下文长度、低并发业务降级关闭追踪。

3. 如何用LangSmith做AI项目回归测试？
   答：搭建标准测试数据集，迭代前后批量跑评测，对比正确率、耗时、Token开销，防止Prompt修改、代码迭代导致效果退化。

4. 线上AI接口报错，优先看LangSmith哪块内容？
   答：优先看Run异常日志、LLM原始入参出参、工具调用入参、各节点耗时，快速区分是代码问题、检索问题还是大模型本身问题。

### 1. 项目中如何用LangSmith解决实际问题（实战案例）

**回答**
我在基于LangChain4j开发企业内部知识库RAG项目中深度使用LangSmith：
1. **问题定位**
线上出现大量回答不准、答非所问问题，通过LangSmith查看Trace里的Retrieval执行节点，发现是文档分片过大、召回相似度阈值设置不合理，导致无关文档混入上下文。
2. **性能优化**
拆分整条RAG链路各Run耗时，发现向量检索环节拖慢整体接口响应，针对性优化向量库索引与召回数量，整体接口耗时下降40%。
3. **效果调优**
使用LLM-as-judge自动评估，批量测试多版Prompt模板，筛选出抑制幻觉、贴合知识库内容的最优提示词，有效降低模型编造内容概率。
4. **成本管控**
统计各业务模块Token消耗，对长对话做上下文截断，精简无效历史消息，整体大模型调用成本降低三成。
5. **线上运维**
通过会话聚合追踪用户多轮对话，快速定位记忆丢失、上下文溢出等线上疑难问题，大幅提升排错效率。

### 2. Trace数据量大、费用高，如何裁剪优化

1. **流量层面**
线上开启**采样上报**，只抽样部分请求上报，非全量采集。
2. **上报模式**
统一开启**异步批量上报**，减少请求次数与网络开销。
3. **字段精简**
关闭完整Prompt、全量对话文本上报，只上报关键指标、耗时、Token数，屏蔽冗余内容。
4. **环境区分**
开发测试全开调试，**生产环境按需关闭非核心链路追踪**。
5. **数据清理**
配置Trace自动过期清理，定期归档删除老旧无用链路数据。
6. **业务降级**
简单问答、低优先级业务直接关闭LangSmith追踪，只保留核心业务链路。
7. **拆分项目**
按业务拆分Project，精准管控不同业务上报量级。

### 3. 用LangSmith做回归测试，防止改坏Prompt

1. 提前在平台创建**标准测试数据集**，录入行业常用提问与标准答案。
2. 迭代修改Prompt、调整检索逻辑、更换模型前，先在数据集批量跑一遍评测。
3. 对比修改前后：回答准确率、忠实度、幻觉率、耗时、Token消耗五大指标。
4. 若评测分数大幅下降，立即回滚Prompt与业务逻辑，避免线上效果崩盘。
5. 将回归评测接入开发流程，每次版本发布前强制跑完批量用例，形成AI应用版本门禁。
6. 留存每一轮版本评测报表，方便追溯是Prompt还是代码逻辑引发效果退化。
