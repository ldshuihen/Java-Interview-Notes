

# LangServe

**LangServe 是 LangChain 官方的生产级部署框架**，基于 FastAPI，一键把任意 LangChain **Runnable（Chain/Agent/LangGraph）**打包成**带文档、可流式、可观测的 REST API**，让 AI 应用从本地脚本快速变成可公网调用的微服务。

> ⚠️ 官方建议：新项目优先用 **LangGraph Platform**；LangServe 进入维护模式，只修 bug、不加新功能。

---

## 一、核心定位与架构

### 1）解决什么问题

- 不用手写 FastAPI/Flask 封装 LLM 逻辑
- 统一接口规范：**invoke/stream/batch**
- 自带文档、调试页、可观测性
- 支持**高并发、负载均衡、弹性扩缩容**

### 2）架构（三层）

```
客户端（Python/JS/HTTP）
   ↓ HTTP
LangServe（FastAPI 路由+序列化+验证+流式）
   ↓ 内部调用
LangChain 逻辑（Runnable/Chain/Agent/LangGraph）
```
- **接入层**：自动生成路由、Swagger 文档、Playground
- **核心层**：请求分发、异步执行、SSE 流式、回调聚合
- **集成层**：对接 LangSmith（追踪/评估/反馈）、认证、日志

---

## 二、核心功能（开箱即用）

### 1）自动生成标准 API 端点

- `POST /invoke`：同步调用，返回最终结果
- `POST /stream`：SSE 流式返回（逐字输出，打字机效果）
- `POST /batch`：批量调用，提升吞吐
- `POST /stream_log`：流式返回**中间步骤**（调试/审计）
- `POST /stream_events`：简化版流式事件（推荐）

---

#### 0. 统一基础规则

所有接口都是 **POST** 请求  
所有请求/返回都是 **JSON**  
`/stream` 系列是 **SSE（Server-Sent Events）** 流式推送

路径格式：
```
http://localhost:8000/{path}/{endpoint}
```
例如：
```
http://localhost:8000/chat/invoke
```

---

#### 1. **POST /invoke**

##### 作用

**同步、一次性、完整返回结果**

##### 原理

1. 接收请求
2. 执行完整链/模型计算
3. 等待全部结束
4. **一次性返回最终结果**

##### 请求体

```json
{ "input": "你好" }
```

##### 返回体

```json
{ "output": "你好！我是AI助手" }
```

##### 特点

- 最简单
- 最稳定
- 无流式
- 适合后端服务调用

##### 使用场景

- 结构化提取
- 文本分类
- 翻译
- 批量任务
- 后端内部调用

---

#### 2. **POST /stream**

##### 作用

**SSE 流式输出 → 逐字返回答案（ChatGPT 效果）**

##### 原理

- 模型每生成一个字符，就立刻推送到前端
- 不用等全部计算完
- 体验更流畅

##### 返回格式（SSE）

```
data: {"output":"你"}
data: {"output":"好"}
data: {"output":"！"}
```

##### 特点

- 只返回最终输出内容
- 不返回中间步骤
- 前端最容易处理

##### 使用场景

- 聊天界面
- 对话机器人
- 需要流畅打字机效果

---

#### 3. **POST /batch**

##### 作用

**一次性传入多个输入 → 批量执行 → 批量返回结果**

##### 请求体

```json
{
  "inputs": ["问题1", "问题2", "问题3"]
}
```

##### 返回

```json
[
  {"output": "回答1"},
  {"output": "回答2"},
  {"output": "回答3"}
]
```

##### 特点

- 一次请求处理 N 个任务
- 提升吞吐量
- 减少网络开销

##### 使用场景

- 批量生成
- 批量摘要
- 批量提取
- 批量数据处理

---

#### 4. **POST /stream_log**

##### 作用

**流式返回 + 完整内部执行日志（调试神器）**

##### 会返回什么？

- 每一步执行的节点
- LLM 调用
- 工具调用
- 检索结果
- 思考过程
- 耗时
- 输入输出

##### 返回格式（SSE）

```
data: {"steps": [...], "output": "..."}
```

##### 特点

- **信息最全**
- 适合调试 Agent / RAG
- 可解释 AI
- 审计日志

##### 使用场景

- 调试复杂链
- 展示 AI 思考过程
- 审计、日志、可观测

---

#### 5. **POST /stream_events**

##### 作用

**官方推荐：标准化、轻量、生产级流式事件**

##### 特点

- 结构统一
- 轻量简洁
- 不冗余
- 跨平台兼容
- **生产环境首选**

##### 返回事件类型

- `on_chain_start`
- `on_llm_stream`
- `on_tool_start`
- `on_chain_end`

##### 使用场景

- **正式上线**
- 前端标准化交互
- 企业级应用

#### 一、核心定义

Playground 是 LangServe **自动内置的可视化调试网页**，无需自己编写任何前端代码，服务启动后直接通过浏览器访问，在线调试 Chain、RAG、Agent 等所有挂载的可运行逻辑，是开发阶段最便捷的自测工具。

```

```

#### 三、核心功能详解

##### 1. 可视化参数输入

- 自动识别当前 Runnable 定义的**输入结构**，自动生成输入表单
- 支持普通文本、JSON 结构化参数、多字段入参，无需手动拼接请求体
- 可自由填写对话问题、业务参数、配置信息，贴合真实业务调用场景

##### 2. 多种调用模式一键切换

页面内置五大接口调用模式，和后端接口完全对应：
1. **Invoke**：同步一次性执行，等待完整结果返回
2. **Stream**：流式逐字输出，模拟聊天打字机效果
3. **Batch**：批量输入多组内容，批量获取结果
4. **Stream Log**：实时查看**全链路中间执行日志**
5. **Stream Events**：标准轻量化事件流式输出

##### 3. 查看完整执行链路

最核心调试能力：
- 清晰展示执行顺序：提示词渲染 → 大模型调用 → 工具调用 → 结果拼接
- 查看完整 Prompt 原文、模型真实入参、工具调用参数
- 查看知识库检索召回文档、Agent 思考决策过程
- 展示每一步执行耗时、Token 消耗，快速定位性能卡点

##### 4. 原生支持 Agent 与复杂流程调试

- 自动展示 Agent 思考、行动、观察完整思维流程
- 直观看到模型选择了哪个工具、传入什么参数、工具返回结果
- 循环调用、多步骤任务、分支逻辑都能完整可视化呈现

##### 5. 会话配置

支持手动填写 `config` 配置项：
- 传入 `session_id` 测试多轮对话记忆
- 自定义上下文参数、追踪标签、环境配置
- 模拟线上真实调用配置，本地调试和生产环境一致

#### 二、第1点：用 Pydantic 自动推断输入/输出 Schema

##### 什么是 Pydantic？

Python 最强**数据验证库**，用来定义：
- 字段是什么类型
- 哪些字段必填
- 字段格式是否正确

##### LangServe 做了什么？

**它会自动读取你 Chain / Agent / Model 的输入输出结构，生成标准 JSON Schema。**

比如你定义：
```python
class Input(BaseModel):
    question: str
    user_id: str
```

LangServe **自动识别**：
- 输入必须包含 `question` 和 `user_id`
- 都是字符串
- 不传就报错
- 传 int 也报错

##### 好处

- AI 服务**不会收到脏数据**
- 提前拦截错误，**节省 Token、节省成本**
- 接口稳定、可维护、可协作

---

#### 三、第2点：强校验，错误信息友好

##### 什么是“强类型校验”？

**你传错参数，接口直接拒绝，不会进入大模型。**

例如：
```json
{
  "question": 12345  // 应该传字符串，你传了数字
}
```

返回：
```json
{
  "detail": [
    {
      "loc": ["body", "input", "question"],
      "msg": "Input should be a valid string",
      "type": "string_type"
    }
  ]
}
```

##### 优点

- 前端一看就知道错在哪
- 后端不用写校验逻辑
- 稳定、安全、工程化

#### 二、第1点：基于 FastAPI + Uvicorn，异步非阻塞

##### 1. FastAPI 是什么？

**目前 Python 世界性能最强、速度最快、生产最稳的 Web 框架**
比 Flask、Django 快 **2~10 倍**

##### 2. Uvicorn 是什么？

**高性能异步 ASGI 服务器**
专门处理高并发、长连接、流式输出（SSE）

##### 3. 异步非阻塞是什么意思？（超级重要）

**同步（旧方式）：**
一个用户请求 → 等着大模型返回 → 期间别人都要排队
→ **慢、卡、并发低**

**异步（LangServe方式）：**
请求进来 → 交给模型 → **不等待，直接处理下一个请求**
模型算好 → 自动回来返回结果
→ **高并发、不卡顿、资源利用率极高**

##### 4. 对AI服务的巨大好处

- 大模型推理很慢，但**异步不会卡住服务**
- 同时处理 100/1000 个请求都没问题
- 流式输出（SSE）超级稳定
- 内存占用低、CPU利用率高

## Langserve现状

**没有完全替代，但官方已将 LangServe 放入“维护模式”，新项目优先用 LangGraph Platform；简单 LCEL 链仍推荐用 LangServe，复杂/有状态/Agent 场景才必须用 Platform**。

### 一、官方定位：并存但分场景

- **LangServe现状**：
  - ✅ 状态：**维护模式（Maintenance）**，只修 bug、不加新功能。
  - ✅ 适用：**简单无状态 LCEL 链/Runnable**（如单轮问答、摘要、简单 RAG）。
  - ✅ 优势：**极简、轻量、上手最快**，几行代码即可暴露 API。
- **LangGraph Platform现状**：
  - ✅ 状态：**主力开发（Active）**，LangGraph 应用的**官方默认部署方案**。
  - ✅ 适用：**有状态、多轮、Agent、人机交互、长流程**（如对话机器人、工具调用、多智能体）。
  - ✅ 优势：内置**持久化、记忆、中断、线程管理、高级流式、定时任务、Webhook**等企业级能力，这些 LangServe 没有。

### 二、核心差异（为什么没完全替代）

| 维度           | LangServe                  | LangGraph Platform                   |
| -------------- | -------------------------- | ------------------------------------ |
| **适用对象**   | 无状态 LCEL Chain/Runnable | 有状态 LangGraph（Agent/多轮）       |
| **状态管理**   | ❌ 无内置状态，需自己做     | ✅ 原生支持 Thread/Checkpointer/Store |
| **人机交互**   | ❌ 不支持中断/审批          | ✅ 原生支持 Human-in-the-loop         |
| **部署模型**   | FastAPI 插件（add_routes） | 独立服务器（langgraph dev/cloud）    |
| **学习成本**   | 极低（会 LCEL 就会用）     | 中等（需了解 LangGraph 概念）        |
| **生产成熟度** | 稳定但功能有限             | 企业级，持续迭代                     |

### 三、官方建议（2026年最新）

1. **新项目**：
   - 简单链 → **LangServe**（最快、最轻）。
   - Agent/多轮/有状态 → **LangGraph Platform**（必须）。
2. **旧项目**：
   - LangServe 可继续用，**长期支持 bug 修复**。
   - 如需复杂能力（如记忆、中断）→ 逐步迁移到 LangGraph + Platform。
3. **混合部署**：
   - 同一 FastAPI 应用可同时挂载 **LangServe 路由（简单链）+ 手写 LangGraph 路由（复杂流程）**，中型团队常用此方案。

**LangGraph Platform 是 LangGraph（Agent）的替代，不是 LangServe（简单链）的完全替代**。  
- 简单无状态 → 继续用 LangServe（最快）。  
- 复杂有状态/Agent → 必须用 LangGraph Platform。

- 适配复杂业务流程迭代与流程重构

#### 不适合

- 简单单轮接口、追求极致轻量化、只想快速写接口

---

### 三、一句话决策口诀

1. **流程固定、单轮无状态、快速出接口 → LangServe**
2. **流程灵活、多轮有状态、智能体复杂编排 → LangGraph Platform**

### 四、极简选型对照表

| 业务特征   | 选 LangServe      | 选 LangGraph Platform  |
| ---------- | ----------------- | ---------------------- |
| 代码风格   | LCEL 链式写法     | 节点+边状态图写法      |
| 会话记忆   | 自己手动实现      | 原生自带持久化记忆     |
| 流程结构   | 直线串行          | 分支/循环/并行         |
| 人工干预   | 不支持            | 原生支持暂停、审批     |
| 部署难度   | 极低，嵌入FastAPI | 标准服务部署，规范更强 |
| 迭代复杂度 | 简单需求稳定够用  | 复杂业务长期迭代首选   |
| 新项目倾向 | 简单功能首选      | 智能体类业务强制首选   |

### 五、官方最终建议

1. 纯 LangChain 简单业务 → 维持 **LangServe** 不变，稳定够用
2. 新项目做 Agent、多轮对话、复杂流程 → 直接上 **LangGraph + LangGraph Platform**
3. 不要强行用 LangServe 部署 LangGraph 流程，会丢失状态、断点、线程等核心能力

# LangGraph Platform

LangGraph Platform 是 LangChain 推出的**企业级 AI Agent 托管与部署平台**，核心是把用 LangGraph 编写的状态化工作流，一键变成可生产、可扩展、可观测的 HTTP 服务，彻底解决自研 Agent 的部署、状态管理、容错与多客户端接入难题。

简单说：**LangGraph 是写 Agent 的框架，LangGraph Platform 是跑 Agent 的生产运行时**。

---

## 一、核心定位与价值

### 解决的核心痛点
- **状态丢失**：服务重启/宕机后，多轮对话、长任务上下文全丢。
- **不可控流程**：Agent 死循环、无限调用工具、无法人工干预。
- **部署复杂**：需自行处理异步、队列、持久化、负载均衡、流式输出。
- **多端接入难**：Java/Go/前端/移动端无法标准化调用。

### 核心价值
- **一键生产化**：本地 Graph → 自动生成高可用 HTTP 服务，无需写 FastAPI/Uvicorn 代码。
- **原生状态管理**：托管 PostgreSQL 持久化，自动检查点（Checkpoint），支持故障恢复、时间旅行调试。
- **企业级可控性**：内置人机协作（Human-in-the-Loop）、中断/恢复、审批流。
- **全链路可观测**：深度集成 LangSmith，追踪每步 Trace、Token 消耗、延迟，支持在线调试与评估。
- **多语言客户端**：Python/JS/任意 HTTP 客户端（含 Java 微服务）无缝调用。

## 二、核心架构（四大组件）
### 1. LangGraph Server（运行时核心）

- **标准化 API**：自动生成 `/threads`（会话）、`/runs`（执行）、`/invoke`（同步）、`/stream`（流式）、`/batch`（批量）接口。
- **托管持久化**：内置 PostgreSQL 存储状态、会话历史、长期记忆，自动检查点（每步执行后快照）。
- **异步任务队列**：高并发、突发流量削峰，自动重试、断点续跑，避免任务丢失。
- **流式输出引擎**：原生支持 Token 级流式（SSE），适配前端打字机效果。

LangGraph Server 是整个 Platform 的**服务运行中枢**，承载工作流调度、会话管理、数据持久化、流量调度、流式推送全部核心能力，是 LangGraph 工作流从本地代码转为线上稳定服务的核心载体。

#### 一、标准化API接口

自动生成全套规范REST接口，统一调用范式，多语言无缝对接（Python/JS/Java等）
1. **`/threads` 会话管理接口**
    专门用来创建、查询、删除、管理独立对话线程
    - 每个`thread_id`代表**独立会话上下文**，多轮对话依靠线程ID绑定历史状态
    - 支持新建空会话、查询会话历史、清空会话、会话归档
    - 作用：隔离不同用户、不同场景对话，实现千人千面独立记忆，互不干扰
2. **`/runs` 任务执行管理**
    管控所有工作流运行实例
    - 发起执行、查询运行状态、终止运行、暂停/恢复任务
    - 可查看任务耗时、执行节点、失败原因，适配长流程任务监控
3. **`/invoke` 同步执行**
    全流程跑完一次性返回最终结果，阻塞式调用
    - 适用：数据提取、文案生成、结果汇总、后端业务同步对接
    - Java微服务、后端系统最常用对接接口
4. **`/stream` 流式执行**
    基于SSE协议，逐Token实时推送输出内容
    - 实现聊天界面打字机实时输出，用户体验最优
    - 全程长连接，不断流、不丢字
5. **`/batch` 批量执行**
    一次性批量传入多组输入，批量调度工作流统一执行
    - 提升批量数据处理吞吐量，减少重复请求开销
    - 适用批量文档解析、批量问答、批量内容生成

#### 二、托管持久化

1. **底层存储**
    内置托管PostgreSQL数据库，无需用户自行搭建数据库、设计表结构
    自动完成数据表创建、索引优化、数据读写
2. **存储内容**
    - 会话维度：对话历史、用户输入、模型输出
    - 状态维度：Graph全局自定义状态、节点中间运算结果
    - 记忆维度：短期会话记忆、用户长期偏好记忆
3. **自动检查点（Checkpoint）**
    工作流**每走完一个节点**，自动快照保存当前全部状态
    - 服务宕机、进程重启、网络中断，重启后自动从最近检查点恢复执行，无需从头重新运行
    - 支持时间旅行调试：回溯任意历史节点状态，复盘执行过程，快速定位逻辑bug
4. **优势**
    彻底解决普通AI服务**重启丢上下文、多轮对话失效、长任务中断作废**痛点，适配小时级、天级超长业务流程。

#### 三、异步任务队列

1. **核心作用**
    内置分布式异步队列，承接突发高并发请求，实现流量削峰填谷
2. **核心能力**
    - **并发隔离**：大量请求涌入时排队有序执行，不会压垮推理服务
    - **自动重试**：模型调用超时、网络异常、临时报错，配置重试策略自动重试
    - **断点续跑**：队列中暂停的任务，恢复后继续从断点执行，不重复消耗Token
    - **任务隔离**：区分优先级任务，核心业务优先执行，非核心任务延后调度
3. **业务价值**
    轻松应对活动峰值、用户集中访问场景，保障服务稳定性，避免雪崩。

#### 四、流式输出引擎

1. **协议支持**
    原生原生适配**SSE（服务端推送事件）**标准流式协议，行业通用无兼容问题
2. **输出粒度**
    支持**Token级精细化推送**，模型每生成一个字词立即推送前端
3. **适配场景**
    - Web聊天窗口、小程序对话、APP智能助手实现实时打字效果
    - 支持前端实时渲染思考过程、工具调用进度
4. **兼容性**
    前端JS、Java后端转发、移动端均能标准解析流数据，无需自定义解析规则。

#### 五、整体运行逻辑总结

客户端通过`threads`创建独立会话 → 调用`invoke/stream`触发Graph工作流 → Server调度异步队列分发任务 → 执行过程中每节点自动存入PG检查点 → 同步直接返回结果、流式实时推送内容 → 全程自动记录状态与会话历史，异常自动重试恢复。

#### **一、一句话讲透 StateGraph**

**StateGraph = 带状态的流程图引擎**
你画一张流程图（节点 + 跳转逻辑），所有节点**共享同一个全局状态（记忆）**，流程可以**循环、分支、暂停、恢复**。

它和传统工作流（DAG）最大区别：
**传统工作流：只能单向执行，不能回头，没有记忆。
StateGraph：可以循环、可以跳转、有全局记忆、永不丢失上下文。**

---

#### **二、第 1 点：有向有环图（真正的智能体核心）**

##### 普通工作流（DAG）：

只能 → → →
**不能回头、不能循环、不能重新执行节点**
只能做简单线性任务。

##### StateGraph（有向有环图）：

可以：
- **循环**（思考 → 查工具 → 再思考 → 再查工具…直到完成）
- **条件分支**（if/else 逻辑：用户问天气→走天气流程；问新闻→走新闻流程）
- **并行执行**（同时查天气、查新闻、查日历）
- **子图嵌套**（大流程里套小Agent，小Agent套小小Agent）
- **任意跳转**（从节点D直接跳回节点A）

##### 这意味着什么？

**只有 StateGraph 能做真正的自主 Agent！**
比如：
- 写论文：搜索 → 整理 → 撰写 → 检查 → 再搜索 → 再撰写
- 客服：理解 → 查询 → 回答 → 反问 → 确认
- 自动化：判断 → 执行 → 验证 → 重试 → 完成

**没有循环，就没有真正的智能体。**

### 4. 记忆管理（Memory）
- **短期记忆**：会话内上下文（对话历史、中间结果），线程隔离。
- **长期记忆**：跨会话持久化（用户偏好、历史交互、知识积累），向量检索支持。
- **记忆压缩**：自动精简冗余信息，避免状态膨胀，提升性能。

#### 一、核心概述

LangGraph Platform 内置分层记忆体系，把对话与业务数据按**使用时效**划分为短期、长期两类，同时自带记忆压缩优化机制，既保障上下文连贯、用户个性化记忆留存，又避免数据臃肿拖慢流程执行，是实现拟人化连续对话、个性化智能服务的核心支撑。

#### 二、短期记忆

1. **存储范围**
专属单个会话线程内的临时数据，包含实时对话问答记录、节点执行产生的中间运算结果、临时业务参数、本轮流程交互信息。
2. **隔离机制**
基于 `thread_id` 实现严格线程隔离，不同用户、不同会话的短期记忆完全独立互不干扰，不会出现对话串线、数据混淆问题。
3. **生命周期**
依附当前会话流程存在，仅用于本轮多轮对话、流程串联，服务运行期间实时读写，依托检查点随流程状态同步保存，会话结束后仅做留存，不跨会话主动复用。
4. **核心作用**
无缝承接多轮问答，让AI读懂上下文语义，连贯完成连续提问、追问、流程递进类交互，是日常聊天、流程式任务必备基础记忆。

#### 三、长期记忆

1. **存储范围**
脱离单一会话限制，全局持久化存储用户固定属性与历史沉淀数据，包括用户使用偏好、历史交互习惯、常用指令、个人资料、业务常用信息、行业专属知识等。
2. **存储与检索**
底层对接向量数据库，自动将非结构化交互内容向量化存储；发起新会话时，可通过语义相似度检索，快速匹配调取对应长期记忆，实现跨对话、跨时间段的个性化应答。
3. **生命周期**
长期永久持久化保存，不受会话关闭、服务重启、设备切换影响，用户再次发起对话可自动唤醒历史沉淀信息。
4. **核心作用**
打造专属个性化AI服务，无需用户重复重复告知自身需求、习惯与信息，大幅提升交互体验，适配私人助手、企业专属智能客服、长期业务助理等场景。

#### 四、记忆压缩

1. **解决痛点**
长时间多轮对话、复杂长流程执行后，对话历史、中间数据会持续堆积，造成全局State状态体积膨胀，不仅增加大模型输入Token消耗、拉高调用成本，还会拖慢推理速度、降低流程响应性能。
2. **压缩执行逻辑**
系统自动识别冗余重复内容、无效闲聊、过时中间数据，精简提炼核心关键信息，保留语义核心与关键业务数据，剔除无用冗余内容。
3. **压缩形式**
支持摘要精简、历史合并、无效信息过滤、对话浓缩等多种模式，在**不丢失核心上下文语义**的前提下，大幅缩减记忆数据体量。
4. **实际价值**
平衡上下文完整性与服务性能，既保证对话逻辑连贯不断层，又严控输入长度、降低成本、提升整体流程运行效率，适配高频长期连续交互场景。

#### 五、整体协同流程

1. 用户发起对话，系统优先加载**短期记忆**承接本轮上下文交互；
2. 同时通过向量检索调取匹配**长期记忆**，融入回答实现个性化适配；
3. 交互持续进行中，后台自动执行**记忆压缩**，清理冗余数据；
4. 会话结束后，有效交互数据沉淀更新至长期记忆，短期记忆随会话归档留存，等待下次唤醒复用。

#### 六、业务落地优势

1. 分层记忆架构清晰，适配C端用户聊天、B端企业流程等不同业务场景；
2. 原生集成向量检索，无需额外开发记忆调取逻辑，快速搭建个性化RAG智能问答；
3. 自动记忆压缩无需人工干预，兼顾体验与性能，适配高并发、长周期AI服务；
4. 记忆数据可对接Java业务系统，同步用户业务档案，实现AI记忆与企业业务数据打通联动。

### 5. 多代理协作（Multi-Agent）

#### 一、一句话讲透：什么是多代理协作？

**一个任务太复杂 → 拆成多个小AI专家 → 分工合作、互相通信、一起完成任务。**
就像一个**公司团队**：
- **产品经理（规划Agent）**：定方案
- **程序员（执行Agent）**：干活
- **测试（验证Agent）**：检查对错
- **老板（主控Agent）**：总调度

**LangGraph 让多个AI像人一样团队协作干活。**

#### **二、第1点：节点级 Agent（每个节点都是独立AI专家）**

##### 核心含义

**Graph 里的每一个节点，都可以是一个完整独立的小 Agent**。
每个节点都能单独设置：
- 用哪个大模型（GPT-4o / DeepSeek / 通义千问）
- 用哪些工具（搜索、数据库、Excel、API）
- 用什么提示词（角色、性格、任务）
- 独立逻辑

##### 典型分工（超级实用）

1. **规划节点（Planner Agent）**
   理解需求 → 拆解步骤 → 制定计划
2. **工具调用节点（Tool Agent）**
   查资料、算数据、搜文档
3. **生成节点（Writer Agent）**
   写文案、写报告、写代码
4. **校验节点（Checker Agent）**
   检查错误、事实核查、格式校验

##### 为什么要这样？

**专业的人干专业的事，效果比单个AI强10倍。**

#### **三、第2点：状态共享协作（不用消息队列，极简通信）**

##### 传统多Agent通信痛点

- 用消息队列
- 发消息、收消息、等待消息
- 复杂、容易乱、容易丢

##### LangGraph 革命性方案

**全局状态 State = 共享黑板**
- 所有Agent都看同一块黑板
- A写完 → B直接看
- B写完 → C直接用
- **不用发消息！不用调用！不用等待！**

##### 协作流程（超级丝滑）

1. 规划Agent把计划写进State
2. 工具Agent读取计划 → 执行 → 把结果写回State
3. 生成Agent读取结果 → 生成内容 → 写回State
4. 校验Agent读取内容 → 检查 → 写回State

**全程自动流转，零通信代码，零复杂度。**

## 五、Java 微服务调用（重点）
LangGraph Platform 暴露**标准 HTTP/JSON 接口**，Java 微服务可直接调用，支持**同步/流式/批量**三种模式。

### 1. 同步调用（/invoke）
```java
// Spring Boot + RestTemplate
String url = "https://your-langgraph-platform.com/your-graph/invoke";
String requestBody = "{\"input\": {\"messages\": [{\"role\": \"user\", \"content\": \"你好，介绍 LangGraph Platform\"}]}}";

RestTemplate restTemplate = new RestTemplate();
String response = restTemplate.postForObject(url, requestBody, String.class);
System.out.println(response);
```

### 2. 流式调用（/stream，实现打字机效果）
Java 后端可通过 `SseEmitter` 或 OkHttp 订阅 SSE 流，逐字接收输出。

### 3. 会话管理（/threads）
- 创建会话：`POST /threads` → 返回 `thread_id`
- 继续对话：`POST /threads/{thread_id}/runs` → 携带上下文，实现多轮对话
