# 设计范式

现代 LLM 智能体工程里，公认的**四大经典设计范式**是：**ReAct、Plan-and-Solve（Plan & Execute）、Reflection（Reflexion）、Tree of Thoughts（ToT）**；学术基础分类则从简单到复杂分为：**简单反射、模型反射、目标导向、效用导向、学习型智能体**。下面把两套体系都讲清楚。

---

## 一、学术基础五层范式（Russell & Norvig）

来自《人工智能：一种现代方法》，按“智能程度/架构复杂度”递进：

### 1）简单反射智能体（Simple Reflex Agent）
- 逻辑：**感知 → 直接动作**，无记忆、无模型
- 特点：规则硬编码、环境**完全可观测**、只能做“if-then”
- 例子：空调恒温器、简单聊天机器人（关键词触发）
- 局限：环境部分观测/变化时失效

### 2）模型反射智能体（Model-Based Reflex Agent）
- 逻辑：**感知 + 内部世界模型 + 状态记忆 → 动作**
- 特点：维护环境状态，能处理**部分可观测**
- 例子：自动驾驶（定位+地图+障碍物跟踪）
- 核心：世界模型 = 环境状态 + 转移规则

### 3）目标导向智能体（Goal-Based Agent）
- 逻辑：**当前状态 + 目标 → 搜索/规划 → 动作序列**
- 特点：明确目标，主动找路径，支持**长序列规划**
- 例子：路径规划（A*）、游戏AI（通关目标）
- 核心：目标驱动，多步推理

### 4）效用导向智能体（Utility-Based Agent）
- 逻辑：**状态 + 目标 + 效用函数（偏好/代价） → 最优动作**
- 特点：不止“可达”，还要**最优/最省/最稳**；处理不确定性
- 例子：推荐系统、投资决策、自动驾驶（安全/速度权衡）
- 核心：效用函数 = 价值 + 风险 + 成本

### 5）学习型智能体（Learning Agent）
- 逻辑：**感知 → 动作 → 反馈 → 学习 → 优化策略/模型**
- 特点：**数据驱动、自主进化、泛化能力强**
- 组成：学习器（改进）、性能器（执行）、评论家（反馈）、问题生成器（探索）
- 例子：强化学习（AlphaGo）、LLM 微调、自适应推荐
- 核心：从经验中持续优化，突破初始编程限制

## 二、大模型时代四大工程范式（最常用）
LLM 作为“大脑”，聚焦**思考、行动、规划、反思**的组合方式：

### 1）ReAct

（Reasoning + Acting）——边想边做（最主流）

- 核心：**Thought → Action → Observation → Repeat**，思考与行动交替循环
- 流程：
  1. Thought：LLM 推理、拆解任务、决定下一步
  2. Action：调用工具（API/函数/数据库）
  3. Observation：获取结果，回到 Thought
- 优点：**动态自适应、适合开放/不确定环境、自然交互**
- 缺点：多轮调用、**成本高、易循环**
- 场景：对话助手、信息检索、客服、代码助手

#### 一、核心定义

**ReAct = Reasoning（推理思考） + Acting（行动执行）**
是**大模型智能体最经典、落地最多**的交互执行范式，打破纯文本空想，让大模型**一边思考、一边调用工具行动、一边接收环境反馈**，循环完成复杂任务。

#### 二、底层核心逻辑

固定闭环三元循环：
**Thought 思考 → Action 行动 → Observation 观察反馈 → 回到思考**
无限迭代直到任务结束。

#### 三、逐环节深度拆解

##### 1. Thought 思考层（大模型大脑）

1. 理解当前全局任务
2. 复盘历史所有对话、行动、结果
3. 判断**当前缺什么信息**
4. 自主决策：下一步要不要调用工具、调用哪个工具、传什么参数
5. 禁止直接编造答案，优先选择行动获取真实信息

**思考输出格式**：
我现在需要XX信息，我决定调用XX工具，传入参数XX

##### 2. Action 行动层（外部能力接口）

由代码层解析大模型思考内容，**自动执行动作**
常见行动类型：

- 联网搜索
- 数据库查询
- 计算器运算
- 接口API调用
- 文件读写
- 代码执行
- 天气/日期/地图查询

**特点**：大模型只做决策，不执行，执行交给程序。

##### 3. Observation 观察反馈层（环境回执）

把**工具执行后的真实结果**原样回传给大模型
- 成功：带回数据、文本、数值
- 失败：带回报错、无结果、权限不足
- 空结果：告知无相关信息

作用：**修正大模型幻觉，用真实事实约束思考**。

#### 四、完整运行流程（实战流程）

1. 用户下发原始任务
2. 智能体进入第一轮 Thought：分析任务，确定要调用工具
3. 解析指令，执行 Action 工具调用
4. 拿到 Observation 结果
5. 第二轮 Thought：结合新结果继续推理，判断是否完成
6. 未完成继续行动，已完成整理答案输出
7. 任务终止

#### 五、典型运行示例

**用户**：查今天北京气温，再算比郑州高几度
1. Thought：需要先查北京今日气温，调用天气工具
2. Action：调用天气API 查询北京
3. Observation：北京28℃
4. Thought：还需查询郑州气温
5. Action：调用天气API 查询郑州
6. Observation：郑州24℃
7. Thought：数据齐全，调用计算器做差值
8. Action：计算28-24
9. Observation：结果4
10. 整合所有信息，输出最终答案

#### 六、核心优势

1. **极强动态适配**
环境信息可变、任务中途变需求都能自适应调整行动
2. **彻底抑制幻觉**
答案全部依赖工具真实结果，不靠模型脑补
3. **天然适配工具调用**
是函数调用、插件系统最匹配的架构
4. **逻辑贴近人类做事习惯**
想一步、做一步、看结果、再想下一步
5. **通用百搭**
几乎所有业务智能体都能基于ReAct搭建

#### 七、明显缺点

1. **多轮调用，token 消耗大，推理成本高**
2. **容易陷入死循环**
重复调用同一个工具、无效反复思考
3. **依赖模型决策能力**
模型思考出错，行动就会全错
4. **长任务上下文容易溢出**
轮次越多历史信息越长
5. **执行速度偏慢**
多轮串行调用，无法并行

#### 八、最适用业务场景

1. 智能问答、知识库问答
2. 联网搜索助手、学术文献检索
3. 企业客服智能工单处理
4. 代码调试、工具自动化助手
5. 数据查询+统计分析类任务
6. 实时资讯、行情、天气类智能体
7. LangChain / LangGraph 通用业务智能体

#### 九、不适合场景

1. 固定流程、步骤不变的批量任务（优先Plan执行式）
2. 极致低成本极简问答
3. 高强度数学推理、复杂逻辑博弈（优先ToT）
4. 文案精修、严谨公文写作（优先Reflection反思）

#### 十、ReAct 与普通提示词、普通函数调用区别

1. 普通提示词：纯文本回答，无行动无反馈
2. 简单函数调用：一次性调用，无连续思考闭环
3. **ReAct**：自主连续决策 + 多轮行动 + 反馈迭代 = 真正自主智能体

#### 十一、工程落地架构分层

1. **大模型层**：负责Thought思考决策
2. **解析层**：提取Action工具名+参数
3. **工具调度层**：统一管理所有外部能力
4. **记忆层**：存储历史Thought、Action、Observation
5. **终止判断层**：判断是否结束任务

#### 十二、优化改良方向（工业级常用）

1. 增加**最大轮次限制**，防止死循环
2. 加入**结果校验**，过滤无效Observation
3. 搭配**短时记忆+长时记忆**提升连贯性
4. ReAct + Reflection 结合：行动后自我纠错
5. 前置轻量规划，减少无效行动

### **langchain实现ReAct**

#### 1. 安装依赖

```bash
pip install langchain langchain-openai python-dotenv
```

#### 2. 完整可运行代码

```python
from dotenv import load_dotenv
import os
from langchain_openai import ChatOpenAI
from langchain_community.tools import CalculatorTool
from langchain.agents import AgentExecutor, create_react_agent
from langchain_core.prompts import PromptTemplate

# 加载环境变量
load_dotenv()

# 1. 初始化大模型
llm = ChatOpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url=os.getenv("OPENAI_BASE_URL"),
    model="gpt-3.5-turbo",
    temperature=0
)

# 2. 定义可用工具
tools = [CalculatorTool()]

# 3. ReAct 标准提示词模板
react_prompt = PromptTemplate.from_template("""
Answer the following questions as best you can. You have access to the following tools:

{tools}

Use the following format:

Question: the input question you must answer
Thought: you should always think about what to do
Action: the action to take, should be one of [{tool_names}]
Action Input: the input to the action
Observation: the result of the action
... (this Thought/Action/Action Input/Observation can repeat N times)
Thought: I now know the final answer
Final Answer: the final answer to the original input question

Begin!

Question: {input}
Agent scratchpad: {agent_scratchpad}
""")

# 4. 创建 ReAct 智能体 + 执行器
agent = create_react_agent(llm, tools, react_prompt)
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 5. 调用测试
if __name__ == "__main__":
    res = agent_executor.invoke({"input": "计算 123 乘以 456 再加 789 等于多少"})
    print("\n最终结果：", res["output"])
```

#### 3. 新建 `.env` 文件

```env
OPENAI_API_KEY=你的密钥
OPENAI_BASE_URL=https://api.openai.com/v1
```

#### 4. 运行效果（ReAct 标准流程）

自动走完闭环：
1. **Thought**：需要用计算器计算算式
2. **Action**：调用 Calculator
3. **Action Input**：`123 * 456 + 789`
4. **Observation**：算出数值
5. **Thought**：已知答案
6. **Final Answer**：输出结果

#### 5. 核心要点

- 严格遵循 ReAct 三元循环：`Thought → Action → Observation`
- `verbose=True` 可打印完整思考过程
- 新增工具直接往 `tools` 列表加即可
- 天然支持多轮连续工具调用

#### 6. 扩展加搜索工具示例

```python
from langchain_community.tools import DuckDuckGoSearchRun
tools = [CalculatorTool(), DuckDuckGoSearchRun()]
```
即可实现：**先联网查数据 → 再计算** 完整 ReAct 业务流程。

这是**最标准、最简洁、可直接运行**的 **LangGraph 版 ReAct 智能体**。
完全遵循 **Thought → Action → Observation → Thought** 循环，是工业界最常用的实现。

### LangGraph ReAct 智能体

#### 安装依赖

```bash
pip install langgraph langchain-openai python-dotenv
```

#### 完整代码

```python
from dotenv import load_dotenv
import os
from typing import TypedDict, Annotated
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langgraph.graph import StateGraph, START, END
from langchain_core.messages import HumanMessage, AIMessage, ToolMessage
import operator

# 加载环境
load_dotenv()

# 1. 初始化大模型
llm = ChatOpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url=os.getenv("OPENAI_BASE_URL"),
    model="gpt-3.5-turbo",
    temperature=0
)

# ===================== 2. 定义工具（ReAct 核心：能调用外部能力）=====================
@tool
def calculator(expression: str) -> str:
    """
    计算器工具，用于数学计算
    参数: expression (str): 数学表达式，例如 "3*5+2"
    """
    try:
        return str(eval(expression))
    except:
        return "计算错误"

@tool
def search(query: str) -> str:
    """
    模拟联网搜索
    参数: query: 搜索关键词
    """
    # 这里模拟真实搜索，你可以替换成真实搜索引擎
    if "北京气温" in query:
        return "北京当前气温 28℃，晴天"
    if "上海气温" in query:
        return "上海当前气温 25℃，多云"
    return "未找到信息"

# 绑定工具
tools = [calculator, search]
llm_with_tools = llm.bind_tools(tools)

# ===================== 3. 定义状态 =====================
class AgentState(TypedDict):
    messages: Annotated[list, operator.add]  # 消息历史
    final_answer: str

# ===================== 4. ReAct 核心节点 =====================

# 节点1：思考 + 行动（Thought → Action）
def think_and_act(state: AgentState):
    messages = state["messages"]
    response = llm_with_tools.invoke(messages)
    return {"messages": [response]}

# 节点2：执行工具，返回观察结果（Observation）
def execute_tool(state: AgentState):
    last_message = state["messages"][-1]
    tool_call = last_message.tool_calls[0]
    
    # 找到对应工具并执行
    for tool in tools:
        if tool.name == tool_call["name"]:
            result = tool.invoke(tool_call["args"])
            break
    
    # 返回工具执行结果
    return {
        "messages": [
            ToolMessage(content=result, tool_call_id=tool_call["id"])
        ]
    }

# ===================== 5. 路由判断 =====================
def should_continue(state: AgentState):
    last_message = state["messages"][-1]
    # 如果有工具调用 → 继续执行
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "execute_tool"
    # 否则 → 结束，给出答案
    return "end"

# ===================== 6. 构建 LangGraph 流程图 =====================
builder = StateGraph(AgentState)

# 添加节点
builder.add_node("think", think_and_act)
builder.add_node("execute_tool", execute_tool)

# 连线
builder.add_edge(START, "think")
builder.add_conditional_edges(
    "think",
    should_continue,
    {"execute_tool": "execute_tool", "end": END}
)
builder.add_edge("execute_tool", "think")  # 执行完回到思考

# 编译
graph = builder.compile()

# ===================== 7. 测试运行 =====================
if __name__ == "__main__":
    print("=== LangGraph ReAct 智能体运行 ===")
    
    # 测试问题：需要调用多个工具
    query = "北京和上海的气温差是多少？"
    
    events = graph.stream(
        {"messages": [HumanMessage(content=query)]},
        stream_mode="values"
    )

    # 打印执行过程
    for event in events:
        last_msg = event["messages"][-1]
        print("\n---")
        if isinstance(last_msg, AIMessage):
            print(f"🤖 思考/行动: {last_msg.content}")
            if last_msg.tool_calls:
                print(f"🔧 调用工具: {last_msg.tool_calls[0]['name']}")
        elif isinstance(last_msg, ToolMessage):
            print(f"📊 观察结果: {last_msg.content}")

    # 最终答案
    print("\n✅ 最终答案:", event["messages"][-1].content)
```

#### 运行效果（标准 ReAct 流程）

1. **Thought**：我需要先查北京气温，再查上海气温
2. **Action**：调用 `search` 查北京
3. **Observation**：北京 28℃
4. **Thought**：再查上海
5. **Action**：调用 `search` 查上海
6. **Observation**：上海 25℃
7. **Thought**：计算差值
8. **Action**：调用 `calculator` 算 28-25
9. **Observation**：3
10. **Final Answer**：温差 3℃

#### 核心特点（面试/考试必背）

- **纯原生 LangGraph**，无黑盒封装
- 严格遵循 **Thought → Action → Observation** 循环
- 可**无限轮次工具调用**
- 自带**路由判断**，自动决定继续行动还是结束
- 可无限扩展工具（搜索、数据库、API、代码执行）



### 2）Plan-and-Solve（Plan & Execute）——三思而后行
- 核心：**先全局规划，再分步执行**，规划与执行解耦
- 流程：
  1. Plan：LLM 生成完整步骤链（DAG/清单）
  2. Execute：按计划逐一执行（可轻量模型/代码）
  3. Solve：汇总结果交付
- 优点：**可控、可调试、成本低（规划一次）、长任务稳定**
- 缺点：**环境变化时僵化、规划错误导致整体失败**
- 场景：报告生成、数据分析、项目管理、论文写作



#### 一、核心定义

**Plan-and-Solve** 即**先全局统筹规划，再按流程分步执行，最后整合结果解决问题**
是大模型智能体**结构化任务首选范式**，把**思考规划**和**落地执行**彻底拆分成两个独立阶段，区别于ReAct边想边做的即时决策模式。

#### 二、核心思想

做事逻辑：**定好全盘路线 → 严格按路线走 → 收尾汇总答案**
全程**一次规划、批量执行**，中途不随意更改整体流程，秩序性极强。

#### 三、完整三层标准流程

##### 1. Plan 全局规划阶段（大模型主导）

1. 深度拆解用户复杂需求
2. 梳理任务依赖关系、先后顺序
3. 输出**有序步骤清单/任务流程图/DAG依赖图**
4. 明确每一步需要调用什么工具、获取什么数据
5. 预判潜在卡点，提前预留处理逻辑
**输出产物**：标准化执行方案、分步任务列表

##### 2. Execute 分步执行阶段（代码/轻量模型主导）

1. 严格按照规划好的顺序依次执行
2. 调用对应工具、接口、数据库、脚本完成单步任务
3. 只执行不做全局决策，仅处理单步小异常
4. 依次收集每一步执行结果并缓存

##### 3. Solve 整合求解阶段

1. 汇总所有步骤产出的结果、数据、内容
2. 统一梳理排版、逻辑串联、格式规整
3. 去除冗余信息，形成完整最终答案
4. 任务闭环结束

#### 四、极简执行示例

用户需求：整理本周销售数据，做统计报表并撰写工作总结
1. **Plan规划**
   步骤1：调取本周全部销售订单数据
   步骤2：统计总销量、客单价、营收总额
   步骤3：筛选热销产品与滞销产品
   步骤4：对比上周数据完成涨跌分析
   步骤5：根据数据撰写正式工作总结
2. **Execute执行**：按1-5顺序逐一调取数据、统计、筛选
3. **Solve整合**：拼接数据报表+分析内容，输出完整工作报告

#### 五、核心优势

1. **流程高度可控**
步骤透明可查看、可修改、可人工干预，极易调试排错
2. **推理成本更低**
仅前期做一次全局规划，无需多轮大模型思考，大幅节省Token
3. **长周期任务极度稳定**
适合步骤多、链路长、流程固定的大型任务，不易混乱

```python
from dotenv import load_dotenv
import os
from typing import TypedDict, List
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END
from langchain_core.prompts import ChatPromptTemplate

load_dotenv()

# 1. 初始化大模型
llm = ChatOpenAI(
    api_key=os.getenv("OPENAI_API_KEY"),
    base_url=os.getenv("OPENAI_BASE_URL"),
    model="gpt-3.5-turbo",
    temperature=0
)

# 2. 定义状态结构体
class PlanState(TypedDict):
    user_query: str
    plan_list: List[str]       # 生成的执行计划
    current_step: int          # 当前执行步骤
    step_result: List[str]     # 每步执行结果
    final_answer: str          # 最终汇总答案

# 3. 节点1：全局规划节点（Plan）
plan_prompt = ChatPromptTemplate.from_messages([
    ("system", "你是任务规划专家，把用户需求拆分成有序简洁执行步骤，只输出步骤列表，不要多余解释"),
    ("user", "需求：{user_query}，请拆分详细执行步骤")
])

def plan_node(state: PlanState) -> PlanState:
    query = state["user_query"]
    res = llm.invoke(plan_prompt.format(user_query=query))
    # 按行拆分步骤
    plans = [p.strip() for p in res.content.split("\n") if p.strip()]
    state["plan_list"] = plans
    state["current_step"] = 0
    state["step_result"] = []
    return state

# 4. 节点2：分步执行节点（Execute）
execute_prompt = ChatPromptTemplate.from_messages([
    ("system", "严格按照给定步骤完成任务，输出该步骤对应的执行结果即可"),
    ("user", "执行任务步骤：{step_content}")
])

def execute_node(state: PlanState) -> PlanState:
    idx = state["current_step"]
    step_content = state["plan_list"][idx]
    res = llm.invoke(execute_prompt.format(step_content=step_content))
    state["step_result"].append(res.content)
    state["current_step"] += 1
    return state

# 5. 节点3：汇总求解节点（Solve）
solve_prompt = ChatPromptTemplate.from_messages([
    ("system", "整合所有步骤结果，整理成通顺完整最终答案"),
    ("user", "所有执行结果：{all_results}，原始需求：{user_query}")
])

def solve_node(state: PlanState) -> PlanState:
    all_res = "\n".join(state["step_result"])
    final = llm.invoke(solve_prompt.format(all_results=all_res, user_query=state["user_query"]))
    state["final_answer"] = final.content
    return state

# 6. 路由判断：是否执行完所有步骤
def route_func(state: PlanState):
    if state["current_step"] >= len(state["plan_list"]):
        return "solve"
    return "execute"

# 7. 构建LangGraph流程图
builder = StateGraph(PlanState)
builder.add_node("plan", plan_node)
builder.add_node("execute", execute_node)
builder.add_node("solve", solve_node)

# 流程连线
builder.add_edge(START, "plan")
builder.add_conditional_edges("plan", route_func, {"execute":"execute", "solve":"solve"})
builder.add_conditional_edges("execute", route_func, {"execute":"execute", "solve":"solve"})
builder.add_edge("solve", END)

# 编译图
graph = builder.compile()

# 8. 测试运行
if __name__ == "__main__":
    input_data = {"user_query": "整理一份本周学习计划，包含早中晚安排，再写一段学习激励语"}
    result = graph.invoke(input_data)
    print("===== 生成执行计划 =====")
    for i,p in enumerate(result["plan_list"],1):
        print(f"{i}.{p}")
    print("\n===== 最终整合答案 =====")
    print(result["final_answer"])
```

### 4）Tree of Thoughts（ToT）——思维树（深度推理）
- 核心：**多分支推理 + 搜索 + 剪枝**，像树一样探索多条思路
- 流程：
  1. 分解问题为**思维步骤**
  2. 每步生成**多个候选**（分支）
  3. 评估分支，**剪枝坏路径**
  4. 递归深入，直到解出答案
- 优点：**复杂推理强、全局最优、减少局部最优陷阱**
- 缺点：**计算量大、实现复杂、需启发式评估**
- 场景：数学证明、逻辑谜题、战略决策、复杂代码

#### 一、核心定义

**ToT 思维树**
打破线性单一路径思考，把问题拆解成多层思维节点，**每层发散多条推理分支**，通过评估择优、淘汰劣路，逐层向下深度探索，最终找到全局最优解，是大模型**最强深度推理范式**。

#### 二、核心思想

把人类**多角度思考、试错筛选、排除错误思路**的思维过程，抽象成树形结构：
**分层拆解 → 多分支发散 → 打分评估 → 剪枝淘汰 → 深度递归求解**

#### 三、标准四步执行流程

1. **问题拆解**
将复杂难题拆分为**连续递进的思维子步骤**，确定思考层级。
2. **分支生成**
每一个思考节点，一次性生成**多条不同解题思路**，形成树分支。
3. **分支评估**
对所有分支思路做合理性、可行性、正确率打分排序。
4. **择优剪枝**
保留高分优质路径，砍掉错误、低效、走不通的劣质分支，继续向下递归推导，直至得出最终答案。

#### 四、运行逻辑类比

普通CoT思维链：**一条路走到黑**
ToT思维树：**同时走多条路，走不通立刻掉头，只走最优路线**

#### 五、通俗实战示例

需求：高难度奥数逻辑推理题
1. 拆分：分为审题、列式、推导、验算四层思维
2. 每层生成3种不同解题思路
3. 评判思路对错、难易程度
4. 删掉错误思路，沿着正确分支继续往下推
5. 多层筛选后得到唯一正确答案

#### 六、核心优点

1. **超强复杂逻辑推理能力**
专治多约束、多条件、高难度推理任务
2. **规避局部最优陷阱**
不会陷入单一错误思路死磕
3. **全局择优，答案准确率极高**
多路径验证，结果严谨可靠
4. **具备试错探索能力**
适合无固定解法、开放性难题
5. 推理过程透明，可追溯每一条思考路径

#### 七、明显缺点

1. **算力与Token消耗极大**
多分支并行思考，调用量远超其余范式
2. **实现难度最高**
需要设计评分机制、剪枝策略、递归调度
3. **推理速度最慢**
多层多分支筛选，耗时最长
4. **依赖强推理大模型**
小模型无法完成高质量分支生成与评估

#### 八、精准适用场景

1. 高等数学推导、奥数难题、数理证明
2. 复杂逻辑谜题、博弈推理、策略博弈
3. 商业战略规划、多方案决策研判
4. 复杂算法设计、多层逻辑代码编写
5. 多条件约束最优路径规划
6. 科学论证、严谨科研推理

#### 九、不适用场景

1. 日常简单问答、闲聊对话
2. 流程固定的流水线任务
3. 低成本大批量文案生成
4. 实时快速响应类业务

#### 十、四大智能体范式 终极汇总（面试必背）

| 范式             | 核心逻辑               | 核心口诀 | 定位             |
| ---------------- | ---------------------- | -------- | ---------------- |
| ReAct            | 思考→行动→观察循环     | 边想边做 | 通用工具交互首选 |

## 三、四大工程范式对比（速记）

| 范式 | 核心节奏 | 优势 | 劣势 | 最佳场景 |
