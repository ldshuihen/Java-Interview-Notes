### 1. 什么是 LangChain？核心定位是什么

**定义**
LangChain 是一款面向**大语言模型(LLM)** 的**应用开发与流程编排框架**，基于Python/Java开发，提供一套标准化、可复用的开发组件，用于快速构建基于大模型的各类智能应用。

**核心定位**
1. 作为**大模型中间层编排框架**，打通大模型与外部世界；
2. 统一封装模型调用、文档处理、向量检索、对话记忆、工具调用、业务流程；
3. 降低LLM应用工程化开发门槛，让开发者快速落地RAG知识库、智能Agent、自动化业务流程、多轮对话系统等；
4. 不训练大模型，只**编排大模型能力**。

### 2. LangChain 解决大模型哪些痛点

1. **知识时效性不足**
原生大模型知识存在时间截止点，无法获取实时数据、企业私有文档、内部业务资料，容易产生**幻觉**。
2. **无法联动外部资源**
大模型原生不能直接读取PDF/Word/Excel、查数据库、调用API、联网搜索、执行代码。
3. **多轮对话上下文难以管理**
原生调用需要手动拼接历史对话，容易超长溢出、上下文丢失、格式混乱。
4. **复杂业务流程无法自动化**
多步骤推理、条件判断、串行/并行任务无统一编排方式。
5. **缺少工程化通用能力**
缺少文本分片、向量化、相似度检索、结果解析、流式输出、异常重试等生产级能力。
6. **模型接入不统一**
兼容OpenAI、通义、文心、星火、本地开源大模型，统一调用接口，无缝切换模型。

### 3. LangChain 核心五大组件分别是什么

1. **Models 模型层**
包含LLM普通文本模型、ChatModel对话模型、Embeddings嵌入模型，负责文本生成与向量化。
2. **Prompts 提示词层**
统一管理提示词模板、角色设定、 Few-shot 示例，标准化输入格式。
3. **Memory 记忆层**
持久化/临时存储对话历史，实现多轮上下文连贯对话，解决记忆丢失问题。
4. **Chains 链路层**
把多个组件按顺序串联，形成固定执行流程，如文档问答链、摘要链、翻译链。
5. **Agents & Tools 智能代理与工具层**
让大模型自主思考、自主判断、自主选择调用外部工具，完成开放式复杂任务。

### 4. Chain 与 Agent 区别

#### Chain 链式流程

- **执行逻辑：固定流程**
  流程、步骤、顺序提前写死，人工定义先做什么再做什么。
- **决策方式：无自主决策**
  大模型只负责生成结果，**不能自主选择工具、不能改变流程**。
- **适用场景**
  流程确定、逻辑简单的标准化任务：RAG问答、文档总结、文本翻译、数据提取。
- **优点**：稳定、可控、易调试、响应快
- **缺点**：无法应对未知、多变场景

#### Agent 智能代理

- **执行逻辑：动态灵活**
  没有固定流程，由大模型**自主思考决策**下一步动作。
- **决策方式：自主推理+工具选择**
  模型判断是否需要调用工具、调用哪个工具、调用几次。
- **适用场景**
  开放式复杂任务：智能助手、数据分析、联网查询、代码执行、复杂业务推理。
- **优点**：通用性强，适配未知问题
- **缺点**：不可控、容易乱调用工具、耗时更长、结果不稳定

**一句话总结**
**Chain = 人定流程，模型执行；Agent = 模型定流程，自主执行。**

### 5. PromptTemplate、ChatPromptTemplate 用法区别

#### 1）PromptTemplate

- 面向**传统文本大模型 LLM**（非对话模型）
- 输入格式：**纯字符串**
- 结构简单，只有一段整体提示词
- 无角色区分，不分系统、用户、助手
- 多用于：文本生成、续写、摘要、简单指令

示例结构：
```
请根据内容回答问题：{context}
问题：{question}
```

#### 2）ChatPromptTemplate

- 面向**对话大模型 ChatModel**（主流GPT、通义、文心均为对话模型）
- 支持**多角色消息结构**
  - SystemMessage：系统角色设定、人设、规则
  - HumanMessage：用户提问
  - AIMessage：模型历史回答
- 严格贴合对话模型交互协议，**多轮对话首选**
- 支持批量拼接历史对话，格式最标准

**核心区别**
1. 适配模型不同：前者适配LLM，后者适配Chat对话模型
2. 结构不同：纯文本 VS 多角色消息列表
3. 业务场景：简单指令用前者，多轮对话、智能问答必用后者

### 6. 什么是 Runnables 流式链式调用

#### 基础概念

Runnables 是 LangChain 新版**统一可运行协议**，所有组件（模型、提示词、检索器、解析器）都统一实现 `Runnable` 接口。

#### 核心作用

1. **统一链式语法**
使用 `|` 管道符快速拼接流程，替代老旧Chain写法，代码更简洁。
```
prompt | model | output_parser
```
2. **统一调用方式**
同步调用、异步调用、批量调用、流式输出**一套接口通用**。
3. **流式链式调用含义**
整条链路**边执行、边输出**，不需要等全部结果生成完毕再返回。
- 从提示词 → 模型推理 → 结果输出全程流式推送
- 前端实现打字机实时回复效果
- 大幅提升用户体验，长文本场景必备

#### 核心优势

- 支持**断点、重试、回调、日志埋点**
- 天然支持异步并发，适合高并发对话服务
- 链路可拆分、可组合、可复用
- 是现在 LangChain 企业开发**主流标准写法**

**一句话总结**
Runnables 就是 LangChain 新一代统一组件规范，**流式链式调用**就是用管道符串联业务链路，实现全程实时流式输出响应。

## 二. 模型层 Model I/O

3. 温度值 temperature、top_p 作用

### 1. ChatModel 与 LLM 区别

#### LLM

1. 传统**文本补全模型**，输入纯字符串，输出纯字符串
2. 无角色概念，不分系统、用户、助手消息
3. 交互形式：单向文本续写，不贴合多轮对话场景
4. 调用方式简单，只传prompt文本
5. 适用：文案生成、摘要、续写、简单指令任务

#### ChatModel

1. **对话交互模型**，主流大模型均为该类型
2. 结构化消息传入：`SystemMessage`、`HumanMessage`、`AIMessage`
3. 原生支持多轮对话上下文格式，符合模型训练对话范式
4. 具备角色人设、对话记忆、对话逻辑能力
5. 适用：智能问答、多轮聊天、RAG对话、Agent交互

#### 核心总结

- **LLM**：纯文本生成，无对话结构
- **ChatModel**：结构化对话交互，企业项目**首选**

### 2. 如何实现本地大模型接入 LangChain

1. **部署本地大模型服务**
    用Ollama、FastChat、LLaMA Factory、vllm 启动模型，暴露本地HTTP接口
2. **使用LangChain兼容对接类**
    - Ollama本地模型：`langchain_ollama.ChatOllama`
    - 兼容OpenAI接口本地模型：`ChatOpenAI` 修改base_url指向本地地址
3. **配置模型参数**
    指定模型名称、本地接口地址、上下文长度、温度等
4. **正常融入流程**
    接入Prompt、Memory、Chain、RAG链路，和在线模型用法完全一致
5. **私有化部署**
    断开外网，本地完成嵌入模型+大模型+向量库全套闭环

### 3. 温度值 temperature、top_p 作用

#### temperature 温度

- 取值：**0 ~ 2**
- 作用：控制模型**随机性、创造性**
- 数值越小：输出越确定、严谨、重复度高，适合问答、知识库、专业答题
- 数值越大：发散性越强、创意越高，适合文案创作、脑洞写作
- 业务常用：RAG知识库设**0.1~0.3**，创作设**0.7~1.0**

#### top_p 核采样

- 取值：**0 ~ 1**
- 作用：按**概率累计**筛选候选词，截断低概率词汇
- 原理：只从累计概率达到top_p的词汇中抽样生成
- 越接近1：选词范围越大，越自由
- 越接近0：选词极度保守
- 一般和temperature搭配使用，控制输出风格一致性

### 4. 大模型上下文窗口限制如何处理

1. **文本预处理分片**
    长文档提前拆分chunk，只送入相关片段，不全量塞入上下文
2. **对话记忆压缩**
    使用`SummaryMemory`摘要总结历史对话，精简篇幅
3. **滑动窗口记忆**
    只保留最近N轮对话，淘汰早期历史
4. **上下文动态裁剪**
    实时统计token数量，超限自动截断最早内容
5. **多级检索召回**
    先粗召再精召、重排过滤，只把高相关内容放进prompt
6. **分层摘要策略**
    长文本先全局摘要+局部细节结合送入模型
7. **使用长上下文模型**
    直接替换更大窗口本地/商用大模型

### 5. 输出解析器 OutputParser 适用场景

#### 作用

统一把大模型**自由文本输出**，解析成**结构化数据**

#### 常用场景

1. **结构化数据提取**
    提取姓名、电话、地址、订单信息、简历字段
2. **固定格式输出**
    强制输出JSON、YAML、列表、枚举、键值对
3. **业务指令分类**
    把用户问题解析成指定枚举类别，用于路由分发
4. **参数抽取**
    抽取函数调用参数、工具入参、接口请求参数
5. **结果标准化**
    统一模型杂乱回答，方便后端代码判断、存储、入库
6. **Agent工具调用格式化**
    规范思考内容+工具名+入参格式，避免调用异常

#### 常用解析器

- StrOutputParser 普通文本解析
- JsonOutputParser JSON结构化解析
- PydanticOutputParser 实体类强类型解析
- CommaSeparatedListOutputParser 列表拆分解析

## 三. 记忆 Memory

1. LangChain 有哪些常用 Memory
2. ConversationBufferMemory / SummaryMemory / VectorStoreRetrieverMemory 区别
3. 会话记忆如何持久化
4. 长对话记忆溢出解决方案
5. 实体记忆 EntityMemory 原理

### 1. LangChain 有哪些常用 Memory

1. **ConversationBufferMemory**：原始缓存记忆，完整存储所有对话
2. **ConversationSummaryMemory**：对话摘要记忆，压缩历史对话
3. **ConversationSummaryBufferMemory**：摘要+缓存结合，兼顾精简与完整
4. **ConversationTokenBufferMemory**：按Token数量限制记忆长度
5. **VectorStoreRetrieverMemory**：向量检索记忆，长历史语义召回
6. **EntityMemory**：实体记忆，抽取人物、事物、关系长期留存
7. **RedisChatMessageHistory**：分布式Redis持久化会话记忆
8. **CombinedMemory**：组合多种记忆混合使用

### 2. ConversationBufferMemory / SummaryMemory / VectorStoreRetrieverMemory 区别

#### 1）ConversationBufferMemory 缓存记忆

- 存储**完整原始对话**，不做任何压缩
- 优点：信息最全、无丢失、逻辑连贯
- 缺点：对话越长Token越多，极易超限
- 适用：短轮次、轻量对话场景

#### 2）ConversationSummaryMemory 摘要记忆

- 不存原文，调用大模型**把历史总结成简短摘要**
- 不断迭代更新摘要，大幅节省Token
- 优点：极致压缩，占用上下文极少
- 缺点：细节丢失过多，容易遗忘细粒度信息
- 适用：超长连续闲聊、日常多轮聊天

#### 3）VectorStoreRetrieverMemory 向量检索记忆

- 将历史对话向量化存入向量库
- 新一轮提问时**语义相似度召回相关历史**
- 不把全部历史塞进上下文，只拿相关片段
- 优点：彻底突破窗口限制，支持万轮级对话
- 缺点：依赖向量库，存在召回不准问题
- 适用：企业知识库、长期用户会话、历史业务溯源

**一句话区分**
- Buffer：全量存
- Summary：精简总结存
- VectorRetriever：向量语义按需取

### 3. 会话记忆如何持久化

1. **文件持久化**
用`FileChatMessageHistory`，将对话写入本地JSON文件
2. **数据库持久化**
MySQL/SQLite 存储会话ID+角色+对话内容
3. **缓存中间件（生产常用）**
`RedisChatMessageHistory`，按`session_id`隔离会话，支持分布式部署
4. **向量库持久化**
把对话历史向量化存入Milvus、FAISS、Chroma，实现长期语义记忆
5. **自定义持久化**
继承`BaseChatMessageHistory`，对接业务自有用户会话表

**核心思路**：以`session_id`作为唯一标识，隔离不同用户对话记录。

### 4. 长对话记忆溢出解决方案

1. **固定窗口截断**：只保留最近N轮对话，删除最早记录
2. **Token阈值截断**：超过设定Token自动裁剪头部历史
3. **摘要压缩**：用SummaryMemory合并精简过往对话
4. **分层记忆**：近期完整保存，远期统一摘要
5. **向量检索隔离**：超长历史存入向量库，仅召回相关内容进上下文
6. **分块会话**：按时间/场景拆分会话，分段记忆
7. **动态Prompt瘦身**：剔除无用话术，精简系统提示词
8. **切换大模型**：改用更大上下文窗口模型



### 5. 实体记忆 EntityMemory 原理

1. **核心思想**
不从整体保存对话，而是**抽取对话里关键实体与关系**长期记忆。

2. **执行流程**
- 大模型自动识别对话中**人物、地点、时间、喜好、习惯、事件**等实体
- 提取实体属性、相互关系存入记忆库
- 新一轮对话时，自动读取实体信息融入问答

3. **特点**
- 遗忘表层对话话术，记住**用户核心信息**
- 跨轮次、跨会话长期记住用户偏好
- 极大节省上下文Token
- 适合私人助理、陪伴聊天、个性化智能客服

4. **使用场景**
记住用户姓名、年龄、喜好、日程、过往需求，实现千人千面个性化应答。

## 四. 文档加载&切片 Document

1. 常用文档加载器：PDF/Word/网页/Excel
2. RecursiveCharacterTextSplitter 分片原理
3. 分片大小、重叠块 chunk_overlap 作用
4. 语义分片 vs 字符分片优缺点
5. 文档预处理清洗方案

### 1. 常用文档加载器

1. **PDF**
`PyPDFLoader`、`PDFPlumberLoader`，读取PDF文本内容，支持分页读取
2. **Word**
`Docx2txtLoader`、`UnstructuredWordDocumentLoader`，解析docx文档
3. **网页**
`WebBaseLoader` 爬取网页正文；`SitemapLoader` 批量爬取站点
4. **Excel**
`UnstructuredExcelLoader`、`CSVLoader` 读取表格数据
5. 其他常用
- `TextLoader`：纯txt文本
- `MarkdownLoader`：md文档
- `DirectoryLoader`：遍历文件夹批量加载所有文件
- `JSONLoader`：读取结构化JSON文档

### 2. RecursiveCharacterTextSplitter 递归字符切片原理

1. **优先级分割**
按照**优先级从大到小**依次尝试分割符拆分文本
分割顺序：`\n\n`(段落) → `\n`(换行) → `.` → `,` → 空格 → 单个字符
2. **递归逻辑**
- 先按最高级分隔符切
- 切完片段仍**大于设定chunk_size**，就用下一级更细分隔符继续切
- 直到所有片段长度都符合限制为止
3. **核心优势**
尽量**保证语义完整性**，优先不拆散完整段落，是RAG项目最通用分片器

### 3. chunk_size 分片大小、chunk_overlap 重叠块作用

#### chunk_size 分片大小

- 含义：单个文本块最大字符/Token长度
- 作用：**适配大模型输入上下文**，防止片段过长超限
- 经验值：通用知识库 **500~1500**

#### chunk_overlap 重叠块

- 含义：相邻两个切片**首尾重复的文本长度**
- 作用：
  1. 避免**语义被拆分断裂**，上下文信息断层
  2. 解决跨片段问答丢信息问题
  3. 提升检索召回完整性
- 经验值：设置为 chunk_size 的 **10%~20%**

### 4. 语义分片 vs 字符分片 优缺点

#### 1）字符分片（递归/固定长度）

**优点**
- 速度快、资源消耗低
- 配置简单、稳定不出错
- 适合大批量文档快速处理
**缺点**
- 容易强行截断句子、打断语义
- 不理解文本逻辑结构，分割生硬

#### 2）语义分片

**原理**：通过嵌入模型计算句子相似度，语义相近划为一块
**优点**
- 严格按照**语义意图**切分，段落逻辑完整
- RAG检索精度更高，问答准确率更好
**缺点**
- 速度慢，需要大量向量计算
- 消耗算力与Token，成本更高
- 长文本处理耗时明显增加

**选型**
- 批量海量文档：字符分片
- 高精度知识库、专业资料：语义分片

### 5. 文档预处理清洗方案

1. **空白清理**
去除多余空行、空格、制表符、换行符
2. **格式过滤**
剔除页眉、页脚、页码、目录、水印、广告无关内容
3. **乱码&特殊字符过滤**

## 五. 向量库&检索 RAG 核心

1. LangChain 支持哪些主流向量库
2. 相似度检索、MMR 多路召回区别
3. 混合检索 Hybrid Search 实现
4. RAG 检索召回不足如何优化
5. 元数据过滤检索实战场景

### 1. LangChain 支持哪些主流向量库

LangChain 几乎兼容所有主流向量数据库，分为**轻量本地版**和**生产分布式版**：

#### 本地轻量型（开发/小数据）

1. **FAISS**：Facebook 开源，速度极快，内存运行，无需服务
2. **Chroma**：轻量开箱即用，调试最方便，适合快速开发
3. **Pinecone**：云端托管，简单易用
4. **FAISS-CPU**：单机百万级数据无压力

#### 生产分布式（企业/海量数据）

1. **Milvus / Zilliz**：国内最主流生产向量库，高可用、分片、索引优化
2. **ElasticSearch**：支持文本检索+向量检索混合查询
3. **Redis**：基于 Redis 向量扩展，速度超快
4. **PGVector**：基于 PostgreSQL 插件，关系库+向量一体
5. **Weaviate**：支持混合检索、实体链接

#### 一句话总结

**开发用 FAISS/Chroma，生产用 Milvus/PGVector/ElasticSearch。**

---

### 2. 相似度检索、MMR 多路召回区别

#### 1）普通相似度检索（Similarity Search）

- **原理**：只看**向量距离**，返回与问题最相似的 topK 条
- **优点**：简单、速度快、精度高
- **缺点**：容易返回**内容高度重复、多样性差**的片段
- **适用**：标准问答、精准检索

#### 2）MMR 最大边际相关性（Maximal Marginal Relevance）

- **原理**：兼顾**相似度 + 多样性**
  1. 先召回一批相似文档
  2. 自动剔除重复、相近内容
  3. 保留最相关 + 最多样的片段
- **优点**：信息更全面、不重复、覆盖度更高
- **缺点**：速度略慢、多一步计算
- **适用**：长文本、综述类问题、需要全面信息的场景

#### 核心区别

- **相似度检索**：追求最像 → 可能重复
- **MMR**：追求像 + 不重复 → 信息最全

---

### 3. 混合检索 Hybrid Search 实现

#### 定义

**混合检索 = 关键词检索（BM25） + 向量检索（语义）**
结合两种检索优势，既懂字面匹配，又懂语义理解。

#### 实现步骤

1. **构建两路检索器**
   - 一路：BM25 关键词检索（字面匹配）
   - 一路：Vector 向量检索（语义匹配）
2. **两路分别召回结果**
3. **融合排序（RRF 召回排序融合）**
   自动给两路结果加权、去重、重新排序
4. **输出最终最优片段**

#### 优点

- 解决**语义检索查不到关键词**问题
- 解决**关键词检索不懂语义**问题
- 企业级 RAG **标配方案**

---

### 4. RAG 检索召回不足如何优化（高频大题）

#### 1）检索前优化（分片层）

- 调整 chunk_size，不要太大/太小
- 增加 chunk_overlap 避免语义断裂
- 使用**语义分片**替代生硬字符分片
- 清洗文档，剔除无效内容

#### 2）检索中优化（向量层）

- 换**更好的嵌入模型**（中英文专用、领域专用）
- 使用 **混合检索**（关键词+向量）
- 使用 **MMR** 提升多样性
- 增加召回数量 topK（5→10→15）

#### 3）检索后优化（后处理）

- 使用 **Rerank 重排模型** 精排
- 过滤低相似度垃圾片段
- 动态拼接上下文，保留最相关信息

#### 4）查询层优化

- **问题改写/扩写**：让问题更清晰
- 复杂问题**拆分子问题**分别检索

### 5. 元数据过滤检索实战场景

#### 元数据 = 给文档打标签（作者、时间、部门、文件名、类型等）

#### 核心作用：**先过滤，再检索**，大幅提升精度与速度

#### 高频实战场景

1. **按部门/权限过滤**
   技术部员工只能查技术文档，不能查财务文档
2. **按时间范围检索**
   “2025年的政策” → 只查2025年的切片
3. **按文档类型过滤**
   只查“合同”、“手册”、“报告”
4. **按产品/项目过滤**
   问A产品 → 只检索A产品资料
5. **按文件名/章节过滤**
   精准定位某一章、某一节

#### 优势

- 缩小检索范围，**速度更快**
- 避免无关信息干扰，**准确率极高**
- 企业级系统**权限控制必备**

#### 一句话总结

**元数据过滤 = 给检索加“精准筛选条件”。**

### 1. Stuff / MapReduce / Refine 三种文档链区别

这三种是 LangChain 处理**多篇/超长文档**时，将上下文喂给大模型的**合并策略**。

#### 1）Stuff (简单填充)

- **原理**：把所有检索到的文档**直接拼接**成一个大 Prompt，一次性喂给模型。
- **优点**：简单、快速、模型能一次性看到全局信息，效果最好。
- **缺点**：文档一多，**立刻超出上下文窗口**，报错。
- **适用场景**：简单问答、小数据量、chunk很小的场景。**（最常用）**

#### 2) MapReduce (分而治之)

- **原理**：分两步。
  1. **Map**：对**每一篇**文档单独调用模型，生成**中间答案/摘要**。
  2. **Reduce**：把所有中间结果汇总，再调用模型生成**最终答案**。
- **优点**：能处理**超大量文档**，突破长度限制。
- **缺点**：速度慢（调用多次API），丢失细节，逻辑复杂。
- **适用场景**：**海量文档总结**、书籍、长篇报告。

#### 3) Refine (迭代优化)

- **原理**：**流水线**作业。
  1. 拿第一个文档生成答案。
  2. 拿着**上一次的答案** + **下一个文档**，让模型迭代**优化、补充、修正**答案。
- **优点**：适合**时序/递进**类信息，答案更精准、连贯。
- **缺点**：极慢（串行调用），依赖上下文传递。
- **适用场景**：逐步完善的写作、多文献综述。

#### 面试必背总结

- **Stuff**: 简单快 -> **短文本首选**
- **MapReduce**: 并行处理 -> **海量总结**
- **Refine**: 迭代优化 -> **精细写作**

---

### 2. RetrievalQA 底层执行流程

`RetrievalQA` 是**最标准的RAG问答链**，面试常让口述流程。

#### 完整执行流程（5步）：

1. **用户提问**：用户输入问题。
2. **检索文档**：Chain 将问题传给**检索器 (Retriever)**，去向量库搜索相关的 `topK` 个文档片段。
3. **文档合并**：使用 `Stuff/MapReduce/Refine` 策略，将检索到的**文档片段**和**用户问题**填充到预设的 Prompt 模板中。
4. **模型推理**：将构造好的 Prompt 发送给 LLM/ChatModel 进行推理。
5. **返回结果**：模型根据提供的**上下文（Context）** 生成答案，返回给用户。

#### 核心逻辑

**RetrievalQA = 检索器 + 文档合并链 + 大模型**
**口诀**：一问、二搜、三拼、四推、五回。

---

### 3. SequentialChain 串行多任务调用

用于处理**多步骤、流水线式**业务，上一步的输出是下一步的输入。

#### 核心概念

- 将多个独立的 `Chain` (如：翻译链->总结链->润色链) 串起来。
- 数据像流水线一样自动流转。
- 必须指定 `input_variables` 和 `output_variables`。

#### 典型场景

1. 提取摘要 -> 翻译成英文 -> 生成标题
2. 分析评论 -> 提取观点 -> 生成报告

#### 特点

- 流程**固定**。
- 必须按顺序执行。
- 适合**确定性**的复杂任务。

---

### 4. 自定义 Chain 实现步骤

面试**手写代码**高频考点。

#### 实现步骤（基于新版 Runnable）：

1. **定义函数**：编写你的业务逻辑函数。
2. **包装成 Runnable**：使用 `@tool` 装饰器 或 `RunnableLambda` 将普通函数变成 Chain 节点。
3. **构建链路**：使用管道符 `|` 串联 **Prompt | Model | Parser | 自定义函数**。
4. **调用执行**：使用 `.invoke()` 方法运行。

#### 极简代码示例（背这个）：

```python
from langchain_core.runnables import RunnableLambda
from langchain_openai import ChatOpenAI

# 1. 自定义函数: 大写转换
def uppercase_func(text: str) -> str:
    return text.upper()

# 2. 包装成 Chain 节点
upper_chain = RunnableLambda(uppercase_func)

# 3. 主模型链
model = ChatOpenAI()

# 4. 组装完整自定义链路
# 流程: 提问 -> 模型回答 -> 转成大写
full_chain = model | upper_chain

# 5. 调用
result = full_chain.invoke("你好")
print(result)
```

### 5. 异步 Chain 调用写法

生产环境**高并发**必备，提升接口吞吐量。

#### 核心方法

- 同步：`.invoke()`
- **异步**：`.ainvoke()`

#### 写法要点：

1. 函数前加 `async`。
2. 调用时加 `await`。
3. 所有组件（Model, Retriever）都支持异步。

#### 代码模板：

```python
# 定义异步函数
async def async_qa(question):
    # 注意: 使用 ainvoke
    response = await qa_chain.ainvoke(question)
    return response

# 在主程序中运行
import asyncio
result = asyncio.run(async_qa("什么是RAG?"))
```

#### 优势

- 非阻塞IO，接口能同时处理更多请求。
- 适合Web服务（FastAPI + LangChain）部署。

## 七. Agent 智能代理

1. Agent 运行整体流程
2. ReAct 代理原理

5. Agent 思考日志 Trace 如何查看

### 1. Agent 运行整体流程

1. **接收用户问题**，带入对话上下文
2. **思考决策**：大模型自主判断当前问题是否需要调用工具
3. **选择工具**：确定调用哪个工具、拼装入参
4. **执行工具**：发起接口/函数调用，拿到外部结果
5. **整合信息**：把工具返回数据+原有信息汇总
6. **循环判断**：若信息不足，继续选工具调用；信息充足则停止
7. **生成最终答案**，整理自然语言回复用户

**核心闭环**：思考→行动→观察→再思考

### 2. ReAct 代理原理

- 全称：**Reason + Act** 思考+行动范式
- 核心思想：让模型把**内心思考过程**明文写出来，再决定执行动作
- 执行逻辑：
  1. Thought：分析用户需求，梳理解题思路
  2. Action：选定具体工具与入参
  3. Action Input：填写调用参数
  4. Observation：接收工具返回结果
  5. 循环往复，直到可直接给出答案
- 优势：逻辑透明、易于调试，是目前工业界最主流Agent实现范式
- 适用：通用工具调用、联网查询、数据查询、代码执行等场景

### 3. ZeroShotAgent / StructuredToolAgent 区别

#### ZeroShotAgent 零样本代理

- 无需额外示例，零学习成本直接使用
- 仅支持**单字符串**格式入参，不能传结构化多参数
- 工具传参简单，复杂参数容易解析出错
- 轻量化、上手快，适合简单单参数工具调用

#### StructuredToolAgent 结构化工具代理

- 支持**JSON结构化入参**，可定义多个不同类型参数
- 严格校验参数名、参数类型、是否必填
- 适配企业复杂业务工具，入参规范不易出错
- 兼容性更强，生产环境业务开发首选

**总结**
简单工具用 ZeroShot，多参数复杂业务工具必用 StructuredToolAgent

### 4. 自定义工具 Tool 开发流程

1. **导入基础依赖**：从`langchain_core.tools`引入`tool`装饰器
2. **编写业务功能函数**，定义入参、返回值、业务逻辑
3. **使用@tool装饰器**注册为Agent可识别工具
4. **填写工具名称、功能描述**（描述决定模型会不会调用该工具）
5. **统一规范入参类型**，便于结构化解析
6. **加入工具列表**，传入Agent初始化配置
7. **测试调用**，验证模型能否自动识别并正确传参调用

**极简模板**
```python
from langchain_core.tools import tool

@tool("天气查询工具")
def get_weather(city: str) -> str:
    """用于查询指定城市实时天气，入参为城市名称"""
    return f"{city}今日天气晴朗"
```

### 5. Agent 思考日志 Trace 如何查看

1. **开启LangSmith追踪**（企业主流）
配置环境变量`LANGSMITH_API_KEY`，自动推送全链路日志，网页端可视化查看思考、选工具、调用记录
2. **控制台原生日志**
设置日志级别为INFO/DEBUG，直接打印Thought、Action、Observation完整流程
3. **回调Callback监听**
自定义`AgentCallbackHandler`，拦截思考过程、工具调用前后事件，本地存储日志
4. **内置返回字段**
部分Agent返回结果中自带`intermediate_steps`字段，可直接读取每一步思考与行动记录

### 6. 如何限制 Agent 乱调用工具

1. **精简工具描述**
工具功能描述精准收敛，不写多余能力，避免模型误解用途
2. **限定可用工具白名单**
初始化Agent时只注入当前业务需要的工具，屏蔽无关工具
3. **设置最大迭代次数**
配置`max_iterations`，限制最大工具调用轮次，防止死循环调用
4. **添加强制终止提示词**
在系统prompt中明确规定：信息足够禁止继续调用工具，优先直接回答
5. **参数强校验**
使用结构化工具，对入参做格式、范围校验，错误参数直接拦截
6. **结果阈值判断**
工具返回结果满足条件直接终止流程，不再继续探索
7. **路由分层管控**
简单问题直接走对话链路，复杂问题再进入Agent工具流程

### 1. LangChain 流式输出实现

#### 核心原理

让模型**边生成边返回**，不用等全部结果，前端实现打字机效果。

#### 实现步骤

1. 模型开启 `streaming=True`
2. 使用回调函数 `StreamingStdOutCallbackHandler` 或自定义回调
3. 后端接口必须支持**流式响应**（FastAPI 使用 `StreamingResponse`）
4. 前端通过 EventStream / Fetch 读取流

#### 核心代码（面试必背）

```python
# 1. 模型开启流式
llm = ChatOpenAI(streaming=True, callbacks=[StreamingStdOutCallbackHandler()])

# 2. 构建 chain
chain = prompt | llm | StrOutputParser()

# 3. 流式调用
for chunk in chain.stream("介绍下AI"):
    print(chunk, end="", flush=True)
```

#### 关键点

- 新版 LangChain 使用 `chain.stream()` 极简实现
- 所有组件（Prompt、Parser、Retriever）天然支持流式
- 生产部署必须用流式，大幅提升用户体验

### 2. 对话上下文隔离方案

#### 核心目标

**不同用户、不同会话**互不干扰，数据不串、历史不乱。

#### 实现方案

1. **Session ID 唯一标识**
   每个用户会话分配唯一 `session_id`
2. **Memory 绑定 Session**
   使用 `ConversationBufferMemory` 并指定 `session_id`
3. **持久化存储隔离**
   - Redis：`key = session_id` 存储对话历史
   - 数据库：按 `session_id` 分表/分字段存储
4. **多线程/分布式隔离**
   每次请求从存储中**加载对应会话历史**，用完即释放
5. **无状态设计**
   服务不存内存，完全依赖外部存储（Redis/PG）

#### 总结

**SessionID + 外部存储 = 生产级上下文隔离**

### 3. 生产环境异常捕获与重试

#### 常见异常

- 模型接口超时
- 网络波动
- 上下文超限
- 向量库连接失败
- Token 过期/限流

#### 解决方案

1. **全局 try-except 捕获**
   捕获LLM异常、网络异常、业务异常，返回友好提示
2. **自动重试机制**
   使用 `tenacity` 库重试超时、限流错误
   ```python
   @retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
   def call_llm():
       return llm.invoke(prompt)
   ```
3. **降级方案**
   重试失败 → 返回兜底文案（“服务繁忙，请稍后再试”）
4. **日志记录**
   记录入参、错误信息、耗时，便于排查问题

---

### 4. 大模型接口限流、熔断处理

#### 限流（控制请求速度）

1. **令牌桶/漏桶算法**
   控制每秒请求数 QPS
2. **队列缓冲**
   请求先进队列，匀速消费
3. **接口级别限流**
   按用户ID、IP、接口路径限流

#### 熔断（防止雪崩）

1. **失败率达到阈值 → 切断调用**
2. 直接返回降级结果，保护下游服务
3. 半开探测：恢复后少量请求试探

#### 生产工具

- Python：`limits`、`pybreaker`
- 网关：Nginx、Gateway 限流

---

### 5. LangChain 性能优化方向

#### 1）检索速度优化

- 向量库构建索引（IVF_FLAT、HNSW）
- 混合检索提前过滤（元数据过滤）
- 缓存高频问答结果（Redis 缓存）

#### 2）模型调用优化

- 异步调用 `ainvoke` 提升并发
- 流式输出减少首屏响应时间
- 复用长连接，减少握手开销

#### 3）上下文瘦身

- 精简系统提示词
- 摘要记忆替代全量记忆
- 检索只返回高相关片段

#### 4）架构优化

- 检索与推理服务分离
- 向量库分布式部署
- 无状态服务支持水平扩容

#### 5）资源控制

- 限制最大召回条数
- 限制 Agent 最大迭代次数
- 超时主动中断

1. **流式输出**：`streaming=True` + `chain.stream()`

# LangGraph 面试题

## 一. 基础认知

1. LangGraph 是什么？和 LangChain 关系
2. 为什么需要 LangGraph，Chain 满足不了什么场景
3. 状态图架构核心思想
4. 适用业务场景：多智能体、复杂流程、循环决策

### 1. LangGraph 是什么？和 LangChain 关系

#### 定义

LangGraph 是**基于 LangChain 生态构建的状态编排框架**，专门用来搭建**带状态、可循环、可分支、可中断**的复杂大模型应用，以**状态图**为核心驱动执行流程。

#### 二者关系

1. **从属关系**
LangGraph **完全兼容 LangChain 所有组件**：Prompt、Model、Memory、Retriever、Tool、Runnable 都能直接复用，不用重新学习新语法。
2. **定位区别**
- LangChain：偏向**线性链式编排**，适合串行固定流程
- LangGraph：偏向**图式流程编排**，适合非线性复杂流程
3. **生态关系**
LangGraph 是 LangChain 官方推出的**高阶流程扩展框架**，属于同一套技术体系，无缝互通。

#### 一句话总结

**LangChain 做线性流程，LangGraph 在它基础上做有状态复杂流程图。**

---

### 2. 为什么需要 LangGraph，Chain 满足不了什么场景

#### 传统 Chain 存在的短板

1. **只能单向线性执行**
Chain 只能 `A→B→C` 顺序跑，**不能回头、不能循环**，无法实现“思考失败重新检索、多轮自查纠错”。
2. **无全局统一状态**
Chain 组件之间数据传递零散，无法全局统一管理会话数据、中间结果、全局变量。
3. **不支持条件分支路由**
无法根据模型输出结果**动态选择下一步走哪个分支**，业务逻辑写死不灵活。
4. **不支持流程暂停与人机介入**
Chain 一旦启动无法中途暂停，不能实现**人工审核、人工确认、人机交互**。
5. **难以实现多智能体协作**
无法优雅实现角色分工、任务派发、结果汇总、智能体之间消息流转。
6. **缺少闭环迭代能力**
不能实现「执行→判断→修正→重新执行」的自主迭代闭环。

#### 结论

简单RAG、单轮问答用**LangChain Chain**足够；
**需要循环、分支、状态、暂停、多Agent**的复杂业务，必须用 LangGraph。

---

### 3. 状态图架构核心思想

#### 核心三大要素：状态 + 节点 + 边

1. **统一全局 State 状态**
整个流程共用一个**结构化状态字典/实体类**，存放所有全局数据：用户问题、检索上下文、历史对话、中间结果、决策指令、工具返回值等。
所有节点**只读/修改同一个状态**，数据全局互通。

2. **Node 节点**
每一个独立业务逻辑封装成一个节点：检索节点、大模型思考节点、工具调用节点、总结节点、判断节点。
节点只负责**处理逻辑、修改状态**，不关心流程走向。

3. **Edge 边（流程连线）**
- **普通边**：固定走向，A节点执行完固定走到B节点
- **条件边**：根据**状态里的结果**动态判断下一步去往哪个节点

#### 整体运行思想

**流程不写死，状态驱动流转**
读取状态 → 节点执行业务 → 更新状态 → 条件判断 → 流向对应节点 → 循环直到结束。

#### 极简流程

初始化State → 进入起始节点 → 处理逻辑更新State → 条件路由跳转 → 直至终止节点。

---

### 4. 适用业务场景

#### 1）复杂RAG迭代问答

- 初次检索信息不足 → 自动改写问题二次检索
- 答案不准确 → 重新分片、重新召回
- 实现**自我纠错、自我补全**的高级RAG

#### 2）循环决策类任务

- 代码调试：编写代码→运行报错→分析错误→重新编写
- 数据分析：查询数据→分析结果→发现缺失→继续查询
- 自主规划：制定方案→校验可行性→修改方案

#### 3）人机交互中断流程

- 智能审批：模型生成审批意见 → 暂停流程 → 人工审核确认 → 继续往下执行
- 高危操作二次确认、信息补全弹窗交互

#### 4）多智能体 Multi-Agent 场景（最主流）

- 主管Agent：下发任务、统筹调度、汇总结果
- 执行Agent：检索、查资料、调用工具、执行具体任务
- 评审Agent：校验答案正确性、过滤错误信息
利用状态完成**任务分发、状态同步、结果收拢**。

#### 5）长链路复杂业务编排

智能客服工单流转、办公自动化流程、多步骤业务审批、知识库深度调研、科研文献梳理等长流程系统。

#### 6）分层路由问答

根据用户问题类型，自动分流：闲聊走对话节点、知识问答走RAG节点、业务查询走工具节点。



## 二. 核心架构

1. State 状态结构作用
2. Node 节点、Edge 边、ConditionalEdge 条件边
3. 入口节点、终止节点定义
4. 状态传递与状态合并规则

### 1. State 状态结构作用

#### 定义

State 是**全局统一数据容器**，用 TypedDict / Pydantic 定义结构化字段，贯穿整个图流程。

#### 核心作用

1. **统一数据中心**
所有节点共享同一份状态，代替零散传参，全局数据统一管理。
2. **流程数据持久载体**
存储用户提问、对话历史、检索上下文、模型思考结果、工具返回数据、中间变量、路由标识等。
3. **驱动流程流转**
条件边依据 State 内字段值判断走向，**状态决定下一步走哪条路**。
4. **断点续跑基础**
保存完整流程快照，暂停后可依据 State 恢复现场继续执行。
5. **多节点数据互通**
不同业务节点无需互相耦合，只读写 State 即可完成数据交互。
6. **规范入参出参**
强类型约束字段格式，减少参数混乱、字段错写问题。

#### 常用内置字段

question、chat_history、context、answer、tools_result、next_node 等。

### 2. Node 节点、Edge 边、ConditionalEdge 条件边

#### 1）Node 节点

- 是**最小执行逻辑单元**，封装单一业务功能。
- 只做两件事：读取State数据 → 执行业务逻辑 → 修改/写入新State。
- 无流程控制权，只负责处理业务，不决定走向。
- 常见节点：思考节点、检索节点、工具调用节点、总结节点、判断节点。

#### 2）Edge 普通边

- **固定流向连线**，写死节点执行顺序。
- A节点执行完毕，**无条件直接跳转到指定B节点**。
- 适用：固定串行流程、顺序不可变链路。

#### 3）ConditionalEdge 条件边

- **动态路由边**，根据自定义判断函数结果分流。
- 判断函数读取State字段，返回不同路由标识。
- 匹配标识自动跳转到对应分支节点。
- 支持多分支、循环跳转、分流执行。
- 是实现复杂逻辑、循环、分支的**核心**。

**三者关系**
节点负责干活，普通边负责固定走路，条件边负责智能选路。

### 3. 入口节点、终止节点定义

#### 入口节点 EntryPoint

- 整张状态图**流程启动的首个执行节点**。
- 流程初始化完成后，第一个进入执行的Node。
- 一般用于初始化状态、预处理用户输入、统一请求格式化。
- 一张图**有且仅有一个入口节点**。

#### 终止节点 EndPoint

- 标识**整个业务流程结束**的终点节点。
- 执行到此节点后，不再继续跳转，直接输出最终State结果。
- 所有分支链路最终都可汇聚到终止节点。
- 作用：统一结果整理、格式化输出、清空临时状态。

**使用规范**
所有循环、分支逻辑必须设置终止节点，避免流程死循环。

### 4. 状态传递与状态合并规则

#### 1）状态传递规则

1. 流程启动：初始化空State并赋值初始数据（用户问题、会话信息）。
2. 节点执行：接收**完整父节点State副本**，在副本上做业务修改。
3. 向下传递：处理完成后，将**更新后的完整State**传递给下一个节点。
4. 全程单向数据流：状态自上而下流转，不会逆向篡改历史上层数据。

#### 2）状态合并规则（LangGraph 默认合并策略）

1. **同字段覆盖原则**
后写入的同名字段，直接**覆盖**State中原旧值。
2. **列表类字段追加合并**
数组、对话历史、上下文列表默认**尾部追加**，不覆盖原有数据。
3. **自定义合并器**
可重写 `reducer` 函数，自定义合并逻辑：
- 头部插入、中间插入、去重合并、条件合并、字段过滤。
4. **局部更新**
节点只需修改变更字段，无需重写整个State，未修改字段自动保留原值。
5. **多并行节点合并**
多个并行节点执行完成后，按预设合并策略统一汇总到主State。

#### 核心总结

传递：全量State逐层下发；
合并：普通字段覆盖，序列字段追加，复杂场景自定义reducer。

1. 串行流程、分支流程、循环流程实现
2. 人机交互中断 Human-in-the-loop 原理
3. 流程断点续跑、状态持久化
4. 并行节点执行写法
5. 条件路由判断逻辑设计

---

### 1. 串行流程、分支流程、循环流程实现

#### 1）串行流程（最基础）

**定义**：节点按固定顺序一步一步执行，A→B→C→结束。
**实现**：直接用**普通边（Edge）**连接。
**适用**：简单RAG、数据清洗、固定流水线。
```python
# 代码写法
workflow.add_edge("node_a", "node_b")
workflow.add_edge("node_b", "node_c")
workflow.add_edge("node_c", END)
```

#### 2）分支流程（条件判断）

**定义**：根据状态值，走不同链路（if/else）。
**实现**：使用**条件边（Conditional Edge）**。
**适用**：问题分类、路由分发、权限判断。
```python
# 判断函数：读取状态，决定走哪条路
def route_question(state):
    if state["type"] == "chat":
        return "chat_node"
    else:
        return "rag_node"

# 绑定条件边
workflow.add_conditional_edges(
    "classify_node",  # 从哪个节点出发
    route_question,   # 判断函数
    {                 # 映射：返回值 → 目标节点
        "chat_node": "chat_node",
        "rag_node": "rag_node"
    }
)
```

#### 3）循环流程（核心能力）

**定义**：满足条件就**回到之前节点重复执行**，不满足则退出。
**实现**：条件边指回上游节点。
**适用**：Agent思考、工具重试、答案自检、代码调试。
```python
# 判断是否需要继续循环
def should_continue(state):
    if state["need_retry"]:
        return "retry_node"  # 回到前面重新执行
    else:
        return "end_node"    # 结束

# 循环实现：条件边指回上游
workflow.add_conditional_edges(
    "check_node",
    should_continue,
    {
        "retry_node": "think_node",  # 循环回去
        "end_node": END
    }
)
```

---

### 2. 人机交互中断 Human-in-the-loop 原理

#### 核心定义

流程执行到某节点时**主动暂停**，等待**人工输入/审核/确认**后，再继续执行。

#### 实现原理

1. **中断节点（Interrupt）**
流程遇到 `interrupt=True` 节点时自动暂停。
2. **状态快照保存**
暂停时，**完整 State 状态被保存**（上下文、中间结果、进度）。
3. **人工介入**
人查看状态 → 修改数据 → 审核通过/驳回 → 输入指令。
4. **恢复执行**
调用 `resume()` 方法，从断点继续往下跑。

#### 典型业务场景

- 合同生成 → 暂停 → 人工审核 → 再发送
- 财务审批 → AI初审 → 人工确认 → 自动入账
- 高危操作（删库、发钱、发通知）必须人工确认

#### 核心价值

**AI 负责干活，人负责把关，安全可控。**

---

### 3. 流程断点续跑、状态持久化

#### 1）断点续跑

- 流程可在**任意节点暂停**。
- 保存当前 State 快照。
- 重启服务/断网恢复后，**加载快照直接继续**，不用从头执行。
- 适合长流程、耗时任务、人机交互流程。

#### 2）状态持久化

**定义**：把 State 存入数据库，实现跨服务、跨重启保存。
**主流方案**
- **Redis**：快，适合会话级临时存储
- **SQLite/PostgreSQL**：永久存档
- **文件/对象存储**：离线备份

#### 实现价值

- 支持**关闭页面，明天接着聊**
- 支持**任务排队、异步执行**
- 生产级复杂流程必备

---

### 4. 并行节点执行写法

#### 定义

同时执行多个独立节点，**全部完成后再汇总结果**，提升速度。
#### 适用场景

- 同时检索+同时联网搜索
- 同时调用多个工具
- 同时生成多个方案

#### 代码写法

```python
# 同时并行执行 node_a 和 node_b
workflow.add_edge(START, ["node_a", "node_b"])

# 全部完成后，进入汇总节点
workflow.add_edge(["node_a", "node_b"], "summary_node")
```

#### 优势

**耗时从 A+B 变成 MAX(A,B)**，速度大幅提升。

---

### 5. 条件路由判断逻辑设计

#### 核心思想

**路由不写死在流程里，而是写在状态里，由状态驱动跳转。**

#### 标准设计步骤

1. **在 State 里定义路由字段**
`next_step`, `should_continue`, `node_type`
2. **在业务节点中计算并赋值**
节点根据逻辑，把下一步要走的节点名称写入状态。
3. **写一个通用判断函数**
函数直接读取状态中的 `next_step` 字段，返回节点名。
4. **绑定条件边**
所有分支都由状态决定，流程图极度简洁。

#### 伪代码模板

```python
def router(state):
    # 完全不写逻辑，只读取状态
    return state["next_step"]

workflow.add_conditional_edges(
    "decision_node",
    router,
    {
        "retrieve": "retrieve_node",
        "chat": "chat_node",
        "tool": "tool_node",
        "end": END
    }
)
```

#### 企业级最佳实践

**逻辑在节点，路由看状态，流程超清晰。**

---

#### 本章面试绝杀总结

1. **串行**：普通边，顺序执行
2. **分支**：条件边，if/else路由
3. **循环**：条件边指回上游，Agent核心
4. **中断**：暂停→人工→恢复，Human-in-the-loop
5. **并行**：同时跑多节点，汇总结果
6. **路由**：状态驱动，简洁可控

## 四. 多智能体 Multi-Agent

1. 分工式多智能体搭建思路
2. 主管-员工 Agent 协作架构
3. 智能体之间消息传递方式
4. 多Agent 任务分配与冲突解决

### 1. 分工式多智能体搭建思路

#### 核心思路

按照**业务职责拆分角色**，每个Agent只负责单一领域任务，各司其职，统一由状态图调度，避免单Agent臃肿杂乱。

#### 搭建步骤

1. **角色拆分**
按能力划分专属智能体：检索Agent、工具调用Agent、写作Agent、审核Agent、闲聊Agent等。
2. **统一状态池**
共用全局State，存放任务需求、执行进度、执行结果、流转指令。
3. **职责边界划定**
明确每个Agent能做什么、不能做什么，限定可用工具与权限。
4. **流转规则制定**
设定任务触发条件、执行顺序、结果流转方向。
5. **结果汇聚整合**
各子Agent输出独立结果，由汇总节点统一整理生成最终应答。
6. **异常兜底**
设置兜底Agent，处理超出所有角色能力的问题。

#### 优势

职责清晰、易于维护、单个Agent逻辑简单、便于独立迭代优化、支持灵活扩容新增角色。

### 2. 主管-员工 Agent 协作架构

#### 架构模型

**管理者Agent（主管） + 执行Agent（员工）** 分层架构，企业最常用多Agent模式。
1. **主管Agent（调度中枢）**
- 接收用户整体需求，拆解细分子任务
- 判断需要调动哪些员工Agent
- 下发任务指令、分配执行优先级
- 收集所有员工执行结果
- 校验结果完整性、正确性
- 整合信息，生成最终答案
- 决策是否追加任务、重新执行

2. **员工Agent（执行单元）**
- 只接收主管下发的指定任务，不自主决策全局流程
- 专注完成本职工作：知识库检索、接口查询、数据分析、文案撰写等
- 执行完毕主动回传结果与执行状态
- 无跨角色调度权限

#### 运行流程

用户提问 → 主管拆解任务 → 分发对应员工执行 → 员工返回结果 → 主管校验汇总 → 输出最终答案

#### 适用场景

复杂调研报告、行业数据分析、长文档联合创作、企业智能办公系统、一站式业务问答。

### 3. 智能体之间消息传递方式

1. **全局State状态共享（主流首选）**
所有Agent读写同一个结构化状态对象，任务指令、执行结果、对话上下文全部存入State，天然实现数据互通，解耦性最强，LangGraph原生支持。

2. **专属消息队列传递**
搭建内存/Redis消息队列，Agent之间通过发布订阅模式收发消息，适合异步多Agent协同。

3. **对话历史上下文传递**
以对话消息格式传递指令与结果，沿用对话交互逻辑，兼容性强，适配大模型交互习惯。

4. **回调事件通知**
一个Agent执行完成触发回调，主动通知指定下游Agent开始执行，适合强顺序依赖场景。

5. **结构化指令报文**
定义统一消息格式：角色标识+任务类型+入参+优先级，标准化传递，避免语义理解偏差。

### 4. 多Agent 任务分配与冲突解决

#### 一、任务分配方案

1. **能力匹配分配**
提前定义各Agent技能标签，根据任务类型自动匹配对应擅长Agent执行。
2. **负载均衡分配**
监控各Agent任务繁忙状态，优先分配任务至空闲智能体，避免单点拥堵。
3. **层级优先级分配**
核心紧急任务优先派发，普通后置任务延后执行。
4. **动态自主申领**
主管发布任务池，空闲员工Agent主动认领匹配任务。

#### 二、常见冲突类型

1. **任务重复执行**：多个Agent同时处理同一任务
2. **结果答案冲突**：不同Agent输出结论不一致、相互矛盾
3. **权限资源冲突**：多Agent抢占同一接口、同一数据库资源
4. **流程顺序冲突**：前置任务未完成，后置任务提前执行
5. **职责越界冲突**：员工Agent跨领域处理不属于自身的任务

#### 三、冲突解决策略

1. **任务唯一标识**
给每一条子任务分配唯一ID，避免重复领取、重复执行。
2. **结果统一校验仲裁**
由主管Agent充当仲裁者，对比多方结果，剔除错误答案，整合统一结论。
3. **资源加锁管控**
对接口、数据库等公共资源加执行锁，同一时间仅允许一个Agent调用。
4. **强流程顺序约束**
通过条件边、状态字段严格约束执行先后顺序，禁止乱序执行。
5. **严格权限与职责隔离**
限定每个Agent可调用工具、可访问数据范围，从根源杜绝越界行为。
6. **冲突重试与丢弃**
出现逻辑冲突自动放弃无效结果，重新下发任务二次执行。
7. **冲突日志记录**
记录冲突场景，后续优化角色分工与调度规则。

### 1. LangGraph 实现 RAG 智能问答流程

#### 完整流程（5 节点标准架构，企业通用）

1. **入口节点（Entry）**
   接收用户问题，初始化 State，清理问题格式。
2. **分类节点（Classify）**
   判断问题是闲聊 / 知识库问答 / 工具查询。
3. **检索节点（Retrieve）**
   去向量库召回相关文档片段，存入 State。
4. **生成节点（Generate）**
   大模型根据问题 + 上下文生成答案。
5. **校验节点（Validate）**
   判断是否需要**重新检索 / 改写问题 / 补充召回**。
6. **结束节点（END）**
   返回最终答案。

#### 条件路由

- 闲聊 → 直接生成回答
- 知识库 → 进入检索→生成
- 信息不足 → **循环回到检索**，再次召回

#### 核心优势

传统 RAG 只能一次检索；
**LangGraph RAG 可以自主循环补全、自我纠错，准确率大幅提升。**

### 2. 实现思考-工具调用-再思考闭环

#### 标准 ReAct 闭环（面试必考）

**Thought → Action → Observation → Thought → Finish**

#### LangGraph 实现步骤

1. **思考节点（Think）**
   模型分析问题，判断是否需要调用工具，生成工具参数。
2. **路由节点（Router）**
   判断：调用工具 / 直接回答 / 重新思考。
3. **工具执行节点（Tool）**
   执行检索、联网、查询、计算等工具，返回结果。
4. **观察节点（Observe）**
   将工具结果存入 State。
5. **再思考节点（ReThink）**
   模型根据工具结果判断信息是否足够。
6. **结束**
   信息足够 → 生成答案；不足 → 继续循环。

#### 一句话总结

**思考 → 调用工具 → 拿到结果 → 再思考 → 直到能回答。**
这就是 Agent 的灵魂。

### 3. 流程死循环如何规避

#### 高频面试题，必须背下来

1. **设置最大迭代次数（最有效）**
   限制最多循环 3~5 次，超过强制结束。
2. **状态中增加执行步数计数**
   `step_count` 每执行一次 +1，超限直接退出。
3. **强制终止条件**
   连续相同结果 → 判定无意义循环，强制终止。
4. **工具调用次数限制**
   同一个工具禁止连续调用多次。
5. **模型决策校验**
   让模型判断：是否已经重复、是否陷入循环。
6. **超时中断**
   流程执行超过时间阈值，自动结束。
7. **避免无意义条件跳转**
   路由逻辑必须明确：什么时候继续、什么时候停止。

#### 核心结论

**LangGraph 必须加“最大迭代次数”，否则生产环境必出问题。**

        nodes1 = self.vector_ret.retrieve(query)
        
        # 2. 关键词检索
        nodes2 = self.keyword_ret.retrieve(query)
        
        # 3. 实体关系检索

        # 5. 上下文压缩
        nodes_final = self.compressor.postprocess_nodes(nodes_reranked, query)
        
        return nodes_final

retriever = GraphEnhancedRetriever(

LlamaIndex 原名 GPT Index，是**面向检索增强 RAG 场景而生的轻量化智能数据框架**，主打**私有数据接入、文档索引构建、知识库问答**，核心目标是让大模型高效连接各类本地私有数据，快速搭建专业知识库系统。

1. **设计初衷不同**
    - **LangChain**：**通用型大模型应用编排框架**，覆盖面极广，偏向流程编排、Agent 智能代理、复杂业务链路、多角色协作，RAG 只是它众多能力之一。
    - **LlamaIndex**：**垂直深耕 RAG 领域**，从底层架构围绕文档解析、分片、建索引、语义检索、问答查询设计，天生为知识库而生。
2. **能力侧重点不同**
    - LangChain：强于**流程编排、工具调用、状态管理、多智能体、复杂业务自动化**。
    - LlamaIndex：强于**海量文档结构化索引、细粒度语义检索、多层级文档关联、精准知识库问答**。
3. **使用门槛不同**
    - LangChain 组件繁多、自由度高，自定义程度强，上手偏复杂。
    - LlamaIndex 封装度更高，内置大量开箱即用索引与查询引擎，**搭建RAG速度更快**。
4. **生态偏向**
    - LangChain 生态庞大，第三方工具、中间件、回调、监控集成更全面。
    - LlamaIndex 生态聚焦数据检索，在文档治理、索引优化领域方案更成熟。

### 2. 两者选型场景：项目选LangChain还是LlamaIndex

#### 优先选择 LlamaIndex

1. 项目核心就是**企业知识库、文档问答、本地私有文档检索**
2. 业务以长文本、PDF、书籍、报告、规章文件问答为主
3. 追求**快速落地RAG**，不想过多自定义底层检索逻辑
4. 需要做**文档层级检索、父子节点检索、知识图谱文档关联**
5. 轻量化知识库项目，不需要复杂流程与大量工具调用

#### 优先选择 LangChain

1. 项目不止做RAG，还需要**Agent工具调用、联网搜索、代码执行**
2. 业务存在**多轮复杂流程、分支判断、循环决策、人机交互**
3. 需要搭建**多智能体协作系统、业务自动化流水线**
4. 项目架构复杂，需要深度自定义链路、记忆管理、异步部署
5. 偏向智能助手、自动化办公、复杂业务工单、智能客服全链路系统

#### 混合选型

中小型知识库用LlamaIndex快速搭建检索层，上层业务流程、Agent调度使用LangChain串联，兼顾效率与灵活性。

### 3. LlamaIndex 核心优势：原生RAG、索引体系更强

#### 1）原生RAG架构，适配性拉满

- 框架底层所有模块**围绕文档检索问答设计**，无需像LangChain额外拼装组件。
- 内置完整链路：文档加载→智能分片→向量化建库→检索匹配→答案生成一站式封装。
- 对长文档、杂乱格式文档兼容性更好，开箱即用问答能力更强。

#### 2）索引体系极度丰富（最大核心优势）

LangChain仅侧重向量检索，LlamaIndex拥有**多类型成熟索引**，适配不同文档场景：
1. **向量索引**：主流语义相似度检索，日常知识库首选
2. **树索引**：适合超长书籍、长篇报告，自上而下逐层摘要检索
3. **列表索引**：顺序遍历文档，适合逐条对照查询
4. **关键词索引**：强化实体、专有名词精准匹配
5. **知识图谱索引**：抽取文档实体与关系，实现关联推理问答
6. **多层级父子索引**：大文档粗定位、小片段精检索，大幅提升准确率

#### 3）检索精细化能力更强

- 原生支持**问题改写、子问题拆分、答案重排、上下文压缩**等高阶RAG优化
- 内置大量文档预处理、语义切块、段落关联策略
- 对专业领域文献、技术文档、规范文档的理解与检索精度更优

#### 4）开发效率更高

- 极简API，几行代码即可完成知识库搭建与问答
- 内置大量现成查询引擎、对话引擎，无需手动拼接Prompt与检索器
- 本地化部署、私有化模型接入流程更简洁，适合快速产出业务原型

## 二. 核心索引体系

1. 向量索引 VectorStoreIndex 最常用场景
2. 树索引 TreeIndex 适用长文档摘要
3. 列表索引 ListIndex 作用
4. 关键词索引 KeywordIndex
5. 知识图谱索引 GraphIndex 原理与用途

### 1. 向量索引 VectorStoreIndex

#### 原理

将文档切分为文本块，通过嵌入模型转为向量，存入向量数据库，基于**语义相似度**完成检索匹配。

#### 最常用场景

1. 日常企业知识库、产品手册、规章制度问答
2. 通用业务文档、FAQ 智能问答
3. 短/中等篇幅文档检索，查询意图偏向语义理解
4. 绝大多数标准RAG项目，**生产环境首选默认索引**
5. 搭配重排、混合检索做高精度问答优化

#### 优缺点

- 优点：检索速度快、语义匹配准、接入简单、生态兼容最好
- 缺点：单纯字面专有名词匹配偏弱，超长全文统筹能力不足

### 2. 树索引 TreeIndex

#### 原理

自上而下构建**层级树形结构**，底层是细粒度原文切片，逐层向上汇总生成高层摘要，形成文档摘要树。查询时从顶层摘要逐层向下筛选定位，最终找到对应原文内容。

#### 适用长文档摘要场景

1. 整本书籍、长篇论文、万字级行业报告、合同全文
2. 需要**全局概括 + 局部细节**结合作答的场景
3. 文档整体主旨总结、大纲梳理、全文观点提炼
4. 大篇幅文档精简浓缩、分层精读检索

#### 优缺点

- 优点：擅长把控全文整体逻辑，超长文本处理友好，节省上下文token
- 缺点：构建索引耗时更长，细粒度精准问答不如向量索引

### 3. 列表索引 ListIndex

#### 核心作用

把所有文档片段按**原始顺序**存入线性列表，查询时按顺序遍历筛选匹配内容，无向量相似度计算，偏向顺序检索。

1. 保留文档原有排版、行文顺序，适合顺序类文本查询
2. 逐条核对条款、流程步骤、操作手册顺序类问答
3. 小体量文档快速遍历检索，结构简单易维护
4. 做文档逐条遍历校验、内容逐条比对场景

#### 适用与局限

- 适合：流程文案、条款清单、时序类文档
- 不适合：海量文档、模糊语义查询，检索效率偏低

### 4. 关键词索引 KeywordIndex

#### 原理

提取文档内核心实体、专业名词、专属术语、关键词建立倒排索引，基于**关键词精准命中**检索。

#### 核心用途

1. 专业领域术语、行业黑话、专有名词精准查找
2. 法条、编号、型号、编号类精准匹配查询
3. 弥补向量索引字面匹配不足，常和向量索引组合使用
4. 精准定位指定关键词所在段落与原文位置

#### 搭配用法

常与向量索引组合实现**语义+关键词**混合检索，大幅提升专业文档召回准确率。

### 5. 知识图谱索引 GraphIndex 原理与用途

#### 核心原理

1. 解析文档文本，由大模型自动抽取**实体**（人物、事物、地点、概念、产品）
2. 识别实体之间**关系**（属于、包含、关联、作用、时间先后等）
3. 构建实体-关系-实体三元组，形成结构化知识图谱网络
4. 用户提问时，先解析问题实体与关系，在图谱中做关联路径检索，再结合原文作答

#### 主要用途

1. 知识关联推理类问答，不止匹配文本，还能推导逻辑关系
2. 企业组织架构、人员关系、业务上下游关系查询
3. 产品功能关联、故障原因推导、事件因果梳理
4. 跨文档关联知识串联，打通分散文档之间逻辑
5. 构建行业领域知识库、专业领域知识体系

#### 优缺点

- 优点：具备逻辑推理能力，问答更有深度，适合复杂知识推理
- 缺点：索引构建成本高，抽取实体质量依赖大模型能力，耗时久

---
#### 面试速记总结

1. **VectorStoreIndex**：通用RAG首选，语义问答主力
2. **TreeIndex**：长文档、书籍、全文总结专用
3. **ListIndex**：顺序文本、流程条款顺序检索
4. **KeywordIndex**：专业名词、精准词条命中
5. **GraphIndex**：实体关系抽取，知识推理、关联问答

### 关键词索引/关键字检索

---

#### 1. 核心定义区分

##### 1）LlamaIndex 关键词索引 `KeywordIndex`

- **属于索引结构**，是**离线建库阶段**构建的持久化索引
- 原理：遍历文档/Node，抽取实体、专有名词、行业术语、编号、法条关键字，构建**倒排索引**
- 存储：和向量索引并列存在，固化在知识库中
- 目的：提前把文本拆解成关键词映射关系，为后续检索提供底层数据源
- 定位：**数据层结构**

##### 2）关键词检索

- **属于查询动作**，是**在线问答阶段**执行的查询逻辑
- 原理：从用户问句里提取关键词，去已建好的**关键词索引**中匹配命中段落
- 是一种**检索策略/查询方式**，不是独立存储结构
- 定位：**应用层查询行为**

---

#### 2. 层级关系

```
离线构建：文档 → 分词抽词 → 生成【关键词索引】(存库)
在线查询：用户问题 → 提取关键词 → 执行【关键词检索】→ 从索引里捞对应文本
```
**一句话**：
**关键词索引是“仓库”，关键词检索是“去仓库找东西的动作”**。

---

#### 3. 功能与用法差异

##### 关键词索引 KeywordIndex

1. 搭建时显式创建
```python
from llama_index.core.indices.keyword_table import KeywordTableIndex
index = KeywordTableIndex(nodes)
```
2. 擅长固定词条、编号、专业名词、法条、证件号精准匹配
3. 独立索引类型，可单独用，也可和向量索引组合

##### 关键词检索

1. 是检索模式，不单独建索引
2. 可以作用在任意索引之上：向量索引也能附加关键词检索
3. 常用来做**混合检索**：向量语义匹配 + 关键词精准命中
4. 多用于弥补纯向量检索“字面精准度不足”的问题

---

#### 4. 最容易混淆的场景

1. 只用 `KeywordIndex` 做查询 → 此时**检索动作 = 关键词检索**
2. 日常RAG主流：**VectorStoreIndex + 附加关键词检索**
   - 向量索引做主语义召回
   - 额外抽取关键词二次过滤、加权排序
   - 这里**没有新建关键词索引**，只是查询层面加了关键词匹配逻辑

---

#### 5. 面试精简背诵版

1. **关键词索引**：离线构建的倒排索引数据结构，属于存储层；
2. **关键词检索**：线上基于关键词做匹配查询的执行逻辑，属于调用层；
3. 二者强关联：关键词检索**依赖**关键词索引才能高效查询，但关键词检索不只能用在关键词索引上；
4. 工程实践：纯关键词索引语义弱，一般搭配向量索引，用**关键词检索做增强兜底**，提升专业术语命中率，减少幻觉。

### 1. SimpleDirectoryReader 批量加载文档

#### 是什么

LlamaIndex **最简单、最常用的文档读取器**，专门用来**一键批量读取文件夹**里的所有文件。

#### 核心能力

1. **自动识别格式**
   支持 PDF、DOCX、PPTX、Excel、CSV、TXT、Markdown、JSON 等。
2. **递归读取子文件夹**
   自动遍历目录下所有文件，不用一个个加载。
3. **自动解析文本内容**
   自动提取正文、标题、页码、路径等信息。
4. **返回文档列表**
   输出 `Document` 对象列表，直接进入下一步分片。

#### 代码（面试常写）

```python
from llama_index.core import SimpleDirectoryReader

# 一键加载整个文件夹
documents = SimpleDirectoryReader("./data").load_data()
```

#### 一句话总结

**SimpleDirectoryReader = 文件夹批量文档导入工具，一行代码搞定数据接入。**

### 2. Node 节点概念，和 LangChain Document 对应关系

#### 1）Node 节点是什么？

Node 是 LlamaIndex 中 **最小的数据存储与检索单元**，相当于**文本块（chunk）**。
所有文档最终都会被拆分成若干个 Node。

#### 2）Node 包含内容

- 文本内容 `text`
- 元数据 `metadata`
- 唯一ID `id`
- 上下级关系（父子节点）
- 嵌入向量 `embedding`

#### 3）与 LangChain Document 对应关系

LlamaIndex Node ≈ LangChain Document
**两者完全等价，都是文本块。**

| 特性         | LlamaIndex           | LangChain     |
| ------------ | -------------------- | ------------- |
| 最小检索单元 | **Node**             | **Document**  |
| 存储内容     | 文本 + 元数据 + 向量 | 文本 + 元数据 |
| 分片后结果   | Node 列表            | Document 列表 |
| 检索返回     | Node                 | Document      |

#### 核心区别

LlamaIndex Node **更强、自带层级关系、支持父子结构、自带向量**，是 Document 的升级版。

---

### 3. 细粒度节点拆分、父子节点层级结构

#### 1）细粒度拆分（Chunking）

把大文档拆成小 Node，目的：
- 语义更聚焦
- 检索更精准
- 不超上下文窗口

LlamaIndex 自动拆分：
```python
from llama_index.core.node_parser import SentenceSplitter
parser = SentenceSplitter(chunk_size=512, chunk_overlap=20)
nodes = parser.get_nodes_from_documents(documents)
```

---

#### 2）父子节点层级结构（LlamaIndex 王牌能力）

**父节点（粗粒度）**：摘要、段落、大段文本
**子节点（细粒度）**：短句、细节、小chunk

##### 工作流程

1. 文档 → 生成**父节点**
2. 父节点 → 拆分成多个**子节点**
3. 检索时：
   - 先匹配**父节点**（快速定位章节）
   - 再返回**子节点**（提供精准细节）

##### 优点

- 检索精度大幅提升
- 上下文更完整
- 长文档效果远超普通分片
- **LlamaIndex 核心竞争力**

### 4. 文档元数据管理

#### 元数据是什么

给每个 Node 打标签：
- 文件名
- 文件路径
- 页码
- 作者
- 部门
- 时间
- 文档类型
- 权限等级

#### 作用

1. **过滤检索**
   只查技术部、只查2025年、只查合同
2. **精准溯源**
   回答来自哪篇文档、哪一页
3. **权限控制**
   不同人看到不同内容
4. **排序与优化**
   新文档优先、重要文档优先

SimpleDirectoryReader 导入时**自动填充元数据**，无需手动写。

## 四. 查询引擎 Query Engine

1. 基础查询引擎、对话查询引擎区别
2. 检索模式：精简/摘要/逐段查询
3. 自定义查询改写、问题扩写
4. 子问题拆分查询 SubQuestionQueryEngine

### 1. 基础查询引擎、对话查询引擎区别

#### 基础查询引擎 QueryEngine

- 定位：**单轮独立问答**，无会话记忆
- 特点：每次请求独立，不保存历史对话上下文
- 执行逻辑：仅根据当前问题+检索文档生成答案
- 适用场景：一次性文档问答、知识库静态查询、批量问答
- 缺点：无法承接上文，多轮对话会断逻辑

#### 对话查询引擎 ChatQueryEngine

- 定位：**多轮连续对话**，内置会话记忆能力
- 特点：自动维护聊天历史，能理解指代、省略句、承接前文
- 执行逻辑：拼接历史对话 + 当前问题 + 检索内容统一推理
- 适用场景：智能聊天助手、连续咨询、交互式知识库客服
- 底层依赖：内置对话记忆组件，自动管理上下文窗口

**核心区分**
单轮查资料用基础引擎，连续聊天交互用对话引擎。

### 2. 检索模式：精简/摘要/逐段查询

1. **精简查询 Compact**
    压缩拼接检索到的文本块，精简冗余内容后送入模型，节省Token，平衡速度与信息量，日常最常用。
2. **摘要查询 Summarize**
    先对所有召回文档批量生成全局摘要，再基于摘要作答，适合通篇概括、整体内容总结，弱化细节。
3. **逐段查询 Retrieve Each**
    逐个文本块依次推理校验，逐条匹配问题相关内容，精准度高，适合细节深挖、条款核对、精准定位原文，缺点是调用次数多、速度偏慢。

### 检索方式

#### 向量检索

##### 一、核心定义

向量检索是**基于语义相似度**的检索方式，通过嵌入模型将自然语言文本转为多维浮点向量，利用向量空间距离计算文本语义相近程度，实现**语义层面匹配**，而非单纯字面匹配，是现代RAG系统最主流的检索方案。

##### 二、核心基础：文本向量化

###### 1. 什么是文本向量

任意一段文字，通过**Embedding嵌入模型**，转换成一组固定长度的**多维数字数组**，示例：
```
"门票退票规则" → [0.12, 0.35, -0.21, 0.78, ..., 0.44]
```
- 向量每一个维度代表文本隐含的语义特征
- 语义越相似的文本，生成的向量在空间中距离越近
- 语义差异越大，向量空间距离越远

###### 2. 嵌入模型作用

负责统一语义编码，把人类语言映射到**同一向量空间**，让机器可以量化计算文本相似度。

##### 三、两大核心阶段：离线建库 + 在线检索

###### 阶段1：离线向量化建库（预处理）

1. **文档切片**
   原始长文档按照固定长度切分为文本块`Chunk/Node`，作为最小语义存储单元。
2. **批量文本嵌入**
   调用嵌入模型，对所有分片文本批量生成**语义向量**。
3. **向量入库存储**
   将「文本内容 + 对应向量 + 元数据」统一存入**向量数据库**（Milvus、FAISS、Chroma、PGVector等）。
4. **构建索引结构**
   向量库内部构建**向量索引**（IVF、HNSW等），用于加速后续海量向量相似度查询。

###### 阶段2：在线用户查询检索（实时推理）

1. **用户问句向量化**
   接收用户提问，使用**同一个嵌入模型**生成查询向量。
   
   > 重点：必须和建库使用同款Embedding模型，否则向量空间不统一，检索失效。
2. **向量相似度计算**
   在向量库中，计算**查询向量**与库存所有文档向量之间的距离。
3. **距离排序筛选**
   按照相似度从高到低排序，取出Top-K个最相似文本块。
4. **拼接上下文返回**
   将召回的语义相似文本作为参考上下文，送入大模型生成答案。

##### 四、主流相似度计算原理

###### 1. 余弦相似度（RAG最常用）

衡量两个向量**方向夹角**，忽略向量长度，只关注语义方向一致性。
- 取值范围：`[-1,1]`
- 越接近1：语义高度相似
- 越接近-1：语义完全相反
**业务场景**：日常知识库、口语化提问、模糊语义检索。

###### 2. 欧氏距离

计算向量空间直线直线距离，距离越小语义越接近。
适合向量长度差异小、短文本精准匹配场景。

###### 3. 曼哈顿距离

逐维度差值求和，多用于简单轻量化向量匹配。

##### 五、向量检索核心特点

###### 优势

1. **强语义理解**
   支持同义句、口语化、转述式提问，无需严格匹配关键词
   例：`怎么退门票` 和 `门票能办理退款吗` 可精准匹配。
2. **泛化能力极强**
   适配自然闲聊、模糊提问、开放式问答。
3. **适配海量文档**
   依托向量库索引算法，百万级文档依旧可快速检索。
4. **接入简单**
   LlamaIndex、LangChain原生深度适配，开箱即用。

###### 缺点

1. **字面精准度不足**
   对编号、法条、固定业务术语、硬性规则匹配容易偏差。
2. **易产生语义漂移**
   相近语义但无关业务的文本容易被误召回。
3. **存在检索幻觉源头**
   容易召回看似语义相近、实际不符合业务规则的内容，诱导大模型编造答案。
4. **依赖嵌入模型质量**
   模型效果差，直接导致整体检索精度暴跌。

##### 六、向量检索 和 关键词检索 核心对比

| 对比维度   | 向量检索                     | 关键词检索                 |
| ---------- | ---------------------------- | -------------------------- |
| 匹配逻辑   | 语义相似度匹配               | 字面关键词精准匹配         |
| 依赖模型   | 需要Embedding嵌入模型        | 无需任何大模型             |
| 擅长场景   | 口语提问、模糊查询、语义问答 | 业务规则、术语、编号、法条 |
| 泛化能力   | 强                           | 无                         |
| 抗幻觉能力 | 弱                           | 极强                       |
| 检索速度   | 中等                         | 极快                       |

##### 七、项目工程落地用法（贴合你的智能体项目）

1. **主检索**：使用向量检索承接用户自然语言、口语化票务/科普提问，完成大范围语义粗召回；
2. **辅助兜底**：搭配关键词检索，锁定退票、改签、申诉、处罚等硬性业务规则；
3. **优化升级**：向量粗召回 + 重排精筛 + 实体关系校验，构成完整图增强检索，大幅抑制向量检索带来的问答幻觉。

##### 八、面试精简背诵版

向量检索依托嵌入模型将文本转为多维语义向量，离线完成全文档向量化并存入向量数据库；用户查询时将问句转为同空间向量，通过余弦相似度计算语义距离，召回最相似文本片段。该方式具备优秀语义理解与泛化能力，适配日常自然语言问答，但专业规则精准匹配较弱，实际项目中常与关键词检索组合使用，兼顾语义泛化与业务精准性。

#### LlamaIndex 关键词检索 

##### 一、核心定义

关键词检索依托**关键词索引（KeywordTableIndex）**实现，属于**词级精准匹配检索**，脱离向量语义，基于文本分词、关键词映射、倒排索引完成文档段落匹配，擅长专业名词、业务规则、固定术语、编号类场景查询，是RAG体系中弥补向量检索字面匹配不足的核心检索方式。

##### 二、底层核心原理：倒排索引（Inverted Index）

###### 1. 正向索引（普通存储）

常规存储格式：**文档ID → 文档全部关键词**
- 文档1：退票、手续费、24小时
- 文档2：申诉、订单号、处罚结果
缺点：查询时遍历所有文档，速度极慢，海量数据无法使用

###### 2. 倒排索引（关键词检索核心）

反向映射格式：**关键词 → 所属文档/文本节点列表**
- 退票 → [节点1, 节点5]
- 申诉 → [节点3, 节点7]
- 手续费 → [节点1, 节点9]
**核心优势**：输入关键词直接命中对应文本块，无需全库遍历，查询效率极高。

##### 三、全流程分步原理（离线建库 + 在线查询）

###### 阶段1：离线构建关键词索引（预处理阶段）

1. **文本分片**
将原始业务文档（票务规则、申诉条例、科普文案）切分为固定大小`Node`文本块，作为最小检索单元。
2. **分词提取关键词**
对每个`Node`文本执行分词处理，过滤停用词（的、了、吗、和等无意义词汇），**抽取核心实体词、业务关键词**。
    - 票务场景关键词：退票、改签、余票、有效期、手续费
    - 申诉场景关键词：违规、处罚、申诉流程、撤销、举证
3. **建立关键词映射关系**
遍历所有文本节点，把提取出的关键词与所属`Node`进行绑定。
4. **生成持久化倒排索引表**
统一汇总所有关键词，构建全局关键词字典，固化存储，完成索引搭建。

###### 阶段2：在线用户查询检索（推理阶段）

1. **用户问句分词**
接收用户自然语言提问，对问句做同样分词处理，提取用户核心查询关键词。
    示例：`当天购买的门票能退票吗？` → 提取关键词：**门票、退票**
2. **关键词精准匹配**
拿着提取的关键词，去离线建好的**倒排索引表**中进行完全字符匹配。
3. **聚合命中文本节点**
匹配到关键词后，快速拉取所有绑定该关键词的`Node`文本块，作为候选上下文。
4. **结果排序输出**
按照**关键词命中数量**排序，命中关键词越多的文本块，优先级越高，最终返回top-k检索结果。

##### 四、LlamaIndex 关键词检索专属运行逻辑

1. 仅匹配**字面字符**，不做语义理解
    同义词、近义词无法匹配，例：`退款` 无法命中关键词 `退票`。
2. 支持多关键词联合检索
用户问句提取多个关键词时，优先召回**同时包含多个关键词**的文本节点，提升匹配精准度。
3. 无向量距离计算
全程不调用嵌入模型、不生成向量，**纯文本字符串匹配**，推理速度快、资源消耗低。
4. 适配结构化规则文本
对条例、规章、业务硬性规则适配性极强，不会出现语义偏差。

##### 五、核心优缺点

###### 优点

1. **匹配精度极高**：固定业务术语、编号、规则条款零偏差；
2. **检索速度快**：基于倒排索引，毫秒级返回结果；
3. **零语义幻觉**：只匹配已有文本关键词，不会衍生无关内容；
4. **轻量化运行**：无需依赖嵌入模型，离线离线均可使用。

###### 缺点

1. **无语义泛化能力**：无法理解同义句、口语化提问；
2. **对长句模糊提问适配差**：自然闲聊、宽泛问题检索效果差；
3. **依赖分词质量**：分词不准会直接导致关键词遗漏，检索失效。

##### 六、业务场景适配（贴合你的AI智能体项目）

1. **票务业务**：查询退票规则、改签条件、票价标准，优先使用关键词检索锁定硬性规则；
2. **申诉业务**：匹配处罚类型、申诉材料、受理流程等固定条例；
3. **科普业务**：专业景区名词、景点专有名称精准定位；
4. **工程最佳实践**：**关键词检索 + 向量检索混合使用**
    - 向量检索：承接用户口语化、模糊化提问，做语义粗召回；
    - 关键词检索：精准锁定业务硬性规则、专业术语，兜底防幻觉；
    - 二者结合构成图增强检索的核心底层能力。

##### 七、面试精简背诵版

关键词检索基于**倒排索引**实现，分为离线建库与在线查询两大流程：离线阶段对文档分片、分词提取核心关键词，搭建关键词与文本节点的反向映射表；在线查询时提取用户问句关键词，通过字符精准匹配快速召回对应文本。该检索方式字面匹配准确率高、查询速度快，擅长业务规则、专有名词检索，但缺乏语义理解能力，项目中常与向量检索组合，兼顾语义泛化与规则精准度，有效降低大模型问答幻觉。

##### 1. **向量检索（VectorIndexRetrieval）**

###### 原理

文本 → 向量 → 相似度匹配

###### 适用场景

90% 企业知识库、日常问答、语义理解
###### 优点

语义理解强、泛化能力好
###### 缺点

专业术语、编号、精准匹配弱
###### 关键词

**语义检索、泛化提问、日常问答首选**

---

##### 2. **关键词检索（KeywordTableIndex）**

###### 原理

倒排索引，关键词精准命中
###### 适用场景

术语、法条、型号、编号、专有名词、精准匹配

###### 优点

**绝对精准、不跑偏、不幻觉**
###### 缺点

无语义能力
###### 关键词

**精准匹配、专业术语、强规则业务**

##### 3. **列表检索（ListIndexRetrieval）**

###### 原理

按顺序遍历所有文本
###### 适用场景

###### 优点

保持原文顺序
###### 缺点

海量文档极慢
###### 关键词

**顺序检索、流程文档、条款核对**

##### 4. **树检索（TreeIndexRetrieval）**

###### 原理

逐层摘要，从上往下查找
###### 适用场景

书籍、长篇报告、论文、总结类问题
###### 优点

全局视野、省 token、适合超长文本
###### 缺点

构建慢
###### 关键词

**长文档、全文总结、书籍问答**

---

##### 5. **知识图谱检索（KnowledgeGraphIndex）**

###### 原理

实体-关系-实体 三元组检索
###### 适用场景

推理、因果、关联关系、组织架构
###### 优点

**能推理、不幻觉、逻辑严谨**
###### 缺点

建库成本高
###### 关键词

**逻辑推理、知识关联、低幻觉问答**

#### 🔥 三、第二类：查询引擎执行模式（同样检索，不同回答方式）

**这是面试官最爱问、新手最容易懵的地方！**

##### 1. **compact 精简模式（默认）**

把所有检索结果**拼在一起**给模型读

###### 优点

快、省 token
###### 缺点

内容太多可能被淹没
###### 适合

**通用问答、日常使用**

先把所有文档生成**摘要**，再回答
###### 优点

适合全文总结
###### 缺点

细节丢失
###### 适合

**总结、概括、报告解读**

**一段一段问模型**，每段都独立回答一次，最后汇总
###### 优点

细节最全、最精准、不漏信息
###### 缺点

最慢、调用次数多

###### 适合

**条款检索、合规问答、票务/申诉/法律类**

**大幅降低幻觉、提升精准度**

### 3. 自定义查询改写、问题扩写

#### 作用

解决用户提问口语化、表述模糊、问题太短导致**检索召回不准**的问题。
#### 查询改写

把口语化、歧义问题，改写为标准书面专业问句，对齐知识库话术，提升匹配度。
#### 问题扩写

基于原问题拓展同义问句、补充相关维度、补全缺失限定词，生成多条相似查询语句多路检索。
#### 实现方式

1. 自定义Prompt指令，让大模型统一规整问题格式
2. 封装改写工具函数，查询前统一预处理问句
3. 内置QueryTransform组件一键开启扩写改写
#### 业务价值

大幅降低因用户提问不规范导致的RAG回答错误。

### 4. 子问题拆分查询 SubQuestionQueryEngine

#### 核心原理

遇到**复杂复合型大问题**时，引擎自动拆解为多个独立简单子问题，分别检索答案，最后汇总整合所有子结果，生成完整最终回答。
#### 执行流程

1. 接收用户复杂总问题
2. LLM自动拆解成多个细分子问题
3. 每个子问题单独执行检索+问答
4. 收集全部子问题答案
5. 融合梳理、去重整合，输出完整答案
#### 适用场景

- 对比类问题、多维度分析问题
- 跨多个文档、多个知识点的综合提问
- 行业调研、方案分析、数据汇总类问答
#### 优缺点

- 优点：解决复杂问题答不全、答片面的痛点，逻辑拆解清晰
- 缺点：拆分+多次检索耗时更长，资源消耗更高

### 1. 重排 Rerank 在 LlamaIndex 集成方式

#### 核心作用

向量检索是**粗召回**，容易混入无关片段；重排模型对召回结果二次打分排序，筛选高相关性内容，剔除低质量文本，大幅提升问答精准度。

#### 集成实现方式

1. **全局接入重排器**
内置`SentenceTransformerRerank`、`CohereRerank`、`BGE-Rerank`等主流重排模型。
2. **流程嵌入**
检索器先召回TopK文本 → 传入重排模型打分 → 截取高分TopN送入大模型。
3. **代码接入流程**
```python
from llama_index.core.postprocessor import SentenceTransformerRerank
# 初始化重排器
rerank = SentenceTransformerRerank(top_n=3, model="BGE-rerank-base")
# 挂载到查询引擎
query_engine = index.as_query_engine(node_postprocessors=[rerank])
```
4. **灵活配置**
可自定义重排数量、本地离线模型、云端API模型，适配私有化部署。

#### 适用场景

专业知识库、技术文档、企业规章等高精准问答场景。

### 2. 上下文窗口自适应压缩

#### 原理

当检索拼接后的文本总量**超出大模型上下文窗口**时，自动智能压缩文本，在保留核心信息前提下精简内容。
#### 实现方案

1. **文本精简裁剪**
自动剔除冗余语句、重复内容、无效修饰词，保留核心语义。
2. **语义摘要压缩**
对长文本块批量生成短句摘要，替代原文。
3. **动态阈值截断**
实时统计Token数量，临近上限自动压缩，不强行截断语句。
4. **分层压缩策略**
远距离历史内容深度压缩，近距离核心内容完整保留。
#### 优势

完美适配小窗口大模型，无需手动调整分片大小，自动适配各类模型。

### 3. 多文档跨库联合检索

#### 定义

同时对接**多个不同索引、不同向量库、不同业务知识库**，一次提问并行检索多数据源，整合结果统一作答。
#### 实现方式

1. **组合索引 CombineIndex**
聚合向量索引、树索引、关键词索引，统一入口查询。
2. **多检索器融合**
绑定多个独立知识库索引，并行发起检索。
3. **结果合并去重**
汇总多库召回内容，剔除重复片段，统一排序整合。
4. **数据源路由筛选**
根据问题类型，定向检索指定知识库，减少无效查询。
#### 业务场景

企业多部门知识库合并查询、产品多系列文档联合问答、内部资料+外部资料融合检索。

### 4. 缓存检索结果提升速度

#### 优化思路

对高频重复问题、固定问答内容做结果缓存，避免重复检索、重复向量化、重复模型推理。
#### 主流缓存方案

1. **内存本地缓存**
开发测试环境使用，进程内缓存问答结果。
2. **Redis分布式缓存（生产首选）**
以用户问题为Key，答案与检索上下文为Value，设置过期时间。
3. **语义缓存**
不止精准匹配问句，相似问句命中缓存，复用结果。
4. **检索片段缓存**
缓存高频热门文本块向量检索结果，减少向量库查询压力。
#### 生效逻辑

用户提问 → 优先查询缓存 → 命中直接返回 → 未命中再走完整RAG流程。
#### 注意点

设置缓存过期时间，文档更新后主动清理缓存，保证内容时效性。

### 5. 幻觉抑制常用方案

1. **严格上下文限定**
在系统提示词强制要求：**仅依据检索到的文档内容作答**，无相关内容直接告知不知道，禁止编造信息。
2. **引用原文溯源**
回答末尾标注内容来源文档、段落位置，约束模型不乱编。
3. **精简冗余上下文**
过滤无关、低相关检索片段，减少模型被无用信息误导。
4. **启用答案校验机制**
生成答案后二次校验，核对答案内容是否能在知识库找到依据。
5. **使用领域专用嵌入与大模型**
行业微调模型专业性更强，幻觉概率远低于通用模型。
6. **拆分细粒度检索**
缩小文本块粒度，让模型只聚焦精准细节内容。
7. **关闭高创造性参数**
调低temperature温度值，让模型输出更严谨、保守。
8. **知识图谱辅助校验**
借助实体关系三元组，判断答案逻辑是否符合文档真实关系。

### 1. LlamaIndex 会话记忆实现

LlamaIndex 专门提供 **ChatMemory + ChatEngine** 实现多轮对话记忆，比 LangChain 更轻量、更开箱即用。

#### 核心组件

1. **ChatMemory**
   对话记忆容器，负责存储历史消息（用户问 + 助手答）。
2. **ChatEngine**
   对话引擎，自动把**历史对话 + 当前问题 + 检索上下文**拼给模型。
3. **MemoryTokenLimit**
   自动按 Token 长度裁剪历史，防止溢出。

#### 最简实现代码

```python
from llama_index.core.memory import ChatMemory
from llama_index.core.chat_engine import SimpleChatEngine

# 1. 创建记忆
memory = ChatMemory.from_defaults()

# 2. 创建对话引擎（自带记忆）
chat_engine = SimpleChatEngine.from_defaults(memory=memory)

# 3. 多轮对话
response1 = chat_engine.chat("你好")
response2 = chat_engine.chat("我刚刚说了什么")  # 能记住上一句
```

#### 核心特点

- 自动维护历史
- 自动管理上下文长度
- 与 RAG 查询引擎无缝结合
- 开箱即用，无需手动拼接消息

### 2. 历史对话融入检索上下文

#### 为什么要融？

用户多轮提问会出现**指代、省略、缩写**：
```
用户：什么是RAG？
AI：xxx
用户：它的缺点是什么？ → 这里“它”必须依赖历史才能理解
```
不融入历史 → 检索完全不准 → 回答错误。

#### LlamaIndex 如何实现？

1. **历史对话重写（最常用）**
   把**历史对话 + 当前问题**丢给 LLM，自动合成**完整标准问题**。
   ```
   历史：什么是RAG？
   当前：它的缺点是什么？
   → 重写后：RAG的缺点是什么？
   ```
2. **将重写后的问题送入检索**
   检索更精准，不会丢失指代信息。
3. **对话引擎自动完成**
   `CondenseQuestionChatEngine` 就是专门做这件事的。

#### 核心引擎

```python
from llama_index.core.chat_engine import CondenseQuestionChatEngine

chat_engine = CondenseQuestionChatEngine.from_defaults(
    query_engine=query_engine,  # RAG检索
    memory=memory,             # 记忆
    condense_question_prompt=custom_prompt
)
```

#### 一句话总结

**历史对话 → 问题重写 → 精准检索 → 正确回答**
这是多轮 RAG 对话的**黄金链路**。

### 3. 持久化对话历史

生产环境必须把历史存起来，不能存在内存里（重启丢失）。

#### LlamaIndex 持久化方案

1. **SimpleChatStore + 本地JSON（轻量）**
   最简单的持久化，存成文件。
   ```python
   from llama_index.core.chat_store import SimpleChatStore
   from llama_index.core.memory import ChatMemory

   # 保存到JSON
   chat_store = SimpleChatStore.from_persist_path("./chat_history.json")
   
   # 绑定记忆
   memory = ChatMemory(chat_store=chat_store, session_id="user_123")
   ```

2. **RedisChatStore（生产首选）**
   分布式、高性能、支持过期、支持多服务。
   ```python
   from llama_index.core.chat_store import RedisChatStore

   chat_store = RedisChatStore(redis_url="redis://localhost:6379")
   ```

3. **DatabaseChatStore**
   PostgreSQL / MySQL 存储，适合需要长期归档的企业系统。

#### 关键：session_id

所有持久化都必须靠 **session_id** 区分不同用户：
- user_001
- user_002
- session_20250405_1234

不同用户对话完全隔离，不串聊。

### 5. 评估体系：RAG准确率、召回率评测

#### 1. 核心评测指标

- **召回率Recall**
衡量**相关文档被成功检索出来的比例**，判断是否漏找关键信息。数值越高，遗漏有效资料越少，解决答不全问题。
- **准确率Precision**
衡量**召回的文档中有效相关内容占比**，判断检索结果是否混杂无关信息，数值越高垃圾内容越少。
- **答案准确率**
人工+评测模型判定最终生成答案是否贴合标准答案、是否存在编造内容。
- **幻觉率**
统计答案中无文档依据的编造内容占比，是RAG核心风控指标。

#### 2. 主流评测方案

1. **人工标注数据集**
构建标准问答数据集，包含问题、标准答案、关联参考文档，批量跑测统计指标。
2. **框架内置评测工具**
使用LlamaIndex自带评测模块，自动批量计算检索指标与问答得分。
3. **第三方评测模型打分**
借助专业评测大模型，自动评判回答完整性、准确性、逻辑性、忠实度。
4. **线上灰度评测**
线上收集用户真实提问，抽样复盘检索结果与回答质量，持续优化分片、检索策略。

#### 3. 优化方向

召回率低：优化分片、扩充召回数量、启用问题扩写、混合检索
准确率低：增加重排过滤、精简上下文、优化嵌入模型、收紧提示词约束

## LlamaIndex 实现RAG 完整全流程详解

### 一、整体核心流程总览

**原始文档 → 数据加载 → 文档清洗预处理 → 文本分片 → 生成向量嵌入 → 构建索引 → 用户提问 → 问题优化 → 语义检索召回 → 结果重排过滤 → 拼接Prompt上下文 → LLM推理生成答案 → 结果输出**

---

### 二、分步详细拆解

#### 1. 数据源加载

负责读取各类本地/在线业务文档
- 核心工具：`SimpleDirectoryReader`
- 支持格式：PDF、Word、TXT、Markdown、Excel、网页、JSON等
- 能力：自动遍历文件夹、递归读取子目录、自动提取正文
- 输出：批量 `Document` 文档对象列表

#### 2. 文档预处理清洗

剔除无效内容，提升后续检索精度
1. 去除空行、多余空格、乱码、特殊符号
2. 过滤页眉、页脚、页码、水印、目录、致谢、参考文献
3. 内容去重、统一排版格式
4. 自动提取**元数据**：文件名、路径、页码、时间、业务标签

#### 3. 文本节点拆分（Chunk分片）

将长文档切分为适合检索与模型输入的最小单元
1. 核心组件：`SentenceSplitter` 句子分割器
2. 核心参数
    - `chunk_size`：单块文本长度
    - `chunk_overlap`：相邻块重叠字符，防止语义断裂
3. 高级分片
    - 语义分片：按语义相似度切割
    - 父子节点分片：粗粒度父节点+细粒度子节点，长文档最优方案
4. 输出：大量结构化 **Node 节点**（LlamaIndex最小检索单元）

#### 4. 嵌入向量化

将文本语义转为浮点向量，实现语义匹配
1. 接入嵌入模型
    - 通用：BGE、Sentence-BERT
    - 私有化：本地自定义Embedding模型
2. 执行逻辑：遍历所有Node，批量生成向量
3. 作用：文字无法直接计算相似度，向量可通过距离计算语义相近程度

#### 5. 构建索引（LlamaIndex核心特色）

根据业务场景选择对应索引，存入向量库
1. **VectorStoreIndex 向量索引（主流首选）**
    基于向量相似度检索，通用知识库、日常问答首选
2. **TreeIndex 树索引**
    构建层级摘要树，适合书籍、长篇报告、全文总结
3. **KeywordIndex 关键词索引**
    构建倒排索引，精准匹配专业术语、编号、专有名词
4. **GraphIndex 知识图谱索引**
    抽取实体+关系三元组，实现知识推理问答
5. 存储介质：FAISS、Chroma、Milvus、PGVector、Elasticsearch

#### 6. 构建查询/对话引擎

封装检索+问答能力，对外提供调用入口
1. 单轮问答：`query_engine` 基础查询引擎
2. 多轮对话：`chat_engine` 对话引擎，自带会话记忆
3. 复杂问题：`SubQuestionQueryEngine` 自动拆分子问题查询

#### 7. 用户发起提问

用户输入自然语言问题，进入查询链路

#### 8. 查询前置优化

解决口语化、指代不清、问题过短导致检索不准
1. **问题改写**：把口语化问句转为标准专业问句
2. **问题扩写**：生成多条同义问句，多路检索
3. **历史对话重构**（多轮必备）
    融合上文对话，将省略句、指代句补全为完整问题，再进行检索

#### 9. 多路文档检索召回

从向量库中找出和问题最相关的文本块
1. 基础：向量语义相似度检索
2. 进阶：向量+关键词**混合检索**
3. 前置过滤：依靠元数据过滤时间、部门、文档类型、权限范围，缩小检索范围
4. 初步召回：取出TopK高相似度Node节点

#### 10. 后置重排Rerank（高阶优化）

对粗召回结果二次精准筛选
1. 作用：剔除语义相近但无关的噪声文本
2. 流程：粗召回Top10 → 重排打分 → 精选Top3~Top5优质上下文
3. 大幅提升答案准确率，减少无关信息干扰

#### 11. 上下文动态压缩

解决上下文过长超出大模型窗口限制
1. 自动统计token数量
2. 超长内容智能摘要精简、冗余删减
3. 保留核心语义，舍弃无效修饰内容

#### 12. 组装提示词Prompt

固定问答规则+参考文档+用户问题统一拼接
固定约束模板：
```
你只能严格依据下面参考文档内容回答用户问题，
禁止编造知识、禁止脱离原文发挥，
没有相关内容直接回复不知道。
参考文档：{检索到的上下文}
用户问题：{用户问题}
```

#### 13. 调用大模型推理生成答案

1. 支持：云端大模型、本地私有化开源大模型
2. 输出模式：普通一次性输出 / 流式逐字输出
3. 依靠约束Prompt严格限制模型行为，从源头降低幻觉

#### 14. 结果后处理与输出

1. 答案格式化整理、去冗余
2. 附加文档来源溯源，标注答案出处
3. 多轮对话自动保存问答记录至记忆组件
4. 对外返回最终回答

#### 15. 长效迭代优化

1. 增量更新文档，增量构建索引
2. 统计召回率、准确率、幻觉率做评测
3. 调整分片大小、嵌入模型、检索策略持续调优

### 三、极简流程背诵版

文档加载 → 清洗 → 切片成Node → 向量化 → 建索引存向量库
用户提问 → 问题改写优化 → 多路检索 → 重排筛选 → 拼装上下文Prompt
大模型依据文档作答 → 输出结果

### 四、核心优势总结

1. 索引种类丰富，适配几乎所有文档场景
2. 原生支持父子节点、多轮对话、记忆管理
3. 开箱即用组件多，RAG落地速度远快于原生LangChain搭建
4. 私有化部署链路完整，全流程可离线实现

## 2. 对比 LangChain / LangGraph / LlamaIndex 技术选型

1. **LlamaIndex**
- 强项：RAG原生能力强、索引丰富、开箱即用、文档处理成熟
- 弱项：流程编排弱、复杂Agent灵活性不足
- 选型：**纯知识库、文档问答、快速落地RAG**首选

2. **LangChain**
- 强项：组件最全、生态最强、记忆/工具/链式调用完善
- 弱项：复杂循环分支、长流程管控一般
- 选型：**通用大模型应用、单Agent、工具调用、普通业务流程**

3. **LangGraph**
- 强项：状态管理、分支、循环、中断、多智能体、长流程闭环
- 弱项：学习成本高，单纯RAG过重
- 选型：**复杂业务流程、多Agent、自主迭代RAG、人机审核流程**

**总结选型**
- 只做知识库 → LlamaIndex
- 工具助手、通用应用 → LangChain
- 复杂闭环、多角色协作、自主思考流程 → LangGraph
- 企业顶配：**LlamaIndex做检索层 + LangGraph做业务编排层**

## 3. 如何解决大模型RAG常见幻觉问题

1. 提示词强约束：只依据参考文档回答，不知道直接拒绝回答
2. 严控上下文，只送入高相关片段，减少干扰信息
3. 调低temperature，让模型输出更严谨保守
4. 开启答案溯源，回答标注文档来源
5. 使用Rerank重排过滤低相关内容
6. 优化分片粒度，语义切块提升上下文精准度
7. 多轮自检机制，生成答案后二次校验依据
8. 接入知识图谱做事实关系校验
9. 缩小模型生成自由性，禁止拓展无关内容
10. 构建评测数据集，长期检测幻觉率持续优化

## 4. 百万级文档海量知识库架构设计

1. **数据分层**
冷数据归档、热数据入库、增量实时更新
2. **分库分表**
按业务/部门/时间拆分向量库，隔离检索范围
3. **多级检索架构**
元数据过滤→粗召回→重排精筛→上下文压缩
4. **索引分层**
全局摘要索引+局部细粒度向量索引，提升检索效率
5. **向量库集群化**
Milvus/ES分布式部署，分片存储、负载均衡
6. **缓存架构**
热门问题、高频检索片段Redis缓存
7. **异步建库**
文档上传异步解析分片向量化，不阻塞业务接口
8. **权限隔离**

1. **检索层（数据层）**
文档解析、切片、向量化、向量库存储、多路检索、重排过滤
负责提供可靠、准确的参考上下文

2. 本地部署嵌入模型，全程离线向量化
3. 部署本地向量库：FAISS、Chroma、Milvus单机版
4. 替换框架内所有外网模型地址，指向本地内网接口
5. 关闭框架内置联网搜索、云端知识库、远程评测等外网依赖功能
6. 内网完成文档导入、切片、建索引

10. 配置内网权限、日志、本地监控，全程断网可用

## 8. 多轮对话RAG上下文丢失解决方案
1. **统一会话ID隔离对话，不同用户历史互不干扰**
2. **对话问题压缩重构**
将历史对话+当前问题合并，改写为完整无歧义新问题再检索
3. **分级记忆策略**
近期对话完整保留，远期对话摘要压缩，节省Token
4. **滑动窗口记忆**
只保留最近有效轮次，防止历史过长淹没当前问题
5. **检索侧带入历史语义**
不仅用当前问句检索，同时融合对话历史语义检索
6. **上下文动态裁剪**
临近模型窗口上限自动精简历史，不丢失核心指代信息
7. **实体记忆留存**
抽取用户信息、专有名词长期记忆，避免轮次冲刷遗忘
8. **多轮专属对话引擎**
使用CondenseQuestion专用对话引擎自动处理指代、省略语句
