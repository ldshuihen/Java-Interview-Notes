理论落地到实际架构，最考验候选人的是"分布式事务"怎么处理。这是微服务架构中最难啃、也最能拉开差距的一块。

###  Redis 面试深挖链路

这是一条非常经典的 Redis 面试深挖链路，面试官会顺着你的亮点一路追问。我来帮你梳理每个环节的核心答案和逻辑。---

![image-20260327093614281](面试链路.assets/image-20260327093614281.png)

#### ① 为什么引入 Redis？

面试官在问你**决策依据**，要回答"业务痛点 → 选型理由"的逻辑。

典型答法：项目存在高频读操作（如商品详情、用户 Session），直接打 DB 响应慢且 QPS 撑不住。Redis 基于内存、读写时延在微秒级，且支持丰富数据结构，适合做旁路缓存（Cache-Aside）。

------

#### ② 雪崩 / 穿透 / 击穿

| 问题     | 本质                                   | 核心方案                                                     |
| -------- | -------------------------------------- | ------------------------------------------------------------ |
| **雪崩** | 大量 key 同时过期，请求全打 DB         | TTL 加随机偏移量；多级缓存；熔断降级                         |
| **穿透** | 查询根本不存在的数据，缓存永远 miss    | 布隆过滤器拦截；缓存空值（短 TTL）                           |
| **击穿** | 单个热点 key 过期瞬间，大量并发涌入 DB | 互斥锁（分布式锁）保证只有一个线程重建；或对热点 key 设置"逻辑过期"永不真正过期 |

------

#### ③ 缓存与数据库一致性

这是最容易展开的考点，有几个方案要说清楚取舍：

**Cache-Aside（最常用）**：写操作先更新 DB，再删除缓存（而不是更新缓存）。删除而非更新是为了避免并发写导致缓存被旧值覆盖。

**延迟双删**：写 DB 前删一次缓存，写完 DB 后延迟一段时间再删一次，解决极短窗口内读请求把旧值写回缓存的问题。

**Canal 订阅 binlog**：异步监听 MySQL binlog，变更事件触发缓存失效或更新，实现最终一致性，适合对强一致要求不高但希望解耦的场景。

------

#### ④ Redis 挂了怎么办？

要从四个层面回答：

**持久化**：RDB（快照，适合备份，重启快）+ AOF（追加日志，数据更完整，可配置每秒 fsync）。生产通常两者同时开。

**主从复制**：一主多从，读写分离，从库异步同步。

**哨兵（Sentinel）**：监控主库存活，主库宕机自动选举新主，客户端通过哨兵发现新主地址。

**Cluster 集群**：数据分片到 16384 个 slot，每个分片一主多从，彻底解决单点和容量问题。面试说得出来"哨兵解决高可用、Cluster 解决扩展性"这句话就很加分。

------

#### ⑤ 热 key / 大 key

**热 key**（某个 key 请求量极大）：会打爆单个 Redis 节点，因为 Redis 单线程处理命令。解法：本地缓存（Caffeine）做二级缓存挡住大部分请求；或把热 key 复制成多个分片 key（`key_0` `key_1`…）分散到不同节点。

**大 key**（value 体积很大，如几 MB 的 Hash/List）：会阻塞 Redis 主线程（尤其是 `DEL` 大 key）、占用大量网络带宽、导致主从同步慢。解法：拆分为多个小 key；用 `UNLINK`（异步删除）代替 `DEL`；序列化时考虑压缩。

发现手段：`redis-cli --hotkeys`、`--bigkeys`，或 slowlog 分析。

#### 各层问题的答题要点

##### ① 为什么用线程池？

核心答法是讲三个代价：频繁创建销毁线程的系统开销大（每次 `new Thread` 都要内核态切换）；不加管理的并发数量失控会打垮下游；线程池统一管理任务队列、监控、拒绝策略，可观测可控制。

##### ② 线程数怎么定？

面试标准答法：CPU 密集型任务设 `N+1`（N 是 CPU 核心数，+1 是防止偶发缺页中断浪费 CPU），IO 密集型设 `2N`，但这是估算起点。生产环境推荐结合压测结果和**动态线程池**（如美团的 `DynamicTp`）来运行时调参，不要写死。

##### ③ 为什么不推荐 `Executors`？

`Executors.newFixedThreadPool` 背后用的是 `LinkedBlockingQueue`，队列容量是 `Integer.MAX_VALUE`，任务堆积会 OOM；`newCachedThreadPool` 则是线程数无上限，也会 OOM。阿里开发规范明确禁止使用，要手动 `new ThreadPoolExecutor` 显式传入所有参数。

##### ④ `execute()` 关键顺序——容易答错的地方

很多人以为"核心线程满了就扩容"，实际顺序是：**核心线程不够 → 先入队 → 队列满了才扩容到 max → max 也满了才拒绝**。非核心线程不是优先创建的，是最后的扩容手段。

##### ⑤ `Worker` 底层是什么？

`Worker` 实现了 `Runnable` 并继承 `AbstractQueuedSynchronizer`（AQS）。它持有一个 `Thread` 和一个 `firstTask`。启动后进入 `runWorker()` 死循环，先跑 `firstTask`，然后不断从 `workQueue.poll()` 或 `take()` 取任务。非核心线程用带超时的 `poll(keepAlive, unit)`，超时返回 null 后 Worker 自动退出，这就是"非核心线程空闲回收"的底层机制。

#### 第一张图：JVM 内存结构——「你知道 JVM 内存怎么分区的吗？」

这是所有追问的起点，面试官要的不只是背出区域名字，而是要你知道**每个区存什么、为什么会 OOM**。

记住一句话可以防止答错：**堆和方法区线程共享，虚拟机栈/本地方法栈/程序计数器线程私有。** JDK 8 之后方法区改名元空间（Metaspace），从 JVM 堆内存移到了**本地内存**，这个变化是高频考点。

#### 第二张图：GC 全景——「从 Minor GC 到 Full GC，对象经历了什么？」

这里是最容易追死的地方。面试官不满足于你说"新生代用复制算法，老年代用标记整理"，他会追问**对象是怎么从新生代晋升到老年代的，Full GC 的触发条件到底有几个**。

![image-20260327094503739](面试链路.assets/image-20260327094503739.png)

GC 这块最容易答漏的是 Full GC 的五个触发条件，以及动态年龄判断规则（不是只有 age ≥ 15 才晋升，Survivor 中相同年龄对象总大小超过 Survivor 空间一半时也会直接晋升）。

#### 第三张图：线上调优——面试官最想听的「实战感」

能说出内存结构和 GC 算法的人很多，但能说出"我在项目里遇到了什么问题、用哪些命令排查、怎么调参"的人才是他们想要的。

![image-20260327094520087](面试链路.assets/image-20260327094520087.png)

#### 把三张图串成一套完整答法

面试时的推荐表达顺序是这样的：

**第一层——内存结构**（30秒）：JVM 运行时数据区分为线程共享的堆和方法区，以及线程私有的虚拟机栈、本地方法栈、程序计数器。堆又分新生代和老年代，JDK8 以后方法区改为 Metaspace，放到了本地内存，不再受堆大小限制。

**第二层——GC 机制**（60秒）：新对象优先在 Eden 分配，Eden 满了触发 Minor GC，用复制算法把存活对象移到 Survivor，age 加一。age 达到阈值（默认 15）或者触发动态年龄判断则晋升老年代。老年代满了触发 Full GC，STW 时间长，高并发场景是性能杀手。垃圾回收器选型上，JDK8 低延迟场景用 CMS，JDK11+ 推荐 G1，超大堆低停顿可以上 ZGC。

**第三层——线上调优**（最加分，要主动提）：说一个你实际遇到的场景，比如"我们项目曾经出现过频繁 Full GC 导致接口超时，我用 `jstat -gcutil` 监控发现老年代使用率一直在 90% 以上，`jmap -histo` 看到某类对象数量异常多，最后定位到一个 ThreadLocal 没有 remove 导致的内存泄漏，修复后 Full GC 频率降到了每天个位数"。

B+ 树的关键答分点：叶节点之间有**双向链表**，这是它能支持范围查询和 `ORDER BY` 的根本原因；非叶节点只存键不存数据，所以每页能存更多键，树更矮（通常只有 3~4 层），一次查询只需 3~4 次磁盘 IO。

#### 把这四张图串成一套完整答法

面试中遇到"你了解 MySQL 索引吗"时，推荐这样展开：

**第一层——为什么用索引**（15秒）：索引本质是用有序的数据结构减少扫描行数，代价是额外的存储空间和写入时的维护成本，适合读多写少的查询列。

**第二层——为什么是 B+ 树**（30秒）：B+ 树非叶节点只存键，扇出大、树矮，减少磁盘 IO；叶节点之间有双向链表，天然支持范围查询和排序。这是哈希索引做不到的。

**第三层——聚簇索引和回表**（30秒）：InnoDB 主键是聚簇索引，叶节点存整行数据；二级索引叶节点只存主键值，查询非索引列时需要**回表**——拿着主键再查一次聚簇索引树。这就是为什么要设计**覆盖索引**，把查询列全塞进联合索引，让 `EXPLAIN` 的 `Extra` 显示 `Using index`，从而彻底消除回表。

**第四层——项目实战**（最加分）：主动说"我在项目里遇到一个慢查询，`EXPLAIN` 看到 `type: ALL`，`rows` 预估扫了 80 万行。分析发现是对 `create_time` 做了 `YEAR()` 函数导致索引失效，改成范围查询后 `type` 变成 `range`，查询从 3 秒降到 20 毫秒"。这句话的信息密度，是面试现场最值钱的内容。

![image-20260328202218875](面试链路.assets/image-20260328202218875.png)

#### 第三张图：MVCC 核心机制——ReadView 可见性判断

这是整条链路里最难讲清楚的地方。大多数人能说出"undo log 版本链"，但说不出 ReadView 的四个字段和可见性判断逻辑。**RC 和 RR 最本质的区别就在 ReadView 的生成时机**：RC 每次 SELECT 都生成一个新的 ReadView，所以能读到其他事务刚提交的最新数据（导致不可重复读）；RR 在事务内第一次读时生成 ReadView 并一直复用，后续无论其他事务怎么提交，看到的都是同一个快照（解决不可重复读）。这一句话能把面试官问的"为什么 RR 能解决不可重复读"解释得清清楚楚。

这是整条链路里最容易被追死的地方——"RR 解决了幻读吗"是个有陷阱的问题。---

#### 把四张图串成完整答法

这条链路的答题黄金顺序是：**ACID 底层 → 隔离级别矩阵 → MVCC 原理 → 幻读真相**。每一层都有一个容易被追死的细节。

##### ACID 的底层（15秒，说出机制才加分）

原子性靠 **undo log** 回滚，持久性靠 **redo log** WAL 预写（先写日志再写磁盘，宕机后可重放），隔离性靠 **锁 + MVCC** 组合实现，一致性是前三者共同保证的目的。能说出"每个特性靠哪个底层机制"这个层次，立刻拉开差距。

##### 隔离级别（20秒，说出区别而不只是背定义）

默认是 **RR 可重复读**。RC 和 RR 的核心区别是 ReadView 的生成时机：RC 每次 SELECT 都生成新快照，所以能读到别人最新提交的（不可重复读）；RR 事务首次读时生成并一直复用，所以前后读的是同一个快照（解决不可重复读）。

##### MVCC 的关键答分点（30秒）

每行记录有两个隐藏列：`trx_id`（最后修改的事务 ID）和 `roll_pointer`（指向 undo log 上一版本），形成版本链。ReadView 有四个字段：`creator_trx_id`（自己）、`m_ids`（活跃未提交事务集合）、`min_trx_id`（最小活跃 ID）、`max_trx_id`（下一个待分配 ID）。可见性判断：`trx_id < min_trx_id` 则一定可见；`trx_id >= max_trx_id` 则一定不可见；在 `m_ids` 中说明未提交，不可见；等于 `creator_trx_id` 是自己修改，可见。

##### 幻读真相——最容易被追死（说清楚才是满分）

面试官问"RR 解决幻读了吗"，正确答案是**部分解决**——这是最容易答错的地方。MVCC 的快照读确实看不到幻行，但**当前读**（`SELECT FOR UPDATE`、`UPDATE`、`DELETE`）会读最新数据，此时若没有锁保护，仍然能看到其他事务新插入的行。InnoDB 在 RR 下用 **Next-Key Lock**（行锁 + 间隙锁）来堵住这个漏洞：`SELECT FOR UPDATE` 会在扫描范围加 Next-Key Lock，阻止其他事务在间隙内 INSERT，彻底防止当前读场景下的幻读。

#### Spring 面试深挖链路全解

这条链路的追问顺序基本固定：**IOC 概念 → AOP 代理机制 → Bean 生命周期 → 循环依赖三级缓存 → 事务失效场景 → SpringBoot 自动配置**。每一层都会从你的回答中找漏洞继续往里追。

------

#### ① IOC — 控制反转与依赖注入

面试官想听的不是定义，是**你理解它解决了什么问题**。

IOC（Inversion of Control）的核心是：对象的**创建权和依赖关系的管理权**，从业务代码手里转移给 Spring 容器。DI（依赖注入）是 IOC 的具体实现方式——容器在创建对象时主动把依赖塞进去，而不是让对象自己 `new`。

**BeanFactory vs ApplicationContext** 是高频追问：

|                 | `BeanFactory`                 | `ApplicationContext`                          |
| --------------- | ----------------------------- | --------------------------------------------- |
| Bean 初始化时机 | 懒加载（第一次 `getBean` 时） | 容器启动时预初始化单例 Bean                   |
| 功能            | 基础 IOC 功能                 | 继承 BeanFactory + 事件发布 + 国际化 + AOP 等 |
| 使用场景        | 测试/轻量环境                 | 生产实际使用的                                |

**三种注入方式的取舍**：构造器注入（推荐，依赖不可变，便于测试）、Setter 注入（可选依赖）、字段注入（`@Autowired` 直接打在字段上，Spring 官方已不推荐，因为无法做单元测试）。

------

#### ② AOP — 面向切面编程

核心考点是**代理实现原理**，面试官会追问"JDK 动态代理和 CGLIB 的区别，什么时候用哪个"。

JDK 动态代理要求目标类**实现接口**，代理对象和目标对象实现同一接口，通过 `InvocationHandler.invoke()` 拦截方法调用。CGLIB 通过**继承目标类**生成子类作为代理，不要求接口，但 `final` 类和 `final` 方法无法代理。

Spring 5.x 默认优先使用 CGLIB（即使有接口），Spring Boot 2.x 也是默认 CGLIB。可以通过 `proxyTargetClass=false` 切换回 JDK 代理。

**五种通知类型**，面试要能区分执行顺序：

```
@Before → 目标方法 → @AfterReturning（正常）/ @AfterThrowing（异常）→ @After（无论如何）
@Around 包裹全程（最强大，包含以上所有节点的控制权）
```

**AOP 失效的常见场景**（高频追问）：同一个类内部方法调用，比如 `this.methodB()` 调用不走代理，AOP 不生效。解法是注入自身 `@Autowired ApplicationContext` 再 `getBean`，或者用 `AopContext.currentProxy()`。

------

#### ③ Bean 生命周期——面试最高频的题

能背出完整顺序的人就已经赢了大多数人，能说出**每个步骤的实际应用场景**才是满分。

```
1. 实例化（Instantiation）
   → 调用构造方法，创建 Bean 对象（此时属性还是默认值）

2. 属性填充（Populate Properties）
   → 依赖注入，@Autowired / @Value 填充

3. Aware 接口回调
   → BeanNameAware.setBeanName()
   → BeanFactoryAware.setBeanFactory()
   → ApplicationContextAware.setApplicationContext()

4. BeanPostProcessor.postProcessBeforeInitialization()
   → 所有 Bean 的前置处理（AOP 代理就在这一带创建）

5. 初始化
   → @PostConstruct 方法
   → InitializingBean.afterPropertiesSet()
   → init-method（XML 配置指定）

6. BeanPostProcessor.postProcessAfterInitialization()
   → 后置处理（此时 Bean 已完全就绪）

7. Bean 使用阶段（放入单例池 singletonObjects）

8. 销毁
   → @PreDestroy 方法
   → DisposableBean.destroy()
   → destroy-method
```

**关键记忆点**：AOP 代理是在 `BeanPostProcessor` 后置处理阶段（第 6 步）创建的，不是在实例化时。这就是为什么循环依赖要引入三级缓存——需要在属性注入阶段提前暴露还没被代理的早期 Bean。

------

#### ④ 循环依赖与三级缓存——最容易被追死的题

先说清楚**场景**：A 依赖 B，B 依赖 A，两个都是单例。

Spring 的解法是**三级缓存**，三个 Map 各司其职：

```java
// 一级缓存：完全初始化好的单例 Bean
Map<String, Object> singletonObjects;

// 二级缓存：早期暴露的 Bean（已实例化，未完成初始化）
Map<String, Object> earlySingletonObjects;

// 三级缓存：Bean 的工厂对象（ObjectFactory，用于创建早期引用）
Map<String, ObjectFactory<?>> singletonFactories;
```

**完整解决流程**（这个顺序是面试得分关键）：

```
① 开始创建 A
② A 实例化（new A()），此时 A 还没注入依赖
③ 把 A 的 ObjectFactory 放入三级缓存 singletonFactories
④ A 开始注入依赖，发现需要 B → 去创建 B
⑤ B 实例化，把 B 的 ObjectFactory 放入三级缓存
⑥ B 注入依赖，发现需要 A
⑦ 从三级缓存取出 A 的 ObjectFactory，调用 getObject() 得到 A 的早期引用
   （如果 A 需要被代理，这里返回的是代理对象）
⑧ 把早期 A 放入二级缓存，从三级缓存移除
⑨ B 完成初始化，放入一级缓存
⑩ A 拿到完整的 B，完成属性注入和初始化，放入一级缓存
```

**为什么需要三级缓存，两级不够？** 这是最容易被追问的点。二级缓存可以解决普通循环依赖，但如果 A 需要被 AOP 代理，在 B 注入 A 时就需要拿到 A 的**代理对象**而不是原始对象。三级缓存中的 `ObjectFactory` 能在需要时按需创建代理，保证 B 拿到的和最终放入一级缓存的是同一个代理对象，保持一致性。如果只有二级缓存，要么提前代理（破坏 Bean 生命周期顺序），要么无法保证引用一致性。

**哪些循环依赖 Spring 解决不了**：

- 构造器注入的循环依赖——实例化时就需要对方，但对方还没创建，三级缓存此时还没有任何缓存可用，直接报 `BeanCurrentlyInCreationException`
- `prototype` 作用域的循环依赖——每次都新建，没有缓存机制，也解决不了

------

#### ⑤ Spring 事务失效场景——"你们项目踩过坑吗？"

这道题是考实战经验的，要主动说出你遇到过的场景：

**方法必须是 public**：`@Transactional` 基于 AOP 代理，`private`/`protected` 方法代理无法拦截，事务不生效。

**同类内部调用**：`this.methodB()` 绕过了代理对象，事务注解失效。这是最常见的坑。

**异常类型不匹配**：默认只回滚 `RuntimeException` 和 `Error`，`checked Exception`（如 `IOException`）不回滚，需要 `rollbackFor = Exception.class`。

**异常被吞掉**：`try-catch` 把异常捕获后没有重新抛出，Spring 感知不到异常，不回滚。

**传播行为理解错误**：`REQUIRES_NEW` 会挂起当前事务开启新事务，两个事务独立提交/回滚，新事务回滚不影响外部事务。`NESTED` 是嵌套事务，子事务回滚只影响到 savepoint，外部事务可以捕获后继续。

------

#### ⑥ SpringBoot 自动配置原理

`@SpringBootApplication` = `@SpringBootConfiguration` + `@EnableAutoConfiguration` + `@ComponentScan`。

核心是 `@EnableAutoConfiguration`，它通过 `@Import(AutoConfigurationImportSelector.class)` 触发自动配置加载。`AutoConfigurationImportSelector` 读取 `META-INF/spring.factories`（SpringBoot 3.x 改为 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`）文件中所有 `EnableAutoConfiguration` 的配置类，但不是无脑全加载——每个配置类上都有 `@ConditionalOnClass`、`@ConditionalOnMissingBean` 等条件注解，只有条件满足时才真正生效。

这套机制让引入一个 `starter` 依赖就能自动配置好所有 Bean，而不需要手动 XML 配置。

------

#### 串成面试答法的关键句子

遇到"说说 Spring 核心原理"时，推荐这个结构：

> "Spring 核心是 IOC 容器，负责管理 Bean 的创建和依赖注入。AOP 是基于动态代理实现的横切关注点，默认用 CGLIB。Bean 生命周期从实例化、属性注入、Aware 回调、BeanPostProcessor 前置、初始化、BeanPostProcessor 后置一直到销毁，AOP 代理在后置处理阶段生成。循环依赖通过三级缓存解决，三级缓存的 ObjectFactory 能按需生成代理对象保证引用一致性，但构造器注入的循环依赖无法解决。事务实现也是基于 AOP，同类内部调用和 checked 异常未声明是最常见的失效场景。"

这段话 100 字，能覆盖面试官 80% 的追问入口。说完后面试官会选一个点往里钻，你再展开对应层的细节。

#### 把四张图串成完整答法

面试时遇到"说说你们项目里 ES 是怎么用的"，推荐这个展开结构：

**为什么用 ES**（15秒）：MySQL `LIKE %关键词%` 无法走索引，全表扫描百万行太慢；ES 的倒排索引专为全文检索设计，分词后词项定位到文档列表，复杂的多字段聚合查询 MySQL 也吃力。

**倒排索引核心原理**（30秒）：文档写入时经分词器切词，建立"词 → 文档列表"的映射。Term Index 用 FST 压缩后常驻内存，避免磁盘 IO；Posting List 用 Roaring Bitmap 压缩，集合求交/并极快。这就是为什么 ES 全文搜索比 MySQL LIKE 快几个数量级。

**写入不实时的原因**（30秒）：写入先进 In-memory buffer，同时写 translog 保证宕机可恢复。每隔 1 秒执行一次 refresh，把 buffer 转成内存 segment 才能被搜索到——所以 ES 是**近实时（NRT）**，不是严格实时。flush 操作才把 segment fsync 到磁盘同时清空 translog。

**MySQL 同步方案**（最加分，说自己项目用的）：我们用 Canal 订阅 MySQL binlog，Canal 伪装成从库接收 binlog 事件，解析后投入 Kafka，消费端幂等写入 ES（用 MySQL 主键作为 ES 文档 ID，天然支持 upsert）。这样业务代码完全不需要改动，Canal 支持断点续传，即使 ES 短暂宕机也不会丢数据。定期跑一个对账任务比较两边数据条数，发现差异自动补偿。

面试遇到"说说 Java 并发锁"，推荐按这个层次展开：

##### synchronized 的核心答法

`synchronized` 底层依赖 JVM 对象头的 **Mark Word**，实现了三级锁升级。只有一个线程访问时用**偏向锁**，Mark Word 只存一次线程 ID，后续进入只需比较 ID 不需要 CAS；有第二个线程竞争时升级为**轻量级锁**，失败线程自旋 CAS 等待；自旋超过阈值升级**重量级锁**，线程挂起进入 OS 互斥量，排队等在 `ObjectMonitor` 的 `EntryList` 里。升级单向不可逆，JDK 15 废弃了偏向锁。

##### ReentrantLock vs synchronized

`ReentrantLock` 能做到 `synchronized` 做不到的几件事：响应中断（`lockInterruptibly()`）、超时获取（`tryLock(timeout)`）、公平锁（按入队顺序）、绑定多个 `Condition` 实现精细化等待/通知。但必须手动 `unlock()`，要放在 `finally` 里。默认选 `synchronized`，需要这些高级特性才换 `Lock`。

##### AQS 一句话讲清楚

AQS 是一个**模板框架**：用 `volatile int state` 表示同步状态，用 CLH 双向链表存等待线程，框架提供入队/出队/阻塞/唤醒的通用逻辑，子类只需实现 `tryAcquire`/`tryRelease` 定义"什么情况算抢到锁"。`ReentrantLock` 的 state 是重入次数，`Semaphore` 的 state 是剩余许可数，`CountDownLatch` 的 state 是倒计时计数——同一套框架，不同语义。

##### CAS 的关键答分点

CAS 底层是一条 CPU 原子指令 `CMPXCHG`，读-比较-写三步不可打断。ABA 问题用 `AtomicStampedReference` 加版本号解决。CAS 是乐观并发，适合冲突少的场景；冲突严重时自旋白耗 CPU，这时候反而不如锁。

##### volatile 的最常见坑

`volatile` 保证可见性和有序性，但**不保证原子性**。`i++` 是读-改-写三步，即使 `i` 是 `volatile`，多线程下仍不安全，需要换 `AtomicInteger`。双重检查锁（DCL 单例）里 `instance` 必须加 `volatile`，否则 `new` 操作的指令重排序会导致其他线程拿到一个未完全初始化的对象。

##### 服务拆分（15秒）

拆分核心原则是**单一职责+高内聚低耦合**，按业务能力拆，不按技术层拆。粒度不是越小越好——服务间网络调用的代价远高于进程内调用，拆太细会导致调用链过长、分布式事务问题爆发。DDD 的限界上下文是比较好的拆分参考边界。

##### 注册发现（30秒）

国内主流选 Nacos。服务启动时向 Nacos 注册实例（IP+端口+元数据），定期发心跳续约；消费方订阅服务列表，Nacos 推送变更通知，本地缓存实例列表；调用时经过 Ribbon/LoadBalancer 做负载均衡，OpenFeign 在上面封装了声明式调用。Nacos 的核心优势是同时支持 AP（临时实例，注册中心）和 CP（持久实例，配置中心），一个中间件两种能力。

##### 熔断限流（30秒）

选 Sentinel。熔断器三态：CLOSED 正常→失败率超阈值→OPEN 全部拒绝走降级→计时到期→HALF_OPEN 放一个探测请求→成功回 CLOSED。限流算法推令牌桶，支持突发流量；漏桶平滑但不容突发。降级兜底要提前写好 Fallback，不能返回空或报错影响用户体验。

##### 分布式事务（最重点，要主动说选型理由）

用 Seata AT 模式——业务代码零侵入，一阶段执行业务 SQL 同时记录 undo_log，二阶段提交直接删 undo_log，回滚用 undo_log 反向补偿，整个过程框架自动完成。对强一致要求高的场景（比如金融扣款）才用 TCC，因为需要业务侵入写 Try/Confirm/Cancel 三套逻辑，成本高但隔离性强。高并发下允许最终一致的场景用**本地消息表+MQ**，把消息插入和业务 SQL 放一个本地事务，定时任务扫描投递，消费端做幂等，简单可靠。

> 说完分布式事务，面试官大概率会追问"TCC 的幂等性怎么保证"或"Seata 的全局锁冲突怎么解决"——点击图中节点可以继续深挖这些细节。

##### ① 为什么限流（10秒）

限流是系统的**最后一道防线**，保护下游不被打垮。它和熔断降级的区别在于：限流是**主动预防**（我知道系统最多承受多少，提前卡住），熔断是**被动响应**（发现下游出问题了，自动断开）。两者不互斥，生产环境通常一起用。

##### ② 四大算法（面试核心，30秒）

计数器最简单但有**临界突刺**（两个窗口边界可瞬间通过 2 倍流量）；滑动窗口解决了临界问题，Sentinel 默认用它；漏桶出口速率恒定，适合**调用有严格 QPS 限制的第三方接口**；令牌桶允许突发消费积累的令牌，**大多数场景首选令牌桶**。面试中说出"漏桶不容突发，令牌桶可以"这个核心区别就够了。

##### ③ Sentinel（30秒）

Sentinel 的核心是 **Slot 链**：每个请求依次经过 NodeSelector、ClusterBuilder、**Statistic**（用 LeapArray 滑动窗口统计 QPS/线程数/RT）、ParamFlow（热点参数）、System（系统自适应）、Flow/Degrade（流控和熔断判断）。任意 Slot 拦截则抛 BlockException，走 BlockHandler 降级。流控效果中最常考的是 **Warm Up**——令牌桶预热，冷启动时从低阈值渐进升到目标值，防止流量突增压垮刚启动的服务。

##### ④ 分布式限流（最加分，说项目里怎么做的）

单机限流在集群部署时各节点独立计数，无法保证全局阈值。生产最常用的是 **Redis + Lua 脚本**：滑动窗口用 ZSet（score 存时间戳），每次请求先清除过期数据、统计数量、判断是否超限、写入当前请求，这四步封装在 Lua 脚本里原子执行，避免多命令之间的并发问题。SpringCloud Gateway 内置的 `RequestRateLimiter` 底层也是这套，可以按 IP / 用户 / 路由设置不同阈值，是业务无侵入的推荐方案。

#### 把四张图串成完整答法

##### ① 为什么用 MQ（15秒）

三个核心价值：**异步**（下单后发消息通知库存/积分/短信，主流程不等待，响应快）、**解耦**（生产者不依赖消费者接口，消费者宕机不影响生产者）、**削峰**（秒杀瞬间流量打到 MQ，消费者按自身节奏消费，保护下游数据库）。

##### ② 消息丢失（30秒，三段说清楚）

生产者侧：Kafka 配 `acks=all`，等所有 ISR 副本写入后才算成功；发送失败重试，配合本地消息表兜底。Broker 侧：Kafka 开多副本，`min.insync.replicas=2`；RocketMQ 用同步刷盘 + 主从双写。消费者侧：**关闭自动提交 offset**，业务逻辑处理完成后再手动 `commitSync()`；消费失败不要吞异常，走死信队列 + 告警人工处理。

##### ③ 重复消费（30秒，说出方案取舍才加分）

根本原因是 MQ 的 At-least-once 语义——宕机重启后重投消息，无法避免重复。解法在消费端设计幂等。最推荐**数据库唯一索引**（`INSERT IGNORE` 消息日志表，与业务操作同一事务，强一致）；高并发场景用 **Redis SETNX**（性能好但有极小不一致窗口）；有业务状态机的用 **WHERE status='pending' 的条件 UPDATE**（影响行数为 0 说明已处理，天然幂等）。

##### ④ 顺序消费（30秒，说出局部有序是核心）

全局有序（1 个分区 1 个消费者）吞吐太低，生产不用。实际做法是**局部有序**：按业务 Key（如 `orderId`）哈希路由到固定分区，同一分区内消息天然有序，消费时单线程串行处理该分区。常见坑是消费者拉取消息后用线程池并行处理导致乱序——解法是按 Key 哈希分配到内存队列，每个内存队列单线程消费。

##### ⑤ 消息积压紧急处理（加分项）

主动说"我们生产遇到过"：优先扩分区再扩消费者实例（消费者数不能超过分区数，超了没用）；如果消费者处理耗时过长，临时开快速消费者把消息转储到新的大容量 Topic，再多开消费者并行消化；已过期的消息不可恢复，只能从 DB 对账补发，消费端做好幂等。

##### ⑥ 选型建议（10秒）

大数据/日志采集选 **Kafka**（百万 TPS，顺序写+零拷贝）；电商/金融选 **RocketMQ**（支持事务消息和延迟消息，阿里系生态好）；灵活路由/任务队列选 **RabbitMQ**（Exchange 模型灵活，延迟最低但吞吐相对弱）。

##### 单例（30秒，代码级才加分）

五种写法从差到好：懒汉（线程不安全）→ 同步方法（性能差）→ 饿汉（简单可靠但无懒加载）→ DCL 双重检查锁（需要 `volatile`，原因是 `new` 的指令重排序）→ **枚举（最优解）**。枚举由 JVM 保证线程安全和单次初始化，还能防反射攻击和序列化破坏（其他写法都有这两个漏洞），是 Effective Java 明确推荐的方式。

##### 工厂三兄弟（20秒，说出演进驱动力）

简单工厂（`switch` 决定创建哪个）→ 问题是新增产品要改工厂类，违反开闭原则 → 升级到工厂方法（每种产品对应一个工厂，新增只加类不改类）→ 需要保证一族产品的一致性时（比如换数据库品牌，所有相关对象都要换）→ 升级到抽象工厂（一个工厂方法创建一族相关产品）。

##### 代理模式（30秒，说出底层原理）

JDK 动态代理要求目标类有接口，运行时生成 `$Proxy0` 实现同一接口，每个方法调用转发到 `InvocationHandler.invoke()`，再通过反射调用目标方法。CGLIB 用 ASM 字节码生成目标类的子类，`MethodInterceptor.intercept()` 拦截方法，不要求接口但 `final` 类/方法无法代理。Spring 5.x 默认用 CGLIB。

##### Spring 中的模式（最加分的综合回答）

工厂：`BeanFactory` 是工厂方法，`FactoryBean` 是生产 Bean 的工厂（注意两者是完全不同的概念，这是高频考点）。单例：Spring Bean 默认单例，存在 `DefaultSingletonBeanRegistry` 的 `ConcurrentHashMap` 里。代理：AOP 在 `AbstractAutoProxyCreator` 后置处理阶段生成代理，`@Transactional` 通过 `TransactionInterceptor` 拦截实现。模板方法：`JdbcTemplate` 封装 JDBC 骨架，`RowMapper` 回调处理可变部分。观察者：`ApplicationEvent` + `@EventListener`，用户注册送积分/发邮件的典型解耦场景。责任链：Spring Security `FilterChain` 和 `BeanPostProcessor` 处理链。策略：用 `Map<String, Strategy>` 注入所有策略实现，按 key 选策略，消除支付方式判断的大量 if-else。

##### ① 什么时候分库分表（20秒，要说出判断标准）

不是一上来就分，分库分表引入的复杂度极高。优先顺序应该是：加索引优化 SQL → 读写分离 → 归档历史数据 → 垂直分库/分表 → 最后才是水平分表。单表数据超过**千万级且已经索引优化过、响应时间仍不达标**，才考虑分表。分库解决的是写并发瓶颈（单库 QPS 上限），分表解决的是单表数据量问题，两个问题要分清楚。

##### ② 分片策略（30秒，说出选型理由）

生产电商最常用的是**用户 ID hash 分库 + 时间 range 分表**的复合策略：按用户 ID 哈希确保同一用户数据在同一库（"我的订单"查询不散射），按月/季度 range 分表控制单表大小（历史表冷归档）。分片键选择的核心原则是高基数 + 查询高频 + 值不可变，用户 ID 和订单 ID 是最优选择。

##### ③ 跨库查询难题（最容易被追问，40秒）

说三个最重要的：JOIN 拆成多次单表查询（业务层 In 查询）；深分页 `LIMIT 100000,10` 改为游标分页 `WHERE id > last_id`，因为深分页在 N 个分片上每片都要取 100010 条再合并，代价不可接受；AVG 绝对不能各分片平均再求平均，必须返回 `(SUM, COUNT)` 汇总后计算。非分片键的复杂搜索通常引入 ES，把查询结果的主键返回后再回数据库查详情。

##### ④ 分布式 ID（30秒，雪花算法要说出结构和坑）

推荐**雪花算法**：64 位 long = 1 位符号位 + **41 位毫秒时间戳**（约 69 年）+ **10 位机器 ID**（最多 1024 节点）+ **12 位序列号**（每毫秒 4096 个 ID），每秒约 400 万个，趋势递增，对 B+ 树友好，无中心化依赖。最大的坑是**时钟回拨**——NTP 同步可能让时间倒退，导致生成重复 ID。解法：小回拨（≤5ms）等待追上；大回拨直接报错快速失败，或用扩展位计数器区分。生产中用美团 Leaf 或百度 UidGenerator，已经解决了这些问题。UUID 无序会导致 B+ 树频繁页分裂，写性能差，不推荐做 MySQL 主键。

##### ⑤ 平滑迁移（面试加分项）

单库迁移到分库分表不能停服，标准做法是**双写迁移**：代码改造为同时写旧库和新分库，存量数据后台 Job 全量迁移，迁移完成后对账校验一致性，最后切读流量（先 1% 灰度→10%→100%），确认稳定后停止写旧库。整个过程可随时回滚。

### HashMap的面试深挖链路

这是 Java 面试里追问最深、最容易被"扣细节"的一条链路。分四张图讲透：追问路径 → 数据结构与 put 流程 → 扩容机制 → 线程安全演进。

#### 第一张图：面试追问路径

![image-20260330214624466](面试链路.assets/image-20260330214624466.png)

#### 第二张图：数据结构 + put 完整流程——「说说 put 的过程」

这是面试官最常考的核心题，能把 put 流程的每一步说清楚，就已经超过大多数人了。hash 扰动函数 `h ^ (h >>> 16)` 的目的是让高 16 位也参与到下标计算中。因为数组长度通常不大，`(n-1) & hash` 实际上只用到了 hash 的低位，高位信息全部浪费，碰撞概率更高。异或后高低位信息都被混入，分布更均匀。

![image-20260330214635995](面试链路.assets/image-20260330214635995.png)

#### 第三张图：扩容机制 + JDK7 死循环——面试最深的坑

JDK8 改了头插为尾插，消灭了死循环，但**数据覆盖**和 **size 统计不准**的问题依然存在——所以多线程环境下 JDK8 的 HashMap 仍然不安全，必须换 ConcurrentHashMap。

![image-20260330214644812](面试链路.assets/image-20260330214644812.png)

#### 第四张图：ConcurrentHashMap 演进 + Map 家族全对比---

![image-20260330214653323](面试链路.assets/image-20260330214653323.png)

#### 把四张图串成完整答法

##### ① 数据结构（20秒）

JDK7 是**数组 + 链表**；JDK8 是**数组 + 链表 + 红黑树**。当链表长度 ≥ 8 且数组长度 ≥ 64 时链表转红黑树（查询 O(n) → O(log n)）；节点数 ≤ 6 时红黑树退化回链表。数组长度 < 64 时优先扩容而不树化——这个细节大多数人答不出来。

##### ② put 流程（30秒，步步说到位）

① 计算 `hash = h ^ (h >>> 16)`（扰动，让高位参与寻址）→ ② `index = (n-1) & hash`（位运算代替取模，要求 n 是 2 的幂次）→ ③ 数组为空则 `resize()` 初始化（懒加载）→ ④ 桶位为空直接放 → ⑤ key 相同则覆盖 → ⑥ 链表尾插，长度 ≥ 8 且容量 ≥ 64 则树化 → ⑦ `++size > threshold` 则扩容。

##### ③ 扩容（30秒，说出 JDK8 的优化）

扩容两倍，JDK8 用 `(e.hash & oldCap) == 0` 判断节点是否搬移——等于 0 留原位，不等于 0 移到 `index + oldCap`，**无需重新计算 hash**，是很精妙的位运算优化。JDK7 头插法在并发扩容时会产生**环形链表导致 get 死循环**（CPU 100%）；JDK8 改成尾插法消灭了死循环，但并发 put 仍有数据覆盖和 size 统计不准的问题，多线程必须用 CHM。

##### ④ ConcurrentHashMap（30秒，说出演进差异）

JDK7 分段锁 Segment（默认 16 段，最大并发 16），本质是粗粒度的 `ReentrantLock`；`size()` 需要遍历所有 Segment，复杂。JDK8 彻底重写：桶位**为空用 CAS 插入**（无锁），**非空用 synchronized 锁桶头节点**（锁粒度从 Segment 缩到单个桶），size 用 `CounterCell` 分段计数（类 LongAdder）。`get` 全程不加锁——Node 的 `val` 和 `next` 都是 volatile，保证读到最新值。

##### ⑤ ConcurrentHashMap 为什么不允许 null（加分细节）

HashMap 允许 null 键/值，CHM 不允许。原因是在并发场景下存在**二义性**：`get(key)` 返回 null 无法区分"key 不存在"还是"value 本身是 null"——单线程下可以用 `containsKey` 确认，但并发环境中两次操作之间可能发生修改，这个判断没有意义，所以 CHM 直接禁止 null，避免混淆。

这是最容易用"说了半天说不清楚"来暴露知识深度的题。要从**线程模型**和**系统调用**两个维度来讲。epoll 比 select/poll 快的本质原因：select/poll 每次调用都要把 fd 集合从用户态拷贝到内核，内核 O(n) 扫描所有 fd 找就绪的；epoll 用红黑树维护 fd（增删 O(log n)），内核回调把就绪 fd 加入链表，`epoll_wait` 直接返回就绪链表，应用无需再遍历，**时间复杂度 O(1)**。

##### ① BIO 的问题（15秒）

BIO 每个连接需要一个线程，1 万个连接就需要 1 万个线程——线程创建销毁开销大，上下文切换频繁，内存消耗巨大（每个线程默认 1MB 栈），这就是经典的 C10K 问题。

##### ② IO 模型演进（30秒）

五种 IO 模型核心是两个维度：**是否阻塞数据就绪阶段**（BIO 阻塞等，NIO 立即返回）和**是否阻塞数据拷贝阶段**（同步 IO 都阻塞，AIO 不阻塞）。生产最常用的是 IO 多路复用——一个线程用 epoll 监听多个 fd，有事件时唤醒处理，这就是 NIO 的底层。epoll 用红黑树管理所有 fd，就绪时通过内核回调加入就绪链表，`epoll_wait` 直接返回就绪的 fd，O(1) 复杂度，支持百万连接。Netty 没有用 AIO 是因为 Linux 的 AIO 实现不成熟，epoll 的性能已经足够。

##### ③ Netty（30秒，说出解决了什么）

主从 Reactor 模型：BossGroup 只负责 accept 新连接（通常 1 个 EventLoop），WorkerGroup 负责 IO 读写（默认 CPU×2 个 EventLoop）。每个 EventLoop 是单线程事件循环，Channel 的所有操作都在同一线程内，无需加锁。ChannelPipeline 是责任链，Handler 按顺序处理编解码和业务逻辑。Netty 最重要的是解决了：ByteBuffer 的 flip/clear 易错问题（换成双指针 ByteBuf）、epoll 空轮询 Bug（计数超阈值重建 Selector）、粘包拆包（LengthFieldBasedFrameDecoder）。

##### ④ TCP 三次握手（30秒，说出"为什么"才加分）

三次握手是为了：双方都确认对方的发送和接收能力正常，且消除历史连接的干扰。两次握手不够是因为：一旦旧的 SYN 延迟到达服务端，服务端会误以为是新连接而分配资源，浪费且无法消除。四次挥手是因为 TCP 全双工，主动方发 FIN 只表示自己不再发数据，被动方可能还有数据要发，两个方向独立关闭需要四步。TIME_WAIT 等 2MSL 是为了确保最后的 ACK 到达对方，以及让网络中残留报文彻底消失，避免新连接收到旧连接的报文。

##### ⑤ 零拷贝（加分项）

传统文件发送经历 4 次拷贝（磁盘→内核缓冲区→用户缓冲区→Socket 缓冲区→网卡）和 4 次用户态/内核态切换。`sendfile` 系统调用让数据直接从内核缓冲区传到网卡，跳过了用户态，减少到 2 次拷贝、2 次切换。Kafka 用 sendfile 发送日志文件，Netty 用 CompositeByteBuf 避免多个 ByteBuf 合并时的内存拷贝。

四大核心：`ChatModel`（文本对话）、`EmbeddingModel`（文本向量化）、`VectorStore`（向量存储检索）、`ChatMemory`（对话记忆）。`ChatClient` 是更高层的流式 API，通过 Advisor 链实现横切关注点。Anthropic 的 Prompt Caching 支持最多 90% 的成本节省，这类模型级特性都通过统一接口透明暴露。

##### ③ RAG（面试核心，40秒）

RAG 分两阶段：**离线构建**（ETL Pipeline：读文档 → 文本分割 → Embedding 向量化 → 存入向量数据库）和**在线查询**（用户问题向量化 → 相似搜索 TopK → 检索结果拼入 Prompt → LLM 基于上下文生成答案）。`QuestionAnswerAdvisor` 自动完成在线阶段的检索注入。RAG 的核心价值是让 LLM 回答基于企业私有知识库而不是只靠训练数据，同时减少幻觉。优化方向：混合搜索（向量+关键词）、Reranking 重排序、查询改写。

##### ④ Function Calling 和 MCP（30秒）

`@Tool` 注解标注一个 Spring Bean 方法，Spring AI 自动生成工具描述发给 LLM，LLM 决定是否调用、参数是什么，Spring AI 反射执行并把结果回传，LLM 基于工具返回值生成最终自然语言答案。这让 AI 从"只能聊天"变成"能操作系统"。Spring AI 1.1 的重大特性是 MCP（Model Context Protocol）集成，它是 Anthropic 提出的跨厂商标准化工具协议，比 Function Calling 更通用，可以把 Spring Boot 应用直接暴露为 MCP Server 供任何支持 MCP 的 AI 客户端调用。

##### ⑤ 生产实战（加分项，主动说出来）

Token 成本控制：开启 Prompt Caching（相同 system prompt 缓存，成本降 90%）；简单任务用小模型（gpt-4o-mini 比 gpt-4o 便宜 20 倍）；非核心场景部署本地 Ollama。幻觉防治：RAG 提供可信上下文 + Prompt 明确约束"只基于以下内容回答" + temperature 调低到 0.1-0.3。可观测性：Spring AI 内置 Micrometer 集成，自动记录 Token 消耗和调用延迟，接 Prometheus + Grafana 监控。
