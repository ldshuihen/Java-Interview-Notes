------

### 1. 突发流量（Traffic Spike）

#### 定义

在**极短时间内流量突然暴涨**，远高于系统平时的流量水平，但持续时间通常较短。

通常表现为：

- QPS 在几秒内快速增长
- 系统瞬时负载急剧上升

例如：

```
平时 QPS：2000
突发 5 秒内：20000
```

------

#### 典型场景

1. APP 推送消息
2. 社交媒体传播
3. 热点新闻
4. 明星事件
5. 短视频爆款

例如：

某明星结婚 → 搜索接口瞬间暴涨。

------

#### 技术特征

特点：

- **增长速度快**
- **持续时间短**
- **瞬时并发极高**

系统压力集中在：

- API Gateway
- 应用服务器
- Redis
- 数据库

------

#### 风险影响

如果系统没有做好保护，可能导致：

- 线程池耗尽
- CPU 飙升
- 服务响应变慢
- 服务不可用

严重情况：

```
突发流量 → 服务阻塞 → 请求堆积 → 服务雪崩
```

------

#### 常见治理手段

常见方案：

1. 限流（Rate Limit）

例如：

```
令牌桶
漏桶
Guava RateLimiter
```

1. 接口降级
2. 缓存热点数据
3. CDN 缓存
4. 弹性扩容

面试高分回答：突发流量（高并发突增）怎么扛、怎么解决、怎么兜底

#### 一、核心本质

**瞬时请求量远超系统常规承载能力**，容易引发：接口超时、CPU打满、DB打爆、MQ堆积、服务雪崩、OOM。
处理原则：**先限流兜底、再削峰缓冲、最后扩容优化**。

#### 二、应急处理（线上突发，立刻止血）

1. **限流**
   接口限流、IP限流、用户维度限流、网关层全局限流，拒绝非法/低频非核心请求，保护核心业务。
2. **降级**
   非核心功能降级（首页推荐、积分、活动、评论），只保下单、支付、核心链路。
3. **熔断**
   第三方接口、弱依赖服务熔断，避免慢接口拖垮整体链路，防止连锁雪崩。
4. **横向扩容**
   快速加服务节点、加MQ消费者实例，提升整体吞吐。
5. **读写分离/临时缓存兜底**
   热点数据全部走Redis，减少DB直接查询压力。

#### 三、架构层解决方案（长效抗突发）

##### 1. 网关层（第一层防护）

- 统一入口：**Spring Cloud Gateway / Nginx**
- 实现：限流、黑名单、流量整形、请求过滤、防刷
##### 2. 缓存削峰

- 热点商品、首页数据、配置全部缓存
- 缓存多级：本地Caffeine + 分布式Redis
- 缓存预热、热点隔离，避免缓存击穿/穿透/雪崩
##### 3. 消息队列异步削峰

- 高并发写请求**同步改异步**
- 秒杀、下单、日志、通知通过MQ削峰，错峰消费
- 应对瞬间洪峰，削峰填谷
##### 4. 数据库抗压

- 读写分离：主库写、从库读
- 分库分表：大表拆分，分散压力
- 禁止大事务、禁止全表查询、优化慢SQL
- 热点行隔离、库存Redis预扣减
##### 5. 服务隔离

- 核心/非核心业务集群隔离
- 线程池隔离、资源隔离，互不影响

#### 四、三大经典高并发问题配套解决

1. **缓存穿透**：布隆过滤器、空值缓存、接口校验
2. **缓存击穿**：互斥锁、热点数据永不过期
3. **缓存雪崩**：过期时间加随机值、集群高可用、多级缓存

#### 五、提前预防（日常优化）

1. 压测：定期全链路压测，摸清系统极限QPS
2. 监控告警：QPS、CPU、负载、队列堆积、DB连接数实时告警
3. 弹性扩容：接入容器编排，自动弹性伸缩
4. 流量预估：大促、活动提前预案，提前扩容+预热缓存

#### 六、极简面试背诵版

遇到突发流量，首先在网关做限流、非核心业务降级、弱依赖服务熔断，防止服务雪崩；
通过横向扩容提升服务能力，利用Redis缓存扛热点、MQ异步削峰填谷；
数据库做读写分离、优化慢SQL保障底层稳定；
日常通过压测评估系统上限、完善监控，大促提前扩容预热，从源头抵御突发流量。

#### 定义

在某一时间段出现 **持续的大规模访问流量高峰**。

和突发流量不同：

| 类型     | 特点                 |
| -------- | -------------------- |
| 突发流量 | 突然出现、持续时间短 |
| 流量洪峰 | 持续时间长、规模大   |

------

#### 典型场景

1. 电商大促

如：

- 双十一
- 618

1. 直播活动
2. 春节抢票
3. 游戏开服

------

#### 技术特征

特点：

- 持续高 QPS
- 长时间高负载
- 系统资源长期紧张

例如：

```
平时 QPS：5000
活动期间：50000

```

------

#### 风险影响

容易导致：

- CPU 长时间满负载
- 数据库压力过大

------

主要策略：

1. **容量规划**

提前压测：

```
压测目标 = 峰值流量 × 安全系数
```

1. **水平扩容**

例如：

```

```

1. **CDN 分流**

静态资源走 CDN。

1. **缓存化**

热点数据缓存。

1. **异步化**

使用 MQ 削峰。

### 3 流量倾斜（Traffic Skew）

#### 定义

流量**没有均匀分布**，而是集中在：

- 某几个节点
- 某几个数据分片
- 某些 Key

------

#### 典型场景

##### 1 Redis 热点 Key

例如：

```
商品ID=10001
访问量：100万
其他商品：1000
```

##### 2 数据库热点行

大量用户访问同一条数据。

##### 3 分库分表热点

```
user_id % 4
```

某个分片流量特别大。

------

#### 技术特征

典型表现：

- 某个节点 CPU 100%
- 某个 Redis Key QPS 极高
- 某个数据库实例压力极大

------

#### 风险影响

导致：

```
整体流量不高
但局部节点崩溃
```

例如：

```
Redis 单 Key QPS 50W
```

------

#### 常见治理手段

1. **Key 拆分**

例如：

```
hot_key_1
hot_key_2
hot_key_3
```

1. **本地缓存**

使用：

```
Caffeine
Guava Cache
```

1. **多级缓存**

```
浏览器 → CDN → Redis → DB
```

1. **数据分片优化**

避免 Hash 倾斜。



面试高频：流量倾斜 满分简答（好背、Java微服务/MQ/Kafka通用）

#### 一、什么是流量倾斜

整体流量正常，但**压力集中在某一个节点/分区/实例**，
部分节点负载爆表、CPU/内存打满、响应卡顿，其他节点空闲，整体集群利用率失衡，容易单点宕机、引发连锁问题。

#### 二、常见场景&原因

##### 1. 微服务集群流量倾斜

- 负载均衡算法不合理：默认轮询失效、IP哈希固化路由
- 用户/商户ID固定哈希，固定打到同一实例
- 服务实例配置不一（有的配置低、有的配置高）
- 注册中心权重未配置、健康检查不及时

##### 2. Kafka / MQ 消息队列倾斜（最常考）

- **分区数过少**，消费组节点多，部分消费者空闲
- key 哈希分区不合理，热点key全部打到同一个分区
- 顺序消息强制单分区，所有流量压单节点
- 分区leader副本集中在同一台机器

##### 3. 数据库流量倾斜

- 分库分表路由规则不合理，热点用户/订单落在同一张表
- 读写分离失效，读流量没分散到从库
- 热点商品、秒杀数据集中单库单表

#### 三、解决方案

##### 1. 服务层（负载均衡优化）

- 改用**随机、加权轮询、最小连接数**负载策略
- 避免固定哈希路由，关键业务增加随机打散
- 统一集群机器配置，配置权重负载
- 完善健康检查、剔除故障/负载过高节点

##### 2. MQ/消息队列解决倾斜

- 合理规划**分区数量**：分区数 ≥ 消费实例数
- 热点key增加**加盐、后缀随机**打散分区
- 拆分顺序消息，非顺序业务并行分区
- 手动调整leader副本分布，打散物理节点

##### 3. 数据层解决倾斜

- 分库分表合理路由，对热点ID做哈希打散
- 热点数据单独隔离、独立库表、独立缓存
- 完善读写分离，多读从库，分散查询压力

##### 4. 兜底防护

- 单机CPU、内存、连接数监控告警
- 节点负载阈值告警，提前发现倾斜
- 核心服务单机限流，防止单点被打垮

#### 四、极简面试背诵版

流量倾斜是集群中流量集中在个别节点，导致单点压力过大、负载失衡。
主要原因有负载均衡策略不合理、热点key哈希扎堆、MQ分区分配不均、分库分表路由集中。
解决方式：优化负载均衡算法，对热点key加盐打散；合理设置MQ分区数、均匀分配leader；数据层做好分表路由与热点隔离，同时监控单机负载，提前告警规避单点故障。

### 4 热点 Key / 热点数据

#### 定义

某些数据访问量远高于其他数据，形成 **访问热点**。

例如：

```
一个商品访问 100 万次
其他商品 100 次
```

------

#### 典型场景

1. 热门商品
2. 热门帖子
3. 热门视频
4. 热门明星
5. 排行榜数据

------

#### 技术特征

表现：

```
Redis 单 Key QPS 极高
```

例如：

```
GET hot_product:10001
QPS = 30W
```

------

#### 风险影响

可能导致：

- Redis CPU 打满
- 网络 IO 瓶颈
- 单实例崩溃

------

#### 常见治理手段

1. **本地缓存**

```
应用缓存
```

1. **Key 多副本**

例如：

```
product:1001_1
product:1001_2
product:1001_3
```

1. **CDN 缓存**



面试高频：热点Key / 热点数据 完整答案（超好背）

#### 一、什么是热点数据/热点Key

**少量数据被超高频次访问**，读QPS极高、流量极度集中，比如：秒杀商品、首页爆款、活动配置、热搜、爆款库存。
会导致：**缓存单点打爆、缓存击穿、DB压力集中、流量倾斜、节点CPU打满**。

---

#### 二、带来的问题

1. **缓存击穿**：同一个热点Key千万次并发直达DB
2. **Redis热点分片倾斜**：所有请求打在同一台Redis节点
3. **服务单机压力过载**：哈希路由固定打到同一实例
4. **DB行锁竞争**：热点商品单行频繁更新，锁等待、事务堵塞
5. **缓存过期瞬间雪崩**

---

#### 三、完整解决方案（核心必背）

##### 1. 本地多级缓存（最常用）

**Caffeine + Redis 二级缓存**
- 热点数据先查JVM本地缓存，不打Redis
- 大幅减少远程缓存请求与网络开销
- 本地缓存设置短期过期，保证最终一致

##### 2. 热点Key 分片/拆分（打散压力）

对同一个热点Key**加随机后缀/加盐哈希**：
```
goods_1001_01、goods_1001_02 ...
```
- 一份热点数据拆成多份Key
- 分散到不同Redis槽位、不同节点
- 并发请求均匀打散，解决Redis单点热点

##### 3. 永不过期 + 主动更新

- 热点商品**不设置固定过期时间**，规避缓存击穿
- 采用**主动更新**：后台定时/变更推送刷新缓存
- 避免大量请求同时等待缓存重建

##### 4. 本地锁 + 分布式锁防击穿

- 缓存失效瞬间，用**本地锁+分布式锁**
- 只放行一个请求查DB、重建缓存
- 其余请求直接走旧缓存兜底

##### 5. 服务层隔离 & 限流

- 热点接口单独线程池隔离
- 接口粒度、用户维度限流
- 防止热点流量拖垮普通业务

##### 6. 数据库热点行优化

- 热点库存**Redis预扣**，减少DB写
- 单行热点更新：**队列串行化、批量合并更新**
- 拆分热点商品、异地多活、热点库隔离

##### 7. 网关层拦截与优化

- 热点接口流量监控、异常告警
- 流量整形、拒绝非法高频请求

---

#### 四、极简背诵版（面试直接说）

热点数据是少量Key被超高并发访问，容易造成缓存击穿、Redis节点倾斜、DB单行压力过大。
解决方案：
采用**本地二级缓存**减少Redis压力；对热点Key加盐分片打散节点；
热点数据设置**永不过期+主动刷新**防击穿；
DB热点更新异步化、合并落库，配合接口隔离限流，彻底解决热点流量问题。

3. 导出堆快照：

##### 2. 代码根治

- 用完集合及时 `clear()`、解除对象引用
- 大数据量**分页/流式/分批**处理，杜绝一次性加载
- 缓存增加：过期时间、LRU淘汰、最大容量限制
- 资源关闭：IO、连接、ByteBuffer 手动释放
- 控制动态代理、动态类生成频率

##### 3. JVM优化

- 合理设置 `-Xmx/-Xms` 堆大小

- **连接池隔离**：核心业务、批量任务、定时任务**独立连接池**互不影响；
- **合理配置参数**：最大连接数、空闲超时、等待超时、定期回收无效连接。

##### 3. 监控预防

- 监控连接池使用率、等待线程、泄漏次数；

 面试简答：网络带宽耗尽（高频线上问题，精简好背）

#### 一、现象

机器出口/入口带宽打满，**请求延迟飙升、丢包、超时、TCP重传、接口卡顿、微服务调用超时、文件/大报文传输卡死**。

#### 二、核心原因

1. **大报文传输**
消息体过大、JSON冗余、返回全量字段、大List一次性返回。
2. **批量拉取/全表查询**
一次性返回几十万条数据，网络IO暴涨。
3. **日志打印泛滥**
全量日志、异常堆栈刷屏、远程日志采集占用带宽。
4. **频繁短轮询、无效心跳**
高频重复请求、定时轮询过密，挤占带宽。
5. **大文件/附件、图片未压缩**
未做压缩、未走CDN，直接走业务服务器带宽。
6. **流量异常、爬虫/攻击**
恶意爬虫、刷量、CC攻击挤占正常业务带宽。

#### 三、排查方式

1. 服务器：`iftop、nload、sar -n` 看上下行带宽占用
2. 抓包分析：定位大报文、高频请求来源
3. 查看接口日志：找出返回体超大的接口
4. 监控带宽曲线、丢包率、重传率

#### 四、解决方案

##### 1. 紧急止血

- 临时封禁爬虫、恶意IP
- 限制大接口QPS、临时关闭非核心文件下载
- 收紧日志级别，暂停远程日志推送

##### 2. 业务报文优化（核心）

- **数据压缩**：Gzip、Snappy压缩请求/响应
- 接口按需返回、字段精简、避免全量序列化
- 分页、流式返回、分批拉取，杜绝一次性大数据

##### 3. 资源动静分离

- 图片、静态资源、附件全部上 **CDN/对象存储**
- 大文件单独域名、单独带宽隔离，不占用业务网卡

##### 4. 协议与调用优化

- 短轮询改长轮询 / WebSocket 按需推送
- 精简心跳包、缩小请求头、减少无效请求
- MQ消息体压缩、大消息拆分

##### 5. 安全与防护

- 网关层限流、防爬、爬虫拦截、UA校验
- 带宽阈值告警，提前发现流量异常

##### 6. 架构兜底

- 网卡带宽扩容、业务与日志/采集流量网卡隔离
- 内网、外网流量分离，避免互相抢占

#### 五、极简背诵版

网络带宽耗尽，多由大报文、全量数据返回、日志泛滥、静态资源未分离、恶意爬虫导致。
排查通过带宽监控与抓包定位耗流量源头；

服务调用类问题的本质是：

```text

```

当某个服务出现问题时，可能会通过 **调用链传播**，最终导致 **系统整体故障**。

典型链路：

```

   ↓
网关
   ↓

   ↓
```

### 1 服务雪崩

#### 定义

一个服务出现故障后，导致 **大量请求堆积并传播到其他服务**，最终引发 **整个系统大面积不可用**。

------

#### 典型场景

例如：

```
服务A → 服务B → 服务C
```

如果 **服务C宕机**：

1. 服务B调用C失败
2. 请求在B堆积
3. B线程池耗尽
4. A调用B失败
5. 整个系统崩溃

------

#### 技术特征

表现为：

- 大量请求失败
- 服务响应时间急剧增加
- 线程池耗尽
- CPU升高

------

#### 风险影响

最终可能导致：

```
系统整体不可用
```

------

#### 常见治理手段

1. 服务限流
2. 服务降级
3. 服务熔断
4. 缓存兜底
5. 隔离策略

面试简答：服务雪崩（满分背诵版）

#### 一、定义

**服务雪崩**：微服务调用链中，**某个下游服务故障/超时/宕机**，导致上游请求阻塞、线程耗尽、资源打满，逐级向上蔓延，最终整个链路、甚至全集群瘫痪。

传播链路：
下游挂 → 上游阻塞 → 线程耗尽 → 服务不可用 → 调用方连锁崩溃

#### 二、产生原因

1. 下游服务响应慢、超时、报错，上游无限制重试
2. 未做超时控制、无熔断、无降级
3. 线程池不隔离，一个弱依赖拖垮核心业务
4. 高并发下大量请求阻塞，连接/线程占满
5. 接口同步调用过多，链路串行阻塞

#### 三、核心解决方案（四字口诀：**隔、熔、降、限**）

##### 1. 服务隔离

- **线程池隔离**：不同业务、不同下游单独线程池，互不影响
- **资源隔离**：核心业务与非核心业务集群物理/逻辑隔离

##### 2. 熔断（CircuitBreaker）

下游错误率、超时率达到阈值，**自动熔断**，不再调用故障下游，快速失败，避免阻塞。

##### 3. 降级

非核心接口、非必要功能临时关闭/返回兜底数据，**保住核心链路**。

##### 4. 限流

网关/接口层限流，拦截过量请求，保护服务承载力。

##### 5. 超时+重试控制

- 所有RPC、HTTP调用**必须配置超时时间**
- 禁止无限重试，加重试次数、间隔、幂等

##### 6. 异步解耦

非强依赖调用改为**异步/MQ**，避免同步阻塞等待。

#### 四、常用技术组件

- 熔断降级：Sentinel、Resilience4j、Hystrix
- 网关限流：Gateway、Nginx
- 隔离：线程池隔离、信号量隔离

#### 五、极简口述背诵版

服务雪崩是下游服务故障，因无超时、无熔断、无限重试，导致上游线程阻塞、资源耗尽，逐级蔓延引发全链路瘫痪。
解决方案核心是**隔离、熔断、降级、限流**：
通过线程池隔离避免相互影响，下游异常时触发熔断停止调用，非核心业务降级兜底，接口加限流保护；
同时统一配置调用超时、限制重试，非强依赖异步化，从调用链上杜绝雪崩扩散。

#### 定义

#### 典型场景

```

```

```
B失败
↓
A失败
```

------

#### 技术特征

- 调用链错误逐层扩大
- 系统整体错误率升高

------

#### 风险影响

可能导致：

```
多个服务同时不可用
```

------

#### 常见治理手段

1. 服务隔离
2. 熔断机制
3. 限流
4. 降级处理

面试简答：服务级联失败（通俗又叫**级联故障、连锁故障**）

#### 一、定义

某个**下游服务出现故障、超时、宕机、响应变慢**，
上游没有防护机制，大量请求阻塞、资源耗尽，
故障顺着调用链**向上扩散**，最终多个服务集体不可用，就是**服务级联失败**。

> 和服务雪崩：级联失败是过程，雪崩是最终结果。

#### 二、核心原因

1. **无超时控制**
RPC、HTTP、数据库调用无超时，线程一直阻塞等待。
2. **无熔断、无限重试**
下游报错不断重试，放大流量，压垮上下游。
3. **无资源隔离**
所有接口共用线程池、连接池，弱依赖拖垮核心业务。
4. **同步调用过多**
长链路同步串行，一环卡死，全链卡死。
5. **容量不足、未限流**
突发流量下，单点故障快速扩散。

#### 三、解决方案（四大核心：隔离、熔断、降级、限流）

1. **超时+重试控制**
所有远程调用必设超时；限制重试次数、间隔、禁止无限重试。

2. **熔断机制**
下游错误率/超时率过高，自动熔断，停止调用，快速失败，阻断故障向上传播。

3. **资源隔离**
- 线程池隔离：不同下游、不同业务单独线程池
- 连接池隔离：核心、非核心库/中间件连接池拆分
- 集群隔离：秒杀、活动独立部署

4. **降级兜底**
非核心功能降级，返回默认数据、缓存兜底，保住核心链路。

5. **异步解耦**
非强依赖改用 MQ、异步线程处理，切断同步阻塞链路。

6. **网关限流**
入口层限流、防刷，避免故障被大流量放大。

#### 四、排查思路

1. 看监控：接口超时率、错误率、响应时间陡增
2. 查线程堆栈：大量线程阻塞在下游调用处
3. 追踪调用链：SkyWalking/PinPoint 定位最先出问题的下游服务

#### 五、极简背诵版

服务级联失败是下游服务异常，因缺少超时、熔断、隔离机制，导致请求阻塞、资源耗尽，故障逐级向上扩散。
解决核心是**超时限制、熔断阻断、资源隔离、业务降级、网关限流**；
同时非核心场景异步解耦，避免同步调用链式阻塞，从源头阻止故障蔓延。

面试简答：服务依赖故障（满分精简版）

#### 一、概念

当前服务**强/弱依赖的第三方、下游服务、中间件、数据库、缓存**出现宕机、超时、报错、不可用，
导致自身接口异常、阻塞、性能下降，称为**服务依赖故障**。

#### 二、常见依赖类型

- 微服务下游接口
- Redis、MQ、Nacos、注册中心
- MySQL、第三方支付/短信/OSS
- 内网网关、配置中心

#### 三、根本原因

1. 依赖服务宕机、重启、网络波动
2. 依赖接口慢、超时、报错率飙升
3. 无超时、无重试策略、无容错
4. 未做隔离、未熔断，弱依赖拖垮主线程
5. 依赖突发限流、节点负载过高

#### 四、解决方案（核心五点）

1. **超时控制**
所有RPC、HTTP、中间件、DB调用**强制配置超时时间**，避免线程永久阻塞。

2. **熔断容错**
使用Sentinel/Resilience4j，依赖错误率、超时率达标自动熔断，快速失败，不持续拖压下游。

3. **依赖隔离**
- 不同下游单独线程池、信号量隔离
- 核心业务与弱依赖资源分开，互不影响

4. **优雅降级**
弱依赖故障时，走缓存兜底、默认值、空数据，**保核心业务可用**。

5. **重试与幂等**
限制重试次数、间隔时间，杜绝无限重试；
所有外部调用保证幂等，防止重复下单、重复扣款。

6. **监控与告警**
监控依赖健康度、RT、错误率、连接数；
依赖不可用及时告警，提前感知故障。

#### 五、极简背诵版

服务依赖故障是下游服务、中间件、数据库等依赖组件异常不可用引发的问题。
通过**统一超时、熔断降级、资源隔离、合理重试、监控告警**实现容错；
弱依赖故障自动降级兜底，避免单一依赖故障拖垮整体服务，保证核心链路稳定。

### 8 RPC 重试风暴

#### 定义

RPC 调用失败后自动重试，导致 **请求数量被放大**。

------

#### 典型场景

例如：

```
请求 = 1000
每次重试 = 3
```

```

```

如果服务已经过载：

```
重试会加剧系统压力
```

------

#### 技术特征

表现为：

- 请求量暴涨
- 服务压力加大

------

#### 风险影响

可能导致：

```
系统彻底崩溃
```

------

#### 常见治理手段

1. 限制重试次数
2. 指数退避
3. 熔断机制
4. 限流

面试高频：RPC重试风暴 极简满分答案

#### 一、什么是重试风暴

下游服务**超时、报错、不可用**时，
上游客户端**自动重试、无限重试、多节点集体重试**，
瞬间放大几倍流量，压垮下游，形成**恶性循环**，最终服务雪崩。

#### 二、产生原因

1. RPC调用未限制**重试次数、重试间隔**
2. 下游抖动/短暂超时，上游全员疯狂重试
3. 无熔断机制，故障时持续硬打
4. 重试未做**退避策略**，所有请求同一时间重试
5. 接口未做幂等，重试引发重复下单、扣款、数据错乱

#### 三、核心危害

- 流量瞬间放大数倍，下游彻底扛不住
- 服务负载飙升、连接池/线程池耗尽
- 超时→重试→更堵→更多超时，恶性循环
- 引发数据重复、业务错乱

#### 四、解决方案（必背）

1. **限制重试次数**
关闭无效重试，只允许**有限次数重试**，非核心接口直接禁重试。

2. **指数退避 + 随机延迟**
重试间隔逐步拉长，加**随机抖动**，避免集群同一时刻集中重试。

3. **熔断拦截**
下游超时率、错误率超标立刻熔断，停止调用，杜绝重试流量。

4. **区分异常类型**
只对**网络超时、瞬时抖动**可重试；
业务异常、参数错误**禁止重试**。

5. **全局幂等**
所有RPC接口增加幂等令牌、唯一流水号，防止重试产生脏数据。

6. **超时优化**
合理设置RPC超时时间，避免误触发大量超时重试。

#### 五、极简口述背诵版

RPC重试风暴是下游异常时，上游无限制、集中式重试导致流量雪崩。
解决手段：**限制重试次数、采用指数退避加随机抖动、按异常类型选择性重试**；
配合熔断机制阻断故障扩散，接口保证幂等，从源头避免重试恶性循环与数据重复问题。

5. 死锁
6. 连接数打满
7. 热点行
8. 写放大
9. 索引失效
10. 数据库主从延迟

数据库问题通常是 **高并发系统最容易成为瓶颈的一层**，因为数据库通常是：

```text
集中式资源
```

当访问量过高或设计不合理时，很容易成为 **系统性能瓶颈或故障源**。

------

### 1 慢 SQL

#### 定义

SQL 执行时间过长，超过系统可接受的阈值。

例如：

```text
SQL执行时间 > 500ms
```

很多系统会定义：

```text
慢SQL阈值 = 1s
```

------

#### 典型场景

1. 未使用索引
2. 全表扫描
3. JOIN 表过多
4. 数据量过大
5. ORDER BY / GROUP BY 未使用索引

示例：

```sql
SELECT * FROM order WHERE user_id = 10001;
```

如果 `user_id` 没有索引：

```text
全表扫描
```

------

#### 技术特征

表现为：

- SQL 执行时间长
- 数据库 CPU 升高
- 查询延迟增加

------

#### 风险影响

慢 SQL 在高并发场景下会导致：

```text
连接被长时间占用
↓
连接池耗尽
↓
系统无法访问数据库
```

------

#### 常见治理手段

1. 添加索引
2. SQL 优化
3. 分页查询
4. 数据分库分表

------

### 2 大事务

#### 定义

事务中包含 **大量数据操作或执行时间很长**。

例如：

```text
事务持续时间 > 10s
```

------

#### 典型场景

1. 批量更新

```sql
UPDATE order SET status = 1 WHERE status = 0;
```

1. 批量删除

```sql
DELETE FROM log WHERE create_time < '2023-01-01';
```

1. 复杂业务逻辑

------

#### 技术特征

表现为：

- 事务持续时间长
- 锁持有时间长

------

#### 风险影响

可能导致：

```text
锁竞争
↓
其他事务等待
↓
系统性能下降
```

------

#### 常见治理手段

1. 拆分事务
2. 批量处理
3. 控制事务范围

------

### 3 锁竞争

#### 定义

多个事务同时争抢同一资源锁。

例如：

```text
多个事务更新同一条记录
```

------

#### 典型场景

1. 更新同一数据

```sql
UPDATE product SET stock = stock - 1 WHERE id = 1001;
```

1. 高频更新操作

------

#### 技术特征

表现为：

- 事务等待
- SQL 执行时间变长

------

#### 风险影响

可能导致：

```text
数据库吞吐下降
```

严重时：

```text
系统整体性能下降
```

------

#### 常见治理手段

1. 减少锁粒度
2. 优化事务逻辑
3. 使用缓存减少数据库访问

------

### 4 行锁 / 表锁

#### 定义

数据库通过锁机制保证数据一致性。

常见锁：

| 锁类型 | 说明         |
| ------ | ------------ |
| 行锁   | 锁定一行数据 |
| 表锁   | 锁定整张表   |

------

#### 典型场景

行锁：

```sql
UPDATE user SET balance = 100 WHERE id = 1;
```

表锁：

```sql
LOCK TABLE user WRITE;
```

------

#### 技术特征

表现为：

- SQL 执行阻塞
- 事务等待

------

#### 风险影响

表锁尤其危险：

```text
整张表无法访问
```

------

#### 常见治理手段

1. 使用行锁
2. 避免长事务
3. 优化索引

------

### 5 死锁

#### 定义

两个或多个事务 **互相等待对方释放锁**。

例如：

```text
事务A 锁定 记录1
事务B 锁定 记录2
事务A 等待 记录2
事务B 等待 记录1
```

形成：

```text
循环等待
```

------

#### 技术特征

数据库会检测死锁：

```text
Deadlock found when trying to get lock
```

数据库通常会 **回滚一个事务**。

------

#### 风险影响

可能导致：

- 事务失败
- 请求失败

------

#### 常见治理手段

1. 固定更新顺序
2. 减少事务范围
3. 设置锁等待时间

------

### 6 连接数打满

#### 定义

数据库允许的连接数达到上限。

例如：

```text
max_connections = 1000
```

连接全部被占用。

------

#### 典型场景

1. 高并发访问
2. 慢 SQL
3. 连接泄漏

------

#### 技术特征

表现为：

```text
Too many connections
```

------

#### 风险影响

导致：

```text
新请求无法连接数据库
```

系统直接不可用。

------

#### 常见治理手段

1. 使用连接池
2. 优化 SQL
3. 增加连接数

------

### 7 热点行

#### 定义

某一行数据被大量并发访问或更新。

例如：

```text
商品库存
```

------

#### 典型场景

秒杀系统：

```sql
UPDATE product SET stock = stock - 1 WHERE id = 1001;
```

大量用户同时更新同一条记录。

------

#### 技术特征

表现为：

- 行锁竞争
- SQL 等待

------

#### 风险影响

可能导致：

```text
数据库吞吐量下降
```

------

#### 常见治理手段

1. Redis 扣库存
2. 队列削峰
3. 数据拆分

------

### 8 写放大

#### 定义

一次业务操作导致 **大量数据库写操作**。

例如：

```text
一个操作写10张表
```

------

#### 典型场景

1. 多表更新
2. 日志写入
3. 审计记录

------

#### 技术特征

表现为：

- 数据库写压力大
- IO 增加

------

#### 风险影响

可能导致：

```text
数据库写入瓶颈
```

------

#### 常见治理手段

1. 异步写入
2. 批量写入
3. 使用消息队列

------

### 9 索引失效

#### 定义

SQL 查询无法使用索引，导致 **全表扫描**。

------

#### 典型场景

例如：

```sql
SELECT * FROM user WHERE name LIKE '%张%';
```

------

#### 技术特征

表现为：

- 执行计划 `type = ALL`
- 查询速度慢

------

#### 风险影响

可能导致：

```text
数据库 CPU 飙升
```

------

#### 常见治理手段

1. 优化 SQL
2. 设计合理索引
3. 使用覆盖索引

------

### 10 数据库主从延迟

#### 定义

主库和从库之间的数据同步存在延迟。

例如：

```text
主库写入
↓
从库同步延迟
```

------

#### 典型场景

读写分离架构：

```text
写 → 主库
读 → 从库
```

------

#### 技术特征

表现为：

```text
主库数据更新
从库查询不到
```

------

#### 风险影响

导致：

```text
数据不一致
```

例如：

用户刚下单：

```text
订单查不到
```

------

#### 常见治理手段

1. 关键业务读主库
2. 延迟读取
3. 半同步复制

------

#### 总结

数据库问题可以归纳为 **五大类核心瓶颈**：

| 类别     | 问题             |
| -------- | ---------------- |
| 查询问题 | 慢 SQL、索引失效 |
| 事务问题 | 大事务、死锁     |
| 锁问题   | 锁竞争、热点行   |
| 连接问题 | 连接数打满       |
| 架构问题 | 主从延迟、写放大 |

数据库问题在高并发系统中的典型演化路径：

```text
慢SQL
↓
连接占用
↓
连接池耗尽
↓
服务无法访问数据库
↓
系统雪崩
```

------

**#5 缓存问题（Redis 类问题）**

包括最经典的：

- 缓存击穿
- 缓存穿透

这些几乎是 **所有互联网系统架构面试必问问题**。

## 5 缓存问题

1. 缓存击穿
2. 缓存穿透
3. 缓存雪崩
4. 热点 Key
5. 缓存失效风暴
6. Redis 单线程瓶颈
7. 大 Key
8. 缓存不一致
9. 缓存重建风暴

缓存层通常是 **高并发系统的第一道保护层**（尤其是 Redis），一旦设计不合理，问题会迅速传导到数据库甚至整个系统。

说明结构：**定义 → 典型场景 → 技术特征 → 风险影响 → 常见治理手段**

缓存层的核心目标是：

```text
减少数据库访问
提升系统吞吐能力
```

但如果缓存设计不合理，就会出现一系列稳定性问题。

------

### 1 缓存击穿

#### 定义

某个 **热点 Key 在缓存中过期的一瞬间**，大量请求同时访问数据库。

即：

```text
热点数据 + 缓存过期 + 高并发
```

------

#### 典型场景

例如：

```text
商品ID = 1001
QPS = 50000
```

缓存过期瞬间：

```text
Redis miss
↓
大量请求访问数据库
```

------

#### 技术特征

表现为：

- Redis 命中率突然下降
- 数据库 QPS 突然暴涨

------

#### 风险影响

可能导致：

```text
数据库被瞬间打爆
```

------

#### 常见治理手段

1️⃣ **互斥锁**

```text
只允许一个线程重建缓存
```

2️⃣ **永不过期 + 异步更新**

3️⃣ **热点数据提前刷新**

------

### 2 缓存穿透

#### 定义

请求访问 **数据库中不存在的数据**，缓存无法命中，每次请求都访问数据库。

------

#### 典型场景

例如：

```text
查询用户ID = -1
```

数据库不存在：

```text
Redis miss
↓
DB miss
↓
每次请求都访问 DB
```

------

#### 技术特征

表现为：

- Redis 命中率低
- 数据库查询大量空结果

------

#### 风险影响

攻击场景下可能导致：

```text
数据库压力极大
```

------

#### 常见治理手段

1️⃣ **布隆过滤器**

提前过滤不存在的数据。

2️⃣ **缓存空值**

```text
key = null
```

设置短 TTL。

------

### 3 缓存雪崩

#### 定义

大量缓存 **同时过期或缓存服务器宕机**，导致请求直接访问数据库。

------

#### 典型场景

例如：

```text
100 万 Key
TTL = 1 小时
```

1 小时后：

```text
全部 Key 同时过期
```

------

#### 技术特征

表现为：

- Redis 命中率急剧下降
- 数据库 QPS 突然暴涨

------

#### 风险影响

可能导致：

```text
数据库被压垮
```

最终：

```text
系统雪崩
```

------

#### 常见治理手段

1️⃣ **随机 TTL**

例如：

```text
TTL = 3600 + random(300)
```

2️⃣ **多级缓存**

```text
本地缓存 + Redis
```

3️⃣ **限流保护**

------

### 4 热点 Key

#### 定义

某个 Key 被 **极高频率访问**。

例如：

```text
GET hot_key
QPS = 100W
```

------

#### 典型场景

1. 热门商品
2. 排行榜
3. 热门视频
4. 秒杀商品

------

#### 技术特征

表现为：

- Redis CPU 升高
- 单 Key QPS 极高

------

#### 风险影响

可能导致：

```text
Redis 单实例瓶颈
```

------

#### 常见治理手段

1️⃣ **Key 拆分**

例如：

```text
hot_key_1
hot_key_2
hot_key_3
```

2️⃣ **本地缓存**

使用：

- Caffeine
- Guava Cache

3️⃣ **CDN 缓存**

------

### 5 缓存失效风暴

#### 定义

大量缓存 **在短时间内失效**，导致大量请求回源数据库。

------

#### 典型场景

例如：

```text
缓存统一 TTL
```

在同一时间失效。

------

#### 技术特征

表现为：

- Redis miss 暴增
- DB QPS 暴涨

------

#### 风险影响

可能导致：

```text
数据库过载
```

------

#### 常见治理手段

1️⃣ **TTL 随机化**

2️⃣ **缓存预热**

3️⃣ **限流**

------

#### 6 Redis 单线程瓶颈

#### 定义

Redis 是 **单线程处理命令**，在高并发下可能成为瓶颈。

------

#### 典型场景

例如：

```text
QPS = 50W
```

单实例 Redis 可能无法承受。

------

#### 技术特征

表现为：

- Redis CPU 高
- 请求延迟增加

------

#### 风险影响

可能导致：

```text
缓存响应变慢
```

最终影响整个系统。

------

#### 常见治理手段

1️⃣ **Redis Cluster**

2️⃣ **读写分离**

3️⃣ **热点 Key 分散**

------

### 7 大 Key

#### 定义

Redis 中 **数据量特别大的 Key**。

例如：

```text
String > 1MB
List > 1万元素
```

------

#### 典型场景

例如：

```text
user:followers
```

包含几十万用户。

------

#### 技术特征

表现为：

- 网络传输慢
- Redis 阻塞

------

#### 风险影响

可能导致：

```text
Redis 延迟升高
```

甚至阻塞。

------

#### 常见治理手段

1️⃣ **Key 拆分**

2️⃣ **分页读取**

3️⃣ **合理数据结构**

------

### 8 缓存不一致

#### 定义

#### 缓存数据与数据库数据 **不一致**。

------

#### 典型场景

例如：

```text
更新 DB
缓存未更新
```

------

#### 技术特征

表现为：

```text
用户看到旧数据
```

------

#### 风险影响

可能导致：

- 数据错误
- 用户体验差

------

#### 常见治理手段

常见策略：

1️⃣ **Cache Aside Pattern**

```text
先更新 DB
再删除缓存
```

2️⃣ **延迟双删**

3️⃣ **消息队列更新缓存**

------

### 9 缓存重建风暴

#### 定义

大量缓存同时失效，多个线程同时 **重建缓存**。

------

#### 典型场景

例如：

```text
Key过期
10000请求同时查询DB
```

------

#### 技术特征

表现为：

- DB QPS 暴涨
- Redis 写入暴涨

------

#### 风险影响

可能导致：

```text
数据库被打爆
```

------

#### 常见治理手段

1️⃣ **互斥锁**

2️⃣ **请求合并**

3️⃣ **单线程构建缓存**

------

#### 总结

缓存问题可以归纳为 **四大类核心问题**：

| 类别           | 问题                        |
| -------------- | --------------------------- |
| 缓存失效问题   | 击穿、雪崩、失效风暴        |

| 性能问题       | 热点 Key、大 Key、Redis瓶颈 |
| 数据一致性问题 | 缓存不一致、缓存重建风暴    |

缓存问题在高并发系统中的典型演化路径：

```text

↓
请求回源 DB
↓
数据库压力暴增
↓
数据库崩溃
↓
系统雪崩
```

## 6 队列 / 异步系统问题

1. 消息堆积
2. 消息重复消费
3. 消息丢失
4. 消费速度慢
5. 分区热点
6. 队列阻塞
7. 顺序消费阻塞
8. Broker 宕机

在高并发系统中，消息队列（如 **Apache Kafka**、**Apache RocketMQ**、**RabbitMQ**）常用于：

```text
削峰填谷
系统解耦
异步处理
```

但 MQ 设计不合理也会成为系统瓶颈或故障源。

------

### 1 消息堆积

#### 定义

生产者发送消息的速度 **远高于消费者处理速度**，导致消息在队列中不断积压。

即：

```text
生产速度 > 消费速度
```

------

#### 典型场景

例如：

```text
生产速度：10000 msg/s
消费速度：2000 msg/s
```

队列积压：

```text
8000 msg/s
```

------

#### 技术特征

表现为：

- MQ backlog 持续增长
- 消息延迟变大
- 消费 lag 增大

在 **Apache Kafka** 中表现为：

```text
consumer lag 持续增长
```

------

#### 风险影响

可能导致：

```text
业务处理延迟
```

例如：

- 订单处理延迟
- 日志延迟
- 通知延迟

严重时：

```text
队列存储被打满
```

------

#### 常见治理手段

1. 增加消费者数量
2. 提升消费并发度
3. 扩展分区数量
4. 降低生产速度

消息堆积如何排查+解决（RabbitMQ/RocketMQ/Kafka通用）

#### 一、先说核心本质

**生产速度 > 消费速度**，就会消息堆积。
三大核心原因：**消费慢、消费挂了、生产突增**。

---

#### 二、完整排查步骤（面试标准流程）

1. 查看MQ控制台：堆积数量、分区/队列消息数、延迟指标
2. 检查**消费者状态**：是否离线、掉线、心跳断连
3. 看消费端日志：报错、重试、阻塞、慢接口、死锁
4. 看消费性能：单条消费耗时、并发数、线程池
5. 看生产侧：是否流量突增、大促、批量发消息
6. 区分：**正常堆积（流量洪峰） / 异常堆积（代码bug）**

---

#### 三、消息堆积常见原因

##### 1. 消费者问题（最常见）

- 消费逻辑**慢**：同步调用第三方、DB慢SQL、复杂业务计算
- 消费报错、无限重试，卡死消费线程
- 消费者线程池过小、并发不足
- 消费者宕机、下线、节点故障
- 消息**幂等/事务**逻辑阻塞

##### 2. 消息本身问题

- 单条消息体过大，序列化/处理耗时久
- 死信未处理，重复投递占用队列
- 顺序消息，单条阻塞后续全部卡住

##### 3. 生产&架构问题

- 流量突增、秒杀/活动瞬时打满
- 生产者无限发消息、死循环投递
- 队列分区/消费组分配不合理，负载不均

---

#### 四、解决方案（先止血、再根治）

##### （一）紧急临时解决（快速削峰）

1. **临时增加消费者实例**
   横向扩容，多加机器，提高整体消费能力。
2. **调高消费并发/线程数**
   调整消费者线程池、批量消费大小。
3. **流量限流、降级**
   生产侧限流、削峰，非核心业务延迟投递。
4. **堆积消息迁移/临时分片队列**
   避免单队列压力过大。

##### （二）根治优化方案

1. **优化消费业务逻辑**
   - 慢SQL优化、加索引
   - 同步改异步、非核心逻辑异步化
   - 减少第三方同步调用，加熔断降级

2. **开启批量消费**
   批量拉取、批量处理，提升吞吐。

3. **异常消息处理**
   - 报错消息投递到**死信队列**，避免无限重试阻塞
   - 设置重试次数、重试间隔
   - 过滤脏数据、无效消息

4. **合理架构优化**
   - 顺序消息拆分，非顺序业务并行消费
   - 大消息压缩、拆分消息体
   - 分区/队列合理分片，负载均衡

5. **监控预警**
   监控队列堆积量、消费延迟、消费TPS，堆积阈值告警。

### 2 消息重复消费

#### 定义

同一条消息被 **消费多次**。

消息队列通常保证：

```text
At-Least-Once
```

即：

```text
至少消费一次
```

------

#### 典型场景

例如：

1. 消费成功但 ACK 失败
2. 消费者重启
3. 网络异常

------

#### 技术特征

表现为：

```text
同一业务被执行多次
```

例如：

```text
订单重复扣库存
```

------

#### 风险影响

可能导致：

- 数据错误
- 重复操作
- 资金问题

------

#### 常见治理手段

1. **幂等设计**

例如：

```text
订单ID唯一
```

1. 去重表
2. Redis 去重

面试简答：消息重复消费（精简超好背）

#### 一、产生原因

1. **消费者offset未提交成功**，重启后重新拉取旧消息
2. 业务处理完，MQ服务端超时未收到ACK，触发重投
3. 网络抖动、断连，导致重复派发
4. 生产者重试、集群分片重试，产生重复消息
5. 消息超时、死信重试、队列重平衡，造成重复投递

#### 二、核心危害

重复下单、重复扣款、数据重复插入、业务错乱、库存超扣。

#### 三、解决方案（核心4点）

##### 1. 消费者手动ACK + 合理提交时机

业务**处理成功后再提交offset**，避免先提交后失败。

##### 2. 接口/消费逻辑做**幂等**（最关键）

- 唯一约束：订单号、业务ID 数据库唯一索引
- 全局唯一ID + 分布式Redis去重
- 状态机判断：已处理/已完成直接过滤

##### 3. 生产者保证幂等

消息添加**全局唯一流水号**，避免重试产生重复消息。

##### 4. 消费端限流、重试策略管控

限制重试次数、间隔，避免无限重投；
区分异常类型，业务异常不入重试队列。

#### 四、极简背诵版

消息重复消费主要由ACK未及时提交、网络抖动、消息重投、集群重平衡导致。
解决方案以**幂等为核心**：利用业务唯一ID、数据库唯一索引、Redis防重做过滤；
配合合理ACK提交时机、管控消息重试策略，保证业务重复执行无副作用。

---
顺带帮你区分：
- 消息丢失：补偿、重试、事务消息、最终一致性
- 消息重复：**幂等**
- 消息积压：扩容消费者、优化消费逻辑、拆分队列

消息丢失｜面试极简满分答案

#### 一、丢失场景&原因

1. **生产者丢消息**
发送后无确认、单向发送、网络异常，消息没抵达MQ服务端。
2. **MQ服务端丢消息**
未持久化、刷盘策略不合理、集群同步未完成、节点宕机。
3. **消费者丢消息**
业务没处理完，**提前ACK/自动提交offset**；
消费异常直接抛异常丢弃，没有补偿。

---
#### 二、解决方案（三层保障）

1. **生产者端**
- 开启**发送确认机制**（同步/异步回调）
- 失败重发、日志记录、本地消息表兜底
- 使用事务消息/半消息，保证生产可靠

2. **MQ服务端**
- 开启消息**持久化**（磁盘落地）
- 集群多副本、同步复制，避免单节点宕机丢数据
- 合理配置刷盘、落库策略

3. **消费者端**
- 关闭自动ACK，**业务处理成功后手动ACK**
- 消费异常不走正常签收，转入重试/死信队列
- 异常消息落地日志，人工补偿

---
#### 三、一句话背诵

消息丢失分为生产者、服务端、消费者三层丢失。
通过**生产者确认+服务端持久化多副本+消费端手动ACK**三层防护；
搭配本地消息表、事务消息、死信队列，保证消息可靠投递，杜绝丢失。

### 4 消费速度慢

#### 定义

```text

```

例如：


------

2. 异步处理
3. 优化业务逻辑

消费速度慢｜面试标准简答

#### 一、原因

1. **消费业务太重**：同步调用、慢SQL、长事务、复杂计算。
2. **单消费者能力不足**：消费实例少、分区/队列数不够，无法横向扩容。
3. **串行处理**：单条顺序消费，无批量、无异步。
4. **资源瓶颈**：消费者CPU、内存、连接池耗尽，线程阻塞。
5. **依赖拖累**：下游DB、Redis、第三方接口响应慢，拖慢消费。
6. **频繁重试/死信堆积**：异常消息反复重试，挤占正常消费。

#### 二、解决方案

1. **横向扩容**
增加消费者实例，保证：**消费者数 ≤ 队列/分区数**，合理扩分片。

2. **批量消费**
开启批量拉取、批量入库，减少网络与IO交互，提升吞吐。

3. **优化消费逻辑**
剥离非核心逻辑异步化；优化慢SQL、索引、减少锁竞争与长事务。

4. **资源与隔离**
优化连接池、线程池参数；消费业务独立资源隔离，避免阻塞。

5. **异常消息治理**
失败消息有限重试，异常直接打入**死信队列**，不阻塞主队列。

6. **顺序解耦**
无需强顺序的业务，改为并行消费；
必须顺序的单独隔离队列，不影响普通消息。

#### 三、极简背诵版

消费速度慢，多由消费逻辑过重、实例不足、队列分区不够、下游依赖慢导致。
优化手段：**扩容消费者、增加队列分区、批量消费、简化消费业务、异常消息入死信**；
剥离非核心逻辑异步处理，解除阻塞，提升整体消费吞吐。

### 5 分区热点

#### 定义

消息队列中的 **某个分区承担了过多消息**。

------

#### 典型场景

例如：

```text

```

```text
partition1 = 80%
partition2 = 10%
partition3 = 5%

```

------

#### 技术特征

表现为：

```text

```

------

#### 风险影响

可能导致：

```text
消费不均衡
```

------

#### 常见治理手段

1. 优化分区策略
2. 使用随机分区
3. 增加分区数量

分区热点｜面试满分简答

#### 一、定义

MQ（Kafka/Rabbit）中，**大量同类热点key、同一业务ID** 经过哈希路由，全部打到**同一个分区**；
该分区读写压力爆表、消费堆积、消费延迟，其他分区空闲，形成**分区流量倾斜**，就是分区热点。

#### 二、产生原因

1. 业务固定 `key` 哈希取分区，热点商品/活动ID扎堆
2. 顺序消息强制单分区，全量流量压在一个分区
3. 分区数量规划过少，无法打散流量
4. Leader分区集中在同一台Broker，单机负载打爆

#### 三、带来问题

- 单分区消息积压、消费延迟持续走高
- 单机CPU、磁盘IO、网络带宽耗尽
- 消费扩容无效（分区数上限决定并发度）
- 引发整体MQ集群性能抖动

#### 四、解决方案

1. **热点Key打散（核心）**
对热点key**加盐、拼接随机后缀**，路由到不同分区，消费端再合并去重。

2. **合理增加分区数量**
分区数 ≥ 消费者实例数，提升整体并行消费能力，打散负载。

3. **拆分顺序消息**
非强顺序业务取消单分区；
必须顺序的单独隔离队列，不影响普通业务。

4. **手动均衡Leader副本**
调整分区Leader分布，避免热点分区集中在单台Broker。

5. **队列隔离**
秒杀、活动、热点业务单独独立Topic，与普通业务物理隔离。

6. **批量消费+优化消费逻辑**
提升单分区处理能力，避免单分区消费瓶颈。

#### 五、极简背诵版

分区热点是热点key哈希扎堆，导致流量集中单个分区，引发消息积压、单机负载过高。
解决：**热点key加盐打散、扩容分区数、隔离热点Topic、均衡Leader节点**；
合理管控顺序消息使用范围，结合批量消费，彻底解决分区倾斜与消费瓶颈。

### 6 队列阻塞

#### 定义

消息队列无法继续接收或处理消息。

表现为：

队列阻塞｜面试极简满分答案

#### 一、现象

队列消息**持续堆积、消费停滞、延迟暴涨**，新消息不断积压，整个队列卡死、无法正常流转。

#### 二、核心原因

1. **消费阻塞**：消费逻辑同步阻塞、慢SQL、长事务、外部RPC超时、锁等待
2. **单分区/单队列限制**：顺序消息单队列，一条失败堵住全队列
3. **异常消息卡死**：脏数据、报错消息无限重试，占用消费线程
4. **消费能力不足**：消费者少、未扩容、无批量消费
5. **资源瓶颈**：消费者连接池/线程池耗尽、CPU打满、网络卡顿
6. **队列重平衡、offset异常**导致消费暂停

#### 三、解决方案

1. **解耦阻塞逻辑**
消费内部**非核心操作异步化**，杜绝同步长耗时调用。

2. **异常消息隔离**
失败消息**有限重试**，多次失败直接转入**死信队列**，不阻塞主流程。

3. **解除顺序绑定**
非强顺序业务取消单分区/单队列；
必须顺序的单独队列隔离，不影响普通队列。

4. **提升消费吞吐**
增加消费者实例、扩容分区/队列数量、开启批量消费。

5. **资源优化**
合理配置线程池、连接池，添加调用超时，避免线程卡死。

6. **运维兜底**
清理堆积、重置偏移量、拆分大队列、监控队列积压告警。

#### 四、极简背诵版

队列阻塞主要由消费逻辑阻塞、异常消息卡死、顺序消息限制、消费能力不足导致。
解决核心：**异步化消阻塞、异常消息入死信、拆分顺序队列、扩容分区与消费者、批量消费**，
防止单点消息故障拖垮整个队列，保障消息持续流转。

---
快速串联
队列阻塞 → 必然引发**消息积压**
根源：消费慢 + 异常不丢弃 + 顺序强串行

顺序消费阻塞｜面试标准简答（超好背）

#### 一、现象

为保证消息顺序，**单分区/单队列串行消费**；
某一条消息处理失败、耗时久、阻塞卡死，**后面所有消息全部排队等待**，整队阻塞、堆积延迟。

4. 下游依赖超时、网络抖动，导致单条消费卡死
5. 无死信隔离，失败消息反复重试，持续占用消费线程

#### 三、核心危害

- 一条堵、全队堵，顺序队列完全停滞
- 消息延迟持续暴涨、大量堆积
- 扩容消费者无效（单分区只能单线程消费）
- 大促下引发连锁MQ瓶颈

#### 四、解决方案

1. **顺序业务隔离**
顺序消费单独独立 Topic/队列，和普通业务队列物理隔离，互不影响。

2. **错误消息隔离**
单条消费失败**有限次数重试**，多次失败直接投递死信队列，不阻塞后续。

3. **消费加超时控制**
DB、RPC、Redis 全链路设置超时，避免单条永久阻塞。

4. **拆分粒度，分段顺序**
不需要全局顺序的，改为**业务维度局部顺序**，拆分多分区并行。

5. **优化串行逻辑**
顺序链路内只保留核心流程，非核心日志、统计、推送全部异步化。

6. **监控预警**
监控单队列堆积、消费耗时、死信数量，及时发现阻塞。

#### 五、极简背诵版

顺序消费阻塞源于单分区串行执行，单条消息异常或耗时过长，卡住整队消息。
解决思路：**顺序队列隔离、失败消息入死信、全链路超时、局部顺序拆分、核心逻辑精简异步化**，
在保证业务有序的同时，避免单点故障拖垮整个顺序队列。

---

顺序消费 = 单分区串行 → 天然容易阻塞 → 死信+隔离 是关键解法

机制。

 Broker 宕机｜面试极简满分答案

#### 一、现象

MQ Broker 节点宕机、进程挂掉、机器故障，引发：
分区 Leader 不可用、读写失败、消息收发异常、队列堆积、分区切换、服务抖动。

#### 二、核心原因

机器断电/系统崩溃、CPU/内存打爆、磁盘满、IO阻塞、进程OOM、版本bug、网络断连。

#### 三、带来问题

1. 该节点上**Leader分区直接不可写**，生产失败
2. 短暂分区选举、重平衡，消费暂停、延迟上涨
3. 未同步的副本数据丢失风险
4. 单节点瓶颈、集群负载不均

#### 四、解决方案

1. **集群多副本部署**
Topic 配置多副本，Leader宕机自动从Follower选举新Leader，保证高可用。
2. **开启持久化+副本同步**
消息落盘持久化，副本同步复制，防止节点宕机数据丢失。
3. **自动故障转移**
依赖MQ集群机制，自动Leader切换、分区重平衡，业务无感知切换。
4. **生产端容错**
生产者配置重试、失败隔离、降级策略，临时规避单节点故障。
5. **磁盘与资源管控**
磁盘容量告警、限制大消息、隔离IO消耗，避免Broker过载宕机。
6. **监控告警**
节点存活、分区状态、副本同步、机器负载实时监控，快速修复。

#### 五、极简背诵版

Broker宕机通过**多副本集群+自动Leader选举**实现故障转移；
配合消息持久化、副本同步防止数据丢失，生产消费增加容错重试，
保障MQ集群节点故障下业务稳定可用。

### 1 分布式锁竞争

#### 定义

多个服务实例在 **争抢同一把分布式锁**，导致大量线程阻塞或重试。

分布式锁通常依赖：

- Redis
- Apache ZooKeeper
- etcd

------

#### 典型场景

例如秒杀库存扣减：

```
多个服务实例
     ↓
争抢库存锁
     ↓
大量线程等待
```

如果并发：

```
10000 请求
```

但锁只允许：

```
1 个线程执行
```

其余线程都会等待或重试。

------

#### 技术特征

表现为：

- QPS突然下降
- CPU空转
- 请求阻塞

------

#### 风险影响

可能导致：

```
系统吞吐量下降
```

甚至：

```
锁争用风暴
```

------

#### 常见治理手段

1. 减少锁粒度
2. 使用无锁设计（CAS）
3. 使用分段锁

例如：

```
库存锁
→ 拆分为
商品ID锁
```

------

### 2 锁失效

#### 定义

分布式锁 **提前失效或被错误释放**，导致多个线程同时执行。

例如使用 **Redis** 的：

```

```

如果：

```
SETNX成功
EXPIRE失败
```

锁就会：

```
永久存在
```

或者：

```
锁过期
业务还未执行完成
```

------

#### 技术特征

表现为：

```
多个线程进入临界区
```

例如：

```
库存被重复扣减
```

------

#### 风险影响

会造成：

- 数据错误
- 并发问题
- 资金问题

------

#### 常见治理手段

1. 使用原子命令

例如：

```
SET key value NX EX 30
```

1. 使用 **RedLock** 算法
2. 续期机制（watch dog）

这类问题不会依赖外部系统，而是**代码本身导致系统崩溃或性能下降**。

### 1 代码死循环

#### 定义

程序逻辑错误导致 **循环无法退出**。

------

#### 典型场景

例如：

```java

}
```


- 线程无法释放
- 请求卡死

------

#### 风险影响

```text
单线程占满CPU → 整个服务性能下降
```

------

#### 常见治理手段

1. 增加退出条件
2. 限制循环次数
3. 监控 CPU 火焰图

------

### 2 内存泄漏

#### 定义

对象已经不再使用，但 **仍然被引用无法回收**。

------

#### 典型场景

#### 技术特征

- JVM Heap 持续增长
- Full GC 频繁

------

#### 风险影响

可能导致：

```text
OutOfMemoryError
```

------

#### 常见治理手段

1. 使用弱引用
2. 清理 ThreadLocal
3. 使用内存分析工具

------

### 3 线程死锁

#### 定义

#### 风险影响

```text
服务完全阻塞
```

#### 常见治理手段

**系统架构问题（高并发系统稳定性中的顶层设计问题）** 的系统化说明。这一类问题本质不是代码问题，而是：

```text
架构设计是否具备“抗压能力 + 容错能力 + 可扩展能力”
```

仍然按统一结构：

> 定义 → 典型场景 → 技术特征 → 风险影响 → 常见治理手段

------

### 1 单点故障（SPOF）

#### 定义

系统中某一个组件一旦宕机，整个系统不可用。

------

#### 典型场景

1. 单台数据库
2. 单台 Redis
3. 单个网关
4. 单个注册中心

例如：

```text
Nginx只有1台
↓
机器宕机
↓
全站不可用
```

------

#### 技术特征

- 无冗余
- 无高可用
- 无容灾

------

#### 风险影响

```text
系统整体不可用
```

------

#### 常见治理手段

1. 集群化部署
2. 主从架构
3. 多机房部署

例如：

- Kubernetes
- Nginx

------

### 2 扩展性不足

#### 定义

系统无法通过增加资源来提升性能。

------

#### 典型场景

1. 单体架构
2. 强耦合服务
3. 单库设计

------

#### 技术特征

- 只能垂直扩展（升级机器）
- 无法水平扩展

------

#### 风险影响

```text
流量增长 → 系统崩溃
```

------

#### 常见治理手段

1. 微服务拆分
2. 无状态设计
3. 分库分表

------

### 3 负载不均

#### 定义

流量或计算负载在节点之间 **分布不均匀**。

------

#### 典型场景

例如：

```text
3台服务
A: 80%
B: 15%
C: 5%
```

------

#### 技术特征

- 某些节点 CPU 100%
- 其他节点空闲

------

#### 风险影响

```text
局部瓶颈导致整体性能下降
```

------

#### 常见治理手段

1. 负载均衡算法优化（如轮询→一致性哈希）
2. 服务拆分
3. 动态权重调整

------

### 4 服务治理缺失

#### 定义

系统缺乏对服务调用的统一治理能力。

------

#### 典型场景

没有以下机制：

- 限流
- 熔断
- 降级
- 超时控制

------

#### 技术特征

- 服务间调用无保护
- 依赖链无限扩展

------

#### 风险影响

```text
局部故障扩散成系统故障
```

------

#### 常见治理手段

1. 服务治理框架

例如：

- Sentinel
- Hystrix

1. 统一网关治理

------

### 5 容量规划不足

#### 定义

系统未根据实际或峰值流量进行资源规划。

------

#### 典型场景

1. 未做压测直接上线
2. 忽略大促流量
3. 低估增长速度

------

#### 技术特征

- CPU / 内存 / DB 不足
- 峰值时崩溃

------

#### 风险影响

```text
流量上涨 → 系统直接崩溃
```

------

#### 常见治理手段

1. 压测（Stress Test）
2. 容量评估模型
3. 峰值预估

------

### 6 限流缺失

#### 定义

系统没有对请求进行流量控制。

------

#### 典型场景

1. 秒杀系统未限流
2. API无限访问
3. 恶意刷接口

------

#### 技术特征

- QPS不可控
- 瞬时流量爆炸

------

#### 风险影响

```text
系统被瞬间打爆
```

------

#### 常见治理手段

1. 网关限流
2. IP限流
3. 用户维度限流

------

### 7 降级缺失

#### 定义

系统在高负载或故障时 **无法关闭非核心功能**。

------

#### 典型场景

1. 推荐系统拖垮主链路
2. 日志系统影响主业务
3. 非核心接口阻塞

------

#### 技术特征

- 全功能运行
- 无优先级控制

------

#### 风险影响

```text
核心业务被拖死
```

------

#### 常见治理手段

1. 功能降级
2. 服务降级
3. 静态兜底

------

### 8 熔断缺失

#### 定义

当下游服务异常时，系统没有自动停止调用。

------

#### 典型场景

```text
服务B异常
A仍持续调用B
```

------

#### 技术特征

- 错误率持续上升
- 请求不断重试

------

#### 风险影响

```text
故障扩散
```

------

#### 常见治理手段

1. 熔断机制

例如：

- Sentinel

1. 自动恢复策略
2. 快速失败（Fail Fast）

------

#### 总结

系统架构问题本质是：

```text
系统是否具备“抗压 + 容错 + 可扩展能力”
```

可以归纳为 4 大类：

| 类别         | 问题                         |
| ------------ | ---------------------------- |
| 高可用问题   | 单点故障                     |
| 扩展问题     | 扩展性不足、容量不足         |

| 容错问题     | 熔断缺失、降级缺失、治理缺失 |

------

架构缺陷的典型故障链：

```text

↓
流量暴涨
↓
无熔断
↓
依赖雪崩
↓
无降级
↓
核心服务崩溃
```

 **网络问题（高并发系统稳定性中的底层基础问题）** 的系统化说明。

网络层问题的本质是：

```text
系统之间通信不可靠 + 不稳定 + 不均匀
```

它通常直接影响：

------

### 1 网络延迟

#### 定义

数据在网络中传输所需时间过长。

例如：

```text
RTT（往返延迟）增加
```

------

#### 典型场景

1. 跨机房调用
2. 跨地域访问
3. 网络拥塞

------

#### 技术特征

- 接口响应变慢
- RPC延迟上升

------

#### 风险影响

```text
整体系统吞吐下降
```

------

#### 常见治理手段

1. 同机房部署
2. CDN加速
3. 减少跨网络调用

------

### 2 网络丢包

#### 定义

网络传输过程中 **部分数据包丢失**。

------

#### 典型场景

1. 网络不稳定
2. 交换机拥塞
3. 无线网络环境

------

#### 技术特征

- TCP重传
- 请求超时

------

#### 风险影响

```text
请求失败或延迟增加
```

------

#### 常见治理手段

1. TCP重传机制
2. 重试机制（应用层）
3. 多路由冗余

### 3 TCP连接积压

#### 定义

大量 TCP 连接处于等待或未处理状态。

------

#### 典型场景

例如：

```text
高并发连接请求
```

但服务端处理能力有限。

------

#### 技术特征

- backlog队列增长
- connect超时

------

#### 风险影响

```text
新连接无法建立
```

------

#### 常见治理手段

1. 调整 backlog 参数
2. 优化服务端处理能力
3. 使用连接池

------

### 4 TIME_WAIT过多

#### 定义

TCP连接关闭后进入 TIME_WAIT 状态，过多导致资源占用。

------

#### 典型场景

1. 短连接高并发
2. HTTP频繁请求

------

#### 技术特征

- netstat大量TIME_WAIT
- 端口占用

------

#### 风险影响

```text
无法建立新连接
```

------

#### 常见治理手段

1. 使用长连接
2. 调整内核参数
3. 连接复用

------

### 5 带宽打满

#### 定义

网络带宽达到物理上限。

------

#### 典型场景

1. 大文件下载
2. 视频流量
3. 数据同步

------

#### 技术特征

- 网络延迟增加
- 丢包率上升

------

#### 风险影响

```text
所有请求变慢
```

------

#### 常见治理手段

1. 流量压缩
2. CDN分流
3. 限流

------

### 6 负载均衡故障

#### 定义

流量分发组件出现异常，导致请求分配错误或失败。

------

#### 典型场景

例如：

- Nginx

- 节点健康检查失效

------

#### 技术特征

- 流量集中到单节点
- 部分服务不可访问

------

#### 风险影响

```text
系统局部或整体不可用
```

------

#### 常见治理手段

1. 多负载均衡节点
2. 健康检查机制
3. 自动摘除故障节点

------

### 7 NAT端口耗尽

#### 定义

NAT设备的端口资源被用完，导致无法建立新连接。

------

#### 典型场景

1. 微服务大量短连接
2. 高并发外部调用

------

#### 技术特征

- 新连接失败
- connect timeout

------

#### 风险影响

```text
无法建立网络连接
```

------

#### 常见治理手段

1. 增加NAT网关
2. 使用长连接
3. 减少连接频率

网络问题本质是：

**运维 / 发布问题（生产环境稳定性中非常关键的一类问题）** 的系统化说明。这一类问题的特点是：

### 2 数据倾斜

#### 定义

数据在分布式存储中 **分布极不均匀**。

------

#### 典型场景

例如：

```text
user_id % 10
```

结果：

```text
某个分片数据量远大于其他分片
```

------

#### 技术特征

- 某些节点压力极大
- 其他节点空闲

------

#### 风险影响

```text
局部热点 → 整体性能下降
```

------

#### 常见治理手段

1. 更均匀的哈希策略
2. 一致性哈希
3. 分片扩容

------

### 3 数据膨胀

#### 定义

数据增长速度过快，导致存储和查询成本急剧上升。

------

#### 典型场景

1. 日志数据无限增长
2. 历史订单未归档
3. 业务数据未清理

------

#### 技术特征

- 表数据持续增长
- 查询性能下降

------

#### 风险影响

```text
数据库查询变慢 + 存储压力增加
```

------

#### 常见治理手段

1. 数据归档
2. 分表分库
3. 定期清理

------

### 4 历史数据过大

#### 定义

历史数据未做分层或归档，导致存储和查询效率下降。

------

#### 典型场景

1. 所有订单数据放在一张表
2. 日志无限累积
3. 用户行为数据长期保存

------

#### 技术特征

- 查询慢
- 索引变大
- IO压力高

------

#### 风险影响

```text
历史查询拖垮系统
```

------

#### 常见治理手段

1. 冷热数据分离
2. 分区表（按时间）
3. 数据归档到数据仓库

------

### 5 分库分表不均

#### 定义

在分库分表设计中，数据分布不均匀导致某些库/表压力过大。

------

#### 典型场景

例如：

```text
user_id % 4
```

导致：

```text
某一张表数据特别多
```

------

#### 技术特征

- 某库CPU/IO爆满
- 查询集中在单表

------

#### 风险影响

```text
系统瓶颈集中在单点分片
```

------

#### 常见治理手段

1. 更优分片算法
2. 动态扩容分片
3. 二次分片

------

### 6 热分片

#### 定义

某一个分库/分表成为 **访问热点**，承载过多请求。

------

#### 典型场景

1. 热门用户数据
2. 热门商品数据
3. 活跃用户集中访问

------

#### 技术特征

- 单分片QPS极高
- 其他分片空闲

------

#### 风险影响

```text
局部热点 → 整体性能瓶颈
```

------

#### 常见治理手段

1. 数据拆分（打散热点）
2. 本地缓存
3. 多副本扩展

------

#### 总结

数据问题本质是：

```text
数据分布不均 + 生命周期设计不合理 + 缺乏治理策略
```

可以归纳为 4 大类：

| 类别           | 问题                   |
| -------------- | ---------------------- |
| 数据正确性问题 | 脏数据                 |
| 数据分布问题   | 数据倾斜、热分片       |
| 数据规模问题   | 数据膨胀、历史数据过大 |
| 架构设计问题   | 分库分表不均           |

------

#### 核心规律（非常重要）

数据问题的典型演化链：

```text
数据增长
↓
未归档
↓
查询变慢
↓
索引变大
↓
系统性能下降
```

------

## 13 安全类问题

1. DDoS 攻击
2. CC 攻击
3. 恶意刷接口
4. 暴力登录
5. Token 滥用

**安全类问题（高并发系统中的攻击面与防护问题）** 的系统化整理。这一类问题的本质是：

```text
系统被“恶意流量 + 非法访问 + 滥用行为”消耗资源
```

------

### 1 DDoS 攻击

#### 定义

分布式拒绝服务攻击（Distributed Denial of Service），通过大量“僵尸流量”耗尽系统资源。

------

#### 典型场景

```text
攻击者控制大量机器
↓
同时访问目标服务
↓
带宽 / CPU / 连接耗尽
```

------

#### 技术特征

- 流量异常暴增
- 带宽打满
- 服务不可用

------

#### 风险影响

```text
系统整体瘫痪
```

------

#### 常见治理手段

1. CDN防护
2. 流量清洗
3. 黑洞路由
4. WAF防火墙

例如：

- Cloudflare

------

### 2 CC 攻击

#### 定义

通过模拟正常用户请求，持续高频访问接口，耗尽应用层资源。

------

#### 典型场景

```text
大量请求登录 / 搜索接口
```

特点是：

- 看起来“像正常用户”
- 但频率极高

------

#### 技术特征

- QPS异常高
- CPU持续100%
- 接口响应变慢

------

#### 风险影响

```text
应用层被拖死
```

------

#### 常见治理手段

1. IP限流
2. 人机识别（验证码）
3. 行为分析
4. 网关限流

------

### 3 恶意刷接口

#### 定义

针对某个业务接口进行自动化高频调用。

------

#### 典型场景

1. 刷点赞
2. 刷浏览量
3. 刷下单接口
4. 刷短信接口

------

#### 技术特征

- 单接口流量异常
- 数据异常增长

------

#### 风险影响

```text
业务数据失真 + 资源浪费
```

------

#### 常见治理手段

1. 接口限流
2. Token校验
3. 行为风控系统
4. 设备指纹识别

------

### 4 暴力登录

#### 定义

通过大量尝试密码组合来破解账号。

------

#### 典型场景

```text
用户名固定 + 密码字典攻击
```

------

#### 技术特征

- 登录失败次数暴增
- 单IP高频请求登录接口

------

#### 风险影响

```text
账号被盗风险
```

------

#### 常见治理手段

1. 登录失败锁定
2. 验证码
3. IP限制
4. MFA（多因素认证）

------

### 5 Token 滥用

#### 定义

Token被非法使用或被恶意重复使用，导致身份伪造或权限滥用。

------

#### 典型场景

1. Token泄露
2. Token长期有效
3. 未绑定设备
4. 被重放攻击

------

#### 技术特征

- 异地登录
- 用户行为异常
- 请求伪造

------

#### 风险影响

```text
身份被盗用 + 权限滥用
```

------

#### 常见治理手段

1. Token过期机制
2. Refresh Token机制
3. 设备绑定
4. IP / 地域校验

------

#### 总结

安全类问题本质是：

```text
恶意流量 + 身份滥用 + 自动化攻击
```

可以归纳为 3 大类：

| 类别         | 问题             |
| ------------ | ---------------- |
| 流量攻击     | DDoS、CC攻击     |
| 业务滥用     | 刷接口、暴力登录 |
| 身份安全问题 | Token滥用        |

------

#### 核心规律（非常重要）

安全攻击的典型链路：

```text
恶意流量进入
↓
接口被刷爆
↓
资源耗尽
↓
正常用户无法访问
↓
系统瘫痪
```
