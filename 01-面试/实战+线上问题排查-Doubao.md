### 1. JDBC 批量插入为什么越插越慢？怎么优化？

**原因：**
1. 未使用 `addBatch/executeBatch`，单条循环插入，网络往返多
2. 自动提交事务，大量小事务频繁刷盘
3. 数据量增大，索引维护成本越来越高
4. redo/undo 日志膨胀，IO 压力持续上升
5. 内存装不下数据页，频繁磁盘 IO

**优化方案：**
1. 使用 `PreparedStatement.addBatch()` + `executeBatch()`
2. 关闭自动提交，分批提交（每 1000~5000 条一次）
3. 临时关闭非关键索引、外键约束
4. 调整连接参数 `rewriteBatchedStatements=true`
5. 大数量用 `LOAD DATA INFILE`
6. 分库分表，避免单表过大

# 2. MySQL 大分页 limit 1000000,10 为什么慢？怎么优化？
**原因：**
MySQL 会先扫描 1000010 条记录，再丢弃前 1000000 条，IO 与排序成本极高。

**优化方案：**
1. **主键定位分页（推荐）**
   ```sql
   select * from t where id > 1000000 limit 10;
   ```
2. 延迟关联：先查 ID，再关联取数据
3. 业务上禁止深翻页，改用滚动/游标
4. 合理加索引，避免 filesort

# 3. 数据库索引失效场景有哪些？
1. 索引列使用函数、运算（`where year(create_time)=2025`）
2. 隐式类型转换（字符串数字混用）
3. 模糊查询以 `%` 开头（`like '%abc'`）
4. 使用 `!=`、`<>`、`not in`、`is not null`
5. 联合索引不满足最左前缀原则
6. 优化器判断全表扫描更快，放弃索引
7. order by / group by 字段与索引不一致
8. 表数据量太小，优化器直接全表扫

# 4. Spring 事务失效场景有哪些？
1. **方法非 public**（Spring AOP 不代理）
2. **同类内部调用**（`this.a()` 调用带 @Transactional 方法）
3. 异常被 catch 吃掉，未抛出
4. 抛出非 RuntimeException（默认只回滚运行时异常）
5. 传播机制设置错误（如 `NOT_SUPPORTED`）
6. 数据库引擎不支持事务（MyISAM）
7. 多数据源未指定事务管理器
8. 类未被 Spring 管理（未加 @Service/@Component）

# 6. 高并发下库存超卖怎么解决？
**本质：** 多线程并发扣减，导致数据不一致。

**解决方案：**
1. **数据库乐观锁（推荐）**
   ```sql
   update stock set num=num-1 where id=? and num>0
   ```
2. **分布式锁（Redis/Redisson）**
   扣减前加锁，防止并发
3. **队列串行化**（RabbitMQ/RocketMQ）
   下单请求排队，单线程扣减
4. 数据库悲观锁（for update，不推荐高并发）
5. Lua 脚本原子扣减（Redis+Lua）

# 1. 线程池参数如何合理设置？
## 核心参数
- **corePoolSize**：核心线程数
- **maximumPoolSize**：最大线程数
- **keepAliveTime**：空闲线程存活时间
- **workQueue**：阻塞队列
- **threadFactory**：线程工厂
- **handler**：拒绝策略

## 如何合理设置？
1. **CPU 密集型**
   线程数 = CPU 核心数 + 1
   避免频繁线程切换。

2. **IO 密集型**（接口调用、数据库、文件）
   线程数 = CPU 核心数 * 2
   或按：`线程数 = CPU核心数 / (1 - 阻塞系数)`，一般设 **20~50** 甚至更高。

3. **队列选型**
   - 小任务：**LinkedBlockingQueue**
   - 避免OOM：**ArrayBlockingQueue**（有界队列）
   - 优先级任务：**PriorityBlockingQueue**

4. **拒绝策略**
   - 默认：Abort（抛异常）
   - 生产常用：CallerRuns（让调用方执行，降低提交速度）

# 2. ThreadLocal 用过吗？内存泄漏怎么避免？
## 作用
- 线程内部变量隔离
- 常用场景：存储用户上下文、事务信息、连接信息等

## 内存泄漏原因
- ThreadLocalMap 的 Entry 是**弱引用 key**，但**value 是强引用**
- 线程复用（线程池）→ value 一直无法回收 → OOM

## 避免方法
1. **使用完手动 remove()**（最重要）
2. 不要在 ThreadLocal 存大对象
3. 避免使用 static ThreadLocal 长期持有
4. 确保线程退出或归还线程池前清理

# 3. 单例模式线程安全写法？
1. **饿汉式**
   类加载就初始化，天生线程安全，但可能浪费内存。

2. **双重校验锁 DCL（最常用）**
   ```java
   public class Singleton {
       private static volatile Singleton instance;
       private Singleton(){}
       public static Singleton getInstance() {
           if (instance == null) {
               synchronized (Singleton.class) {
                   if (instance == null) {
                       instance = new Singleton();
                   }
               }
           }
           return instance;
       }
   }
   ```
   必须加 **volatile** 防止指令重排。

3. **静态内部类**（推荐）
   ```java
   public class Singleton {
       private Singleton(){}
       private static class Holder {
           private static final Singleton INSTANCE = new Singleton();
       }
       public static Singleton getInstance() {
           return Holder.INSTANCE;
       }
   }
   ```
   懒加载 + 线程安全 + 简洁。

4. **枚举单例**
   ```java
   public enum Singleton {
       INSTANCE;
   }
   ```
   最安全，防止反射/反序列化破坏。

# 4. synchronized 和 Lock 区别，实际场景怎么选？
## 区别
- synchronized：JVM 实现，自动加锁解锁，可重入，阻塞不可中断
- Lock：JDK 层面 API，手动 lock/unlock，支持**超时、中断、公平锁、多条件**

## 实际场景选择
- **简单同步、代码简洁** → synchronized
- **需要高级特性**（超时获取锁、尝试锁、多条件队列）→ Lock
- **高并发细粒度控制**（如分段锁）→ Lock / ReentrantLock
- 生产业务一般优先 **synchronized**，简单不易出错；复杂场景用 Lock。

# 5. 线程死锁如何避免？
## 死锁四个条件
互斥、请求与保持、不可剥夺、循环等待。

## 避免方案
1. **固定加锁顺序**（破坏循环等待）
2. **定时锁**，超时放弃（Lock.tryLock）
3. **一次性申请所有资源**
4. **避免锁嵌套**
5. **设置线程等待超时**
6. 线上用 jstack 排查死锁

# 1. 如何保证消息不丢失？
从**生产者 → Broker → 消费者**三个环节一起保证：

1. **生产者端**
   - 开启**发送确认机制**（Confirm 机制）
   - 失败重试 + 本地消息表补偿
   - 处理路由失败消息（Return 机制）

2. **Broker 端**
   - 队列、交换机、消息**全部持久化**
   - 搭建**集群 + 镜像队列/高可用副本**
   - 开启**落盘确认**，不丢数据

3. **消费者端**
   - 关闭自动 ACK，使用**手动 ACK**
   - 业务执行成功再确认，异常拒绝或进入死信
   - 配置死信队列，兜底异常消息

一句话总结：
**生产者确认 + Broker 持久化高可用 + 消费者手动ACK + 死信兜底**

# 2. 消息重复消费怎么解决？
MQ 只能保证 **至少一次投递**，无法避免重复，**必须靠业务幂等解决**。

常见方案：
1. **全局唯一 ID + Redis 判重**
   每条消息带 msgId，消费前先查 Redis 是否处理过
2. **数据库唯一索引**
   利用唯一约束，插入重复直接抛异常并忽略
3. **状态机/乐观锁**
   只有状态符合条件才执行，否则视为已处理
4. **分布式锁**
   同一消息同一时间只允许一个线程处理

核心：**允许重复投递，但不允许重复执行**。

# 3. 如何实现接口幂等？
和上面重复消费本质是同一个问题，通用方案：

1. **token 机制**
   - 前端先申请 token，后端保存到 Redis
   - 请求带上 token，执行前校验并删除 token
2. **唯一索引防重**
   订单号、流水号建唯一索引
3. **Redis 分布式锁**
   以业务唯一标识作为 key，加锁执行
4. **状态机控制**
   待支付 → 支付中 → 已支付，不可逆
5. **乐观锁 version**
   带版本号，只有版本匹配才更新

适用场景：
- 支付、退款、下单、回调、MQ 消费 **必做幂等**。

# 4. 消息积压如何处理？
线上高频事故题，标准排查思路：

1. **先定位原因**
   - 消费者挂了/消费极慢
   - 生产者流量暴增
   - 消费逻辑有阻塞（慢SQL、外部调用超时、死循环）

2. **紧急恢复**
   - 扩容消费者实例，提高并行消费能力
   - 临时优化消费逻辑，去掉慢操作、异步化
   - 清理无效消息，跳过异常消息

3. **避免恶化**
   - 禁止随意重启生产者，防止瞬间暴打
   - 必要时限流生产者
   - 长时间堆积可转移消息到临时队列，慢慢回放

4. **事后优化**
   - 提高消费速度
   - 线程池合理配置
   - 增加监控告警

# 5. 延迟队列实现方案？
## RabbitMQ
- 用**死信队列 + TTL** 实现
  消息设置过期时间 → 到期后进入死信队列 → 消费者监听死信队列
- RabbitMQ 3.5.8+ 支持 **插件化延迟队列**（x-delayed-message）更方便

## RocketMQ
- 内置支持 **18 个固定延迟等级**
  直接设置延迟级别，无需自己构建

## Kafka
- 自身不支持，常用方案：
  1. 时间轮算法
  2. 用 RocksDB 等外部轮询
  3. 按时间分片 + 定时扫描

通用业务场景：
- 订单超时未支付取消
- 收货超时自动确认
- 验证码过期

# 1. 缓存穿透、击穿、雪崩怎么解决？
## 穿透（查不存在的数据，一直打库）
- 原因：查询**不存在的 key**，缓存不命中，全部请求到数据库。
- 解决方案：
  1. **缓存空值**（null/默认值）并设置短过期
  2. **布隆过滤器**过滤不存在的 key
  3. 接口层参数校验、黑名单拦截

## 击穿（热点 key 过期，瞬间打库）
- 原因：某一个**热点 key 过期**，大量并发同时请求数据库。
- 解决方案：
  1. **互斥锁（分布式锁）**：缓存失效时串行查库
  2. **热点 key 永不过期**
  3. 后台定时主动刷新缓存

## 雪崩（大量 key 同时过期 / Redis 宕机）
- 原因：大量缓存**同一时间过期**，或 Redis 集群宕机。
- 解决方案：
  1. 过期时间**加随机值**，避免集体失效
  2. **多级缓存**（本地缓存 + Redis）
  3. Redis **集群高可用、哨兵、主从**
  4. 服务**降级、限流、熔断**
  5. 缓存预热

# 2. Redis 分布式锁如何实现？
最简标准回答：

1. **加锁**
   使用 `SET key value NX EX 超时时间` 原子命令。
   - NX：不存在才设置
   - EX：自动过期，避免死锁

2. **防误删**
   value 设为**唯一 ID（UUID/线程ID）**，解锁时先判断再删。

3. **解锁**
   用 **Lua 脚本**原子执行：判断值 → 匹配则删除。

4. **重入**
   用 `Redisson` 实现可重入锁（内部用 hash + 计数）。

5. **可靠性**
   - 锁超时：Redisson 自动**看门狗续期**
   - 高可用：红锁（生产一般不推荐，用主从+哨兵即可）

一句话：
**SET NX EX 原子加锁 + 唯一值 + Lua解锁 + Redisson 看门狗**

# 3. 缓存与数据库双写一致性方案？
常用三种方案，面试按这个说：

## 方案1：先更新数据库，再删除缓存（最常用）
- 核心思路：不更新缓存，只**删除**，下次读时重建。
- 问题：极短暂时间内可能脏数据，可接受。

## 方案2：延迟双删
- 先删缓存 → 更新DB → 延迟几百ms再删一次
- 降低并发不一致窗口

## 方案3：可靠最终一致性（强一致场景）
- 监听 binlog（Canal）
- 异步更新缓存，保证顺序一致

## 原则
- 强一致性：不用缓存，直接读库
- 高并发：接受**最终一致**，用**删除缓存而非更新缓存**

# 4. Redis 大Key、热Key问题？
## 大Key（value 特别大）
- 危害：网络IO阻塞、Redis 单线程卡顿、迁移超时
- 解决方案：
  1. 拆分为多个小 key
  2. 压缩存储
  3. 避免存储大 json、大列表
  4. 过期打散，避免集中删除

## 热Key（某 key 极高并发）
- 危害：流量集中到一台 Redis 实例，打挂节点
- 解决方案：
  1. **本地缓存（Caffeine/Cache）** 兜底
  2. 热Key **备份多份**（key+随机后缀）分散请求
  3. 拆分数据，减少并发竞争
  4. 服务限流

# 1. 接口限流怎么做？
## 常见限流算法
1. **计数器法**
   固定窗口统计请求数，简单粗暴，有临界突刺问题。
2. **滑动窗口**
   把时间分片，解决突刺问题，比计数器更平滑。
3. **漏桶算法**
   请求匀速处理，超出直接丢弃，保护系统不被打满。
4. **令牌桶算法（最常用）**
   固定速率放令牌，拿不到令牌拒绝，**支持突发流量**。

## 落地实现
- 单机：Guava RateLimiter（令牌桶）
- 分布式：Redis + Lua 实现滑动窗口/令牌桶
- 网关层：Sentinel、Hystrix、Gateway 限流
- 层级限流：网关层 + 应用层 + 数据库层

## 限流策略
- 拒绝请求
- 排队等待
- 服务降级

# 2. 分布式 Session 方案？
1. **Session 统一存储（最常用）**
    session 存在 Redis，多服务共享，Spring Session 一键接入。
2. **Token 无状态方案**
   JWT 存储用户信息，服务端不存 session，适合微服务、前后端分离。
3. **Cookie 加密存储**
   信息存在客户端 cookie，加密签名，不推荐存敏感信息。
4. **粘性 Session**
   Nginx ip_hash 绑定用户到同一台机器，不适合扩缩容。

**企业主流：Spring Session + Redis 或 JWT。**

# 3. 支付接口如何保证安全与幂等？
## 安全方面
- 使用 **HTTPS** 防抓包篡改
- **签名验签**（appId + 时间戳 + nonce + 参数 + 密钥）
- 防重放：时间戳过期 + nonce 一次性校验
- 敏感数据加密（手机号、身份证、银行卡）
- 权限校验、IP 白名单、风控规则

## 幂等方面
- **商户订单号唯一索引**
- **支付状态机**：待支付 → 支付中 → 已支付，不可回退

- 基于 **Redis 分布式锁**，防止重复提交

# 4. 大文件上传如何实现断点续传？
核心思路：**把文件切片，前端分片上传，后端合并校验**。

1. 前端计算文件 **MD5** 作为唯一标识
2. 把文件切成固定大小分片（如 2MB/片）
3. 前端请求后端：**该 MD5 已传了哪些片**
4. 只传未上传的分片，实现断点续传
5. 所有片传完，前端通知后端**合并文件**
6. 后端校验 MD5，确保文件完整

技术点：
- 前端：Blob.slice、并发上传、进度计算
- 后端：临时目录存储分片、原子合并、秒传（判断MD5已存在）

# 1. CPU 100% 如何排查？
**排查流程：**
1. `top` 找到 CPU 最高的 Java 进程 PID
2. `top -H -p PID` 找到最耗 CPU 的线程 TID
3. `printf "%x" 线程TID` 转 16 进制
4. `jstack PID | grep 16进制ID -A 20` 定位代码行

**常见原因：**
- 死循环
- 频繁 GC（FULL GC 狂飙）
- 正则表达式回溯
- 大量线程自旋 / CAS 空转
- 复杂计算、序列化压满 CPU

**解决：**
修复死循环、优化代码、调整 JVM、限流、异步化。

# 2. 内存飙升、OOM 如何排查？
**排查流程：**
1. `jstat -gc PID 1000` 看 Eden、Old 区增长 & FGC 次数
2. 出现 OOM 时加上参数自动 dump：
   `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=.`
3. 用 MAT/JProfiler 分析 dump 文件
   看大对象、GC Roots 引用、泄漏类

**常见原因：**
- 内存泄漏（ThreadLocal、静态集合、连接未关闭）
- 大对象集中创建（List 一次性加载海量数据）
- 堆设置过小
- 流、连接、文件未关闭

**解决：**
定位泄漏代码、分批查询、加大堆、及时释放资源。

# 3. 接口响应特别慢，怎么定位？
**排查思路（从外到内）：**
1. 看接口耗时分布：网关 → Nginx → 应用
2. `arthas trace 类名 方法名` 直接定位方法内部耗时
3. 查看数据库慢查询、外部调用超时（HTTP/Dubbo/MQ）
4. 看线程池是否阻塞、锁竞争严重
5. 看 Redis 阻塞、连接池耗尽

**常见瓶颈：**
- 慢 SQL（无索引、大分页、连表过多）
- 外部接口超时
- 锁等待、死锁
- 连接池打满
- 序列化耗时（超大 JSON）

**解决：**
优化 SQL、加缓存、异步化、超时控制、扩容连接池。

# 4. 应用频繁 FullGC 原因？
**常见原因：**
1. **老年代占用过高**
   内存泄漏，对象无法回收，一直进入老年代
2. **大量大对象**
   直接进入老年代，触发 FullGC
3. **元空间不足**
   大量反射、动态代理导致类加载过多
4. **System.gc() 主动调用**
5. **堆内存配置不合理**
   新生代太小，对象快速晋升老年代

**排查：**
`jstat -gcutil PID 1000`
看 Old 区使用率、FGC 次数、耗时。

**解决：**
找泄漏、优化大对象、调整 Xmx/Xms/Metaspace、禁用显式 GC。

# 5. 线程池耗尽、任务堆积怎么查？
**排查：**
1. `jstack PID | grep -A 20 pool` 看线程状态
2. 看线程池队列是否爆满、线程数达到 maximumPoolSize
3. 看任务是否大量 **WAITING / BLOCKED**
4. arthas `thread` 命令查看线程池状态

**常见原因：**
- 任务执行太慢（慢SQL、外部调用超时）
- 线程池核心线程数设置过小
- 队列设置不合理
- 锁竞争严重，大量阻塞
- 死锁或资源等待（数据库连接池耗尽）

**解决：**
优化任务执行速度、调大线程池、扩容连接池、异步化、降级限流。

# 1. MySQL CPU 100% 如何排查？
## 排查流程
1. `top` 看是否是 mysqld 进程占满 CPU
2. 登录 MySQL 执行：
   ```sql
   show processlist;
   ```
   查看正在执行的 SQL，找到耗时长、State 为 Sending data 的语句
3. 开启慢查询，用 `mysqldumpslow` 分析慢查询日志
4. `explain` 查看执行计划，看是否全表扫描、filesort

## 常见原因
- 大量**慢SQL**（无索引、大表查询、复杂连表）
- 高并发下大量**锁等待、事务长**
- 大量排序、分组（order by / group by 无索引）
- 大量并发请求打满 MySQL 计算能力

## 解决
-  kill 掉慢查询
-  加索引、优化 SQL
-  限流、降级、读写分离
-  分库分表

# 2. 数据库死锁如何排查？
## 排查方式
1. 查看死锁日志：
   ```sql
   show engine innodb status;
   ```
   找到 `LATEST DETECTED DEADLOCK` 段落
2. 查看事务等待：
   ```sql
   select * from information_schema.innodb_locks;
   select * from information_schema.innodb_lock_waits;
   ```

## 死锁常见原因
- 多个事务**交叉更新**不同行，加锁顺序相反
- 范围查询、gap lock 导致隐式锁定范围过大
- 长事务持有锁不释放
- 更新无索引，导致行锁升级为**表锁/间隙锁**

## 避免方案
- 固定**统一加锁顺序**
- 事务尽可能小，快进快出
- 更新语句加合理索引
- 避免长事务、避免在事务中调用远程接口

# 3. 慢查询如何定位与优化？
## 定位
1. 开启慢查询日志，设置阈值（如 1秒）
2. 使用 `mysqldumpslow` 或可视化工具分析
3. `show processlist` 实时观察
4. `explain` 分析执行计划

## 优化三板斧
1. **加索引**
   最左前缀、联合索引、避免索引失效
2. **改写SQL**
   减少连表、减少 select *、大分页优化、子查询转 join
3. **分数据**
   分库分表、分区表、历史数据归档

## 常见优化点
- 避免 select *
- 避免 like '%xx'
- 避免无索引 order by / group by
- 避免大事务、深度分页

# 4. 数据库连接池打满如何排查？
## 现象
应用报：
- 无法获取连接
- 超时等待连接
- 大量线程 BLOCKED

## 排查
1. 查看连接池监控（Druid/SkyWalking/Prometheus）
2. `show processlist` 看 MySQL 连接数、Sleep 连接
3. 查看是否有**慢SQL、长事务、锁等待**占用连接不释放
4. 检查是否未释放连接（try-with-resources、未关闭连接）

## 原因
- 执行太慢，连接长期占用
- 事务太长，不提交
- 连接池参数不合理（max-active 太小）
- 代码bug导致连接泄漏

## 解决
- 优化慢SQL、缩短事务
- 调大连接池上限 + 检查空闲回收
- 增加连接复用、避免连接泄漏
- 熔断、限流、降级

# 5. 主从延迟严重怎么处理？
## 原因
- 从库执行慢，主库写入量大
- 大事务、长SQL、无索引同步
- 从库硬件比主库差
- 主库并发写，从库单线程回放（传统异步复制）

## 解决方案
1. 开启**并行复制**（MTS）
2. 避免大事务，拆成小事务
3. 从库配置升级，与主库相当
4. 开启半同步复制，减少数据不一致风险
5. 分库分表，降低单库写入压力
6. 读写分离合理，避免从库压力过大

# 1. Redis 集群抖动、连接超时如何排查？

## 排查思路

2. `redis-cli --stat` 查看连接数、命中率
3. `info clients` 看连接数是否超限

## 解决方案
- 调整连接池参数：最大连接数、重试次数、超时时间
- 避免频繁创建销毁连接
- 网络优化，内网 DNS 稳定
- 大 Key 拆分，避免阻塞
- 集群扩容，避免热点节点过载

# 2. 缓存命中率低如何优化？
## 原因
- 缓存预热不足，大量冷数据请求
- Key 过期时间太短，频繁失效
- 缓存设计不合理，Key 分散、粒度太细
- 大量**缓存穿透**请求（不存在的数据）
- 缓存更新策略不当，频繁被删

## 优化方案
- 合理设置过期时间，避免短时间集体失效
- 增加缓存**预热**，启动时加载热点数据
- 合并小缓存，优化缓存粒度
- 缓存穿透防护：布隆过滤器 / 缓存空对象
- 热点缓存永不过期，后台主动刷新
- 避免频繁删除缓存，尽量用更新策略

# 3. Redis 阻塞、执行变慢？
## 典型原因
- **主线程被阻塞**（Redis 单线程核心原因）
  - 大 Key 读写 / 删除
  - 使用 `keys *`、`hgetall` 等阻塞命令
  - 大量 O(n) 类指令
- 持久化阻塞：AOF 重写、RDB 生成 fork 耗时
- 内存满，触发主动淘汰，大量 Key 清理
- 热 Key 并发极高，CPU 打满
- 主从同步、全量重同步
- Swap 交换、硬盘 IO 高、机器负载高

## 排查命令
```
info latency
slowlog get 10
info memory
info stats
```

## 解决方案
- 禁止使用 `keys *`、`flushdb` 等阻塞命令
- 大 Key 拆分为小 Key
- 避免大量集中过期，设置随机值
- 使用 pipeline、lua 减少网络开销
- 关闭透明大页 THP
- 开启 lazyfree 惰性删除
- 避免 AOF 每次同步，设置 everysec
- 热 Key 做本地缓存，减轻 Redis 压力

# 1. 前端调用接口异常，后端正常为什么？
典型表现：浏览器/前端报错，但后端日志正常、接口实际执行成功。

常见原因：
1. **跨域 CORS** 被浏览器拦截，请求根本没发出去
2. **返回格式不对**：后端返回 XML/文本/HTML，前端只认 JSON
3. **状态码不是 200**：401/403/405/415 等，前端判定异常
4. **Content-Type 不匹配**：前端 JSON，后端接收 form-data，或反之
5. **Nginx/网关拦截**：限流、WAF、权限、长度限制
6. **响应数据过大**，前端解析超时或截断
7. **HTTPS 证书问题、代理、VPN 导致请求异常**
8. **序列化异常**：循环引用、超长字段、null 处理异常

# 2. 跨域问题如何排查？
## 什么是跨域
协议、域名、端口任意一个不同，浏览器就会跨域拦截。

## 排查步骤
1. 看浏览器控制台：`Access-Control-Allow-Origin` 报错
2. 看 Network 里的 **OPTIONS 预检请求** 是否成功
3. 检查后端是否配置 CORS（Spring MVC / Nginx / 网关）
4. 确认是否允许自定义 Header、Credentials
5. 网关/nginx 是否覆盖了后端的跨域配置

## 解决方案
- Spring Boot 添加 `@CrossOrigin` 或全局配置 CorsFilter
- Nginx 配置 add_header 跨域响应头
- 网关统一处理跨域
- 前端使用代理（devServer proxy）

# 3. 接口超时，网关/nginx/服务分别可能什么原因？
## 网关层超时
- 路由配置错误
- 网关线程池/连接池打满
- 网关限流、熔断触发
- 下游服务注册失效

## Nginx 超时
- proxy_read_timeout 时间太短
- upstream 无响应、节点挂了
- Nginx 负载过高、文件打开数满
- 网络链路拥堵

## 应用服务超时
- 接口执行慢：慢SQL、外部调用、锁等待
- 线程池耗尽、队列积压
- GC 卡顿、OOM 前假死
- 数据库/Redis/MQ 中间件超时

## 一句话排查
**从外到内：网关 → Nginx → 应用 → 中间件 → 数据库**

# 4. 微服务调用失败：4xx/5xx 分别代表什么？
## 4xx 客户端错误（前端/调用方问题）
- 400：参数错误
- 401：未登录、token 无效
- 403：无权限
- 404：接口不存在
- 405：请求方法不对（GET/POST 反了）
- 415：Content-Type 不支持
- 429：限流

## 5xx 服务端错误（后端挂了或异常）
- 500：服务器代码异常
- 502：网关无效响应（下游服务挂了/返回非法）
- 503：服务不可用，过载、熔断、降级
- 504：网关超时，下游没及时返回

**口诀：4 开头是你传得不对，5 开头是服务挂了/慢了。**

# 5. 内网通、外网不通如何排查？
## 典型场景
服务器内部能 ping 通、能访问内部服务，但上不了外网/连不了外部服务。

## 排查思路
1. **DNS 解析问题**：nslookup 域名看是否能解析
2. **防火墙/安全组**：出站端口被限制（80/443/特定端口）
3. **网关/代理配置**：需要 HTTP_PROXY、HTTPS_PROXY
4. **路由问题**：默认网关未配置、路由表错误
5. **白名单限制**：外部服务只允许特定IP访问
6. **NAT/网络ACL**：云服务器出口策略限制

## 常用命令
- ping / telnet
- curl 外网地址
- route -n
- cat /etc/resolv.conf
- iptables -L

# 1. MQ 消息堆积如何快速恢复？
## 快速应急（线上优先）
1. **先扩容消费者**
   同一队列多消费者并行消费，提升消费速度。
2. **检查消费逻辑是否阻塞**
   慢SQL、外部调用超时、死循环、锁等待都会导致消费极慢。
3. **临时跳过异常消息**
   把报错消息丢死信队列，避免一条消息堵死整个队列。
4. **必要时临时限流生产者**
   防止堆积进一步恶化。

## 事后优化
1. 优化消费逻辑，异步化、批量处理
2. 合理设置消费者并发数
3. 增加线程池，提高消费能力
4. 建立堆积监控告警

## 一句话口诀
**堵的原因就两个：消费太慢、消费者太少。**

# 2. 分布式事务失败，数据不一致怎么处理？
## 常见不一致场景
- 订单已创建，库存没扣
- 支付成功，订单状态没改
- MQ 消息丢了或重复

## 处理思路
1. **先对账**
   定时任务对比上下游数据，找出不一致记录。
2. **修复数据**
   脚本/后台手动补偿，修正状态或金额。
3. **根因修复**
   - 消息丢失 → 加可靠消息、本地消息表
   - 超时未确认 → 加重试、幂等
   - 事务未提交 → 改用最终一致性方案

## 常用方案
- **TCC**：Try-Confirm-Cancel
- **可靠消息最终一致性**（最常用）
- **SAGA 长事务**：业务编排+补偿
- **Seata**：AT 模式（低代码接入）

原则：
**分布式不追求强一致，追求最终一致 + 可对账 + 可修复。**

# 3. 微服务链路报错，如何用链路追踪定位？
以 **SkyWalking、Pinpoint、Zipkin** 为例：

1. **拿 traceId**
   前端/日志里拿到唯一追踪 ID。
2. **在链路追踪系统搜索 traceId**
   看到完整调用链：
   网关 → 服务A → 服务B →  Dubbo/HTTP → MQ/DB
3. **看哪一步标红/耗时异常**
   定位是哪一个服务、哪一个接口抛异常或超时。
4. **看具体异常栈、依赖状态**
   是代码错？调用超时？连接失败？限流？
5. **跳转到对应服务日志**
   进一步定位代码行。

一句话：
**traceId 定位链路，颜色定位节点，日志定位代码。**

# 4. 服务雪崩、级联失败如何应对？
## 雪崩原因
一个服务挂了/变慢 → 调用方阻塞 → 线程池打满 → 整个链路不可用。

## 应对三板斧
1. **熔断**
   调用失败达到阈值，直接切断调用，快速失败。
2. **降级**
   非核心接口返回默认值、兜底数据、缓存数据。
3. **限流**
   限制入口流量，保护系统不被打崩。

## 技术实现
- Sentinel
- Hystrix/Resilience4j
- 网关限流 + 单机限流

## 其他配套
- **超时控制**：避免无限等待
- **线程池隔离**：一个服务故障不占用全部线程
- **重试机制**：注意幂等，避免放大流量
- **负载均衡 + 健康检查**：剔除异常节点

口诀：
**熔断防挂、降级兜底、限流抗压、超时保命。**

# 1. 磁盘满了如何清理？
## 排查步骤
1. 看整体磁盘使用
   ```bash
   df -h
   ```
2. 定位大文件目录
   ```bash
   du -sh /*
   du -sh /var/*
   ```
3. 找最大文件
   ```bash
   find / -type f -size +100M | sort
   ```

## 常见占满原因
- 应用日志太大（`/var/log/`、`logs/`）
-  Docker 镜像/容器/日志堆积
-  核心转储文件 `core dump`
-  临时文件未清理（`/tmp/`）
-  数据库 binlog 没清理

## 清理方式
- 清理过期日志、压缩旧日志
- 清理 Docker 无用镜像/卷
   ```bash
   docker system prune -a
   ```
- 配置日志轮转 `logrotate`
- 清理 `/tmp` 临时文件
- MySQL 清理过期 binlog

# 2. 端口被占用怎么查？
## 查找占用进程
```bash
lsof -i:8080
# 或
netstat -tulpn | grep 8080
# 或
ss -tulpn | grep 8080
```

## 拿到 PID 后
```bash
kill -9 PID
```

## 常见原因
- 旧进程没杀死，重复启动
- 端口配置冲突（多个服务用同一个端口）
- 系统服务占用（如 Nginx 80、MySQL 3306）
- 容器端口映射冲突

# 3. 应用启动失败、日志不输出排查思路？
## 1）先看系统级日志（没有应用日志时最有效）
- 查看启动输出日志（nohup.out、console 日志）
- 查看系统日志
  ```bash
  tail -f /var/log/messages
  journalctl -u 服务名
  ```

## 2）常见启动失败原因
- JDK 版本不兼容
- 堆内存 `-Xmx` 过大，机器内存不足
- 配置文件错误（yaml/yml 语法错、中文空格）
- 端口被占用
- 权限不足（日志目录无写权限）
- 依赖包缺失、打包不完整
- 连接池连不上数据库/Redis（但日志没打出来）

## 3）日志不输出原因
- 日志路径配置错误
- 目录无写入权限
- logback/log4j 配置文件损坏
- 启动参数指定了错误的日志配置
- 启动崩溃极早，日志组件还没初始化

# 4. 容器（Docker/K8s）服务异常重启原因？
## Docker 排查
```bash
docker ps -a
docker logs 容器ID
docker inspect 容器ID
```

## K8s 排查
```bash
kubectl get pods
kubectl logs pod名
kubectl describe pod pod名
```

## 常见重启原因
- **OOM 被系统杀死**（内存不够）
- 应用内部崩溃、异常退出
- **健康检查失败**（存活探针/就绪探针失败）
- 启动命令错误，进程直接退出
- 端口冲突、资源不足
- 连接不上依赖中间件，启动失败
- 磁盘满、文件句柄不足

## 重点看两项
- `OOMKilled`：内存溢出
- `RestartCount`、`LastState`、`ExitCode`
