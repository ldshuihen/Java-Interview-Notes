# Java后端面试8大链路

## 第一页：MySQL（必问，核心精简）

### 一、核心连环问（直接背）

\1. 项目MySQL瓶颈：查询慢（全表扫描）、高并发CPU/IO飙升、单表超千万、脏数据、单点风险

\2. 索引作用+原理：加速查询、降压力；InnoDB底层B+树（矮胖、叶子有序、聚簇/二级索引）

\3. 索引失效8场景：函数/运算、隐式转换、违最左前缀、like%开头、!=/is null、or无索引、order/group不遵前缀、优化器放弃

\4. 事务隔离级别：读未提交（三问题）→读已提交（解脏读）→可重复读（MySQL默认，解不可重复读）→串行化（无并发）

\5. MVCC实现：隐藏字段（trx_id/roll_pointer）+undo log版本链+ReadView，无锁并发读

\6. 慢查询优化：开慢日志→explain分析→优化索引/SQL→分批次→架构优化（缓存/读写分离）

\7. 分库分表：垂直（按业务/冷热字段）、水平（哈希/范围分片）；用ShardingSphere解决跨库问题

\8. 主从复制：主binlog→从IO拉取→SQL重放；延迟解决：拆慢SQL、提从库配置、关键读主库

### 二、亮点+难点（面试加分）

亮点：索引/SQL优化、事务控制、分库分表、读写分离

难点：最左前缀、回表/覆盖索引、MVCC底层、死锁（避免：固定顺序、小事务）

串讲：项目解决MySQL查询慢、并发高问题，通过索引/SQL优化、事务与MVCC保安全，高并发用读写分离、分库分表，处理主从延迟、死锁等问题。

## 第二页：JVM + Spring/SpringBoot（高频）

### 一、JVM（核心必背）

\1. 内存模型：线程私有（程序计数器、虚拟机栈、本地方法栈）；线程共享（堆、方法区/元空间、常量池）

\2. 对象分配：Eden→Minor GC→Survivor→年龄15→老年代；大对象直接进老年代

\3. 垃圾回收：算法（标记清除/复制/整理）；收集器（CMS低延迟、G1兼顾吞吐、ZGC超低延迟）

\4. FullGC触发：老年代不足、元空间不足、Minor GC后对象超老年代剩余、System.gc()、CMS并发失败

\5. OOM排查：开dump参数→MAT分析→看GC日志→查代码（集合/连接/ThreadLocal）→优化

\6. 类加载：加载→验证→准备→解析→初始化；双亲委派（父类先加载），破坏场景（SPI、热部署）

\7. 线上参数：-Xms4g -Xmx4g -Xmn2g -XX:MetaspaceSize=512m -XX:+UseG1GC -XX:+HeapDumpOnOutOfMemoryError

### 二、Spring/SpringBoot（核心必背）

\1. IOC+Bean生命周期：IOC控制反转+DI依赖注入；生命周期：实例化→填属性→@PostConstruct→初始化→AOP→使用→销毁

\2. AOP原理：动态代理（JDK需接口、Cglib继承，Spring有接口用JDK）；核心：切点、通知、切面

\3. 事务传播：REQUIRED（默认）、REQUIRES_NEW（新建）、NESTED（嵌套）

\4. 事务失效8场景：非public、try-catch吞异常、非Runtime异常、同类调用、未加注解、多线程、MyISAM、传播配置错

\5. 自动装配：@SpringBootApplication→@EnableAutoConfiguration→读取imports文件→条件装配

\6. 循环依赖：单例+setter注入，靠三级缓存；构造器/多例无法解决

\7. 三级缓存：1级（成熟Bean）、2级（早期Bean）、3级（ObjectFactory，生成代理）

串讲：Spring核心IOC管理Bean，AOP动态代理增强；SpringBoot自动装配简化配置，事务靠代理实现，循环依赖用三级缓存解决。

## 第三页：并发编程 + 分布式锁（高并发必问）

### 一、并发编程（核心必背）

\1. 线程池7参数：corePoolSize、maximumPoolSize、keepAliveTime、unit、workQueue、threadFactory、handler

\2. 线程池OOM：无界队列任务暴涨；解决：有界队列+合理最大线程+拒绝策略

\3. synchronized锁升级：无锁→偏向锁（单线程）→轻量锁（CAS自旋）→重量锁（阻塞），只能升级不降级

\4. volatile：保证可见性、禁止指令重排（DCL单例必须加），不保证原子性

\5. AQS原理：state状态+CLH队列+CAS；工具类：CountDownLatch（倒计时）、CyclicBarrier（齐步走）、Semaphore（控并发）

\6. ReentrantLock可重入：state加减，0释放；公平/非公平（默认）

\7. ThreadLocal内存泄漏：弱引用key+强引用value；解决：使用后remove()

### 二、分布式锁（核心必背）

\1. 为什么用：分布式环境本地锁失效，防止超卖、重复提交

\2. 实现方案：Redis（高并发首选）、ZK（强一致）、数据库（不推荐）

\3. Redis分布式锁：加锁SET lock_key 唯一值 NX PX 30000；解锁Lua脚本（原子）

\4. 锁超时：Redisson看门狗，定时续期（10秒续30秒）

\5. RedLock：向N个独立节点加锁，过半成功；解决主从切换丢锁，性能低

\6. Redis vs ZK：Redis高性能，ZK强一致；高并发选Redis，强一致选ZK

\7. 主从切换丢锁：异步复制导致；解决：业务容忍、ZK锁

串讲：用Redis+Redisson实现分布式锁，SET NX PX+Lua保证原子性，看门狗解决续期，根据业务选Redis/ZK，规避主从丢锁风险。

### 二、高并发系统设计（核心必背）

## 一、简历亮点：为什么项目引入Redis？为什么用Redis做缓存？

### 1. 项目为什么引入Redis？（核心回答）

1. **解决高并发下数据库压力过大问题**
   项目中读多写少（如商品详情、首页数据、用户信息），直接查MySQL会导致数据库CPU/IO飙升，扛不住高QPS，引入Redis做**分布式缓存**，把高频访问数据放内存，大幅减轻DB压力。
2. **提升接口响应速度，优化用户体验**
   MySQL是磁盘IO，响应毫秒级~秒级；Redis是**纯内存操作**，响应微秒级，接口性能提升10~100倍。
3. **支撑分布式场景**
   项目是集群部署，需要**跨进程、跨服务器共享数据**，本地缓存（Caffeine/Map）无法同步，Redis作为集中式缓存完美解决。
4. **丰富的数据结构满足业务需求**
   除了缓存，还用Redis做分布式锁、限流、计数器、排行榜、会话共享等。

### 2. 为什么选Redis做缓存，不选其他？
- **性能极致**：单机QPS 10W+，远超数据库；
- **高性能+高可用**：支持主从、哨兵、集群，保证服务稳定；
- **内存操作**：读写速度远超磁盘型数据库；
- **过期策略**：天然支持缓存过期，适配缓存场景；
- **生态成熟**：Java生态整合简单（Spring Cache/Redisson），企业级标准方案。

✅ 简历写法参考：
**在项目中引入Redis作为分布式缓存，解决高并发读场景下数据库性能瓶颈，提升接口响应速度；同时基于Redis实现分布式锁、会话共享，支撑系统高可用。**

## 二、简历难点：使用缓存后带来的问题（雪崩、穿透、击穿）
这是面试**核心难点**，必须说清：问题定义 + 解决方案。

### 1. 缓存雪崩
**定义**：
大量缓存**同时过期/Redis宕机**，所有请求直接打到数据库，导致数据库压力过载甚至宕机。

**解决方案**：
1. **给缓存过期时间加随机值**，避免批量同时失效；
2. **搭建Redis高可用集群**（主从+哨兵/Redis Cluster），防止单节点故障；
3. **开启服务降级、熔断限流**（Sentinel/Sentinel），保护数据库；
4. **设置缓存永不过期**（热点数据）+ 后台异步更新；
5. 多级缓存：本地缓存(Caffeine) + Redis缓存。

### 2. 缓存穿透
**定义**：
查询**数据库和缓存都不存在的数据**（如id=-1），缓存无法命中，请求每次都穿透到数据库，恶意攻击可打垮DB。

**解决方案**：
1. **接口参数校验**，过滤非法请求；
2. **缓存空值**：查询不存在的数据，缓存`null`，设置短暂过期时间；
3. **布隆过滤器**：将存在的数据key存入布隆过滤器，请求先过过滤器，不存在直接返回。

### 3. 缓存击穿
**定义**：
**一个热点Key过期**，大量并发请求同时访问该Key，直接打到数据库。

**解决方案**：
1. **热点Key永不过期**，后台定时异步刷新；
2. **加互斥锁**（Redisson分布式锁）：缓存未命中时，只允许一个线程去查数据库更新缓存。

## 三、核心难点：缓存和数据库的一致性如何解决？
面试高频压轴题，**必须说清楚方案+适用场景**。

### 标准回答（企业级方案）
我们项目采用：**Cache Aside Pattern（旁路缓存模式）** + **最终一致性**

1. **读流程**：先查缓存 → 缓存命中直接返回 → 未命中查数据库 → 写入缓存；
2. **写流程**：**先更新数据库，再删除缓存**（绝对不做更新缓存！）。

---

### 为什么是「先更库，再删缓存」？
- 更新缓存：会产生脏数据、浪费性能（写多读少场景无效）；
- 删除缓存：下一次查询自动重建最新数据，成本最低。

---

### 极端并发问题解决方案
如果出现「线程A查缓存未命中 → 查库 → 还没写缓存」时，「线程B更新了库并删除缓存」，会导致旧数据入缓存。
解决方案：
1. **延迟双删**：更新库 → 删除缓存 → 延迟几百ms → 再次删除缓存；
2. **可靠消息队列+重试**：通过Canal监听binlog，异步删除缓存，保证最终一致；
3. 业务允许**短暂最终一致性**（互联网99%场景适用）。

✅ 总结：**我们保证缓存与数据库最终一致性，核心方案：先更新数据库，后删除缓存。**

## 四、Redis挂了怎么办？（高可用方案）
### 回答逻辑：事前预防 + 事中兜底 + 事后恢复
1. **架构层面**：提前搭建**Redis主从+哨兵集群 / Redis Cluster集群**
   - 主节点挂了，哨兵自动故障转移，从节点升主，业务无感知；
2. **服务层面**：**熔断、降级、限流**
   Redis不可用时，接口直接降级（返回默认数据/静态页面），禁止请求打到数据库；
3. **缓存层面**：**本地缓存兜底**
   集成Caffeine本地缓存，Redis挂了读取本地缓存，保证核心接口可用；
4. **运维层面**：完善监控告警（CPU、内存、连接数、命中率），提前预警。

## 五、Redis 热Key、大Key 问题（高频难点）
### 1. 什么是热Key？
**访问量极高的Key**（如秒杀商品、热点新闻），大量请求集中在一个Redis节点，导致节点流量过载、网卡打满。

**影响**：
- 单节点压力过大，响应变慢；
- 集群数据倾斜，引发服务雪崩。

**解决方案**：
- 本地缓存（Caffeine）兜底，减少Redis请求；
- 热Key拆分：`key:1` → `key:1:1、key:1:2` 分散到不同节点；
- 使用Redis Cluster读写分离。

---

### 2. 什么是大Key？
**占用内存极大/元素极多的Key**（如String>10KB、Hash/List存几万数据）。

**影响**：
- 内存占用过高，触发内存淘汰，影响其他缓存；
- 网络IO阻塞，Redis性能急剧下降；
- 迁移、备份、过期删除时造成Redis阻塞卡顿。

**解决方案**：
- 大Key拆分：拆分为多个小Key，分批存储/读取；
- 清理无用大Key，规范数据结构；
- 使用scan代替keys，避免阻塞；
- 监控大Key，定时清理。

我在项目中引入Redis做分布式缓存，主要是为了解决**高并发读场景下MySQL性能瓶颈**，提升接口响应速度，同时实现分布式数据共享。

使用缓存后，我们重点解决了**缓存雪崩、穿透、击穿**三大问题：通过随机过期时间、集群高可用、布隆过滤器、分布式锁保证缓存稳定。

## 1. 你们项目 MySQL 遇到什么瓶颈？
**背诵回答：**
项目初期数据量小、并发低没问题，但随着用户量增长，出现几个典型问题：
1. **查询慢**：大表全表扫描，接口响应超时；
2. **数据库压力大**：高并发读导致CPU/IO飙升，影响写入；
3. **数据量大**：单表数据超千万，查询、更新性能急剧下降；
4. **高并发下数据安全问题**：超卖、重复提交、脏数据；
5. **单点风险**：单节点MySQL故障会导致整个业务不可用。

**最终解决方案：**
索引优化、SQL优化、读写分离、分库分表、缓存降级、事务与锁优化。

---

## 2. 为什么要建索引？索引原理是什么？
### 为什么建索引？
- 加快**查询速度**，避免全表扫描；
- 降低数据库CPU/IO消耗，支撑高并发。

### 索引原理（InnoDB）
- 底层数据结构：**B+ 树**
- 特点：
  1. 多路平衡，矮胖结构，减少IO次数；
  2. 叶子节点有序、形成链表，范围查询极快；
  3. 非叶子节点不存数据，只存键，内存利用率高。

### 聚簇索引 vs 二级索引
- **聚簇索引**：主键索引，叶子节点存**整行数据**；
- **二级索引**：普通索引，叶子节点存**主键值**。

---

## 3. 索引失效场景有哪些？（高频）
**背诵 8 大失效场景，直接说：**
1. **where 字段使用函数 / 运算**（如 `where YEAR(create_time)=2025`）
2. **隐式类型转换**（字符串不加引号，`id='123'` 变成 `id=123`）
3. **违反最左前缀原则**（联合索引跳过第一个字段）
4. **like 以 % 开头**（`like '%abc'`）
5. **使用 != / <> / IS NULL / IS NOT NULL**
6. **or 连接无索引字段**
7. **order by / group by 不遵守最左前缀**
8. 优化器判断**全表扫描更快**，放弃索引。

---

## 4. 事务隔离级别？脏读/不可重复读/幻读？
### 四大隔离级别（从低到高）
1. **读未提交**：最低级别，会出现脏读、不可重复读、幻读；
2. **读已提交（RC）**：解决脏读，仍有不可重复读、幻读；
3. **可重复读（RR）**：**MySQL默认**，解决脏读、不可重复读，**通过MVCC解决幻读**；
4. **串行化**：最高级别，无并发问题，但性能差。

### 三大问题
- **脏读**：读到未提交的事务数据；
- **不可重复读**：同一事务内，两次查询结果不一致；
- **幻读**：同一事务内，两次查询**条数不一致**。

---

## 5. MVCC 怎么实现的？（深挖难点）
**背诵回答：**
MVCC = 多版本并发控制，**InnoDB在RR、RC级别下实现无锁并发读**。

### 三大核心组件
1. **隐藏字段**
   - `trx_id`：生成这行数据的事务ID
   - `roll_pointer`：指向undo log历史版本
2. **undo log**：版本链，保存历史数据快照
3. **ReadView**：事务快照，判断版本可见性

### 核心逻辑
- 事务开始时生成ReadView；
- 根据**事务ID规则**判断当前版本是否可见；
- 不可见则顺着undo log版本链找历史快照；
- 实现**无锁、高并发、一致性读**。

---

## 6. 慢查询怎么优化？（万能步骤）
**背诵 6 步，面试官最爱：**
1. **开启慢查询日志**，定位慢SQL；
2. **explain 分析执行计划**，看type、key、rows、Extra；
3. **优化索引**：建立合适索引，避免失效；
4. **优化SQL**：减少子查询、join、select *、大in；
5. **分批次处理**：大SQL拆小，避免长事务；
6. **架构优化**：加缓存、读写分离、分库分表。

## 7. 分库分表怎么做？水平/垂直拆分？
### 垂直拆分
- **垂直分库**：按业务模块拆分（用户库、订单库、商品库）；
- **垂直分表**：大表拆小，把冷热字段分开。

### 水平拆分（最常用）
- 把**一张大表按规则切分成多张结构相同的小表**；
- 常用分片键：`user_id`、`order_id`；
- 分片算法：哈希取模、范围分片、一致性哈希。

### 面临问题
- 跨库分页、排序、聚合函数困难；
- 分布式事务、分布式ID、扩容困难；
- 解决方案：ShardingSphere、Seata。

---

## 8. 主从复制原理？主从延迟怎么解决？
### 主从复制原理
1. 主库写入，生成**binlog**；
2. 从库IO线程拉取binlog，写入**relay log**；
3. 从库SQL线程重放relay log，同步数据。

### 主从延迟原因
- 大事务、慢SQL、从库性能低、并发高。

### 解决方案
1. 减少大事务、拆分慢SQL；
2. 提升从库配置；
3. **读写分离后，关键强一致读主库**；
4. 多从库负载均衡；
5. 引入缓存兜底。

### 1. 最左前缀原则

- 联合索引必须**从左到右依次使用**，跳过前面字段，后面索引失效。

### 2. 回表、覆盖索引、索引下推

- **回表**：二级索引查到主键，再回聚簇索引查整行数据；
- **覆盖索引**：查询的字段都在索引里，**不需要回表**，速度极快；

### 4. 死锁产生与避免

- 产生原因：**多个事务互相等待对方锁资源**；
- 避免：
  1. 按固定顺序访问资源；
  2. 小事务、快提交；
  3. 合理索引，减少锁范围；
  4. 避免长事务。

## 1. 说一下 JVM 内存模型（运行时数据区）
**背诵回答：**
JVM 运行时数据区分为**线程私有**和**线程共享**两部分：

1. **线程私有（随线程创建销毁）**
   - **程序计数器**：记录当前执行行号，不会OOM
   - **虚拟机栈**：方法执行的栈帧（局部变量表、操作数栈）
   - **本地方法栈**：native 方法使用

2. **线程共享**
   - **堆**：存放对象实例，GC 主要区域，**最容易发生 OOM**
   - **方法区**：存放类信息、常量、静态变量（JDK8 后叫元空间 MetaSpace）
   - **运行时常量池**：存放字面量、符号引用

---

## 2. 对象分配流程？什么时候进入老年代？
### 对象分配流程
1. 优先在**Eden 区**分配
2. Eden 满 → **Minor GC**
3. 存活对象移到 **Survivor（From）**
4. 每熬过一次 GC，**年龄+1**
5. 达到阈值（默认15） → 进入**老年代**

### 进入老年代的 4 种情况
1. **年龄达标**（默认15岁）
2. **大对象直接进入**（-XX:PretenureSizeThreshold 设置）
3. **Survivor 区放不下**
4. **动态年龄判断**（一批对象总大小 > Survivor 50%）

---

## 3. 垃圾回收算法？CMS / G1 / ZGC 区别？
### 三大算法
- **标记-清除**：简单、但产生内存碎片
- **标记-复制**：适合新生代，无碎片、浪费空间
- **标记-整理**：适合老年代，无碎片、效率低

### 三大收集器对比（面试必背）
1. **CMS**
   - 老年代回收，**低延迟**
   - 4 阶段：初始标记→并发标记→重新标记→并发清除
   - 缺点：浮动垃圾、内存碎片、CPU 敏感

2. **G1**
   - 服务端常用，**兼顾吞吐量与延迟**
   - 把内存划分为多个 Region
   - 可预测停顿，优先回收价值最大的区域

3. **ZGC**
   - JDK11+，**超低延迟（亚毫秒级）**
   - 几乎全程并发，STW 极短
   - 适合大内存、低延迟场景

**一句话总结：**
CMS 低延迟、G1 兼顾性能、ZGC 超低延迟。

---

## 4. FullGC 什么场景会触发？（高频）
**背诵 6 个触发场景：**
1. **老年代空间不足**
2. **方法区（元空间）不足**
3. **Minor GC 后升入老年代的对象 > 老年代剩余空间**
4. **调用 System.gc()**（只是建议，不保证执行）
5. **CMS 出现 concurrent mode failure**
6. **大对象直接分配，老年代无空间**

**FullGC 是线上故障重灾区！**

---

## 5. 线上 OOM 怎么排查？（最加分题）
**万能排查步骤（背这个）：**
1. **保留现场**：开启参数 `-XX:+HeapDumpOnOutOfMemoryError` 自动生成 dump
2. **分析 dump**：用 MAT / JProfiler 查看大对象、泄漏点
3. **查看 GC 日志**：判断是内存溢出还是内存泄漏
4. **检查代码**：长生命周期集合、未关闭连接、ThreadLocal 未清理
5. **优化**：对象及时释放、加大堆内存、缩短对象生命周期

### 内存溢出 vs 内存泄漏
- **内存溢出 OOM**：实在不够用了
- **内存泄漏**：对象不用了但 GC 回收不了，越积越多导致 OOM

---

## 6. 类加载机制？双亲委派？为什么破坏？
### 类加载流程
加载 → 验证 → 准备 → 解析 → 初始化

### 双亲委派机制
1. 类加载时**先交给父类加载**
2. 父类加载不了才自己加载
3. 保证**核心类不会被重复加载、被篡改**

### 为什么要破坏？
1. **SPI 机制**（JDBC、Spring）需要子类加载
2. **热部署**（如 Tomcat）需要独立加载
3. **模块化加载**需要打破双亲委派

## 7. 线上 JVM 参数怎么配置？
**标准线上配置（直接背）：**
```
-Xms4g -Xmx4g          堆初始值=最大值，避免扩容
-Xmn2g                 新生代大小
-XX:MetaspaceSize=512m
-XX:+UseG1GC           使用G1
-XX:MaxGCPauseMillis=200  最大停顿时间
-XX:+HeapDumpOnOutOfMemoryError  OOM自动dump
```

**原则：**
- 堆内存 **不扩容、不浪费**
- 低延迟用 **G1/ZGC**
- 开启 **dump + GC日志**

## 1. Spring IOC 原理？Bean 生命周期？
### IOC 原理（背诵）
- **IOC 控制反转**：把对象创建、依赖注入的控制权交给 Spring 容器，不再自己 new。
- **DI 依赖注入**：容器自动把依赖对象赋值给 Bean，解耦。
- 核心流程：**扫描包 → 解析 BeanDefinition → 实例化 → 依赖注入 → 初始化 → 放入单例池**。

### Bean 生命周期（高频）
1. 实例化（调用构造方法）
2. 填充属性（依赖注入 @Autowired）
3. 初始化前（@PostConstruct）
4. 初始化（afterPropertiesSet / init-method）
5. 初始化后（AOP 代理）
6. 正常使用
7. 销毁（@PreDestroy / destroy-method）

## 2. AOP 实现原理？JDK 代理 vs Cglib？
### AOP 原理
- 底层是**动态代理**，在不修改源码的前提下增强方法（日志、事务、缓存）。
- 核心：**切点、通知、切面、连接点、织入**。

### 两种代理区别
- **JDK 动态代理**
  - 必须实现**接口**
  - 基于反射调用
- **Cglib 代理**
  - 不需要接口，**继承目标类**
  - 基于 ASM 字节码生成子类
  - 效率更高
- **Spring 规则**：有接口用 JDK，没接口用 Cglib。

## 3. Spring 事务传播机制？
**背 3 个最常用 + 场景**：

2. **REQUIRES_NEW**：**新建独立事务**，外部事务不影响我（扣钱、日志必须成功）

## 5. SpringBoot 自动装配原理？

## 6. Spring 循环依赖怎么解决？
### 背诵回答
- 只支持**单例、setter 注入**的循环依赖。
- 底层靠**三级缓存**提前暴露早期对象。
- 流程：
  1. A 实例化 → 放入三级缓存
  2. A 注入 B → 去创建 B
  3. B 注入 A → 从三级缓存拿到 A 早期引用
  4. B 创建完成 → A 继续创建完成

**构造器注入、多例模式无法解决循环依赖。**

3. **三级缓存（singletonFactories）**：ObjectFactory，用于生成代理对象

**核心作用**：解决循环依赖 + 保证 AOP 代理对象正确生成。

## 1. 线程池参数？核心线程、最大线程、队列、拒绝策略
### 7大核心参数（背诵）
1. **corePoolSize**：核心线程数，常驻不销毁
2. **maximumPoolSize**：最大线程数，队列满了才会新建
3. **keepAliveTime**：非核心线程空闲存活时间
4. **unit**：时间单位
5. **workQueue**：阻塞队列，存放等待任务
6. **threadFactory**：线程工厂，自定义线程名
7. **handler**：拒绝策略

### 执行流程（必背）
提交任务 → 核心线程未满 → 创建核心线程
核心线程满 → 放入队列
队列满 → 创建非核心线程
达到最大线程 → 执行**拒绝策略**

### 4大拒绝策略
1. **Abort**：抛异常（默认）
2. **CallerRuns**：调用者线程执行
3. **Discard**：丢弃任务
4. **DiscardOldest**：丢弃队列最旧任务

---

## 2. 线程池为什么容易 OOM？
**背诵回答：**
- 使用了**无界队列**（如 `LinkedBlockingQueue`），任务无限添加
- 队列无限膨胀 → 内存爆满 → OOM
- 线程数无限制创建也会 OOM

**解决方案：**
- 使用**有界队列**
- 合理设置最大线程数
- 加**拒绝策略**

---

## 3. synchronized 锁升级过程（高频）
**锁升级流程（无锁 → 偏向锁 → 轻量锁 → 重量锁）**
1. **无锁**：无竞争
2. **偏向锁**：只有一个线程用，标记线程ID，无CAS
3. **轻量锁**：竞争出现，CAS自旋，不阻塞
4. **重量锁**：竞争激烈，自旋失败，进入阻塞（OS锁）

**特点：**
- 锁**只能升级不能降级**
- 减少重量级锁的性能消耗

---

## 4. volatile 作用？禁止指令重排？
### 两大作用（背死）
1. **保证可见性**：一个线程修改，其他线程立刻看到
2. **禁止指令重排**（内存屏障）

### 指令重排作用
- 防止CPU优化乱序执行
- **典型场景：DCL单例必须加 volatile**，防止半初始化对象

**注意：volatile 不能保证原子性，不能替代锁！**

---

## 5. AQS 原理？CountDownLatch/CyclicBarrier/Semaphore
### AQS 原理（背诵）
- **AbstractQueuedSynchronizer** 队列同步器
- 核心：**state 状态 + 双向CLH队列 + CAS**
- 抢锁：CAS修改state
- 抢不到：进入队列阻塞等待唤醒

### 三个工具类区别
- **CountDownLatch**：倒计时门栓，**线程等待计数到0**
- **CyclicBarrier**：循环屏障，**大家都到齐再一起走**
- **Semaphore**：信号量，**控制并发数**

---

## 6. ReentrantLock 可重入原理
**背诵回答：**
- 可重入：同一个线程可以多次加锁，不会死锁
- 原理：
  - AQS 的 state 变量
  - 加锁：state+1
  - 解锁：state-1
  - state=0 才真正释放锁

### 公平锁 / 非公平锁
- 公平：按排队顺序
- 非公平：直接抢（默认，吞吐量高）

---

## 7. ThreadLocal 内存泄漏？
### 内存泄漏原因
- ThreadLocalMap 的 Entry 是 **弱引用 key，强引用 value**
- key 被回收，value 还在 → **value 永远无法回收** → 内存泄漏

### 解决方案
- **使用完手动 remove()**
- 线程池场景必须 remove，泄漏极严重

---

## 核心亮点（面试加分）

- 能合理**设置线程池参数**，避免OOM
- 掌握 **synchronized 锁升级** 高并发优化
- 熟练使用 **AQS 工具类** 实现线程同步
- 避免 **ThreadLocal 内存泄漏**

---

## 核心难点（深挖必问）

1. **AQS 底层**：state + CLH队列 + CAS
2. **CAS 自旋**：无锁更新，比较并交换
3. **ThreadLocal 内存泄漏**：弱引用key + 强引用value

这是**高并发面试必问、最容易答砸**的模块，我给你整理成**逐题标准答案**，直接背就能满分。

---

## 1. 为什么用分布式锁？

**背诵回答：**
分布式环境下，多个服务/多个进程同时操作共享资源（比如秒杀减库存、重复支付、重复提交），**本地锁失效**，必须用**分布式锁**来保证：**同一时间只有一个线程执行，防止超卖、脏数据、重复操作**。

---

## 2. 分布式锁实现方案？
**背诵 3 种主流方案：**
1. **Redis 分布式锁**：性能最高、高并发首选
2. **Zookeeper 分布式锁**：一致性强、可靠性高
3. **数据库锁**：简单、性能差，不推荐高并发使用

---

## 3. Redis 分布式锁怎么实现？
### 标准实现（背这个）
1. **加锁**：`SET lock_key unique_value NX PX 30000`
   - `NX`：互斥（只有不存在才能设置成功）
   - `PX 30000`：设置过期时间，防止死锁
   - 必须加**唯一值**，防止误删别人的锁
2. **解锁**：Lua 脚本（判断唯一值 + 删除锁，原子操作）

### 必须满足 3 个条件
- **互斥**
- **防死锁**（过期时间）
- **防误删**（唯一值 + Lua）

---

## 4. 锁超时、任务没执行完怎么办？
**背诵回答：看门狗机制**
- 问题：锁过期了，业务还没执行完 → 锁被释放 → 并发安全问题
- 解决方案：**Redisson 看门狗（WatchDog）**
  - 加锁成功后，**后台开启定时线程**
  - 每隔 10 秒（锁过期时间 1/3）**自动续期 30 秒**
  - 业务执行完，手动解锁，停止续期

---

## 5. RedLock 红锁原理？
**背诵回答：**
- 解决：**Redis 主从切换导致锁丢失**问题
- 原理：
  1. 向 **N 个独立 Redis 节点** 依次加锁
  2. **超过半数节点加锁成功** + **未超时** → 加锁成功
  3. 解锁时向所有节点解锁
- 缺点：性能低、成本高、官方不推荐生产使用

---

## 6. Zookeeper 分布式锁 vs Redis？
| 特性         | Redis 锁         | ZK 锁              |
| ------------ | ---------------- | ------------------ |
| **性能**     | 极高             | 一般               |
| **一致性**   | 最终一致         | **强一致**         |
| **可靠性**   | 主从切换可能丢锁 | 不丢锁             |
| **使用场景** | 高并发、性能优先 | 强一致、可靠性优先 |

**一句话：**
**并发高选 Redis，强一致选 ZK。**

---

## 7. 主从切换导致锁失效？
**背诵回答：**
- 问题：主节点加锁成功，**还没同步到从节点**，主节点宕机
- 从节点升主 → **无锁信息** → 其他线程可加锁 → 锁失效
- 解决方案：
  1. **RedLock 红锁**（不推荐）
  2. **业务容忍最终一致**（大多数互联网场景）
  3. **Zookeeper 强一致锁**

---

## 核心亮点（面试加分）

- 熟练使用 **Redisson 分布式锁**，解决秒杀超卖、重复提交
- 理解 **看门狗自动续期**，解决锁超时问题
- 知道 **主从切换风险**，具备高可用意识
- 能区分 Redis/ZK 锁，合理选型

---

## 核心难点（深挖必问）

1. **锁续期（看门狗）**：定时自动延长锁时间
2. **主从切换丢锁**：异步复制导致锁丢失
3. **分布式事务 vs 分布式锁**：
   - 锁：保证**并发串行执行**
   - 事务：保证**ACID，要么全成功要么全失败**

---

## 一句话串讲（面试直接背）

分布式环境下用分布式锁解决并发安全问题，项目中使用 **Redis + Redisson** 实现分布式锁，通过 **SET NX PX + 唯一值 + Lua 解锁** 保证原子性，用 **看门狗** 解决锁超时续期问题，了解主从切换锁丢失风险，根据业务选择合适方案。

---

## 1. 为什么引入 MQ？（三大核心）
**背诵回答：**
项目引入 MQ 主要为了 **解耦、异步、削峰**：
1. **解耦**：业务不用强依赖其他服务，下游系统自己消费消息
2. **异步**：主流程快速完成，非核心逻辑（通知、日志）异步处理
3. **削峰**：高并发流量先存入队列，避免压垮数据库/下游服务

**典型场景**：秒杀、下单、支付、短信推送、日志收集。

---

## 2. MQ 挂了怎么办？（高可用）
**背诵回答：**
1. **集群部署**：主从/多节点架构，避免单点
2. **限流、降级、熔断**：MQ 不可用时直接返回提示，不阻塞主流程
3. **消息落地**：重要业务先落库，MQ 恢复后补发
4. **多集群容灾**：跨机房部署

---

## 3. 消息重复消费怎么解决？（幂等性）
**背诵回答：**
MQ 无法保证只投递一次，**必须做消费端幂等**：
1. **唯一ID + 数据库唯一约束**（最常用）
2. **Redis 分布式锁**：消费前加锁，执行完释放
3. **状态机判断**：根据订单状态判断是否已处理
4. **全局ID去重表**：记录已消费ID

**核心：保证多次消费结果一致。**

---

## 4. 消息丢失怎么解决？（可靠性）
**三段式背诵：生产者、MQ、消费者**
1. **生产者**：
   - 开启**事务消息 / 确认机制（ACK）**
   - 失败重试 + 告警
2. **MQ 服务端**：
   - 开启**持久化**（磁盘+落盘）
   - 集群同步复制
3. **消费者**：
   - **业务执行成功再返回ACK**
   - 不要自动ACK

---

## 5. 消息顺序性怎么保证？
**背诵回答：**
保证 **同一业务ID 进入同一个队列，单线程消费**：
1. 发送时：按 **orderId / userId** 哈希取模路由
2. 队列：**一个队列只给一个消费者**
3. 消费：**单线程消费**，不并发

**适用场景**：订单创建→支付→发货，必须顺序执行。

---

## 6. 消息积压怎么处理？（高频难点）
**背诵 4 步急救方案：**
1. **紧急扩容消费者**，提高消费能力
2. **增加队列数量**，提高并行度
3. **临时禁止生产端发送**，先消费存量
4. **消息转移**：把积压消息转到新队列，慢慢消费

**根本原因**：消费太慢 / 下游阻塞 / 死循环。

---

## 7. 延时队列实现？
**背诵回答：**
延时队列 = 消息发送后**不立即消费**，等待指定时间再投递。

**使用场景**：
- 订单超时未支付自动取消
- 未评价自动好评

**RocketMQ**：内置18个固定延时等级
**Kafka**：用时间轮+自定义topic实现

2. **幂等性**：重复消费不产生脏数据
3. **消息积压**：高并发下快速恢复方案

---

## 一句话串讲（面试直接背）

## 1. 你们接口QPS多少？怎么扛住高并发？
### 回答模板（安全又专业）
- 普通接口：**几百 ~ 几千 QPS**
- 活动/秒杀：**上万 QPS**

### 高并发抗压方案（背这套）
1. **前端**：CDN、静态资源缓存、防抖节流
2. **网关**：限流、黑白名单
3. **应用层**：
   - 集群扩容

   - Redis 缓存扛读
   - MySQL 索引优化、读写分离、分库分表
5. **兜底**：熔断、降级、限流

一句话：**读多放缓存、写多用异步、流量做削峰、瓶颈做扩容。**

---

## 2. 接口限流怎么做？
### 限流目标
防止流量洪峰打垮服务，保证**核心可用**。

### 四种限流算法（背）
1. **计数器**：简单粗暴，存在临界突刺问题
2. **滑动窗口**：解决突刺，精度更高
3. **漏桶**：流量匀速出水，稳定平滑
4. **令牌桶**：**最常用**，允许突发流量，适合互联网

### 落地实现
- 单机：Guava RateLimiter（令牌桶）
- 分布式：**Redis + Lua** 实现分布式限流
- 框架：Sentinel、Hystrix、Spring Cloud Gateway

---

## 3. 服务降级、熔断怎么实现？
### 熔断（断路器）
- 连续失败达到阈值 → **打开断路器**，拒绝请求

- 目的：**防止级联故障、雪崩**

### 降级

- 目的：**保住核心链路**

### 框架

## 2. 单例模式为什么双重校验（DCL）？
### 背诵回答：
双重校验锁 = **volatile + 两次 null 判断 + 锁**
目的：**高并发下安全 + 高性能**

1. **第一次判空**：避免每次都加锁，提升效率
2. **加锁**：保证线程安全
3. **第二次判空**：防止并发时重复创建
4. **volatile**：**禁止指令重排**，防止返回半初始化对象

**经典代码（背）：**
```java
public class Singleton {
    private static volatile Singleton instance;
    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

## 3. 策略模式解决什么问题？
**背诵回答：**
解决**大量 if else / switch** 导致的代码臃肿、难以维护、难以扩展问题。

**结构：**
- 策略接口
- 多个实现类（不同策略）
- 策略上下文（选择并执行对应策略）

**场景：**
- 支付方式（微信/支付宝/银联）
- 订单优惠（满减/折扣/优惠券）
- 文件解析（PDF/Word/Excel）

**优点：**
- 扩展性极强，加一个策略只加一个类
- 代码清晰，符合开闭原则

## 4. 怎么写出优雅代码？
**背诵 6 条标准（面试官最爱）：**
1. **命名规范**：见名知意，不使用 a、b、temp
2. **方法单一职责**：一个方法只做一件事，不超过 20 行
3. **减少重复代码**：抽取工具类、公共方法
4. **面向接口编程**：便于扩展、便于测试
5. **异常处理统一**：不捕获不处理、不乱抛
6. **注释精简**：只写 why，不写 what

**核心：**
**可读性 > 可扩展性 > 性能**（除非性能瓶颈）

## 5. 如何避免大量 if else？
**万能 4 板斧（背这个直接高分）：**
1. **策略模式 + Map 注册**（最常用）
   提前把策略放入 Map，直接 get 取出执行
2. **工厂模式**
   工厂统一创建实例，外部无判断
3. **枚举 + 逻辑**
   枚举里定义行为，直接匹配
4. **责任链模式**
   按顺序过滤，适合审批、校验

**一句话：**
用**Map 注册 + 策略/工厂/枚举**替代 if else，让代码可插拔、易扩展。
