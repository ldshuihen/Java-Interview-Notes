电商订单系统是 Java / 后端 / 架构面试中**最核心的一类系统设计题**。像 Alibaba、JD.com、Amazon 的电商系统都会围绕订单设计很多复杂问题。

面试官通常会从 **订单 → 库存 → 支付 → 一致性 → 高并发** 一步一步深挖。

下面给你一份 **电商订单系统 12 个经典面试题（面试高频）**。

------

# 一、订单系统整体架构怎么设计？

典型微服务架构：

```
用户
 │
 │
Gateway
 │
 │
订单服务
 │
 ├── 商品服务
 ├── 库存服务
 ├── 支付服务
 ├── 用户服务
 └── 物流服务
```

核心思想：

```
订单系统 = 核心业务中枢
```

订单服务负责：

- 创建订单
- 查询订单
- 取消订单
- 支付状态更新

------

# 二、订单号如何设计？

常见方案：

1️⃣ 数据库自增
2️⃣ Redis INCR
3️⃣ Snowflake
4️⃣ 号段模式

实际生产最常见：

- Snowflake

订单号结构一般包含：

```
时间 + 机器ID + 序列号
```

优点：

- 唯一
- 有序
- 高并发

------

# 三、如何防止重复下单？

例如：

```
用户疯狂点击提交订单
```

解决方案：

### 1 前端防抖

按钮：

```
提交后禁用
```

### 2 Token 防重复提交

流程：

```
生成 token
↓
提交订单携带 token
↓
Redis 删除 token
```

### 3 分布式锁

例如：

```
userId + skuId
```

Redis：

```
SETNX
```

------

# 四、库存如何防止超卖？

最经典面试题之一。

错误方案：

```
select stock
update stock
```

并发会超卖。

正确方案：

### 1 Redis 扣库存

```
DECR stock
```

### 2 Lua脚本原子扣库存

```
判断库存
扣库存
```

### 3 数据库乐观锁

字段：

```
version
```

SQL：

```
update stock
set count = count -1
where version = ?
```

------

# 五、下单流程如何设计？

典型流程：

```
用户下单
 ↓
校验商品
 ↓
锁定库存
 ↓
创建订单
 ↓
支付
 ↓
扣减库存
 ↓
发货
```

很多系统会把：

```
锁库存
```

和：

```
扣库存
```

分开。

------

# 六、订单 30 分钟未支付如何取消？

常见方案：

1️⃣ 定时任务扫描
2️⃣ 延迟消息队列
3️⃣ Redis 延迟队列

大厂最常见：

```
MQ 延迟消息
```

例如：

- Apache RocketMQ

流程：

```
下单
 ↓
发送30分钟延迟消息
 ↓
MQ触发
 ↓
取消订单
 ↓
恢复库存
```

------

# 七、支付回调如何保证幂等？

支付平台例如：

- Alipay
- WeChat Pay

可能：

```
多次回调
```

解决方案：

数据库唯一约束：

```
orderId
```

代码：

```
if(order.status == PAID){
    return;
}
```

确保：

```
重复回调不会重复处理
```

------

# 八、库存扣减在哪一步做？

常见有三种模式。

### 1 下单扣库存

优点：

```
不会超卖
```

缺点：

```
库存锁定时间长
```

------

### 2 支付后扣库存

优点：

```
库存利用率高
```

缺点：

```
可能超卖
```

------

### 3 预扣库存（最常见）

流程：

```
下单 → 锁库存
支付 → 扣库存
取消 → 释放库存
```

------

# 九、订单系统如何抗高并发？

核心手段：

### 1 缓存

```
Redis
```

### 2 MQ 削峰

```
下单请求 → MQ
```

### 3 限流

```
Gateway
```

### 4 分库分表

订单表：

```
order_0
order_1
order_2
```

按：

```
userId hash
```

------

# 十、订单如何分库分表？

订单量非常大。

例如：

```
1天 1000万订单
```

常见策略：

### 1 按用户ID

```
userId % 32
```

优点：

```
查询用户订单快
```

------

### 2 按时间

```
order_2026_01
```

适合：

```
归档
```

------

# 十一、订单状态如何设计？

订单状态机：

```
待支付
 ↓
已支付
 ↓
已发货
 ↓
已完成
```

异常状态：

```
已取消
已退款
```

数据库字段：

```
status
```

------

# 十二、订单系统如何保证一致性？

涉及：

```
订单
库存
支付
```

常见方案：

### 1 分布式事务

例如：

- Seata

### 2 最终一致性

使用：

```
MQ + 补偿机制
```

例如：

```
扣库存失败
→ 回滚订单
```

------

# 面试总结（推荐回答）

如果面试官问：

**“你如何设计一个电商订单系统？”**

可以这样总结：

> 电商订单系统通常采用 **微服务架构**，订单服务作为核心服务，负责订单创建、状态管理等功能。
> 在高并发场景下，通过 **Redis缓存、MQ削峰、分库分表、分布式ID** 等方式提升性能，同时利用 **延迟消息、幂等设计、最终一致性机制** 来保证系统的可靠性。
