#### 1. HashMap（重中之重）

##### 面试官：你看过源码吗？说下 HashMap

**标准满分回答：**

> HashMap 是线程非安全的哈希表实现，JDK1.8 采用数组 + 链表 + 红黑树结构。
>
> put 时先通过 hash 扰动函数计算下标，发生碰撞用链表尾插法，链表长度超过 8 且数组≥64 转为红黑树。
>
> 扩容是 2 倍扩容，阈值是容量 * 0.75，JDK1.8 用尾插法修复了 JDK7 多线程死链问题。
>
> 它的核心是 hash 算法、扩容机制和树化规则，是空间和查询效率平衡的经典设计。

##### 🔥 底层数据结构

- **JDK1.7**：数组 + 链表
- JDK1.8：数组 + 链表 + 红黑树
  - 链表长度 ≥8 且数组长度≥64 → 转为红黑树
  - 红黑树节点 ≤6 → 退化为链表

##### 🔥 put 流程（必背）

1. 首次`put`：初始化数组（默认长度 16）
2. 计算`key`的 hash 值：**高 16 位异或低 16 位**（扰动函数，减少 hash 碰撞）
3. 按`(n-1) & hash`定位数组下标
4. 下标无元素 → 直接插入
5. 下标有元素：
   - key 相同 → 覆盖旧值
   - 是红黑树 → 树结构插入
   - 是链表 → 尾插法插入
6. 插入后判断是否需要扩容（size > 阈值）

##### 🔥 扩容机制

- 扩容条件：`元素个数 > 数组长度 * 加载因子（0.75）`
- 扩容大小：**2 倍扩容**（保证 hash 重新计算下标高效）
- JDK1.7：头插法 → **多线程下会形成环形链表，造成死循环**
- JDK1.8：尾插法 → 修复死链问题，保持链表顺序

##### 🔥 加载因子为什么是 0.75？

- 空间和时间的

  平衡选择：

  - ＞0.75：空间利用率高，但 hash 碰撞大幅增加
  - ＜0.75：碰撞少，但空间浪费严重

- 泊松分布统计：0.75 时碰撞概率最低

#### 2. ArrayList（高频）

- 底层：**动态 Object 数组**
- 初始化：
  - 无参构造：初始为空数组，首次 add 才扩容为 10
- 扩容机制：**1.5 倍扩容**（oldCapacity + oldCapacity >> 1）
- 核心 API：`add()`用`System.arraycopy()`拷贝数组，效率高
- 缺点：**非线程安全**，插入 / 删除中间元素需要移动数组

#### 3. LinkedList

- 底层：**双向链表**
- 每个节点：`item + prev + next`
- 优点：插入删除快，无需扩容
- 缺点：随机访问慢（需要遍历链表）

#### 4. ConcurrentHashMap（高频）

- JDK1.7：**Segment 分段锁**（ReentrantLock），并发度 16

- JDK1.8：

  CAS + synchronized

   \+ 红黑树

  - 锁粒度：从 Segment 降低到**数组头节点**
  - 并发性能大幅提升

- 关键：**key/value/next 都用 volatile 修饰**，保证内存可见性

#### 5. LinkedHashMap

- 继承 HashMap，底层：**哈希表 + 双向链表**
- 链表保存**插入顺序 / 访问顺序**
- 实现 LRU 核心：
  - `accessOrder = true` → 访问元素移到链表尾部
  - 重写`removeEldestEntry()`即可实现 LRU 缓存

#### 一、AQS（AbstractQueuedSynchronizer）—— JUC 核心灵魂

##### 1. 三大核心

1. **state 变量（volatile int）**
   - 同步状态：`0=无锁`，`≥1=持有锁`
   - 重入时`state++`，释放`state--`
2. **CLH 变体双向队列**
   - 阻塞等待线程的队列，**head 是虚节点，tail 指向最后**
   - 线程阻塞用`LockSupport.park()`，唤醒`unpark()`
3. **两种模式**
   - **独占模式（Exclusive）**：`ReentrantLock`
   - **共享模式（Shared）**：`CountDownLatch、Semaphore、CyclicBarrier`

##### 2. 核心原理（一句话）

**尝试修改 state → 成功则持有锁；失败则封装 Node 进入 CLH 队列，park 阻塞，被唤醒后再次竞争锁。**

##### 3. 关键方法

- `tryAcquire(int)`：尝试获取锁（子类实现）
- `acquire(int)`：获取失败→入队→阻塞
- `tryRelease(int)`：尝试释放
- `release(int)`：释放→唤醒后继节点

---

#### 二、ReentrantLock 基于 AQS 实现

##### 1. 可重入原理

通过 **state** 实现：
- 线程已持有锁时，再次获取：`state++`
- 释放时：`state--`，直到`state=0`才算完全释放

##### 2. 公平锁 vs 非公平锁（核心区别）

- **非公平锁（默认）**
  获取锁时**先直接 CAS 抢锁**，不看队列，抢到就执行，**吞吐高、可能饥饿**。
- **公平锁**
  先判断队列是否有等待线程，**有则必须排队**，严格 FIFO，**无饥饿、效率低**。

源码区别：
- 公平：`!hasQueuedPredecessors()` 才 CAS
- 非公平：直接`compareAndSetState(0,1)`

##### 3. 与 synchronized 对比

- ReentrantLock：**AQS+CAS+park**，可公平、可中断、可超时、多 Condition
- synchronized：**JVM 底层、对象头、锁升级**

#### 三、ThreadPoolExecutor（面试满分答案）

##### 1. 7 个核心参数

1. `corePoolSize`：核心线程数
2. `maximumPoolSize`：最大线程数
3. `keepAliveTime`：非核心线程空闲时间
4. `unit`：时间单位
5. `workQueue`：阻塞队列
6. `threadFactory`：线程工厂
7. `handler`：拒绝策略

##### 2. 任务提交流程（背这一段）

1. 任务进来，**核心线程未满 → 创建核心线程执行**
2. 核心已满 → **进入阻塞队列**
3. 队列满了 → **创建非核心线程**（直到最大线程数）
4. 达到最大线程 → **执行拒绝策略**

##### 3. 线程复用原理（源码重点）

- 线程启动后**不是执行一次就销毁**
- 在`runWorker()`中**死循环循环从队列取任务（getTask()）**
- 核心线程：**一直阻塞 take()**
- 非核心线程：**超时 poll()**，超时销毁

##### 4. 4 种拒绝策略

1. **AbortPolicy（默认）**：抛异常
2. **CallerRunsPolicy**：调用者线程执行
3. **DiscardPolicy**：直接丢弃
4. **DiscardOldestPolicy**：丢弃队列最老任务

#### 四、CountDownLatch / CyclicBarrier / Semaphore（AQS 应用）

##### 1. CountDownLatch（倒计时门闩）

- **AQS 共享模式**
- `state` 初始化为 N
- `countDown()`：`state-1`
- `await()`：阻塞直到`state=0`
- **一次性**，不可重置

##### 2. CyclicBarrier（循环屏障）

- **基于 ReentrantLock + Condition**
- 一组线程互相等待，全部到达才一起执行
- **可重置（reset）**，可循环使用
- 支持`Runnable barrierAction`，满了就执行

##### 3. Semaphore（信号量）

- **AQS 共享模式**
- `state` 表示许可证数量
- `acquire()`：获取（state-1，不足阻塞）
- `release()`：释放（state+1）
- 用于**限流、控制并发数**

##### 一句话区分

- **CountDownLatch：线程等待 N 个事件完成（一次性）**
- **CyclicBarrier：N 个线程互相等待（可循环）**
- **Semaphore：控制同时执行的线程数量（限流）**

---

#### 五、volatile（JVM 内存模型核心）

##### 1. 三大特性

1. **可见性**：一个线程修改，其他线程立即看到
2. **禁止指令重排序**：内存屏障
3. **不保证原子性**（i++ 不安全）

##### 2. 底层原理

- **内存屏障 Memory Barrier**
  - LoadLoad、StoreStore、LoadStore、StoreLoad
- 写前：**StoreStore**
- 写后：**StoreLoad**（强制刷回主存）
- 读后：**LoadLoad**（强制从主存读）

##### 3. happens-before（核心）

**volatile 写操作 happens-before 后续 volatile 读操作**
保证写结果对读可见。

---

#### 六、synchronized（锁升级机制 JDK1.8）

##### 1. 锁存在哪里？

**对象头 MarkWord**：
- 哈希码、分代年龄、**锁状态标志位**

##### 2. 锁升级流程（不可逆）

1. **偏向锁**
   - 只有一个线程用锁
   - 标记线程ID，**不加锁、无CAS**
2. **轻量级锁（自旋锁）**
   - 多线程交替执行，无竞争
   - CAS 替换 MarkWord，**自旋等待，不阻塞**
3. **重量级锁**
   - 竞争激烈、自旋失败
   - 内核态**Mutex Lock**，线程阻塞（OS 调度）

##### 3. 锁升级特点

**无竞争 → 偏向锁；轻度竞争 → 轻量锁；重度竞争 → 重量锁**
**只会升级，不会降级**

#### 超级精简背诵版（面试口述 1 分钟）

1. **AQS**：volatile state + CLH 队列 + 独占/共享，是所有锁的基础。
2. **ReentrantLock**：基于 AQS 独占，state 实现重入，公平/非公平区别在于是否直接抢锁。
3. **线程池**：核心线程→队列→最大线程→拒绝策略，复用靠死循环取任务。
4. **三大工具**：CountDownLatch 递减、CyclicBarrier 循环、Semaphore 限流。
5. **volatile**：可见性、禁重排、内存屏障，**不保证原子性**。
6. **synchronized**：对象头，锁升级：**偏向→轻量→重量**。

#### 一、类加载机制 — 双亲委派模型（重点：loadClass 源码）

##### 1. 类加载过程（固定 5 步）

**加载 → 验证 → 准备 → 解析 → 初始化**

- **加载**：读取 class 文件到方法区，生成 Class 对象
- **验证**：校验字节码合法性
- **准备**：为**静态变量分配内存、赋默认值**（int=0，Integer=null）
- **解析**：符号引用 → 直接引用
- **初始化**：执行 `<clinit>()`，静态代码块、静态变量赋值

##### 2. 双亲委派模型（核心）

> **子类先让父类加载，父类加载不了自己再加载。**

1. 先检查是否已加载
2. 没加载 → **交给父类加载器**
3. 父类无法加载 → 自己加载
4. 都不行 → ClassNotFoundException

##### 3. ClassLoader.loadClass 源码逻辑（面试必说）

```java
protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查是否已经加载过
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            // 2. 委派父类加载
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                c = findBootstrapClassOrNull(name);
            }
            // 3. 父类都加载不了，自己 findClass
            if (c == null) {
                c = findClass(name);
            }
        }
        return c;
    }
}
```

##### 4. 为什么要双亲委派？

1. **安全**：防止核心类（如 java.lang.String）被恶意替换
2. **保证类全局唯一**：同一个全类名只加载一次
3. **避免重复加载**

#### 二、String 不可变性、常量池、intern()（超级高频）

##### 1. String 为什么不可变？（源码 3 点）

1. **value 数组被 final 修饰**
   ```java
   private final char value[]; // JDK8及以前
   private final byte[] value; // JDK9+ 压缩为byte
   ```
2. **类被 final**：不能被继承，无法修改行为
3. **无任何修改 value 的方法**：substring/concat 都是**返回新 String**

##### 2. 不可变性好处

- **线程安全**
- **哈希值缓存**，HashMap 高效
- **字符串常量池共享**，节约内存

##### 3. 字符串常量池（StringTable）

- JDK7 及以后：**移到堆内存**
- 存放：**字面量字符串**（"abc"）
- 相同字面量**共用同一个对象**

##### 4. `intern()` 方法（面试必考）

> **将字符串尝试放入常量池，返回常量池中的对象引用。**

规则：
1. 常量池中已存在 → **直接返回池中的引用**
2. 不存在 → **将对象/引用入池，返回池引用**

经典面试题：
```java
String s1 = new String("a") + new String("b");
String s2 = s1.intern();
System.out.println(s1 == s2); 
```
- JDK6：false（复制对象入池）
- **JDK7+：true**（引用入池，s1 就是池引用）

#### 三、Integer 缓存 — IntegerCache

##### 1. 源码核心

```java
private static class IntegerCache {
    static final int low = -128;
    static final int high = 127; // 可通过 -XX:AutoBoxCacheMax 修改
    static final Integer cache[];

    static {
        // 初始化缓存 [-128, 127]
    }
}
```

##### 2. 自动装箱规则

`Integer i = 100;` 等价于 `Integer.valueOf(100);`

**valueOf 源码逻辑：**
```java
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}
```

##### 3. 经典面试题（必背）

```java
Integer a = 100;
Integer b = 100;
Integer c = 200;
Integer d = 200;

System.out.println(a == b); // true  缓存内，同一对象
System.out.println(c == d); // false 超出范围，new 两个对象
```

##### 4. 其他包装类缓存

- **Byte/Short/Long**：缓存 **-128 ~ 127**
- **Character**：缓存 **0 ~ 127**
- **Float/Double**：**无缓存**

#### 一、IoC 容器启动流程 — refresh() 12 个核心步骤（必背）

Spring 容器启动**唯一核心方法：`AbstractApplicationContext.refresh()`**

1. **prepareRefresh()**  
   刷新前准备：标记启动时间、激活配置文件、环境准备。
2. **obtainFreshBeanFactory()**  
   获取新的 BeanFactory：解析 XML/注解，**扫描所有 BeanDefinition**。
3. **prepareBeanFactory(beanFactory)**  
   BeanFactory 准备工作：设置类加载器、忽略自动装配接口、添加**ApplicationContextAwareProcessor**。
4. **postProcessBeanFactory(beanFactory)**  
   后置处理 BeanFactory（空方法，留给子类扩展）。
5. **invokeBeanFactoryPostProcessors(beanFactory)**  
   执行 **BeanFactory 后置处理器**：扫描、解析 `@Configuration`、`@ComponentScan`。
6. **registerBeanPostProcessors(beanFactory)**  
   注册所有 **Bean 后置处理器**（AOP、依赖注入、循环依赖都靠它）。
7. **initMessageSource()**  
   初始化国际化资源。
8. **initApplicationEventMulticaster()**  
   初始化事件广播器。
9. **onRefresh()**  
   模板方法，初始化特殊 Bean（如 Tomcat 端口）。
10. **registerListeners()**  
    注册监听器。
11. **finishBeanFactoryInitialization(beanFactory)**  
    **核心！实例化所有非懒加载单例 Bean**（真正创建 Bean 的地方）。
12. **finishRefresh()**  
    发布容器启动完成事件，启动完成。

> 一句话总结：  
> **准备 → 构建工厂 → 扫描BeanDefinition → 注册处理器 → 实例化Bean → 完成启动**

#### 三、循环依赖 — 三级缓存（Spring 最难点，必问）

##### 三级缓存是什么（源码级）：

```java
// 一级缓存：成品Bean（完全初始化好）
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>();

// 二级缓存：半成品Bean（实例化但未完成初始化）
private final Map<String, Object> earlySingletonObjects = new HashMap<>();

// 三级缓存：工厂对象，用于生成代理（解决AOP循环依赖）
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>();
```

##### 解决流程（面试必说这一段）：

1. A 实例化 → 放入**三级缓存**（`ObjectFactory`）
2. A 注入属性，发现依赖 B
3. B 实例化 → 放入**三级缓存**
4. B 注入属性，发现依赖 A
5. B 去缓存找 A：
   - 一级没有 → 二级没有 → **从三级获取 ObjectFactory**
   - 得到**早期代理对象/原始对象**，放入**二级缓存**，删除三级
6. B 完成初始化 → 加入**一级缓存**
7. A 拿到 B，完成初始化 → 加入**一级缓存**

##### 为什么必须三级？

- **一级：存完整Bean**
- **二级：存早期原始Bean**
- **三级：存工厂，延迟生成AOP代理，保证全局唯一代理**  
**少一级都无法同时解决 循环依赖 + AOP 代理。**

---

#### 四、AOP 实现原理

##### 1. 核心：动态代理

- **JDK 动态代理**：被代理类**实现接口**，生成接口实现类代理
- **CGLIB**：被代理类**无接口**，生成**子类继承**代理

##### 2. JDK vs CGLIB

- JDK：接口、快、无额外依赖
- CGLIB：子类、不能代理final方法、性能略低

##### 3. AOP 织入时机

**在 Bean 生命周期的：`BeanPostProcessor#postProcessAfterInitialization`**  
创建代理对象，替换原始 Bean，放入容器。

##### 4. AOP 核心组件

- `@Aspect` 切面
- `@Pointcut` 切点
- `@Before/@After/@Around/@AfterReturning/@AfterThrowing` 通知
- **Advisor**：**通知 + 切点**（Spring 底层真正存储的对象）

---

#### 五、Spring 事务 — @Transactional 传播机制（源码核心）

##### 1. 事务本质：

**AOP 代理 + ThreadLocal 保存 Connection + 事务管理器**

##### 2. 7 大传播机制（高频必背 3 个）

1. **REQUIRED（默认）**  
   有事务则加入，没有则新建（**最常用**）
2. **REQUIRES_NEW**  
   **挂起当前事务，新建独立事务**（内层失败不影响外层）
3. **NESTED**  
   嵌套事务，**保存点（savepoint）**，内层回滚只回滚到保存点，外层可继续提交
4. SUPPORTS：有事务就用，没有就非事务
5. NOT_SUPPORTED：非事务，挂起事务
6. MANDATORY：必须有事务，否则抛异常
7. NEVER：不能有事务，否则抛异常

##### 3. 事务失效场景（高频面试坑）

- **非public方法**（CGLIB 也不行）
- **同类内部调用（this.method()）** 不走代理
- 异常被**catch 吃掉**
- 抛出**非RuntimeException/Error**（默认只回滚运行时异常）
- 多线程调用（事务基于 ThreadLocal）

##### 4. 源码核心流程

1. `TransactionInterceptor` 拦截
2. `PlatformTransactionManager` 获取事务
3. 根据**传播机制**判断：新建/加入/挂起
4. 执行目标方法
5. 异常 → 回滚；正常 → 提交

---

#### 超级精简背诵版（面试 1 分钟口述）

1. **refresh() 12步**：准备 → 建工厂 → 扫描Bean → 注册处理器 → 实例化Bean → 启动完成
2. **Bean生命周期**：实例化 → 赋值 → Aware → 前置 → 初始化 → 后置(AOP) → 使用 → 销毁
3. **三级缓存**：一级成品、二级半成品、三级工厂；解决循环依赖+AOP代理唯一
4. **AOP**：JDK接口代理、CGLIB子类代理，在Bean后置处理器生成代理
5. **事务**：AOP+Connection+ThreadLocal，默认REQUIRED，同类调用、异常捕获会失效

---

## 第一套：Java基础 + JVM

### Java基础（必会）

**Q1. int 和 Integer 的区别？Integer 缓存范围是多少？**

> 考察点：基本类型与包装类、自动拆装箱、IntegerCache

参考答案要点：

- int 是基本类型，Integer 是包装类，存储位置不同（栈 vs 堆）
- 自动装箱底层调用 `Integer.valueOf()`，拆箱调用 `intValue()`
- IntegerCache 缓存范围 **[-128, 127]**，该范围内 `==` 比较返回 true
- 超出范围每次 new 新对象，`==` 返回 false，应用 `equals()` 比较

------

**Q2. String、StringBuilder、StringBuffer 的区别？**

> 考察点：不可变性、线程安全、性能

参考答案要点：

- String 不可变，每次拼接产生新对象，适合少量固定字符串
- StringBuilder 可变、线程不安全，单线程拼接首选，性能最好
- StringBuffer 可变、线程安全（方法加 synchronized），多线程场景使用
- 字符串常量池：`String s = "abc"` 复用常量池，`new String("abc")` 堆上新建

------

**Q3. == 和 equals() 的区别？hashCode 和 equals 的关系？**

> 考察点：对象比较、契约规范

参考答案要点：

- `==` 比较引用地址（基本类型比值），`equals()` 默认也是比地址，需重写才比内容
- 重写 equals() **必须** 同时重写 hashCode()
- 契约：equals 相等 → hashCode 必须相等；hashCode 相等 → equals 不一定相等
- HashMap 中先比 hashCode 再比 equals，两者不一致会导致 key 找不到

------

**Q4. 说说 final、finally、finalize 的区别？**

参考答案要点：

- `final`：修饰类（不可继承）、方法（不可重写）、变量（不可重新赋值）
- `finally`：try-catch 中必定执行的代码块，用于资源释放
- `finalize()`：GC 回收对象前调用，已过时，不可靠不推荐使用

------

**Q5. 异常体系结构是怎样的？checked 和 unchecked 异常的区别？**

参考答案要点：

- 顶层：`Throwable` → `Error`（系统级，不处理）和 `Exception`
- checked 异常：编译期必须处理，如 `IOException`、`SQLException`
- unchecked 异常：继承 `RuntimeException`，如 `NPE`、`IndexOutOfBoundsException`
- Spring 事务默认只回滚 `RuntimeException`，checked 异常不回滚（可配置）

**Q6. JVM 内存区域是怎么划分的？**

> 考察点：运行时数据区，几乎必问

```
线程私有：程序计数器 / 虚拟机栈 / 本地方法栈
线程共享：堆（对象实例）/ 方法区（类信息、常量池）
JDK8：方法区实现从 PermGen 改为 Metaspace（本地内存）
```

**Q7. 什么是OOM？各区域OOM原因是什么？**

| 区域          | OOM原因                      |
| ------------- | ---------------------------- |
| 堆            | 对象过多无法回收，内存泄漏   |
| 栈            | 递归过深，StackOverflowError |
| 方法区/元空间 | 动态生成大量类（如CGlib）    |
| 直接内存      | NIO ByteBuffer 使用过多      |

------

**Q8. GC 垃圾回收算法有哪些？各有什么优缺点？**

参考答案要点：

- **标记-清除**：产生内存碎片
- **标记-整理**：无碎片，但移动对象开销大
- **复制算法**：效率高无碎片，但浪费一半空间（新生代 Eden/Survivor 用此）
- **分代收集**：新生代用复制，老年代用标记-整理，目前主流

------

**Q9. 说说 CMS 和 G1 收集器的区别？**

| 对比项   | CMS           | G1             |
| -------- | ------------- | -------------- |
| 目标     | 最短停顿时间  | 可预测停顿时间 |
| 内存划分 | 新生代/老年代 | Region 分区    |
| 碎片问题 | 有内存碎片    | 无（整理）     |
| 适用场景 | 4G以下堆      | 大堆（推荐）   |
| JDK9+    | 废弃          | 默认GC         |

------

**Q10. 类加载过程是怎样的？双亲委派模型是什么？**

参考答案要点：

- 加载流程：**加载 → 验证 → 准备 → 解析 → 初始化**
- 双亲委派：子类加载器先委托父类加载，父类找不到再自己加载
- 好处：避免类重复加载，保证核心类库安全
- 打破场景：Tomcat（WebAppClassLoader隔离各应用）、SPI机制、OSGi

## 第二套：并发 + 集合源码

**Q1. HashMap 的底层结构？put 流程是怎样的？**

参考答案要点：

- JDK8：数组 + 链表 + 红黑树（链表长度 ≥8 且数组长度 ≥64 转红黑树）
- put 流程：计算 hash → 找数组下标 → 无冲突直接放 → 有冲突链表/红黑树插入 → 超阈值扩容
- 扩容：2倍扩容，rehash，JDK8 用高低位链表优化，避免死链
- 加载因子 0.75：时间与空间的均衡，太小频繁扩容，太大冲突增多

------

**Q2. HashMap 和 ConcurrentHashMap 的区别？**

参考答案要点：

- HashMap 线程不安全，多线程下可能死循环（JDK7）或数据丢失
- ConcurrentHashMap JDK7 用分段锁（Segment），JDK8 改为 CAS + synchronized（锁单个桶头节点）
- JDK8 中 put 时：无竞争用 CAS，有竞争用 synchronized，粒度更细性能更好
- size() 方法：用 baseCount + CounterCell 数组分散统计，避免竞争

------

**Q3. volatile 关键字的作用？能保证原子性吗？**

参考答案要点：

- 保证**可见性**：写入后立即刷新主内存，其他线程读取主内存最新值
- 保证**有序性**：禁止指令重排序（内存屏障）
- **不保证原子性**：`i++` 是三步操作，volatile 无法保证，需用 AtomicInteger
- 典型应用：双重检查单例的 instance 变量必须加 volatile

**Q4. synchronized 和 ReentrantLock 的区别？**

| 对比项   | synchronized          | ReentrantLock          |
| -------- | --------------------- | ---------------------- |
| 实现     | JVM层（monitorenter） | JDK层（AQS）           |
| 锁释放   | 自动释放              | 必须手动 unlock        |
| 可中断   | 不可中断              | 支持 lockInterruptibly |
| 公平锁   | 不支持                | 支持                   |
| 条件变量 | 一个waitSet           | 多个 Condition         |
| 性能     | JDK6后差距不大        | 高竞争下略好           |

**Q5. 线程池的核心参数有哪些？任务提交流程是怎样的？**

```
corePoolSize     核心线程数（常驻）
maximumPoolSize  最大线程数
keepAliveTime    非核心线程空闲存活时间
workQueue        任务队列
threadFactory    线程工厂
handler          拒绝策略
```

任务提交流程：

```
① 核心线程未满 → 创建核心线程执行
② 核心线程已满 → 放入队列
③ 队列已满且未达最大线程 → 创建非核心线程
④ 达到最大线程且队列满 → 执行拒绝策略
```

**Q6. AQS 是什么？说说它的核心原理**

参考答案要点：

- AbstractQueuedSynchronizer，并发工具的底层框架
- 核心：`state` 变量（表示锁状态）+ CLH 变体队列（等待线程）
- 独占模式：ReentrantLock，acquire/release
- 共享模式：CountDownLatch、Semaphore，acquireShared/releaseShared
- 模板方法模式：子类实现 `tryAcquire`、`tryRelease` 等方法

## 第三套：数据库 + Redis

**Q1. MySQL 索引为什么用 B+ Tree 而不是 B Tree？**

参考答案要点：

- B+ Tree 非叶子节点只存 key，不存数据，每页能存更多 key，树更矮，IO 次数更少
- 叶子节点用链表相连，**范围查询效率极高**
- B Tree 数据分散在各层节点，范围查询需要多次回溯
- 相比红黑树：B+ Tree 多路，层高远低于红黑树，减少磁盘IO

------

**Q2. 什么是回表？如何避免？**

参考答案要点：

- 非聚簇索引叶子节点存的是主键值，查到主键后还需再查聚簇索引获取行数据，称为**回表**
- 避免方式：**覆盖索引**，让查询的字段全部包含在索引中，不需要回表
- `SELECT id, name FROM t WHERE name = 'xx'`，若 name 是索引，只需查索引树即可

------

**Q3. 事务的隔离级别有哪些？MySQL 默认是哪个？**

| 隔离级别         | 脏读 | 不可重复读 | 幻读          |
| ---------------- | ---- | ---------- | ------------- |
| READ UNCOMMITTED | ✅    | ✅          | ✅             |
| READ COMMITTED   | ❌    | ✅          | ✅             |
| REPEATABLE READ  | ❌    | ❌          | ✅（部分解决） |
| SERIALIZABLE     | ❌    | ❌          | ❌             |

- MySQL 默认 **REPEATABLE READ**，通过 MVCC + 间隙锁解决幻读

------

**Q4. MVCC 是什么原理？**

参考答案要点：

- Multi-Version Concurrency Control，多版本并发控制
- 每行数据有 `trx_id`（事务ID）和 `roll_pointer`（指向undo log链）
- 读操作生成 **ReadView**（快照），根据事务ID判断哪个版本可见
- RR 级别：事务开始时生成一次 ReadView，整个事务看同一快照
- RC 级别：每次 SELECT 生成新 ReadView，所以能读到已提交的最新数据

------

**Q5. Redis 缓存穿透、击穿、雪崩的区别和解决方案？**

| 问题 | 原因                        | 解决方案                    |
| ---- | --------------------------- | --------------------------- |
| 穿透 | 查询不存在的key，每次打到DB | 布隆过滤器 / 缓存空值       |
| 击穿 | 热点key过期，大量请求打到DB | 互斥锁重建 / 逻辑过期不删除 |
| 雪崩 | 大量key同时过期             | 过期时间加随机值 / 多级缓存 |

------

**Q6. Redis 分布式锁怎么实现？有什么问题？**

参考答案要点：

- 基础实现：`SET key value NX PX 30000`（原子操作，设置值同时设置过期时间）
- 释放锁：用 Lua 脚本保证"判断+删除"原子性，避免误删其他线程的锁
- 问题：主从切换时锁可能丢失（master未同步到slave就宕机）
- 更可靠方案：**Redisson** 看门狗机制自动续期，或使用 **RedLock** 算法（多节点）

## 第四套：分布式 + 微服务

**Q1. 什么是 CAP 理论？微服务中如何取舍？**

参考答案要点：

- C（一致性）、A（可用性）、P（分区容错性），三者不可同时满足
- 分布式系统 P 必须保证，只能在 CA 间取舍
- **CP**：Zookeeper（选举期间不可用，但数据一致）
- **AP**：Eureka（允许脑裂，但保持可用）、Nacos 可切换

------

**Q2. 常见的分布式事务方案有哪些？**

| 方案       | 原理               | 优缺点                     |
| ---------- | ------------------ | -------------------------- |
| 2PC        | 协调者两阶段提交   | 简单，但有阻塞，协调者单点 |
| TCC        | Try-Confirm-Cancel | 性能好，但业务侵入大       |
| 本地消息表 | 消息+本地事务      | 可靠，实现复杂             |
| Seata AT   | 自动补偿           | 侵入小，适合CRUD业务       |
| Saga       | 长事务拆分补偿     | 适合长流程，不保证隔离性   |

------

**Q3. 服务注册发现是怎么工作的？Nacos 和 Eureka 的区别？**

参考答案要点：

- 服务启动时注册到注册中心，消费者定期拉取服务列表，调用时通过负载均衡选择实例
- Nacos 支持 AP/CP 切换，Eureka 只支持 AP
- Nacos 支持配置中心功能，Eureka 不支持
- Nacos 使用主动探活（心跳+健康检查），Eureka 依赖心跳，有保护模式

------

**Q4. 什么是服务熔断和降级？有什么区别？**

参考答案要点：

- **熔断**：下游服务异常率过高时，自动断开调用，保护上游，类似电路断路器
- **降级**：服务不可用时返回兜底数据（默认值/缓存），保证核心功能可用
- 区别：熔断是触发条件（异常率），降级是处理结果（返回兜底）
- Sentinel：滑动时间窗口统计，支持 QPS 限流、异常比例熔断、热点参数限流

------

**Q5. 如何保证消息队列的消息不丢失？**

参考答案要点（以 RocketMQ 为例）：

```
生产者：同步发送 + 失败重试，确认 Broker 收到
Broker： 同步刷盘（性能低但可靠）/ 主从同步复制
消费者：手动 ACK，消费成功再提交 offset，失败进重试队列
```

------

**Q6. 接口如何设计幂等性？**

参考答案要点：

- **Token 机制**：请求前获取 token，服务端校验并删除（Redis 原子操作）
- **唯一索引**：数据库唯一约束，重复插入直接报错
- **状态机**：订单状态只能向前流转，重复请求幂等
- **去重表**：记录已处理的请求 ID，重复请求直接返回上次结果
