# Claude

## 完整订单流程逻辑

### 一、创建订单 createOrder()

```
风控检查（账号/频率/黄牛）
    ↓
商品校验（是否下架/价格快照）
    ↓
Redis 预占库存
    ↓
计算价格（原价 + 活动价）
    ↓
写入订单 DB（状态: 待付款）
    ↓
发出 30min 超时延迟消息
```

------

### 二、支付阶段

#### 余额支付 balancePay()

```
校验订单状态（是否待付款/是否超时）
    ↓
校验余额是否充足
    ↓
校验优惠券是否有效
    ↓
扣优惠券（核销）
    ↓
扣余额（本地 DB 事务）
    ↓
订单状态（待付款 → 已付款）
    ↓
取消超时关单延迟消息
    ↓
发出"支付成功"消息通知下游
```

#### 第三方支付 thirdPartyPay()

```
校验订单状态（是否待付款/是否超时）
    ↓
校验优惠券是否有效
    ↓
扣优惠券（核销）
    ↓
生成支付订单，跳转支付宝/微信收银台
    ↓
【等待回调】
    ↓
支付回调通知 payCallback()
    ↓
验签（防篡改）
    ↓
幂等校验（防重复回调）
    ↓
订单状态（待付款 → 已付款）
    ↓
取消超时关单延迟消息
    ↓
发出"支付成功"消息通知下游
```

------

### 三、履约阶段 fulfillOrder()

```
消费"支付成功"消息
    ↓
订单路由（选仓/选商家）
    ↓
WMS 接单
    ↓
库存真实扣减（Redis 预占 → DB 扣减）
    ↓
拣货 → 打包 → 出库
    ↓
对接菜鸟物流
    ↓
订单状态（已付款 → 备货中 → 已发货）
```

------

### 四、收货阶段

```
物流轨迹持续更新
    ↓
用户主动签收 or 超时自动签收（15天）
    ↓
订单状态（已发货 → 已完成）
    ↓
触发商家结算（T+N 到账）
    ↓
进入售后保障期
```

------

### 五、超时关单 cancelOrder()

```
延迟消息触发（30min 未付款）
    ↓
校验订单状态（已付款则忽略，幂等）
    ↓
释放 Redis 预占库存
    ↓
释放优惠券（若已预占）
    ↓
订单状态 → 已关闭
```

------

### 六、售后阶段 refundOrder()

```
用户发起退款申请
    ↓
判断类型（仅退款 / 退货退款）
    ↓
商家审核
    ↓
        ├─ 余额支付 → 退回余额（本地 DB）
        └─ 第三方支付 → 调支付宝/微信退款接口
    ↓
库存回补
    ↓
优惠券视规则退还或作废
    ↓
订单状态 → 已退款/已关闭
```

------

### 核心设计原则总结

| 原则                   | 体现                                                   |
| ---------------------- | ------------------------------------------------------ |
| 库存预占与真实扣减分离 | 下单预占，履约才真实扣减                               |
| 优惠券支付时才核销     | 避免锁券浪费，关单无需回滚                             |
| 回调幂等               | 防止第三方重复回调导致重复付款                         |
| 消息驱动解耦           | 支付成功后发消息，履约异步消费                         |
| 延迟消息管理超时       | 多节点超时（关单/发货超时/自动签收）统一用延迟消息处理 |

## 优惠券应该在支付阶段核销

------

### 如果放在创建订单阶段，会有这些问题

**用户行为不确定** 用户下单后可能不付款，券被锁住期间无法用于其他订单，30min 后关单再释放，体验差且逻辑复杂

**用户可能在支付页换券** 下单时选了A券，支付时想换B券，创建阶段核销就无法支持这个场景

**多次下单浪费券** 用户反复下单不付款，券被反复锁定释放，增加系统压力

------

### 但支付阶段也需要一个"预校验"

```
创建订单时
    ↓
校验优惠券是否有效（是否过期/是否已用/是否满足门槛）
不锁定，只校验
    ↓
写入订单（记录用户选择的券 ID）

支付时
    ↓
再次校验券是否依然有效（防止下单到支付期间券过期）
    ↓
核销（真正扣减）
```

------

### 极端情况：两个订单同时用同一张券怎么办？

```
用户 A 下了两个订单，都选了同一张券
订单1 支付时 → 核销成功
订单2 支付时 → 校验发现券已核销 → 支付失败 → 提示用户换券
```

依赖数据库**唯一约束 + 乐观锁**保证券不会被重复核销，而不是靠预占来解决。

------

### 总结

| 阶段     | 优惠券动作       | 原因                 |
| -------- | ---------------- | -------------------- |
| 创建订单 | 只校验，不锁定   | 用户不一定付款       |
| 支付阶段 | 再次校验 + 核销  | 确保付款时券依然有效 |
| 关单     | 无需处理         | 从未锁定过           |
| 退款     | 视规则退还或作废 | 业务规则决定         |

## 当前流程还有几个值得完善的点

------

### 一、库存预占的释放时机不够严谨

当前流程是关单时释放预占库存，但还有几个场景没覆盖

```
支付失败（余额不足/第三方支付失败）→ 是否立即释放？
退款完成 → 库存回补的是预占还是真实库存？
履约异常（仓库无货）→ 需要释放并重新路由还是直接退款？
```

------

### 二、第三方支付回调异常处理缺失

```
回调网络超时 → 订单状态没更新，但钱已扣
用户重复点击支付 → 可能生成两笔支付单
支付成功但回调一直未到 → 需要主动轮询第三方支付状态兜底
```

当前流程依赖回调被动接收，缺少**主动对账**机制

------

### 三、余额支付和第三方支付的混合场景

```
余额不足时，部分用余额+部分用第三方支付（组合支付）
退款时需要按原路径分别退回
```

这个场景当前流程完全没有覆盖

------

### 四、履约失败的逆向流程不完整

```
WMS 接单后发现实际库存不足（预占和真实库存不一致）
    ↓
需要触发退款 → 但此时订单状态已是"备货中"
    ↓
逆向流程和正向售后流程是否共用同一套退款逻辑？
```

------

### 五、结算逻辑过于简单

```
当前只提到 T+N 到账
实际还涉及：
    - 平台佣金抽成
    - 多商家拆单（一个订单买了不同店铺商品）
    - 退款后结算金额的冲正
```

------

### 六、消息可靠性没有保障

```
"支付成功"消息发送失败 → 履约系统收不到 → 订单卡住
当前没有提到：
    - 消息发送失败重试
    - 本地消息表保证至少投递一次
    - 消费方幂等处理
```

------

### 优先级建议

| 优先级 | 问题               | 原因                    |
| ------ | ------------------ | ----------------------- |
| 🔴 高   | 第三方支付主动对账 | 直接涉及资损            |
| 🔴 高   | 消息可靠性         | 履约卡单影响用户体验    |
| 🟡 中   | 库存释放时机       | 库存不准会影响销售      |
| 🟡 中   | 履约失败逆向流程   | 影响退款体验            |
| 🟢 低   | 组合支付           | 复杂度高，非核心MVP功能 |
| 🟢 低   | 拆单结算           | 财务系统独立处理        |

整体来看核心链路是完整的，**资损类问题**（对账、消息可靠性）建议优先补充。

下面给出一个**整理后的完整订单逻辑流程**，只用**函数表示业务步骤**，不涉及具体代码实现，并且**支付流程包含：第三方支付 + 余额支付**。这个流程接近真实互联网电商系统。

## 一、整体订单生命周期

```text
用户下单
   │
placeOrder()
   │
createOrder()
   │
lockStock()
   │
confirmOrder()
   │
payOrder()
   │
handlePaymentCallback()
   │
shipOrder()
   │
completeOrder()
```

```text
未支付
   │
closeTimeoutOrder()
   │
cancelOrder()
```

## 二、下单流程（生成订单）

```text
placeOrder()
    ├── validateOrderRequest()
    ├── loadProductInfo()
    ├── checkProductStatus()
    ├── calculateOrderPrice()
    ├── applyCoupon()
    ├── generateOrderId()
    ├── createOrder()
    ├── lockStock()
    └── returnOrderInfo()
```

说明：

| 函数                 | 作用             |
| -------------------- | ---------------- |
| validateOrderRequest | 校验请求参数     |
| loadProductInfo      | 查询商品信息     |
| checkProductStatus   | 校验商品是否可售 |
| calculateOrderPrice  | 计算订单价格     |
| applyCoupon          | 应用优惠券       |
| generateOrderId      | 生成订单号       |
| createOrder          | 创建订单记录     |
| lockStock            | 锁定库存         |

订单状态：

```text
CREATED
```

## 三、确认订单流程

```text
confirmOrder()
    ├── checkOrderStatus()
    ├── verifyStockLock()
    ├── verifyCouponLock()
    ├── updateOrderToPendingPay()
    └── sendOrderCreatedEvent()
```

说明：

| 函数                    | 作用             |
| ----------------------- | ---------------- |
| checkOrderStatus        | 检查订单状态     |
| verifyStockLock         | 校验库存锁       |
| verifyCouponLock        | 校验优惠券锁     |
| updateOrderToPendingPay | 订单变为待支付   |
| sendOrderCreatedEvent   | 发送订单创建消息 |

```text
PENDING_PAYMENT
```

💡 **一句话总结**

一个完整电商订单系统逻辑：

```text
下单 → 创建订单 → 锁库存 → 确认订单 → 支付
       → (余额支付 / 第三方支付)
       → 支付回调 → 发货 → 完成订单
```

异常流程：

```text
取消订单
超时关闭
退款
```
