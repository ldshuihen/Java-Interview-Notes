# 高并发下服务稳定性问题分类

---

# 一、缓存稳定性问题（Redis 为主）

1. **缓存雪崩**
   大量 key 同时过期 / Redis 宕机 → 流量击穿 DB → 数据库打挂
2. **缓存击穿**
   热点 key 过期 → 高并发请求同时穿透到 DB
3. **缓存穿透**
   查询不存在的数据 → 缓存不命中 → 每次都查 DB
4. **大 Key 问题**
   超大 Value 导致网络阻塞、IO 阻塞、Redis 主线程卡住
5. **缓存污染**
   大量低频数据占满内存 → 逐出热点数据
6. **缓存并发竞争**
   高并发写缓存导致数据不一致

---

## 一、缓存雪崩（Cache Avlanche）

### 本质
大量 Key **同一时刻集体失效**，或 **Redis 整体宕机** → 瞬间巨量请求击穿缓存，直接砸向数据库，DB 被压垮。

### 典型成因
- 过期时间**没有加随机偏移**，例如全用 `EXPIRE key 3600`，导致整点/整小时集体过期。
- Redis 实例集群**宕机/不可用**，缓存层“消失”。
- 机房断电、网络割接等**底层故障**。

### 直观危害
- QPS 曲线呈**尖刺状**打穿 DB，服务级联超时、降级甚至宕机。
- 业务核心链路（下单、支付、详情）**全线不可用**。

### 落地解法（从治标到治本）
1. **过期时间加随机抖动**
   - 在原过期时间基础上加 ±10%~30% 随机偏移，避免集体过期。
   - 示例（伪代码）：
     ```java
     int expireSeconds = 3600;
     int randomOffset = new Random().nextInt(360); // ±10分钟
     redisTemplate.opsForValue().set(key, value, expireSeconds - randomOffset, TimeUnit.SECONDS);
     ```

2. **熔断 + 降级 + 本地缓存**
   - Redis 不可用时，前端/网关层**熔断**，直接返回兜底数据或降级逻辑。
   - 关键热点数据落地**本地缓存**（Caffeine/Guava），给 DB 加一层缓冲。

3. **Redis 高可用架构**
   - 主从 + 哨兵（Sentinel）或 Redis Cluster 集群，避免单点故障。
   - 结合**多 AZ 部署**，规避机房级灾难。

4. **数据库限流与保护**
   - 对 DB 做**读写分离 + 限流**（如 MySQL 慢查询日志、连接数限制）。
   - 必要时引入**服务端限流**（Sentinel/Resilience4j），削峰填谷。

---

## 二、缓存击穿（Cache Breakdown）
### 本质
**单个热点 Key** 在某一时刻**精确过期**，此时高并发请求涌入，全部直接查 DB，把 DB 打“穿”。

> 和雪崩区别：雪崩是**大量 Key 一起挂**，击穿是**一个热点 Key 挂了引发海啸**。

### 典型成因
- 热点商品/热点活动 Key，过期时间**没有续期**。
- 秒杀、直播弹幕等**瞬时高并发**打在单个 Key。

### 直观危害
- 热点接口 DB QPS 暴增，**连接池打满**，慢查询激增。
- 其他业务也被拖慢，出现**线程池耗尽、超时**。

### 落地解法
1. **热点 Key 永不过期 / 主动续期**
   - 核心热点数据设置为**永久有效**，由后台线程异步更新缓存。
   - 采用“**延时双删**”或“**Canal 订阅 Binlog**”机制保证数据一致性。

2. **互斥锁（Mutex Lock）——经典方案**
   - 只有一个线程去 DB 查并回写缓存，其他线程等待或返回旧值。
   - 伪代码逻辑：
     ```java
     String get(String key) {
         String value = redis.get(key);
         if (value == null) {
             // 加锁，保证只有一个线程去查DB
             String lockKey = "lock:" + key;
             try {
                 Boolean locked = redis.set(lockKey, "1", "NX", "EX", 3);
                 if (locked) {
                     value = db.query(key);
                     redis.set(key, value, expireSeconds);
                 } else {
                     // 重试或短暂休眠，等其他线程写入
                     Thread.sleep(10);
                     return get(key); // 递归重试
                 }
             } finally {
                 redis.del(lockKey);
             }
         }
         return value;
     }
     ```

3. **异步更新 + 缓存预热**
   - 提前预热热点数据到缓存。
   - 采用**定时任务**异步刷新缓存，避免过期瞬间击穿。

---

## 三、缓存穿透（Cache Penetration）
### 本质
查询**根本不存在的数据**（Key 既不在缓存，也不在 DB），请求每次都**穿透缓存直达 DB**。

### 典型成因
- 非法请求查询一个**不存在的 ID**，比如 `user_id = -1` 或随机乱 ID。
- 数据确实不存在，缓存没存，DB 也查不到。

### 直观危害
- 大量无效请求**消耗 DB 资源**，导致正常请求被拖慢。
- 被恶意利用，发起**大流量空查询攻击**，直接拖垮数据库。

### 落地解法
1. **布隆过滤器（Bloom Filter）——终极防御**
   - 提前将所有**存在的 Key** 哈希映射到布隆过滤器中。
   - 请求先过过滤器，不存在则直接拦截，不查 DB。
   - 适合**写多读少、Key 固定**的场景（如商品 ID 列表）。

2. **缓存空值 + 短期过期**
   - DB 查不到数据，缓存中写入一个**空值**（`null` 或特定占位符），并设置**短过期时间**（如 1 分钟）。
   - 避免短时间内重复查询同一无效 Key。
   - 注意：空值缓存会占用内存，需控制总量。

3. **接口层校验与限流**
   - 对参数做**合法性校验**（如 ID 范围、格式）。
   - 对非法 IP/请求频率做**限流**，拦截恶意攻击。

---

## 四、大 Key 问题（Big Key）
### 本质
某个 Key 的 **Value 体积过大**（如 String 超过 10MB，Hash/List 元素过多），导致**网络阻塞、IO 瓶颈、主线程卡住**。

### 典型成因
- 一次缓存过大的**结构化数据**，如完整的商品详情页 HTML、大 JSON。
- 未做**分库分表/分片**，将大量数据塞到一个 Key 里。
- 日志、消息队列等**未压缩数据**直接写入 Redis。

### 直观危害
- 单个 Key 操作耗时**远超正常阈值**（>10ms）。
- 阻塞 Redis 主线程，导致**整个实例延迟升高**，其他请求排队。
- 引发**网络带宽打满**，影响集群内其他节点。

### 落地解法
1. **拆分大 Key（最核心）**
   - 把大 Value 拆成多个小 Key，例如按 ID 分片：`user:info:1`、`user:info:2`。
   - 针对 List/Hash，使用**分页查询**，避免一次获取全部元素。

2. **数据压缩**
   - 对 String 类型数据进行**压缩**（如 GZIP），再存入 Redis。
   - 读取时解压，平衡网络传输与内存占用。

3. **避免存储无关数据**
   - 清理缓存中**不必要的字段**，只缓存核心热点数据。
   - 对于大对象，考虑直接落盘到数据库，仅缓存关键索引。

4. **监控与告警**
   - 通过 Redis 命令 `MEMORY USAGE key` 或监控工具（如 Prometheus + Grafana）监控大 Key。
   - 设置阈值告警，及时发现并处理。

---

## 五、缓存污染（Cache Eviction/Stale Data）
### 本质
缓存内存不足，触发**逐出策略**（如 `allkeys-lru`），导致**热点数据被频繁淘汰**，缓存命中率骤降；或**低频脏数据长期占用**。

### 典型成因
- 缓存**内存配置不足**，未根据业务增长扩容。
- 逐出策略配置不当，比如用了 `allkeys-lru` 但大量低频数据涌入。
- 未设置**合理过期时间**，导致垃圾数据长期堆积。

### 直观危害
- 缓存**命中率下降**，大量请求被迫回源查 DB。
- 性能退化，响应时间变长，甚至引发**缓存雪崩**的前兆。

### 落地解法
1. **合理配置内存与逐出策略**
   - 根据业务数据量预估，**扩容 Redis 内存**。
   - 针对热点数据，优先使用 `volatile-lru` 或 `volatile-ttl` 策略，只逐出带过期的 Key。

2. **设置合理过期时间**
   - 给所有 Key 设定**有意义的 TTL**，避免永久不过期的垃圾数据。
   - 对低频数据，适当缩短过期时间，主动清理。

3. **缓存分层**
   - 引入**多级缓存**：本地缓存（Caffeine）+ 分布式缓存（Redis）。
   - 本地缓存存储最热点数据，Redis 存储次热点数据，减少 Redis 内存压力。

4. **主动清理与数据治理**
   - 定期清理**无效/过期数据**，通过定时任务或 Binlog 订阅机制。
   - 监控缓存**命中率、内存使用率**等指标，动态调整策略。

---

## 六、缓存并发竞争（Cache Concurrency）
### 本质
**高并发写场景**下，多个线程同时对同一 Key 进行**读写操作**，导致**数据不一致**或**覆盖更新**。

### 典型成因
- 多线程同时更新同一缓存 Key，后写的覆盖先写的，导致**最终数据错误**。
- 读-改-写（Read-Modify-Write）操作**非原子性**，中间被其他线程打断。

### 直观危害
- 缓存数据与数据库数据**不一致**，导致业务逻辑错误。
- 出现**脏数据、幻读**等问题，影响核心业务（如订单、支付）。

### 落地解法
1. **原子操作（Redis 单线程特性）**
   - 尽量使用 Redis 原生**原子命令**，如 `SETNX`、`INCR`、`HSET` 等，避免多步操作。
   - 对于复杂逻辑，使用 **Lua 脚本** 打包操作，保证原子性。

2. **分布式锁**
   - 对关键业务逻辑加**分布式锁**（如 Redisson 锁），确保同一时间只有一个线程修改缓存。
   - 注意锁的**过期时间、可重入性**和**死锁处理**。

3. **写入策略优化**
   - 采用**延迟双删**：更新 DB 后，延迟一段时间再删除缓存，确保其他线程读取到旧值后能刷新。
   - 优先使用**Canal 订阅 Binlog** 异步更新缓存，减少业务代码耦合。

4. **读写分离与最终一致性**
   - 对于非强一致性要求的场景，可接受短暂数据不一致，通过异步机制保证最终一致。
   - 核心业务（如支付、订单）必须保证强一致性，采用上述锁或原子方案。

---

## 总结：六者关系与应对核心
| 问题         | 核心矛盾                  | 核心解法方向                   |
| :----------- | :------------------------ | :----------------------------- |
| **缓存雪崩** | 大量 Key 集体失效/宕机    | 加随机过期、高可用、熔断降级   |
| **缓存击穿** | 热点 Key 过期导致并发回源 | 永不过期/主动续期、互斥锁      |
| **缓存穿透** | 查询不存在的数据          | 布隆过滤器、缓存空值、参数校验 |
| **大 Key**   | 单个 Key 体积过大         | 拆分 Key、数据压缩、监控告警   |
| **缓存污染** | 内存不足，热点被逐出      | 扩容、合理逐出策略、缓存分层   |
| **并发竞争** | 高并发读写导致数据不一致  | 原子操作、分布式锁、延迟双删   |

**核心记住一句话**：缓存的本质是**平衡性能与一致性**，所有问题最终都要回归到**数据生命周期管理**和**高可用架构设计**上。

# 二、消息队列（MQ）稳定性问题
1. **消息堆积 / 消费阻塞**
   消费能力不足、消费线程卡死、慢SQL、第三方调用阻塞
2. **消息重复消费**
   重试机制导致重复处理，引发脏数据、超卖
3. **消息丢失**
   生产者/ broker/消费者 丢消息
4. **死信队列泛滥**
   异常消息无限重试，堵死正常队列
5. **MQ  broker 阻塞**
   磁盘满、主从同步延迟、队列分区阻塞
6. **顺序消息错乱**
   并发消费导致业务逻辑异常

---

## 1. 消息堆积 / 消费阻塞
### 本质
**消费速度 << 生产速度**，消息在 Broker 里越堆越多，最终导致队列阻塞、服务雪崩。

### 核心原因
- 消费线程数太少，消费能力不足
- 消费逻辑里有**慢 SQL、慢接口、第三方调用超时**
- 消费线程**卡死/抛异常**，单条消息卡住整个队列
- 死信消息无限重试，占用消费线程

### 危害
- 业务实时性失效（订单、通知延迟）
- 磁盘占满 → 新消息无法写入
- 消费延迟引发连锁超时、数据不一致

### 终极解决方案
1. **提高消费并发度**
   - 增加消费线程数（匹配队列分区/队列数）
   - 水平扩容消费实例（Kafka 分区数 = 最大并发消费数）
2. **异步化/批处理**
   - 非核心逻辑丢到线程池异步执行
   - 支持批量消费的 MQ 开启批量拉取
3. **慢操作隔离**
   - 禁止消费逻辑里执行慢 SQL、远程调用
   - 第三方接口必须加**超时控制 + 熔断**
4. **快速跳过异常消息**
   - 失败立即进入死信队列，不阻塞正常消费
5. **监控告警**
   - 监控堆积量、消费延迟、消费成功率

---

## 2. 消息重复消费
### 本质
MQ 为了**不丢消息**，默认开启**重试机制**，网络波动/异常都会重发 → 消费者收到多条相同消息。

### 典型场景
- 消费成功但未提交 ACK
- 消费者超时、重启、抛异常触发重试
- 生产者重试发送

### 危害
- 重复扣款、重复通知、**库存超卖**
- 脏数据、业务逻辑错乱

### 核心原则
**MQ 不保证只投递一次，消费者必须保证【幂等性】**

### 解决方案（通用且落地）
1. **数据库唯一索引**（最简单）
   - 用 `msgId`/`orderId` 建唯一键，重复插入直接报错
2. **Redis 分布式锁**
   - 消费前：`SETNX lock:msgId`
   - 加锁成功才执行业务
3. **状态机判断**
   - 订单状态：待支付 → 已支付，重复消息直接忽略
4. **全局唯一 ID + 消费记录表**
   - 记录已消费的 msgId，先查再处理

---

## 3. 消息丢失
### 本质
消息在 **生产者 → Broker → 消费者** 三个环节任意一环弄丢。

### 三大丢失场景 + 对应解决方案
1. **生产者丢消息**
   - 原因：网络闪断、Broker 未收到
   - 解决：**开启事务/确认机制（ACK）+ 重试**
2. **Broker 丢消息**
   - 原因：Broker 宕机、内存数据未刷盘
   - 解决：**持久化 + 主从同步 + 同步刷盘**
3. **消费者丢消息**
   - 原因：先提交 ACK，再执行业务，中途宕机
   - 解决：**先执行业务，再提交 ACK**

### 一句话保证不丢消息
**生产者确认 + Broker 持久化 + 消费者手动 ACK**

---

## 4. 死信队列泛滥
### 本质
无法被正常消费的**异常消息**，一直重试 → 占用队列、拖慢正常消费。

### 典型原因
- 数据格式错误、业务异常（如订单已取消）
- 依赖服务不可用，消息一直失败重试
- 重试次数无限制，无限循环

### 危害
- 正常消息被“卡死” behind 异常消息
- 消费线程被占满，队列雪崩

### 解决方案
1. **设置最大重试次数**
   - 重试 3~5 次失败 → 立即进入死信队列
2. **独立死信队列 + 监控告警**
   - 死信不影响主队列
   - 人工排查/脚本自动修复
3. **失败消息隔离**
   - 绝不允许失败消息阻塞队列

---

## 5. MQ Broker 阻塞
### 本质
Broker 本身不可用/性能暴跌 → 整个消息系统瘫痪。

### 核心原因
- 磁盘空间写满
- 内存不足、GC 卡顿
- 主从同步延迟、网络拥堵
- 队列分区/队列卡死
- 消息体过大、并发过高

### 危害
- 生产消息失败、服务超时
- 全线业务阻塞（订单、支付、通知）

### 解决方案
1. **磁盘监控 + 定期清理**
   - 过期消息自动删除
2. **Broker 集群高可用**
   - 主从 + 多节点部署，避免单点
3. **流量控制**
   - 大消息拆分、禁止超大消息
   - 生产端限流削峰
4. **同步复制/异步复制合理配置**
   - 核心链路用同步复制保证可靠性

---

## 6. 顺序消息错乱
### 本质
业务要求**消息必须按发送顺序消费**（如订单创建→支付→物流），但并发消费导致乱序。

### 原因
- 多线程并发消费
- 重试消息插队
- 多分区/多队列并行分发

### 危害
- 先支付再创建订单 → 业务报错
- 数据状态错乱

### 解决方案
1. **同一业务 ID 进入同一队列/分区**
   - 如按 `orderId` 哈希路由
2. **单线程消费**
   - 该队列只允许一个线程消费
3. **MQ 官方顺序消息机制**
   - Kafka：分区内有序
   - RocketMQ：严格顺序消息
4. **业务层面容忍最终有序**
   - 用状态机避免乱序影响

# 三、数据库稳定性问题
1. **数据库连接池耗尽**
   慢查询、事务过大、连接未释放
2. **慢 SQL / 索引失效**
   高并发下锁等待、IO 飙升、CPU 100%
3. **数据库雪崩**
   大量请求打库 → 行锁/表锁竞争 → 全部阻塞
4. **超卖/并发安全问题**
   库存、余额、优惠券并发扣减错误
5. **主从延迟**
   读从库读到旧数据，导致业务异常
6. **事务过大/长事务**
   锁持有时间长，引发大量阻塞

## 1. 数据库连接池耗尽
### 核心本质
业务获取数据库连接后**长期不释放**，连接池最大连接数被占满，新请求无法获取连接，接口全部阻塞、报错。

### 产生原因
1. 慢SQL执行时间过长，连接一直占用不归还；
2. **长事务/大事务**，事务迟迟不提交/回滚；
3. 代码异常未捕获，`Connection`、Session 未手动关闭；
4. 批量操作一次性抢占大量连接，无隔离限制；
5. 第三方接口阻塞嵌套在事务内，连接无限等待。

### 造成危害
- 服务大量请求卡死、超时、线程池积压；
- 全局服务不可用，引发上下游连锁雪崩；
- 数据库TCP连接数暴增，DB负载飙升。

### 解决方案
1. 合理配置连接池参数：最大连接数、空闲超时、获取连接超时；
2. 杜绝慢SQL，设置SQL执行超时时间；
3. 事务尽量短小，避免事务中嵌套远程调用、复杂业务；
4. 统一异常处理，保证连接、资源**finally强制释放**；
5. 监控连接池活跃连接数、等待队列，及时告警。

---

## 2. 慢 SQL / 索引失效
### 核心本质
SQL 未走索引、索引设计不合理、SQL写法糟糕，导致全表扫描、磁盘IO暴涨、CPU打满。

### 产生原因
1. 字段函数运算、隐式类型转换、左右模糊查询，导致**索引失效**；
2. 索引区分度低、联合索引违背最左匹配原则；
3. 大表无索引、数据量百万级以上全表扫描；
4. limit 超大分页、多表关联无合理关联条件；
5. 批量批量in数据过多、order by/group by 无索引排序。

### 造成危害
- 数据库CPU、IO 100%，整体实例性能下降；
- 行锁等待、锁竞争加剧，引发阻塞；
- 单条慢SQL拖垮整个库，影响所有业务；
- 接口响应极慢，请求堆积。

### 解决方案
1. 合理设计索引，遵循最左匹配、高频查询字段优先建索引；
2. 避免索引失效场景：禁止索引列函数、统一字段类型、避免%前缀模糊；
3. 开启慢查询日志，定时巡检、优化劣化SQL；
4. 大表分页优化、分批查询、避免大in；
5. 读写分离，复杂统计、报表查询走从库。

---

## 3. 数据库雪崩
### 核心本质
高并发流量直接直击数据库，大量写/读请求竞争锁、资源耗尽，SQL大量阻塞等待，数据库完全不可用。

### 产生原因
1. 缓存失效、缓存击穿/雪崩，**流量全部打透到DB**；
2. 热点商品、秒杀活动大流量无限流、无削峰；
3. 大量并发写操作，行锁升级表锁，锁竞争严重；
4. 无降级、无限流，弱业务和核心业务共用数据库；
5. 大批量批量更新、批量删除，引发锁范围扩大。

### 造成危害
- 数据库连接爆满、锁等待队列堆积；
- 所有业务接口超时、服务宕机；
- 数据库CPU、内存、磁盘IO满载，实例卡死。

### 解决方案
1. 强化多层缓存：本地缓存+Redis，阻挡大部分流量；
2. 网关/业务层**限流、削峰、排队**，秒杀使用队列削峰；
3. 读写分离，分担查询压力；
4. 大表拆分、分库分表，分散单库压力；
5. 业务分级降级，非核心业务流量直接降级兜底；
6. 优化锁粒度，缩小更新范围，避免表锁。

---

## 4. 超卖 / 并发安全问题
### 核心本质
多线程/多请求同时操作共享资源（库存、余额、优惠券），并发更新无互斥，覆盖更新导致数据错乱。

### 产生原因
1. 先查后改 非原子操作：`查库存→判断→扣减` 三步无锁；
2. 数据库普通隔离级别，无悲观/乐观控制；
3. 消息重复消费、接口重复提交加剧并发冲突；
4. 分布式多实例部署，单机锁失效。

### 造成危害
- 商品超卖、优惠券超发、余额负数；
- 资金、库存数据错乱，直接造成业务资损。

### 解决方案
1. **数据库乐观锁**：版本号version / 库存字段原子扣减
   ```sql
   update stock set num = num - 1 where id=? and num >= 1
   ```
2. 分布式悲观锁：Redisson分布式锁控制并发；
3. Redis+Lua 脚本，实现库存原子扣减，拦截大部分并发；
4. 接口防重、幂等设计，防止重复提交；
5. 高并发场景最终通过**分布式事务+锁**兜底。

---

## 5. 主从延迟
### 核心本质
主库写入数据后，Binlog同步到从库存在时间差，短时间内从库查询还是旧数据。

### 产生原因
1. 主库写入QPS过高，Binlog生成速度 > 从库回放速度；
2. 从库机器配置低、SQL单线程回放（MySQL5.7之前）；
3. 大事务、大批量写入，单条Binlog体积过大；
4. 网络带宽不足、主从网络波动；
5. 从库存在慢SQL，占用资源拖累回放。

### 造成危害
- 写主库、立刻查从库，查询不到最新数据；
- 业务逻辑异常、数据展示不一致；
- 严重延迟会导致数据一致性大幅偏差。

### 解决方案
1. 拆分大事务，避免大批量一次性写入；
2. 升级MySQL8.0，开启**并行复制**，提升从库回放能力；
3. 核心强一致场景：**写后读走主库**；
4. 优化从库配置、隔离从库慢查询；
5. 监控主从延迟阈值，超过阈值告警；
6. 中间件（Sharding-JDBC）实现动态路由。

---

## 6. 事务过大 / 长事务
### 核心本质
事务开启后，执行链路过长、耗时太久，迟迟不提交/回滚，长时间持有锁与MVCC快照。

### 产生原因
1. 事务中包含大量查询、循环、文件IO、远程调用；
2. 批量操作不拆分，一次性放入同一个事务；
3. 异常未捕获，事务卡死不结束；
4. 错误使用全局事务，无关业务强行绑定事务。

### 造成危害
1. 行锁长期不释放，大量并发请求锁等待、死锁；
2. Undo日志堆积，数据库磁盘暴涨、清理滞后；
3. MVCC快照过多，数据库性能下降；
4. 极易引发连接池耗尽、数据库阻塞。

### 解决方案
1. **事务最小化**：只把写库核心逻辑放进事务；
2. 禁止事务内调用第三方接口、MQ、文件操作；
3. 大事务拆分、分批提交；
4. 设置事务超时时间，自动回滚卡死事务；
5. 监控长事务，及时拦截告警。

# 四、JVM & 应用内存稳定性问题
1. **OOM 内存溢出**
   Java heap space、metaspace、直接内存、线程过多 OOM
2. **内存泄漏**
   静态集合、ThreadLocal、线程池任务、连接未释放
3. **频繁 Full GC / GC 长时间停顿**
   CPU 飙升、接口响应超时、服务假死
4. **线程池耗尽**
   任务堆积、线程死锁、阻塞队列满
5. **死锁 / 活锁 / 锁竞争严重**
   synchronized、ReentrantLock 竞争导致接口阻塞

---

## 1. OOM 内存溢出
### 核心本质
应用内存资源耗尽，堆内存、元空间、直接内存、线程资源等达到上限，JVM 无法分配新内存，直接抛出 OOM 崩溃。

### 分类&成因
1. **Java heap space**
   堆对象过多、大对象、内存泄漏、缓存无过期、集合无限扩容。
2. **Metaspace 元空间溢出**
   动态生成类过多、频繁热部署、反射/代理滥用、大量动态字节码。
3. **直接内存 OOM**
   Netty、NIO 未手动释放 DirectBuffer、堆外内存不回收。
4. **线程过多 OOM**
   无限制创建原生线程，线程栈内存耗尽。

### 危害
服务直接崩溃重启、业务中断、接口全部不可用。

### 解决方案
- 合理设置 JVM 堆、元空间、直接内存参数；
- 避免创建超大对象、批量分批处理数据；
- 堆外内存手动释放，Netty 资源规范关闭；
- 禁止无限制新建线程，统一使用线程池；
- 线上监控内存、堆快照排查，定位大对象/泄漏点。

---

## 2. 内存泄漏
### 核心本质
**无用对象无法被 GC 回收**，长期常驻内存，内存占用持续上涨，最终触发 OOM。

### 常见成因
1. 静态集合（static Map/List）长期存放数据，不主动清理；
2. **ThreadLocal 使用后不 remove**，线程复用导致强引用残留；
3. 线程池核心线程常驻，绑定业务对象无法回收；
4. 数据库、Redis、MQ 等连接资源未关闭，连接持有对象；
5. 匿名内部类、回调、长生命周期对象短对象强引用绑定。

### 危害
内存缓慢上涨、老年代塞满、FullGC 频繁、最终 OOM 宕机。

### 解决方案
- 静态容器手动定时清理过期数据；
- ThreadLocal 使用完必须 `remove()`，避免内存泄漏；
- 资源使用 try-with-resources 自动关闭；
- 合理设置线程池参数，避免任务强引用堆积；
- 定期 dump 堆内存，MAT 分析泄漏对象。

---

## 3. 频繁 Full GC / GC 长时间停顿
### 核心本质
老年代内存持续不足、堆内存不合理、内存泄漏，导致频繁触发 Full GC；GC 标记整理耗时过长，STW 停顿严重。

### 成因
1. 内存泄漏导致老年代持续被占满；
2. 大对象直接进入老年代，频繁触发扩容与回收；
3. 年轻代过小，MinorGC 频繁，晋升到老年代压力大；
4. 垃圾收集器选择不当，STW 时间过长；
5. 元空间、堆外内存占用过高，挤压堆空间。

### 危害
- 业务线程长时间 STW 暂停；
- CPU 飙高、接口 RT 暴涨、超时、服务假死；
- 集群整体吞吐量暴跌，引发上游雪崩。

### 解决方案
- 优化 JVM 参数：合理划分年轻代/老年代，调整晋升阈值；
- 改用低延迟收集器：G1、ZGC、Shenandoah；
- 杜绝内存泄漏、控制大对象产生；
- 优化代码：避免短生命周期大集合、对象频繁创建；
- 监控 GC 次数、停顿时间、GC 耗时告警。

---

## 4. 线程池耗尽
### 核心本质
核心线程+最大线程全部打满，阻塞队列塞满，新任务无法提交，触发拒绝策略，接口阻塞报错。

### 成因
1. 线程池参数配置不合理：核心线程、队列过小；
2. 任务执行阻塞：包含慢SQL、第三方超时、同步等待；
3. 任务无超时，线程长时间卡死不释放；
4. 无界队列滥用，任务无限堆积，内存打爆；
5. 业务手动新建线程，资源不可控。

### 危害
接口请求排队、大量超时、线程堆积、CPU 飙升、服务瘫痪。

### 解决方案
- 严格自定义线程池，禁止使用 Executors 原生工具类；
- 合理配置核心线程、最大线程、阻塞队列、拒绝策略；
- 所有异步任务增加超时时间、熔断降级；
- 耗时任务隔离线程池，核心业务与非核心业务池隔离；
- 监控线程池活跃线程数、队列积压、拒绝次数。

---

## 5. 死锁 / 活锁 / 锁竞争严重
### 核心本质
- **死锁**：多个线程互相持有对方需要的锁，互相等待，永久阻塞；
- **活锁**：线程反复重试、互相谦让，始终无法执行业务；
- **锁竞争**：高并发下大量线程争抢同一把锁，串行执行，吞吐量暴跌。

### 成因
1. 多把锁**获取顺序不一致**，造成循环等待；
2. synchronized、ReentrantLock 滥用，锁粒度过大；
3. 悲观锁大范围锁定，并发串行化；
4. 循环重试不加退让，引发活锁；
5. 分布式锁超时、续约异常，加剧竞争。

### 危害
线程卡死、接口无限阻塞、吞吐量急剧下降、服务卡顿、级联超时。

### 解决方案
- 统一多把锁的**加锁顺序**，破坏死锁必要条件；
- 缩小锁粒度，只锁定临界代码块；
- 优先使用乐观锁、CAS、无锁设计降低竞争；
- 活锁加入随机休眠、退让机制；
- 定时检测死锁、线上线程堆栈监控排查；
- 高并发场景用分布式锁+限流兜底。

# 五、网络通信稳定性问题
1. **TCP 连接阻塞 / 全连接队列溢出**
   服务端 accept 不过来，大量 SYN 超时
2. **连接池耗尽**
   HTTP、Dubbo、Feign、OkHttp 连接不够用
3. **大量 TIME_WAIT / CLOSE_WAIT**
   文件描述符耗尽，无法新建连接
4. **超时风暴 / 重试风暴**
   超时+重试放大流量，压垮上下游
5. **网络丢包、延迟高**
   跨运营商、机房抖动、带宽打满

## 1. TCP 连接阻塞 / 全连接队列溢出
### 核心本质
TCP 三次握手完成后，**全连接队列(accept队列)** 满，服务端消费连接太慢，新连接无法被 `accept` 处理，导致连接建立失败、SYN 超时。

### 产生原因
- 服务端业务处理慢、消费连接能力弱，accept 不及时；
- 全连接队列、半连接队列内核参数配置过小；
- 瞬间高并发建连，TCP 队列承载不足；
- 服务线程阻塞、卡顿，无法及时取走连接。

### 危害
- 客户端连接超时、握手失败、接口报错；
- 大量 SYN 重传，网络拥堵；
- 新流量无法接入，服务对外不可用。

### 解决方案
- 调整内核参数：`somaxconn`（全连接队列）、`tcp_syncookies`；
- 优化服务端处理速度，避免IO阻塞、慢逻辑；
- 增加服务实例扩容，分摊建连压力；
- 接口层限流，防止突发海量连接冲击。

---

## 2. 连接池耗尽（HTTP/Dubbo/Feign/OkHttp）
### 核心本质
短连接滥用或连接池参数不合理，**长连接数量打满**，新请求拿不到连接，全程阻塞、超时。

### 产生原因
- 连接池最大连接数设置过小；
- 调用下游接口无超时、线程卡死占用连接不释放；
- 同步串行调用多依赖，连接占用时间过长；
- 未复用长连接，频繁创建销毁短连接；
- Dubbo/Feign 底层连接未合理回收。

### 危害
- 连锁调用阻塞，服务雪崩；
- 大量请求排队、线程积压、CPU 飙升；
- 下游依赖不可用直接拖垮当前服务。

### 解决方案
- 合理配置各类客户端连接池：最大连接、空闲超时、获取超时；
- 所有远程调用**强制加超时、熔断、降级**；
- 核心/非核心依赖**线程池隔离、连接池隔离**；
- 开启长连接复用，减少频繁建连断连；
- 监控连接池活跃数、等待队列、拒绝数。

---

## 3. 大量 TIME_WAIT / CLOSE_WAIT
### 核心本质
- **TIME_WAIT**：主动关闭方需等待2MSL，短连接大量频繁关闭，导致套接字资源占满；
- **CLOSE_WAIT**：被动关闭，服务端收到FIN后**未主动close**，连接僵死挂起。

### 产生原因
1. TIME_WAIT
   - 业务大量短连接，请求完立即断开；
   - 接口调用频繁，不断新建/关闭TCP连接；
2. CLOSE_WAIT
   - 代码未手动关闭流、连接、Response；
   - 对方提前断开，我方业务卡死不执行关闭逻辑；
   - 连接资源未使用 try-with-resources 自动回收。

### 危害
- 系统**文件描述符FD耗尽**，无法新建TCP连接；
- 端口资源耗尽，服务无法对外通信；
- 接口大面积连接异常、服务瘫痪。

### 解决方案
- 业务改用**长连接复用**，减少短连接；
- 优化内核：开启 `tcp_tw_reuse`、`tcp_tw_recycle`；
- 代码规范：所有网络连接、IO流自动关闭；
- 定时清理僵死连接，设置连接空闲销毁时间；
- 监控FD数量、TCP状态分布告警。

---

## 4. 超时风暴 / 重试风暴
### 核心本质
下游服务卡顿超时 → 上游统一重试、批量重试 → 流量瞬间翻倍、放大，层层传导压垮整条调用链。

### 产生原因
- 未做重试限制，超时、异常默认无限重试；
- 无退避策略，失败立即暴力重试；
- 集群所有实例同时重试，流量叠加；
- 缺少熔断、降级，故障无法快速隔离。

### 危害
- 单节点故障扩散为**全链路雪崩**；
- 下游压力成倍暴增，彻底打挂依赖服务；
- 响应时间持续恶化，全线业务不可用。

### 解决方案
- 禁止无条件重试，限制**最大重试次数**；
- 加入**指数退避+随机休眠**，避免同时重试；
- 引入熔断组件（Sentinel/Resilience4j），故障节点快速熔断；
- 超时、异常场景快速降级，拒绝无效流量；
- 网关层限流、削峰，控制入口流量。

---

## 5. 网络丢包、延迟高
### 核心本质
物理网络、机房链路、带宽瓶颈导致数据包丢失、传输耗时暴涨，跨机房/跨区域调用体验极差。

### 产生原因
- 跨机房、跨运营商、跨地域远程调用；
- 交换机/路由器硬件故障、链路抖动；
- 网卡带宽打满、流量挤占、大报文传输；
- 防火墙、安全策略、路由配置异常；
- 容器环境网络隔离、NAT转发损耗。

### 危害
- 接口RT飙升、随机超时、报文重传；
- 分布式事务、MQ、RPC通信不稳定；
- 数据传输不完整、业务偶发异常。

### 解决方案
- 核心服务**同机房部署**，就近调用，减少跨机房；
- 带宽扩容、大报文拆分、数据压缩传输；
- 优化路由、隔离业务网络与运维网络；
- 关键RPC/MQ开启重传机制+超时控制；
- 监控网络延迟、丢包率、带宽使用率。

---

# 六、网关 & 中间件稳定性问题
1. **Gateway/Nginx 阻塞**
   worker 线程打满、backlog 满、upstream 超时
2. **注册中心（Nacos/Eureka）异常**
   服务发现失败、调用错误节点
3. **配置中心推送延迟**
   限流、降级规则不生效
4. **分布式锁死锁/阻塞**
   Redisson 锁竞争、超时、未释放

---

## 1. Gateway / Nginx 阻塞
### 本质
网关层接入能力耗尽，请求排队、转发阻塞，成为全局流量瓶颈，导致所有下游接口超时。

### 核心成因
1. Nginx worker 进程数不足、Gateway 业务线程池打满；
2. upstream 下游服务慢、超时、不可用，连接挂住不释放；
3. TCP 全连接队列 backlog 过小，连接排队溢出；
4. 缺少超时、重试、熔断，异常下游拖垮网关；
5. 大请求、大文件、长连接占用网关线程。

### 危害
- 全局接口响应变慢、大面积超时；
- 网关层请求堆积，流量入口瘫痪；
- 单点网关故障引发全站不可用。

### 解决方案
1. 合理配置 Nginx worker 数、连接数、backlog 队列；
2. Gateway 自定义线程池，隔离读写、快慢接口；
3. 给下游路由统一配置**连接超时、读取超时**；
4. 网关层熔断、降级、重试、负载均衡剔除故障节点；
5. 网关集群化部署，避免单点；
6. 限制单IPQPS、请求体大小，防恶意大流量攻击。

---

## 2. 注册中心（Nacos / Eureka）异常
### 本质
服务注册、心跳、服务拉取异常，导致**服务列表不准**，调用到下线/故障节点。

### 核心成因
1. 心跳超时、网络抖动，实例被错误剔除；
2. 注册中心集群单点故障、同步延迟；
3. 客户端缓存服务列表刷新不及时；
4. 健康检查机制不合理，误判健康状态；
5. 大批量服务上下线，推送风暴。

### 危害
- 远程调用报错、连接失败、随机接口异常；
- 负载均衡路由到无效节点，报错率飙升；
- 微服务集群调用链路不稳定。

### 解决方案
1. 注册中心集群高可用部署，避免单点；
2. 合理调整**心跳间隔、健康检查、剔除时间**；
3. 客户端本地缓存服务列表，降级容错；
4. 开启快速失败+重试，结合负载均衡故障剔除；
5. 限制批量上下线，避免推送风暴。

---

## 3. 配置中心推送延迟
### 本质
限流、降级、开关、参数配置推送不及时，规则未实时生效，故障无法快速止血。

### 核心成因
1. 配置长轮询拉取延迟、网络波动；
2. 客户端缓存配置、未及时监听刷新；
3. 配置中心节点同步延迟；
4. 未主动触发推送，依赖定时拉取；
5. 大配置、大批量配置推送阻塞。

### 危害
- 线上突发问题：降级、限流、开关不能立即生效；
- 故障止血滞后，放大故障影响范围；
- 多环境配置不一致，业务逻辑异常。

### 解决方案
1. 监听配置变更事件，**实时动态刷新**，无需重启；
2. 缩短长轮询间隔，保证配置秒级推送；
3. 配置中心集群部署，提高推送稳定性；
4. 核心限流/降级配置独立拆分，轻量化推送；
5. 本地配置兜底，配置中心不可用时降级。

---

## 4. 分布式锁死锁 / 阻塞
### 本质
基于 Redis 的分布式锁，出现**锁无法释放、过期异常、锁竞争激烈**，导致业务串行阻塞、死锁。

### 核心成因
1. 加锁后服务宕机、异常崩溃，**锁未主动释放**；
2. 锁过期时间设置过短，业务没执行完锁自动过期；
3. 无锁续期（看门狗），长任务提前释放锁；
4. 未保证原子性：加锁、解锁逻辑非原子；
5. 大量并发争抢同一把锁，严重锁竞争。

### 危害
- 接口串行执行、吞吐量暴跌、大量阻塞等待；
- 锁永久残留形成**分布式死锁**，业务永久卡死；
- 并发控制失效，引发超卖、数据不一致。

### 解决方案
1. 使用 Redisson 自带**可重入锁 + 看门狗自动续期**；
2. 锁必须设置过期时间，防止宕机死锁；
3. 保证解锁原子性：Lua 脚本比对 value 再删除；
4. 缩小锁粒度，只锁住临界资源；
5. 高并发场景分段锁、本地限流减少锁竞争；
6. 加锁等待超时，避免无限阻塞。

---

# 七、高并发流量类稳定性问题
1. **流量洪峰 / 秒杀压垮服务**
2. **接口限流未做导致被打满**
3. **级联故障（服务雪崩）**
   一个服务挂 → 整条调用链挂
4. **服务熔断未开启导致全盘阻塞**
5. **线程池、队列无界导致 OOM**

---

## 1. 流量洪峰 / 秒杀压垮服务
### 本质
瞬时超大流量（秒杀、活动、热点事件）远超系统设计承载上限，各层组件资源快速耗尽，服务直接瘫痪。
### 成因
- 未做流量预估、无压测，系统容量不足；
- 突发海量请求直达应用、数据库，无层层拦截；
- 秒杀热点商品、热点接口无特殊防护；
- 无削峰、无排队机制，瞬时并发打爆链路。
### 危害
- CPU、内存、连接池、MQ、数据库全部打满；
- 接口大面积超时、报错、服务不可用；
- 活动崩盘、资损、用户体验雪崩。
### 解决方案
- **多层削峰**：网关限流 + 队列削峰 + 令牌桶排队；
- **热点隔离**：秒杀独立集群、独立库表、隔离普通业务；
- **流量拦截**：页面静态化、CDN、本地缓存、Redis 拦截大部分请求；
- **异步化**：下单、扣库存、消息通知全部 MQ 异步解耦；
- **容量规划**：压测摸底、弹性扩容、定时扩容预案。

---

## 2. 接口限流未做导致被打满
### 本质
无QPS限制、无访问控制，外部/内部无限请求涌入，线程、连接、资源被恶意/异常流量占满。
### 成因
- 网关、应用、接口均未配置限流规则；
- 对内服务信任过度，未做内部调用限流；
- 爬虫、恶意刷接口、重复提交无拦截；
- 突发业务流量无管控，资源无保护。
### 危害
- 核心接口被非核心流量挤占；
- 线程池、连接池快速耗尽；
- 正常用户请求被拖垮，全局响应变慢。
### 解决方案
- **全链路限流**：网关全局限流 + 接口单机限流 + 分布式限流；
- 限流算法：令牌桶/漏桶，控制匀速流量；
- 接口分级：核心接口高阈值、非核心严格限制；
- 防刷控制：IP限流、用户维度限流、验证码防刷；
- 统一兜底：超限直接返回降级提示，不阻塞。

---

## 3. 级联故障 / 服务雪崩
### 本质
下游单个服务故障、卡顿、超时，上游未做防护，大量请求堆积、线程卡死，逐级向上传导，整条调用链全部瘫痪。
### 成因
- 微服务强依赖，同步调用链路过长；
- 下游异常无隔离、无快速失败；
- 无限重试、无超时，放大故障；
- 服务间没有强弱依赖划分。
### 危害
- 单点故障演变为全站故障；
- 所有依赖方线程积压、连接耗尽；
- 分布式链路全部阻塞，系统整体宕机。
### 解决方案
- **强弱依赖划分**：非核心依赖故障直接降级；
- 超时控制：所有RPC、HTTP调用强制超时；
- 集群隔离：业务拆分、机房隔离、集群隔离；
- 异步解耦：非强一致场景改用MQ异步调用；
- 故障演练：常态化压测、混沌工程，提前暴露薄弱点。

---

## 4. 服务熔断未开启导致全盘阻塞
### 本质
下游服务持续异常、超时、报错，上游未熔断，仍持续大量调用，导致自身线程/连接一直阻塞等待。
### 成因
- 未接入熔断组件（Sentinel/Resilience4j/Hystrix）；
- 异常阈值、失败率未配置，故障无法识别；
- 调用方无脑重试，持续打故障服务；
- 依赖服务长时间不可用，调用不终止。
### 危害
- 故障服务持续拖垮所有调用方；
- 阻塞线程不断累积，服务假死；
- 小问题演变为大规模服务不可用。
### 解决方案
- 接入熔断框架，基于**失败率、超时率、异常数**自动熔断；
- 熔断→降级→恢复半开，自动恢复机制；
- 故障节点负载均衡自动剔除；
- 熔断期间返回兜底数据、默认结果；
- 核心服务优先保障，非核心快速熔断。

---

## 5. 线程池、队列无界导致 OOM
### 本质
使用**无界队列、无限创建线程**，高并发下任务无限堆积，对象持续膨胀，最终堆内存溢出。
### 成因
- 错误使用 `Executors` 快捷创建线程池（无界队列/无限线程）；
- 自定义线程池队列设置为无界 `LinkedBlockingQueue`；
- 异步任务、定时任务无限提交，无拒绝策略；
- 消费速度远小于生产速度，任务持续堆积。
### 危害
- 任务对象、上下文、大对象常驻内存；
- 老年代快速占满，频繁FullGC，最终OOM崩溃；
- 服务卡死、无法自动恢复。
### 解决方案
- **强制使用有界队列 + 自定义线程池**；
- 合理配置：核心线程、最大线程、队列容量、拒绝策略；
- 拒绝策略兜底：队列满时限流、丢弃、降级；
- 快慢任务线程池隔离，防止互相挤压；
- 监控线程池队列积压、拒绝次数、内存占用。

---

# 八、分布式事务 & 数据一致性问题
1. **分布式事务悬挂/超时**
2. **TCC 悬挂、空回滚、幂等失败**
3. **最终一致性不一致**
   订单/支付/库存不一致

---

## 1. 分布式事务悬挂 / 事务超时
### 核心本质
分布式事务（Seata AT/XA）执行过程中**长时间未提交/未回滚**，事务处于悬挂状态，资源不释放；或全局事务超时强制结束，数据状态割裂。

### 成因
1. 微服务调用链路过长，某节点阻塞、超时、宕机；
2. 分支事务执行卡住、第三方接口阻塞、慢SQL；
3. Seata TC 通信异常、网络抖动，全局事务无法决议；
4. 全局事务超时时间设置不合理，业务未跑完已过期；
5. 事务未正常结束，分支锁、行锁长期持有。

### 危害
- 全局事务悬挂不结束，**行锁、全局锁持续占用**；
- 引发大量锁等待、接口阻塞、死锁；
- 回滚/提交错乱，出现中间态脏数据；
- 数据库连接池积压，性能急剧下降。

### 解决方案
1. 合理配置**全局事务超时时间**，自动触发超时回滚；
2. 所有远程调用加超时、熔断，避免分支事务卡死；
3. 保证 TC 高可用集群部署，防止单点故障；
4. 增加定时补偿任务，扫描悬挂事务强制闭环；
5. 缩短业务事务周期，避免长分布式事务。

---

## 2. TCC 三大经典问题：悬挂、空回滚、幂等失败
### （1）TCC 悬挂
#### 本质
**Try 未执行成功**，但远端收到 Cancel 请求，空 Cancel 先执行；后续 Try 又正常执行，数据无法回滚，导致数据悬挂不一致。
#### 原因
网络超时、重试、调用延迟，Cancel 先于 Try 到达。
#### 解决
- 业务事务状态表标记；
- Cancel 阶段先判断是否有 Try 记录，无则直接忽略。

---

### （2）TCC 空回滚
#### 本质
分支事务**Try 完全没执行**，却收到 Cancel 回滚请求，错误执行无效回滚逻辑，引发异常。
#### 原因
半阶段网络异常、上游取消事务，分支未做任何操作却被要求回滚。
#### 解决
- 新增**事务控制表**，记录 Try 执行状态；
- 无 Try 记录时，Cancel 直接放行，不执行业务回滚。

---

### （3）TCC 幂等失败
#### 本质
重试机制导致 Try / Confirm / Cancel 重复执行，出现重复扣款、重复扣库存、重复生成单据。
#### 原因
MQ 重试、RPC 重试、网络重传，TCC 三阶段无防重。
#### 解决
- 全局事务ID 作为唯一键，做**幂等控制**；
- 状态机判断：已完成阶段禁止重复执行；
- 操作前加分布式锁+唯一索引兜底。

---

## 3. 最终一致性不一致（订单/支付/库存）
### 核心本质
本地事务无法跨多服务、多库生效，订单、支付、库存、优惠券、物流**多节点数据更新不同步**，部分成功、部分失败，最终数据割裂。

### 成因
1. 微服务拆分后，跨库跨服务无原子事务；
2. 异步消息丢失、消费失败，补偿逻辑未执行；
3. 定时补偿任务漏扫、执行失败；
4. 缓存、DB、MQ 多组件状态不同步；
5. 业务异常未捕获，流程中断导致半截事务。

### 典型业务乱象
- 订单已取消，库存未返还；
- 支付成功，订单未更新为已支付；
- 扣了库存，订单创建失败；
- 优惠券核销成功，订单关闭未退回。

### 解决方案
1. **可靠消息最终一致性（主流）**
   半消息+本地事务+消息投递+消费回执，保证消息可靠收发。
2. **TCC 柔性事务**
   手动补偿，Try/Confirm/Cancel 三阶段手动控制正负业务。
3. **Seata AT 事务**
   无侵入分布式事务，适合短链路、低并发场景。
4. **定时对账+补偿任务**
   定时扫描异常数据、差异数据，自动修复兜底。
5. **状态机驱动业务**
   订单/支付全链路状态流转，异常自动回滚或重试。

# 九、安全与风控类
1. **恶意刷接口、爬虫、CC攻击**
2. **重复提交导致超卖、重复发放奖品**

## 1. 恶意刷接口、爬虫、CC攻击
### 本质
恶意客户端/脚本高频、伪造、批量请求，无节制冲刷业务接口，挤占正常用户资源，拖垮应用与数据库，属于**业务层流量攻击**。

### 成因

- 接口无防刷、无频次限制、无身份校验；
- 爬虫批量爬取数据，高频遍历接口；
- CC攻击伪造大量正常请求，消耗连接、线程、带宽；
- 对外接口暴露过全，参数无校验、无拦截。

### 危害
- 服务器CPU、带宽、连接池被打满；
- 数据库查询激增，引发慢SQL、库压力雪崩；
- 正常用户请求延迟、超时、服务瘫痪；
- 敏感数据泄露、活动福利被恶意薅羊毛。

### 解决方案
1. **多层限流防刷**
   网关层 + 应用层，按**IP、用户ID、设备号**维度限流；
2. **人机识别**
   验证码、滑块、行为校验、接口签名、Token鉴权；
3. **爬虫防护**
   User-Agent校验、高频封禁、动态密钥、数据加密返回；
4. **CC攻击防护**
   黑名单封禁、异常流量清洗、网关WAF、恶意IP拉黑；
5. **接口加固**
   入参校验、请求体限制、非法参数拦截、内网接口隔离；
6. **监控告警**
   异常QPS、短时间重复请求、高频错误实时告警。

---

## 2. 重复提交导致超卖、重复发奖/重复业务
### 本质
同一操作短时间多次提交，无防重机制，并发+重复请求叠加，引发**重复业务执行**，破坏数据一致性。

### 成因
- 前端未做按钮置灰、防抖；
- 后端无全局防重、无唯一幂等标识；
- 网络卡顿、用户频繁点击、页面回退重提；
- MQ重复消费、RPC重试、接口重试导致重复执行；
- 并发扣减业务无锁、无原子控制。

### 危害
- 商品超卖、优惠券/积分/奖品重复发放；
- 重复下单、重复扣款、重复生成订单；
- 数据错乱、资损、业务纠纷。

### 解决方案
1. **前端防抖**
   按钮禁用、定时冷却、禁止短时间连续提交；
2. **后端全局防重**
   唯一流水号/请求ID + Redis 分布式防重锁（SETNX 短过期）；
3. **数据库幂等**
   唯一索引、业务状态机判断（已完成直接拦截）；
4. **并发控制**
   库存扣减使用**SQL原子更新 / Redis+Lua**；
5. **全局幂等设计**
   所有写接口、领奖、下单、支付统一接入幂等组件；
6. **限制提交频率**
   单用户核心操作短时间内限制次数。
