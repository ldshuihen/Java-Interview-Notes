# 一、支付系统核心业务流程（必须先说）

面试第一步先讲 **完整支付链路**。

### 标准支付流程

```
用户下单
   ↓
订单系统创建订单（状态：待支付）
   ↓
用户点击支付
   ↓
支付系统创建支付单
   ↓
调用第三方支付（支付宝/微信）
   ↓
用户完成支付
   ↓
第三方回调支付结果
   ↓
支付系统更新支付单
   ↓
通知订单系统修改订单状态
   ↓
订单完成
```

核心系统：

```
用户
  │
  ▼
订单系统
  │
  ▼
支付系统
  │
  ├── 支付渠道（微信/支付宝）
  │
  ├── 账户系统
  │
  ├── 对账系统
  │
  └── 风控系统
```

# 四、如何保证支付不重复（幂等）

支付系统 **最重要的问题之一：幂等性**

例如：

```
用户重复点击支付
支付回调重复
网络重试
```

解决方案：

### 方案1：数据库唯一约束

```
unique(order_id)
```

保证一个订单只有一个支付单。

------

### 方案2：幂等Token

用户支付：

```
生成 requestId
```

表：

```
request_id
order_id
```

重复请求直接返回。

------

### 方案3：状态机幂等

```
只有 PAYING 状态才能更新 SUCCESS
```

SQL：

```
update payment
set status = SUCCESS
where payment_id = ?
and status = PAYING
```

更新行数=1 才成功。

# 五、退款幂等问题（重点）

退款非常容易出现 **重复退款**。

例如：

```
用户多次点击退款
系统重试
回调重复
```

解决方案：

### 1 退款单唯一约束

```
unique(order_id, refund_request_id)
```

------

### 2 状态机控制

只允许：

```
INIT → REFUNDING → SUCCESS
```

SQL：

```sql
update refund
set status='SUCCESS'
where refund_id=?
and status='REFUNDING'
```

------

### 3 金额校验

必须保证：

```
已退款金额 ≤ 已支付金额
```

例如：

```
支付100
已退款80
最大只能再退20
```
