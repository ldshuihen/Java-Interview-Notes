# 实战+线上问题排查

## 总体架构

![image-20260423173043114](实战+线上问题排查-Claude.assets/image-20260423173043114.png)实战

以下是七大分类的要点说明，点击图中任意卡片可以展开深入探讨。

**一、JVM 问题** 是最基础的排查能力，核心工具链是 `jmap`（生成 heap dump）→ MAT 分析、`jstack` 定位死锁/热点线程、`jstat -gcutil` 观察 GC 频率。

**二、性能问题** 通常从现象（接口慢、TPS 低）反推瓶颈，黄金路径是：接口 RT → 链路追踪找慢节点 → 慢 SQL / 锁竞争 / 外部依赖逐一排除。

**三、并发问题** 难在复现，需要借助 `jstack` 抓 thread dump、`async-profiler` 做 CPU 火焰图，并结合代码走查。

**四、数据库/存储** 重点关注慢查询（`explain` + 索引）、锁等待（`information_schema.innodb_trx`）和主从延迟（`show slave status`）。

**五、分布式/微服务** 排查要有全局视角，核心是链路追踪（TraceId 串联）+ 服务调用拓扑，配合熔断降级策略兜底。

**六、日志/监控** 是所有排查的眼睛，建议做到：结构化日志 → ELK 检索 → Prometheus 指标 → Grafana 大盘 → 告警通知的完整闭环。

**七、安全/发布** 在变更管控上，灰度 + 快速回滚是线上稳定性的最后防线，K8s 探针配置不当是常见的部署踩坑点。

> **排查万能公式**：现象描述 → 指标定量 → 工具采集 → 根因分析 → 修复验证 → 故障复盘

**排查总原则：**

遇到 JVM 线上问题，第一步永远是"先保留现场"——在重启之前抓 heap dump 和 thread dump，否则证据全消失。推荐的预防性 JVM 启动参数组合：

```
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/oom-$(date +%Y%m%d%H%M%S).hprof
-Xlog:gc*:file=/tmp/gc.log:time,uptime:filecount=5,filesize=20m
-XX:+UseG1GC -XX:MaxGCPauseMillis=200
```

这四行参数能让你在出问题时有据可查，是每个 Java 服务的标配。

**性能排查的核心心法：**

性能问题 99% 都可以用一句话概括——"请求在哪里等了最久"。排查时遵循这个优先级：

1. 先看监控时间轴，把问题发生时间圈出来

## **三、并发问题**

并发问题最难在于**难复现、难感知**——数据在极端时序下才出错，单线程测试永远正常。先用一张图建立整体认知，再逐专题深入。

![image-20260423174722668](实战+线上问题排查-Claude.assets/image-20260423174722668.png)

并发问题的根源只有三个：**可见性**（一个线程的修改另一个线程看不到）、**原子性**（复合操作被打断）、**有序性**（指令重排导致执行顺序与预期不符）。下面六个专题从这三个维度展开：



并发问题的难点在于"偶发、难复现"。先用一张全景图建立知识框架，再深入每个专题。

![image-20260424095714179](实战+线上问题排查-Claude.assets/image-20260424095714179.png)

并发问题从根上分三类：可见性、原子性、有序性。所有具体的 bug（死锁、数据竞争、ThreadLocal 泄漏）都能归入这三类。下面是完整的排查手册：---

**并发问题的核心心法：**

并发 bug 之所以难，在于它们"概率出现"，本地压测没问题，线上高并发才暴露。排查时记住这个优先级：

1. 先问"这是不是可见性问题"——两个线程读写同一变量却没加 `volatile`/锁，几乎必然出问题
2. 再问"操作是不是原子的"——`i++`、`check-then-act`、`if-absent-then-put` 全都不是原子的
3. 最后问"有没有锁依赖顺序"——两把锁不按固定顺序加，迟早死锁

工具链选择上，**死锁用 `jstack`**，**锁竞争热点用 `async-profiler -e lock`**，**在线方法级排查用 Arthas `trace`**。分布式场景则首选 Redisson，自研锁必须用 Lua 脚本保证原子解锁，并且所有关键业务必须做幂等兜底。

架构分为四层：客户端 → 网关 → 服务集群 → 基础设施（MQ/注册中心/配置中心），链路追踪横贯全局。点击架构图中任意节点可展开该专题，六大排查手册如下：---

**分布式/微服务问题的核心心法：**

微服务最难的不是单个问题，而是"问题跨边界传播"。排查时遵循三步：

1. 先用 TraceId 串联全链路日志，把"哪个服务、哪个操作、耗时多久"的瀑布图画出来
2. 找到最先出现异常的那个 Span，那就是根源服务，其他超时都是连锁反应
3. 根因确定后，按隔离 → 熔断 → 降级的顺序恢复，最后再修复根本问题

工具优先级：线上排查优先看 SkyWalking 链路 + Sentinel 控制台 + 注册中心实例状态，这三个面板能覆盖 90% 的分布式问题。想深入某个专题（比如 Kafka 积压的完整处理流程、Nacos 集群部署、雪崩演练），点击面板内的按钮或直接告诉我。

## **六、日志/监控**

日志/监控是整个可观测性体系的眼睛。先看完整的可观测性三支柱架构，再深入每个专题。

![image-20260424171857012](实战+线上问题排查-Claude.assets/image-20260424171857012.png)

可观测性的核心是三支柱联动：指标发现异常 → 日志定位细节 → 链路还原现场。六大专题完整手册如下：---

**日志/监控的核心心法：**

可观测性的价值在于"问题发生前就知道"，而不是"出问题了才去找"。建立完整闭环的关键是：

**指标 → 告警 → 日志 → 链路**，四步联动缺一不可：指标发现异常触发告警，日志定位具体错误行，链路还原完整调用现场。

几个建设优先级建议：

1. 先把结构化日志做好（JSON + TraceId），这是一切的基础，ELK 才能发挥价值
2. 接入 Micrometer + Prometheus，监控黄金四指标（延迟/流量/错误率/饱和度）
3. 告警从少到多，先保证 P0/P1 告警 100% 准确，再逐步补充 P2/P3
4. Arthas 是线上救火的瑞士军刀，提前熟悉 `trace`、`watch`、`thread -b` 三个命令，能解决 80% 的线上偶现问题
