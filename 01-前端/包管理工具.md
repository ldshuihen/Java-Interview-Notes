## 一、前端 / Node.js 生态

### npm run dev/bulid/serve

#### 1. `npm run serve` 是干什么的？

**作用：启动一个本地预览服务器，用来跑「打包后的 dist 文件夹」**
也就是：**跑已经 build 好的上线代码**，不是跑源码。

---

#### 2. 三个命令最清晰对比（必看）

##### ① `npm run dev`

- **跑源码**
- 开发时用
- 热更新（改代码自动刷新）
- 不压缩、方便调试
- 对应：**pnpm dev**

##### ② `npm run build`

- **打包源码 → 生成 dist 文件夹**
- 压缩、混淆、优化
- 用于上线
- 对应：**pnpm build**

##### ③ `npm run serve`

- **运行 dist 文件夹**
- 预览**最终上线效果**
- 没有热更新
- 用来检查打包是否正常
- 对应：**pnpm serve**

---

#### 3. 最标准的工作流（你以后就按这个来）

```
1. 写代码 → pnpm dev
2. 写完打包 → pnpm build
3. 检查打包结果是否正常 → pnpm serve
4. 没问题 → 上线 dist
```

---

#### 4. 为什么有的项目是 dev，有的是 serve？

因为**框架不同**：

- **Vue CLI 老项目**：`npm run serve`
- **Vite / Vue 3 / React / 新项目**：`npm run dev`
- 功能**完全一样**，都是启动开发服务器

---

#### 5. 对应 pnpm 命令（直接记）

- `npm run dev`    →   `pnpm dev`
- `npm run build`  →   `pnpm build`
- `npm run serve`  →   `pnpm serve`

---

#### 一句话总结

- **dev = 开发跑源码**
- **build = 打包生成 dist**
- **serve = 预览 dist（上线前检查）**

#### 三、npm 两个核心文件（必须懂）

##### 1. `package.json`（项目配置清单）

**所有项目的灵魂文件**，记录：
- 项目名、版本
- **项目依赖 dependencies**
- **开发依赖 devDependencies**
- **运行脚本 scripts**（npm run dev）

示例结构：
```json
{
  "name": "my-project",
  "version": "1.0.0",
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  },
  "dependencies": {
    "vue": "^3.4.0"
  },
  "devDependencies": {
    "vite": "^5.0.0"
  }
}
```

##### 2. `package-lock.json`（锁定版本文件）

- **自动生成**，不要手动改
- **锁定所有依赖版本**，保证团队/服务器安装的版本完全一致
- 提升安装速度

#### 一、pnpm 是什么？

- 作者：Zoltan Kochan（2016 年发布）
- 全称：**Performance NPM**（高性能 npm）
- 官网：[pnpm.io](https://pnpm.io)
- 核心：**硬链接 + 符号链接 + 内容寻址存储（CAS）**

解决 npm/yarn 三大痛点：
1. **磁盘浪费**：每个项目复制完整依赖
2. **安装慢**：重复下载、解压、复制
3. **幽灵依赖**：能引用未声明的依赖，导致崩溃

---

#### 二、核心原理（必须懂，面试高频）

##### 1. 全局内容寻址存储（~/.pnpm-store）

- 所有包**按内容哈希（SHA-512）存一份**，**全网唯一**
- 相同版本包只存一次，**10 个项目省 70%+ 空间**
- 路径示例：`~/.pnpm-store/v3/files/sha512-xxx`

##### 2. 硬链接（Hard Link）

- 项目依赖**不复制，只硬链接到全局存储**，几乎不占空间
- 所有项目共享同一份物理文件

##### 3. 符号链接（Symlink）构建 node_modules

- 项目 `node_modules` 结构**非常干净**：
  ```
  node_modules
  ├── react -> .pnpm/react@18.2.0/node_modules/react （软链接）
  ├── vue -> .pnpm/vue@3.4.0/node_modules/vue          （软链接）
  └── .pnpm/ （硬链接真实文件区）
  ```
- **严格隔离**：只能访问 `package.json` 显式声明的依赖，**彻底杜绝幽灵依赖**

##### 4. 工作原理一句话

**全局存一份 → 项目硬链接 → 软链接暴露声明依赖 → 极速+省空间+安全**

#### 一、Bun 是什么？

- **全称**：无官方全称，就叫 Bun（发音 /bʌn/）
- **官网**：[bun.com](https://bun.com)
- **内核**：基于 **Safari 的 JavaScriptCore 引擎**（不是 V8），天生更快
- **核心定位**：**替代 Node.js + npm + Webpack + Jest**，一套工具全搞定
- **支持平台**：macOS、Linux、Windows（WSL 推荐，原生 Windows 支持一般）

##### Bun 的四大核心能力

1. **运行时**：替代 Node.js，启动快、性能高
2. **包管理器**：`bun install`，**比 npm 快 30 倍**，比 pnpm 更快
3. **打包器**：`bun build`，零配置打包，比 Webpack 快 10–100 倍
4. **测试运行器**：`bun test`，替代 Jest/Vitest，内置覆盖率

#### 二、Bun 包管理器核心原理（为什么这么快）

##### 1. 全局二进制缓存（~/.bun/install/cache）

- 所有包**按内容哈希存一份**，全局唯一，多项目共享
- 相同版本包只存一次，**磁盘占用极低**（接近 pnpm）

##### 2. 硬链接 + 写时复制（Copy-on-Write）

- macOS：用 **clonefile 系统调用**，瞬间复制，不占空间
- Linux：用 **硬链接**，几乎零开销
- 比 pnpm 的软链接**更快、更底层**，系统调用更少

##### 3. 并行解析 + 并行下载（核心提速）

- **并行解析依赖树**（npm/pnpm 默认串行）
- **HTTP/2 多路复用**，同时下载多个包
- 安装时间：**Bun 0.4s vs pnpm 6.4s vs npm 10.6s**（实测 Remix 项目）

##### 4. 零幽灵依赖 + 严格隔离

- `node_modules` 结构干净，**只能访问声明依赖**
- 默认禁用 `postinstall` 脚本，**安全性更高**

##### 5. bun.lockb（二进制锁文件）

- 替代 `package-lock.json` / `pnpm-lock.yaml`
- **二进制格式**，解析速度比 JSON/YAML 快 100 倍
- 锁定精确版本，保证团队环境一致

#### 五、Bun 五大核心优势（碾压 npm/pnpm）

##### ⚡ 1. 安装速度**全球最快**（比 npm 快 30 倍）

- 并行解析 + 并行下载 + 硬链接，**大型项目 1 秒内完成**
- CI/CD 环境优势巨大，**显著缩短构建时间**

##### 💾 2. 磁盘占用**极低**（接近 pnpm）

- 全局单例缓存，**多项目共享**，几乎无冗余
- 比 npm 节省 70%–90% 空间

##### 🛡️ 3. **更安全**（默认禁用 postinstall）

- 阻止恶意脚本执行，**减少 supply chain 攻击风险**
- 严格依赖隔离，**无幽灵依赖**

##### 🧩 4. **全能工具链**（一套搞定所有）

- 无需安装 Node.js、Webpack、Jest，**开箱即用**
- 原生支持 TypeScript、JSX、.env 文件，**零配置**

##### 🚀 5. **运行时性能极强**（比 Node.js 快 3 倍）

- JavaScriptCore 引擎，启动快、内存占用低
- HTTP 性能碾压 Node.js（Express 测试：59k vs 19k 请求/秒）

#### 六、Bun 缺点（2026 现状）

##### ❌ 1. **生态尚未完全成熟**

- 部分 Node.js 原生模块（如 `child_process`、`net`）兼容有问题
- 小众库可能不支持，**大型企业项目需谨慎**

##### ❌ 2. **Windows 原生支持一般**

- 推荐用 **WSL2**，原生 Windows 可能有兼容性问题

##### ❌ 3. **社区与文档相对较少**

- 遇到问题时，Stack Overflow 解答比 npm/pnpm 少

##### ❌ 4. **Monorepo 支持不如 pnpm 成熟**

- 支持 workspaces，但**高级功能（如版本自动管理）较弱**

#### 八、适用场景（什么时候用 Bun）

##### ✅ 推荐用 Bun

- 新项目（追求极速开发体验）
- 全栈项目（前端 + 后端，一套工具）
- TypeScript 项目（零配置支持）
- CI/CD 环境（大幅缩短构建时间）
- 学习/个人项目（体验新技术）

##### ❌ 不推荐用 Bun

- 大型企业级项目（依赖复杂原生模块）
- 纯 Windows 环境（无 WSL）
- 重度 Monorepo 项目（pnpm 更成熟）
