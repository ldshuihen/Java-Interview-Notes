# LangChain 

LangChain 是一个构建 LLM 应用的模块化框架，整体可从**包结构**、**核心组件**、**高层编排**三条线来理解。



## 一、包结构分类（工程层面）

- **langchain-core**：最底层核心，定义基础接口、Runnable、消息模型、LCEL 原语。
- **langchain**：高层组件（ Chains、Agents、Memory 等）。
- **langchain-community**：社区维护的第三方集成（各种模型、向量库、工具）。
- **langchain-openai / langchain-anthropic / …**：官方维护的轻量专属包。
- **langgraph**：图式编排，用于复杂、有状态的多轮流程。
- **langserve**：将链/模型部署为 REST API。
- **langsmith**：调试、评估、监控平台。

### 1. langchain-core

- **定位**：底层核心基石包
- **核心能力**
  定义统一抽象接口、消息结构体、**Runnable 规范**、LCEL 表达式原语、通用数据模型
- **特点**：轻量化、无第三方强依赖，所有上层包都依赖它

### 2. langchain

- **定位**：传统高层聚合包
- **核心能力**
  内置经典高层组件：Chain、Agent、Memory、Prompt 等老式封装
- **现状**：新版本逐步弱化，官方推荐**拆分独立模型包 + LCEL** 替代

### 3. langchain-community

- **定位**：社区通用集成仓
- **核心能力**
  海量第三方组件：小众大模型、向量数据库、文档加载器、工具、外部连接器
- **特点**：开源社区维护，组件多但稳定性参差不齐

### 4. langchain-xxx 系列（langchain-openai、langchain-anthropic）

- **定位**：官方独立轻量化模型包
- **核心能力**
  各大厂商模型专属封装（ChatModel、Embedding、工具调用）
- **优势**：解耦、按需引入、版本独立迭代、官方维护更稳定

### 5. langgraph

- **定位**：有状态流程编排框架
- **核心能力**
  基于图结构，支持**分支、循环、状态持久化、多轮迭代**
- **适用场景**：复杂 Agent、多步骤任务、人机交互、长链路业务流程

### 6. langserve

- **定位**：工程化部署组件
- **核心能力**
  将 Chain、Runnable、Agent 快速发布为 HTTP/REST API，提供标准化调用
- **适用场景**：LLM 应用后端服务化、前后端分离、微服务集成

### 7. langsmith

- **定位**：全链路可观测平台
- **核心能力**
  链路追踪、日志查看、Prompt 调试、效果评估、性能监控、异常排查
- **适用场景**：LLM 应用线上运维、问题排查、迭代优化

---

### 补充关键知识点（面试高频）

1. **LangChain 架构演进**
旧版：统一大而全 `langchain` 包
新版：**核心拆分 + 模型独立包 + 社区包**，轻量化、解耦设计

2. **核心依赖关系**
所有业务包（openai/community/langgraph）**都依赖 langchain-core**

3. **开发选型建议**
- 简单 RAG/对话：`langchain-core` + 对应模型包
- 多数据源/小众组件：引入 `langchain-community`
- 复杂智能体、循环任务：搭配 `langgraph`
- 线上部署：搭配 `langserve + langsmith`

## 二、六大核心组件（功能层面）

#### 1. Model I/O（模型输入输出）

负责与大模型交互的统一层。
- **LLM**：输入字符串 → 输出字符串（旧式模型）。
- **Chat Model**：输入消息列表 → 输出消息（ChatGPT 类）。
- **Embeddings**：文本 → 向量，用于语义检索。
- **Prompt Templates**：动态构建提示词。
- **Output Parsers**：把 LLM 输出转成 JSON/对象等结构化数据。

#### 2. Retrieval（检索 / RAG 相关）
把私有数据接入 LLM 的核心流水线。
- **Document Loaders**：加载 PDF/Markdown/数据库/网页等。
- **Text Splitters**：长文档切分（递归字符、语义切分等）。
- **Vector Stores**：向量库（Chroma、FAISS、Pinecone 等）。
- **Retrievers**：从向量库/知识库召回相关片段。

#### 3. Chains（链）
把多个组件“串”成可复用流程。
- **LLMChain**：Prompt + LLM 的最基础链。
- **RetrievalQAChain**：RAG 问答链（检索 → 拼上下文 → 问答）。
- **SequentialChain**：串行多步骤。
- **LCEL（Runnable）**：现代写法，用 `|` 组合任意组件。

#### 4. Memory（记忆）
给无状态 LLM 加上对话上下文。
- **ConversationBufferMemory**：保存全部历史。
- **ConversationBufferWindowMemory**：只保留最近 K 轮。
- **ConversationSummaryMemory**：用摘要压缩历史，解决长会话超限。
- **VectorStoreRetrieverMemory**：把历史向量化召回，适合超长记忆。

#### 5. Agents（智能代理）
让 LLM 能**自主决策、调用工具、循环执行**。
- **Tools**：搜索、计算器、API、数据库查询等能力。
- **ReAct Agent**：思考 → 行动 → 观察 → 再思考。
- **Tool Calling Agent**：利用模型原生函数调用能力。
- **Plan-and-Execute**：先规划任务，再分步执行。

#### 6. Callbacks（回调）
贯穿全链路的**事件钩子**，用于日志、监控、流式输出、评估。
- 事件：`on_llm_start`、`on_chain_end`、`on_tool_start` 等。
- 用途：埋点、耗时统计、可视化调试、LangSmith 集成。

### 1. Model I/O 模型输入输出

**核心作用**：统一封装大模型交互，标准化入参、出参、提示词、结构化解析。
- LLM：纯文本输入输出，传统补全模型。
- Chat Model：消息列表结构（System/Human/AI），对话式大模型主流。
- Embeddings：文本转向量，为RAG语义检索提供基础。
- Prompt Templates：动态拼接提示词，实现变量注入、话术复用。
- Output Parsers：解析大模型自由文本，转为 JSON、实体、对象等结构化数据。

### 2. Retrieval 检索（RAG核心）

**核心作用**：私有/外部数据接入大模型，解决知识时效性、私有数据问答。
- Document Loaders：多格式文档加载（PDF、网页、数据库、本地文件）。
- Text Splitters：长文本切块，解决上下文限制，包含递归分割、语义分割。
- Vector Stores：向量数据库，存储文本向量，提供相似度检索。
- Retrievers：检索器，封装向量库查询逻辑，召回相似度最高的上下文片段。

### 3. Chains 链路编排

**核心作用**：组合多个组件，封装成完整、可复用的业务流程。
- LLMChain：最基础链路，Prompt + 大模型组合。
- RetrievalQAChain：标准RAG链路，检索上下文+拼接提示词+问答生成。
- SequentialChain：串行执行多段任务，多步骤流水线。
- LCEL/Runnable：新版核心写法，管道符`|`链式组合，灵活、易扩展、支持流式。

### 4. Memory 对话记忆

**核心作用**：弥补大模型无状态特性，维护对话上下文。
- ConversationBufferMemory：存储完整对话历史，适合短对话。
- ConversationBufferWindowMemory：滑动窗口，只保留最近N轮，控制token。
- ConversationSummaryMemory：摘要压缩历史，解决长对话上下文溢出。
- VectorStoreRetrieverMemory：对话历史向量化存储，适配超长会话记忆。

### 5. Agents 智能代理

**核心作用**：赋予大模型自主思考、决策、调用外部工具的能力。
- Tools：外部能力扩展（联网搜索、数据库、接口、计算器等）。
- ReAct Agent：思考-行动-观察循环推理，通用型代理。
- Tool Calling Agent：依赖模型原生函数调用，精准匹配工具。
- Plan-and-Execute：先整体规划、再分步执行，适配复杂长任务。

### 6. Callbacks 回调机制

**核心作用**：全链路事件监听，贯穿模型、链、代理、工具全流程。
- 核心事件：模型开始/结束、链路执行、工具调用等钩子方法。
- 实际用途：日志打印、耗时监控、流式输出、链路追踪、LangSmith 调试。

## 三、高层编排范式（应用模式）

- **RAG（检索增强生成）**：私有文档 → 向量库 → 检索 → 问答。
- **Agent + Tools**：自主解决需外部信息/操作的问题。
- **Multi-Agent**：多个 Agent 分工协作（如 Planner + Executor）。
- **LangGraph 工作流**：循环、分支、状态持久化的复杂流程。

### 1. RAG 检索增强生成

**核心流程**：私有文档加载 → 文本分割 → 向量嵌入 → 存入向量库 → 相似度检索 → 上下文拼接 → LLM 问答。
**核心价值**：解决大模型**知识滞后、私有数据盲区、幻觉**问题，是企业最常用落地方案。

### 2. Agent + Tools 智能工具调用
**核心逻辑**：LLM 自主思考判断，按需调用外部工具（搜索、数据库、接口、计算等），循环迭代直至任务完成。
**核心价值**：突破模型能力边界，完成实时查询、数据操作、复杂计算等外部行为。

### 3. Multi-Agent 多智能体协作
**核心逻辑**：多个Agent分工解耦，典型架构：**规划Agent(Planner) + 执行Agent(Executor)**，各司其职、协同完成复杂任务。
**核心价值**：拆解复杂业务，职责隔离，降低单Agent上下文压力与决策难度。

### 4. LangGraph 图工作流编排
**核心逻辑**：基于图结构管理全局状态，天然支持**条件分支、循环迭代、状态持久化、人机介入**。
**核心价值**：适配超长链路、复杂决策、多轮交互、需要断点续跑的高阶 LLM 应用。

# LangGraph

LangGraph 是 LangChain 生态**有状态、图式编排框架**，核心用来解决：普通 Chain/Agent 无法实现的**分支、循环、持久化、人机干预、复杂多轮**流程。

## 一、核心定位分类
1. **编排模式分类**
- 线性编排：普通 Chain、LCEL，单向顺序执行，无分支循环
- 图式编排：LangGraph，**节点+边+全局状态**，支持分支、循环、回退

2. **核心特性分类**
- 有状态：全局 State 贯穿全流程，数据共享、持久化
- 可控性：可中断、人工审核、断点续跑
- 可迭代：天然支持 Agent 思考-行动循环

---

## 二、核心组成要素（底层结构分类）
1. **State 状态**
全局共享数据容器，分为：
- 基础 State：字典结构，存储对话、问题、上下文
- 结构化 State：自定义 Pydantic 模型，强类型、复杂字段
- 累加式 State：列表追加、覆盖更新、合并更新策略

2. **Nodes 节点**
流程中的最小执行单元，分类：
- LLM 节点：调用大模型、生成决策
- Tool 节点：调用工具、外部接口
- 自定义节点：业务逻辑、数据处理、格式解析
- 人机节点：人工确认、输入干预

3. **Edges 边**
控制节点流转逻辑，分为：
- 普通边：固定流向 A→B
- 条件边：根据 State 字段判断分支走向（if/else）
- 循环边：回到上游节点，实现多轮迭代

---

## 三、主流应用架构分类
### 1. 基础单图流程
简单顺序+少量分支，适合：简单问答、条件判断类任务

### 2. 循环 Agent 架构（最常用）
经典 ReAct 闭环：
`LLM决策节点` → 条件判断 → `工具调用节点` → 回到 LLM 节点
解决：工具调用、多轮推理、自主迭代任务

### 3. 多智能体 Multi-Agent 架构
- 分工型：Planner规划节点 + Executor执行节点
- 协作型：评审Agent、写作Agent、检索Agent多角色流转
- 适合：复杂长任务、文案生成、数据分析、项目拆解

### 4. 人机协同工作流
内置 `interrupt` 断点机制：
自动暂停 → 人工审核/修改 → 恢复执行
适合：敏感操作、审批流程、高危工具调用

### 5. 长链路复杂业务流
嵌套子图、状态持久化、多步骤任务编排
适合：企业级复杂 LLM 业务、自动化流程

---

## 四、运行模式分类
1. **同步模式**：阻塞执行，适合后端简单接口
2. **异步模式**：async 异步，高并发服务推荐
3. **流式模式**：节点分步流式输出，前端实时展示
4. **持久化模式**：结合 Checkpointer，保存会话状态，断线续跑

---

## 五、与传统 Chain / Agent 区别（面试高频）
1. Chain：线性、无状态、无循环分支，适合简单流程
2. 原生 Agent：黑盒循环，自由度低、不可控
3. LangGraph：**可视化流程、状态可控、逻辑透明、支持人工介入**，企业复杂应用首选

---

## 极简背诵总结
1. 三大核心要素：**State状态、Node节点、Edge边**
2. 三大核心能力：**分支、循环、有状态**
3. 四大典型场景：**循环Agent、多智能体、人机交互、复杂业务编排**
4. 核心优势：流程透明、可控可中断、支持持久化，弥补原生 Agent 短板

# RAG

## 一、按架构层级分类
### 1. 基础 RAG（朴素RAG）
流程：文档加载 → 切块 → 向量化 → 向量库存储 → 相似度检索 → 上下文拼接 → LLM生成
特点：架构简单、易落地、成本低
缺点：检索精度差、上下文无关、容易幻觉

### 2. 进阶 RAG
在基础版上做优化：
- 优化切块：语义分割、分层切块
- 优化检索：混合检索、重排序
- 优化生成：上下文压缩、Prompt 约束

### 3. 高级 RAG（企业级）
包含：路由、改写、自省、多轮优化、多数据源融合
代表：**自适应RAG、自省RAG、模块化RAG**

---

## 二、按检索策略分类
### 1. 语义检索（向量检索）
基于 Embedding 向量相似度匹配，解决**语义相似**，不依赖关键词。

### 2. 关键词检索
基于 BM25、分词匹配，擅长**精确词条、专有名词**检索。

### 3. 混合检索（Hybrid RAG）
向量语义检索 + 关键词检索 结合，兼顾语义+精准匹配，工业界主流。

### 4. 重排序 RAG（Rerank）
先粗召回多条片段，再通过 Rerank 模型打分筛选高相关内容，大幅提升精度。

---

## 三、按文档切分粒度分类
### 1. 固定切块
字符分割、递归字符分割
优点：简单高效
缺点：容易割裂语义

### 2. 语义切块
依据句子语义边界分割，保证片段语义完整。

### 3. 分层切块 / 层级RAG
大块做摘要索引、小块做精细检索，兼顾全局信息+细节内容。

### 4. 细粒度切块
短句、实体级切片，适合问答、精准知识点查询。

---

## 四、按交互&问答形式分类
### 1. 单轮 RAG
单次问题、无上下文，一问一答。

### 2. 多轮 RAG
结合对话历史，**问题改写 + 上下文记忆**，实现多轮连贯问答。

### 3. 对话式 RAG
融合 Memory 机制，维护会话状态，适合智能客服、知识库问答。

---

## 五、按技术方案&增强能力分类
### 1. 改写 RAG（Query Rewrite）
用户问题模糊/简短时，LLM 自动改写、补全、拆分，提升检索命中率。

### 2. 路由 RAG（RAG Router）
根据问题类型，自动分发到不同知识库/检索器：
- 简单问题 → 直接LLM回答
- 专业知识 → 向量库检索
- 数据查询 → 数据库检索

### 3. 自省 RAG（Self-RAG）
模型自行判断：
- 是否需要检索
- 检索内容是否够用
- 答案是否合理、是否需要二次检索

### 4. 多模态 RAG
支持文本+图片+表格+图文混合文档检索与问答。

---

## 六、按数据存储&索引方式分类
### 1. 稠密向量 RAG
基于 Embedding 稠密向量，主流方案（FAISS、Chroma、Milvus、ES向量）。

### 2. 稀疏向量 RAG
基于词袋、稀疏嵌入，适合关键词强场景。

### 3. 图谱 RAG（Graph RAG）
结合知识图谱，存储实体&关系，解决**逻辑推理、关联查询**。

### 4. 数据库 RAG
结构化数据（MySQL）+ 非结构化文档联合检索，适配业务系统数据问答。

---

## 七、按部署规模分类
1. **轻量RAG**：本地向量库、单机部署，个人/小项目
2. **企业级RAG**：分布式向量库、权限隔离、增量更新、监控运维
3. **分布式RAG**：多数据源、分片索引、高并发查询

---

## 极简面试总结（背诵）
1. 架构：**基础RAG → 进阶RAG → 高级RAG**
2. 检索：**向量、关键词、混合、重排序**
3. 切块：**固定、语义、分层、细粒度**

# LlamaIndex 

LlamaIndex 定位是**大模型数据层框架**，核心强项：**索引体系丰富、数据接入强、检索策略灵活、原生支持复杂RAG**。下面从**核心定位、索引类型、核心组件、检索策略、应用模式、与LangChain区别**六大类整理。

---

## 一、核心定位分类（vs LangChain）
- **LangChain**：**流程编排框架** → 擅长 Chain/Agent/Graph 组装
- **LlamaIndex**：**数据索引框架** → 擅长**文档结构化、混合索引、深度检索、长文本处理**

核心差异：LangChain 管“怎么跑”，LlamaIndex 管“数据怎么找”。

---

## 二、核心索引类型分类（最重点，面试必背）
### 1. 向量索引（VectorStoreIndex）— 最常用
- 原理：文档切块→Node→Embedding→向量库（FAISS/Chroma/Milvus）
- 检索：问题向量相似度匹配，召回 Top-K
- 特点：**语义强、支持同义词/歧义、RAG标配**
- 适用：通用问答、语义检索、知识库


### 2. 树索引（TreeIndex）— 层级/长文档专用
- 原理：文档→多层摘要树（根→中间→叶子），高层存摘要、底层存细节
- 检索：自顶向下选分支，递归定位最相关叶子
- 特点：**长文档友好、全局+细节兼顾、摘要能力强**
- 适用：书籍、报告、论文、章节化文档


### 3. 摘要索引（SummaryIndex/ListIndex）— 顺序/全量遍历
- 原理：Node 线性链表存储，无复杂结构
- 检索：遍历所有节点（或前N个），适合小数据
- 特点：**最简单、开销小、顺序访问友好**
- 适用：短文、小批量文档、简单列表查询


### 4. 关键词索引（KeywordTableIndex）— 精准匹配
- 原理：提取关键词→倒排表（词→Node映射）
- 检索：关键词精确匹配（类似BM25）
- 特点：**专有名词/代码/数值强、语义弱**
- 适用：技术文档、API手册、含大量实体的文本


### 5. 知识图谱索引（PropertyGraphIndex）— 关系推理
- 原理：抽取实体+关系→图结构存储
- 检索：多跳关联查询、逻辑推理
- 特点：**解决“关系型问题”、可解释强**
- 适用：医疗、法律、金融、知识图谱场景

### 6. 多模态索引（MultiModalVectorStoreIndex）— 图文混合
- 原理：文本+图片→统一向量空间
- 检索：跨模态匹配（文搜图/图搜文）
- 适用：图文报告、PPT、扫描文档、多模态RAG

### 7. 组合索引（Composable Graph）— 企业级混合
- 原理：**向量+关键词+树+图谱** 任意组合
- 检索：路由分发、多索引结果融合
- 特点：**精度最高、适配复杂企业数据**
- 适用：大型知识库、多源异构数据

---

## 三、核心组件分类（模块化）
### 1. 数据连接层（Data Connectors）
- 功能：100+数据源接入→统一 Document 对象
- 类型：PDF/Markdown/Word/数据库/Notion/Slack/网页
- 特点：**加载能力极强、企业数据适配首选**

### 2. 节点与切块（Nodes/Chunking）
- Document → Node（带文本/向量/元数据/关系）
- 切块策略：固定字符、递归字符、语义切块、分层切块

### 3. 索引层（Indexes）— 见第二部分

### 4. 存储层（Stores）
- DocStore：原始 Node 存储
- IndexStore：索引元数据
- VectorStore：向量库对接

### 5. 查询引擎（Query Engines）
- 基础查询：单索引检索+生成
- 高级查询：多跳检索、递归检索、路由查询、重排序
- 响应合成：Refine/Compact/Tree Summarize 等模式

### 6. 记忆与Agent
- Memory：对话上下文管理
- Agents：工具调用+检索增强（比LangChain轻量）

---

## 四、检索策略分类（RAG核心优化）
### 1. 基础检索
- 向量检索（默认）
- 关键词检索（BM25）
- 混合检索（向量+关键词，工业界主流）

### 2. 高级检索（LlamaIndex强项）
- **递归检索（Recursive Retrieval）**：先查高层摘要→再查底层细节，长文档神器
- **句子窗口检索（Sentence-Window）**：召回句子+上下文窗口，减少语义割裂
- **元数据过滤（Metadata Filtering）**：按时间/作者/来源过滤，精准定位
- **重排序（Rerank）**：粗召回→Cross-Encoder 重打分→精选，提升精度

---

## 五、应用模式分类
### 1. 基础RAG（单索引）
- 流程：加载→切块→向量索引→检索→生成
- 适用：小型知识库、简单问答

### 2. 高级RAG（多索引/混合）
- 自适应路由：简单问题→直接LLM；复杂问题→检索；关系问题→图谱
- 自省RAG：自我判断是否需要检索、结果是否足够、是否二次检索

### 3. 长文档RAG（TreeIndex/递归检索）
- 处理书籍、论文、报告，支持全局总结+细节问答

### 4. 多模态RAG
- 图文混合文档问答、OCR+检索+生成

### 5. Agentic RAG
- 检索+工具调用+循环推理，解决复杂任务

# 知识图谱 

## 一、核心定义
知识图谱（Knowledge Graph）是一种**结构化语义知识库**，以「**实体-关系-实体**」三元组为核心，描述现实世界事物、属性与关联关系，用于**逻辑推理、关联检索、复杂问答**。

---

## 二、核心组成要素分类
### 1. 实体（Entity）
现实客观事物，图谱基本节点。
- 人物、部门、文档、商品、指标、业务模块等

### 2. 关系（Relation）
实体与实体之间的关联边。
- 从属、包含、属于、负责、关联、时序、因果等

### 3. 属性（Property）
实体自身特征描述。
- 名称、时间、类型、状态、编号、简介等

### 4. 三元组（Triple）
图谱最小存储单元：
**头实体 + 关系 + 尾实体**

---

## 三、知识图谱架构分层分类
1. **数据源层**
结构化数据、非结构化文本、文档、表格、日志、业务数据库。
2. **知识抽取层**
从原始数据自动提取实体、关系、属性。
3. **知识融合层**
实体消歧、实体对齐、冲突消解、数据合并。
4. **知识存储层**
图数据库、向量库、关系型数据库混合存储。
5. **知识推理层**
规则推理、向量语义推理、多跳关联推理。
6. **应用服务层**
图谱问答、智能检索、关联分析、RAG增强、决策辅助。

---

- 实体对齐

3. **知识加工**
